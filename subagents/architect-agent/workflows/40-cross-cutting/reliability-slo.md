# reliability-slo

**Group:** 40-cross-cutting  ·  **Runs as:** subagent: ../.claude/agents/reliability-architect.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
Set the reliability **targets** the rest of the architecture is held to, and prove the system can meet them. Turn "it should be reliable" into a measurable contract: an **SLI** (what we measure), an **SLO** (the target on that SLI), an **error budget** (the allowed unreliability that gates change), and — where the SLO is external-facing — an **SLA**. Then size **recoverability**: state the **RTO** (how fast we restore service after a disaster) and **RPO** (how much data loss we tolerate), pick a **DR strategy** off the ladder (backup-restore → pilot-light → warm-standby → hot/multi-site active-active) that demonstrably meets those numbers, and justify the tier's cost against the driver. This flow owns the *recoverability + availability-target* leaves of the utility tree; `availability-resilience.md` owns the *structural* means (redundancy, no-SPOF, degradation) that the SLO consumes. The design question: **what are our reliability numbers, how do we measure them, and what recovery tier meets the RTO/RPO at a justified cost?**

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target (per the config — e.g. a single-region deployment with a single-primary data store and **no replica, no managed DB, no second region yet**, or whatever topology the config states), the scale/SLO seed, the compliance regime (e.g. PII or financial data, per the config), the artifact locations (`architecture/<slug>/`, the ADR log, the diagram source).
- The **drivers / utility tree** from `00-drivers/quality-attribute-scenarios.md` — specifically the *Reliability › Availability* and *Reliability › Recoverability* leaves, each a 6-part scenario ending in a response measure. The config flags where these are **not yet stated in the source-of-truth docs** — an unstated RTO/RPO/availability target is an **open driver question**, not an assumption.
- Preconditions: the utility tree exists (no drivers → downgrade to `status:error`, surface the open questions, author nothing); the deployment topology (or its candidate) from `../50-infrastructure/deployment-topology.md` is readable, since RTO/RPO are argued against a *topology* (single-AZ vs multi-AZ vs multi-region); the data stores and their backup posture are known (what is backed up, how often, where, and **whether a restore has ever been tested**).
- Hard rule from the config: **never invent an RTO/availability/RPO number.** If the source-of-truth docs (BRD / PRD / understanding log, per `project-architecture-config.md`) don't state it, this flow's primary output is the *open driver question with a recommended starting point and its cost*, for the controller to confirm with the user — not a guessed nine.

## Oracle (source of truth)
The reliability **contract** — SLI/SLO/error-budget per availability scenario, and RTO/RPO with a DR strategy that provably meets them — traced to a stated business goal or a documented NFR (the SRE model; the DR-strategy ladder by RTO/RPO+cost). "It's reliable / it's highly available / it won't lose data" is **not** an oracle; "99.5% monthly availability measured as `successful_core_API_requests / total` at the edge proxy, error budget 3h 39m/month; RTO ≤ 4h, RPO ≤ 15min, met by a warm-standby restored from a 15-min PITR backup" is. No number + no traced driver = a preference, not a finding (`shared/quality-attribute-rubric.md`).

**hard:**
- **Every reliability scenario has an SLI + an SLO + a measurement method.** Each availability/latency-reliability scenario names a precise **SLI** (a ratio of good events to valid events — e.g. `availability = good_requests / valid_requests`), an **SLO** target on it (`≥ 99.5% over 28 days`), and *where/how it is measured* (the edge proxy, the synthetic probe, the request log) per `observability-architecture.md`. An SLO with no SLI, or an SLI with no measurement point, blocks — it is unmeasurable, therefore unenforceable.
- **Every SLO has an error budget, and the budget gates change.** `error_budget = 1 − SLO` is computed (a number: minutes/month or a failure ratio), and there is a stated **budget policy** — what happens when it's exhausted (freeze risky change / shift to reliability work). An SLO with no derived budget, or a budget that gates nothing, blocks (an SLO no one is held to is decoration).
- **RTO and RPO are stated, and the DR strategy demonstrably meets them.** Both numbers exist (traced to a driver, not invented), and the chosen DR tier's *recovery-time math* is shown to satisfy them — restore-from-backup time < RTO, backup interval / replication lag ≤ RPO. A DR tier asserted to "meet RTO" with no recovery-time estimate blocks; an RTO/RPO with no DR strategy behind it blocks.
- **Backups exist with a *tested* restore path.** Every stateful store on a recoverability scenario's critical path (the primary data store; any future store) has a defined backup (cadence, location — ideally a different failure domain, retention) **and a documented, exercised restore procedure**. "We take backups" with no tested restore is an **un-accepted SPOF on recovery** and blocks (an untested backup is a hope, not a control — see `shared/quality-attribute-rubric.md` → *No un-accepted SPOF*, *Failure mode defined*).
- **The DR tier is cost-justified against the RTO/RPO driver.** The tier carries an order-of-magnitude run-cost (CBAM: benefit = RTO/RPO utility gained × priority, cost = standby infra + ops), and the choice is the *cheapest tier that meets the numbers* — not the most resilient one available. A multi-site active-active design for a system whose driver says `RTO ≤ 4h` blocks as accidental complexity (`cost_bounded` + the right-sizedness rubric).

**soft:** the SLOs are **realistic and achievable** (an SLO tighter than the system's components can deliver, or tighter than the business needs, is theatre — 99.99% on a single-region single-primary-store stack is not real); the DR tier **matches** the RTO/RPO (no warm-standby where backup-restore meets a 24h RTO; no hot multi-site where a 4h RTO is the bar); the **error budget genuinely gates** change (the policy is honored, not nominal); the SLI is **user-centric** (measures what the user experiences — e.g. a successful order/checkout submission — not an internal proxy like CPU); the reliability targets are **tiered to the journey** (the core transactional path may carry a tighter SLO than an audit-log read).

## Standards & techniques
- **The SRE reliability model (SLI / SLO / SLA / error budget):** an **SLI** is a quantitative measure of service level, almost always a ratio `good_events / valid_events` (Google SRE: availability, latency, throughput, correctness as the canonical SLI types). An **SLO** is the target/range on an SLI over a window (`99.5% over 28 days`). An **SLA** is the *external, contractual* promise with consequences — set the SLO **tighter** than the SLA so the budget burns before the contract breaches. The **error budget** = `1 − SLO`, the allowed unreliability; it is the bridge between reliability and velocity — while budget remains, ship; when it's spent, the **budget policy** freezes risky change and redirects to reliability. Pick the window (rolling 28-day is the SRE default) and the SLI specification (event-based vs time-based) explicitly.
- **RTO / RPO (the recoverability definition):** **RTO** = *Recovery Time Objective* = the maximum tolerable time to restore service after a disaster (drives *how fast you fail over / restore*). **RPO** = *Recovery Point Objective* = the maximum tolerable **data loss**, measured as time (drives *how often you back up / how tightly you replicate*). RPO ≈ backup interval or replication lag; RTO ≈ detect + decide + restore + validate. These are *recoverability* response measures and become utility-tree leaves under *Reliability › Recoverability*.
- **The DR-strategy ladder (choose by RTO/RPO + cost — AWS DR whitepaper):** four tiers, cheapest/slowest → costliest/fastest:
  1. **Backup & restore** — periodic backups to a durable, separate failure domain; on disaster, *provision fresh + restore*. **RTO hours, RPO = backup interval.** Cheapest (storage only). Right when the RTO is hours and some data loss is tolerable.
  2. **Pilot light** — core data continuously replicated to a minimal, *switched-off* standby (DB replica live, app tier dormant); on disaster, *scale up the dormant tier*. **RTO tens of minutes, RPO low (replication lag).** Modest cost (data replication + minimal idle infra).
  3. **Warm standby** — a scaled-down but *running* full copy in the recovery site; on disaster, *scale it to full + cut over*. **RTO minutes, RPO seconds–minutes.** Higher cost (always-on, reduced capacity).
  4. **Hot / multi-site active-active** — full capacity running in ≥2 sites serving live traffic; on disaster, *route away* (often zero manual failover). **RTO ≈ 0, RPO ≈ 0.** Costliest (≥2× infra + cross-site consistency complexity → PACELC tradeoff). Justified only by a near-zero-RTO driver.
  The deliverable is **the tier whose RTO/RPO matches the driver at the lowest cost** — climbing the ladder buys speed at super-linear cost; do not climb past the driver.
- **Backup strategy + restore testing:** define cadence (full + incremental / continuous PITR), **retention** (and where a compliance regime sets it — e.g. PII or financial-record retention, per the config, is a driver here), **location** (a *different* failure domain than the primary — a backup in the same AZ as the primary that the disaster takes out is no backup), and the **3-2-1 heuristic** as a baseline (3 copies, 2 media, 1 off-site). The **restore test** is the actual oracle: a backup is only proven by a periodic, timed restore drill whose elapsed time validates the RTO and whose recovered data validates the RPO. An untested backup does not count toward the hard oracle.
- **Graceful-degradation tie-in:** the SLO is met not only by *not failing* but by *failing partially well*. Cross-link `availability-resilience.md` — when a downstream (a deferred integration seam, an external payment or document-processing provider) is down, degrade the dependent journey (read-only mode, queued action, cached view) rather than fail the whole request, so the SLI's "good events" ratio survives a partial outage. The degradation modes are *named per scenario* and become part of the SLI's definition of a "good" event.
- **Composing the SLO from the dependency chain:** an SLO is bounded by the *product* of its hard dependencies' availabilities (a request that needs the API **and** the primary data store can be no more available than `A_api × A_store`); a same-origin single-primary-store stack's realistic ceiling is the single primary's availability minus the restore window — surface this when an SLO is set above what the topology can deliver (the *realistic SLO* soft oracle).
- **MADR for every target as a decision:** the availability SLO, the RTO/RPO, and the DR-tier choice are each architecturally-significant decisions recorded as ADRs (`shared/adr-template.md`) — context (the driver), the options (the ladder rungs / candidate nines), the decision, and the consequences (the cost, the ops burden, the budget policy). One-way-door flag where the DR tier locks topology (a multi-region choice reshapes data residency + consistency — flag it).

## Step sequence
- **audit:** Frame the reliability drivers from the utility tree (the availability + recoverability leaves) → for each, check the **hard structural facts**: is there an SLI with a measurement point? an SLO with a derived error budget and a budget policy? are RTO/RPO stated and traced (or are they open questions the source-of-truth docs never answered)? does the *current* topology meet them — walk the project's actual stack (per `project-architecture-config.md`) against each recoverability scenario (what is the real RTO of "the primary data store is lost" today? what is the RPO of the current backup cadence — or is there no backup?) → confirm a *tested* restore path exists (flag "backups taken, restore never tested" as an un-accepted recovery SPOF) → check the implied DR tier is cost-matched to the driver (or over/under-built) → emit findings + a risk register (read-only; authors nothing, edits no source, runs no restore). The headline audit findings on a young/single-region system are typically **open driver questions** (no stated RTO/RPO/availability) and **structural gaps** (a single-primary data store = unbounded RTO on primary loss; backup posture undefined).
- **design:** **Frame** (extract/confirm the availability + recoverability scenarios; where the number is unstated, *propose a starting target with its cost* as the open question — never silently assume; read-only) → **Explore** (≥2 divergent DR directions as concrete specs — e.g. *Direction A: backup-restore with 15-min PITR to a managed store in another AZ, RTO ~2–4h*; *Direction B: pilot-light with a live replica + dormant app tier, RTO ~20min* — each with its RTO/RPO math and run-cost, read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each direction vs the RTO/RPO driver and the cost budget via ATAM+CBAM, order-swapped; names the **tradeoff point** — *replication lag vs RPO vs cost*, *cross-site consistency vs RTO* (PACELC) — and the **sensitivity point** — *RTO is sensitive to whether the restore is automated vs manual*; **penalizes climbing past the driver**) → **Document** (the **one writer** `solution-architect` folds the winner into `architecture/<slug>/`: the SLI/SLO/error-budget table in `quality-attributes.md`, the RTO/RPO + DR-tier ADR(s), a `runbook`-shaped recovery sequence + the DR deployment diagram as diagrams-as-code, and the **restore-test fitness function** spec — reuse the ladder + the SRE model, don't invent a bespoke scheme; preserve conceptual integrity) → **Verify** (hard gates: every scenario has SLI+SLO+budget; RTO/RPO stated + the DR math meets them; a tested-restore path is specified with a CI/periodic restore-drill fitness function; the cost model bounds the tier; diagrams compile; ADRs complete — then the evaluator re-scores realism + tier-match from the artifacts) → loop ≤ 10 or pass.

## Assertions & exit gate
- Every prioritized reliability scenario has an **SLI** (a good/valid ratio), an **SLO** (target + window), a **measurement method/point**, and a derived **error budget** with a stated **budget policy** that gates change.
- **RTO and RPO are stated** (traced to a driver, or explicitly surfaced as an open question with a recommended starting value + cost — never silently invented).
- The chosen **DR tier's recovery math meets the RTO/RPO** (restore/failover time < RTO; backup interval / replication lag ≤ RPO), and the tier is **the cheapest that meets the numbers** — no over-build (no multi-site for an hours-RTO system), no under-build (no backup-restore for a minutes-RTO driver).
- Every stateful store on a recoverability path has a **backup** (cadence, off-domain location, retention honoring the compliance regime) **and a tested, timed restore path** (an untested backup fails the gate).
- The DR tier carries an **order-of-magnitude cost** against the budget; the SLO is **achievable** given the dependency-chain availability ceiling; graceful-degradation modes are named per scenario (cross-linked to `availability-resilience.md`).
- Each reliability target (SLO, RTO/RPO, DR tier) is an **ADR** with context · options · decision · consequences (incl. cost + budget policy); diagrams (DR deployment + recovery sequence) **compile**; one-way-door tier choices flagged.
- **Gate:** hard oracles green (SLI+SLO+budget per scenario · RTO/RPO stated · DR math meets them · backup with tested restore · cost bounded · ADRs complete · diagrams compile) AND (design) rubric mean ≥ 0.8 (realistic SLOs · tier matches RTO/RPO · budget genuinely gates).

## Output
Write `artifacts/reliability-slo/report.json` per `shared/report-format.md` — findings/risks per `shared/finding-schema.md` (`type:"availability"` for SLO/budget gaps, a dedicated recoverability framing for RTO/RPO/DR, `type:"cost"` where the tier is over/under-built, `type:"risk"` for an untested-restore or single-primary recovery SPOF; `oracle:"hard"` for a scenario with no SLI/SLO/budget, an unstated-and-unsurfaced RTO/RPO, a DR tier with no recovery-time math, an untested restore path, or an unbounded DR cost; `driver_ref` to the `QAS-AVAIL-*`/`QAS-RECOV-*` scenario — a finding with no driver is dropped). Unstated reliability numbers (RTO, RPO, availability target) are listed as **open driver questions with a recommended starting target + its cost**, for the controller to confirm with the user — never assumed. The design-mode `verification` block carries the traceability (every reliability scenario → SLO/DR decision), the fitness-function results (the restore-drill spec, the budget-burn check), the hard-oracle results, and the rubric score. Authored artifacts land in `architecture/<slug>/` per `project-architecture-config.md`: the SLI/SLO/error-budget table in `quality-attributes.md`, the RTO/RPO + DR-tier ADR(s) in `adr/`, the DR-deployment + recovery-sequence diagrams in `diagrams/`, the cost line in `capacity-model.md`/`tradeoffs.md`, and the restore-drill + budget-burn checks in `fitness-functions.md` — never scattered.

## Guardrails
Per `shared/guardrails.md`: **propose & document the reliability contract — never write product code, never run a restore, never apply or touch a live store or backup.** This flow authors SLO definitions, RTO/RPO targets, DR-tier ADRs, recovery runbooks/diagrams, and restore-drill *fitness-function specs* into `architecture/<slug>/` (and a runnable drill check into the project's `tests/`/`fitness/` dir for the controller/operator to wire into CI) — it produces the *plan*; the operator runs the actual DR drill, the controller builds the instrumentation. **Ground every target in a driver** — no SLO/RTO/RPO without a stated business goal or NFR; if it's unstated, it's an **open question with a costed recommendation**, not a guessed number (the config forbids inventing nines). **Reuse the SRE model + the DR ladder** before inventing a bespoke reliability scheme; **right-size** — the cheapest tier that meets the driver wins, and propose the over-engineering risk if asked for a tier the driver doesn't demand (per the config's locked "right-size, don't résumé-drive" rule). Read-only audit authors nothing. **Record the tradeoff and the rejected ladder rungs** (RPO vs cost, cross-site consistency vs RTO per PACELC), never just the chosen tier. Never write to project memory — lessons go in the report and the ADRs.
