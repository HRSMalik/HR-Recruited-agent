# availability-resilience

**Group:** 40-cross-cutting  ·  **Runs as:** subagent: ../.claude/agents/reliability-architect.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
Design the system to **stay up under partial failure** — and to fail *gracefully* when it can't. Answers a specific decision: *for each availability/RTO scenario in the utility tree, where does this design break, what is redundant, what degrades, and what is the bounded blast radius — does the composite availability meet the target?* This is the **FMEA + redundancy + graceful-degradation + blast-radius** flow: every failure mode gets a mitigation or a signed accepted risk, no component on an availability-SLO path is a silent single point of failure, degraded-mode behavior is defined (not "it's down"), and the redundancy is **sized to the driver** — not gold-plated multi-region for a system whose drivers never asked for it. It owns the *availability* limb of the utility tree; `reliability-slo.md` owns the SLI/SLO/RTO/RPO *numbers* this flow designs against, and `../80-patterns/resilience-patterns.md` owns the *mechanisms* (circuit-breaker, bulkhead, retry, fallback) this flow selects and composes.

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment topology (the project's stack — per `project-architecture-config.md`; e.g. a modular-monolith pair — one backend service over a **single-primary datastore** behind a same-origin reverse proxy — single-AZ, no broker/queue today, or whatever the project's actual topology is), the team topology (small team — favor a recoverable monolith over an unoperable HA sprawl), the constraint that any dev-only compose/Makefile is not the prod-deploy topology and a real prod-deploy topology is a **separate ADR**, and where the architecture package lives (`architecture/<slug>/`).
- The **drivers / utility tree** (`00-drivers/quality-attribute-scenarios.md`): the prioritized 6-part **availability scenarios** ending in a measurable response — e.g. `QAS-AVAIL-02 (RTO ≤ 5 min on primary-DB loss)`, `QAS-AVAIL-03 (a core write action still completes during a downstream-integration outage)`, plus the SLO/RTO/RPO from `reliability-slo.md`. **A flow with no availability driver downgrades scope and says so** — you cannot size redundancy without the availability number it must satisfy. If the availability target / RTO / RPO are *not yet stated in the source-of-truth docs*, and the controller hasn't resolved them, surface them as **open driver questions** and do not assume a number (per `project-architecture-config.md` Reliability driver). Right-size: most internal/enterprise systems are not consumer-scale — do **not** design active-active multi-region absent a concrete driver.
- Target: the system/subsystem (or greenfield scope) under design/review, its critical-path components for each availability scenario, and every cross-process / external dependency (the primary datastore, any third-party integration, any auth/secret store).
- Preconditions: the source-of-truth docs read (the BRD/PRD/understanding-log locations named in `project-architecture-config.md`); a current deployment view (C4 Deployment) or enough topology to build one; the cross-service failure-handling contract from `../30-integration/service-communication-resilience.md` available so this flow doesn't re-derive timeout/retry policy.

## Oracle (source of truth)
The **availability quality-attribute scenarios** (each with a response measure) + the **FMEA register** (every failure mode → effect → mitigation-or-accepted-risk) + the **composite-availability computation** against the target. Never "it's resilient" / "it's highly available" with no number and no traced failure mode (the grounding rule, `../shared/quality-attribute-rubric.md`).

- **hard:**
  - **No un-accepted SPOF** — every component on the critical path of an availability-SLO scenario has redundancy/failover, OR the single point of failure is an **explicit, signed-off accepted risk** in the register with the scenario it threatens. A silent SPOF on an availability-SLO path **blocks** (the rubric's *No un-accepted SPOF* oracle). *Structural check:* for each `QAS-AVAIL-*`, trace its request path on the C4 Deployment view; any node with cardinality 1 and no failover is a SPOF — name it or accept it. (A single-primary datastore with no replica is exactly such a node — it is a SPOF for any RTO scenario until a replica-set/managed-failover decision or a recorded accepted risk exists.)
  - **Every failure mode has a disposition** — the FMEA covers each component + each cross-process/external call with a **mitigation** (redundancy, failover, retry, fallback, degraded mode) **or** an `accept/monitor` risk entry; no failure mode is silently undispositioned. *Traceable:* `failure_modes_identified == failure_modes_with_disposition`.
  - **Degraded-mode behavior is defined** — for every dependency whose loss does **not** take the whole system down, the design states *what still works and what is shed* (a named feature-toggle / fallback / read-only mode), not an implicit crash. "When a document-processing provider is down, intake still saves; extraction is queued and the field is marked pending" is a defined degraded mode; an unhandled 500 is not. *Structural:* each external/cross-process call on a non-fatal path names its degraded behavior (ties to the *Failure mode defined* oracle — bounded timeout + retry-with-backoff + fallback/fail-fast, never an unbounded synchronous chain).
  - **Composite availability computed against the target** — availability is **multiplicative across serial dependencies** (`A_system ≈ Π A_i` for components in series; redundant pairs raise a leg via `1 − (1 − A)ⁿ`). The design carries the back-of-envelope number in `capacity-model.md` and it **meets the target** (e.g. a serial chain of three 99.9% components yields ~99.7%, not 99.9% — if the driver demands 99.9% the chain must be shortened or a leg made redundant). A scenario whose computed availability < target with no redundancy decision **blocks** (the *Response-measure met* oracle).
  - **Blast radius bounded** — a single fault does not cascade system-wide: the design names its **bulkhead/cell** boundaries (connection-pool isolation per dependency, resource partitioning, load-shedding) so one saturated dependency can't exhaust the shared pool and take down unrelated paths. *Structural:* no single shared resource (one DB connection pool, one thread pool) fronts all critical paths with no isolation, where a driver requires one path to survive another's failure.
  - **Fallacies honored** — partial failure is the *normal* state (fallacy #1, `../shared/fallacies-and-tradeoffs.md`): the design has a partial-failure path for every remote/external call, not only a happy path. Assuming a dependency is always up **blocks**.
  - **ADR-shaped** — each redundancy / degradation / DR decision is an ADR (context · ≥2 options · decision · positive **and** negative consequences) traced to its `QAS-AVAIL-*` (`../shared/adr-template.md`).
- **soft:** resilience is **sized to the availability driver** (CBAM-checked, `evaluation-method.md`) — no multi-region / active-active where no scenario demands it; the redundancy ROI clears 1 against the *actual* availability target, not the maximum imaginable. Degradation is **graceful, not all-or-nothing** — the system sheds the failing feature and keeps the core working (a user can still read/work the core business entity when a downstream document-processing provider is down), rather than a single dependency loss blanking the app. Operability of the failure path — the degraded mode is observable (a health/readiness signal flips, the toggle state is visible) and runbook-able, not a silent half-broken state.

## Standards & techniques
- **FMEA (Failure Mode and Effects Analysis)** — enumerate, per component and per cross-process/external dependency: the **failure mode** (crash / hang / timeout / omission / slow / wrong response / partition), its **effect** on each availability scenario, its **cause**, and its **mitigation or accepted-risk**. Optionally rank by **RPN = Severity × Occurrence × Detection** to prioritize which modes get a structural mitigation vs an accepted/monitored risk. The FMEA register is the hard-oracle artifact: *every mode has a disposition.*
- **Availability math (the "nines" + dependency multiplication)** — translate the target to allowed downtime (99.9% ≈ 8.77 h/yr ≈ 43.8 min/mo; 99.99% ≈ 52.6 min/yr) and compute the **composite**: serial dependencies multiply (`Π A_i` — more hops = lower availability, the cost of distribution), redundancy within a leg raises it (`1 − (1 − A)ⁿ`). This is what proves a scenario's response-measure is met *by a model*, not asserted. Cross-link `reliability-slo.md` for the SLO↔error-budget translation.
- **Redundancy topologies** — **N+1** (spare capacity for one loss), **active-passive** (standby promoted on failover — cheaper, an RTO gap = detection + promotion time), **active-active** (all serve, no promotion gap — costlier, needs state/consistency handling). Choose by the RTO/RPO the scenario demands and the cost driver (CBAM); for a single-primary-datastore system the first move is usually a **replica set with automatic failover** (active-passive within an AZ/managed cluster), not a leap to multi-region.
- **SPOF elimination** — walk each availability-scenario request path on the **C4 Deployment view**; any node with no redundant peer and no failover is a SPOF. Each SPOF is either made redundant (with its cost tradeoff) or recorded as a **signed accepted risk** naming the scenario it threatens and the monitoring that detects it. Silent SPOFs are the cardinal availability failure.
- **Graceful degradation + feature toggles** — design **degraded modes**, not binary up/down: shed non-essential features under partial failure and keep the core path serving. A **feature toggle / kill-switch** lets the system (or an operator) disable a failing dependency's feature and fall back (cached/stale read, queued write, read-only mode, a "pending" placeholder) rather than cascade a 500. The fallback is itself a designed path with its own correctness bound (e.g. stale-read staleness window).
- **Blast-radius containment — bulkheads & cells** — partition resources so a fault is contained: **bulkhead** (isolate connection/thread pools per dependency so one slow dependency can't exhaust the pool shared with healthy paths — the ship-compartment metaphor), **cell-based / shuffle-sharding** (partition tenants/users across independent cells so one cell's failure affects a fraction, not all), **load-shedding / rate-limiting** at the edge to fail fast under overload rather than collapse. Mechanism details live in `../80-patterns/resilience-patterns.md`; this flow decides *where the boundaries go* against the blast-radius driver.
- **Chaos-engineering readiness** — the design is **structured to be tested under failure**: failure modes are *injectable* (a dependency can be killed/delayed in a non-prod env), steady-state is *measurable* (the SLI exists to detect a regression — ties to `observability-architecture`), and a hypothesis-driven experiment ("kill the primary datastore → failover completes within RTO, no data loss within RPO") can be run by `qa-agent`'s chaos/resilience verification. This flow does **not run** chaos experiments (that's QA against a running SUT) — it *specifies the failure-mode coverage and the steady-state hypotheses* as fitness-function-shaped checks for QA to execute.
- **Canonical tradeoffs** (`../shared/fallacies-and-tradeoffs.md`) named and *with the point chosen*: **Cost ↔ Resilience** (every redundant leg / region costs money — size to the driver), **Consistency ↔ Availability** (CAP/PACELC — active-active and async-replicated failover trade RPO/consistency for availability; state which per data store), **Latency ↔ Availability** (synchronous replication buys a smaller RPO at write-latency cost), **Simplicity ↔ Resilience** (each failover mechanism is operational surface area the small team must run — Conway).

## Step sequence
- **audit:** Frame the availability drivers (the `QAS-AVAIL-*` scenarios + their RTO/RPO from `reliability-slo.md`; if unstated, flag as an open driver question, don't assume) → build/read the **C4 Deployment view** and walk each scenario's request path → run an **FMEA** over every component + cross-process/external call (mode → effect → cause → existing disposition) → flag every **silent SPOF** on an SLO path, every failure mode with **no mitigation and no accepted risk**, every dependency with **no defined degraded mode** or an unbounded synchronous call, and any **shared resource with no bulkhead** → **compute composite availability** and compare to target → emit findings + a risk register (read-only, authors **no** artifact, edits **no** source).
- **design:** **Frame** (extract/confirm the availability scenarios + RTO/RPO + cost constraint; build the deployment view, read-only) → **Explore** (N divergent redundancy/degradation directions as concrete specs — e.g. *(a)* single-AZ datastore replica set + active-passive failover + degraded read-only mode; *(b)* managed multi-AZ cluster + bulkheaded pools per external dependency; *(c)* multi-region active-active — each with its topology, RTO/RPO, blast-radius boundary, and cost, read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each vs the availability scenarios + hard oracles via ATAM; finds the **sensitivity points** — e.g. RTO is sensitive to failover-detection time — and the **tradeoff points** — sync replication improves RPO but worsens write latency; runs **CBAM** so multi-region is rejected when its ROI < 1 against the actual driver; order-swapped to kill position bias; penalizes over-engineering) → **Document** (the **one** `solution-architect` writer authors: the **FMEA register** + **degraded-mode catalog** + **blast-radius/bulkhead boundaries** into `design-doc.md`, the composite-availability math into `capacity-model.md`, the **C4 Deployment view with redundancy + failure paths** into `diagrams/`, one **ADR per redundancy/degradation/DR decision** with its rejected alternatives, and the **failure-mode-coverage + steady-state-hypothesis fitness functions** into `fitness-functions.md`; reuses the resilience patterns from `../80-patterns/resilience-patterns.md`, preserves conceptual integrity) → **Verify** (the hard gates below as a deterministic check; the evaluator re-scores from the *artifacts*, not the prose) → loop ≤ 10 or pass.

## Assertions & exit gate
- Every availability-SLO scenario's request path is traced on the deployment view; **no node on it is a silent SPOF** — each is redundant/failover-capable or a **signed accepted-risk** entry naming the threatened scenario.
- **Every failure mode in the FMEA has a disposition** — a mitigation (redundancy/failover/retry/fallback/degraded-mode) or an `accept/monitor` risk; `identified == dispositioned`.
- **Degraded-mode behavior is defined** for every dependency whose loss is non-fatal (named fallback/toggle/read-only mode); every cross-process/external call has a **bounded timeout + retry-with-backoff + fallback/fail-fast** (no unbounded synchronous chain).
- **Composite availability is computed** in `capacity-model.md` (serial multiplication + redundancy uplift) and **meets the target** for each scenario; a scenario below target with no redundancy decision fails.
- **Blast radius is bounded** — bulkhead/cell boundaries named; no single shared pool fronts all critical paths where a driver requires one to survive another's failure.
- **Fallacies honored** (partial-failure path exists for every remote call); each redundancy/degradation/DR decision is a **complete ADR** (context · ≥2 options · decision · positive + negative consequences) traced to its `QAS-AVAIL-*`; the deployment diagram **compiles**; resilience is **right-sized** (no un-driven multi-region — CBAM ROI ≥ 1).
- **Gate:** hard oracles green (no un-accepted SPOF · every failure mode dispositioned · degraded modes defined · composite availability ≥ target · blast radius bounded · fallacies honored · ADRs complete · diagram compiles) **AND** (design) rubric mean ≥ 0.8.

## Output
Write `artifacts/availability-resilience/report.json` per `shared/report-format.md` — findings/risks per `shared/finding-schema.md` (`type:"availability"`, `iso25010:"Reliability › Availability"`/`"Recoverability"`/`"Fault Tolerance"`; `oracle:"hard"` for a silent SPOF on an SLO path, an undispositioned failure mode, a missing degraded mode / unbounded synchronous call, composite availability below target, or an unbounded blast radius; `oracle:"soft"` for over-sized redundancy / all-or-nothing degradation). Each finding carries its `driver_ref` (the `QAS-AVAIL-*`), the **traced structural fact** in `evidence` (the SPOF node + cardinality, the un-mitigated mode, the computed composite vs target), and a `tradeoff` + `alternatives_rejected` (e.g. "datastore replica set + active-passive failover raises cost ~Xx and adds failover/operability surface; buys RTO from *unbounded* → ≤ 5 min and RPO ≤ replication lag · multi-region active-active rejected: 3× cost, no driver demands it"). The verification block includes the FMEA-coverage count, the composite-availability computation, the SPOF-trace result, and the failure-mode-coverage fitness functions. A `type:"risk"` entry is emitted for each **accepted** SPOF/failure mode (likelihood × impact, disposition `accept/mitigate/monitor`, the scenario it threatens). Authored design artifacts (FMEA register, degraded-mode catalog, redundancy/DR ADRs, the C4 Deployment-with-failure-paths diagram, the availability math, the failure-mode fitness functions) land in **`architecture/<slug>/`** per `project-architecture-config.md`, never scattered. If the availability target / RTO / RPO is unstated, list it as an **open driver question** for the controller — never silently assume a number.

## Guardrails
Per `shared/guardrails.md`: **propose & document — never write product code and never apply infra** (no `terraform apply`, no replica-set provisioning, no failover config on a live cluster — author the *topology and the FMEA*; the controller builds, the operator applies). **Ground every redundancy/degradation decision in an availability driver** — no failover, bulkhead, or region exists without a `QAS-AVAIL-*` that demands it (un-driven redundancy is accidental complexity and cost — flag it, don't ship it; right-size to the project's actual scale per `project-architecture-config.md`). **Reuse the named resilience patterns** (`../80-patterns/resilience-patterns.md` — circuit-breaker, bulkhead, retry-with-jitter, fallback, load-shedding) before inventing a bespoke failover mechanism. **Record the tradeoff and the rejected alternatives** on every decision (what the redundancy costs on cost/latency/consistency/operability, and what topology it beat). **One writer** authors the failure model so it has conceptual integrity; the read-only `architecture-evaluator` scores it and never edits it. This flow **specifies** the chaos/failure-mode coverage as fitness functions for `qa-agent` to execute — it never runs chaos experiments against a live system itself.
