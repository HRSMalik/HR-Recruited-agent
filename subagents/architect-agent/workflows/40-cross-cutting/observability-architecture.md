# observability-architecture

**Group:** 40-cross-cutting  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Design observability into the system from the start — structured logs, distributed traces with correlation IDs propagated across every service boundary, and metric instrumentation anchored to the four golden signals and RED/USE — so every SLO scenario is measurable in production and an on-call engineer who didn't build the system can diagnose a failure from first principles. Answers: *which signals are emitted where, how do they correlate across hops, and which SLO thresholds are instrumented as error-budget burn rates?*

## Inputs & preconditions

- From `project-architecture-config.md`: the project's stack (services, frontend/backend/datastore topology, proxy/gateway layer), deployment target, compliance regime (if the domain handles PII or regulated data → structured logs must not emit raw sensitive fields), constraints, and the `architecture/<slug>/` artifact locations.
- The architecture **drivers**: the prioritized quality-attribute scenarios from `workflows/00-drivers/quality-attribute-scenarios.md` — specifically the scenarios that carry measurable response-measures (e.g. page/dashboard render latency, a long-running processing step's latency, a critical write-path action's latency, per the source-of-truth NFRs named in `project-architecture-config.md`) and any stated availability/reliability targets. A flow with no SLO targets downgrades to producing skeleton instrumentation anchored to the four golden signals only and flags the open driver questions explicitly.
- The utility tree (the complete set of prioritized QAS leaves from `architecture/<slug>/quality-attributes.md` if authored) — every leaf that has a `p99`/latency/error-rate response-measure is an SLO to instrument.
- The C4 Container diagram (L2) and any authored Component views (L3): every container boundary and every cross-service call is a telemetry emission point. If the C4 diagram has not yet been authored, run the audit pass against the inferred topology (per `project-architecture-config.md` — e.g. frontend → gateway → backend service(s) → datastore) and flag missing diagrams as a prerequisite for the design pass.
- Target: the observability layer of the system under design or review — specifically every service boundary and cross-service call in the C4 L2 Container view (per the project's topology in `project-architecture-config.md`), every user-visible latency path named in the QAS, and the telemetry pipeline (OTel SDK + Collector + backend). The relevant methods are: the four golden signals + RED/USE instrumentation framework, OTel tracing with W3C Trace Context propagation, SLO→SLI→error-budget derivation (Google SRE Workbook ch.5), and the `shared/quality-attribute-rubric.md` hard-oracle set (trace-ID propagation, golden-signal coverage, SLO instrumentation, structured-log schema, bounded cardinality, PII-clean telemetry, fitness functions, diagram compiles, ADR complete).
- Preconditions: `project-architecture-config.md` read; `architecture/<slug>/quality-attributes.md` read if present; the relevant QAS leaves and their response-measures confirmed before sizing cardinality budgets or SLO burn-rate windows.

## Oracle (source of truth)

The hard oracle is **structural and traceable**: a correlation/trace ID is propagated across every service boundary the design shows; each service emits the golden-signal metrics plus structured logs plus spans; every SLO scenario has a named instrument (metric + SLI expression + burn-rate alert) such that you can determine in production whether the scenario is being met. Observability is in the design — a component that emits no signal is a gap, not a choice.

- **hard:**
  - **Trace-ID propagation:** every cross-service call (per the project's topology in `project-architecture-config.md` — e.g. frontend → gateway → backend service, backend → datastore driver, any future async hop) carries a W3C `traceparent` header (or equivalent OTel propagation format); the trace-ID is present on every structured log line emitted within that request's context; no hop is a telemetry dead-zone. Structural check: the C4 L2 diagram shows every synchronous relationship — each one must have a corresponding instrumentation decision (span + log correlation).
  - **Golden-signal coverage:** every service boundary emits all four golden signals — **latency** (p50/p95/p99 histogram, not just average), **traffic** (request rate), **errors** (non-2xx + timeout count, structured as an explicit field not a log parse), **saturation** (active connections / queue depth / datastore connection-pool utilization). Missing a signal on a service boundary is a hard gap. Cross-reference `workflows/40-cross-cutting/performance-capacity.md` for the latency budgets these instruments must satisfy.
  - **RED per service, USE per resource:** for each request-handling layer (the gateway/ingress, each backend API module per `project-architecture-config.md`), emit Rate + Error-rate + Duration histograms. For each resource (the datastore, the application worker pool, the gateway upstream pool), emit Utilization + Saturation + Errors. These are the diagnostic tools the on-call uses after the golden-signal alert fires — they must exist before an incident, not be retrofitted during one.
  - **SLO instrumentation:** every QAS leaf with a response-measure (e.g. `page render p99 ≤ 2s`, `document-processing p99 ≤ 30s`) has a named SLI expression (a ratio query over the latency histogram), a burn-rate alert at ≥2×burn over a 1h window (fast-burn) and ≥1×burn over a 6h window (slow-burn), and an error-budget gauge visible on the runbook dashboard. If a QAS response-measure is unstated, this is an open driver question (surface it; do not invent a threshold). Cross-reference `workflows/40-cross-cutting/reliability-slo.md` for the SLO→SLI→error-budget derivation rules.
  - **Structured events (not free-form strings):** every log line is a structured JSON event with a fixed schema: `trace_id`, `span_id`, `service`, `level`, `timestamp` (ISO 8601 UTC), `event` (semantic name), `duration_ms` (where applicable), `user_id` (opaque ID — never raw PII), plus the project's core entity identifiers (opaque IDs only, per `project-architecture-config.md`). Free-form `printf`-style log strings that require regex parsing to extract a field are a hard gap.
  - **Cardinality bounded:** no label or log field has unbounded cardinality (e.g. a raw entity ID or `user_id` as a Prometheus label dimension — these belong in trace attributes, not metric labels). Metric label sets must be reviewed for cardinality before finalizing the instrumentation design; any label whose worst-case series count (max labels × max values per label) would push the total active-series budget above **10 M series** on a Prometheus-style backend is a hard gap and must be resolved (move the dimension to trace attributes or add a recording rule to reduce it). Cross-reference `workflows/40-cross-cutting/performance-capacity.md` for the cost-sizing model used to validate this threshold.
  - **PII-clean telemetry:** structured log fields and trace attributes must not emit raw PII (name, DOB, SSN/national ID, financial account numbers, or whatever sensitive fields the domain defines). Opaque IDs (`user_id`, the core entity ID, `tenant_id`) are acceptable. Any field that could contain PII is classified and scrubbed at the instrumentation layer before emission. Compliance driver: the compliance regime named in `project-architecture-config.md` (e.g. PII, financial, health data).
  - **CI/CD pipeline telemetry:** build + deploy events are emitted to the same observability backend (DORA metrics: deployment frequency, lead-time for change, MTTR, change-failure rate). This is the supply-chain observability gate; cross-reference `workflows/50-infrastructure/cicd-architecture.md`.

- **soft:**
  - Observability boundaries are drawn at the points that matter for the real failure modes — every cross-service call, every I/O, every user-visible latency path — not sprinkled uniformly for coverage theater.
  - Metric cardinality is cost-aware: the designer has thought through label explosion (team, environment, route are fine; raw entity IDs are not) and has made a deliberate choice, not a default.
  - The system is debuggable by an on-call who didn't build it: a runbook dashboard exists (named), the trace carries enough context to follow the core business entity through its lifecycle, and the log schema is documented in the architecture artifacts.
  - The instrumentation design is evolvable: OTel's vendor-neutral SDK is the instrumentation API (not a vendor SDK), so the backend (Jaeger, Grafana Tempo, AWS X-Ray, Honeycomb, Datadog) can be swapped without changing application code.

## Standards & techniques

- **The three pillars + structured events** (Charity Majors / Cindy Sridharan): **logs** (structured JSON events, not free-form strings), **metrics** (time-series counters/histograms/gauges), **traces** (distributed request spans with parent-child relationships). The fourth pillar — **structured events** — is the source from which logs and metrics derive; emit the canonical event once and fan it out, never emit the same data twice in different formats.
- **OpenTelemetry (OTel)** as the single instrumentation standard: the OTel SDK + auto-instrumentation for the project's backend framework (per `project-architecture-config.md`) and the HTTP/RPC layers; the OTel Collector as the telemetry pipeline (receive → process/scrub → export to backend). No vendor SDK in application code — the Collector handles the backend routing. This is the one-way-door choice (once auto-instrumentation is wired, migrating away requires re-instrumenting every service); document it as an ADR.
- **W3C Trace Context** (`traceparent` / `tracestate` headers) as the propagation format across HTTP hops (frontend fetch → gateway → backend, per the project's topology). For the datastore: OTel driver instrumentation auto-propagates context into database span attributes where supported. Fallacy 1 (network reliability) and Fallacy 8 (network homogeneity) from `shared/fallacies-and-tradeoffs.md` — the propagation header must survive the gateway/proxy pass without being stripped; assert this in a fitness function.
- **The four golden signals** (Google SRE Book ch.6): latency, traffic, errors, saturation — the minimum set that lets an on-call triage any service. These are the design targets for instrumentation; RED (Rate/Errors/Duration per service) and USE (Utilization/Saturation/Errors per resource) are the operational decompositions for diagnosis once the golden signal has fired.
- **SLO instrumentation + error-budget burn** (Google SRE Workbook ch.5): express each SLO scenario as a ratio-based SLI (e.g. `proportion of requests completing in < 2s`), compute the error budget (1 − SLO × period), and alert on burn-rate multiples (fast-burn: 2× for 1h; slow-burn: 1× for 6h) so alerts fire before the budget is exhausted, not after. A bare threshold alert (`p99 > 2s for 5m`) without a budget-burn framing is a soft gap — it fires too late and too noisily.
- **Cardinality cost model** (OpenTelemetry Metrics Specification §Cardinality): every metric instrument's label set is evaluated for worst-case cardinality (max labels × max values per label = series count); Prometheus-style backends become expensive above the **10 M active-series threshold** set as the hard oracle in the Cardinality bounded check above. Core entity IDs, user IDs, and request URLs are high-cardinality dimensions — they belong in trace attributes and log fields, not metric labels. The cardinality threshold is the governing constraint; see `workflows/40-cross-cutting/performance-capacity.md` for the cost-sizing derivation.
- **Diagrams-as-code** (`shared/diagramming-standards.md`): the C4 dynamic/sequence view (`diagrams/obs-trace-propagation.puml`) is authored in PlantUML with the C4-PlantUML stdlib (`!include <C4/C4_Dynamic>`). Every element is labeled with name + type + one-line responsibility; every relationship is labeled with verb + protocol (e.g. "propagates traceparent via, W3C Trace Context / HTTP"); sync relationships use solid lines, async dashed. The diagram must **compile and render without error** before it counts as a completed artifact — the diagram-compile hard oracle from `shared/quality-attribute-rubric.md` applies. A PNG export is not an architecture artifact; the `.puml` source lives in `architecture/<slug>/diagrams/`.
- **ATAM sensitivity/tradeoff points for observability**: synchronous log writes are a sensitivity point for request latency (fallacy 2: latency is not zero); async log buffering raises throughput at the cost of possible log loss on crash (consistency ↔ latency tradeoff from `shared/fallacies-and-tradeoffs.md`). The ADR must name the buffer-flush strategy (async with bounded buffer + crash-safe drain vs synchronous) and the consequence for log completeness during a crash. The OTel Collector in-process vs sidecar is a second tradeoff point: in-process is simpler (fewer moving parts, correct for a single-service or modular-monolith topology) but the Collector sidecar offers better isolation and backpressure — flag as a two-way door (either can be adopted later without re-instrumenting application code because OTel SDK is the stable interface).
- **AKF Scale Cube relevance**: confirm the current topology's scale axis (per `project-architecture-config.md` — e.g. a single-instance X-axis modular monolith vs multiple service replicas); observability design must not assume multiple replicas today but must not preclude it either. The trace-ID and golden-signal metrics work identically across N replicas with no design change — confirm this holds before documenting.
- **Fitness functions** for the observability characteristic (per `workflows/60-evaluation/fitness-functions.md`): (1) a dependency check confirming no component in the frontend or backend source uses a vendor-specific telemetry SDK (only OTel); (2) a schema-conformance test confirming every log line emitted in the test harness matches the structured log schema; (3) a header-propagation test confirming `traceparent` survives the gateway/proxy pass; (4) a cardinality check confirming no metric label set exceeds the defined budget.

## Step sequence

- **audit:** Read `project-architecture-config.md` and the C4 diagram if present → enumerate every cross-service call (per the project's topology — e.g. frontend → gateway → backend → datastore) and every user-visible latency path from the QAS → for each boundary, check whether a trace-ID propagation mechanism is designed, whether structured log emission is specified, and whether a golden-signal metric exists → check whether the QAS response-measures have named SLI expressions + burn-rate alerts → check structured log field schema for PII leakage → check metric label cardinality → emit findings per `finding-schema.md` (`type: "observability"`; `oracle: "hard"` for missing trace propagation, free-form logs, missing golden signal on a boundary, uninstrumented SLO, raw-PII telemetry field, unbounded label cardinality; `oracle: "soft"` for on-call debuggability gaps, vendor SDK lock-in, missing runbook dashboard). Read-only; no artifact authored.
- **design:** **Frame** — confirm the QAS response-measures that are SLOs; surface any unstated targets as open driver questions; confirm the C4 topology (which hops exist, per `project-architecture-config.md`) and the compliance regime (PII scrubbing rules). **Explore** — produce ≥2 divergent instrumentation directions: e.g. (A) in-process OTel SDK + stdout JSON → Collector agent sidecar → Grafana Tempo/Prometheus/Loki stack; (B) OTel SDK + OTLP direct-export to a managed backend (AWS X-Ray + CloudWatch Metrics + CloudWatch Logs Insights). Each direction specifies the propagation format, the log schema, the metric label sets with their cardinality estimates, the SLI expressions for each QAS, and the burn-rate alert configuration. **Evaluate** (separate skeptical evaluator, order-swapped) — score each direction against: trace-ID propagation completeness, golden-signal coverage, SLO instrumentability, cardinality cost, PII-clean emission, vendor lock-in (OTel neutrality), and operational simplicity given the project's current topology (per `project-architecture-config.md`). Pick a winner; graft the best SLI expressions from the runner(s). **Document** (one writer — solution-architect) — author the observability design doc, the structured log schema, the SLI/SLO instrumentation spec (one entry per QAS leaf), the metric catalog (instrument name + type + labels + cardinality estimate), the ADR for OTel as the instrumentation standard, and the fitness-function specs into `architecture/<slug>/`. Author a C4 supplementary view (dynamic/sequence) showing trace propagation across the project's service hops. **Verify** — run the four fitness functions (vendor-SDK dependency check, log-schema conformance, traceparent header propagation, metric-label cardinality check); evaluator re-scores the authored artifacts against hard oracles; loop ≤ 10 iterations or pass.

## Assertions & exit gate

- Every cross-service call in the C4 L2 diagram has a corresponding instrumentation decision: a named span, a `traceparent` propagation mechanism, and a structured log correlation.
- Every service boundary emits all four golden signals (latency histogram p50/p95/p99, traffic rate, error count + rate, saturation indicator); RED per API service; USE per resource (the datastore connection pool, the application worker pool, the gateway upstream pool).
- Every QAS leaf with a response-measure has a named SLI expression and a burn-rate alert at 2×/1h (fast-burn) and 1×/6h (slow-burn); if a QAS has no stated response-measure, it is listed as an open driver question, not silently uninstrumented.
- Every log line matches the structured log schema (`trace_id`, `span_id`, `service`, `level`, `timestamp`, `event`, optional `duration_ms`, opaque IDs only — no raw PII fields).
- No metric label has unbounded cardinality (entity IDs are trace/log attributes, not Prometheus labels); the worst-case series count across all instruments is **≤ 10 M active series** (the hard-oracle threshold from the Cardinality bounded check; derived from the OTel Metrics Specification §Cardinality cost model and validated against `workflows/40-cross-cutting/performance-capacity.md`).
- All four fitness functions pass: no vendor SDK in application source; log-schema conformance; `traceparent` survives the gateway/proxy pass; label-cardinality within budget.
- The OTel instrumentation standard is documented as an ADR per `shared/adr-template.md` — with `Status`, `Date`, `Deciders`, `Driver refs` (traced to the relevant QAS leaves), `One-way door?`, `Context and problem statement`, `≥2 Considered options` (e.g. OTel SDK + Collector vs a vendor agent SDK), `Decision outcome`, and `Consequences` (positive: vendor-neutral, backend-swappable; negative: OTel SDK + Collector is more moving parts than a single-vendor agent). An ADR missing any mandatory section (context · ≥2 options · decision · positive AND negative consequences) fails the `adrs_complete` hard oracle.
- The C4 dynamic/sequence diagram (`diagrams/obs-trace-propagation.puml`) **compiles and renders without error** per `shared/diagramming-standards.md`; a syntax error or missing render is a hard-oracle failure.
- The observability design does not emit raw PII (name, DOB, national ID, financial account numbers, or whatever sensitive fields the domain defines) in any telemetry channel.
- **Gate:** hard oracles green (trace-ID propagation on every hop · golden-signal coverage on every boundary · SLO instrumentation for every measured QAS · structured log schema · bounded cardinality ≤ 10 M active series · PII-clean · fitness functions pass · diagram compiles/renders · ADR complete per `shared/adr-template.md`) AND (design) rubric mean ≥ 0.8 (on-call debuggability, cardinality cost-awareness, OTel vendor-neutrality, evolvability of the telemetry pipeline).

## Output

Write `artifacts/observability-architecture/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (type `"observability"`; hard findings for uninstrumented hops, missing golden signals, raw-PII fields, unbounded cardinality, uninstrumented SLOs; soft findings for on-call debuggability gaps, vendor lock-in, missing runbook dashboard), plus the verification block (fitness-function results, traceability coverage — every QAS leaf → named SLI → named instrument, hard-oracle pass/fail, rubric score) for design mode.

Design artifacts land in `architecture/<slug>/` per `project-architecture-config.md`:

```
architecture/<slug>/
├── observability-design.md        log schema, metric catalog, SLI/SLO instrumentation spec, burn-rate alert config, runbook dashboard spec
├── adr/NNNN-otel-instrumentation.md   ADR: OTel SDK + Collector over vendor SDK (one-way door: false; the OTel SDK is the stable interface; Collector backend is swappable)
├── diagrams/obs-trace-propagation.puml  C4 dynamic view: traceparent propagation across the project's service hops (per `project-architecture-config.md`)
└── fitness-functions.md           four checks: vendor-SDK dep scan, log-schema conformance, traceparent header propagation, label-cardinality budget
```

The run report (`artifacts/`) is the machine record; the architecture package (`architecture/<slug>/`) is the durable deliverable the controller implements from.

## Guardrails

Per `shared/guardrails.md`: propose and document — never write product instrumentation code in the project's backend or frontend source directories (per `project-architecture-config.md`); author the log schema, metric catalog, SLI expressions, ADRs, C4-as-code diagrams, and fitness-function *specifications* into `architecture/<slug>/`, and runnable fitness-function *check scripts* into `tests/fitness/` for CI enforcement. The controller wires the OTel SDK; the operator configures the Collector and backend.

Ground every instrumentation decision in a driver: if a golden signal, SLI, or log field has no QAS or compliance driver behind it, it is accidental complexity — flag it rather than including it. Right-size: match the instrumentation footprint to the project's actual scale and topology (per `project-architecture-config.md`) — do not propose a multi-backend fan-out, a full distributed tracing mesh, or a ML-based anomaly-detection pipeline absent a driver that demands it. Boring, proven instrumentation (OTel + a single backend, structured JSON logs to stdout, Prometheus-style metrics) is the correct default.

Do not log or trace raw PII; do not hardcode backend endpoints or credentials in any architecture artifact — inject via env, reference the secret-store pattern from the compliance regime in `project-architecture-config.md`. The evaluator is separate from the writer and never grades its own output. Bound the design loop to ≤ 10 iterations.
