# fitness-functions

**Group:** 60-evaluation  ·  **Runs as:** subagent: ../.claude/agents/architecture-evaluator.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
Turn the architecture's characteristics into **executable, automatable checks with a pass/fail threshold** — the fitness functions that objectively guard each architecturally-significant invariant (dependency rules, layering, coupling ceilings, latency budgets, security invariants like "no PII in logs") and fail loudly when a commit drifts from the intended structure. This is the **design's hard oracle made runnable and wired into CI**: it converts "the domain must not depend on the framework" or "checkout p99 ≤ 200ms" from prose in an ADR into a check a pipeline can red-light. Answers: *which characteristics are objectively measurable, what is the threshold, which function guards each, and is it wired into the gate?* A fitness function with no characteristic it guards is noise; a characteristic that matters with no function guarding it is unguarded drift.

## Inputs & preconditions
- From `project-architecture-config.md`: the project's stack and architecture style (e.g. a modular monolith vs microservices — a modular monolith skews the functions toward *module-boundary* and *layering* rules over service-topology rules), the diagram/ADR/artifact locations, the CI flow and where a runnable check belongs (a project `fitness/`/`tests/` dir — never `src/**`), and the locked rule that a new tool/datastore needs a driver-justified ADR.
- The **drivers**: the utility tree from `../00-drivers/quality-attribute-scenarios.md` (`architecture/<slug>/quality-attributes.md`) — every `(H,*)` 6-part scenario with its **response measure** is a candidate fitness function; the prioritized leaves are where guarding effort goes. A flow with no utility tree **downgrades scope and says so** — you cannot write a fitness function without the measurable characteristic it guards (`status:error`, not a guessed threshold).
- The **decisions**: the ADR log (`architecture/<slug>/adr/`) — each ADR's `Follow-ups / fitness functions` block names the check that enforces it in CI (the *function-per-ADR* pattern); the structural rules from `../10-styles-decomposition/dependency-management.md` (acyclic deps, stable-dependencies, the dependency rule), and the boundary/threat invariants from `../40-cross-cutting/security-architecture.md`.
- Target: the existing/proposed architecture under guard (the module set, the auth/RBAC boundary, the core write path for the project's central business entity — per `project-architecture-config.md`) and the CI pipeline the functions must run in (see `../50-infrastructure/cicd-architecture.md`). Preconditions: the utility tree + ADRs read; the codebase readable for the *audit* path (dependency graph, import graph, coupling metrics) but **never written back** (`guardrails.md`).

## Oracle (source of truth)
The architecture's **own measurable characteristics** — each fitness function asserts a structural fact or a response measure that is deterministic, not a matter of taste. The function *is* the oracle; "the design is clean" is not. Grounding rule (`quality-attribute-rubric.md`): **no number + no traced driver = a preference, not a fitness function** — drop it.
- **hard:**
  - **Every architecturally-significant decision/invariant has a fitness function** — each `(H,*)` scenario's response measure and each structural ADR maps to a check that is (a) **automatable** (runs unattended, no human judgement — the *invocation* axis defaults to automated), (b) carries an explicit **pass/fail threshold** (a number, a forbidden-edge set, a boolean structural assertion — `cycles == 0`, `domain→framework imports == 0`, `p99 ≤ 200ms`, `PII-in-log matches == 0`, `inbound-coupling(module) ≤ N`), and (c) is **traced** — `driver_ref` to a `QAS-*` or `decision_ref` to an ADR. A characteristic the architect calls significant with **no guarding function** is an `oracle:"hard"` gap; a function with **no traced driver/ADR** is dropped as a preference.
  - **Every function is classified on all 5 taxonomy dimensions** (`Building Evolutionary Architecture` 2nd ed. — see *Standards & techniques*): **scope** (atomic|holistic), **cadence** (triggered|continuous|manual), **result** (static|dynamic), **invocation** (automated|manual), **proactivity** (intentional|emergent). A function with any dimension unstated is an incomplete spec, not a finished fitness function — the register (below) carries the 5-tuple as columns, not prose.
  - **The functions are wired into the CI gate** — every *triggered* function is referenced from the pipeline stage that runs it (see `../50-infrastructure/cicd-architecture.md`) and **fails the build** on breach; every *continuous* function names its monitoring target (the dashboard/alert it feeds) so it isn't merely aspirational. A fitness function that exists but blocks/alerts nothing is dead — it must be enforced, not advisory.
  - **The check is real, not a stub, and wired to a concrete executable tool** — for the *design* path the authored spec names the concrete tool + the exact assertion so the controller can implement it verbatim: **ArchUnitTS** or **dependency-cruiser** (the layering/cycle/connascence rule for a JS/TS stack — swap for the project's actual stack per `project-architecture-config.md`), **import-linter**/**pytestarch** (the equivalent contract for a Python stack), **OPA/conftest** (a policy over the IaC/manifest/Dockerfile/ADR front-matter — cross-ref `../shared/policy-as-code.md` for the Rego library and package IDs), **Spectral + oasdiff** (an API-contract lint + breaking-change diff — cross-ref `../10-styles-decomposition/api-first-contracts.md`), **cosign** (a signed-provenance/SBOM verification gate — cross-ref `../shared/supply-chain-contract.md`), or a **perf-budget gate** on the p-percentile from the load run — never a vague "add a test for coupling."
- **soft:** the function **set covers the characteristics that matter** — every `(H,H)` and `(H,M)` leaf of the utility tree plus the key structural rules (the dependency rule, acyclic modules, the auth boundary, no-PII-in-logs) has a guard, with no significant characteristic left to manual review; the functions are **cheap enough to run continuously** (the atomic structural checks complete in seconds so they can gate every commit, not just nightly); the taxonomy mix is **defensible** — a register that is all-manual or all-holistic where a cheap atomic/automated check would prove the same invariant is a craft gap; and they **fail loudly and specifically** — a breach names the offending edge/module/percentile, not just "fitness check failed," so the controller can act without re-deriving the cause.

## Standards & techniques
- **Evolutionary architecture + fitness functions** (Ford/Parsons/Kua, *Building Evolutionary Architecture*, 2nd ed.) — the founding frame: an *architectural fitness function* is "an objective integrity assessment of some architectural characteristic." The 2nd edition's canonical taxonomy is **five dimensions, not one axis** — classify **every** fitness function on all five before it counts as specified:
  1. **Scope — atomic vs holistic.** Atomic = one characteristic in isolation (a single cyclic-dependency check, a single layering rule). Holistic = a combination that only matters *together* (e.g. "coupling stays bounded **and** p99 stays inside budget as load scales" — either alone can pass while the pair fails).
  2. **Cadence — triggered / continuous / manual.** Triggered = runs on a pipeline event (commit, PR, merge, release). Continuous = always-on production monitoring (an SLO burn-rate alert, a live coupling-drift dashboard). Manual = a documented human review step (the exception, for a characteristic no tool can yet express — e.g. "the append-only audit trail reads as tamper-evident to a reviewer").
  3. **Result — static vs dynamic.** Static = a fixed threshold that doesn't change with context (`cycles == 0`, `domain→framework imports == 0`). Dynamic = the threshold itself is computed from current context (a percentile band relative to a rolling baseline, an error-budget burn rate relative to the SLO window) — dynamic functions need a stated computation, not just a number, or they aren't specified yet.
  4. **Invocation — automated vs manual.** Automated = runs unattended, no human judgement (the default; required for anything gating a commit). Manual = a human executes the check per a documented procedure — permitted only where no tool can structurally express the assertion, and always paired with a stated cadence so it doesn't silently lapse.
  5. **Proactivity — intentional vs emergent.** Intentional = authored up front to guard a known characteristic (everything in this flow's register). Emergent = discovered *after* an incident or a conformance-audit finding (`architecture-conformance.md`) reveals a characteristic that needed guarding and didn't have one — an emergent function is back-filled into the register with its `driver_ref` traced to the incident/finding that surfaced it, not left as a one-off patch.

  Record the 5-tuple per function in the register (below). **Atomic + triggered + static + automated + intentional is the default profile** — cheapest to run, gates every commit; deviate from it only when the characteristic genuinely demands holistic/continuous/dynamic/manual/emergent, and say which axis and why.
- **Architecture characteristics as testable invariants** — each `-ility` is reframed as a deterministic assertion: *modifiability* → "no cyclic module dependencies" + "the dependency rule: `domain` imports no framework/IO"; *performance* → "p99 of the critical read/write path ≤ budget"; *security* → "no secret literal in source," "no PII field in a log statement," "every route under an auth dependency"; *scalability/coupling* → "inbound coupling of a module ≤ N," "afferent/efferent ratio within the stable-dependencies band." The reframe is the deliverable — vague `-ilities` don't gate.
- **The tool catalog — wired to a runnable command, not just named** (cite the source so the controller can wire the real thing verbatim; every entry below is a concrete gate, not a category label). The examples below assume a frontend/backend split per `project-architecture-config.md`; substitute the equivalent tool for the project's actual stack:
  - **ArchUnitTS + dependency-cruiser (JS/TS frontend, illustrative — swap for the project's stack per `project-architecture-config.md`).** `ArchUnitTS` expresses layering/dependency-rule assertions as executable TypeScript tests (`projectFiles().inFolder('stores').shouldNot().dependOnFiles().inFolder('components')`) runnable under the project's existing test runner — no new CI concept, just another test suite. `dependency-cruiser` expresses the same class of rule declaratively in `.dependency-cruiser.js` (`forbidden: [{ name: 'no-store-to-component', from: { path: '^src/stores' }, to: { path: '^src/components' } }]`) and doubles as **cycle detection** (`dependency-cruiser --validate` with a `no-circular` rule) and **connascence-boundary checks** (a `forbidden` rule matching a cross-boundary import pattern stronger than Connascence-of-Name, mirroring `shared/quality-attribute-rubric.md` → *Connascence ≤ contract*). Run both: ArchUnitTS for assertion-style rules colocated with the test suite a developer already runs locally; dependency-cruiser for the full-graph `--validate` sweep (cycles + forbidden edges) as a dedicated CI step with a machine-readable (`--output-type json`) exit code. Prefer these over `madge`/`import-linter` as the *primary* enforcers on a JS/TS stack — `madge` remains useful for a quick visual graph, not as the gate of record.
  - **import-linter / pytestarch (Python backend, illustrative — swap for the project's stack per `project-architecture-config.md`)** — the equivalent layering/dependency-rule contracts on the backend side (`Contract`/`layers` in `.importlinter`, or `pytestarch`'s fluent API run under the existing test runner) where a Python-side boundary needs the same class of check dependency-cruiser provides on the TS side.
  - **OPA / conftest (policy-as-code)** — the enforcer for **infra/security/ADR gates** that aren't expressible as a source-level import-graph rule: Terraform-plan JSON, Kubernetes manifests, Dockerfiles, and **ADR front-matter completeness** (a `conftest` policy asserting every accepted ADR's YAML front-matter carries `status`/`decision-makers`/`driver_refs`). The full Rego policy library, package layout, and CI-stage wiring are specified in `../shared/policy-as-code.md` — this flow's register **references** those policy IDs by ID (`SEC-TF-001`, `DOC-ADR-001`, …), it does not re-author them; one policy, one place.
  - **Spectral + oasdiff (API contracts)** — `Spectral` lints the OpenAPI spec against a ruleset (naming, required fields, security-scheme presence) as a fast pre-merge gate; `oasdiff breaking` (or `openapi-diff`) diffs the current contract against the PR's changed spec and **fails the build on any breaking change** (a removed field, a narrowed enum, a changed required-ness) with the exact path/operation in its output. Wire both as sequential steps in the same CI stage — Spectral first (style/completeness), oasdiff second (breaking-change gate) — cross-link `../10-styles-decomposition/api-first-contracts.md`.
  - **cosign (supply-chain)** — `cosign verify-attestation --type slsaprovenance ...` (and `--type cyclonedx`/`--type spdx` for the SBOM attestation) as the pre-promotion gate verifying a signed, non-falsifiable provenance chain before an artifact is deployed. The full SLSA-level target, Sigstore/Rekor topology, and SBOM-format decision are specified in `../shared/supply-chain-contract.md` — this flow's register carries the `cosign` command as the executable check and references that contract for the ADR-level design.
  - **perf-budget gates** — assert a p-percentile from the load run (`../50-infrastructure/cicd-architecture.md` pipeline → the qa-agent's load result) against the QAS budget; **bundle-size budgets** for a client-side frontend, where applicable.
- **The function-per-ADR pattern** — every accepted ADR that asserts a structural invariant ships a fitness function in its `Follow-ups / fitness functions` block (`shared/adr-template.md`), so the decision is *enforced*, not just *recorded*. The ADR is the why; the fitness function is the guard that stops the why from silently eroding. An ADR with a structural invariant and no follow-up function is an incomplete decision.
- **Relationship to `architecture-conformance.md`** — that flow runs the **same static tools** (ArchUnitTS/dependency-cruiser, import-linter/pytestarch) as a **one-time or periodic audit** of the as-built system against the as-documented reflexion model; this flow is where a conformance rule that recurs **graduates into a standing, CI-wired fitness function**. A divergence `architecture-conformance.md` reports repeatedly (the same rule breached across audit runs) is the concrete trigger for an **emergent** (proactivity axis) function here — back-filled with `driver_ref` traced to the conformance finding that surfaced it, never left as a one-off re-fix.
- **Right-sizing (the locked rule)** — match the function set to the architecture style actually named in `project-architecture-config.md`: a modular monolith over a single primary datastore skews the set toward **module-boundary + layering + perf-budget + security-invariant** checks, not a service-mesh contract matrix. Do **not** author k8s/multi-region/mesh fitness functions absent a driver — that's accidental complexity made executable (`shared/quality-attribute-rubric.md` → simplicity). Prefer the cheap static check (import graph, cycle detection) over a heavyweight runtime probe where it proves the same invariant.

## Step sequence
- **audit:** Frame the drivers (read the utility tree + ADRs) → for each `(H,*)` scenario and each structural ADR, ask **"is there an automatable check with a threshold guarding this, and is it wired into CI?"** → run the read-only structural checks against the *existing* codebase (cycle detection / import-graph / coupling metrics via the tools named in `project-architecture-config.md` for each side of the stack — e.g. **ArchUnitTS**/**dependency-cruiser** for a JS/TS frontend, **import-linter**/**pytestarch** for a Python backend, in report mode) to find characteristics that are *already* drifting → emit findings: ungoverned characteristics (an `(H,*)` leaf with no function), un-wired functions (a function that exists but doesn't gate), threshold-less checks (a "test" with no pass/fail bound), functions with an **incomplete 5-dimension classification**, and any **current breach** the structural read surfaced (a cycle that exists today, a `domain→framework` import already present). Read-only — authors no spec, edits no source.
- **design:** **Frame** (extract/confirm the `(H,*)` scenarios + the structural invariants from the ADRs, read-only) → **Explore** (for each characteristic, ≥2 candidate guards as concrete specs — e.g. the dependency rule via an **import-linter** contract vs a **dependency-cruiser** `forbidden` rule; a perf budget as a triggered pre-release gate vs a continuous SLO burn-alert; an infra/security invariant via a hand-authored **OPA/conftest** policy vs reusing an existing package from `../shared/policy-as-code.md` — each a real tool + assertion + full 5-tuple classification, not the same check relabeled) → **Evaluate** (the skeptical `architecture-evaluator` scores each candidate vs the utility tree + the hard oracle: is it automatable, does it carry a threshold, is the 5-dimension classification complete and defensible, is it cheap enough to gate every commit, does it fail specifically; order-swapped to kill bias; penalize a heavyweight/holistic/continuous function where a cheap atomic/triggered/static check proves the same invariant) → **Document** (ONE writer authors `architecture/<slug>/fitness-functions.md` — the **fitness-function register**: one row per function with `id · characteristic · driver_ref/decision_ref · tool+command · exact assertion+threshold · scope(atomic|holistic) · cadence(triggered|continuous|manual) · result(static|dynamic) · invocation(automated|manual) · proactivity(intentional|emergent) · CI stage · failure message`; folds the chosen guard into each ADR's `Follow-ups` block; reuses an established check from the tool catalog — ArchUnitTS/dependency-cruiser, import-linter/pytestarch, OPA/conftest, Spectral/oasdiff, cosign — before inventing one) → **Verify** (the *verification* hard gates: **traceability** — every `(H,*)` leaf + every structural ADR maps to ≥1 function and every function maps back to a driver/ADR, no orphans; **automatability** — each function names a real tool + command + a deterministic assertion; **threshold-present** — each carries a number/forbidden-set/boolean; **taxonomy-complete** — every function's 5-tuple is filled in, not defaulted silently; **CI-wired** — each triggered function references the pipeline stage in `../50-infrastructure/cicd-architecture.md`; the `architecture-evaluator` re-scores the *register*, not the prose) → loop ≤10 or pass. The writer additionally authors a **runnable check** per function — the ArchUnitTS/dependency-cruiser config, the `.importlinter`/pytestarch contract, the Rego policy (referencing `../shared/policy-as-code.md`'s package IDs), the Spectral ruleset + `oasdiff` CI step, or the `cosign verify-attestation` command (referencing `../shared/supply-chain-contract.md`) — dropped into the project's `fitness/`/`tests/` dir for the controller to wire into CI (a verification artifact, **not** product code — `guardrails.md` §1), never into `src/**`.

## Assertions & exit gate
- **Coverage:** every `(H,H)` and `(H,M)` quality-attribute scenario in the utility tree, and every structural ADR (the dependency rule, acyclic modules, the auth boundary, no-PII-in-logs, the API contract), has **≥1 fitness function** guarding it — no ungoverned significant characteristic.
- **Automatability + threshold:** every function names a **concrete tool + runnable command** (ArchUnitTS/dependency-cruiser, import-linter/pytestarch, OPA/conftest, Spectral+oasdiff, cosign, or a perf-budget gate) and an **explicit pass/fail threshold** (a number, a forbidden-edge set, a boolean structural assertion) — no human-judgement "function," no threshold-less "just check it."
- **Taxonomy-complete:** every function is classified on **all 5 dimensions** — scope (atomic|holistic), cadence (triggered|continuous|manual), result (static|dynamic), invocation (automated|manual), proactivity (intentional|emergent) — recorded as register columns. A function missing any dimension is an incomplete spec and blocks the gate the same as a missing threshold.
- **Traceability (both directions):** every function carries a `driver_ref` (`QAS-*`) or `decision_ref` (`ADR-NNNN`); every significant decision/invariant has a function pointing at it. No orphan function (a check no driver demands — accidental complexity), no orphan invariant (a characteristic no check guards — unguarded drift).
- **CI-wired + tool-real:** every *triggered* function is referenced from a concrete pipeline stage (`../50-infrastructure/cicd-architecture.md`) and **fails the build** on breach; every *continuous* function names the monitor/alert it feeds; the cheap atomic/static/automated checks gate **every commit**, the heavier holistic/dynamic ones gate at merge/pre-release or run continuously in prod. Infra/security/ADR-gate functions reference the exact `../shared/policy-as-code.md` package + policy ID they reuse or extend; API-contract functions reference `../10-styles-decomposition/api-first-contracts.md`; supply-chain functions reference `../shared/supply-chain-contract.md`'s SLSA/SBOM design — no function invents a tool the shared contracts already specify.
- **Fails loudly:** each function's failure message names the offending edge/module/percentile/field — actionable without re-deriving the cause.
- **Audit-path current state:** the structural read reports any invariant **already breached today** (an existing cycle, a present `domain→framework` import) as a `critical`/`major` finding, not a future risk.
- **Gate:** hard oracles green (every `(H,*)` scenario + structural ADR has an automatable, thresholded, taxonomy-complete, traced, CI-wired function; no orphans) AND (design) rubric mean ≥ 0.8.

## Output
Write `artifacts/fitness-functions/report.json` per `shared/report-format.md` — findings/risks per `shared/finding-schema.md` (`type:"evolvability"` for a coverage/dependency-rule gap, `type:"performance"`/`"security"` for a missing perf-budget/PII-in-log guard; `oracle:"hard"` for an ungoverned `(H,*)` characteristic, a threshold-less check, an un-wired function, an incomplete 5-dimension classification, or a structural invariant breached today; each finding's `evidence` = the traced characteristic + the structural fact (the missing function, the live cycle, the un-gated stage), `recommendation` = the ADR-shaped guard (tool + command + exact assertion + threshold + the 5-tuple classification + CI stage), with `tradeoff` (the run-cost/false-positive cost the check adds) and `alternatives_rejected` (the cheaper/heavier guards considered, including a rejected cadence/scope choice — e.g. "continuous SLO monitor rejected in favor of a triggered pre-release gate: no driver demands live burn-rate alerting yet")). For **design** mode the report's `verification.fitness_functions[]` block lists each authored function with its `result` (`pass`/`modeled-pass`/`fail`) and its 5-tuple, and the path to the authored **`architecture/<slug>/fitness-functions.md`** register (and any runnable spec/script dropped in the project `fitness/`/`tests/` dir). Authored artifacts — the register and the per-ADR `Follow-ups` updates — land in **`architecture/<slug>/`** per `project-architecture-config.md`, never scattered. The run report stays in `artifacts/`; the architecture (the register) lives in `architecture/<slug>/`.

## Guardrails
Per `shared/guardrails.md`: **propose & document, never write product code** — this flow authors the *fitness-function register* and per-ADR follow-ups (architecture artifacts), and may drop a *runnable check spec/script* (an ArchUnitTS/dependency-cruiser config, an `.importlinter` contract, a Rego policy, a Spectral ruleset, a `cosign` command) into the project's `fitness/`/`tests/` dir for the controller to wire into CI; it **never** edits `src/**`/`frontend/**`/`backend/**`, **never** modifies the actual pipeline config, **never** runs `conftest`/`cosign`/a live scan itself, and **never** runs `terraform apply` / a migration / a live cloud change (the operator wires and runs the gate). **Ground every function in a driver** — no fitness function without a measurable characteristic it guards (a `QAS-*` response measure or a structural ADR); a check no driver demands is accidental complexity, flag it, don't author it; a number with no traced driver is a preference, not a finding. **Reuse the established check** from the tool catalog (ArchUnitTS/dependency-cruiser, import-linter/pytestarch, OPA/conftest per `../shared/policy-as-code.md`, Spectral/oasdiff per `../10-styles-decomposition/api-first-contracts.md`, cosign per `../shared/supply-chain-contract.md`) before inventing a bespoke probe. **Classify every function on all 5 taxonomy dimensions** — an unclassified function is an incomplete spec. **One writer** authors the register so the function set has conceptual integrity. **Right-size** — guard the characteristics the project's actual architecture style (per `project-architecture-config.md`) has (module boundaries, layering, perf budgets, security invariants); do not author service-mesh/multi-region fitness functions, or continuous/holistic checks where a triggered/atomic one proves the same invariant, absent a driver. The audit path **reads** the codebase (dependency/import graph, coupling, cycles) but writes nothing back. **Never write to project memory** — lessons go in the returned report and the register.
