# wardley-evaluation

**Group:** 60-evaluation  ·  **Runs as:** subagent: ../.claude/agents/architecture-evaluator.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Map the value chain on the evolution axis (Wardley mapping) to make build-vs-buy-vs-rent a reasoned architectural input rather than taste. Commodity components are bought or rented; genesis and differentiating components are built. Every sourcing recommendation emerges from a component's evolutionary stage and its differentiation status — not convention, familiarity, or resume-driven preference. This flow answers: *which components should we build, which should we buy, and which should we rent — and what does the architecture look like when those choices are respected?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target, scale/SLO targets, compliance regime, team topology (small team — Conway: sourcing choices must be operatable by this team), cost constraints and budget horizon, and where `architecture/<slug>/` + the ADR log live.
- The architecture **drivers**: the prioritized quality-attribute scenarios (the utility tree from `workflows/00-drivers/quality-attribute-scenarios.md` → `architecture/<slug>/quality-attributes.md`) and the constraints the sourcing decisions must satisfy. Every `(H,*)` scenario component must appear on the map — a map that omits a scenario-critical component is incomplete.
- The **anchor user need** — the expressed user capability at the top of the value chain (e.g. "user submits a request and receives a decision"). Without a named anchor the map has no orientation; surface as an `error` if absent.
- The system's components — from the C4 container/component diagram (`workflows/50-infrastructure/deployment-topology.md`) if available, or from the functional scope. Every component that appears in a `(H,*)` scenario must appear on the map; others are in-scope if they are material dependencies.
- Preconditions: the source-of-truth docs (BRD / PRD / understanding log named in `project-architecture-config.md` — real constraints; **never invent evolution placements**) read; `shared/wardley-map-template.md` read and applied to every component placement; if a component's industry maturity is genuinely contested, that uncertainty is surfaced as a finding, not resolved by assertion.

## Oracle (source of truth)
The sourcing recommendation for each component is grounded in its **evolutionary stage** (Genesis / Custom-built / Product / Commodity) on the *industry axis*, combined with its **differentiation status** (does this component uniquely serve the system's value proposition?) — per Simon Wardley's value-chain × evolution framework. The map is a **soft input** to the architecture: it informs `architecture-explorer` option-generation and the cost-finops TCO model; it does not independently gate a design.

- **hard (deterministic — traceable / structural):**
  - The map anchors on a **named real user need** at the top of the chain — "the system" is not an anchor; the concrete user capability is. A map with no anchor is rejected (`status:error`).
  - Every component that appears in a `(H,*)` quality-attribute scenario or in the C4 container/component diagram is placed on the map. A component from a high-priority scenario that is absent from the map blocks (`oracle:"hard"`, `type:"risk"`).
  - Each component carries: its **evolution stage** (I-Genesis / II-Custom / III-Product / IV-Commodity) with the *industry-maturity rationale in one clause* (not the team's familiarity); its **differentiation status** (yes / no / partial); its **sourcing recommendation** (build / buy / rent / watch / commission); and its **primary driver** — either a quality-attribute scenario ID or "none — commodity default." A sourcing recommendation with no driver and no evolution-stage justification is not a finding; it is a preference and is dropped.
  - Every **evolution-gap finding** (component at Stage III–IV on the industry axis, built or operated in-house with no differentiating driver) is raised as a finding (`ARCH-WARD-NNN`, `type:"cost"` or `type:"operability"`, `oracle:"soft"`) with a TCO evaluation prompt pointing to `workflows/90-specialized/cost-finops-architecture.md`. The gap is flagged; the TCO decision is the controller's.
  - Every **inertia flag** (component resisting rightward movement despite no driver demanding it) is named explicitly. Inertia is not silently accepted — the map makes the cost and risk of the resistance visible.
  - The completed sourcing-recommendation set feeds `architecture-explorer` as a **build-vs-buy-vs-rent posture constraint**: Stage III–IV components with no differentiating driver → the explorer's option set must include a managed/utility variant; options that build these from scratch must cite the driver that justifies the deviation.

- **soft (rubric / craft — scored only after hard oracles are green):**
  - **Honest placement.** Placement reflects *industry maturity*, not the team's familiarity. A component built in-house is not automatically Genesis. A contested placement (where reasonable practitioners would disagree) is flagged with the range and carried as a soft finding — not resolved by assertion.
  - **Sourcing aligns with strategic drivers.** Components classified as differentiating are built; commodity components are rented or bought. Building a commodity and renting a differentiator are both soft failures.
  - **Commodity hygiene.** The map does not rationalize building a commodity (compute, object storage, email delivery, managed identity) absent a driver (compliance residency, latency, cost at extreme scale, regulatory). Every in-house-built commodity has a named justifying driver or a `watch` + inertia flag.
  - **FinOps alignment.** Sourcing choices align with the cost/FinOps constraints in `project-architecture-config.md` and the strategic drivers — do not let the map rationalize accidental complexity or résumé-driven technology choices. Cross-reference: `workflows/90-specialized/cost-finops-architecture.md`.
  - **Time-sensitivity surfaced.** Components moving from Custom-built toward Product within the planning horizon are flagged as `watch` items; the ADR for them carries a review date. A map that ignores emerging commoditization under-serves the architecture horizon.

## Standards & techniques
- **Wardley mapping — value chain × evolution** (Simon Wardley): the two-axis framework (visibility to user need [vertical] × evolutionary stage [horizontal, I–IV]) that positions every component on a commodity curve. The map is read for sourcing strategy and fed as a posture constraint into option-generation. The canonical stages and the placement + sourcing rules are in `shared/wardley-map-template.md`; this flow applies that template and does not re-derive the framework.
- **Situational awareness** (Wardley): the map reveals *where the landscape is now* and *where it is moving*. Components moving rightward are commoditizing; building them in the face of that trajectory is waste. Surfacing this movement (the `watch` flag) is the flow's second-order value.
- **Sourcing strategy by evolutionary stage**: Genesis / early Custom-built → build (or spike; no viable external option); late Custom-built → build if differentiating, commission if not; Product / Rental → buy (license) or rent (SaaS/managed), evaluate TCO + lock-in; Commodity / Utility → rent or utility-consume; building is accidental complexity and an operability burden.
- **Pioneers / Settlers / Town-Planners** (Wardley organizational lens): Genesis and differentiating Custom-built work belongs to *pioneers* (explorers, high uncertainty); maturing Custom-built belongs to *settlers* (productizers); Product/Commodity work belongs to *town-planners* (efficiency operators). Where team topology (`project-architecture-config.md`) mismatches the component's evolutionary stage — pioneers operating a commodity, or town-planners forced to pioneer — that is a Conway-law and operability finding.
- **Evolution-gap and inertia analysis** (per `shared/wardley-map-template.md`): systematic scan for (a) components at Stage III–IV operated in-house without a driver — evolution gaps; (b) components resisting rightward movement despite no differentiating driver — inertia. Both are `ARCH-WARD-NNN` findings.
- **Build↔Buy tradeoff** (`shared/fallacies-and-tradeoffs.md`): the canonical framing this flow operationalizes. The tradeoff is not generic — it is resolved *per component* by evolutionary stage + differentiation + TCO. The rejected path (the alternative sourcing option) is always named.
- **Fallacy #7 — transport cost is not zero** (`shared/fallacies-and-tradeoffs.md`): relevant when a rented managed service crosses a network boundary; egress and inter-service latency are in the sourcing-recommendation evidence, not assumed zero.
- **CBAM / TCO confirmation** (`shared/evaluation-method.md`, `workflows/90-specialized/cost-finops-architecture.md`): the map produces a *positional hypothesis* for sourcing; the TCO number from cost-finops confirms or overrides it. A sourcing recommendation that contradicts a grounded TCO model yields to the TCO model — cost evidence beats positional hypothesis. This flow prompts the TCO evaluation; it does not produce the cost model.

## Step sequence
- **audit:** Read the existing architecture description and the utility tree (read-only) → identify the anchor user need; flag absent if missing → place every component that appears in a `(H,*)` scenario or the C4 diagram on the evolution axis using *industry maturity* as the placement criterion (not team familiarity) → apply the differentiation test to each Custom-built component → scan for evolution gaps (Stage III–IV components built or operated in-house with no differentiating driver) → scan for inertia flags (components resisting rightward movement) → flag contested placements with the range → emit findings (`ARCH-WARD-NNN` per `finding-schema.md` — `type:"cost"` / `type:"operability"` / `type:"risk"`, `oracle:"soft"` for evolution-gap and inertia; `oracle:"hard"` for absent anchor or missing `(H,*)` component) and a sourcing-recommendation set → **read-only; no artifact authored; no source edited**.
- **design:** **Frame** (confirm the anchor user need + the utility tree + the component scope, read-only; flag any missing anchor or `(H,*)` component gap before proceeding) → **Explore** (produce the 2D map: place every in-scope component on the value-chain × evolution grid per `shared/wardley-map-template.md`; draw dependency arrows; apply the differentiation test; produce the sourcing-recommendation record per component; flag evolution gaps, inertia, contested placements, and `watch` items — all read-only) → **Evaluate** (skeptical evaluator scores each placement against the *industry-maturity criterion*; verifies that every `(H,*)` scenario component is on the map; checks that commodity sourcing does not build what could be rented; checks pioneer/settler/town-planner alignment with team topology; identifies sensitivity points — placements where a stage shift flips the sourcing recommendation; identifies tradeoff points — components where differentiation and cost pull in opposite directions; verifies the sourcing-recommendation set is driver-grounded — read-only, evaluator never edits what it grades) → **Document** (the **one writer** `solution-architect` authors `architecture/<slug>/wardley-map.md` — the map artifact: the 2D grid or text representation, the dependency arrows, the per-component sourcing-recommendation records in the format from `shared/wardley-map-template.md`, the evolution-gap and inertia findings, the `watch` items with review dates, and the build-vs-buy-vs-rent posture summary for `architecture-explorer`; also authors `adr/NNNN-sourcing-<component>.md` per significant sourcing decision that is a one-way door, per `shared/adr-template.md`; the evaluator never authors artifacts) → **Verify** (re-check the hard oracles from the authored artifact: anchor named, every `(H,*)` component placed, every sourcing recommendation has a driver or "commodity default", every evolution gap raised as a finding, no inertia silently accepted; evaluator re-scores craft — honest placement, commodity hygiene, FinOps alignment, time-sensitivity surfaced; rubric mean ≥ 0.8) → loop ≤10 or pass.

## Assertions & exit gate
- The map has a named **anchor user need** at the top of the chain — a concrete user capability, not "the system."
- Every component that appears in a `(H,*)` quality-attribute scenario or in the C4 container/component diagram is placed on the map with its evolution stage, differentiation status, and sourcing recommendation.
- Every sourcing recommendation carries a **primary driver** — a quality-attribute scenario ID, a constraint, or "none — commodity default." An untraced recommendation is a preference, not a finding, and is dropped.
- Every **evolution gap** (Stage III–IV component operated in-house without a differentiating driver) is raised as an `ARCH-WARD-NNN` finding (`type:"cost"` or `type:"operability"`, `oracle:"soft"`) with a TCO-evaluation cross-reference.
- Every **inertia flag** is named and not silently accepted; the organizational or legacy constraint holding it in place is stated.
- Every **contested placement** is flagged with the stage range and the uncertainty is carried as a soft finding — it is never resolved by assertion.
- Every significant sourcing decision that is a **one-way door** (datastore, auth model, deployed runtime) is an ADR with context · ≥2 options · decision · consequences (positive AND negative) · status.
- The completed sourcing-recommendation set is structured for consumption by `architecture-explorer` (posture constraint: Stage III–IV, no differentiating driver → managed/utility variant in the option set) and by `workflows/90-specialized/cost-finops-architecture.md` (TCO confirmation).
- **Gate:** hard oracles green (anchor named · every `(H,*)` component placed with stage + driver + recommendation · every evolution gap and inertia flag raised as a finding · one-way-door sourcing decisions are ADRs with rejected alternatives) AND (design) rubric mean ≥ 0.8 (honest industry-maturity placement · commodity hygiene · FinOps alignment · pioneer/settler/town-planner alignment checked · time-sensitive watch items surfaced with review dates · tradeoff honesty — rejected sourcing path named and its strengths stated).

## Output
Write `artifacts/wardley-evaluation/report.json` per `shared/report-format.md` — `drivers[]` (the prioritized scenarios whose components are placed on the map), `findings[]` per `shared/finding-schema.md` (`ARCH-WARD-NNN`; `type:"cost"` / `type:"operability"` / `type:"risk"` for evolution-gap, inertia, and contested-placement findings; `oracle:"soft"` for these; `oracle:"hard"` for an absent anchor or a missing `(H,*)` component; every finding carries `driver_ref`, `tradeoff` — what building a commodity costs on operability/FinOps vs what renting costs on control/compliance, `alternatives_rejected` — the rejected sourcing path and what it was better at), `decisions[]` (one-way-door sourcing ADRs with `driver_ref`), and the `verification` block (anchor present, component coverage, per-component sourcing-recommendation count, evolution-gap findings count, inertia-flag count, hard-oracle results, rubric score). Design artifacts land in `architecture/<slug>/`: the map in `wardley-map.md` (per `shared/wardley-map-template.md` — the 2D grid or text representation, dependency arrows, per-component sourcing records, watch list with review dates, build-vs-buy-vs-rent posture summary) and one-way-door sourcing decisions in `adr/NNNN-sourcing-<component>.md`. Never scatter the map or ADRs elsewhere; the machine record stays in `artifacts/wardley-evaluation/`, the durable artifact in `architecture/<slug>/`.

## Guardrails
Per `shared/guardrails.md`: **propose, map, and document — never write product code, never apply infra** (this flow positions components and records sourcing decisions; the controller builds the winners, the operator provisions or procures). Ground every evolution-stage placement in *industry maturity*, never in team familiarity — if the industry landscape is unknown for a component, label it "Genesis (unverified)" and flag it for market research; do not assert a stage. The map is a **soft input** — its findings are `oracle:"soft"` and advisory; a sourcing recommendation from the map that contradicts a grounded TCO model from `workflows/90-specialized/cost-finops-architecture.md` yields to the TCO model. Do not build what could be rented without a named driver; do not rent a differentiating genesis capability without a named driver — both are soft failures. Right-size: `project-architecture-config.md` locked rules apply (the project's context and scale, per `project-architecture-config.md`, bound the map; do not let the map rationalize résumé-driven or commodity-building choices). A separate skeptical evaluator scores craft (never the proposer grading itself). Read-only audit authors nothing; exactly one writer `solution-architect` authors the map artifact and ADRs for conceptual integrity; the evaluator never edits the architecture it grades. Never write to project memory — lessons and findings go in the returned report and ADRs. Cross-references: `shared/wardley-map-template.md` (the canonical map construction rules, stage definitions, sourcing-recommendation record format, and qualitative limits — the soft-input compact); `workflows/10-styles-decomposition/architecture-style-selection.md` (the style decision that follows from the sourcing posture — what you build and what you rent shapes the style); `workflows/90-specialized/cost-finops-architecture.md` (TCO confirmation that accepts or overrides the positional hypothesis; the rigorous cost model this flow prompts but does not replace).
