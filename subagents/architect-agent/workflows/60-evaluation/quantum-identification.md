# quantum-identification

**Group:** 60-evaluation  ·  **Runs as:** subagent: ../.claude/agents/architecture-evaluator.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Identify the **architecture quanta** — the independently-deployable units bounded by synchronous connascence — so that every downstream evaluation flow (especially `architecture-evaluation-atam.md`) scores each quantum against its *own* utility sub-tree, never against a misleading system-wide tree. Answers the prerequisite question: *what are the independently-deployable runtime units, and which quality-attribute scenarios apply per unit vs the whole system?*

This flow implements **Step 0** of `shared/evaluation-method.md`: it must complete and pass before `architecture-evaluation-atam.md` or any per-scenario scoring begins. A system-level utility tree that ignores quantum boundaries misleads; this flow prevents that.

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target (what runtimes/processes are independently deployable?), the team topology (Conway — boundaries should fit what can be independently owned), the constraints (monolith lock vs service decomposition), and where the architecture description / ADR log / C4 diagrams live (`architecture/<slug>/`).
- The **architectural drivers**: the prioritized quality-attribute scenarios (the utility tree) produced by `../00-drivers/quality-attribute-scenarios.md`. Without a utility tree this flow produces the quantum map but cannot assign per-quantum attributes — record that as a blocking open question for the controller.
- The architecture description under review: the C4 container + component views, the service/process topology, the deployment diagram, and the ADR log in `architecture/<slug>/`. In audit mode, the as-built system is scanned from the repo. In design mode, the proposed architecture description is the input.
- Preconditions: source-of-truth docs (`docs/`, BRD/PRD, `project-understanding.md`) read; the call-type of every inter-component interaction is visible (synchronous HTTP/gRPC vs asynchronous event/message) — a call whose synchrony is unknown is a gap that must be surfaced, never assumed.

## Oracle (source of truth)
The **architecture quantum** definition from Richards & Ford (*Fundamentals of Software Architecture*, 2020): an independently deployable artifact with high functional cohesion and **synchronous connascence** — i.e. components that must respond synchronously to serve a request are in the same quantum regardless of how many processes they span. The boundary is *not* a team boundary, a bounded-context boundary, or a code-module boundary — it is the synchronous-call chain that couples components at runtime.

- **hard:**
  - Every inter-component interaction in the architecture is classified as synchronous (same quantum) or asynchronous (quantum boundary) — no interaction left unclassified.
  - Every identified quantum is independently deployable: it can be built, deployed, and started without co-deploying another quantum (i.e. no **static connascence** — shared compile-time artifacts, libraries that force co-deployment — crosses a claimed quantum boundary; if it does, the boundary is wrong or the coupling must be acknowledged as an exception with a driver-justified ADR).
  - Every quantum with **materially different quality-attribute characteristics** (different peak-load profile, different availability target, different latency budget, different compliance boundary, different deployment cadence) has its own utility sub-tree — not sharing a parent tree that averages these differences away.
  - No evaluation or finding in `architecture-evaluation-atam.md` states "the system is scalable," "the system meets p99 ≤ 200ms," or any other system-wide claim that conflates the characteristics of distinct quanta — every such claim names the quantum it applies to.
  - Shared system-wide attributes (e.g. corporate security posture, compliance regime, a single identity provider) stay at the **root utility tree** and are not duplicated into each per-quantum sub-tree.

- **soft:**
  - Quanta are cut at *real* synchronous runtime boundaries, not at convenient module or team lines — if the evidence shows a synchronous call chain crossing a proposed boundary, the boundary is wrong.
  - The quantum map is no more granular than the deployment topology supports: a monolith that deploys as one artifact is one quantum unless a genuinely independent sub-process (e.g. a separate worker process) is also present.
  - The per-quantum sub-trees are scoped to the attributes that *differ* between quanta; attributes shared across all quanta appear once at the root (DRY applies to the utility tree too).
  - The classification rationale for each boundary is stated in one sentence — if it cannot be stated in one sentence, the boundary is likely wrong.

## Standards & techniques
- **Architecture quantum** (Richards & Ford, *Fundamentals of Software Architecture*, 2020, Ch. 6) — the independently-deployable unit with high functional cohesion and synchronous connascence. The term is grounded in **connascence theory** (Page-Jones, *What Every Programmer Should Know About Object-Oriented Design*, 1996): static connascence (coupling at compile/link time) and dynamic connascence (coupling at runtime — synchronous is tighter than asynchronous). A quantum boundary is where *synchronous* dynamic connascence ends.
- **Synchronous vs asynchronous connascence as the boundary rule** — two components share a quantum if and only if they must both be available and responsive in the same request/response cycle (sync). Components coupled only through a durable message broker or event log (async) are in different quanta — the broker absorbs the temporal coupling.
- **Per-quantum quality-attribute scoping** (`shared/evaluation-method.md`, Step 0) — the method for assigning scenarios from the root utility tree to per-quantum sub-trees: a scenario whose response measure (p99, RTO, throughput) is driven by a specific quantum's behavior is assigned to that quantum's sub-tree; scenarios that apply uniformly (e.g. data-classification compliance) stay at the root.
- **C4 container diagram** (Brown, *Software Architecture for Developers*, 2018) — the container-level view (deployable units + their communication style) is the primary source for quantum identification. Synchronous arrows (HTTP, gRPC, direct DB call) collapse containers into one quantum; async arrows (message broker, event stream) are potential quantum boundaries.
- **Static connascence check** — scan the dependency graph (package imports, shared libraries, compile-time schema coupling) for dependencies that cross a proposed quantum boundary. Any static connascence that forces co-deployment is either a misdrawn boundary or an explicit coupling to record as an exception in an ADR.
- **Conway's Law** (Melvin Conway, 1968) — verify that the quantum boundaries fit (or deliberately reshape, with documented intent) the team topology from `project-architecture-config.md`. A quantum that spans two teams with no shared ownership is a governance gap, not just a structural gap.

## Step sequence
- **audit:** Read the architecture description (C4 container/component views, deployment diagram, ADR log) and the source-of-truth docs → enumerate every inter-component interaction and classify each as synchronous or asynchronous → group co-synchronous components into candidate quanta → validate each candidate: (a) independently deployable? (b) no static connascence crossing the boundary? → for each confirmed quantum, compare its quality-attribute characteristics (load profile, latency budget, availability target, deployment cadence, compliance boundary) against the others — flag quanta whose characteristics are materially identical as merge candidates; flag quanta with materially different characteristics as requiring separate sub-trees → map each scenario from the existing utility tree to its owning quantum or to the shared root → emit the quantum map as findings + a proposed scenario-to-quantum assignment table. Read-only: authors no artifact, edits no source.

- **design:** Frame (read the proposed architecture description, the utility tree, and the team topology — confirm that sources exist and that call-type is stated for every interaction; surface unknowns as blocking open questions, never assume) → Explore (enumerate candidate quantum boundaries from the container diagram + the synchronous-call chains; classify every interaction; build N ≥ 2 candidate quantum maps when the boundary is non-obvious — one per plausible cut of ambiguous interactions — each concrete and fully specified, not a list of adjectives) → Evaluate (skeptical evaluator scores each candidate map against the hard oracle — independently deployable? static-connascence-free? per-quantum sub-trees for materially different characteristics? Conway-aligned? pick the map with the fewest unjustified quantum splits; graft the superior boundary rationale from the runner-up candidates where it doesn't break conceptual integrity; order-swapped when comparing two close candidates) → Document (the **one writer** `solution-architect` authors the quantum map artifact: `architecture/<slug>/quantum-map.md` — the list of quanta, each with its deployment unit, its synchronous boundary rationale, its per-quantum utility sub-tree, and a scenario-to-quantum assignment table; raises an ADR if a proposed quantum boundary is a new architectural decision not previously recorded) → Verify (re-score the quantum map from the artifacts: every interaction classified, no static connascence uncrossing a boundary without an ADR, every quantum with differing characteristics has its own sub-tree, every `(H,*)` scenario assigned to exactly one quantum or the shared root; fitness function: the deployment topology in `project-architecture-config.md` matches the claimed independently-deployable artifact count) → loop ≤ 10 or pass.

## Assertions & exit gate
- Every inter-component interaction in the architecture description is classified as synchronous or asynchronous — no unclassified interaction remains.
- Every claimed quantum boundary is justified by the synchronous-connascence rule in one sentence; any static connascence crossing a boundary is either a boundary-correction finding or an ADR-accepted exception.
- Every quantum is verified as independently deployable: it can be started/stopped/deployed without co-deploying another quantum.
- Every quantum with materially different quality-attribute characteristics (load profile, latency budget, availability target, deployment cadence, compliance boundary) has its own utility sub-tree; quanta with identical characteristics share one sub-tree (stated explicitly, not silently merged).
- Every `(H,*)` scenario from the root utility tree is assigned to exactly one quantum's sub-tree or explicitly retained at the shared root — no scenario is stranded without a quantum owner.
- The quantum count is consistent with the deployment topology in `project-architecture-config.md` — no phantom quanta (more claimed than deployable units), no merged quanta (fewer claimed than the deployment topology shows as independently deployable).
- No downstream evaluation finding states a system-wide quality claim without naming the quantum it applies to.
- **Gate:** all hard oracle checks above are green AND (design) the authored quantum map artifact re-scores green with rubric mean ≥ 0.8.

## Output
Write `artifacts/quantum-identification/report.json` per `shared/report-format.md` — findings per `shared/finding-schema.md` (use `type:"decomposition"` for misdrawn quantum boundaries; `type:"driver"` for scenarios with no quantum owner; `oracle:"hard"` for any unclassified interaction, static-connascence boundary violation without an ADR, or quantum with materially different characteristics but no per-quantum sub-tree; `oracle:"soft"` for over-granular quanta or DRY violations in the utility tree), plus the verification block (the interaction-classification table, the quantum-to-scenario assignment table, the static-connascence scan result, the deployment-topology consistency check).

In design mode the authored artifact lands in `architecture/<slug>/quantum-map.md`; any boundary decision not previously in the ADR log is raised as a new ADR in `architecture/<slug>/adr/` (`shared/adr-template.md`). The report lists those paths. Open questions (an interaction whose synchrony is unknown, a deployment unit count inconsistency, a scenario with no measurable response measure that cannot be assigned) are listed explicitly for the controller — never silently assumed.

Cross-references: `../00-drivers/quality-attribute-scenarios.md` (the root utility tree this flow partitions), `architecture-evaluation-atam.md` (the downstream flow that consumes the per-quantum sub-trees), `shared/evaluation-method.md` Step 0 (the authoritative definition this flow implements).

## Guardrails
Per `shared/guardrails.md`: **propose & evaluate — never write product code and never edit the architecture being graded** (this flow is read-only in audit mode; the one writer `solution-architect` authors `quantum-map.md` in design mode, sequenced). Ground every boundary in the synchronous-connascence rule — a quantum boundary justified by "team ownership" or "domain model" alone, without a runtime synchrony rationale, is not a valid cut; name the driver it serves or correct the boundary. Reuse the container diagram and the deployment topology before inferring quanta from code structure — the deployment unit, not the code module, is the unit of the quantum. Record the tradeoff and the rejected alternatives for every boundary decision: a finer-grained cut costs operational complexity and independent-deploy overhead; a coarser-grained cut risks coupling quanta with incompatible characteristics. Never write to project memory — findings and the quantum map live in the report and `quantum-map.md`. The evaluator never closes the revise loop on its own word past the external-signal gate (`shared/evaluation-method.md` — Evaluator discipline).
