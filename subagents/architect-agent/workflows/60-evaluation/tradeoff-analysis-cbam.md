# tradeoff-analysis-cbam

**Group:** 60-evaluation  ·  **Runs as:** subagent: ../.claude/agents/architecture-evaluator.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
When two or more options each clear the drivers, "both work" is not an answer — pick the one with the best **return on the analysis dollar**. This flow quantifies the tradeoff with **CBAM** (Cost-Benefit Analysis Method): score every option against each high-priority quality-attribute scenario in a **decision matrix** (showing *where each wins and loses*, never one blended number that hides the tradeoff), estimate **cost = build + run** over the horizon, compute **benefit = Σ (scenario-utility × priority weight)**, rank by **ROI**, and classify each underlying decision as a **one-way door (irreversible)** or **two-way door (reversible)** so the analysis depth matches what's at stake. The deliverable is a ranked recommendation that says what it costs, what it beat, and what the rejected option was *better* at. It answers: *given the constraints, which option returns the most utility per dollar, and how much rigor does this decision actually deserve?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target, the stated scale/SLO/cost constraints (per `project-architecture-config.md` — **right-size, never résumé-drive**; do not let CBAM rationalize multi-region/Kafka/k8s absent a driver), the compliance regime, the team topology (small team — Conway: every option's **run cost** is bounded by what this team can operate), and where `architecture/<slug>/` + the ADR log live.
- **The utility tree** (`workflows/00-drivers/quality-attribute-scenarios.md` → `architecture/<slug>/quality-attributes.md`): the prioritized 6-part scenarios with **response measures** and **(business value, technical risk)** weights. CBAM weights benefit by these priorities — **no utility tree, no CBAM** (`status:error`, downgrade to qualitative ATAM tradeoff-points only).
- **≥2 candidate options that all clear the hard oracles** — from `architecture-explorer` (design mode) or the two real options on the table (audit mode). CBAM is for choosing *between viable options*, not for rescuing one that misses a response measure: an option that fails a hard oracle is eliminated *before* it enters the matrix, not down-weighted inside it.
- The horizon for cost (e.g. 1–3 yr TCO) and the budget constraint to rank under; the build-cost + run-cost inputs (infra unit prices, eng-effort estimate, operational/opportunity cost). Where cost is itself a first-class driver, defer the rigorous model to `workflows/90-specialized/cost-finops-architecture.md` and consume its numbers here — this flow *uses* the cost model, it doesn't replace it.
- Preconditions: the source-of-truth docs (BRD / PRD / understanding log) named in `project-architecture-config.md` — real NFR numbers, **never invent** — read; the options expressed as concrete specs (topology + key decisions), not slogans.

## Oracle (source of truth)
The **drivers (the utility tree) + the cost model + the reversibility class** — not "the better-architected option," not the more impressive stack. CBAM ranks by ROI against the *priorities the business set*, and the analysis budget is itself gated by whether the decision is a one-way door.

- **hard (deterministic — traceable / measurable / structural):**
  - **Per-scenario matrix, not a blended score.** Every option is scored against *every high-priority `(H,*)` scenario* in a **weighted decision matrix** — the cell shows whether that option meets the scenario's **response measure** and at what tradeoff. A single averaged "8.5 vs 7.2" that collapses a latency win and an availability loss into one number is a **fail** (it hides the tradeoff point — the whole deliverable). The matrix must surface where each option *wins and loses*.
  - **Benefit is computed, not asserted.** `benefit(option) = Σ over scenarios (utility_contribution × priority_weight)`, where `priority_weight` derives from the scenario's `(business value, technical risk)` rank and `utility_contribution` is the marginal utility this option delivers on that scenario (e.g. p99 220ms → 140ms ⇒ utility 60→90 on QAS-PERF-01). Every utility number traces to a scenario in the tree; an untraced utility score is a preference, not a finding.
  - **Cost is estimated (build + run, over the stated horizon).** Each option carries an **order-of-magnitude** `cost = build (eng-effort) + run (infra + operational + opportunity)` against the stated budget. "Scale infinitely, cost unstated" fails `cost_bounded`. Cross-region egress / managed-service fees / operational headcount are in the run figure (fallacy 7 — transport cost ≠ zero; see `shared/fallacies-and-tradeoffs.md`).
  - **ROI ranking under the constraint.** `ROI = benefit / cost`; options are ranked by ROI and the winner is the highest-ROI option **that fits the budget and violates no constraint**. The gold-plated option with higher raw utility but **sub-1 ROI against the actual driver** is explicitly rejected *for that reason* — CBAM exists to justify *not* gold-plating.
  - **Reversibility classified per decision.** Each decision under analysis is labeled **one-way door (irreversible/expensive to undo — the datastore, the auth model, the public API contract, the prod-deploy topology)** or **two-way door (reversible/cheap to undo)**, with the cost-to-reverse stated. The classification sets the rigor: a misclassification that spends full CBAM rigor on a trivially reversible config choice, *or* that fast-tracks a one-way door, is a hard finding.
  - **Winner is ADR-shaped.** The recommendation names the chosen option + the response-measures it meets + the **tradeoff** + the **rejected alternatives and what each was better at** (`shared/adr-template.md`); a recommendation with only upside, or no recorded losers, fails `adrs_complete`.

- **soft (rubric / craft — scored only after hard oracles are green):**
  - **Reversible decisions are made fast and cheap.** Two-way doors are *decided*, not over-analyzed — pick the default, note it's reversible, move on. Spending a full decision matrix on a config flag is **analysis paralysis** (Bezos: most decisions are two-way doors and should be made by people with ~70% of the information; waiting for 90% is slowness). Penalize over-analysis of the reversible.
  - **The analysis budget is spent on the one-way doors.** The depth, the sensitivity analysis, and the explore-fan-out concentrate on the irreversible/high-blast-radius decisions; that's where rigor pays.
  - **The recommendation states what the rejected options were *better* at** — not just that they lost. "Buy-managed won on TCO and time-to-market; we chose build for the data-residency control it can't give us" is honest craft; "build is better" is not.
  - **Right-sizedness honored** — CBAM is not a license to launder accidental complexity past the rubric; if the high-ROI option is the *boring* one, say so plainly.

## Standards & techniques
- **CBAM (SEI Cost-Benefit Analysis Method)** — the economic extension of ATAM: each architectural strategy carries a **benefit** (Σ scenario-utility × priority) and a **cost** (build + run over the horizon); strategies are ranked by **ROI = benefit/cost** and chosen under the budget constraint. CBAM's job is to make the *cost* of utility visible so the team stops buying utility no driver demanded (`shared/evaluation-method.md` → CBAM).
- **Weighted decision matrix** — scenarios (rows, weighted by utility-tree priority) × options (columns); each cell = met/unmet + the marginal utility + the tradeoff. The matrix's value is the *shape of the win/loss profile per option*, not the column total. Position-bias controls apply (see below).
- **One-way-door vs two-way-door framing (Bezos / "Type 1 vs Type 2" decisions)** — irreversible, consequential decisions warrant heavy, deliberate analysis; reversible decisions should be made quickly and cheaply and *un-made* if wrong. The reversibility class is the throttle on how much CBAM rigor to spend — and the antidote to **analysis paralysis** on the two-way doors.
- **ATAM sensitivity & tradeoff points** (`shared/evaluation-method.md`) — CBAM inherits ATAM's vocabulary: a **sensitivity point** is a decision where one property strongly determines whether a scenario is met (e.g. ROI is *sensitive* to the assumed eng-day rate or the cache hit-rate); a **tradeoff point** is a decision that helps one attribute and hurts another (the matrix's cross-scenario rows). Name them; they drive the risk register.
- **The canonical tradeoffs you choose but never escape** (`shared/fallacies-and-tradeoffs.md`) — Build↔Buy, Cost↔Resilience/Performance, Flexibility↔Simplicity (YAGNI), Consistency↔Availability/Latency. CBAM quantifies the *point chosen* on the curve; the rejected option sits at a different, justified point.
- **AWS/Azure Well-Architected — Cost Optimization pillar** as the cost-modeling frame (right-sizing, build-vs-buy economics, serverless-vs-provisioned, the cost of each decision); feed/consume the rigorous figures from `workflows/90-specialized/cost-finops-architecture.md`.
- **Sensitivity / what-if on the assumptions** — re-rank under the pessimistic and optimistic cost/utility inputs; if the ROI ranking *flips* under a plausible assumption, that assumption is a sensitivity point and a risk (don't ship a winner that only wins under the rosy estimate).

## Step sequence
- **audit:** Frame the two real options + pull the utility tree and the cost inputs → build the **weighted decision matrix** (each option × each `(H,*)` scenario, met/unmet + utility + tradeoff) → compute benefit, estimate build+run cost, derive ROI → classify each decision one-way/two-way + state cost-to-reverse → run the sensitivity what-if on the cost/utility assumptions → emit findings (the recommended option with its tradeoff + rejected alternatives, the tradeoff/sensitivity points, the reversibility risks) **read-only — authors no artifact, edits no source**.
- **design:** **Frame** (confirm drivers + the viable options that already clear the hard oracles, read-only) → **Explore** (if <2 viable options, request divergent directions from `architecture-explorer`; CBAM does not invent options, it ranks them — read-only) → **Evaluate** (build the decision matrix scored against the utility tree; compute benefit/cost/ROI; classify reversibility; locate sensitivity + tradeoff points; **judge in one order then swap the column order and judge again** to kill position bias; penalize the more-elaborate option's accidental complexity — read-only, order-swapped) → **Document** (the **one writer** `solution-architect` authors `architecture/<slug>/tradeoffs.md` = the decision matrix + benefit/cost/ROI table + the one-way/two-way classification + sensitivity/tradeoff points + the risk register, and the winning decision as a MADR ADR per `shared/adr-template.md` with the rejected options and *what each was better at*; preserves conceptual integrity, reuses the known tradeoff vocabulary; the evaluator never edits the architecture it grades) → **Verify** (re-check the hard oracles from the artifacts: every `(H,*)` scenario is a matrix row, benefit/cost/ROI are computed not asserted, reversibility classified, the ADR carries rejected alternatives + tradeoff; the evaluator re-scores craft from the artifacts, not the prose) → loop ≤10 or pass.

## Assertions & exit gate
- Every high-priority `(H,*)` scenario in the utility tree is a **row** in the decision matrix; each option's cell states met/unmet vs the **response measure** + the tradeoff — no single blended score stands in for the per-scenario profile.
- **Benefit** = Σ(utility × priority) is computed and every utility number traces to a scenario; **cost** = build+run is estimated to order-of-magnitude against the stated budget; **ROI** = benefit/cost is the ranking key.
- The recommended winner is the **highest-ROI option that fits the budget and violates no constraint**; any higher-utility/sub-1-ROI option is explicitly rejected *as gold-plating against the actual driver*.
- Each decision is classified **one-way door / two-way door** with its cost-to-reverse; analysis depth matches the class (reversible decisions decided fast; rigor concentrated on the irreversible).
- A **sensitivity what-if** was run on the cost/utility assumptions; if the ranking flips under a plausible input, that assumption is logged as a sensitivity point + risk.
- The winning decision is an **ADR** (context · ≥2 options · decision · consequences incl. **negative**) that records the rejected alternatives **and what each was better at**.
- **Gate:** hard oracles green (per-scenario matrix not a blended number · benefit+cost+ROI computed and traced · reversibility classified · winner is highest-ROI-under-constraint · ADR complete with rejected alternatives + tradeoff) AND (design) rubric mean ≥ 0.8 (reversible decisions not over-analyzed · budget spent on the one-way doors · rejected option's strengths stated · no laundered accidental complexity).

## Output
Write `artifacts/tradeoff-analysis-cbam/report.json` per `shared/report-format.md` — `drivers[]` (the prioritized scenarios used as matrix rows), `findings[]` per `shared/finding-schema.md` (`type:"risk"` for the reversibility/sensitivity risks with likelihood×impact; `oracle:"hard"` for a blended-score-hiding-the-tradeoff, an asserted-not-computed benefit/cost, an unclassified one-way door, or a winner picked on raw utility over ROI; each finding carries `driver_ref`, `tradeoff`, and `alternatives_rejected`), the `decisions[]` index (the chosen ADR + driver_ref), and the `verification` block (matrix completeness, ROI ranking, reversibility classification, sensitivity result, rubric score). The design artifacts land in **`architecture/<slug>/`**: the option matrix + benefit/cost/ROI table + one-way/two-way classification + sensitivity/tradeoff points + risk register in `tradeoffs.md`, and the winning decision as `adr/NNNN-*.md`. Never scatter the matrix or the ADR elsewhere; the machine record stays in `artifacts/`, the durable artifact in `architecture/<slug>/`.

## Guardrails
Per `shared/guardrails.md`: **propose, quantify, and document — never write product code, never apply infra** (this flow ranks options and records the decision + its tradeoff; the controller builds the winner, the operator provisions it). Ground every benefit number in a utility-tree scenario and every cost number in the cost model — **no invented NFR or price figures**; if a cost input is unknown, surface it as an open question and run the sensitivity what-if, don't guess a winner on a fabricated number. CBAM ranks *viable* options only — eliminate any option that fails a hard oracle before the matrix, never down-weight it inside. Right-size: do not let an ROI calculation rationalize accidental complexity no driver demanded (per the project's locked rules in `project-architecture-config.md`). A separate skeptical evaluator scores craft (never the proposer grading itself), order-swapped to kill position bias and penalizing the more-elaborate option's complexity. Record the tradeoff and the rejected alternatives + what each was better at — a recommendation with only upside is incomplete. Read-only audit authors nothing; exactly one writer `solution-architect` authors the matrix + ADR for conceptual integrity, and the evaluator never edits the architecture it grades; never write to project memory — lessons go in the returned report/ADR.
