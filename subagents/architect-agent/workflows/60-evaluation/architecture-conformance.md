# architecture-conformance

**Group:** 60-evaluation  ·  **Runs as:** subagent: ../.claude/agents/architecture-evaluator.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Verify the **built** system conforms to its **own** architecture — the ADR decisions, the layering/dependency rules, and the documented module boundaries — via a **static-tool-first, LLM-reasoning-second** pass that is **read-only** on the codebase. Answers: *does what's actually in the repo still match what the ADRs and the dependency rules say it should be, and where it drifted, which decision does the drift break?* This is the reflexion check between the **as-documented** architecture (`architecture/<slug>/`) and the **as-built** one (the import graph, the layering) — conformance checking closes the loop that a design-time ADR alone cannot: a decision can be perfectly reasoned and still silently erode one commit at a time.

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack (per the project's config — e.g. a modular monolith with a client tier, an API tier, and a single primary datastore, so conformance is checked at the **module-boundary/layering** level, not a service-mesh contract level), the artifact locations (`architecture/<slug>/`), and the locked rule that a new tool/datastore needs a driver-justified ADR (so an undocumented dependency is itself a conformance gap, not just a style nit).
- The **documented architecture**: the ADR log (`architecture/<slug>/adr/`) — every ADR whose `Follow-ups / fitness functions` or `Decision outcome` states a structural rule (a layering rule, an allowed/forbidden import direction, an ownership boundary) is a conformance rule to check; the C4 Component view / dependency graph from `../10-styles-decomposition/dependency-management.md` (`architecture/<slug>/diagrams/dependency-graph.*`) is the **reflexion model** — the intended-architecture graph the as-built graph is diffed against.
- The **as-built system**: the actual import/dependency graph of the running codebase (client-side module tree, server-side module imports — per the project's layout in `project-architecture-config.md`), extracted by a **static conformance tool**, never inferred from prose or from an LLM reading source files in isolation.
- A flow with no ADRs and no dependency-graph model to check against **downgrades scope and says so** (`status:"error"`) — conformance cannot be checked against a design that was never documented; recommend `../10-styles-decomposition/dependency-management.md` and `../70-documentation/architecture-decision-records.md` run first.

## Oracle (source of truth)
The **static conformance tool's output** is the hard gate — deterministic, re-runnable, exit-coded. The LLM (the `architecture-evaluator`) never *decides* pass/fail on layering; it explains, prioritizes, and clusters what the tool already found, and separately flags **candidate** architectural smells as soft advisory. This split exists because **pure-LLM architectural-smell/violation detection is precision-limited** (~40% F1 in published smell-detection studies) — an LLM reading code and asserting "this violates the layering rule" without a structural model produces too many false positives/negatives to gate a build on. Gate on the tool; use the LLM for the second-order job (why it matters, what to fix first) it is actually reliable at.

- **hard:**
  - **Dependency/layering conformance is checked by a STATIC tool** — the tool appropriate to the project's stack per `project-architecture-config.md` (e.g. ArchUnitTS / `dependency-cruiser` for a JS/TS client; `deptrac` or `import-linter`/`pytestarch` for a Python backend; the equivalent for whatever languages the project actually uses) runs against the real import graph and asserts the **reflexion model**: the intended layering (from `component-design.md` / `dependency-management.md`'s dependency-graph diagram) vs the actual graph. Every **divergence** the tool reports — a forbidden edge, a layer skip, a module importing something the intended graph disallows — is a `hard` finding. The LLM does not re-derive this graph from memory; it consumes the tool's JSON/exit-code output.
  - **Every violation maps to the ADR/rule it breaks** — a tool-reported divergence with no traceable `decision_ref` (an ADR) or documented rule (`dependency-graph.*`) is either (a) matched to the nearest documented rule it contradicts, or (b) surfaced as an **undocumented-but-real constraint** (the code enforces a boundary no ADR records) — never silently dropped. No orphan violations.
  - **The conformance gate is the tool's exit code**, not the LLM's read of the diff — `status:"fail"` is set by the tool run (non-zero exit / rule violations > 0), and the LLM's role is recorded separately in the report so a reviewer can see the gate was deterministic, not a judgment call.
- **soft:**
  - **Candidate architectural smells the LLM flags** (God-module concentration, an ADR whose rule the static tool *can't* structurally express — e.g. a semantic rule like "the event log must be append-only" — a naming/ubiquitous-language drift from the domain model) are `oracle:"soft"`, **explicitly labeled precision-limited**, and never block the gate on their own. They are advisory input for the controller to triage, not a finding the controller must act on.
  - The static tool's rule set itself is **complete enough** to cover the ADRs that state a structural rule (no ADR with a checkable structural rule left unencoded) — a coverage gap here is soft (it's a completeness/craft gap in the check set, not a live violation) but should trend to zero.

## Standards & techniques
- **Architecture conformance checking** (Passos et al.; Terra & Valente's *ArchLint*/*Arcan* lineage) — the discipline of comparing an **intended** architecture (a documented module/layer model) against the **as-implemented** one (the extracted dependency graph), reporting **divergences** (edges present in code but not in the model), **absences** (edges in the model never exercised — dead intent), and **convergences** (the confirmed-correct edges). This flow runs the divergence/absence/convergence triage explicitly, not just a pass/fail count.
- **Reflexion modeling** (Murphy, Notkin & Sullivan, 1995 — *Software Reflexion Models*) — the canonical technique: build the high-level model (here: the ADR-documented layering + the `dependency-graph.*` diagram), map source elements to model elements, compute the reflexion (model edges vs code edges), and triage each divergence. This flow's `hard` oracle *is* a reflexion-model diff, executed by a static tool, not eyeballed.
- **The static-first, LLM-second pattern** — run the deterministic tool first (the ArchUnitTS/dependency-cruiser/deptrac class of tool appropriate to the project's stack); feed its structured output (not the raw source) to the LLM for **explanation and prioritization** (which violation is worst, which ADR/driver it threatens, what the minimal fix is) and for a **separate, clearly-marked** smell-candidate pass. This is the load-bearing method citation for why the LLM is *not* the conformance oracle: published architectural-smell-detection studies put pure-LLM precision/F1 in the ~40% range — too low to gate a build; gating stays on the tool, the LLM adds judgment on top of a trustworthy signal.
- **Architectural smell taxonomy** (cross-ref `../80-patterns/anti-patterns-fallacies.md`) — God-component, unstable dependency, cyclic dependency, ambiguous interface, scattered functionality — named smells the LLM pass may flag as candidates; each candidate cites the specific structural evidence (afferent/efferent counts, the offending module pair) the same way a `finding-schema.md` entry requires, even though it's soft.
- **The dependency rule** (Martin, *Clean Architecture*) and the **acyclic-dependencies / stable-dependencies principles** (`../10-styles-decomposition/dependency-management.md`) — the concrete layering rules the static tool encodes: source dependencies point only inward/toward stability; no cycle.
- **Fitness functions as the executable form of a conformance check** (`fitness-functions.md`) — a conformance rule that recurs (rather than a one-time audit) should graduate into a fitness function wired into CI; this flow's audit pass is the one-time/periodic check, `fitness-functions.md` is where a rule becomes a standing CI gate.

## Step sequence
- **audit:** Read the ADR log + the `dependency-graph.*` reflexion model (or derive the intended layering from `component-design.md` if no diagram exists yet, and say so) → run the **static conformance tool** (the tool appropriate to each tier of the project's stack per `project-architecture-config.md` — e.g. `dependency-cruiser`/madge with a `forbidden`-rules config for a JS/TS client; `import-linter`/`pytestarch`/`deptrac` for a Python backend) against the real import graph in **report mode** (read-only, no config or source changed) → parse the tool's structured output into divergences/absences/convergences → map each divergence to the ADR/rule it breaks (or flag as undocumented-but-real) → **only then** hand the divergence list to the LLM to explain/prioritize/cluster by severity and blast radius, and run a **separate** smell-candidate pass (God-module, cyclic, scattered-functionality) explicitly labeled `oracle:"soft"` and precision-limited → emit findings grouped hard (tool divergences) then soft (smell candidates), plus a risk register for any undocumented-but-real constraint. Read-only throughout — no artifact authored, no source or tool config edited.
- **design:** **Frame** (confirm the ADR log + the reflexion model are current enough to check against; if the dependency-graph diagram predates recent ADRs, flag staleness as a precondition gap, read-only) → **Explore** (`architecture-explorer` proposes ≥2 remediation directions for the highest-severity divergences — e.g. (a) update the code to match the documented rule — the controller's job to execute, this flow only specifies it, (b) accept the divergence as a superseding decision and author a new ADR that supersedes the old rule, (c) tighten the static tool's rule config where the tool itself under- or over-constrains vs the true intent — each a concrete spec, read-only) → **Evaluate** (`architecture-evaluator` scores each direction against the driver the broken ADR served + the hard oracle — does the direction actually close the tool-reported divergence, does it avoid opening a new one; order-swapped to kill bias) → **Document** (ONE writer — `solution-architect` — authors: an ADR per accepted divergence that becomes a new decision, superseding the old one per `adr-template.md`; an updated `architecture/<slug>/diagrams/dependency-graph.*` reflexion model where the intended graph itself was stale; an updated conformance-rule config **specification** — the exact `dependency-cruiser`/`import-linter` rule text — for the controller to apply to the real tool config, never edited live by this flow) → **Verify** (re-run the static tool against the proposed rule/model changes in a dry mode; confirm every previously-open hard divergence now maps to either a closed rule or a superseding ADR; diagram-compile check on the updated reflexion model; evaluator re-scores from the artifacts, not from re-reading source) → loop ≤10 or pass.

## Assertions & exit gate
- The static conformance tool ran successfully (non-error exit distinguishing "ran, found violations" from "couldn't run") against the real import graph — a tool-run failure is `status:"error"`, never inferred as a pass.
- Every tool-reported divergence maps to a `decision_ref` (the ADR/rule it breaks) or is explicitly logged as an undocumented-but-real constraint — no orphan divergence silently dropped.
- The gate value (`pass`/`fail`) is set **from the tool's output**, never from the LLM's independent read of the violation list; the report shows both the tool's raw result and the LLM's explanation/prioritization as distinct fields so the split is auditable.
- Every smell-candidate finding is tagged `oracle:"soft"` and carries an explicit precision-limited caveat in its `evidence` field — it is never promoted to `hard` on its own, and it never blocks the gate by itself.
- For design mode: every remediated divergence is closed by either a re-run tool pass showing zero remaining violations for that rule, or a complete superseding ADR (context · ≥2 options · decision · consequences) that documents the accepted new state.
- **Gate:** hard oracles green (static tool ran; every divergence traced to a decision or logged as undocumented-but-real; gate value derived from the tool, not the LLM) AND (design) rubric mean ≥ 0.8.

## Output
Write `artifacts/architecture-conformance/report.json` per `shared/report-format.md` — findings per `shared/finding-schema.md` (`type:"decomposition"` for a layering/dependency-rule divergence, `type:"documentation"` for an undocumented-but-real constraint, `type:"risk"` for an accepted divergence pending disposition; `oracle:"hard"` for every static-tool-reported divergence, `oracle:"soft"` for every LLM-flagged smell candidate — each soft finding's `evidence` states the precision-limited caveat explicitly). Each finding's `evidence` names the tool + the exact rule id/edge (e.g. `"dependency-cruiser rule 'no-orphans': src/stores/sessionStore.ts → src/components/EntityTable.tsx, forbidden by ADR-0004 layering rule"`), `decision_ref` names the broken (or superseding) ADR, `tradeoff` and `alternatives_rejected` are mandatory for any design-mode remediation. The report's `verification` block records the tool name/version, the raw divergence/absence/convergence counts, and the rubric score. For design mode, authored artifacts (superseding ADRs, the updated reflexion-model diagram, the rule-config specification) land in `architecture/<slug>/` per `project-architecture-config.md`; the run report stays in `artifacts/`.

## Guardrails
Per `shared/guardrails.md`: **propose & document — never write product code**. This flow **never** edits the project's source directories (the client/server/service trees named in `project-architecture-config.md`), and it **never** edits the live static-tool config file used by CI — it authors the rule-config *specification* (the exact rule text) for the controller to apply, the same way a fitness-function spec is authored, not applied. The static tool itself only **reads** the import graph (report/dry mode); it is never invoked in a mode that rewrites source or auto-fixes imports. **Ground every finding in a driver/decision** — a divergence with no traced ADR is logged as undocumented-but-real, not silently upgraded to a finding with an invented rationale. **Static gates, LLM explains** — the evaluator (the LLM) is never the sole source of a `hard` verdict; it is always the tool's exit code / structured output. **One writer** authors any superseding ADRs and the updated reflexion model so the decision trail keeps conceptual integrity. **Reuse the established tool catalog** (ArchUnitTS, dependency-cruiser, deptrac, import-linter/pytestarch, or the equivalent for the project's stack) before proposing a bespoke conformance checker — a new checker needs a driver-justified reason the catalog tool can't express the rule. Cross-reference `fitness-functions.md` (a recurring conformance rule graduates into a CI-wired fitness function), `../10-styles-decomposition/dependency-management.md` (the acyclic/stable-dependencies rules this flow checks conformance against), and `../80-patterns/anti-patterns-fallacies.md` (the named smell catalog the soft LLM pass draws its candidate labels from). **Never write to project memory** — findings and smell candidates go in the returned report only.
