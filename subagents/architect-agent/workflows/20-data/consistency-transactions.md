# consistency-transactions

**Group:** 20-data  ·  **Runs as:** subagent: ../.claude/agents/data-architect.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
Decide the **consistency model per write path** — strong vs eventual vs causal — and pick the **mechanism for any cross-boundary write**: a saga + compensating transactions, an outbox + CDC, or a redrawn boundary that makes the transaction local again. The design question this flow answers: *for each thing the system writes, what does the reader see after the write, and when two systems must both change, how do we keep them in agreement without a silent dual-write?* The cardinal failure mode it exists to kill: a `db.save()` immediately followed by a `bus.publish()` (or a second `collection.update()`) with nothing tying the two together — the dual-write that loses data on partial failure and that no amount of "it usually works" makes correct.

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack and its consistency primitives (e.g. a single-primary document store today — note that a *single document/row* update is atomic but a write spanning two documents/tables is **not** a transaction unless explicitly wrapped in a multi-document/distributed transaction; where there is no replica set, no broker, no queue, any "publish after save" is already a latent dual-write); the near-term bounded contexts to keep seams for (e.g. documents, payments, closure/fulfillment) where cross-context writes will appear; the compliance regime (PII + financial data — money/settlement writes are the high-value invariants).
- The **drivers / utility tree** (`workflows/00-drivers/quality-attribute-scenarios.md`): the prioritized 6-part quality-attribute scenarios this decision must satisfy — specifically the data-integrity scenarios (e.g. *QAS-DATA-01: no financial transaction is ever lost or double-applied; QAS-DATA-02: order/invoice-number minting never produces a duplicate*) and any availability scenario the consistency choice trades against (PACELC: strong consistency costs latency/availability). A write path with **no stated invariant** downgrades scope — you cannot pick a consistency model without knowing what must stay true.
- The **target**: the specific write path(s) under review/design — what mutates, across how many storage roots / services / external systems (the primary store, a future event seam, an external payment processor), and the business invariant each must preserve. The relevant methods: CAP/PACELC, ACID-vs-BASE, the saga and outbox patterns, idempotency, DDD aggregate boundaries.
- Preconditions: the utility tree exists (else `status:error`); the data model / aggregate boundaries from `data-modeling` are read (the aggregate is the **natural unit of a local transaction** — consistency choices follow aggregate boundaries, not the other way round); the current write paths are enumerable from the code/schemas (audit mode reads them, never edits them).

## Oracle (source of truth)
The **business invariant** each write must preserve, made into a hard, structurally-checkable contract — never "the data is consistent." A money write (a settlement, a payment capture, a balance adjustment) and an analytics/audit write are governed by *different* invariants and therefore *different* models; the oracle is the invariant, expressed against CAP/PACELC and the named pattern's delivery guarantee.

- **hard** (measurable / traceable / structurally checkable):
  - **Every write path declares its consistency model** — `strong` | `eventual` | `causal` — and, where the write spans more than one storage root / service / external system, **its mechanism**: `saga+compensation` | `outbox+CDC` | `single-document/local-transaction` | `2PC`. A write path with no stated model **blocks** (the `consistency_stated` hard oracle, `../shared/quality-attribute-rubric.md`).
  - **No silent dual-write.** No code/design path writes to two systems (two databases, or a database **and** a message bus / external API) inside one logical operation without an outbox+CDC, a saga, or an explicit idempotent two-phase reconciliation. A `save()` then `publish()` (or two un-transacted `update()`s across collections that must agree) with no atomicity tying them is a **critical** hard finding — it loses the second write on a crash between them, and a retry of the first double-applies. This is the structural check that defines the flow.
  - **No distributed 2PC without explicit, signed-off justification.** Two-phase commit / `XA` across services or heterogeneous stores is rejected by default (it blocks on the coordinator — a violation of *fallacy #1 the network is reliable* and *#2 latency is zero*, and a synchronous availability sink); choosing it requires an ADR whose decision drivers show no saga/outbox/redrawn-boundary alternative holds the invariant.
  - **Cross-service / cross-aggregate consistency is eventual-with-compensation, OR the boundary is redrawn to make the write local.** If two things must change atomically and a single local transaction can't cover them, the design either (a) accepts eventual consistency and provides a **complete** compensation/forward-recovery path for every saga step that can fail, or (b) redraws the aggregate/bounded-context boundary so both changes land inside one local transaction. There is no third option of "we'll just do both writes and hope." Per CAP: under partition you get C **or** A, stated per path.
  - **Idempotency on every ret, ed/replayed write.** Any consumer of an outbox/event, any saga step, and any retried external call carries an idempotency key (or a natural dedup key) so at-least-once delivery does not become apply-twice. A retry-able write with no idempotency key is a hard finding (it defeats the outbox's at-least-once guarantee). Where ordering matters, causal/ordered delivery is stated, not assumed.
  - **Isolation level stated where concurrent writers contend.** For a write path with concurrent mutation of shared state (e.g. a sequence-counter store that mints an order/invoice number — a textbook lost-update if read-modify-write under the wrong isolation), the design names the isolation level / the atomic primitive (an atomic increment/find-and-modify primitive, optimistic-version check, or a serializable transaction) that prevents the anomaly (lost update / dirty read / write skew). "It's fine under load" with no named primitive blocks.
- **soft** (craft, rubric-scored only after hard oracles are green):
  - The consistency choice **matches the business invariant**: strong where correctness of money/orders is non-negotiable; eventual where staleness is harmless (dashboards, analytics, audit fan-out, read models). Picking strong consistency for an analytics write (paying latency/availability for nothing) or eventual for a settlement balance (risking a lost/double payment) is a craft failure even if it "works."
  - The **saga compensations are complete and semantically correct** — every step that can fail has a compensating action that actually restores the invariant (a *semantic* undo: refund, not delete; release-hold, not roll-back-time), and steps that cannot be compensated are ordered last (the pivot) or made idempotently retriable instead.
  - The **simplest mechanism that holds the invariant** is chosen (right-sizedness / YAGNI): a redrawn boundary that makes the write a single local transaction beats a saga; a single-document/row update beats a multi-document/multi-row transaction beats an outbox+CDC beats 2PC — climb that ladder only as far as the invariant forces. No saga where the two writes belong in one aggregate; no outbox+broker introduced where there is no second system to write to yet.

## Standards & techniques
- **CAP / PACELC** (`../shared/fallacies-and-tradeoffs.md`) — under a **P**artition choose **C** or **A**; **E**lse choose **L**atency or **C**. State the point on the curve **per write path / per store**, not once for the system. Synchronous replication buys consistency at latency's cost; async buys latency at consistency's. Naming the point chosen *is* the deliverable.
- **ACID vs BASE** — ACID (atomicity, consistency, isolation, durability) is the local-transaction contract within one store/aggregate; BASE (basically-available, soft-state, eventually-consistent) is the cross-boundary reality once a single transaction can't span the write. The flow's job is to keep each write **inside an ACID boundary where it can be**, and make it **explicitly BASE with a recovery path where it can't**.
- **The saga pattern** — a sequence of local transactions where each step publishes/triggers the next; on failure, **compensating transactions** semantically undo the completed steps. Two coordination styles, both valid, traded explicitly (`../30-integration/event-driven-patterns.md`):
  - **Orchestration** — a central saga coordinator drives the steps; visible, controllable, testable end-to-end, but a coupling point and a thing to operate.
  - **Choreography** — each service reacts to events and emits the next; decoupled and no central component, but the end-to-end flow is implicit and harder to reason about / debug.
- **The outbox pattern + CDC** — to write to a store **and** emit an event atomically, write the event row into an **outbox table/collection in the same local transaction** as the business write, then a separate relay (Change-Data-Capture tailing the oplog / a poller) publishes it to the broker. This converts a dual-write into a single local write + an at-least-once relay — the canonical cure for the dual-write problem. Pairs with an **idempotent consumer**.
- **Idempotency** — the property that applying a write more than once has the same effect as applying it once (idempotency key + dedup store, or a naturally idempotent operation like a `set`/upsert). The precondition that makes at-least-once delivery safe; without it, retries corrupt.
- **The dual-write problem** — the named anti-pattern this flow exists to detect: two independent writes to two systems in one operation with no atomic link, so a failure between them leaves the systems disagreeing and a retry double-applies. The fix is always outbox+CDC, a saga, or redrawing the boundary — never "add a retry."
- **Isolation levels & concurrency anomalies** — read-uncommitted → read-committed → repeatable-read → serializable, and the anomalies each prevents (dirty read, non-repeatable read, phantom, lost update, write skew). Name the anomaly the contended write path is exposed to and the level/primitive that closes it (single-document/row atomicity, an atomic increment/find-and-modify primitive, or multi-document/multi-row transactions with snapshot isolation; optimistic-concurrency version fields).
- The **8 fallacies** and the **canonical tradeoffs** (`../shared/fallacies-and-tradeoffs.md`) — consistency-vs-availability/latency, synchronous-vs-asynchronous, orchestration-vs-choreography, normalization-vs-read-performance: name the tradeoff *and the point chosen on it* for every decision.

## Step sequence
- **audit:** Frame the invariants from the utility tree (which writes are money/order-grade, which are analytics/audit-grade) → enumerate every write path in the existing/proposed architecture and, for each, walk: *how many storage roots / services / external systems does this one operation mutate?* → flag every path that mutates **two systems with no atomic link** as a dual-write (critical), every path with **no declared consistency model** (hard), every **retried/replayed write with no idempotency key** (hard), every **contended write with no named isolation primitive** (hard, e.g. the counter mint), and any **2PC used without a justifying ADR** → emit findings + the risk register (read-only; authors nothing, edits no source).
- **design:** **Frame** (extract/confirm the per-path invariants + the consistency-relevant scenarios from the utility tree, read-only) → **Explore** (≥2 divergent mechanisms per problematic write path as concrete specs — e.g. for a future order→payment write: *(A) redraw the boundary so payment state lives inside the order aggregate → single local transaction*; *(B) orchestrated saga: the order service coordinates payment via the external processor with a compensating refund/void step*; *(C) outbox+CDC: write the `PaymentRequested` event in the same transaction as the order state-change, relay to the payment consumer, idempotent by order-id — each a real spec with its CAP/PACELC point and failure path, read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each against the invariant + the hard oracles via ATAM — does the compensation actually restore the invariant? where's the sensitivity point (e.g. the saga pivot step that can't be undone)? the tradeoff point (saga = decoupled + resilient *but* eventually-consistent + harder to trace)? — order-swapped to kill bias; the simplest mechanism that holds the invariant wins, accidental complexity penalized) → **Document** (the **one writer**, `solution-architect`, authors: the per-write-path consistency table, the saga/outbox sequence diagram as diagrams-as-code, and one MADR ADR per significant choice with its rejected alternatives) → **Verify** (the hard gates below as fitness functions + the evaluator re-scores from the artifacts) → loop ≤10 or pass.

## Assertions & exit gate
- **Every write path has a row** in the consistency table: `{path, invariant, model (strong|eventual|causal), mechanism (local-txn|multi-doc-txn|outbox+CDC|saga[orch|choreo]|2PC), idempotency key, isolation/primitive}`. No blank cells on any architecturally-significant write.
- **No silent dual-write survives** — every operation that mutates two systems is covered by an outbox+CDC, a saga, a redrawn-to-local boundary, or an explicit accepted-risk in the register with a reconciliation job. (Structural check; the cardinal gate.)
- **No 2PC without a justifying ADR** whose rejected-alternatives show saga/outbox/redrawn-boundary don't hold the invariant.
- **Every cross-service write is eventual-with-compensation or local** — each saga step that can fail has a **complete, semantically-correct compensating action** (or is the idempotently-retriable pivot); no orphan step with no recovery path.
- **Every retried/replayed write is idempotent** (named key or naturally idempotent op); ordering/causality stated where it matters.
- **Every contended write names its isolation primitive** that closes the specific anomaly (the counter mint shows lost-update is prevented).
- Each consistency decision is an **ADR** (context · ≥2 options · decision · positive+negative consequences · the CAP/PACELC point · driver refs) and traces to a `QAS-DATA-*`; diagrams compile (`../shared/diagramming-standards.md`).
- **Gate:** hard oracles green — **consistency-stated on every write path, zero silent dual-writes, no unjustified 2PC, idempotency on every retriable write, isolation stated on every contended write** — AND (design) rubric mean ≥ 0.8 (invariant-fit + compensation-completeness + simplest-mechanism).

## Output
Write `artifacts/consistency-transactions/report.json` per `../shared/report-format.md` — findings/risks per `../shared/finding-schema.md` (`type:"data"`; `oracle:"hard"` for a dual-write, an unstated consistency model, an unjustified 2PC, a non-idempotent retriable write, or an unnamed isolation primitive on a contended path; `driver_ref` = the `QAS-DATA-*` invariant it serves; each finding carries its `tradeoff` and `alternatives_rejected`, ADR-shaped). The `verification` block lists the fitness-function results (the no-dual-write structural check, the consistency-table-complete check, the idempotency check) + hard-oracle results + the rubric score. Design-mode authored artifacts land in `architecture/<slug>/` per `project-architecture-config.md`: the per-path **consistency table** in `design-doc.md` (or `tradeoffs.md`), the saga/outbox **sequence diagrams-as-code** in `diagrams/`, and one **MADR ADR** per significant decision in `adr/NNNN-*.md` — never scattered. Open questions (an unstated invariant, an undecided RPO on a settlement write) are listed for the controller, never silently assumed.

## Guardrails
Per `../shared/guardrails.md`: **propose & document — never write product code and never apply a migration or change a live datastore.** This flow authors the consistency table, the ADRs, and the saga/outbox diagrams; the controller implements the outbox/idempotency/saga code and the operator runs any migration. Ground every consistency choice in a **driver** (the business invariant from the utility tree) — no model without an invariant it protects, no invariant without a declared model. **Reuse the named pattern** (outbox+CDC, saga, idempotent consumer) before inventing a bespoke reconciliation; prefer the **simplest mechanism that holds the invariant** (redraw-to-local > single-doc > multi-doc txn > outbox+CDC > 2PC) and right-size to the actual stack — do **not** introduce a broker/CDC pipeline where there is no second system to write to yet (surface it as a seam for the future documents/payment/closure contexts named in `project-architecture-config.md` instead). Record the **tradeoff and the rejected alternatives** on every decision (CAP/PACELC point, sync↔async, orchestration↔choreography), never just the winner. Read-only audit edits nothing; one writer preserves conceptual integrity.
