# caching-strategy

**Group:** 20-data  ·  **Runs as:** subagent: ../.claude/agents/data-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Decide *where* and *how* to cache deliberately — choosing the pattern (cache-aside, read-through, write-through, write-behind, refresh-ahead), the invalidation strategy (TTL, write-invalidate, versioned keys), the staleness window the system can accept as a correctness tradeoff, and the stampede-protection mechanism for each cache site. Answers: *which data paths need a cache, at which tier (client/CDN/application/database), with what staleness tolerance tied to a named scenario, and what happens on a cache miss or failure?*

A cache is never added reflexively for "speed." Every cache site must be grounded in a latency or load scenario from the utility tree (`workflows/00-drivers/quality-attribute-scenarios.md`) that the unaugmented data path cannot meet, and the staleness it introduces must be named and accepted as a correctness tradeoff. Invalidation is the hard problem — this flow names it, never waves it away.

Caching decisions are only justified by concrete latency or load scenarios with measurable response measures (e.g. "dashboard render p95 ≤ 2s at N concurrent users" — an NFR from the source-of-truth docs named in `project-architecture-config.md`). A flow with no utility tree downgrades to audit-only and surfaces the missing scenarios as open driver questions.
- The **access-pattern inventory** produced by `workflows/20-data/data-modeling.md`: which queries are high-frequency (list-records per tenant, dashboard counts, user lookups), which data is read-heavy vs write-heavy, and what the read/write ratio is per path. Without this inventory, cache placement decisions are guesses.
- The **storage selection** and consistency model from `workflows/20-data/storage-selection.md`: the datastore's read characteristics (collection/table scan vs covered-index latency at current and projected data volume) provide the miss-path cost, which determines whether a cache earns its operational complexity.
- The **performance/capacity model** from `workflows/40-cross-cutting/performance-capacity.md` (if authored): the current p95 latency per critical path against the SLO. If this model does not yet exist, this flow produces the back-of-envelope sizing needed to justify each proposed cache site.
- **Preconditions:** `project-architecture-config.md` read; the utility tree (or stub with open questions) available; the access-pattern inventory available; the existing API routes and data-access schemas (per `project-architecture-config.md`) read to identify hot read paths. If the access-pattern inventory is absent, downgrade scope to audit (flag the missing precondition) and do not recommend specific cache sites.

## Oracle (source of truth)

The **utility tree's latency and load scenarios** (with their response measures) and the **miss-path cost** (the observed or modeled latency of the uncached data path). Every proposed cache is justified by a named scenario it enables; the staleness it tolerates is named as an accepted correctness tradeoff.

- **hard:**
  - Every cache site **declares the staleness it tolerates**, tied to a named quality-attribute scenario. "Cache for 5 minutes" with no driver is not an oracle pass — "dashboard summary counts tolerate 60s staleness per QAS-PERF-02 (last-updated notice displayed)" is. A cache with unstated staleness tolerance blocks.
  - Every cache site **declares its invalidation strategy** — one of: TTL (time-based expiry), write-invalidate (invalidate on write to the backing store), or versioned keys (a new key per version of the underlying entity; old keys expire by TTL). "Invalidate when data changes" with no mechanism named blocks.
  - Every cache site **declares its stampede-protection mechanism**: probabilistic early expiry (PER / XFetch), mutex/distributed-lock (one filler, others wait), request-coalescing (deduplicated in-flight requests), or jittered TTL (randomized expiry windows to spread recomputation). A cached item with high concurrency, a non-trivial miss-path cost, and no stampede protection blocks.
  - **The cache is never the source of truth.** The backing store (the project's primary datastore, per `project-architecture-config.md`) is the authoritative record. The cache is a read-acceleration layer. Any write path that updates the cache without updating the backing store (or vice versa) blocks. The dual-write / write-behind pattern must name its consistency exposure and carry an ADR.
  - **A cache miss or cache failure path is defined, not assumed.** On a miss: degrade gracefully to the backing store (slower but correct); do not crash or return stale data past the declared tolerance window. On a cache failure: fail open to the backing store (availability over cache benefit) or fail closed with a circuit-breaker (if the backing store cannot absorb the fallout) — name which and why.
  - Every cache that holds **PII or other sensitive/regulated data** is flagged. The eviction / TTL strategy must be consistent with the retention/erasure obligations in `project-architecture-config.md`; cached PII must expire at most at the retention window, and erasure of the backing-store record must invalidate the cached copy.
  - Every ADR for a caching decision is **complete**: context · ≥2 options · decision · consequences (positive AND negative) · the staleness tolerance explicitly stated as a consequence. An ADR with only performance upside and no staleness/invalidation downside blocks.

- **soft:**
  - Caching is added **only where a latency or load scenario demands it** — not reflexively, not for general "speed," not because a cache exists elsewhere. The canonical tradeoff ("Performance ↔ Simplicity" from `shared/fallacies-and-tradeoffs.md`) is named: every cache site adds invalidation complexity and a new operational component; the benefit must exceed that cost for the specific scenario. A cache site with no traced scenario is accidental complexity.
  - **Invalidation is treated as the hard problem**, not elided. The chosen strategy honestly names what it cannot handle: TTL tolerates up to TTL-window staleness; write-invalidate requires that all writers go through a single invalidation path (fan-out to distributed replicas is not free); versioned keys leave orphan entries until TTL expires. The design does not claim "always fresh" with a TTL strategy.
  - The cache **hierarchy is deliberate**: client-side (browser cache, SPA in-memory store), CDN/edge (static assets, public API responses), application-level (in-process LRU, a broker/cache like Redis or Memcached), database-level (the datastore's own query cache / buffer pool, per `project-architecture-config.md`). Each tier is chosen by the scenario it serves — CDN for public assets; app-level for expensive cross-collection/cross-table aggregations; no external cache introduced without a driver-justified ADR given the stack-lock constraint.

## Standards & techniques

- **Cache patterns** (name the pattern explicitly in every ADR):
  - **Cache-aside (lazy population):** the application checks the cache first; on a miss, reads from the backing store, populates the cache, returns the result. Read-heavy, tolerant of initial-miss latency; cache is always eventually consistent. The application owns the cache interaction — fine for app-level; requires discipline to not bypass.
  - **Read-through:** the cache layer intercepts reads and populates itself on a miss; the application never reads the backing store directly. Simplifies application code; coupling between cache and store; supported by some Redis client abstractions (e.g. cache-aside emulated via client-side read-through middleware).
  - **Write-through:** every write goes to the cache AND the backing store synchronously. Cache is always warm; write latency is higher; no staleness on the cached copy. Appropriate where read-after-write consistency within the TTL window is required.
  - **Write-behind (write-back):** write hits the cache, the backing store is updated asynchronously. Lower write latency; data durability window — a cache failure before the flush loses writes. Inappropriate for financial or otherwise durability-critical data where durability is a hard driver. Name the window and the flush-failure recovery path if considered.
  - **Refresh-ahead (pre-fetch):** the cache refreshes an entry before it expires, proactively, based on predicted access. Eliminates the cold-miss latency spike on expiry; the prediction must be accurate enough to justify the read amplification (pre-fetching data that is never accessed wastes resources).

- **Invalidation strategies** (the hard problem — choose honestly):
  - **TTL (time-to-live):** simplest; tolerates up to TTL-window staleness; no coordination with writers required. Correct choice when the business can accept the staleness window and writes are infrequent or uncoordinated.
  - **Write-invalidate (active invalidation):** the writer (or a post-write hook) deletes or updates the cache entry. Zero staleness window if all writers participate; breaks if a writer bypasses the invalidation path. Requires that every write path is known and instrumented; note that the 8 Fallacies Fallacy 1 (network is reliable) and Fallacy 5 (topology doesn't change) mean the invalidation message can fail — design a fallback TTL floor.
  - **Versioned keys (cache-busting):** the key encodes a version or ETag (`entity:{id}:v{version}`); a write increments the version, making the old key unreachable. Old entries expire by TTL; the store is append-only in terms of keys. Eliminates stale reads at the cost of higher storage churn and needing to propagate the new version to readers.

- **Cache stampede / thundering-herd mitigation** (required when concurrency is non-trivial and miss cost is high):
  - **Probabilistic Early Recomputation (PER / XFetch):** each reader, as TTL approaches zero, has a probability of recomputing early proportional to the recomputation cost. Prevents synchronized expiry without coordination; no lock required; the math is in the Vattani et al. 2015 paper (used by Redis documentation).
  - **Mutex / distributed lock:** on a miss, one thread/request acquires a lock, recomputes, populates the cache, releases; others wait or serve a slightly-stale copy. Eliminates thundering herd but adds latency tail for waiters and a lock dependency. Use when miss cost is very high and a short wait is acceptable.
  - **Request coalescing (deduplication):** in-process or in-gateway deduplication of identical in-flight cache misses — only one backend request is issued; all waiting requests share the result. Effective at application level (e.g. using a `promise` / `asyncio.Event` or a singleflight-pattern abstraction); does not require a distributed lock.
  - **Jittered TTL:** instead of a single TTL for all instances of a key (which causes synchronized expiry at scale), apply a random jitter (`TTL ± jitter%`) so expiry is spread across a time window. Simple; no coordination; reduces but does not eliminate concurrent misses at the expiry boundary.

- **Cache hierarchy** (model which tier owns which concern):
  - **Client-side:** browser HTTP cache headers (`Cache-Control`, `ETag`, `Last-Modified`) for API responses; SPA in-memory store (per the project's frontend stack) for session-scoped data. Zero network cost; stale on tab reuse.
  - **CDN / edge:** static assets (JS bundles, images) with long `max-age` + content-hash bust; public API responses where applicable. Inappropriate for RBAC-scoped or tenant-scoped data.
  - **Application-level (in-process or an external cache like Redis/Memcached):** expensive aggregations (dashboard counts, tenant-scoped summaries), reference data (org metadata, user role lookups). In-process (e.g. an LRU decorator, an async TTL dict) is the default for a single-process deployment; an external cache is a one-way-door decision requiring a driver-justified ADR under the stack-lock constraint.
  - **Database-level:** the datastore's own buffer pool / working-set cache and query plan cache (per `project-architecture-config.md`). Not application-controlled; sized per the datastore's tuning knobs; effective for repeated index-covered queries. Referenced but not designed here — `workflows/20-data/storage-selection.md` owns the DB-tier tuning.

- **ATAM / utility tree** — every proposed cache site is scored against the prioritized quality-attribute scenarios. Sensitivity points (e.g. "dashboard p95 is *sensitive* to the cache hit-rate: below 85% the 2s SLO is breached") and tradeoff points (e.g. "write-through is a tradeoff point: it buys read-after-write freshness but costs write latency — the opposite of write-behind") are named explicitly per `shared/evaluation-method.md`. The evaluator locates these before the writer authors any ADR.

- **The 8 Fallacies as applied to caching** (`shared/fallacies-and-tradeoffs.md`):
  - *Fallacy 1 (network is reliable)* — write-invalidate messages can be dropped; design a TTL floor even when active invalidation is in use.
  - *Fallacy 2 (latency is zero)* — in-process caching eliminates a network hop; an external cache (e.g. Redis) re-introduces a hop; measure the net gain per scenario.
  - *Fallacy 4 (network is secure)* — a shared external cache is a trust boundary; PII cached there must be encrypted at rest and in transit; access-controlled per the project's RBAC model (per `project-architecture-config.md`).
  - *Fallacy 5 (topology doesn't change)* — cache invalidation fan-out must handle nodes coming and going (e.g. application workers restarting); in-process cache is lost on restart; account for cold-start stampedes.

## Assertions & exit gate

- Every proposed cache site is traced to a named quality-attribute scenario from the utility tree (the driver that demands it); a cache site with no traced scenario is removed from the design.
- Every cache site declares: (a) the staleness it tolerates, tied to a scenario; (b) its invalidation strategy (TTL, write-invalidate, or versioned keys), naming what the strategy cannot guarantee; (c) its stampede-protection mechanism (PER, mutex, request-coalescing, or jittered TTL) where concurrency at expiry is non-trivial.
- The cache is not the source of truth: every cache-write path is paired with a backing-store write; every cache-invalidation failure falls back to the TTL floor (write-invalidate always has a TTL floor).
- The miss path and the failure path are both defined: miss → degrade to backing store (latency hit, not a crash); cache failure → circuit-break or degrade, never cascade.
- PII and other sensitive/regulated data in any cache tier has TTL ≤ the data's retention window; backing-store erasure triggers cache invalidation.
- Every caching ADR is complete: context · ≥2 options · decision · consequences (positive: which scenario is now met, at what response measure; negative: the staleness tolerance accepted, the invalidation complexity added, the operational surface area of any new component). An ADR with only upside blocks.
- The cache-hierarchy diagram compiles without error (Mermaid / PlantUML per `project-architecture-config.md`); every cache tier in the diagram corresponds to a decision in an ADR.
- The performance/capacity model (back-of-envelope) confirms the cache hit-rate assumption the scenario depends on (e.g. "85% hit-rate assumption → p95 = 0.85 × cache_latency + 0.15 × DB_latency = 0.85×5ms + 0.15×120ms = 22ms p95 → meets 2s dashboard SLO with headroom").
- **Gate:** hard oracles green (staleness declared + traced; invalidation strategy named; stampede protection present where required; miss/failure paths defined; cache not source of truth; PII TTL ≤ retention; ADRs complete; diagram compiles) AND (design) rubric mean ≥ 0.8 (caching is demand-driven not reflexive, invalidation is honest about its limits, hierarchy is right-sized for the deployment model, no external cache introduced without a driver-justified ADR under the stack-lock constraint).

## Output

Write `artifacts/caching-strategy/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (`type:"data"` or `type:"performance"`; `oracle:"hard"` for: undeclared staleness tolerance, unnamed invalidation strategy, missing stampede protection where concurrency is non-trivial, undefined miss/failure path, cache used as source of truth, PII TTL exceeding retention window, incomplete ADR; `oracle:"soft"` for: cache added without a traced scenario, invalidation strategy that understates its guarantees, over-engineered cache tier for the current deployment model); the verification block (fitness-function results, hard-oracle results, capacity-model hit-rate check, rubric score) for design mode. Open driver questions (staleness tolerance per data type if unstated, p95 latency budget per route if not stated, acceptable staleness for org/user reference data) are listed explicitly for the controller to resolve — never silently assumed.

## Guardrails

Per `shared/guardrails.md`:

- **Propose & document — never write product code.** The data architect authors ADRs, the cache-strategy design-doc section, diagrams-as-code, and fitness-function specifications. It does not write application decorator code, external-cache client configuration, or any file under the project's source directories (per `project-architecture-config.md`). The controller implements the chosen design.
- **Ground every cache site in a driver.** No cache is added reflexively. Every site traces to a named quality-attribute scenario (a latency or load response measure from the utility tree). A cache with no driver is accidental complexity — flag it, don't recommend it. The canonical tradeoff ("Performance ↔ Simplicity") is named: the cache adds invalidation complexity and operational surface area; the benefit must exceed that cost for the specific scenario (`shared/fallacies-and-tradeoffs.md`).
- **Invalidation is the hard problem — name it honestly.** No strategy claims "always fresh" unless it is write-through with synchronous invalidation *on every write path* — and even then, the 8 Fallacies (network is reliable, topology doesn't change) mean a TTL floor is still warranted. TTL-based caching is honest about its staleness window; write-invalidate is honest about the coordination requirement. The design does not wave away the hard problem.
- **Stack-lock constraint.** An external cache (e.g. Redis or Memcached) is a new technology requiring a driver-justified ADR. Under the current stack (per `project-architecture-config.md`), the default is in-process caching (an in-language memoization/LRU utility, an async TTL dict, or the framework's in-process cache backend). In-process cache is simpler, has lower operational cost, and is sufficient for a single-process deployment. An external cache is appropriate only when a driver demands cache sharing across multiple workers/replicas and the operational cost is accepted.
- **Cross-reference siblings.** Storage characteristics that determine miss-path cost → `workflows/20-data/storage-selection.md`. Performance/capacity model that proves a cache site earns its complexity → `workflows/40-cross-cutting/performance-capacity.md`. Cloud-design patterns (cache-aside is in the Azure CAT catalog; retry/circuit-breaker on cache failure are in the resilience-patterns catalog) → `workflows/80-patterns/cloud-design-patterns.md`. PII retention/erasure obligations that constrain TTL → `workflows/40-cross-cutting/privacy-compliance-architecture.md`. Do not duplicate their oracles; reference the paths.
- **Record the tradeoff and the rejected alternatives** on every significant caching ADR. The staleness tolerance is a *consequence*, not a footnote — state it as a negative consequence alongside the performance gain. The write-behind pattern's durability exposure for financial data is a hard risk; record it, don't elide it.
- **One writer preserves conceptual integrity.** The `data-architect` is the sole writer for caching-strategy artifacts. Proposer and evaluator agents are read-only. A second writer (e.g. `adr-author`) runs only sequenced, never concurrent, on a non-overlapping artifact.
- **Non-destructive and safe.** Read-only on any live environment. Never flush a cache, alter external-cache configuration, or change the primary datastore's buffer-pool size on a live system. The data architect produces the *design*; the controller/operator applies it.
