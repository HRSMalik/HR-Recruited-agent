# data-mesh

**Group:** 20-data  ·  **Runs as:** subagent: ../../agents/data-architect.md  ·  **Mode:** audit + design   ·   **Default model:** sonnet

## Purpose

Decide whether **Data Mesh fits at all** — it is not a universal architecture, it is a socio-technical answer to a specific pain (a centralized data team/lake that has become the bottleneck for many domains publishing to many consumers) — and, only where a cost-benefit trigger justifies it, design its **4 principles**: domain ownership, data-as-a-product, the self-serve data platform, and federated computational governance, with **data contracts (ODCS)** as the product interface. Answers: *does this system have enough domains/consumers/org scale to justify decentralizing data ownership, and if so, what does each domain's data product, the self-serve platform, and the governance model concretely look like?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack + deployment target, the domain/team count, the compliance regime, and the constraint that a new architecture style needs a driver-justified ADR (boring-tech-first, `shared/guardrails.md` §3). For a small single-team system on a single datastore, this flow's likely output is "not recommended" — that is a valid, complete answer, not a downgrade.
- The **domain map** from `workflows/00-drivers/domain-analysis.md` — bounded contexts, their owning teams, and the context map (upstream/downstream, shared-kernel, ACL relationships). Data Mesh's domain-ownership principle is meaningless without a real bounded-context/team mapping; if domain-analysis has not run, this flow reads the codebase's module boundaries directly and flags the missing domain map as a precondition gap.
- The architecture **drivers**: the utility tree (`architecture/<slug>/quality-attributes.md`) — specifically any scenario naming data-platform bottleneck, cross-domain analytics latency, data-team lead-time, or a stated multi-domain/multi-team scale. A mandate to "evaluate Data Mesh" with no scale/pain driver stated is itself a finding — surface the missing driver rather than assume the answer is yes.
- **Existing data landscape:** the current data platform (a central warehouse/lakehouse, ETL pipelines, or none) and its known pain points (lead time to add a new dataset, cross-team dependency backlog, ownership ambiguity) — from `workflows/20-data/data-modeling.md`, `storage-selection.md`, and `data-governance-lifecycle.md` outputs where they exist.
- **Preconditions:** `project-architecture-config.md` read; the domain map (or its absence, flagged) available; the org/team topology (`workflows/10-styles-decomposition/team-topology-decomposition.md`) read where authored — Data Mesh is a Conway-shaped decision, not a purely technical one.

## Oracle (source of truth)

The fit decision and the design (when adopted) are judged against Zhamak Dehghani's four Data Mesh principles and their measurable structural consequences — never against "data mesh is modern" or "everyone's doing microservices for data now." A mesh with no data contracts, no named domain owners, or a central bottleneck team gatekeeping every dataset is not a mesh — it is a distributed data swamp with mesh vocabulary.

- **hard:**
  - **Adoption gate (cost-benefit trigger) is explicit and checked first.** Data Mesh is recommended **only if** at least two of: (a) domain count ≥ ~5–6 independently-owned bounded contexts each producing analytical data, (b) consumer count/diversity is high enough that a single central team is a demonstrated bottleneck (a stated or evidenced lead-time/backlog pain), (c) org scale supports dedicated platform-engineering investment (a self-serve platform team can be funded). Absent the trigger, recommending Data Mesh is itself a `critical` hard finding (résumé-driven / accidental-complexity per `shared/quality-attribute-rubric.md` → simplicity) — the design falls back to a simpler warehouse/lakehouse + data contracts.
  - **Domain ownership is real, not nominal.** Every data product traces to exactly one **owning domain** (a bounded context from `domain-analysis.md` with a named accountable team) — no data product owned by "the data team" or by no one. An orphaned or centrally-owned "domain" data product blocks.
  - **Every data product is a full ODCS data contract**, not a bare schema dump. Per `shared/data-contract-template.md`: the contract carries Section 1 (metadata/ownership), Section 2 (schema), Section 3 (quality rules), Section 4 (SLA — availability/latency/freshness/retention), and Section 5 (security/classification). A published "data product" missing the SLA or security/classification section is not a data product — it is an unmanaged dataset and blocks.
  - **Data-product self-description is verifiable, not asserted.** Each data product's contract + registry entry demonstrably satisfies the 6 data-product properties (below): discoverable (has a registry/catalog entry), addressable (a stable, versioned URI/table/topic name), trustworthy (quality rules + SLA in the contract), self-describing (schema + semantics in the contract, no tribal knowledge required), interoperable (schema uses shared, documented types/standards across domains — not domain-private encodings for cross-domain fields), secure (classification + access control per Section 5). A data product failing any one property is a `major` finding naming which property is unmet.
  - **The self-serve data platform is a named, scoped capability**, not "the domain team figures it out." At minimum it provides: (a) infrastructure provisioning for a domain to stand up its own pipeline/store, (b) a contract registry/catalog for publish + discovery, (c) a shared observability/quality-monitoring layer, (d) policy-as-code enforcement hooks (below). A "platform" that is just a shared wiki page is a `major` finding (cross-ref `shared/idp-capability-checklist.md` §11 Data Platform — apply its capability-maturity scale to the mesh's platform).
  - **Governance is federated AND computational — never a central bottleneck.** Global policies (interoperability standards, PII classification scheme, access-control baseline, a shared taxonomy for cross-domain keys) are decided by a federation of domain representatives (not one central architecture-review team) and are **enforced automatically** via policy-as-code (`shared/policy-as-code.md`) at publish/consume time — not via a manual central approval gate. A governance model that routes every new data product through one central team's sign-off before it can be consumed **fails this oracle** — it has recreated the centralized bottleneck the mesh exists to remove.
  - **Data contracts are ODCS-conformant and machine-readable.** Every contract validates against the ODCS schema shape in `shared/data-contract-template.md` (Sections 1–8 as applicable); a contract expressed only as free-form prose or a slide is not a hard artifact and blocks.
  - **PII/compliance obligations still hold per domain.** Federating ownership does not federate away the compliance oracle — every data product carrying PII/PHI/PCI still satisfies every hard oracle in `workflows/20-data/data-governance-lifecycle.md` (classification, retention, erasure, lineage), now enforced *per domain* via the platform's policy-as-code layer rather than centrally.
  - **Every mesh-adoption decision is an ADR** with the cost-benefit trigger evidence, ≥2 considered options (mesh vs simpler alternative, and — if mesh — which domain-boundary cut), decision, and consequences (positive AND negative, including the added operational overhead). An ADR that only lists mesh's upside is incomplete.

- **soft:**
  - **Right-sizing is the primary craft judgment.** Data Mesh is recommended *only* where the org scale/domain count genuinely justifies the operational overhead of decentralization (per-domain pipeline ownership, a federation governance body, a platform team). For a small single-team system (a modular monolith over a single datastore is a common default shape — confirm the project's actual shape per `project-architecture-config.md`), a **simpler warehouse/lakehouse + data contracts** (contracts without the full domain-ownership/federation apparatus) is the correct, higher-scoring answer — Data Mesh here would be over-engineering even if technically well-executed.
  - **Data product craft** — is the product genuinely useful to consumers without a support ticket (self-serve discovery actually works), or does "self-serve" mean "read the source code"? Score against the `idp-capability-checklist.md` DX metrics (time-to-first-consume analog of TTFD) where a platform exists.
  - **Federation governance is lightweight and fast**, not a second bureaucracy — the federated body should set standards (interoperability, security baseline) and automate enforcement, not become a new approval queue. A federation that meets weekly to rubber-stamp every contract change has re-centralized under a different name.
  - **Interoperability standards are genuinely shared** (a common way to represent a customer ID, a common event envelope, a common timestamp/timezone convention) rather than each domain quietly diverging — cross-domain joins/consumption should not require a bespoke ACL per pair of domains.
  - **Evolvability** — can a new domain join the mesh (publish its first data product) without a platform-team ticket, using the golden path? If not, the self-serve platform principle is unmet in practice even if documented in principle.

## Standards & techniques

- **Data Mesh (Zhamak Dehghani)** — the foundational architecture, introduced to solve the failure mode of the monolithic centralized data lake/warehouse team becoming an organizational bottleneck as domain and consumer count grows. The **4 principles**:
  1. **Domain ownership** — analytical data is owned and served by the domain that produces it (the same team that owns the operational bounded context), not extracted and centralized by a separate data team.
  2. **Data as a product** — each domain treats its published dataset as a product with real consumers, applying product thinking: discoverability, documented semantics, a quality bar, and a named owner accountable for it.
  3. **Self-serve data platform** — a shared, domain-agnostic infrastructure layer (provisioning, cataloging, observability, access control) so a domain team can publish/consume without needing deep data-engineering specialization or a platform-team ticket for every step.
  4. **Federated computational governance** — global concerns (interoperability, compliance, security standards) are decided by a federation of domain representatives and *automated* via computational policy enforcement (policy-as-code) rather than a central manual gate.
- **The 6 data-product properties** (discoverable · addressable · trustworthy · self-describing · interoperable · secure) — the acceptance criteria for "is this actually a data product" vs. just a dataset with a Data Mesh label on it.
- **Data contracts (ODCS)** — the standard, machine-readable interface for a data product; see `shared/data-contract-template.md` for the full section-by-section template (metadata, schema, quality rules, SLA, security/classification, pricing/ownership, roles/consumer registry, lifecycle/versioning). ODCS is what makes a data product's contract *executable* rather than a wiki description.
- **Federated computational governance via policy-as-code** — see `shared/policy-as-code.md`: the federation's decisions (a shared PII-classification taxonomy, a shared interoperability schema, a minimum SLA floor) are expressed as Rego/OPA rules run by `conftest` at publish time, so compliance is checked automatically per domain rather than by a central reviewer reading every contract.
- **ATAM / utility tree** — the fit decision (mesh vs. simpler alternative) and, where adopted, the domain-boundary cut are scored against the utility tree using ATAM sensitivity/tradeoff points. The canonical tradeoff: **domain autonomy/evolvability ↔ operational and governance overhead** — mesh buys each domain the ability to model, own, and evolve its data independently, at the cost of duplicated platform investment, a federation governance process, and cross-domain interoperability discipline that a single central team didn't need to negotiate.
- **Conway's Law** — Data Mesh is as much an org-design decision as a technical one; a mesh drawn across domain boundaries the org doesn't actually operate independently (no real per-domain data-engineering capacity) is a mesh on paper, not in practice. Cross-ref `workflows/10-styles-decomposition/team-topology-decomposition.md`.
- **MADR** — the adoption decision and each domain-boundary cut are ADRs (`shared/adr-template.md`), tracing `Driver refs` to the cost-benefit trigger evidence in `architecture/<slug>/quality-attributes.md`.

## Step sequence

- **audit:** Read `project-architecture-config.md` + the domain map (`domain-analysis.md` output or the inferred module boundaries) + the existing data platform description → check the **adoption gate first**: does the current or proposed architecture already claim "Data Mesh" without the domain-count/consumer-scale/platform-investment trigger being evidenced? → for any dataset labeled a "data product," check it against the 6 properties and the ODCS-contract hard oracle (does a contract exist, is it complete per `data-contract-template.md` Sections 1–5?) → check the governance model: is there a federation, or has a "mesh" recreated a central approval bottleneck? → check per-domain compliance: does each PII-bearing data product still satisfy `data-governance-lifecycle.md`'s hard oracles? → emit findings + risks per `finding-schema.md`, grouped by `type:"data"` (missing/incomplete contracts, unmet data-product properties) and `type:"decomposition"` (ownership ambiguity, re-centralized governance) then severity (critical: a mesh recommended/claimed with no adoption-gate evidence; a PII data product with no classification/erasure path — major: a "data product" missing SLA or security section; a central-approval governance bottleneck masquerading as federation — minor: interoperability drift between domains, a platform capability at maturity Level ≤1 per `idp-capability-checklist.md`). Read-only, no artifact authored, no source edited.

- **design:** Frame (read the utility tree + domain map + org/team topology + existing data-platform pain points; confirm or surface as an open question: domain count, consumer diversity/count, stated data-team bottleneck pain, platform-investment appetite; Direction B: mesh cut along a coarser subset of "core" domains first (a phased rollout) with a fuller self-serve platform (provisioning + catalog + observability) reserved for phase 2; each direction specifies its domain-boundary cut, its data-product contract template fill-in, its platform capability scope per `idp-capability-checklist.md` §11, and its federation governance model; every data product traces to exactly one owning domain with a named accountable role [no orphan/centrally-owned product]; the 6 data-product properties are each demonstrably met per product [not asserted]; ## Assertions & exit gate

- The adoption gate (domain-count / consumer-scale / platform-investment trigger) is evaluated **before** any mesh design work, with its evidence recorded in the ADR; a mesh recommended without it is blocked.
- Every data product traces to exactly one named owning domain (from `domain-analysis.md`) and one accountable owner role — no product owned by a central data team or by no one.
- Every data product has a complete ODCS data contract (`shared/data-contract-template.md` Sections 1–5 at minimum) — schema, quality rules, SLA, and security/classification are all present; a contract missing any of these blocks.
- Every data product demonstrably satisfies all 6 properties (discoverable, addressable, trustworthy, self-describing, interoperable, secure); a property asserted but not evidenced (e.g. "self-describing" with no schema/semantics in the contract) blocks.
- The self-serve platform's capability scope is named against `shared/idp-capability-checklist.md` §11, with a stated target maturity level per capability — no "the platform will handle it" hand-wave.
- Governance is federated (a named federation body, not one central team) AND computational (policy-as-code rules per `shared/policy-as-code.md` enforce the federation's standards automatically); a manual central-approval gate on routine publish blocks.
- Every PII/PHI/PCI-bearing data product also passes every hard oracle in `workflows/20-data/data-governance-lifecycle.md` (classification, retention, erasure, lineage) — federation does not exempt compliance.
- The mesh-adoption ADR (and each domain-boundary-cut ADR) has context, ≥2 options (mesh vs. the simpler alternative, at minimum), decision, and consequences positive AND negative, with `Driver refs` tracing to the cost-benefit trigger evidence in `architecture/<slug>/quality-attributes.md`.
- The mesh topology diagram compiles without error.
- **Gate:** hard oracles green (adoption-gate evidence present; every data product domain-owned with a complete ODCS contract meeting all 6 properties; governance federated + computational; PII compliance holds per domain; ADRs complete; diagram compiles) AND (design) rubric mean ≥ 0.8 — including the right-sizing judgment that mesh (or its rejection in favor of the simpler alternative) is the correctly-scoped answer for this system's actual domain/consumer scale.

## Output

Write `artifacts/data-mesh/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (`type:"data"` for contract/property gaps and PII-compliance-per-domain gaps; `type:"decomposition"` for ownership ambiguity or re-centralized governance; `oracle:"hard"` for the blocking cases enumerated above — including a mesh recommendation with no adoption-gate evidence; `oracle:"soft"` for right-sizing, federation lightweightness, and interoperability-discipline craft); the verification block (fitness-function results — contract-schema validation, 6-property coverage, domain-ownership completeness, PII-compliance-per-domain check, diagram-compile result, ADR completeness, rubric score) for design mode. Where the fit gate fails, the report's primary output is the **rejection ADR** recommending the simpler warehouse/lakehouse + data-contracts alternative — this is a complete, successful run, not a partial one. Open driver questions (unconfirmed domain count, consumer scale, platform-investment appetite) are listed explicitly for the controller to resolve — never silently assumed.

Authored design artifacts (when the gate passes) land in `architecture/<slug>/` per `project-architecture-config.md`:
- `architecture/<slug>/adr/NNNN-data-mesh-adoption.md` — the fit decision, cost-benefit trigger evidence, considered options (mesh vs. simpler alternative), consequences.
- `architecture/<slug>/adr/NNNN-mesh-domain-boundary-cut.md` — which domains publish first, the phasing rationale if phased.
- `architecture/<slug>/data-contracts/<domain>-<product>.yaml` — one ODCS contract per data product, per `shared/data-contract-template.md`.
- `architecture/<slug>/design-doc.md` — a Data Mesh section: the platform capability scope (`idp-capability-checklist.md` §11 maturity targets), the federated-governance model and its policy-as-code enforcement (`shared/policy-as-code.md`), the domain-ownership map.
- `architecture/<slug>/diagrams/data-mesh-topology.md` — domains → data products → consumers, the self-serve platform layer, and the federation governance boundary, as diagrams-as-code (compiles without error).

## Guardrails

Per `shared/guardrails.md`: propose & document — this flow authors ADRs, ODCS contract stubs, the platform-capability scope, the federation-governance design, and Rego policy *specs*; it never provisions a real data platform, never runs a live `conftest`/OPA enforcement, and never writes product code (`backend/**`, `frontend/**`, or any live pipeline). Ground every recommendation in the cost-benefit adoption trigger — a mesh proposed with no domain-count/consumer-scale/platform-investment evidence is accidental complexity, not a decision, and is rejected in favor of the simpler warehouse/lakehouse + data-contracts alternative. Reuse the existing data-contract template (`shared/data-contract-template.md`) rather than inventing a bespoke contract shape; reuse `shared/idp-capability-checklist.md` §11 for platform scope rather than re-deriving platform capabilities from scratch. One writer (`data-architect`) preserves conceptual integrity across the domain-ownership map, the contracts, and the governance design. Record the tradeoff (domain autonomy/evolvability ↔ operational and governance overhead) and the rejected alternatives on every ADR — an adoption or boundary-cut ADR with only upside is incomplete.
