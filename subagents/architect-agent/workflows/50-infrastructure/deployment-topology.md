# deployment-topology

**Group:** 50-infrastructure
**Runs as:** subagent: ../.claude/agents/infrastructure-architect.md
**Mode:** audit + design
**Default model:** sonnet

## Purpose
Design and audit the **C4 deployment view (L4)** — the mapping of every container onto its runtime node, AZ, region, and environment tier — sized to the availability and scale scenarios in the utility tree. Answers the specific decision: *for each production environment, which containers run where, how are they placed across failure domains, what release strategy ensures zero-downtime and reversal, and do the environments (dev / staging / prod) have enough parity to trust a staged promotion?* This flow owns the *deployment topology* limb of the infrastructure concern; `../40-cross-cutting/availability-resilience.md` owns the FMEA and redundancy design this topology must accommodate, `cicd-architecture.md` owns the pipeline that executes the release strategy this flow selects, and `cloud-well-architected.md` owns the Well-Architected pillar sweep this topology is evaluated against.

## Inputs & preconditions
- From `project-architecture-config.md`: the project's stack (e.g. an SPA frontend / API backend / primary data store / reverse-proxy same-origin — per the config), the current deployment posture (single-host vs multi-host, local-compose dev only vs deploy-ready images, real prod deploy = separate ADR not yet built or already live), the team topology (per the config — small team favors operational simplicity per Conway), the constraint that any dev-only compose/Makefile/scripts stay dev-only (don't sync to org, if applicable), and where the architecture package lives (`architecture/<slug>/`).
- The **drivers / utility tree** from `../00-drivers/quality-attribute-scenarios.md`: the prioritized 6-part **availability and scale scenarios** with their response measures — specifically `QAS-AVAIL-*` (RTO/RPO targets), `QAS-PERF-*` (the response-time NFRs from the source-of-truth docs), and any stated SLO target. **If the availability target / RTO / RPO is not stated, surface it as an open driver question and do not assume a number** (per `project-architecture-config.md` Reliability driver). Read the project's actual scale posture from `project-architecture-config.md`: an internal/low-scale tool should not default to multi-region or Kubernetes absent a concrete driver; a consumer-scale system may need it — do NOT assume either without a driver.
- The **existing or proposed C4 Container view (L2)** — every Container that must be placed: the frontend container (e.g. static assets served via a reverse proxy), the API/backend service(s), the primary data store, the reverse proxy / gateway, and any future containers (e.g. a document-processing provider, a payment/settlement integration, a deferred integration seam). Containers not yet in L2 are named as planned placements with a forward-compatibility note.
- The **redundancy decisions** from `../40-cross-cutting/availability-resilience.md` — what the FMEA demanded (replica set, N+1, standby node) so the deployment view reflects the agreed failure-domain strategy, not a naive single-node placement.
- Preconditions: the source-of-truth docs read (BRD / PRD / understanding log, as named in `project-architecture-config.md`); the C4 Container view available or derived; the availability-resilience output available; the cloud/hosting target confirmed from `project-architecture-config.md` before placing nodes (managed cloud vs bare-metal vs PaaS — do not assume AWS/Azure/GCP; read the config).

## Oracle (source of truth)
The **availability quality-attribute scenarios** (each with a response measure) + the **diagram-as-code deployment view** (every container mapped to a node/AZ/region, compiling without error) + the **release strategy verification** (zero-downtime and reversible, demonstrable by structural check). Never "the topology scales" or "it's highly available" with no number and no traced failure domain (`../../shared/quality-attribute-rubric.md` grounding rule).

- **hard:**
  - **Deployment diagram is diagrams-as-code and compiles** — the C4 deployment view is authored in Mermaid, PlantUML+C4-PlantUML, Structurizr DSL, or D2 (per `project-architecture-config.md` diagram tooling) and renders without error. Every Container from the L2 view appears in the deployment view mapped to a named node/AZ/region (or explicitly marked `planned: true`). An unlabeled node, an unplaced Container, or a DSL syntax error **blocks** (`../../shared/diagramming-standards.md` *Diagram compiles* oracle).
  - **Topology demonstrably meets the availability scenario** — for each `QAS-AVAIL-*` scenario, trace the request path on the deployment diagram and confirm: (a) no node on the critical path is a silent SPOF without an accepted-risk record (cardinality-1 node with no peer = SPOF — it must have a redundant peer, a managed failover, or a signed accepted-risk entry naming the scenario it threatens and the RTO it implies); (b) failure domains are real — nodes claimed to be in different AZs must be in genuinely independent AZs, not two instances on the same host; (c) the composite availability (serial multiplication per `../40-cross-cutting/availability-resilience.md`) meets the target at the stated topology. A topology that claims two AZs but places both containers on one physical host fails this check. *Structural:* trace `QAS-AVAIL-*` path → enumerate nodes → confirm each node has a failure-domain distinct from its redundant peer.
  - **Release strategy is zero-downtime and reversible** — the chosen strategy (blue-green / canary / rolling) for the production environment: (a) allows serving traffic during deployment (no maintenance window required); (b) has a defined rollback path that reverts within a stated time bound; (c) has no protocol-level incompatibility with the existing session/state model (stateless API = rolling-safe; sticky sessions or in-flight file uploads require blue-green or canary with affinity). *Structural:* verify the strategy handles the project's frontend + API container pair as described in `project-architecture-config.md` (e.g. a statically-served SPA plus an independently deployable API — they can deploy independently, but a version skew window must be bounded or handled with backward-compatible contracts).
  - **Environment parity (dev/staging/prod)** — every environment (local dev, staging, production) uses the **same build definition** (e.g. the same Dockerfile) and the **same base image tag**; the only differences are externalized config (env vars, secret store references, origins) — no code-path differences by environment baked into the image. Building from the same definition must produce a deployable artifact for all environments. *Traceable:* the deployment diagram labels each environment's topology; diff of dev vs prod images is config-only.
  - **Failure domains are real** — "multi-AZ" in the diagram corresponds to AZs in the cloud provider's sense (independently powered/networked facilities), not replicas on the same host or in the same data-center rack. A claim of AZ-level isolation must name the provider/region/AZ; if the target environment doesn't yet have this, the diagram marks the topology as `single-AZ` with an `open driver question` for the controller. Fallacy #1 and #5 (network reliability, topology doesn't change) are honored: the placement assumes node failure is possible and the topology handles it without manual intervention or accepts the risk.
  - **Fallacies honored at the placement boundary** — Fallacy #5 (`../../shared/fallacies-and-tradeoffs.md`, topology doesn't change): containers are placed via service-discovery-aware mechanisms (not hardcoded host:port); Fallacy #7 (transport cost is non-zero): cross-AZ/region egress cost is acknowledged in the cost model and factored into multi-AZ/multi-region decisions; Fallacy #4 (network is hostile): inter-container communication on the deployment view names its security boundary (private subnet / mTLS / network policy) — containers in the same trusted network vs crossing a trust boundary are visually distinct per `../../shared/diagramming-standards.md`.
  - **ADR complete per topology/release decision** — the choice of release strategy (blue-green vs canary vs rolling), the AZ/region topology, and the environment-parity approach each have a complete ADR (context · ≥2 options · decision · positive **and** negative consequences) traced to a `QAS-*` driver (`../../shared/adr-template.md`).
  - **Cost bounded** — the topology carries an order-of-magnitude monthly cost estimate (compute + storage + egress per environment tier: dev/staging/prod); per the team topology and scale posture in `project-architecture-config.md`, multi-AZ must clear a CBAM ROI ≥ 1 against the actual availability driver before being recommended. No node "runs for free."

- **soft:**
  - **Simplest topology that meets the driver** — multi-AZ before multi-region; multi-region only with an explicit driver that single-region cannot satisfy (e.g. data-residency, RTO ≤ 1 min across continent). When the availability target is unstated and the project is low/internal-scale (per `project-architecture-config.md`): a **single-AZ deployment with a primary-store replica set + active-passive failover** is the right default; active-active or multi-region would be accidental complexity. The evaluator penalizes over-engineering explicitly (the simplicity / right-sizedness rubric, `../../shared/quality-attribute-rubric.md`).
  - **Failure domains are intellectually honest** — a topology labeling two nodes as independent AZs when they share a power circuit, a VPC NAT gateway, or a shared database on one host is not a real failure domain — it's a diagram lie. The soft oracle flags a topology that *claims* redundancy but doesn't isolate the actual failure-propagation path.
  - **Operability of the release strategy** — the chosen release strategy fits the team size: a blue-green deploy for a small team with no automated traffic-shifting mechanism is operationally risky (the double-infra cost + manual cut-over step). The right-sized default for a small-team project is a **rolling deploy** (reverse proxy + API worker pool, SPA static re-serve) unless the version-skew window is a real problem, in which case blue-green with a managed load balancer; canary is appropriate when a progressive traffic split is automated.
  - **Evolvability seam for a deferred integration** — the deployment topology preserves the integration seam for any deferred/future integration layer named in `project-architecture-config.md` (the Evolvability driver): the API service boundary and the frontend's `/api/*` proxy are placed so a future sidecar / gateway / integration service can be inserted without re-topologizing the entire deployment view.

## Standards & techniques
- **C4 deployment view** (Simon Brown, `../shared/diagramming-standards.md`) — the fourth C4 supplementary view: maps every Container (L2) onto a deployment node (physical server, VM, container runtime, PaaS slot, managed service region/AZ). Labels every node with its type (Docker host / Kubernetes node / managed PaaS / CDN edge), every container with its runtime image tag, and every relationship with its protocol and network boundary (private subnet / public internet / mTLS). Every element and relationship must be labeled and typed; sync vs async visually distinct; trust boundaries drawn.
- **Multi-AZ / multi-region topology by RTO/RPO** — AZ-level redundancy (active-passive or active-active within a region) is the first tier of availability investment; it protects against single-facility failure at low egress cost. Multi-region is the second tier (protects against full-region loss / data-residency requirements) at 2–3× cost and significant consistency/latency tradeoff (PACELC). Size the tier to the **actual driver**: if RTO ≤ 5 min and the driver is internal business continuity, a same-region primary-store replica set with automatic failover achieves it; active-active multi-region is not justified. Document the decision as an ADR citing the CBAM ROI.
- **Blue-green / canary / rolling release** — three named patterns, chosen by the version-skew risk, the team's automation capability, and the stateful-session model:
  - **Rolling** (replace instances one by one; both old+new run simultaneously for the roll duration) — lowest infra cost, safe when the API is backward-compatible across the skew window; the default for a stateless API behind a reverse proxy with no sticky sessions.
  - **Blue-green** (full parallel environment; traffic cut atomically at the load balancer) — zero skew window, instant rollback (flip LB pointer back); costs double infra during deploy; correct when API versions are breaking or when the frontend+API must always be at the same version.
  - **Canary** (progressive traffic split: 1% → 10% → 50% → 100%) — validates in production at real traffic before full roll; requires automated traffic-shifting (weighted routing at the LB/gateway), SLO-gated promotion, and automated rollback on error-rate threshold. Worth the overhead when the change is high-risk and the team has the automation; over-engineering for a small team without a CI/CD gate in place.
  - The **rollback path** for each strategy must be stated: rolling → re-deploy the prior tag; blue-green → LB pointer flip; canary → drain new-version weight to 0 and remove. Rollback must be executable within the stated RTO.
- **Zero-downtime release** — a release is zero-downtime iff: (a) health/readiness probes are wired (the load balancer / orchestrator removes a container from rotation before it receives traffic on the new version, and waits for readiness before routing to the new instance); (b) graceful-drain is implemented (in-flight requests complete before the old instance is killed — a `SIGTERM` handler with a bounded drain timeout); (c) the database schema changes are backward-compatible with the old version still running (expand-then-contract, `../80-patterns/distributed-systems-patterns.md` expand-contract pattern). A deployment that restarts the API with no health probe, no drain, and a destructive migration is not zero-downtime.
- **Environment parity (the three-tier principle from The Twelve-Factor App, factor X)** — dev, staging, and prod differ only in config (env vars / secret references / origins / resource sizes), not in code, runtime, or OS. Building the deploy artifact from the same source produces the same image for all tiers. "Works on my machine" failures between staging and prod are a parity gap — name and close them. Per `project-architecture-config.md`: confirm whether the existing deploy artifacts (e.g. Dockerfiles) are already deploy-ready; note whether any dev-only compose/Makefile/scripts stay dev-only (do not sync to org); real prod secrets are env-driven, not baked in. Environment parity is what makes "it passed staging" a trustworthy signal.
- **Failure-domain reasoning** — a failure domain is the blast radius of a single failure event. Independent AZs share no power, network uplink, or cooling — a rack failure, power-unit outage, or top-of-rack switch failure does not cross AZ boundaries. A "multi-AZ" deployment places the redundant instances in genuinely different AZs (different `availability_zone` attributes in cloud APIs, not merely different subnets in the same AZ). Where the hosting target is a single VPS or bare-metal host, name the failure domain honestly: it is a **single physical host** with a SPOF at the hardware layer — the accepted-risk entry must say so.
- **AWS/Azure/GCP Well-Architected — Reliability and Operational Excellence pillars** (`cloud-well-architected.md`) — the deployment topology is the primary artifact the Reliability pillar's "design for failure" and "test recovery procedures" checks apply to; Operational Excellence's "automate operations" check anchors the release-strategy automation requirement. The topology is the design; the Well-Architected review (`cloud-well-architected.md`) is the evaluation pass; they are companion flows.
- **The AKF scale cube** (`../40-cross-cutting/scalability-architecture.md`) — a deployment across AZs is an **X-axis clone** (identical copies behind a load balancer); a sharded/partitioned placement is a **Z-axis split**. The deployment view makes explicit which axis the topology is scaling on and why (a driver must demand it — no accidental complexity).
- **Canonical tradeoffs** (`../../shared/fallacies-and-tradeoffs.md`) applied at the topology level:
  - **Cost ↔ Resilience** — every additional AZ/region/standby node costs money and operational surface; size the topology to the **driver**, not the maximum imaginable.
  - **Consistency ↔ Availability** (PACELC) — async primary-store replication (low RPO cost, non-zero data-loss window on primary failure) vs synchronous (zero data-loss, write-latency cost); state the point chosen per data store.
  - **Simplicity ↔ Resilience** — a blue-green deploy with a managed LB is more resilient than a rolling deploy but adds infra, automation, and runbook complexity; for a small team this is a real cost.
  - **Latency ↔ Availability** — cross-AZ calls add ~1–5 ms of network latency; cross-region adds ~10–100 ms; factor into the latency-budget check for `QAS-PERF-*` scenarios when the topology distributes containers across AZs or regions.

## Step sequence

- **audit:** Read the existing or proposed deployment configuration (build definitions, compose files, any IaC) → derive the **current deployment view** as a C4 deployment diagram (diagrams-as-code, even if the current posture is single-host) → for each `QAS-AVAIL-*` scenario, trace the request path on the diagram and enumerate every cardinality-1 node (SPOF check) → verify each SPOF has a signed accepted-risk entry or a redundant peer in a real failure domain → check the release strategy (if any) for zero-downtime guarantees (health probe, graceful drain, schema backward-compat) → check environment parity (same build definition / image tag across tiers; config-only diffs) → check fallacy compliance (#1, #4, #5, #7 at the deployment boundary) → check each topology/release ADR for completeness → compute composite availability from the observed topology and compare to any stated target → emit findings + risks per `../../shared/finding-schema.md` (read-only, authors **no** artifact, edits **no** source).

- **design:** **Frame** (extract/confirm the availability and scale drivers from `../00-drivers/quality-attribute-scenarios.md`, the RTO/RPO from `../40-cross-cutting/availability-resilience.md`; read the redundancy decisions already made; confirm the hosting target from `project-architecture-config.md`; surface any unstated availability target as an open driver question — do not assume a number; read-only) → **Explore** (N divergent deployment topologies as concrete diagrams-as-code specs — e.g. *(a)* single-AZ single-host (honest SPOF, accepted-risk record, rolling deploy, lowest cost); *(b)* single-AZ primary-store replica set + active-passive failover + managed reverse-proxy with rolling deploy, no double-infra; *(c)* multi-AZ managed cluster with blue-green frontend/API deploy + automated health-gate promotion — each with its real failure-domain claim, RTO/RPO implication, and order-of-magnitude cost; read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each direction against the utility tree + hard oracles using ATAM; identifies **sensitivity points** — e.g. RTO is sensitive to the primary store's failover-detection latency, zero-downtime is sensitive to whether readiness probes are configured — and **tradeoff points** — async replication improves write latency at the cost of a non-zero RPO; runs CBAM to reject multi-AZ/multi-region when ROI < 1 against the actual driver; order-swapped to kill position bias; penalizes accidental complexity; picks a winner and grafts superior ideas from runners-up) → **Document** (the **one** `solution-architect` writer authors: the **C4 deployment diagram** as diagrams-as-code in `architecture/<slug>/diagrams/` mapping every Container to its node/AZ/region and labeling each relationship's protocol and boundary; one **ADR per deployment-topology decision** and one **ADR for the release strategy** with ≥2 options and full consequences; the **environment-parity matrix** (dev / staging / prod: same image? config-only diff? secret store?) in the design doc; the **cost estimate** (order-of-magnitude per tier) and the **composite availability computation** in `capacity-model.md`; the **zero-downtime release sequence** (health probe → drain → rollout → readiness gate → rollback path) as a fitness-function spec in `fitness-functions.md`; reuses patterns from `cloud-well-architected.md` and resilience patterns from `../40-cross-cutting/availability-resilience.md`; preserves conceptual integrity — one coherent topology, not a committee's compromise) → **Verify** (the hard gates below run as a deterministic check; the evaluator re-scores from the *artifacts*, not from prose) → loop ≤ 10 or pass.

## Assertions & exit gate

- **Diagram compiles:** the C4 deployment view DSL renders without error; every Container from the L2 view appears in it (mapped to a node or explicitly marked `planned: true`); every node and relationship is labeled and typed; trust boundaries are drawn; sync/async visually distinct.
- **No un-accepted SPOF on an availability-SLO path:** for each `QAS-AVAIL-*` scenario, every cardinality-1 node on the traced path has a redundant peer in a genuinely independent failure domain **or** a signed accepted-risk entry naming the threatened scenario and the implied RTO.
- **Failure domains are real:** every node claimed to be in a distinct AZ is in an independently powered/networked facility (not the same physical host or the same data-center rack with a shared uplink). A "multi-AZ" topology that collapses to a single host fails this check structurally.
- **Release strategy is zero-downtime and reversible:** health/readiness probes are specified; graceful-drain timeout is bounded; rollback path is defined and executable within the availability-scenario RTO; no breaking database migration ships without an expand-contract sequence.
- **Environment parity:** dev, staging, and prod use the same Dockerfile and base image tag; all differences are externalized config (env vars / secret store references / origins). Verified by a structural diff of the build artifact spec across environments.
- **Fallacies honored (#1, #4, #5, #7):** no hardcoded host:port in container placement; inter-container trust boundaries labeled on the diagram; cross-AZ/region egress cost acknowledged in the cost model; the topology handles node failure without manual intervention or the risk is accepted.
- **Every topology and release-strategy decision is a complete ADR:** context · ≥2 options · decision · positive **and** negative consequences · traces to a `QAS-*` driver; rejected alternatives recorded.
- **Cost bounded:** order-of-magnitude monthly estimate per environment tier; multi-AZ CBAM ROI ≥ 1 against the actual availability driver.
- **Composite availability computed and meets the target** in `capacity-model.md` (serial multiplication + redundancy uplift); any scenario whose computed value < target with no accepted-risk entry fails.

**Gate:** hard oracles green (diagram compiles · no un-accepted SPOF · real failure domains · zero-downtime + reversible release · environment parity · fallacies honored · ADRs complete · cost bounded · composite availability ≥ target or open-driver-question declared) **AND** (design) rubric mean ≥ 0.8.

## Output

Write `artifacts/deployment-topology/report.json` per `../../shared/report-format.md` — findings/risks per `../../shared/finding-schema.md` (`type:"availability"` / `type:"operability"` / `type:"cost"` / `type:"documentation"`; `iso25010:"Reliability › Availability"` / `"Operational Excellence"` / `"Maintainability › Modifiability"`; `oracle:"hard"` for: diagram doesn't compile, un-accepted SPOF on an SLO path, claimed failure domain that is not independent, release strategy without health probes or rollback path, environment-parity gap with code-path diffs baked into the image, fallacy violated at the deployment boundary, ADR missing options/consequences, composite availability below target, cost unbounded; `oracle:"soft"` for: over-engineered topology relative to the driver, operability mismatch between release strategy and team size, evolvability seam missing for a deferred integration). Each finding carries `driver_ref` (the `QAS-AVAIL-*` / `QAS-PERF-*`), the **traced structural fact** in `evidence` (the SPOF node + cardinality + the AZ claim vs the physical truth, the missing health-probe spec, the parity gap), `tradeoff` + `alternatives_rejected` (e.g. "blue-green doubles infra cost during deploy and requires LB automation; buys zero-skew-window rollback — rolling rejected: safe for stateless API, insufficient when breaking API version shipped"). A `type:"risk"` entry is emitted for each **accepted** SPOF (likelihood × impact, disposition `accept/monitor`, the scenario it threatens, the implied RTO). The verification block includes: diagram-compile result, SPOF-trace per `QAS-AVAIL-*`, failure-domain independence check, release-strategy structural check (health probe / drain / rollback), environment-parity diff, composite-availability computation, and cost estimate. Authored artifacts (C4 deployment diagram, topology ADRs, release-strategy ADR, environment-parity matrix, cost estimate, zero-downtime fitness-function spec) land in **`architecture/<slug>/`** per `project-architecture-config.md`, never scattered. If the availability target / RTO / RPO is unstated, list it as an **open driver question** for the controller — never assume a number.

## Guardrails

Per `../../shared/guardrails.md`: **propose & document — never write product code and never apply infra** (no `terraform apply`, no container re-deployment, no reverse-proxy config change on a live host — author the topology and the ADRs; the controller writes code, the operator applies infra). **Ground every topology decision in a driver** — no AZ, no region, no standby node exists in the design without a `QAS-*` that demands it; un-driven redundancy is accidental complexity and wasted cost (flag it, don't ship it; right-size to the team topology and scale posture in `project-architecture-config.md`). **Reuse named patterns** (blue-green, canary, rolling, expand-contract) from `../80-patterns/cloud-design-patterns.md` and the `cicd-architecture.md` flow before inventing a bespoke release mechanism. **Record the tradeoff and the rejected alternatives** on every ADR (what the topology costs on latency, cost, consistency, and operability; what alternative topology it beat and why). **One writer** authors the C4 deployment view and the ADRs so the topology has conceptual integrity; the `architecture-evaluator` scores read-only and never edits. **Failure-domain claims must be verifiable** — if the target hosting environment isn't known, mark the node `hosting-target: TBD` and surface it as an open driver question rather than drawing a spurious multi-AZ diagram. Cross-reference `../40-cross-cutting/availability-resilience.md` for the FMEA and redundancy decisions this topology must reflect, `cicd-architecture.md` for the pipeline that executes the release strategy, and `cloud-well-architected.md` for the Well-Architected pillar sweep that evaluates this topology.
