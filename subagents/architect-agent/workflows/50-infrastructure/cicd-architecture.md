# cicd-architecture

**Group:** 50-infrastructure  ·  **Runs as:** subagent: ../.claude/agents/infrastructure-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Architect the delivery pipeline — trunk-based flow, build/test/deploy stages, progressive delivery, and supply-chain security (SLSA, SBOM, signed artifacts) — with quality and fitness-function gates wired in at the right stage. Answers the decision: *what pipeline shape, promotion model, and supply-chain controls satisfy the deployment-frequency, lead-time, security, and rollback-time drivers in the utility tree, at the right cost for this team size?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack (per the project's config — e.g. a web SPA + an API service + a database, behind a reverse proxy), the deployment target (containerized or not, deploy-ready build artifacts, real prod-deploy topology may be a **separate ADR — not yet built**), the team topology (as stated in the config), the constraint that any dev-only local tooling (e.g. a root compose/Makefile) does not sync to a shared/org runtime, and that introducing a new CI/CD runtime needs a driver-justified ADR (boring-tech-preferred).
- The **drivers / utility tree** (`../00-drivers/quality-attribute-scenarios.md`): the prioritized 6-part delivery scenarios — deployment frequency, change lead time, change-failure rate, mean time to restore (MTTR) — each ending in a measurable response measure. If DORA targets or a deployment-frequency / MTTR SLO are **not stated in the BRD/PRD**, surface them as **open driver questions** and do not assume a number. If the source-of-truth docs (per `project-architecture-config.md`) state performance NFRs (e.g. a page-render or a request-latency budget), these become pipeline gate thresholds (the fitness functions gate must keep these green); if availability/MTTR are unstated, surface as open.
- The **security + compliance driver** (`../40-cross-cutting/security-architecture.md`): PII / regulated data, RBAC scoping, audit logging are hard drivers where the project's domain calls for them; secrets (signing keys, encryption keys, DB creds) must never appear in pipeline logs, artifact metadata, or diagram source.
- The **EU Cyber Resilience Act (CRA)** as a named compliance driver for the supply-chain design (`../shared/supply-chain-contract.md` §1): Regulation (EU) 2024/2847, in force since 2024-12-10, vulnerability-reporting obligations apply from 2026-09-11 — a manufacturer must notify ENISA of an actively-exploited vulnerability within **24 hours**, which is only realistic with an SBOM + provenance chain already in place. Surface CRA applicability as an open driver question if EU market/customer scope is unconfirmed; do not assume it away.
- The **fitness functions** from `../60-evaluation/fitness-functions.md`: the pipeline's gate layer is the enforcement mechanism for the architecture's fitness functions — dependency rules, coupling metrics, perf budgets, security scan thresholds. Read those specs before designing the stage layout.
- The **IaC topology** from `infrastructure-as-code.md`: the environment topology (local/dev → staging → prod) defines the promotion chain; the pipeline must match it. If `infrastructure-as-code.md` does not yet exist, surface the prod-deploy topology as a **prerequisite ADR** — a pipeline cannot be fully designed without knowing the environments it promotes through.
- Preconditions: the source-of-truth docs read (the BRD / PRD / understanding log named in `project-architecture-config.md`); the repo structure scanned (branch strategy, current Actions/CI config if any, any documented org↔personal or fork↔upstream sync pattern); current artifact build scripts inspected. The team's branching model (per `project-architecture-config.md`) is a named input — do not re-derive.

## Oracle (source of truth)
The **delivery quality-attribute scenarios** (DORA four-key metrics + the SLSA level target + the BRD NFR thresholds as gate anchors) + the **supply-chain compliance requirements** (SBOM, provenance, signed artifacts). Never "the pipeline is fast / secure" with no number and no traced driver (`../shared/quality-attribute-rubric.md` grounding rule).

**hard:**
- **Artifacts are built once, promoted — not rebuilt per env.** An artifact rebuilt per environment can differ between what was tested and what was deployed — this is a structural correctness violation, not a style preference. *Structural check:* the container image digest (or the npm build hash) used in staging must be the **same** SHA promoted to prod; no `docker build` at deploy-time in any env after the initial CI build. An artifact rebuilt per env **blocks**.
- **Pipeline gates on the architecture's fitness functions before promotion.** The gate layer is not optional quality: it is the execution point of the `../60-evaluation/fitness-functions.md` checks — dependency rules, layering, RBAC/auth coverage, perf-budget assertions, accessibility thresholds. A promotion that bypasses the fitness-function gate destroys the evolutionary architecture contract. *Structural check:* each promotion step (staging → prod) must have an explicit gate block that references the fitness-function spec IDs it enforces; no gate = a blocked path.
- **Supply chain secured at SLSA level ≥ 2 (target: level 3 where toolchain permits), per `../shared/supply-chain-contract.md`.** SLSA (Supply-chain Levels for Software Artifacts) — **v1.2, APPROVED November 2025** (supply-chain-levels.dev/spec/v1.2; cite version + date, never "in development") — defines a progressively stronger provenance guarantee: Level 1 = build script generates provenance; Level 2 = hosted build service, signed provenance; Level 3 = hardened build service, non-falsifiable provenance. The default target is **SLSA L2 minimum, L3 where the toolchain permits** (a hosted CI runner, e.g. GitHub Actions, meets L2 today; `slsa-github-generator` closes the gap to L3 — state which is chosen in the SLSA-level ADR, per the contract's CBAM guidance). *Structural checks:* (a) the build runs on a hosted, non-self-hosted runner (SLSA L2 minimum — local builds are L0); (b) every produced artifact has a **signed provenance attestation** (Sigstore/cosign, keyless via Fulcio OIDC), verifiable by the deployment step before promotion; (c) a **SBOM** (Software Bill of Materials — CycloneDX or SPDX format) is generated at build time and attached to the artifact **as a signed attestation** (`cosign attest`), not a loose file; (d) every signature and attestation is appended to **Rekor** (Sigstore's immutable, publicly-auditable transparency log) — non-repudiation and anomaly detection depend on this, not just the signature itself; (e) all dependencies are scanned against a vulnerability database (OWASP Dependency-Check, Trivy, or equivalent) before the artifact is released; (f) container images are scanned for known CVEs. **Release-gate verification:** `cosign verify-attestation --type slsaprovenance ...` (exact command in `../shared/supply-chain-contract.md` §3) run as the pre-promotion gate step — exit 0 required to promote; exit non-zero **blocks**. An artifact promoted without provenance attestation, without a Rekor log entry, or without a signed SBOM **blocks**.
- **Deploys are reversible within the MTTR target.** Every deployment to a production-like environment must have a documented rollback path — either a blue-green traffic switch (≤ 1 min cutover), a feature-flag disable, or a prior-artifact redeploy — and the rollback must be automatable (no manual DB migration required as a prerequisite to rolling back). If rollback requires a DB migration, that migration must be backward-compatible before the deployment proceeds. *Structural check:* each ADR-specified deployment pattern names its rollback mechanism and its estimated rollback time; a deployment with no defined rollback **blocks**.
- **Secrets never in pipeline outputs.** Auth/session signing secrets, encryption keys, database credentials, and any signing key must be injected via the CI secret store (GitHub Actions Secrets / Vault) and must not appear in: build logs, SBOM artifact metadata, provenance annotations, pushed Docker layer history (`ENV`/`ARG` in Dockerfile), or diagram source. *Structural check:* scan pipeline YAML and Dockerfile for hardcoded secrets; any hit **blocks** (ties to `../40-cross-cutting/security-architecture.md` *Trust boundary secured* hard oracle).
- **Every cross-stage call has a timeout and a failure path.** A CI job that hangs indefinitely blocks the pipeline indefinitely — this is the distributed-systems fallacy that latency is zero, applied to pipeline steps. *Structural check:* every job in the pipeline YAML has an explicit `timeout-minutes`; every external call (package registry, container registry, Sigstore transparency log, deployment API) has a bounded retry with exponential backoff and a fail-fast on terminal errors. A job without a timeout **blocks** (fallacy #2 — `../shared/fallacies-and-tradeoffs.md`).
- **ADR-complete** — trunk-based vs feature-branch strategy, artifact promotion model, SLSA level target, progressive-delivery mechanism, and rollback topology are each an ADR with context · ≥2 options · decision · positive **and** negative consequences (`../shared/adr-template.md`). A decision without recorded alternatives **blocks**.

**soft:**
- The pipeline is the **path of least resistance to production-with-confidence** — a developer who fixes a bug can get it to prod in ≤ the lead-time target without manual steps, approvals, or tribal knowledge. If the process is the bottleneck rather than the code, the pipeline design has failed.
- **Gates are automated, not manual** — a quality gate that requires a human to check a dashboard and click "approve" degrades with team size and fatigue. Manual gates are acceptable only where a driver explicitly demands a human sign-off (e.g. a regulatory release approval); every other gate is a script with a pass/fail exit code.
- **Rollback is fast** — the preferred deployment pattern is one that makes rollback as fast as forward-deploy (blue-green, feature-flag) rather than one that makes it expensive (in-place redeploy requiring a rebuild). The architecture should make the right thing (rolling forward safely) easier than the wrong thing (hotfixing prod directly).

## Standards & techniques
- **Trunk-based development (TBD)** — the only long-lived branch is the trunk (e.g. `main`/`master`); feature work lands on **short-lived branches** (days, not weeks) merged via PR. The single-trunk model eliminates merge debt, reduces integration-failure blast radius, and keeps the integration gate fast. For the project's current branching model (per `project-architecture-config.md`), this is the correct shape — the architecture formalizes and gates it, not changes it. Pair with **feature flags** to decouple deployment from release (code ships dark; flag enables it per-tenant or per-user).
- **Artifact promotion (build once, promote)** — the build stage produces a single canonical artifact (a container image SHA, a versioned npm build) that is **promoted** through environments by changing deployment config, not by rebuilding. This is the DORA "build once, deploy everywhere" principle; it eliminates the "works in staging but broke in prod" class of failures. Promotion records the artifact SHA, the build provenance, the SBOM, and the fitness-function gate results in the artifact registry.
- **Progressive delivery** — new code reaches prod through a **controlled blast radius**: (a) **canary / percentage rollout** — route a small fraction (e.g. 5–10%) of traffic to the new version, hold, assert on error-rate + RED metrics, promote or roll back; (b) **feature flags** — deploy the code decoupled from enabling it (a feature-flag service or a simple config toggle), enabling per-tenant or per-user without a redeploy; (c) **blue-green switch** — maintain two identical prod slots, switch traffic at the load-balancer level, keep the old slot warm for instant rollback. Right-size the mechanism to the team/deployment topology in `project-architecture-config.md` — e.g. for a small team running a single internal app behind a same-origin reverse proxy, a **blue-green with feature flags** is often the right-sized default (canary requires a traffic-splitting layer that may not be in scope; multi-region blue-green can be over-engineered for that shape). Document which mechanism is chosen and why via an ADR.
- **Supply-chain security (SLSA + SBOM + Sigstore + Rekor)** — full contract in `../shared/supply-chain-contract.md`; treat the pipeline itself as an attack surface: (a) **SLSA levels** (v1.2, approved Nov 2025) — provenance assertions tell a consumer *how* an artifact was built, not just *what* it contains; Level 2 (hosted runner + signed provenance) is the minimum bar for a system handling PII or other regulated data (per the project's data-classification driver); Level 3 (hardened, non-falsifiable provenance, e.g. GitHub Actions with OIDC-signed provenance via `slsa-github-generator`) is the target; (b) **SBOM** — generate a CycloneDX or SPDX SBOM at build time (via `syft`, `cdxgen`, or `trivy sbom`) and attach it to the OCI image **as a signed attestation** (`cosign attest --type cyclonedx`/`--type spdx`), never a loose CI-artifact file; the SBOM enables dependency-level vulnerability tracking post-release and is the artifact that makes the CRA's 24-hour disclosure window realistic; (c) **Sigstore/cosign + Rekor** — sign the container image and its provenance attestation with a keyless (Fulcio OIDC-token-bound) signature; every signature is appended to **Rekor**, Sigstore's immutable public transparency log, giving non-repudiation and enabling anomaly detection (`rekor-monitor` alerts on signatures from an unexpected workflow identity); the deployment step runs `cosign verify-attestation` (exact command in the contract §3) as a **release gate** before pulling; (d) **Dependency scanning** — run `pip-audit` (Python backend), `npm audit` / `trivy fs` (frontend), and `trivy image` (container layer scan) as blocking gates on the build stage, not post-merge advisories; (e) **Secret scanning** — run `gitleaks` or `trufflehog` on every PR to block committed secrets before they reach the registry or the pipeline logs; (f) **EU Cyber Resilience Act** — the compliance driver that makes (a)–(c) non-optional for any EU-market system: 24-hour actively-exploited-vulnerability disclosure to ENISA depends on the SBOM + provenance chain already existing at release time, not built after the fact.
- **Fitness-function gate placement** — fitness functions (`../60-evaluation/fitness-functions.md`) are executable architecture characteristic checks; they live in the pipeline, not just in the design doc. Gate placement follows **shift-left + blast-radius** principles: (a) **PR gate** — fast checks: unit tests, lint, import/dependency rules (no circular deps, no UI layer importing internal service-layer modules directly), secret scan, license check; target ≤ 5 min; (b) **Integration gate (post-merge to trunk)** — fitness functions: coupling metrics, RBAC coverage check, performance-budget smoke (p95 within the response-time budget stated in the BRD/PRD NFRs, per `project-architecture-config.md`), accessibility gate (axe-core ≤ 0 critical violations), contract tests, SBOM generation, vulnerability scan; target ≤ 15 min; (c) **Pre-promotion gate (staging → prod)** — full fitness-function suite + canary/blue-green health check (error-rate ≤ threshold, p95 ≤ SLO for hold duration) + provenance-attestation verification + SBOM diff vs previous release; human sign-off only if a regulatory driver demands it. Each gate stage is a named, versioned job; skipping a stage requires an explicit `allow-list` override that is audited.
- **QA gate wiring** — the `qa-agent`'s smoke, regression, and performance-load workflows are the execution engine for the integration and pre-promotion gates; the pipeline invokes them, receives a structured pass/fail + `report.json`, and gates on it. The pipeline YAML references the QA invocation by the flow name; the architecture flow specifies *where* QA runs, not *how* (that's `qa-agent/`'s lane).
- **DORA four key metrics** as the fitness-function target set for the pipeline itself: **Deployment Frequency** (how often code ships to prod), **Lead Time for Changes** (commit → prod), **Change Failure Rate** (% deployments causing a prod incident), **Mean Time to Restore** (time to restore prod after a failure). These are the delivery-performance oracle; the pipeline design must demonstrate how it serves each one. If the BRD does not state targets, surface them as open driver questions.
- **Expand-contract for DB schema changes** — any migration that changes a shared database schema (relational or document store) must follow the expand-contract (parallel-change) pattern: (1) expand (add the new field/index while the old shape still works — backward-compatible deploy); (2) migrate data (async, online if possible); (3) contract (remove the old field only after all services have adopted the new shape). A breaking schema change deployed in a single step makes rollback require a reverse migration — a hard oracle violation (deploy not reversible). This is the pipeline-level enforcement of the consistency model from `../20-data/consistency-transactions.md`.
- **Immutable infrastructure + GitOps** — infra configuration lives in version control (`infrastructure-as-code.md`); the pipeline applies changes via the IaC tool (never SSH + manual apply). An infrastructure drift (live state diverges from repo state) is detectable by the pipeline's drift-detection step and surfaced as a finding, not silently tolerated. Where the project's current state has dev-only local tooling (e.g. a Compose file), deploy-ready build artifacts, but no prod-deploy topology built yet, the IaC design is a prerequisite for the prod-promotion stage — flag as a **blocked stage** until the topology ADR exists.

## Step sequence
- **audit:** Frame the delivery drivers (DORA targets, deployment-frequency goal, MTTR from `../40-cross-cutting/reliability-slo.md`, security/SLSA level — surface any unstated as open driver questions; craft review passes when rubric mean ≥ 0.8 for any design artifacts under review) → inspect the existing pipeline YAML (GitHub Actions config or equivalent), the Dockerfiles, the build scripts, and the branching model → check the **build-once-promote** contract (are artifacts rebuilt per env?), the **supply-chain controls** (provenance, SBOM, vulnerability scan — present/absent/blocking?), the **gate placement** (are fitness functions gated at the right stage?), the **timeout coverage** (every CI job has a `timeout-minutes`?), the **secret hygiene** (secrets in pipeline YAML or Dockerfile layers?), and the **rollback definition** (is it documented and automatable?) → compute the **lead-time and change-failure-rate** from commit history if available → emit findings + risks per `../shared/finding-schema.md` (read-only, authors **no** artifact, edits **no** source). Flag each violation by its hard oracle name and the driver it threatens.
- **design:** **Frame** (extract/confirm delivery drivers + DORA targets + SLSA level target + compliance constraints; read `../60-evaluation/fitness-functions.md` + `infrastructure-as-code.md`; surface unknowns as open questions) → **Explore** (≥2 divergent pipeline directions as concrete specs — e.g. *(a)* GitHub Actions trunk-based with blue-green + SLSA L2 + gitleaks + dependency-audit tooling + cosign signing, targeting the current team's shape per `project-architecture-config.md`; *(b)* same trunk + canary rollout via a weighted reverse-proxy upstream + SLSA L3 with `slsa-github-generator` + full Sigstore transparency-log integration, targeting a higher security posture; each direction names its pipeline topology, stage count, gate placement, supply-chain controls, rollback mechanism, and estimated lead-time + cost, read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each direction vs the delivery scenarios + hard oracles via ATAM; names the **sensitivity points** — e.g. lead-time is sensitive to the integration-gate duration; change-failure-rate is sensitive to the gate coverage — and the **tradeoff points** — SLSA L3 buys stronger provenance at higher pipeline complexity / slower builds; blue-green buys instant rollback at double-infra cost; feature flags decouple deploy from release at the cost of flag-management overhead; runs **CBAM** so L3 is chosen only when the security driver demands it over L2's lower complexity; order-swapped to kill position bias; penalizes over-engineering) → **Document** (the **one** `solution-architect` writer authors: the **pipeline topology diagram** — stages, gates, promotion steps, rollback paths — as diagrams-as-code into `architecture/<slug>/diagrams/`; one **ADR per significant pipeline decision** — branching model, artifact-promotion model, SLSA level, progressive-delivery mechanism, rollback topology — into `architecture/<slug>/adr/`; the **gate specification** — which fitness-function IDs are enforced at which stage, which QA flows are invoked, what the pass/fail criteria are — into `architecture/<slug>/design-doc.md`; the **supply-chain controls catalog** (per `../shared/supply-chain-contract.md`) — SLSA target level + toolchain, SBOM format + generation tool + attestation method, Sigstore/cosign + Fulcio signing config, Rekor transparency-log backend + anomaly-detection step, vulnerability-scan tool + threshold, secret-scan tool, and the `cosign verify-attestation` release-gate command — into `design-doc.md`; the **delivery fitness functions** — deployment-frequency counter, lead-time measurement, change-failure-rate tracker, MTTR measurement — into `architecture/<slug>/fitness-functions.md`; reuses named patterns from `../80-patterns/cloud-design-patterns.md` and SLSA/SBOM standards, preserves conceptual integrity) → **Verify** (the hard gates below as a deterministic check; the evaluator re-scores from the *artifacts*, not the prose; the diagram-compile oracle run on the pipeline topology diagram) → loop ≤ 10 or pass.

## Assertions & exit gate
- **Build-once-promote confirmed** — the pipeline YAML shows the artifact SHA (image digest or build hash) is captured at build time and referenced by digest at every downstream stage; no `docker build` or `npm run build` appears in a deploy step after the initial CI build job.
- **Every promotion stage has an explicit gate block** — a named job that enforces the fitness-function IDs from `../60-evaluation/fitness-functions.md` at that stage; no gate-skip path without an audited allow-list.
- **Supply-chain controls are blocking gates, not post-merge advisories** (per `../shared/supply-chain-contract.md`) — provenance attestation generated + signed (Sigstore/cosign, Fulcio OIDC) + appended to **Rekor** + verified via `cosign verify-attestation --type slsaprovenance ...` as the **release gate**, SBOM (CycloneDX/SPDX) generated + attached to the image as a signed attestation (`cosign attest`), dependency scan (pip-audit + npm audit + Trivy CVE scan) exits non-zero on CVSS ≥ threshold, secret scan (gitleaks/trufflehog) runs on every PR. Each control is in a build/integration stage step, not a nightly advisory job. The **EU Cyber Resilience Act** is named as the compliance driver requiring these controls exist *before* release, not retrofitted — a design with no SBOM/provenance for an EU-market system blocks under `Data lifecycle lawful` / `Supply-chain provenance` (`../shared/quality-attribute-rubric.md`).
- **SLSA level target achieved** — SLSA **v1.2 (approved November 2025)** is the cited spec version; provenance attestation is hosted-runner-signed (L2 minimum); if L3 is the target, `slsa-github-generator` or equivalent non-falsifiable provenance toolchain is used and named in the ADR, with Rekor as the named transparency-log backend.
- **Every CI job has `timeout-minutes`** — no job hangs indefinitely; every external call (registry, Sigstore log, deployment API) has a bounded retry with backoff and a fail-fast exit code. A timeout-less job **blocks**.
- **Secrets hygiene clean** — no secret appears in: pipeline YAML values (only `${{ secrets.NAME }}`), Dockerfile `ENV`/`ARG` layers pushed to registry, SBOM metadata, provenance annotation, or diagram source. The secret-scan step exit code is 0.
- **Rollback is defined, automatable, and within MTTR target** — each deployment ADR names its rollback mechanism (blue-green traffic switch / feature-flag disable / prior-image redeploy) and its estimated rollback time; any schema migration involved is expand-contract (no breaking change without a backward-compatible expand step first).
- **Every pipeline decision is an ADR** — branching model, artifact-promotion model, SLSA level, progressive-delivery mechanism, rollback topology — each with context · ≥2 options · decision · positive + negative consequences + follow-up fitness functions. A decision with no recorded alternatives **blocks**.
- **Pipeline topology diagram compiles** — the diagrams-as-code (Mermaid or PlantUML) renders without error; every stage, gate, and promotion step is labeled with name + type + relationship verb; sync/async stages are visually distinct; trust boundaries (where signing/verification occurs) are drawn (`../shared/diagramming-standards.md`).
- **DORA metrics are traceable** — for each of the four key metrics, the design names the measurement point (where deployment-frequency is counted, where lead-time starts and ends, how change-failure-rate is defined, where MTTR is tracked) and the fitness function that enforces it. Unstated DORA targets are listed as open driver questions, not silently omitted.
- **Cost bounded** — the pipeline design carries an order-of-magnitude cost estimate against a stated budget or team-size constraint: runner minutes (hosted vs self-hosted), registry storage, SLSA/Sigstore toolchain overhead, and progressive-delivery infrastructure (blue-green slot, feature-flag service) are each estimated; no component is "scale infinitely, cost unstated." If cost is an explicitly unconstrained driver for this team, that is stated; otherwise a missing cost model **blocks** (hard oracle `Cost bounded` in `../shared/quality-attribute-rubric.md`).
- **Gate:** hard oracles green (build-once-promote · gates enforce fitness functions · supply-chain controls block · SLSA level met · every job timed out · secrets clean · rollback defined · ADRs complete · diagram compiles · DORA metrics traceable · cost bounded) **AND** (design) rubric mean ≥ 0.8.

## Output
Write `artifacts/cicd-architecture/report.json` per `../shared/report-format.md` — findings/risks per `../shared/finding-schema.md` (`type:"operability"` / `"security"` / `"evolvability"` / `"compliance"` / `"risk"` as appropriate; `oracle:"hard"` for a rebuild-per-env violation, a gate bypass, an un-signed artifact promoted to prod, a job without a timeout, a secret in pipeline output, a deployment with no rollback definition, or an incomplete ADR; `oracle:"soft"` for over-engineered gate complexity, manual gates where automation is feasible, or a pipeline shape mismatched to the team size). Each finding carries its `driver_ref` (the DORA metric, `QAS-DEPLOY-*`, `QAS-SEC-*`, or the SLSA level constraint), the **traced structural fact** in `evidence` (e.g. "deploy-prod.yml line 42 runs `docker build` — artifact rebuilt, not promoted from staging digest"), and a `tradeoff` + `alternatives_rejected` (e.g. "adding Sigstore transparency-log verification as a blocking gate adds ~30s to the pre-promotion stage and requires network connectivity to the Rekor log; buys non-falsifiable deployment provenance; alternative — skip transparency log, sign locally — rejected: does not meet SLSA L3 non-falsifiable provenance requirement"). The verification block includes the supply-chain-controls coverage table (SLSA level, SBOM format, signing tool, scan tools + thresholds), the build-once-promote trace (artifact SHA continuity across stages), the gate-placement matrix (stage → fitness-function IDs enforced), the secret-hygiene scan result, and the rollback-mechanism table per environment. Authored design artifacts (pipeline topology diagram, delivery ADRs, gate-spec, supply-chain-controls catalog, delivery fitness functions) land in **`architecture/<slug>/`** per `project-architecture-config.md`, never scattered. If DORA targets / MTTR SLO / SLSA level are unstated, list them as **open driver questions** for the controller — never silently assume a number. A `type:"risk"` entry is emitted for each deferred control (e.g. prod-deploy topology not yet built → pre-promotion stage is a blocked stage until `infrastructure-as-code.md` ADR exists), with likelihood × impact and disposition `monitor/mitigate`.

## Guardrails
Per `../shared/guardrails.md`: **propose & document — never write product code and never apply infra** (no `terraform apply`, no pipeline execution, no registry push, no live CI/CD config change — author the *pipeline design and the ADRs*; the controller wires the YAML, the operator applies infra). **The supply-chain design/target is authored by the fleet; verification is run by the operator** — the fleet writes the SLSA-level ADR, the SBOM/signing/Rekor design, and the exact `cosign verify-attestation` release-gate command (`../shared/supply-chain-contract.md` §7 table); it never signs an artifact, never pushes to Rekor, never holds a signing key, and never executes `cosign` itself. **Ground every gate, stage, and supply-chain control in a driver** — no SLSA level, no SBOM, no progressive-delivery mechanism exists without a `QAS-*` or compliance constraint that demands it; an un-driven gate is accidental complexity (flag it, don't ship it). **Reuse named standards and tools before inventing** — SLSA (v1.2), Sigstore/cosign/Fulcio/Rekor, CycloneDX/SPDX SBOM, DORA metrics, GitHub Actions OIDC provenance, Trivy, gitleaks are the known pattern set; a bespoke signing scheme needs a driver-justified ADR showing why the standard tool does not fit. **Record the tradeoff and the rejected alternatives** on every pipeline decision — what the chosen shape costs on lead-time / complexity / cost / security posture, and what it beat. **Secrets are never in the deliverable** — pipeline YAML examples in ADRs and diagrams use placeholder names (`${{ secrets.JWT_SECRET }}`), never real values. **One writer** authors the pipeline design so it has conceptual integrity; the read-only `architecture-evaluator` scores it and never edits it. This flow **specifies** the fitness-function gate IDs and the QA-invocation hooks for `qa-agent` and `../60-evaluation/fitness-functions.md` to execute — it never runs the pipeline or the tests itself. Cross-reference `infrastructure-as-code.md` (the environment topology the promotion chain must match), `../40-cross-cutting/security-architecture.md` (the trust boundaries the supply-chain controls must secure), and `../shared/supply-chain-contract.md` (the full SLSA/Sigstore/SBOM/Rekor contract and the EU Cyber Resilience Act driver this flow's supply-chain hard oracle enforces) before authoring any stage design — these are not optional reads; a pipeline divorced from its infra topology, its security threat model, and its supply-chain contract is a broken artifact.
