# networking-architecture

**Group:** 50-infrastructure  ·  **Runs as:** subagent: ../../agents/infrastructure-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Design the network as a set of explicit **failure and security domains** — VPC/subnet layout, ingress/egress controls, DNS and service discovery, CDN/edge, and east-west (service-to-service) zero-trust segmentation — honoring the two network fallacies that kill the most designs: "the network is reliable" and "the network is secure." Answers the *decision* "which topology partitions blast radius, satisfies the security drivers (PII/sensitive data, RBAC scoping, STRIDE-identified threats), and meets the availability + latency response measures without over-engineering for a scale the system doesn't have?"

## Inputs & preconditions
- From `project-architecture-config.md`: the existing deployment shape (per the project's stack lock — e.g. a web SPA behind a reverse proxy, an API tier, a primary datastore; deploy-ready containerization if present; prod deploy may be a **separate, not-yet-built path** — managed DB, real secrets, env-driven origins); the stack lock (the project's stack — a new network component needs a driver-justified ADR); the compliance regime (PII/sensitive data if applicable; RBAC scoping is a hard security driver where stated; audit logging is a compliance driver where stated); the team topology (per `project-architecture-config.md` — Conway: the network the team can operate, not the one that looks impressive).
- The architecture **drivers** — the prioritized utility tree from `../00-drivers/quality-attribute-scenarios.md`. Pull each scenario that the network topology is on the critical path of. Specifically: any `QAS-AVAIL-*` (the LB/NAT path, the datastore connectivity path, the ingress path must have no un-accepted SPOF); any `QAS-PERF-*` (a stated render/response-time budget — the latency budget is spent on hops the network adds; cross-AZ latency is a tradeoff point); any `QAS-SEC-*` (PII/sensitive data in transit; trust boundaries at every ingress and service boundary; STRIDE-derived network-layer controls); any `QAS-COST-*` (cross-AZ/region egress is not zero — fallacy #7; the cost model must account for it). **If a driver (RTO, availability target, cloud target, specific scale) is not stated, surface it as an open question — do not invent a number.**
- The C4 deployment view from `deployment-topology.md` (environments, regions, AZ layout) and any authored `architecture/<slug>/threat-model.md` (STRIDE trust boundaries that constrain the segmentation design). Read both before proposing subnet or trust-boundary choices.
- Preconditions: the source-of-truth NFRs read (the BRD / PRD / understanding log named in `project-architecture-config.md`); the cloud target identified (if none, surface it as an open driver — do not pick a provider reflexively); the shared-responsibility boundary established (who secures the network fabric vs the application-layer controls vs us, per the managed-vs-self split decided in `cloud-well-architected.md`).

## Oracle (source of truth)
Every network decision is scored against the **quality-attribute scenario it serves** — not "industry best practice" in the abstract. The grounding rule from `shared/quality-attribute-rubric.md`: no number + no traced driver = a preference, not a finding.

**hard:**
- **Segmentation creates real failure and security domains** — private subnets for the data tier (the primary datastore) are structurally isolated from the public ingress tier; the data subnet has no route to the internet (no internet-gateway route, no public IP); this is checkable in the route table, not asserted in prose. A design where the database is reachable from the public subnet without a bastion or VPN is a hard oracle failure, traced to the PII/compliance driver.
- **No un-accepted SPOF on the availability-SLO path** — the ingress path (LB → reverse proxy → API tier) and the NAT path (for managed-service egress) must have no single point of failure on any `QAS-AVAIL-*` path, OR the SPOF is an explicit accepted risk in the register. A single NAT gateway in one AZ when an availability driver exists is a hard failure (`shared/quality-attribute-rubric.md` → `no_unaccepted_spof`). The LB must be redundant (multi-AZ or an HA-pair) if an availability scenario is active.
- **East-west traffic is authenticated (zero-trust)** — service-to-service calls (client → API tier → datastore) are not implicitly trusted by network position; each hop has an explicit authn/authz control (a token at the API boundary; client TLS or IP-allow-list for the datastore; no "trusted because same VPC" assumption). Checked structurally against the STRIDE analysis in `../40-cross-cutting/security-architecture.md` — every trust boundary has a stated control, and "same subnet = trusted" is not a control.
- **Egress is controlled** — all outbound traffic from the application tier flows through a controlled egress point (NAT gateway or an explicit egress proxy) with an allowlist or security-group-level restriction; no subnet with a direct internet-gateway route unless it is the public ingress tier by design. The data subnet has no egress to the internet at all.
- **DNS and service discovery handle topology change** (fallacy #5 — topology doesn't change) — no hardcoded IP addresses or host literals in application config or in network rules; service endpoints are resolved by name (internal DNS / service-discovery, environment-variable-injected hostnames); the design gracefully handles node IP changes, scale-out rebalancing, and container restarts without a config change.
- **Fallacies honored** — the design explicitly accounts for all 8 fallacies of distributed computing (`shared/fallacies-and-tradeoffs.md`) where they apply to the network layer: (1) every remote call (client → API, API → datastore, API → an external payment/integration endpoint) has a timeout + retry policy; (2) the latency budget per scenario is spent across hops — cross-AZ latency is accounted; (3) bandwidth is not infinite — large payload paths (document processing, bulk data export) are identified and a claim-check or pagination strategy is noted; (4) the network is not secure — mTLS or a VPN channel for the datastore, TLS termination at the LB, a verified token at every API trust boundary; (5) topology change handled via DNS/service-discovery; (6) there is one administrator is not assumed — the team topology (per `project-architecture-config.md`) may split ownership across independent deploy cadences; the network design uses explicit versioned API contracts and no-coordinated-deploy assumptions at every service boundary, so one side's container restart or config change does not require a synchronised deploy from the other; (7) cross-AZ/region egress cost in the cost model; (8) the network is not homogeneous — at any external integration boundary (e.g. a payment provider) the protocol, payload format, and schema version are foreign and may evolve independently; an anti-corruption layer (ACL) with explicit schema validation and a versioned adapter sits at that boundary; mixed TLS cipher suites and HTTP versions across LB, reverse proxy, and API tier are declared and reconciled, not assumed uniform.
- **STRIDE controls at every trust boundary** — at minimum: the public ingress boundary (internet → LB/WAF → reverse proxy), the application-tier boundary (reverse proxy → API tier), and the data boundary (API tier → datastore). Each boundary has a stated STRIDE-derived control or an accepted risk. A boundary with no STRIDE coverage blocks (traced to the PII/compliance driver). Cross-references `../40-cross-cutting/security-architecture.md`.
- **ADR complete** per `shared/adr-template.md` for every one-way-door network decision: the VPC/subnet topology (hard to restructure without downtime), the zero-trust segmentation model (hard to relax without a security review), the DNS/service-discovery mechanism (tied to the deployment platform). Two-way-door decisions (security group rules, CIDR block sizing within a VPC) are decided cheaply and not over-analyzed.
- **Diagram compiles** — the network topology diagram (C4 deployment view showing subnet layout, trust boundaries, ingress/egress paths, service-discovery flow) is diagrams-as-code (PlantUML + C4-PlantUML or Mermaid) and renders without error (`shared/diagramming-standards.md`). Every trust boundary is drawn as a boundary box; every relationship is labeled with the protocol + direction + authn control.

**soft (rubric-scored, after hard oracles are green):**
- **Segmentation is driver-justified, not ceremonial** — the number of subnets, the tier count, and the cross-AZ spread are driven by the blast-radius + compliance requirements and the stated scale. A three-tier VPC with six subnets for a single-instance modular monolith with no multi-AZ availability driver is accidental complexity (`shared/quality-attribute-rubric.md` — penalize over-engineering). Conversely, a flat single-subnet design when PII/compliance demands tier isolation is under-engineered and fails this rubric.
- **The network honors the fallacies in its failure-handling design** — retries on transient network failures are configured at the application layer (not patched at the LB layer with blind retries that cause double-posting on non-idempotent endpoints); circuit breakers or bulkheads exist at the application tier, not assumed in the network.
- **Right-sized for the team** — the network topology is operable by the stated team (per `project-architecture-config.md`; no zero-trust service mesh requiring a dedicated mesh-operations specialist unless the driver demands it); managed network components (managed LB, managed NAT, managed DNS) are preferred over self-operated equivalents where the TCO is lower and the driver does not demand bespoke control.

## Standards & techniques
- **VPC / subnet / AZ layout** — the canonical three-tier segmentation: **public** (internet-facing: load balancer, WAF, optionally CDN origin); **private / application** (not internet-routable: the API-tier containers, the reverse proxy if internal); **data** (private, no internet route: the primary datastore). Each tier maps to a separate subnet (or subnet group) with a corresponding route table. AZ spread (at least two AZs for any `QAS-AVAIL-*` driver) is the structural mechanism for the no-SPOF hard oracle. Per the AWS Well-Architected Framework (Reliability pillar) and the Azure WAF equivalent.
- **Ingress patterns** — **LB/WAF** (Layer-7 application load balancer with WAF rules: OWASP Top-10, rate limiting, geo-block if the compliance regime requires) as the only public-facing entry point; TLS termination at the LB; the application tier never has a public IP. **API gateway** as an ingress variant when the driver demands edge-side authn, rate-limiting per client, or request transformation — for a project with today's shape per `project-architecture-config.md` (e.g. internal, single API service, same-origin SPA), a WAF + LB is often the right-sized default; an API gateway is a two-way-door decision that can be added later with an ACL/strangler-fig seam.
- **Egress control** — NAT gateway (managed, HA in each AZ if availability-driver exists) for application-tier outbound (dependency updates, external integration endpoints, an optional managed datastore service); VPC endpoints (AWS PrivateLink / Azure Private Endpoint) for cloud-managed services that support them, eliminating internet-routed egress to those services and reducing egress cost. Security groups / NACLs enforce the *deny-by-default* model; allow rules are explicit, narrowly scoped.
- **DNS and service discovery** — environment-variable-injected hostnames only; no hardcoded IPs (fallacy #5). Internal DNS (Route 53 Private Hosted Zone / Azure Private DNS / CoreDNS in a container platform) resolves service names within the VPC. Health-check-backed DNS failover (Route 53 health checks or equivalent) is the mechanism for AZ-level failover where no active-active LB exists. The discovery contract: a service registers a name, not an IP; callers resolve on each connection attempt (TTL-aware, not cached-forever).
- **CDN / edge** — a CDN (CloudFront / Azure CDN / Cloudflare) in front of the SPA static assets reduces origin load and latency for the initial page render (traced to a `QAS-PERF-*` render/response-time scenario); the API is never behind a CDN cache layer (stateful, authenticated, not idempotent-GET-only). CDN origin is the LB, not the container directly. Cache-control headers govern asset TTL; the API path is excluded from CDN caching by path rule.
- **East-west (service-to-service) vs north-south (client-to-service) traffic** — north-south: TLS-terminated at the LB, token-verified at the API boundary; no direct-to-container access. East-west (any future inter-service call, or a deferred integration seam per `project-architecture-config.md`): zero-trust by default — mTLS or a scoped service token, not "trusted because same VPC." A single-service modular monolith typically has no east-west service calls; the seam for a future service boundary is an ACL or a sidecar entry point, not a flat network trust.
- **Zero-trust microsegmentation** — applied at the granularity the driver demands. Illustrative pattern: security groups enforce that only the application-tier subnet can reach the datastore on its port; the application tier can only reach the data tier and the controlled egress point (NAT/VPC endpoint); the public ingress tier can only reach the application tier on the HTTPS port. "Same VPC = trusted" is explicitly rejected. This is the structural encoding of fallacy #4 (the network is not secure) and the STRIDE threat-model controls for the data-tier trust boundary.
- **STRIDE per trust boundary** — threat categories applied to each network boundary: **S**poofing (token/mTLS at the API boundary; TLS at the LB), **T**ampering (TLS in transit; WAF at ingress), **R**epudiation (audit logs at the API layer; flow logs at the VPC level), **I**nformation disclosure (private subnets; no publicly reachable datastore; encryption in transit), **D**enial of service (WAF rate limits; LB connection limits; security group port restriction), **E**levation of privilege (security groups deny lateral movement; IAM least-privilege for managed services). Each category has a stated control or an accepted risk. References `../40-cross-cutting/security-architecture.md` — this flow does not duplicate the full threat model; it grounds the network controls in it.
- **The 8 fallacies as a network design checklist** (`shared/fallacies-and-tradeoffs.md`) — walked per network path: ingress (LB timeout + connection limit + health-check retry), application-to-data (datastore connection pool timeout + retry-on-transient-error), application-to-external (an external integration endpoint, e.g. a payment provider: timeout bounded < 60s, retry idempotent-only with backoff+jitter, circuit-breaker), cross-AZ cost (bandwidth + NAT cost in the cost model), one-administrator (#6 — if the team topology splits ownership across independent deploy cadences per `project-architecture-config.md`: API versioning + no-coordinated-deploy contracts at the reverse-proxy → API boundary; backward-compatible config changes documented in the ADR), homogeneous-network (#8 — any external-integration ACL with explicit schema versioning + wire-format declarations; TLS version + cipher suite compatibility matrix at LB/reverse-proxy/API tier stated in the ADR). Not assumed away on any path.
- **AKF scale cube** — for networking: X-axis (horizontal LB across instances — the LB pool); Y-axis (future bounded-context decomposition — the seam for a deferred service boundary lives at an ACL or sidecar, not a flat network trust); Z-axis (tenant/organization partitioning at the application layer, not the network layer — the modular monolith + RBAC scoping handles this without per-tenant subnets). Right-sized: no per-tenant VPC unless a multi-tenancy compliance driver demands it.
- **MADR ADRs** (`shared/adr-template.md`) for one-way-door network decisions: VPC/subnet topology, zero-trust segmentation model, DNS/service-discovery mechanism, CDN strategy. Two-way-door decisions (security group CIDRs, LB listener rules) decided lightly and not over-ADR'd.

## Assertions & exit gate
- **Segmentation is structural** — the data subnet has no internet-gateway route; the application subnet has no direct path to the internet (only via NAT/egress-proxy); this is checkable in the route-table diagram, not asserted in prose.
- **No un-accepted SPOF** on the ingress path (LB → reverse proxy → API tier) or the NAT/egress path for any active `QAS-AVAIL-*` scenario; each SPOF is either redundant (multi-AZ LB, HA NAT) or an explicitly accepted risk in the register.
- **East-west authn at every trust boundary** — no "same VPC = trusted" assumption; each boundary has a stated authn/authz control traced to the STRIDE model in `../40-cross-cutting/security-architecture.md`.
- **Egress is controlled** — no subnet with a direct internet-gateway route unless it is the designated public ingress tier; the data subnet has no outbound internet route; all application-tier egress flows through the NAT or a VPC endpoint.
- **DNS / service discovery** — zero hardcoded IPs in any network rule, application config, or diagram relationship; all service names are DNS-resolved; the discovery mechanism handles container restarts and scale-out without a config change.
- **All 8 fallacies addressed** per network path — (1) timeout + retry on every cross-process call (client→API, API→datastore, API→any external integration endpoint); (2) latency budget traced to a `QAS-PERF-*` scenario with per-hop accounting; (3) large-payload paths (document processing, bulk export) identified with a claim-check or pagination strategy; (4) no "same VPC = secure" assumption — zero-trust authn at each boundary; (5) topology-change handled via DNS/service-discovery with no hardcoded IPs; (6) if the team topology splits ownership across independent deploy cadences (per `project-architecture-config.md`) — versioned API contracts and no-coordinated-deploy assumption at the reverse-proxy→API boundary, documented in the ADR; (7) cross-AZ egress cost captured in the capacity model; (8) any external-integration boundary carries a versioned ACL adapter with explicit schema validation — no homogeneous-stack assumption. Each fallacy accounted per path in the fallacy coverage matrix in `report.json`.
- **STRIDE controls at every trust boundary** — internet→LB, LB→application, application→data — each with a stated control or accepted risk per category; no boundary silently unmodeled.
- **ADRs complete** for every one-way-door network decision — VPC/subnet topology, zero-trust segmentation model, DNS/service-discovery mechanism, CDN strategy — each with context · ≥2 options · decision · positive AND negative consequences · driver refs.
- **Diagrams compile** — the C4 deployment view with subnet tiers, trust boundaries, ingress/egress paths, DNS resolution, and CDN layer renders without error; every element and relationship is labeled and typed; sync/async distinguished; trust boundary boxes present.
- **Cost bounded** — an order-of-magnitude estimate for the network layer (LB, NAT, VPC endpoints, CDN, cross-AZ egress, VPC flow logs) against the stated budget or surfaced as an open driver question; no component is "elastic, cost unstated."
- **Gate:** hard oracles green (segmentation structural · no un-accepted SPOF on `QAS-AVAIL-*` paths · east-west authn at each trust boundary · egress controlled · DNS/discovery topology-safe · 8 fallacies honored · STRIDE per trust boundary · ADRs complete · diagrams compile · cost bounded) AND (design) rubric mean ≥ 0.8 (segmentation is driver-justified not ceremonial · fallacies honored in failure-handling design · right-sized for the team and the stated scale).

## Output
Write `artifacts/networking-architecture/report.json` per `shared/report-format.md` — `drivers[]` (the `QAS-AVAIL-*`/`QAS-SEC-*`/`QAS-PERF-*`/`QAS-COST-*` scenarios the network is on the critical path of), `findings[]` per `shared/finding-schema.md`. Each finding carries all required schema fields: `id` (`"ARCH-NET-NNN"`); `type` (`"availability" | "security" | "performance" | "cost" | "operability" | "risk"`); `severity` (`"critical"` for un-accepted SPOF or violated hard security oracle; `"major"` for missing egress control, hardcoded host, violated fallacy, missing STRIDE boundary control, incomplete ADR, non-compiling diagram; `"minor"` for unbounded cost without an open-question record; `"cosmetic"` for documentation gaps — severity drives the gate: critical/major hard findings block, minor/cosmetic are advisory); `oracle` (`"hard"` for SPOF, missing egress control, hardcoded hosts, violated fallacy, missing STRIDE control, incomplete ADR, non-compiling diagram, unbounded cost; `"soft"` for ceremonial over-segmentation or over-engineered topology without a driver); `driver_ref` (the `QAS-*` scenario the finding is traced to — a finding with no driver_ref is a preference, not a finding); `iso25010` (the ISO/IEC 25010 characteristic — e.g. `"Reliability › Fault Tolerance"` for SPOF findings; `"Security › Confidentiality"` for egress/disclosure; `"Performance Efficiency › Time Behaviour"` for latency-budget gaps; `"Maintainability › Modifiability"` for hardcoded hosts); `component` (the network boundary or tier the finding is scoped to — e.g. `"data-subnet / egress route"`, `"reverse-proxy → API trust boundary"`, `"payment-integration ACL"`); `decision_ref` (the ADR id this finding traces to or demands — e.g. `"ADR-0001 (vpc-subnet-topology)"`; omit only for pre-ADR findings where no decision exists yet); `issue` (one sentence stating what is wrong vs the driver/standard — e.g. `"The data subnet route table contains an internet-gateway route, violating the PII isolation driver QAS-SEC-02"`); `evidence` (the traced driver + the structural fact — topology element, route-table entry, missing config — not prose assertion); `tradeoff` (what the fix costs on other attributes); `alternatives_rejected` (options considered and why not); `recommendation` (ADR-shaped: pattern/topology/control + the response-measure it now satisfies); `status` (`"open"`). Include a **network boundary coverage matrix** (boundary → STRIDE category → control or accepted risk) and a **fallacy coverage matrix** (fallacy #1–#8 → network path → accounting mechanism — each cell must be non-empty or marked `"N/A: [reason]"`). The **risk register** (`type:"risk"`) carries items whose `recommendation` is accept/mitigate/monitor with `severity` = likelihood×impact.

## Guardrails
Per `shared/guardrails.md`: **propose and document — never write product code and never apply infrastructure.** This flow authors network topology ADRs, diagrams-as-code, and STRIDE boundary controls into `architecture/<slug>/`; it never edits `src/**`, `backend/**`, `frontend/**`; it never runs a cloud CLI, a Terraform apply, a VPC change, or a security-group modification — the controller builds, the operator applies the plan. **Ground every network decision in a driver** — no subnet without a blast-radius or compliance driver; no CDN without a latency driver; no multi-AZ without an availability driver. Segmentation that no driver demands is accidental complexity (`shared/quality-attribute-rubric.md`) — flag it, don't ship it. **Right-size, don't résumé-drive** — per `project-architecture-config.md`, default to managed, boring network components (managed LB, managed NAT, managed DNS) over self-operated equivalents unless the stated scale and team topology demand more; do not propose a service mesh, a dedicated zero-trust network overlay, or per-tenant VPCs without a concrete driver. Accept lock-in in managed network components only where the leverage (operational simplicity, the SLA, lower TCO for the stated team) outweighs the reversal cost — state the tradeoff. **Record the tradeoff and the rejected alternatives on every one-way-door decision, never just the winner.** Cross-reference sibling flows rather than duplicating them: `../40-cross-cutting/security-architecture.md` (the full STRIDE threat model the network controls are grounded in — this flow reads it, does not re-derive it), `deployment-topology.md` (the C4 deployment view and AZ/region layout the subnet design is argued against), `shared/fallacies-and-tradeoffs.md` (the 8 fallacies each network path is checked against). A read-only audit makes no edits; exactly one writer (`solution-architect`) authors the design artifacts; the proposer and the evaluator are read-only throughout.
