# realtime-streaming

**Group:** 90-specialized  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Architect real-time and low-latency delivery paths — WebSocket, SSE, long-poll, and server-push — together with the event-streaming infrastructure at scale (partitioning, consumer groups, backpressure) and time-window processing (windowing, watermarks, exactly-once vs idempotent) that feeds them. Answers the decision: *what delivery mechanism, connection topology, and fan-out strategy satisfy the stated freshness and tail-latency drivers, and what does each cost on consistency, throughput, operational complexity, and connection-state management?* This flow owns the real-time path from the broker (or source-of-truth store) through the push channel to the client — where `../20-data/streaming-data.md` owns the pipeline that produces events, and `../40-cross-cutting/performance-capacity.md` owns the latency budget that constrains the total hop.

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack + deployment target (per `project-architecture-config.md` — e.g. an SPA behind a reverse proxy → an API service → a primary datastore; whether a broker or WebSocket server is already in the stack); the compliance regime (per `project-architecture-config.md` — push channels cross a trust boundary and must carry authn/authz per `security-architecture.md`); team topology (per `project-architecture-config.md`; Conway applies — no streaming infrastructure without a driver-justified ADR and a team that can operate it); the artifact locations (`architecture/<slug>/`).
- The architecture **drivers**: the prioritized quality-attribute scenarios (utility tree from `workflows/00-drivers/quality-attribute-scenarios.md`) that real-time delivery must serve — specifically: (a) the **freshness scenario** (e.g. "a manager receives a status change within N seconds of the transition, measured p99 at expected concurrent subscriber count") and (b) the **tail-latency scenario** (e.g. "a push notification from the server to an active client is delivered within M ms p99 at K concurrent connections"). Both must be 6-part scenarios with response measures. A flow with no stated freshness or tail-latency driver **downgrades scope** and defers real-time machinery — polling or a short-TTL cache is simpler and correct when no sub-minute freshness scenario is stated.
- The **functional scope**: which events are pushed to which client roles (per the project's RBAC roles in `project-architecture-config.md` — e.g. status updates to an assigned owner/manager, audit-log entries to an admin dashboard, transaction confirmations). The distinction between *notification events* (something happened — client re-fetches) vs *event-carried-state* (the new state is in the push payload) changes the consistency contract and the payload-size design.
- The **connection and fan-out topology**: how many concurrent connections at peak, the subscriber-to-event-source fan-out ratio, and whether connections are per-user or per-resource (a record's owner vs all records in a dashboard). Fan-out is a sensitivity point: 1 event → N connections → N writes is where the topology fallacy (`../shared/fallacies-and-tradeoffs.md` §5) bites — topology changes (reconnections, pod restarts, node rebalances) during a fan-out must be designed, not assumed away.
- The **capacity model seed**: the `peak concurrent WebSocket/SSE connections`, the `event rate at peak` (events/sec into the fan-out), and the `average payload size` — needed to prove the tail-latency budget. These are inputs from the utility tree; if unstated, surface as open driver questions and do not assume numbers (`project-architecture-config.md` locked rule: never invent an NFR).
- **Preconditions**: the utility tree exists with measurable freshness and tail-latency scenarios; `project-architecture-config.md` read; `workflows/20-data/streaming-data.md` consulted for the upstream pipeline (backpressure, delivery semantics, and the event-bus topology the real-time path consumes from); `workflows/40-cross-cutting/performance-capacity.md` consulted for the per-hop latency budget (the real-time path is one leg of the total response-measure budget); `workflows/30-integration/async-messaging.md` consulted when a broker is in scope.

## Oracle (source of truth)

The driver-traced quality-attribute scenarios from the utility tree, proven against a capacity model and a connection-topology design — never the writer's claim that it "is real-time" or "scales to N connections."

**hard:**

- **Tail-latency budget met by a capacity model (the math, not an assertion).** The real-time path's tail-latency driver (`p99 ≤ M ms from event-commit to client-receipt`) is decomposed into a per-hop budget — source store/broker → push server → connection write → client ACK — and the arithmetic (Little's Law applied to the connection-write queue, serialization cost, per-hop network round-trip from the powers-of-ten table) closes at peak connection count and peak event rate. A scenario "met" by bare assertion with no capacity model **blocks** (`response_measures_met` hard oracle from `../shared/quality-attribute-rubric.md`). The model gates on **p99** (never mean) and accounts for **fan-out tail amplification**: where one event fans out to N connections, the server-observed latency for the last connection write in the fan-out is the deterministic tail — design the fan-out topology so this stays inside the budget. Cross-reference: `../40-cross-cutting/performance-capacity.md` for the Little's-Law / M/M/1 method and fan-out tail analysis.

- **Connection state, fan-out topology, and reconnection designed (the topology fallacy).** Fallacy 5 — topology doesn't change — is the cardinal failure mode of real-time systems: nodes restart, connections drop, pod-to-pod routing changes, load-balancers time out idle WebSockets. The design must account for: (a) *sticky routing or connection-state sharing* — if the push server is horizontally scaled, the mechanism for routing inbound events to the correct connection-holding replica is explicit (e.g. a shared pub/sub channel per user, a consistent-hash load-balancer, a Redis pub/sub fan-out layer); (b) *client reconnection with exponential backoff + jitter* — clients reconnect automatically on disconnect; the server tolerates reconnect storms (a cold-start of K clients all reconnecting simultaneously is a load spike the capacity model must include); (c) *missed-event recovery on reconnect* — the design states whether a reconnecting client replays missed events (cursor/offset from a durable log) or re-fetches state (simpler, correct when event-carried-state is not required). An architecture with stateful connections and no topology-change handling **fails `fallacies_honored`**.

- **Backpressure prevents unbounded buffering.** If the event rate into the fan-out exceeds the connection-write capacity (a slow client, a slow network path, or a temporary spike), the overflow strategy is explicit and bounded: (a) per-connection send-buffer cap with a drop-and-reconnect policy; (b) upstream back-pressure signal to the event source (e.g. TCP flow-control for WebSocket; bounded async queue between the event ingestion path and the connection writer); (c) load-shedding (drop notifications, client re-fetches on reconnect) as the graceful-degradation path. An architecture that silently grows in-process buffers without bound violates Fallacy 3 (bandwidth is not infinite) and the `failure_modes_defined` hard oracle. Cross-reference: `../20-data/streaming-data.md` §Backpressure for the upstream pipeline contract.

- **Consistency / latency tradeoff chosen explicitly per write path.** The CAP/PACELC point is stated: push channels are inherently on the Latency side (PA/EL) — the client may hold a slightly stale view between the push event and the client read. The design declares: (a) whether the push payload carries the new state (event-carried state — client can render immediately, payload must be self-consistent) or a notification-only token (client re-fetches — strongly consistent but adds a round-trip latency); (b) the **bounded staleness window** (the maximum lag between a write committed at the source store and a client receiving the push, expressed as a p99 duration under peak event rate); (c) what happens on a partition (`CAP` theorem: during a broker or push-server partition, does the system drop pushes and rely on client re-fetch, or queue and deliver with a freshness penalty?). "It's real-time" with no stated consistency model **blocks** (`consistency_stated` hard oracle).

- **Failure mode defined on every hop.** Every cross-process call in the real-time path — event source → push server → client — has a bounded timeout, a retry policy (with backoff+jitter where safe), and a fallback. Specifically: (a) the WebSocket/SSE server-side write has a per-connection write timeout (not infinite); (b) the upstream event subscription (broker consumer or a database change-stream cursor) has reconnect logic with bounded retry; (c) the fallback for a failed push is explicit — typically: drop the push, client polls or re-fetches on the next interaction (this is the acceptable graceful-degradation path for non-critical notifications). A real-time path with an unbounded write or an infinite consumer reconnect loop **blocks** (`failure_modes_defined`).

- **Trust boundary secured on the push channel.** A WebSocket or SSE connection crosses a trust boundary: the initial HTTP upgrade or SSE GET carries the auth token; the server validates it before establishing the connection; per-connection authorization is enforced (a manager's connection only receives events for their assigned records — not a global broadcast). Secrets (JWT, session tokens) are never logged. Per-connection STRIDE: spoofing (unauth upgrade), information disclosure (over-broad fan-out), denial of service (connection exhaustion). Cross-reference `workflows/40-cross-cutting/security-architecture.md`. A push channel with no per-connection authz is a `critical` security finding.

- **Driver traceability complete.** Every component of the real-time path — the push server, the fan-out mechanism, the connection-state store (if any), the reconnect strategy, the client library — traces to a quality-attribute scenario it serves. A component with no driver is accidental complexity; flag it, do not design it.

**soft:**

- **Mechanism matches the freshness driver (SSE/long-poll before WebSocket if sufficient).** WebSocket is a full-duplex, stateful, long-lived connection: it is the right answer when the client sends frequent messages server-ward (e.g. collaborative editing, bidirectional control signals) or when the latency budget is sub-100ms p99. SSE is a unidirectional server-push over HTTP/1.1 or HTTP/2: simpler to proxy (works through standard HTTPS reverse proxies without WebSocket upgrade issues), automatically reconnects, and fits any scenario where the traffic is server→client only. Long-poll is the simplest: no persistent connection, no proxy complications, correct for freshness windows ≥ 5–10s. Proposing WebSocket for a unidirectional notification scenario with a 5s freshness driver is accidental complexity — the evaluator penalizes it (YAGNI; `../shared/fallacies-and-tradeoffs.md` Flexibility↔Simplicity). The chosen mechanism must be the simplest that meets the stated freshness and tail-latency scenario.

- **Design is honest about p99 and p999, not just the average.** The capacity model presents the full latency distribution — p50, p95, p99, and (for high-connection-count systems) p999 — with the fan-out tail explicitly named. A design that presents only the average or "typical" delivery latency while silently ignoring the tail at high fan-out is incomplete craft. The evaluator penalizes any capacity model that omits the tail or that sweeps fan-out amplification under "it depends on the network."

- **Connection lifecycle is boring and resilient by default.** Heartbeat/ping-pong, per-connection idle timeout, and graceful drain on rolling deploys are designed in from the start (not bolted on). The design names the idle-connection timeout value and the rationale (keep-alive vs load-balancer timeout vs cost-of-idle-memory). A design that omits these is operability-incomplete.

## Standards & techniques

- **WebSocket (RFC 6455)** — full-duplex, long-lived TCP connection upgraded from HTTP. State: per-connection state on the server; requires sticky load-balancing or a shared pub/sub backplane (e.g. Redis pub/sub, a broker consumer group with per-node fan-out) for horizontal scale. Proxy/firewall support: widely supported but requires HTTP Upgrade; some corporate proxies block. Use when bidirectional or sub-100ms p99 is a stated driver.
- **SSE (Server-Sent Events, HTML Living Standard)** — unidirectional server→client stream over HTTP/1.1 keep-alive or HTTP/2 multiplexed streams. Built-in reconnect (`Last-Event-ID` header for cursor-based replay). Simpler proxy story than WebSocket. Use for server→client notification when the freshness driver is ≥ 1s p99 and the client has no need to send messages on the same channel. Prefer SSE over WebSocket for a unidirectional status-notification use case unless a driver demands otherwise.
- **Long-poll** — client issues a request; server holds it open until an event is available or a timeout fires, then responds; client immediately re-issues. No persistent connection; works through all HTTP/1.1 proxies. Correct for freshness windows ≥ 5–10s where a broker-less implementation is preferred. The fallback tier when SSE is operationally inconvenient (e.g. shared nginx config that can't set `proxy_buffering off`).
- **Event streaming at scale — partitioning and consumer groups.** When event volume demands horizontal scale on the push-server side, the fan-out is distributed via a broker (Kafka, Redis Streams, or a cloud-native equivalent): (a) a **consumer group** partitions event streams across push-server replicas — each replica owns a partition and handles fan-out to the connections it holds; (b) **partition count** must be ≥ the expected push-server replica count to avoid idle replicas; (c) **consumer lag** (the delta between the latest event in the partition and the last consumed offset) is a RED metric for the real-time path — it must be bounded to meet the freshness scenario. AKF scale cube X-axis: more push-server replicas + more partitions. Y-axis: separate partitions by event type (status updates vs audit-log entries vs transaction confirmations). Cross-reference: `../20-data/streaming-data.md` for partitioning, consumer groups, and backpressure contracts; `workflows/30-integration/async-messaging.md` for broker selection and delivery semantics.
- **Backpressure in push systems.** Pull-based consumers (a broker poll loop) provide backpressure natively — the consumer fetches at its own pace, and the broker holds the queue. Push-based delivery (WebSocket write, SSE write) requires explicit flow control: (a) TCP send-buffer fills if a client is slow → the OS blocks the write → the server-side coroutine/thread blocks → the event-processing loop stalls. Solution: a bounded async queue between event ingestion and the connection writer; if the queue is full, drop-and-notify (the client re-fetches on reconnect) or apply backpressure upstream. (b) For high-connection fan-out, use a non-blocking I/O model (async/event-loop server per the project's stack, or a dedicated async push tier) so one slow client does not block the event loop for all others.
- **Stream processing — windowing and watermarks.** For real-time aggregations pushed to dashboards (e.g. "orders completed this hour," "average cycle time over the last 5 minutes"): tumbling windows (non-overlapping, fixed size — suitable for throughput metrics), sliding windows (overlapping — heavier, suitable for moving averages), session windows (activity-gap based — natural for a business entity's lifecycle spans). Watermarks (`max_event_time_seen − max_lateness_bound`) drive window close; the max_lateness_bound is derived from observed source delay, not invented. Cross-reference: `../20-data/streaming-data.md` §Windowing and Watermarks for the full treatment. This flow owns *pushing the window result to the client*; streaming-data owns producing it.
- **Exactly-once vs idempotent delivery to the client.** At-least-once delivery to the client (push may be duplicated on reconnect) combined with an idempotent client render (event ID dedup, event-carried-state rendering by version/sequence number) is the correct default — simpler, lower latency, and sufficient for UI notification use cases. Exactly-once-to-the-client machinery (client-side ACK + server-side retention + re-delivery tracking) is warranted only when a missed or duplicate notification causes a correctness failure in the client state that cannot be recovered by a re-fetch. State the delivery contract explicitly; "it's real-time" without a stated delivery guarantee blocks. Cross-reference: `../20-data/streaming-data.md` §Delivery and Processing Semantics for at-least-once + idempotency design.
- **Tail-latency engineering — p99/p999.** The tail at high fan-out is deterministic: with K concurrent connections and a p99-single-write of T ms, the expected p99 of the fan-out batch is approximately `T + HOL_blocking_overhead`. Hedged writes (writing to the first N/2 connections while the second N/2 are in flight) reduce the tail at the cost of ordering complexity. Tighter per-connection timeouts (with a drop-and-reconnect fallback) bound the tail at the cost of a higher reconnect rate. The design names which lever it pulls and what it costs. Never model only the average. Cross-reference: `../40-cross-cutting/performance-capacity.md` §Tail latency & fan-out amplification.
- **The 8 fallacies applied to real-time paths** (`../shared/fallacies-and-tradeoffs.md`): Fallacy 1 (network is reliable) → every connection can drop; reconnect is the normal path, not the exception. Fallacy 2 (latency is zero) → each push adds a network hop; budget it. Fallacy 3 (bandwidth is infinite) → per-connection send-buffer is finite; backpressure required. Fallacy 5 (topology doesn't change) → push servers restart, pods reschedule, LB sessions reset; the design accounts for this explicitly. Fallacy 4 (network is secure) → the push channel is a trust boundary; authn/authz on upgrade + per-event authorization enforcement.
- **CAP/PACELC for real-time serving.** The push path is inherently PA/EL: during a partition or push-server restart, the system favors Availability (it keeps serving the HTTP API) and Latency (it does not block writes waiting for push-delivery ACK). The consistency model is eventual — the client may hold stale state between the last push and the next re-fetch. The design names this point and confirms it is acceptable given the utility tree's consistency driver priority (which for many internal business tools is typically lower priority than availability and latency — confirm against `project-architecture-config.md`). A design that claims strong consistency on the push path while running an async fan-out is inconsistent — block it.
- **MADR ADR format** (`../shared/adr-template.md`) for the mechanism decision (WebSocket vs SSE vs long-poll), the fan-out topology decision (direct vs broker-mediated), the connection-state design (sticky vs shared backplane), and the delivery-contract decision (at-least-once + idempotent vs exactly-once). Each ADR records the rejected alternatives and what they cost — a recommendation with only upside is incomplete.

## Assertions & exit gate

- Every client-facing real-time or low-latency path has a freshness or tail-latency driver from the utility tree; any path proposed without one is flagged as accidental complexity (`ARCH-RT-NNN`, `oracle:"soft"`, severity `minor`).
- The real-time delivery mechanism (WebSocket / SSE / long-poll) matches the freshness and directionality driver — SSE or long-poll for unidirectional server→client with freshness ≥ 1s; WebSocket only when bidirectional or sub-100ms p99 is a stated driver. Mechanism over-engineering (WebSocket where SSE suffices) is a `minor` soft finding.
- The tail-latency budget (p99) is proven by a capacity model: Little's Law applied to the connection-write queue at peak connection count and event rate; fan-out tail amplification (max-of-N or explicit fan-out batch analysis) is modeled, not assumed away. A model on the mean, or without fan-out analysis, blocks.
- Connection state, topology change, and reconnection are designed: sticky routing or a shared fan-out backplane is specified; client reconnect with exponential backoff+jitter is specified; missed-event recovery on reconnect (replay vs re-fetch) is named. Absence of topology-change handling is a `major` `availability` finding.
- Backpressure is designed on the push path: per-connection send buffer is bounded; overflow strategy is explicit (drop + reconnect / upstream backpressure signal / load-shed); no unbounded in-process queue between the event source and the connection writer.
- The consistency/latency tradeoff is explicitly chosen: event-carried-state vs notification-only is stated; the bounded staleness window (p99 from event-commit to client-receipt at peak) is named; the CAP/PACELC point (PA/EL) is declared.
- Every hop in the real-time path has a bounded timeout and a defined failure/fallback: push-server write timeout; upstream subscription reconnect; client-side fallback (re-fetch on reconnect). Absence of any bounded timeout on the push-write path is a `major` `performance` finding.
- The push channel trust boundary has per-connection authn (token validated on upgrade) and per-event/per-connection authz (connections receive only events scoped to their role/resource); STRIDE is covered per trust boundary.
- Every significant decision is an ADR with context · ≥2 options · decision · consequences (positive AND negative) and a traced `Driver ref` to a `QAS-*` in the utility tree.
- All diagrams-as-code compile and render without error; the Container view labels every connection with mechanism + sync/async + protocol; the Dynamic view shows the end-to-end fan-out for the key tail-latency scenario.
- The capacity model states the saturation point — the peak-connection or peak-event-rate at which the push server's async write queue saturates — and the headroom between the design load and that saturation point.

**Gate:** hard oracles green (tail-latency budget proven by capacity model at p99 + fan-out · connection-state + topology + reconnection designed · backpressure bounded · consistency/latency tradeoff chosen and CAP/PACELC point stated · failure modes defined on every hop · push channel trust boundary secured with authn+authz + STRIDE · ADRs complete · diagrams compile · driver traceability complete) AND (design) rubric mean ≥ 0.8 (`../shared/quality-attribute-rubric.md`).
