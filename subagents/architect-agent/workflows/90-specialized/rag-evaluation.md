# rag-evaluation

**Group:** 90-specialized · **Runs as:** subagent: ../.claude/agents/solution-architect.md · **Mode:** audit + design · **Default model:** sonnet

## Purpose
Specify and run the RAG-pipeline evaluation battery so retrieval quality and generation faithfulness are **measured oracles, not assumed properties**. This flow owns: the RAGAS-style metric set (Faithfulness, Context Precision, Context Recall, Response/Answer Relevancy, and Tool-Call Accuracy for agentic designs); the construction and maintenance of the golden evaluation set; the CI-gated promotion policy; and the localizable failure diagnostic (retrieval failure vs generation failure vs tool-use failure are distinct — the design must expose which component failed). The governing question: *does this RAG pipeline demonstrably meet its correctness and grounding drivers, with a measured threshold that gates promotion and a fallback for low-confidence answers on any decision path?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack and data stores (per the project's stack — any vector store is a new component requiring a driver-justified ADR); the compliance regime (per the project's data-sensitivity classification — e.g. PII or other regulated data — all evaluation inputs must be de-identified); the scale/SLO targets; the team topology; and where the architecture description / ADR log / diagrams live (`architecture/<slug>/`).
- The **utility-tree drivers** from `00-drivers/quality-attribute-scenarios.md` — specifically: a **correctness / quality scenario** (faithfulness ≥ θ; answer relevancy ≥ θ; context recall ≥ θ — each ending in a measurable threshold, traced to a business goal or NFR; if the source-of-truth docs state no threshold, surface it as an open driver question — **never invent a number**), a **latency scenario** (end-to-end including retrieval + rerank + generation), a **cost scenario** (per-query token + retrieval cost), and a **safety/PII scenario** (no unredacted PII egressed to the model provider). A flow with no quality-threshold driver from the utility tree downgrades scope and says so explicitly.
- The **RAG system description** to evaluate or design: the chunking strategy, embedding model, vector store, retrieval topology (top-k, optional rerank), context-assembly logic, orchestration layer, and the LLM / generation step. For an agentic design, also: the tool definitions, the tool-call routing logic, and the tool-call accuracy target.
- A **golden evaluation set** (or a specification for building one): question–context–ground-truth-answer triples for reference-based metrics (Faithfulness, Context Precision/Recall) and question–answer pairs for reference-free metrics (Response Relevancy). If no golden set exists, the design step produces the specification and seed for the controller to build. An eval without a golden set is an open driver question, not a blocked flow.
- Preconditions: the AI-system drivers are read from `00-drivers/quality-attribute-scenarios.md`; the source-of-truth docs (BRD / PRD / understanding log) named in `project-architecture-config.md` are read; the RAG system description or the in-scope design scope is confirmed.

## Oracle (source of truth)
The prioritized quality-attribute scenarios from the utility tree, made concrete as RAGAS-style metric thresholds, structural pipeline checks, and CI-gate contracts. Split:

**hard:** (deterministic — measurable, traceable, or structurally checkable)
- **The RAG design declares its eval battery and CI gate.** The pipeline documentation names: (a) which RAGAS-style metrics it is evaluated against (at minimum: Faithfulness + Context Precision + Context Recall + Answer Relevancy; Tool-Call Accuracy added for any agentic/tool-use path); (b) a **project-calibrated threshold per metric** traced to the utility tree — e.g. "Faithfulness ≥ 0.85 (traced to QAS-CORRECTNESS-01: ≤ 5% hallucinatory answers on end-user-facing queries)"; (c) the golden evaluation set (or a tracked open question to build one); and (d) a CI pipeline step that runs the battery on the golden set and blocks model/prompt promotion on a threshold miss. A RAG design with no declared metric set, no threshold, or no CI gate is a hard finding (`oracle:"hard"`). Cross-ref `../shared/quality-attribute-rubric.md` → *RAG grounded* hard oracle and the AI-system hard oracles table.
- **Retrieval failure and generation failure are separable.** The eval battery reports retrieval metrics (Context Precision, Context Recall) and generation metrics (Faithfulness, Answer Relevancy) independently. A single blended score that hides the component is a hard finding — the pipeline must expose which layer (retrieval vs generation vs tool-use) caused a quality regression, or fixing it is blind. This maps directly to the *Failure mode defined* oracle: the failure path must be localizable and actionable, not opaque.
- **A low-confidence / failure fallback exists on every decision path.** For any answer surface where the RAG pipeline output influences a human or system decision (a recommendation, a record classification, a workflow trigger): when the pipeline produces a low-confidence answer (Faithfulness < threshold, any guardrail trigger, or a generation/retrieval error), there is a **defined, bounded response** — a typed refusal, a human-in-the-loop handoff, a cached/deterministic fallback — **never** a silent hallucinated answer or an unbounded wait. The absence of a fallback on a decision path is a hard finding (`oracle:"hard"`). Cross-ref `../90-specialized/genai-risk-assessment.md` (Confabulation risk dimension) and `../90-specialized/ml-ai-system-architecture.md` (fallback oracle).
- **Thresholds are project-calibrated and traced.** Each metric threshold is derived from a stated business goal or NFR in the utility tree (e.g. "end-user-facing decisions must be grounded — internal testing showed a Faithfulness score of 0.85 correlates with < 5% flagged errors in human review"). A threshold that was invented rather than traced is an open driver question, not a gate. Where no threshold is stated by the source-of-truth docs, the design documents it as **open** and proposes a calibration protocol (annotate N samples, measure human-rated groundedness vs the metric, set the threshold at the observed breakpoint) — the open question blocks promotion until the calibration is done.
- **Tool-Call Accuracy evaluated for agentic designs.** Any RAG pipeline that routes to tools (document lookup, a domain-data API, calculation functions) must declare a **Tool-Call Accuracy** metric on the golden set: the fraction of queries where the pipeline calls the correct tool(s) with correct arguments. Incorrect tool selection is a distinct failure mode from retrieval or generation failure — it must be tracked separately. An agentic RAG design with no Tool-Call Accuracy target is a hard finding. Cross-ref `../30-integration/agent-interop-protocol.md` (tool-call contract) and `../90-specialized/agentic-threat-model.md` (tool-poisoning).
- **Eval runs on de-identified data under the compliance regime.** Evaluation inputs (questions, retrieved contexts, ground-truth answers) must not contain unredacted sensitive data (personal identifiers, financials, or whatever the project's data-sensitivity classification covers) — the golden set is built on de-identified or synthetic data under the project's compliance regime. A golden set that contains live sensitive data sent to an external eval LLM is a hard finding (`oracle:"hard"`), same class as the unredacted-PII-egress oracle in `../shared/quality-attribute-rubric.md`.
- **Reference-based vs reference-free split is explicit.** The eval design states which metrics require a ground-truth reference answer (Context Precision, Context Recall, Faithfulness — reference-based) and which do not (Answer Relevancy — reference-free, using the LLM-as-judge pattern). The LLM judge for reference-free metrics must be a different model or model family from the generator to avoid self-preference inflation (per `../shared/evaluation-method.md` → *Evaluator discipline & debiasing*). Using the same model to judge its own generation is a hard finding.

**soft:** (craft — rubric-scored by `architecture-evaluator` after hard oracles are green)
- **Thresholds are project-calibrated starting points, not cargo-culted absolutes.** The "right" Faithfulness threshold depends on the domain, the stakes of an error, and the tolerance of the user population — a 0.90 threshold is not universal; 0.75 may be correct for low-stakes suggestions, 0.95 for a decision-path trigger. The soft oracle checks that the thresholds are *argued* (what does missing the threshold mean for the user? what is the cost of a false negative?), not just copied from a benchmark.
- **The eval set covers the real query distribution.** The golden set samples from the actual or anticipated query distribution (real end-user question types, real source-document structures, edge cases like ambiguous queries, short-context misses, cross-document reasoning) — not just easy, clean examples. A golden set skewed toward easy queries gives an optimistically biased eval that masks production failures.
- **Eval coverage per failure mode.** The golden set includes examples designed to surface the known RAG failure modes: (a) **retrieval failure** — the correct context is not retrieved (tests Context Recall); (b) **faithfulness failure** — the answer contradicts or hallucinates beyond the retrieved context (tests Faithfulness); (c) **relevancy failure** — the answer is irrelevant or off-topic for the query (tests Answer Relevancy); (d) **precision failure** — irrelevant contexts are retrieved and pollute generation (tests Context Precision); (e) **tool-routing failure** — the wrong tool or wrong arguments are selected (tests Tool-Call Accuracy, for agentic paths). A golden set that only tests the happy path is a soft finding.
- **Conceptual integrity of the eval harness.** One consistent eval framework (e.g. RAGAS, a project-standard wrapper, or a lightweight implementation of the same metric definitions) runs all metrics — not a different ad-hoc script per feature or per team member. Eval results are version-controlled alongside the golden set and the model/prompt version they evaluated.

## Standards & techniques
- **RAGAS (Retrieval-Augmented Generation Assessment)** — the primary metric framework. Four core metrics, each with a defined computation:
  - **Faithfulness** — the fraction of claims in the generated answer that are entailed by (can be inferred from) the retrieved context. A reference-based metric: requires the retrieved context and the generated answer; uses an LLM judge to decompose the answer into atomic claims and verify each against the context. Faithfulness = (claims supported by context) / (total claims). A low score signals hallucination — the generator is adding information not grounded in retrieval.
  - **Context Precision** — the fraction of the retrieved context chunks that are relevant to the query. Measures retrieval specificity / signal-to-noise: are the chunks returned actually needed, or is there retrieval noise that can mislead generation? Computed as a ranked precision (weights higher-ranked relevant chunks more heavily — average precision).
  - **Context Recall** — the fraction of the ground-truth answer's information that is present in the retrieved context. Requires a reference answer. Measures whether retrieval is finding all of the relevant content needed to answer the query completely. A low score signals the retriever is missing required context — the generation step cannot compensate for what was never retrieved.
  - **Answer/Response Relevancy** — the degree to which the generated answer addresses the question (is it on-topic and complete?), evaluated without a reference answer using an LLM to generate back-questions from the answer and measure semantic alignment with the original query. A reference-free metric: does not require ground-truth, but requires a separate judge model.
  - **Tool-Call Accuracy** (agentic extension) — for pipelines with tool-use steps: the fraction of queries where the correct tool(s) are called with correct arguments, evaluated against the golden set's annotated tool call labels. Distinguishes routing errors (wrong tool selected) from argument errors (right tool, wrong parameters).
- **Reference-based vs reference-free evaluation** — the principled split. Reference-based metrics (Faithfulness, Context Precision, Context Recall) compare the pipeline output against a ground-truth signal (the golden set's reference answers or labeled relevant chunks) — they are higher-fidelity oracles but require a curated golden set. Reference-free metrics (Answer Relevancy, some fluency/coherence checks) use the LLM-as-judge pattern without a reference — they scale without annotation but carry the LLM-judge failure modes (style bias, self-preference, position bias; per `../shared/evaluation-method.md` → *Evaluator discipline & debiasing*). Both types are needed: reference-based for the hard CI gate; reference-free for coverage where annotation is impractical.
- **LLM-as-judge discipline** — applied to any RAGAS metric that uses an LLM to score. Per `../shared/evaluation-method.md`: the judge must not be the same model (or model family where possible) as the generator; style-normalize the candidate before judging; score per-criterion, not holistically; anchor soft scores to typed evidence (the specific claim, the specific context chunk). The self-preference inflation for Faithfulness (the generator judging its own claims against its own context) is a known failure mode — use a cross-family judge or strip identifying markers.
- **The retrieval-then-generation failure mode taxonomy** — the two-stage RAG pipeline has two independently failing components, and the failure modes compose:
  - *Retrieval miss* (low Context Recall) → the generator produces a faithful-but-incomplete answer (or hallucinates to fill the gap).
  - *Retrieval noise* (low Context Precision) → the generator is polluted with irrelevant context, degrading faithfulness and relevancy.
  - *Generation hallucination* (low Faithfulness, high Context Recall) → good retrieval, but the generator fabricates claims beyond what the context supports.
  - *Irrelevant generation* (low Answer Relevancy) → the generator produces an on-context but off-question answer (e.g. answers a related but different question).
  - *Tool-routing error* (low Tool-Call Accuracy) — agentic only: the orchestrator calls the wrong tool or passes incorrect arguments, causing the retrieval or action step to be wrong before generation.
  Localizing which failure mode is firing directs the fix to the right component (retrieval tuning, reranker, prompt tuning, model upgrade, tool-definition revision) — without this split, all fixes are blind.
- **Golden evaluation set construction** — the eval set is a version-controlled asset: question–context–reference-answer triples (for reference-based metrics) and question–tool-annotation pairs (for Tool-Call Accuracy). Construction protocol: (a) sample from the real or anticipated query distribution; (b) include adversarial examples per failure mode (retrieval-miss, retrieval-noise, hallucination triggers, off-topic queries, tool-routing edge cases); (c) de-identify all PII under the compliance regime; (d) version alongside the model/prompt/index version it was built against; (e) review and annotate by a domain expert, not just an LLM auto-labeler (the LLM-generated reference answer is a soft start, not a hard ground truth). The set grows incrementally as production failure modes are discovered.
- **CI-gated promotion policy** — the eval battery runs as a non-optional CI step before any model version, prompt version, or retrieval-index change is promoted to staging or production. The gate: if any declared metric falls below its calibrated threshold, the promotion is blocked and a finding is emitted to the controller with the component diagnosis (which metric failed + which failure mode). Shadow / canary rollouts (`../90-specialized/ml-ai-system-architecture.md` → MLOps) sample live traffic for online eval signals (sampled Faithfulness + Answer Relevancy on production queries, de-identified) to detect regressions the golden set did not cover. Cross-ref `../50-infrastructure/cicd-architecture.md` for pipeline wiring.

## Step sequence
- **audit:** Frame the RAG quality drivers (read from the utility tree; flag any quality/faithfulness/recall/relevancy driver with no response measure as an open driver question) → walk the existing/proposed RAG pipeline against each hard check: (1) Is an eval battery declared with named metrics and thresholds? (2) Are the thresholds traced to a driver/NFR, or invented? (3) Is the CI gate wired to block promotion on a miss? (4) Are retrieval metrics (Precision, Recall) and generation metrics (Faithfulness, Relevancy) reported separately so failures are localizable? (5) Is a fallback defined on every decision path for low-confidence / error outputs? (6) For agentic designs: is Tool-Call Accuracy declared and tracked? (7) Is the LLM judge a different model/family from the generator? (8) Does the golden set exist, is it version-controlled, and is it de-identified? → emit findings + a risk register, read-only, author nothing, edit no source. Group findings by type then severity (critical → cosmetic).
- **design:** **Frame** (confirm the utility-tree quality/correctness scenarios + the RAG subsystem's in/out scope; surface any unstated thresholds as open driver questions — read-only) → **Explore** (≥2 genuinely divergent eval-architecture directions as concrete specs, e.g.: *RAGAS fully-automated CI battery with a cross-family judge* vs *human-in-the-loop annotation-driven golden set with LLM pre-scoring* vs *lightweight reference-free online-only eval*; each direction includes the metric set, the golden-set construction protocol, the CI-gate wiring, the judge model choice, the fallback definition, and the compliance handling for the golden set; all read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each vs the utility tree + hard oracles per `../shared/evaluation-method.md`: does it meet the declared quality threshold? is the failure localized? does the judge avoid self-preference? is the golden set de-identified and representative? names sensitivity points — *Faithfulness score is sensitive to judge-model choice; Context Recall is sensitive to chunk size and top-k; Tool-Call Accuracy is sensitive to tool-definition clarity* — and tradeoff points — *automated LLM judge is fast and cheap but carries style+self-preference bias; human annotation is high-fidelity but slow and expensive; reference-free online eval scales but can't catch hallucinations the golden set would*; order-swapped to kill position bias) → **Document** (ONE writer authors the artifacts: the eval-battery spec naming metrics + thresholds + judge model + golden-set spec + CI-gate wiring; the ADRs for the key decisions — metric-set choice, judge-model choice, golden-set construction protocol, promotion-gate policy; the fallback design per decision path; the compliance handling for evaluation data; references the capacity/cost model for eval token spend) → **Verify** (all declared metrics are computable against the golden set; the CI gate is specified with blocking logic; the judge-model is cross-family from the generator or style-markers are stripped; the fallback covers every decision path; the golden set spec is de-identified; ADRs complete; the evaluator re-scores from the artifacts) → loop ≤ 10 or pass.

## Assertions & exit gate
- Every quality/correctness driver from the utility tree traces to a declared RAGAS-style metric; every metric has a calibrated threshold traced to a driver/NFR (or the threshold is explicitly open, with a calibration protocol specified).
- Retrieval metrics (Context Precision, Context Recall) and generation metrics (Faithfulness, Answer Relevancy) are reported independently — the failure mode is localizable to the retrieval layer or the generation layer without ambiguity.
- A fallback is defined and bounded for every RAG output surface on a decision path (no silent hallucinated answer, no unbounded wait on error or low-confidence).
- For agentic designs: Tool-Call Accuracy is declared with a threshold and is tracked separately from retrieval and generation metrics.
- The CI gate is specified: the eval battery blocks promotion when any declared metric misses its threshold; the gate runs on the version-controlled golden set.
- The LLM judge is a different model or model family from the generator; or, where the same family is unavoidable, style markers are stripped and a soft warning is emitted.
- The golden evaluation set is version-controlled, de-identified under the compliance regime, and covers the known failure modes (retrieval miss, retrieval noise, generation hallucination, irrelevant generation, tool-routing error where applicable).
- Thresholds are argued (tied to a business consequence of missing them) — not cargo-culted from a benchmark.
- (design) Diagrams compile; every ADR (metric-set choice, judge-model choice, golden-set protocol, promotion-gate policy) has context · decision drivers · ≥2 considered options · decision · consequences (positive AND negative) per `../shared/adr-template.md`.
- **Gate:** hard oracles green (eval battery declared with named metrics + traced thresholds · CI gate specified and blocking · retrieval/generation failure separable · fallback on every decision path · Tool-Call Accuracy for agentic paths · LLM judge cross-family or style-normalized · golden set de-identified + version-controlled · diagrams compile · ADRs complete) AND (design) rubric mean ≥ 0.8 (thresholds argued · golden set covers real distribution + failure modes · one coherent eval harness · reference-based/reference-free split explicit and discipline-enforced).

## Output
Write `artifacts/rag-evaluation/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (use the full finding `type` enum; for a RAG eval audit the load-bearing types are: `performance` (latency including retrieval + rerank + generation end-to-end) · `observability` (missing or un-alertable eval metrics, absent online drift signal — an eval battery with no CI wiring is an `observability` finding, not just a process gap) · `data` (golden set absent, not version-controlled, or not de-identified; retrieval pipeline undesigned) · `integration` (tool-call contract not evaluated; judge-model same family as generator) · `security` (PII in the golden set; PII egressed to the judge model) · `compliance` (PII/HIPAA on the eval data path) · `availability` (no fallback on a decision path — a missing fallback is a hard availability finding, same class as an un-accepted SPOF) · `evolvability` (golden set not version-controlled alongside model/prompt version; no calibration protocol for threshold revision) · `risk` (unstated thresholds as open driver questions; uncalibrated thresholds) · `documentation` (missing ADR for metric-set / judge-model / gate-policy decisions); `oracle:"hard"` for: no declared eval battery, no CI gate, no fallback on a decision path, un-separable retrieval/generation failure, missing Tool-Call Accuracy on an agentic path, judge model same family as generator with no style normalization, PII in the evaluation data, an uncalibrated or invented threshold; `oracle:"soft"` for: golden set skewed toward easy examples, thresholds not argued against business consequences, reference-free metrics only (no reference-based gate)), plus the verification block for design mode (metric-set coverage · CI-gate specification · failure-localizability check · fallback coverage per decision path · judge-model provenance · golden-set compliance check · ADR completeness · rubric score). Authored artifacts for design mode land in `architecture/<slug>/` — the eval-battery spec (`eval-battery.md`), the golden-set construction protocol (`golden-set-spec.md`), the ADRs (metric-set choice, judge-model choice, golden-set protocol, promotion-gate policy), and the fallback design per decision path. Open driver questions (unstated quality thresholds, no golden set, no cost ceiling on eval token spend) are listed explicitly for the controller — never silently assumed.

Cross-reference `../shared/quality-attribute-rubric.md` (the *RAG grounded* hard oracle and the AI-system hard oracles table that references this file), `../90-specialized/ml-ai-system-architecture.md` (the broader ML/AI architecture context: the index pipeline, the inference path, the drift signal, the fallback oracle, the provider SPOF), and `../90-specialized/genai-risk-assessment.md` (the Confabulation and Information-Integrity NIST AI 600-1 risk dimensions that the faithfulness + fallback oracles serve).

## Guardrails
Per `../shared/guardrails.md`: **propose & document — never write product code, never call a live model or retrieval endpoint, never build or modify the golden set itself** (the eval-battery spec and golden-set construction protocol are the artifacts; the controller builds the harness and the domain expert annotates the set). Ground every metric choice in a driver — no metric declared without a quality-attribute scenario it serves; if an eval threshold is unstated in the source-of-truth docs, it is an **open driver question, not a guessed number**. Treat all evaluation inputs as potentially sensitive: never log PII from retrieval contexts or generated answers in audit mode; assume the compliance regime applies to the golden set. One writer preserves the conceptual integrity of the eval harness (one framework, one judge protocol, one CI gate). Record the tradeoff and the rejected alternatives on every eval-architecture decision (automated LLM judge vs human annotation, reference-based vs reference-free, RAGAS vs bespoke metric) — never just the winner. The evaluator is a **separate skeptical agent** from the writer; it scores from the artifact facts (the declared metrics, the golden-set spec, the CI-gate wiring), not from the writer's prose claims.
