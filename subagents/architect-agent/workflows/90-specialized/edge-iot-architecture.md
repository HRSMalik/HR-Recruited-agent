# edge-iot-architecture

**Group:** 90-specialized  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Architect edge and IoT systems — deciding where compute lives (device, gateway, or cloud), how the device→gateway→cloud topology is structured, how the system behaves when connectivity is absent or intermittent, and how constrained-resource devices are provisioned, updated, and kept secure at fleet scale. Answers: *for each processing concern, is the edge-cloud placement justified by a latency, bandwidth, or sovereignty driver — and is intermittent connectivity a first-class design input, not an afterthought?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target, cloud region(s), compliance regime (data sovereignty, PII residency), team topology, constraints, and where the architecture description / ADR log / diagram source live.
- The architecture **drivers** — the prioritized quality-attribute scenarios (the utility tree from `../00-drivers/quality-attribute-scenarios.md`), specifically the scenarios that govern: end-to-end processing latency (the stimulus-to-response measure that determines whether edge placement is warranted), available uplink bandwidth per device/gateway class, data-sovereignty or regulatory residency constraints (which data may leave the local zone), device resource limits (flash, RAM, CPU class, battery budget), fleet size and growth trajectory, and OTA update reliability targets. A flow with no stated latency/bandwidth/sovereignty driver **cannot** justify an edge placement — downgrade scope and surface the missing driver as an open question.
- Target: the system/subsystem under design or review — the device tier, the gateway tier, the cloud tier, and the data flows between them; the relevant methods (compute-placement analysis, IoT topology mapping, store-and-forward design, OTA update architecture, device-identity threat model via STRIDE per `../40-cross-cutting/security-architecture.md`).
- Preconditions: the C4 Context view (or a topology diagram) is available so the device→gateway→cloud data flows are explicit; if it is absent the audit begins by reconstructing it from source/config before evaluating placements. The utility tree exists; if quality-attribute scenarios lack response measures (latency budget, bandwidth cap, sovereignty constraint), surface them as open driver questions — do not invent numbers.

## Oracle (source of truth)
The **quality-attribute scenarios with response measures** from the utility tree — a latency scenario that names the p99 budget across the full device-to-response path, a bandwidth scenario that names the uplink cap per device or gateway, a sovereignty constraint that names the data category and the jurisdiction it must not leave, a fleet-update scenario that names the maximum rollout window and the safe-failure rate — never "edge is better" or "cloud scales." The edge-cloud split is justified only when a traced driver demands it; moving processing to the edge without a driver is accidental complexity.

- **hard** (deterministic — traceable, measurable, or structurally checkable):
  - **Every edge-placement decision is justified by a named driver.** Compute placed at the edge (gateway or device) traces to a latency scenario (`p99 ≤ Xms end-to-end at the device, infeasible if the round-trip to cloud adds Yms`), a bandwidth scenario (`uplink capped at Z kbps; pre-aggregating at the gateway reduces payload by N×`), or a sovereignty constraint (`category-A telemetry must not leave jurisdiction J`). A placement with no driver trace is accidental complexity and blocks.
  - **Intermittent connectivity is designed for, not assumed away.** The design names the maximum disconnection window (the connectivity QAS response measure) and provides a **store-and-forward path** (local durable buffer at device or gateway), **idempotent sync** (replay-safe message IDs / sequence numbers / upsert semantics), and a **conflict-resolution policy** (last-write-wins / CRDT / server-authoritative with merge) for any write path that can diverge during a partition. A write path with no offline behaviour is a violation of fallacy #1 (*the network is reliable*, `../../shared/fallacies-and-tradeoffs.md`) and blocks.
  - **Device identity and mutual authentication exist at every trust boundary.** Each device and gateway carries a cryptographic identity (X.509 device certificate, TPM-backed key, or equivalent) and the cloud endpoint authenticates the device before accepting any data. An unauthenticated device-to-cloud channel violates fallacy #4 (*the network is secure*) and blocks (`../40-cross-cutting/security-architecture.md` STRIDE trust-boundary check). Per-device revocation is described.
  - **OTA update path is described and safe-failure-bounded.** The design names the update delivery mechanism (delta/full firmware, signed artifact, staged rollout), the rollback trigger (health-check failure within N minutes), the maximum blast radius of a bad update (% of fleet allowed to receive a staged wave before abort), and the update verification check (signature + hash). An OTA path with no rollback or no signature verification blocks.
  - **Constrained-resource limits are respected.** For each device/gateway class the design states the flash, RAM, CPU, and (where applicable) battery budget, and every component placed on that class is bounded to fit. A design that places an unbounded-memory cache or an unconstrained TLS stack on a microcontroller without a stated resource model blocks.
  - **Fallacies honored for every device-to-cloud hop.** The 8 fallacies apply with heightened severity at the edge: latency is never zero (multi-hop WAN, cellular, LoRaWAN), bandwidth is physically finite (LPWAN may be bytes/day), topology changes constantly (devices roam, sleep, and reboot), and the network is hostile (physically accessible hardware). Each is accounted for — not assumed away.
  - **Consistency model stated per write path.** Every write path that can diverge during a disconnected period declares its consistency model (eventual with CRDT merge / last-write-wins / server-authoritative reconciliation) and its mechanism. "It syncs when reconnected" with no conflict-resolution policy blocks (`../20-data/consistency-transactions.md`).
  - **Diagrams compile.** The device→gateway→cloud topology diagram (C4 Deployment view or equivalent) is diagrams-as-code (Mermaid / PlantUML / Structurizr) and renders without error per `../../shared/diagramming-standards.md`. Trust boundaries (device/gateway perimeter, cloud ingress) are drawn as boundary overlays.
  - **Every ADR is complete.** Each significant decision (edge-cloud split, gateway topology, store-and-forward mechanism, OTA update strategy, device-identity approach, conflict-resolution policy) is an ADR with context · ≥2 options · decision · positive and negative consequences per `../../shared/adr-template.md`.
- **soft** (craft, rubric-scored after hard oracles are green):
  - **Compute is placed where the driver demands — not everywhere.** Edge compute is applied precisely where a latency, bandwidth, or sovereignty driver justifies it; the rest stays in the cloud where it is cheaper to operate, easier to observe, and simpler to update. An architecture that pushes all processing to the edge because "edge is faster" is over-engineered and scores low on simplicity/right-sizedness.
  - **The offline path is first-class, not an afterthought.** The store-and-forward buffer, the sync protocol, and the conflict-resolution policy are explicit design decisions with their own ADRs — not a footnote that says "it retries." The system behaves correctly during disconnection, not just during a sunny-day network.
  - **Operability at fleet scale is designed in.** Device observability (telemetry, health signal, last-seen timestamp), fleet grouping for staged OTA rollout, remote diagnostics, and graceful degradation are structural properties, not bolt-ons.

## Standards & techniques
- **Compute placement analysis (edge-cloud split)** — the decision rule: place a workload at the edge *only* if a driver (latency, bandwidth, sovereignty) demands it; otherwise the cloud wins on operability, cost, and evolvability. The three canonical edge-placement justifications are: (1) **latency** — the round-trip to cloud exceeds the response-measure budget; (2) **bandwidth** — shipping raw data to cloud exceeds the uplink budget (pre-aggregate, filter, or compress at the edge); (3) **data sovereignty** — a regulatory or contractual constraint prevents certain data categories from leaving a jurisdiction. Each placement must name which justification applies.
- **IoT device→gateway→cloud topology** — the three-tier reference topology: *devices* (constrained, often battery-powered, LPWAN/BLE/Zigbee/Wi-Fi, limited compute/storage); *gateways* (protocol translation, local store-and-forward, pre-processing, always-on); *cloud* (fleet management, data ingestion pipeline, analytics, OTA distribution, long-term storage). The gateway is the critical decoupling seam — it absorbs intermittent device connectivity and translates heterogeneous device protocols to a cloud-facing standard (MQTT/AMQP/HTTP). A design that direct-connects every device to cloud skips the gateway seam and must justify that choice with a driver (e.g. devices have stable Wi-Fi and sufficient resources — document the decision).
- **Intermittent connectivity + store-and-forward + local-first/offline** — the offline-first pattern: write locally to a durable queue (on device or gateway), sync to cloud when the channel is available, replay idempotently. Key decisions: buffer size (device flash budget), eviction policy (FIFO / priority / TTL-based drop), sync protocol (MQTT QoS 1/2, CoAP with confirmable messages, a custom sequence-number protocol), and **conflict resolution** — last-write-wins (simple, loses concurrent writes), CRDT (mergeable without coordination, adds complexity), or server-authoritative reconciliation (server decides, requires a merge API). The choice is a one-way-door tradeoff (changing the conflict model after deployment is expensive); record an ADR.
- **Constrained-resource design** — MCU-class devices (Cortex-M, ESP32, nRF52): flash typically 512 KB–4 MB, RAM 64–512 KB, CPU ≤ 200 MHz, battery finite. Design controls: static allocation over heap (no unbounded alloc), firmware delta updates over full-flash OTA, TLS libraries sized for the target (mbedTLS / WolfSSL — not OpenSSL), protocol overhead minimized (MQTT over CoAP/LwM2M for ultra-low-power). Gateway-class nodes (Raspberry Pi / industrial Linux gateway) have fewer constraints but are still bounded compared to cloud — state the resource class per tier.
- **Fleet provisioning + device identity + OTA updates** — zero-touch provisioning (certificate injection at manufacture or first-boot via a bootstrap PKI); per-device X.509 certificates (or TPM-sealed keys) — never shared symmetric secrets across a fleet (a single-secret compromise revokes the entire fleet; that is a hard oracle failure). OTA: artifact signing (asymmetric — private key in an HSM, public key burned into firmware), staged rollout (canary → ring 1 → full fleet with abort on error-rate threshold), rollback on health-check failure within a bounded window. Cloud-side OTA distributor patterns: AWS IoT Jobs / Azure IoT Hub Device Update / custom MQTT topic hierarchy.
- **Protocol selection** — MQTT (publish/subscribe, QoS 0/1/2, small overhead, broad broker support — Mosquitto, AWS IoT Core, HiveMQ); CoAP (UDP-based, confirmable messages, low-overhead REST-like — good for LPWAN); AMQP 1.0 (heavier, enterprise broker compatibility); LoRaWAN / NB-IoT / LTE-M (wide-area, sub-kbps uplink, duty-cycle constraints — bandwidth fallacy is acute here); DDS (publish/subscribe with QoS for deterministic real-time — industrial/automotive). Protocol choice is a driver-traced ADR: latency budget × bandwidth × device power class × broker ecosystem.
- **The bandwidth + topology fallacies at the edge** — fallacy #2 (latency is zero) is violated at every WAN hop and especially on LPWAN (seconds of transmission time); fallacy #3 (bandwidth is infinite) is violated on LoRaWAN (payload capped ~50–250 bytes/message, duty-cycle < 1%); fallacy #5 (topology doesn't change) is violated constantly (devices roam, sleep, power-cycle, reboot into firmware updates). These are not edge cases — they are the normal operating mode. Every design decision must account for them explicitly (`../../shared/fallacies-and-tradeoffs.md`).
- **CAP/PACELC at the edge** — an edge node under partition (the common case) must choose: reject writes (consistency) or buffer locally and accept divergence (availability). The offline-first/store-and-forward pattern chooses **AP** during partition and reconciles on reconnection. State the choice per write path (`../20-data/consistency-transactions.md`).
- **STRIDE per IoT trust boundary** — four boundaries dominate: (1) device ↔ gateway (physical access risk: **S** device impersonation, **T** firmware tampering, **E** privilege escalation via serial/JTAG); (2) gateway ↔ cloud ingest (WAN: **I** data exfiltration, **D** spoofed gateway flooding the ingest endpoint); (3) OTA distribution ↔ device (supply-chain: **T** malicious firmware, **S** rogue update server); (4) cloud management plane ↔ operators (**E** over-broad fleet-admin grants). Each cell is controlled or accepted per `../40-cross-cutting/security-architecture.md`.
- **AKF scale cube** — the X-axis (horizontal clone) applies to the cloud ingest tier and the OTA distributor; the Z-axis (data partitioning) applies by device group / geography / tenant for multi-tenant IoT platforms. The edge tier does not scale horizontally by adding more gateways to one site — it scales by deploying more gateways geographically. State the scaling axis per tier.
- **Cross-references:** low-latency event delivery and back-pressure for high-throughput telemetry streams → `workflows/90-specialized/realtime-streaming.md`; STRIDE threat model per trust boundary, device identity, secrets management → `workflows/40-cross-cutting/security-architecture.md`; the 8 fallacies and canonical tradeoffs that govern every edge-cloud placement → `../../shared/fallacies-and-tradeoffs.md`.

## Step sequence
- **audit:** Frame the edge-IoT drivers (the latency/bandwidth/sovereignty QAS + the connectivity QAS + the device-resource constraints + the fleet-update SLO) from the utility tree → reconstruct or confirm the device→gateway→cloud topology diagram → for each compute placement, check: is the placement justified by a named driver (latency/bandwidth/sovereignty)? → walk every write path for offline behaviour (store-and-forward buffer, idempotent sync, conflict-resolution policy) — flag any path where disconnection is not handled → check device identity (per-device cryptographic identity, mutual auth at each boundary, revocation path) → check OTA update path (signing, staged rollout, rollback trigger, blast-radius bound) → verify constrained-resource model exists per device/gateway class → run STRIDE per trust boundary (device↔gateway, gateway↔cloud, OTA path, management plane) → check the 8 fallacies — flag any hop where latency/bandwidth/topology change is assumed away → emit findings + risk register. Read-only; authors nothing, edits no source.
- **design:** **Frame** (extract/confirm the latency/bandwidth/sovereignty/connectivity QAS + resource constraints + fleet-update SLO; surface unstated drivers as open questions; read-only) → **Explore** (N ≥ 2 divergent placements: e.g. *thin-device + fat-gateway with local ML inference* vs *thick-device with cloud-offload only for analytics* vs *serverless edge functions at a regional PoP with direct-device MQTT*; each is a concrete spec — topology, protocol, store-and-forward mechanism, OTA path, resource model — not the same boxes relabeled; read-only) → **Evaluate** (the skeptical `architecture-evaluator` using ATAM per `../../shared/evaluation-method.md` scores each option against the utility tree and hard oracles; identifies **sensitivity points** — e.g. "end-to-end latency is *sensitive* to whether inference runs on-gateway vs round-trips to cloud"; **tradeoff points** — e.g. "on-gateway ML inference meets the latency driver but demands a gateway-class CPU, raising BOM cost and OTA complexity"; **risks** — e.g. "store-and-forward buffer overflow if disconnection exceeds flash budget at the stated event rate"; order-swapped to kill position bias; picks a winner and grafts superior elements from runners-up; read-only) → **Document** (the ONE writer — `solution-architect`, advised read-only by `infrastructure-architect` and `security-architect` — authors the C4 Deployment diagram (device/gateway/cloud tiers with trust-boundary overlays, diagrams-as-code), one **MADR ADR** per significant decision (edge-cloud split, gateway topology, store-and-forward mechanism and eviction policy, sync protocol, conflict-resolution model, device-identity approach, OTA update strategy), the capacity model (buffer sizing at stated event rate × max disconnection window, uplink utilization at stated device count × payload size), and the threat model (STRIDE per boundary) into `architecture/<slug>/`; reuses named patterns — store-and-forward, zero-touch provisioning, staged OTA rollout, CRDT merge — before inventing) → **Verify** (fitness functions: topology-diagram compiles; every compute placement traces to a named driver in the ADR; every write path has a named conflict-resolution policy; the capacity model shows the buffer fits within device flash budget; every STRIDE cell is controlled or accepted; OTA ADR names signing algorithm + rollback trigger + blast-radius cap; evaluator re-scores from artifacts) → loop ≤ 10 or pass.

## Assertions & exit gate
- Every compute placement at the edge (device or gateway) traces to a named latency, bandwidth, or sovereignty driver in an ADR — no unjustified edge placement.
- Every write path that can diverge during a disconnection period has a named store-and-forward buffer (with a bounded size), an idempotent sync protocol (replay-safe IDs or sequence numbers), and a conflict-resolution policy (LWW / CRDT / server-authoritative) — no write path where "reconnect and sync" is the complete description.
- Every device has a per-device cryptographic identity (X.509 / TPM-backed key); the gateway-to-cloud channel is mutually authenticated; per-device revocation is described; no shared symmetric secret spans the full fleet.
- The OTA update path names the artifact signing algorithm (asymmetric, private key in HSM), the staged rollout policy (canary wave size + abort threshold), the rollback trigger (health-check failure within a bounded window), and the signature verification check on the device — no OTA path without a rollback or without signature verification.
- The constrained-resource model exists per device/gateway class: flash, RAM, CPU class, and (where applicable) battery budget are stated; every component placed on a class is bounded to fit within those limits.
- The 8 fallacies are accounted for at every hop — especially fallacies #1 (network reliable), #2 (latency zero), #3 (bandwidth infinite), and #5 (topology stable) — which are violated by design at the edge. No hop where any of these is implicitly assumed.
- The consistency model is stated per write path (AP during partition, reconcile on reconnection; or reject-on-disconnect for strong-consistency paths) with its mechanism named.
- Every STRIDE category per trust boundary (device↔gateway, gateway↔cloud ingest, OTA path, management plane) is mitigated by a named control or recorded as an explicit accepted risk — no empty cell.
- Every significant decision is a complete ADR (context · ≥2 options · decision · positive and negative consequences); the topology diagram (C4 Deployment with trust-boundary overlays) compiles/renders without error.
- **Gate:** all hard oracles green (edge placement driver-traced · offline write paths have store-and-forward + idempotent sync + conflict-resolution · per-device cryptographic identity + mutual auth + revocation · OTA signed + staged + rollback-bounded · resource model per class · fallacies honored · consistency stated · STRIDE cells controlled-or-accepted · ADRs complete · diagram compiles) AND (design) evaluator rubric mean ≥ 0.8.

## Output
Write `artifacts/edge-iot-architecture/report.json` per `../../shared/report-format.md` — findings and risks per `../../shared/finding-schema.md` (`type` from: `"security"` for device-identity or STRIDE gaps; `"availability"` for offline-path gaps; `"performance"` for unjustified or missing edge-placement drivers; `"operability"` for missing fleet-update or observability design; `"risk"` for accepted residual risks; `oracle:"hard"` for any of the gate-blocking conditions above; each finding carries `driver_ref` to the relevant `QAS-*`, the `iso25010` characteristic, the `tradeoff` — what the control or decision costs on other attributes — and `alternatives_rejected`), plus the verification block (placement-traceability coverage, write-path offline-coverage count, STRIDE-cell completeness, OTA-safety checklist pass/fail, diagram-compile result, hard-oracle summary, rubric score).

Authored design artifacts land in `architecture/<slug>/` (per `project-architecture-config.md`):
- `README.md` — edge-IoT decision summary + utility tree at a glance
- `design-doc.md` — arc42 architecture description covering the three-tier topology, compute-placement rationale, offline/sync design, fleet management design
- `quality-attributes.md` — the utility tree: edge-IoT prioritized scenarios (latency budget, bandwidth cap, sovereignty constraint, connectivity QAS, fleet-update SLO, resource constraints)
- `adr/NNNN-*.md` — one MADR per significant decision: edge-cloud split rationale, gateway topology, store-and-forward mechanism + eviction policy, sync protocol, conflict-resolution model, device-identity approach, OTA update strategy
- `diagrams/` — C4 Deployment view (device/gateway/cloud tiers, protocol labels on all relationships, trust-boundary overlays) as diagrams-as-code (Mermaid / PlantUML); dynamic/sequence views for the store-and-forward sync path and the OTA rollout path
- `threat-model.md` — STRIDE matrix per trust boundary (device↔gateway, gateway↔cloud ingest, OTA path, management plane)
- `capacity-model.md` — buffer-size calculation (event rate × max disconnection window × message size ≤ device flash budget), uplink utilization (device count × payload size × reporting frequency ≤ bandwidth cap)
- `fitness-functions.md` — the architectural-characteristic checks for CI enforcement: every edge-placement ADR traces to a QAS; every write-path in the design doc has a named offline policy; OTA ADR names signing algo + rollback trigger; STRIDE matrix has no empty cell; diagram compiles
- `tradeoffs.md` — option matrix + sensitivity/tradeoff points (compute placement, offline strategy, conflict model, OTA rollout) + risk register (accepted/residual risks with likelihood×impact and disposition)

The explore→evaluate→verify run reports go to `artifacts/edge-iot-architecture/` (machine record); the architecture package lives in `architecture/<slug>/`.

## Guardrails
Per `../../shared/guardrails.md`: **propose and document — never write product code, never apply infra, never flash a device, never touch a live fleet.** The fleet authors ADRs, topology diagrams, the threat model, the capacity model, and fitness-function specifications; the *controller* implements the firmware and the gateway software; the *operator* applies the provisioning and OTA pipeline. **Ground every edge placement in a driver** — no decision without a latency/bandwidth/sovereignty scenario it serves; a placement no driver demands is accidental complexity, flag it. **Reuse named patterns** (store-and-forward, zero-touch provisioning, staged OTA rollout, CRDT merge, ambassador/ACL at the gateway boundary) before inventing a bespoke mechanism. **Never invent response measures** — latency budgets, bandwidth caps, disconnection window targets, fleet-update SLOs that are unstated are open driver questions for the controller, not assumptions. **One writer** (`solution-architect`) preserves conceptual integrity across the device→gateway→cloud design; `infrastructure-architect` and `security-architect` advise read-only and verify their dimensions. **Record the tradeoff and rejected alternatives** on every decision — the offline strategy, the conflict model, the OTA approach, the device-identity mechanism — never just the winner. **Never write to project memory** — lessons, risks, and residual findings go in the returned report and the ADRs.
