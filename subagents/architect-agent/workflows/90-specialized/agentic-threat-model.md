# agentic-threat-model

**Group:** 90-specialized · **Runs as:** subagent: ../.claude/agents/security-architect.md · **Mode:** audit + design · **Default model:** sonnet

## Purpose
Threat-model an LLM or agentic system beyond classical STRIDE — sweeping OWASP Top 10 for LLM Applications (2025), the CSA MAESTRO 7-layer agentic framework, and MITRE ATLAS — to surface and control the novel attack surface introduced by foundation models, retrieval-augmented pipelines, tool-equipped agents, and multi-agent topologies. Answers: *which threats are live on each trust boundary, what control closes each one, and where is the residual risk signed off?* Every identified threat resolves to a named architectural control or an ADR-shaped accepted-risk entry — "we will add monitoring" is not a control.

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target, the compliance regime (PII classes, regulatory scope — HIPAA / GDPR / SOC 2 / PCI as applicable), the team topology, and where the architecture description / ADR log / diagrams live (`architecture/<slug>/`).
- The **architecture drivers**: the prioritized quality-attribute scenarios from `00-drivers/quality-attribute-scenarios.md` — in particular any security scenario (e.g., `QAS-SEC-01: 0 unredacted-PII egress to model provider under peak load`), availability scenario (the agent path is on an SLO path), and compliance scenario. A flow with no security QAS downgrades to producing an open driver question, not an assumed scope.
- **Target scope**: the in-scope LLM/agentic components — foundation model(s), embedding model(s), RAG/vector pipeline, agent orchestrator, tool manifest, multi-agent topology, inference endpoint, model registry, and the trust boundaries between them and between the system and its users / external services. Read the C4 L2/L3 container/component diagram from `architecture/<slug>/diagrams/` before beginning; if none exists, the first output of audit mode is a trust-boundary enumeration as a data-flow diagram.
- **Preconditions**: the agentic-threat taxonomy (`../shared/agentic-threat-taxonomy.md`) read; the cross-cutting security-architecture workflow (`../40-cross-cutting/security-architecture.md`) and the agent interop protocol (`../30-integration/agent-interop-protocol.md`) consulted for existing STRIDE coverage and inter-agent call contracts; `rag-evaluation.md` consulted for retrieval trust-boundary specifics. The flow does not re-document classical STRIDE controls already authored by `security-architecture.md` — it cites them by ADR/path and extends with LLM/agentic-specific sweep.

## Oracle (source of truth)
The OWASP Top 10 for LLM Applications 2025, the CSA MAESTRO 7 layers, MITRE ATLAS, and NIST AI 600-1 are the external truth — per `../shared/agentic-threat-taxonomy.md`. Each framework's coverage requirement is the measurable gate.

**hard:** (deterministic — traceable, measurable, or structurally checkable; block on failure)
- **Full OWASP LLM01–LLM10 sweep, per component.** Every LLM/agent component is swept against all ten OWASP LLM categories. For each: the threat is present (in-scope) or explicitly out-of-scope with a reason; if in-scope it has a named architectural control (pattern + component + verifiable configuration) or an ADR-shaped accepted-risk entry. Particular focus on **LLM01** (prompt injection — every data-ingestion path to the model is tagged with a trust tier; no untrusted content enters the instruction plane without a sanitization + output-validation step, verifiable by tracing the data-flow diagram), **LLM06** (excessive agency — every tool in every agent's manifest has a documented permission scope that is the minimum required; irreversible-write actions are gated behind a human-in-the-loop confirmation token), **LLM07** (system-prompt leakage — output scanning for system-prompt fingerprints is a post-generation gate), and **LLM08** (vector/embedding weaknesses — cross-tenant retrieval bypass returns empty, verifiable by the retrieval-authorization fitness function). An OWASP item that is in-scope with no control and no accepted-risk ADR is `type:"security"`, `severity:"critical"`, `oracle:"hard"`.
- **Full MAESTRO L1–L7 sweep.** Walk the component diagram layer by layer (Foundation Model → Data → Agent Frameworks → Deployment Infrastructure → Observability & Monitoring → Security Controls → Agent Ecosystem). For each layer, enumerate every component present, list the applicable threats per `../shared/agentic-threat-taxonomy.md` § Part 2, and confirm the minimum coverage requirement for that layer is met. A layer with in-scope components and no documented sweep is `type:"security"`, `severity:"critical"`, `oracle:"hard"`.
- **Every agent↔tool and agent↔agent boundary carries authn + a trust scope + excessive-agency limits.** For each directed call in the agent topology: (a) the calling agent is authenticated to the tool/sub-agent (not ambient trust); (b) the sub-agent's output is treated as untrusted input by the receiver before any further action; (c) the tool's permission scope is documented and is the minimum required; (d) an explicit action budget (max tool calls per session, max irreversible-write actions per session) is enforced by the orchestrator, not by model self-governance. A boundary with missing authn, ambient trust, or no action budget is `type:"security"`, `severity:"critical"`, `oracle:"hard"`.
- **MITRE ATLAS tactic sweep per ML component.** For every ML component (foundation model, embedding model, reranker, classifier/guardrail, agentic orchestrator), run the six-tactic ATLAS sweep (ML Model Access, Reconnaissance, Resource Development, Initial Access, ML Attack Staging, Impact) per `../shared/agentic-threat-taxonomy.md` § Part 3. For each applicable technique: (a) the attack path is assessed as present or absent; (b) if present, the control is named with an ATLAS technique ID; (c) the result is recorded in the findings. An ML component with no ATLAS sweep is `type:"security"`, `severity:"major"`, `oracle:"hard"`.
- **Every identified threat resolves to a control or an accepted-risk ADR.** A threat documented with no control and no accepted-risk ADR (context, rationale, re-evaluation conditions) is a `critical` finding. Accepted-risk ADRs are created in `architecture/<slug>/adr/` and linked from the threat-model artifact. "We will add monitoring" is not a control — a control is a named pattern, a specific system component, or a verifiable configuration.
- **RAG/vector pipeline trust boundaries are fully enumerated.** If a RAG pipeline is in scope: the document-ingestion → embedding → vector-store path is a trust boundary; ABAC at the retrieval layer is verified (not delegated to the model); cross-tenant retrieval isolation is confirmed by a named fitness function reference (e.g., `tests/fitness/rag-authz-isolation.test.ts`). Cross-ref `../shared/agentic-threat-taxonomy.md` § LLM08 and `rag-evaluation.md`.

**soft:** (craft — rubric-scored by the evaluator after hard oracles are green)
- **The control set is proportionate to the autonomy and data-sensitivity of the system.** A low-autonomy, read-only agent acting on non-sensitive data warrants lighter controls than a fully autonomous agent with write access to financial or PII records. The threat model names the autonomy level and data-sensitivity class of each agent and calibrates control depth accordingly — a uniformly maximal control set on a low-risk system is as wrong as a uniformly minimal one on a high-risk system.
- **Guardrails are designed in, not bolted on.** Input-side (prompt sanitization, injection defense) and output-side (PII scanning, grounding/schema validation, toxicity classifier) guardrail components appear in the C4 container diagram as first-class components — not described as future work. The MAESTRO L6 (Security Controls layer) threat surface is addressed by the guardrail topology, not assumed to be closed by model self-governance.
- **Human-in-the-loop gates are right-sized.** Irreversible-write and high-impact actions have an explicit HITL gate defined in the agent design (the approval token the orchestrator must receive before tool invocation). The HITL trigger threshold is calibrated to the action class (reversible-read → no gate; reversible-write → soft gate / audit log; irreversible-write → hard gate / human approval) — not applied uniformly to all actions (which defeats usability) or to none (which defeats safety).

## Standards & techniques
- **OWASP Top 10 for LLM Applications 2025** (published November 2024) — the authoritative application-layer vulnerability taxonomy for LLM-integrated systems. Each of LLM01–LLM10 is a required sweep item per `../shared/agentic-threat-taxonomy.md`. Citation: owasp.org/www-project-top-10-for-large-language-model-applications.
- **CSA MAESTRO — Multi-Agent Environment Security Threat and Risk Ontology** (Cloud Security Alliance AI Safety Working Group) — the 7-layer agentic threat-decomposition framework (Foundation Model / Data / Agent Frameworks / Deployment Infrastructure / Observability & Monitoring / Security Controls / Agent Ecosystem). Mandates layer-by-layer coverage; each layer has a minimum coverage requirement. Citation: cloudsecurityalliance.org/research/working-groups/artificial-intelligence/maestro.
- **MITRE ATLAS — Adversarial Threat Landscape for AI Systems** (atlas.mitre.org) — the ML-specific analog of ATT&CK; 6 tactics and technique-level coverage (AML.T0051.000 LLM Prompt Injection, AML.T0019 Training Data Poisoning, AML.T0025 Model Inversion, AML.T0010 ML Supply Chain Compromise, AML.T0029 Denial of ML Service, etc.). Used for technique-level tracking per ML component.
- **NIST AI RMF 1.0 + AI 600-1 Generative AI Profile** (NIST, January 2023 + July 2024) — the Govern / Map / Measure / Manage lifecycle for AI risk management; AI 600-1 adds 12 GenAI-specific risks (Confabulation, Data Privacy, Harmful Bias, Human-AI Configuration, Information Integrity, Information Security, Intellectual Property, Harmful Content, Overconsumption, Privacy, Value Chain / Third-Party Risks, Unreliable/Unsafe Behavior). Used to position architectural controls within governance framing and surface accepted risks that need ADR treatment.
- **Classical STRIDE** (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) — applied to every trust boundary as the base layer before the LLM/agentic extensions. STRIDE per trust boundary is a hard gate in `shared/quality-attribute-rubric.md` (`trust_boundaries_secured`); this flow extends it, does not replace it. Cross-ref `../40-cross-cutting/security-architecture.md`.
- **LINDDUN** (Linkability, Identifiability, Non-repudiation, Detectability, Disclosure of information, Unawareness, Non-compliance) — applied where PII flows through the model context or retrieval pipeline (LLM02, LLM07, LLM08). Pairs with the privacy-compliance architecture (`../40-cross-cutting/privacy-compliance-architecture.md`).
- **Trust boundary enumeration for agent↔tool and agent↔agent.** Every directed call in the multi-agent topology is a trust boundary: it crosses when a message crosses agent/tool isolation. Each boundary carries: (a) the authenticated identity of the caller, (b) the scope of permissions the callee accepts from that caller, (c) the treatment of the callee's output as untrusted by the caller. Agent collusion (two agents acting on ambient trust without per-call authn) is modeled as an LLM06 + MAESTRO L7 finding. Cross-ref `../30-integration/agent-interop-protocol.md` for the inter-agent call contract.
- **Principle of least privilege + action reversibility classification.** Tools / API scopes / file permissions are categorized as read-only, reversible-write, or irreversible-write. The excessive-agency control (LLM06) is operationalized through a least-privilege matrix: each agent × tool × permission scope, with the business justification for any write scope. Irreversible actions require a HITL gate.
- **Data-flow diagram as the threat-model base.** Per standard threat-modeling practice (OWASP Threat Dragon, the Shostack data-flow diagram method), the threat model is grounded in the data-flow diagram, not the narrative. Every trust boundary is a line on the DFD; every threat is anchored to a boundary. A threat not on the DFD is not documented.

## Step sequence
- **audit:** Read the existing architecture description, C4 diagrams, and any prior threat model → enumerate trust boundaries from the data-flow / C4 L2 diagram (create a boundary list if none exists — this is the first read-only deliverable) → sweep each LLM/agent component against **OWASP LLM01–LLM10** and flag gaps → walk **MAESTRO L1–L7** for each component layer and flag uncovered layers → run the **MITRE ATLAS tactic sweep** per ML component and record technique-level findings → map findings to **NIST AI 600-1** risk categories for governance framing → check every agent↔tool / agent↔agent boundary for authn, trust scope, and action budget → check the RAG/vector pipeline for retrieval ABAC and cross-tenant isolation → emit a prioritized findings list + risk register per `finding-schema.md` (critical → cosmetic) with a **per-framework coverage matrix** (OWASP × MAESTRO × ATLAS — which items are covered / gaps / accepted). Read-only, authors nothing, edits no source.
- **design:** **Frame** (confirm the in-scope LLM/agentic components, their trust boundaries, the security quality-attribute scenarios, and the compliance regime — read-only; surface any missing security QAS as open driver questions) → **Explore** (≥2 divergent control architectures — e.g., *centralized guardrail sidecar* vs *per-agent embedded policy engine*; or *HITL on all writes* vs *HITL only on irreversible actions + anomaly-based escalation* — each as a concrete spec: where the guardrail lives in the C4 topology, what the inter-agent authn mechanism is, how the least-privilege matrix is enforced, read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each direction against the security QAS + hard oracles per `evaluation-method.md`; names sensitivity points — e.g., *a centralized guardrail sidecar is a SPOF on the agent path*; tradeoff points — e.g., *per-agent embedded policy is more resilient but harder to update consistently*; order-swapped; penalizes security theater over grounded controls) → **Document** (ONE writer authors the threat-model artifact `architecture/<slug>/threat-model.md` extending the existing STRIDE baseline with the OWASP/MAESTRO/ATLAS findings; the **control ADRs** for each significant architectural decision — guardrail topology, inter-agent authn mechanism, HITL gate policy, action budget enforcement — with tradeoffs + rejected alternatives; the **least-privilege matrix** for each agent × tool scope; the **accepted-risk ADRs** for threats explicitly not mitigated; the **coverage matrix** as a machine-readable table; the **fitness-function specs** for verification; reuses named patterns from `../80-patterns/` and `../shared/agentic-threat-taxonomy.md`) → **Verify** (run the OWASP LLM red-team probe set against the staged system — 20 standardized jailbreak + extraction attempts per LLM01 and LLM07; run the retrieval-authorization fitness function for cross-tenant isolation; check inter-agent authn on a test topology; run the ATLAS technique checklist; evaluator re-scores from the authored artifacts) → loop ≤ ~10 or pass.

## Assertions & exit gate
- Every LLM/agent component has a completed OWASP LLM01–LLM10 sweep with each item marked covered (control named), out-of-scope (reason given), or accepted-risk (ADR linked).
- Every MAESTRO layer (L1–L7) has been walked; every in-scope component at each layer has a documented threat assessment and minimum-coverage confirmation.
- Every ML component has a completed MITRE ATLAS tactic sweep with technique IDs recorded.
- Every agent↔tool and agent↔agent boundary in the component diagram has: (a) a named authn mechanism, (b) a documented permission/trust scope, (c) an action budget enforced by the orchestrator.
- Every identified threat resolves to a named architectural control (pattern + component + verifiable configuration) or an ADR-shaped accepted-risk entry; zero threats left in a "to be addressed later" state.
- RAG/vector pipeline (if in scope): retrieval ABAC confirmed by a named fitness function reference; cross-tenant isolation verified.
- A per-framework coverage matrix (OWASP × MAESTRO × ATLAS) is present in the threat-model artifact — machine-readable, gap-free or with explicit accepted-risk entries for each gap.
- All control ADRs are complete (context / ≥2 options / decision / positive + negative consequences — per `../shared/adr-template.md`).
- The red-team probe results (design mode) are present as a pass/fail record naming each probe, the expected behavior, and the observed behavior.
- **Gate:** hard oracles green (all sweep items covered or accepted; all boundaries authenticated; retrieval isolation verified; red-team probes pass) AND (design) evaluator rubric mean ≥ 0.8.

## Output
Write `artifacts/agentic-threat-model/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (type: `"security"` or `"risk"`, with `driver_ref` tracing each finding to a security QAS or compliance constraint, `component` naming the boundary or layer, and `oracle:"hard"` for uncovered OWASP/MAESTRO items or unauthenticated boundaries). The verification block (design mode) carries: `owasp_coverage_matrix` (LLM01–LLM10 × component → control / out-of-scope / accepted), `maestro_layer_sweep` (L1–L7 → covered / gap), `atlas_sweep` (tactic × ML-component → technique-id + control), `red_team_probes` (probe-id + expected + observed + pass/fail), `retrieval_authz_fitness` (pass/fail), `inter_agent_authn` (pass/fail per boundary), and `rubric_score`. Authored artifacts (design mode) land in `architecture/<slug>/`:
- `threat-model.md` — the full threat model: data-flow diagram (trust-boundary enumeration), OWASP sweep table, MAESTRO layer walk, ATLAS sweep table, per-boundary STRIDE + LLM/agentic extension, accepted-risk register.
- `adr/NNNN-guardrail-topology.md` — the guardrail placement decision (input + output filtering, where in the C4 topology).
- `adr/NNNN-inter-agent-authn.md` — the inter-agent and agent↔tool authn mechanism decision.
- `adr/NNNN-hitl-gate-policy.md` — the HITL gate policy: which action classes trigger a human approval step.
- `adr/NNNN-<accepted-risk-slug>.md` — one ADR per accepted risk (threat explicitly not mitigated), with re-evaluation conditions.
- `fitness-functions.md` — the executable/automatable checks enforcing the controls in CI: retrieval-authorization isolation test, inter-agent authn contract test, action-budget enforcement check, output-scanner regression, red-team probe suite reference.

## Guardrails
Per `shared/guardrails.md`: this flow proposes and documents the threat model and its controls — it never writes product code (`src/**`, `frontend/**`, `backend/**`) and never applies infrastructure. It authors architecture artifacts (threat model, ADRs, fitness-function specs) for the controller to implement. Every control decision traces to a security driver (no control without a QAS or compliance constraint it satisfies; no driver without a decision that satisfies it — `guardrails.md` § 2). The oracle is external — OWASP, MAESTRO, ATLAS, NIST AI 600-1 — not the agent's judgment about what "seems secure." One writer (`security-architect`) authors the threat-model artifact and the control ADRs; the evaluator is read-only and scores against the frameworks. Accepted risks are proposed by this flow but the final accept/mitigate/monitor disposition is the controller's/human's call. Findings from the red-team probe step are reported; the controller decides which to act on — this flow does not self-implement fixes between the audit and verify steps.
