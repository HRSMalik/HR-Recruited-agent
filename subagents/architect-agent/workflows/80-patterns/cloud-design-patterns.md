# cloud-design-patterns

**Group:** 80-patterns
**Runs as:** subagent: ../../agents/solution-architect.md
**Mode:** audit + design   ·   **Default model:** sonnet

## Purpose

Maintain the cloud design-pattern catalog as the system's reusable vocabulary for cloud-native concerns — resilience, messaging, data management, design-and-implementation, security, and carbon-aware sustainability. Every structural decision in the system is grounded in a named pattern with a known contract, known preconditions, and known tradeoffs; no bespoke mechanism where a catalog pattern suffices. Answers: *which catalog pattern satisfies this driver, under what preconditions, at what tradeoff, and have its failure modes been accounted for?*

## Inputs & preconditions

- From `project-architecture-config.md`: the project's stack, the deployment target, the team topology (small team, modular-monolith default unless configured otherwise), the constraints (no new datastore/broker without a driver-justified ADR), and the artifact locations (`architecture/<slug>/`).
- The active **drivers** from `workflows/00-drivers/quality-attribute-scenarios.md`: the prioritized utility tree (quality-attribute scenarios with response measures). Each pattern application must trace to at least one named driver (`QAS-*`). No pattern without a driver is recommended — that is accidental complexity.
- The **existing architecture description** (`architecture/<slug>/design-doc.md`, `architecture/<slug>/adr/`) and the integration surface (`workflows/30-integration/integration-patterns.md`, `workflows/30-integration/service-communication-resilience.md`) — read before recommending; do not repeat patterns already adopted.
- Preconditions: the utility tree exists (or `workflows/00-drivers/quality-attribute-scenarios.md` has been run); the C4 container-level diagram is available so pattern placements are traceable to real containers; the catalog group relevant to the mandate (resilience vs messaging vs data-mgmt vs security) is identified before stepping through it.

## Oracle (source of truth)

The Azure/AWS cloud design-pattern catalog (grouped by concern) and the patterns' own documented contracts — each pattern names its *problem*, *solution*, *when-to-use*, *when-not-to-use*, and *known tradeoffs*. An applied pattern is only a hard-oracle pass when the pattern is matched to a driver, its preconditions are met in the design, and its tradeoff is recorded.

**hard:**
- Every recommended pattern is named (from the catalog) and matched to a concrete driver (`QAS-*`) — no orphan pattern recommendation.
- Each pattern's **preconditions are verified present** in the design before it is applied: circuit-breaker requires a defined fallback (a response to return when the circuit is open, not an unbounded wait); competing-consumers requires idempotent message processing (a duplicate delivery must not corrupt state); queue-based load-leveling requires the downstream consumer to tolerate the latency added by the queue (a driver that demands sub-second synchronous response cannot absorb this pattern without explicit scope analysis); claim-check requires a durable, accessible store for the payload and a reliable reference passed on the message; strangler-fig requires a routing façade at the entry point that can redirect traffic at the granularity of migration slices; sidecar/ambassador require a platform that can co-locate a proxy process per instance; anti-corruption layer requires owning the translation layer and isolating the foreign model so it does not bleed into the domain.
- Every applied pattern records its **tradeoff against at least one other driver** — a circuit-breaker that improves availability (`QAS-AVAIL-*`) costs observability (the open-circuit state must be instrumented or failures are silent); queue-based load-leveling that smooths throughput (`QAS-PERF-*`) adds latency (violates synchronous response measures unless scoped to async paths); competing-consumers that scale throughput (`QAS-SCALE-*`) requires idempotency (consistency cost); cache-aside that reduces latency (`QAS-PERF-*`) introduces staleness (consistency tradeoff named in `workflows/20-data/caching-strategy.md`).
- The 8 fallacies of distributed computing (`shared/fallacies-and-tradeoffs.md`) are honored per pattern: retry + backoff + jitter accounts for fallacy 1 (network not reliable); circuit-breaker accounts for fallacy 1 + partial failure; claim-check accounts for fallacy 3 (bandwidth not infinite); ambassador/sidecar account for fallacy 8 (network not homogeneous); bulkhead accounts for partial failure cascades.
- No pattern is applied without a **failure-mode definition**: what happens when the pattern's mechanism fails (the circuit stays open; the queue backs up beyond its depth; the claim-check store is unreachable; the sidecar proxy crashes).
- Cross-service calls introduced or formalized by a pattern carry a bounded timeout, retry policy (with backoff+jitter where idempotent), and fallback (`shared/quality-attribute-rubric.md` → *Failure mode defined* hard oracle).
- Every pattern application becomes an ADR entry (`shared/adr-template.md`): context · considered options (include "bespoke mechanism" and "do nothing") · decision · consequences (positive AND negative) · one-way-door flag where the pattern introduces a structural dependency hard to remove (strangler-fig routing layer; ACL at an integration boundary).
- Diagrams-as-code updated to show the pattern placement: which containers are affected, what new component (the circuit-breaker wrapper, the queue, the sidecar, the ACL) is introduced, and the sync/async distinction per `shared/diagramming-standards.md`. Diagram renders without error.

**soft:**
- The catalog is **reused over bespoke designs** — if a named pattern exists for the problem, it is preferred over an invented mechanism with unknown failure modes. A bespoke design where a catalog pattern applies is a finding (`oracle:"soft"`, `severity:"minor"` minimum).
- Patterns are **composed coherently** — not cargo-culted. A retry wrapping a circuit-breaker wrapping a bulkhead wrapping a queue is over-engineered unless each layer solves a distinct named driver. The evaluator penalizes accidental complexity; the simpler sufficient pattern wins.
- The **simplest sufficient pattern** is chosen: retry alone is simpler than circuit-breaker; circuit-breaker alone is simpler than circuit-breaker + bulkhead; gateway-aggregation is simpler than a bespoke fan-out orchestrator. Choose up from simple, not down from elaborate.
- Pattern names and placement are **consistent across ADRs and diagrams** — the same component is called the same thing everywhere (Brooks — conceptual integrity). A pattern named "retry wrapper" in the ADR and "resilience decorator" in the diagram is a finding.
- **Appropriate coupling & cohesion** — patterns are applied at the correct granularity: a sidecar/ambassador is justified only when the same cross-cutting concern applies consistently across multiple services (not introduced for a single service where inline handling is simpler); an ACL boundary is drawn on business-capability / volatility criteria, not technical-layer boundaries; no pattern application creates a "distributed monolith" where the caller and callee must deploy together. Dependencies introduced by patterns point toward stability.
- **Operability & observability** — every introduced pattern mechanism is instrumented: circuit-breaker state (Closed/Open/Half-Open) exposed as a metric; queue depth alarmed with a threshold tied to the consumer's processing-rate SLO; retry+backoff invocations tracked (rate of retries is an early-warning signal for fallacy-1 violations); sidecar/ambassador proxy health surfaced to the service mesh health-check; failure-mode runbooks documented per pattern (`architecture/<slug>/fitness-functions.md`). A pattern that adds operational complexity without adding its own observability is a finding (`oracle:"soft"`, `severity:"minor"` minimum).
- **Sustainability (carbon-aware)** — per `shared/quality-attribute-rubric.md` → *Sustainability*: where sustainability is a stated driver, temporal/spatial shifting, demand shaping, and right-sizing are considered and an SCI estimate is attempted; where it is *not* a stated driver, absence of a carbon-aware pattern is a non-finding. This oracle is **always soft and uncertainty-flagged** (embodied carbon and grid-intensity forecasts are hard to source precisely) — it never blocks the gate unless the project's `project-architecture-config.md` explicitly promotes it to a hard constraint (e.g. a contractual carbon budget). A carbon-aware pattern that violates a higher-priority latency/availability response-measure is itself a finding — carbon-awareness never overrides a hard oracle.

## Standards & techniques

**The Azure/AWS cloud design-pattern catalog, grouped by concern** — the canonical source, cross-referenced by concern group, not alphabetically, so gaps in a concern area are visible:

- **Resilience group** (account for fallacies 1, 5, partial failure): **Retry** — transient failure → retry with exponential backoff + jitter; precondition: the operation is idempotent (or a duplicate is safe); tradeoff: adds latency per retry and amplifies load under systemic failure; pair with circuit-breaker to bound total retry storms. **Circuit Breaker** — repeated failure → open the circuit and return a fallback immediately; three states (Closed → Open → Half-Open); precondition: a meaningful fallback exists (cached response, degraded mode, error page — not an infinite wait); tradeoff: improves availability and latency-under-failure but requires instrumentation (open-circuit is invisible without metrics) and adds state the service must manage. **Bulkhead** — isolate failure domains by pool/thread/process so one slow downstream does not exhaust the caller's resource pool; precondition: the caller has multiple downstream dependencies worth isolating; tradeoff: adds resource partitioning overhead; simplest form is a separate connection pool per dependency.

- **Messaging group** (account for fallacies 1, 2, 3): **Queue-Based Load Leveling** — absorb burst load via a durable queue so the downstream consumer runs at a steady rate; precondition: the producer–consumer relationship tolerates the added latency (async path only — never apply to a synchronous response-measure driver without explicit scope); tradeoff: smooths throughput but adds queuing latency (latency ↔ throughput canonical tradeoff, `shared/fallacies-and-tradeoffs.md`); failure mode: queue depth grows unbounded if consumer is slower than producer — requires a depth alarm and backpressure. **Competing Consumers** — multiple consumer instances read from a single queue to scale throughput; precondition: message processing is idempotent (duplicate delivery is inevitable under at-least-once semantics); tradeoff: horizontal throughput scale but consistency cost — ordering guarantees weaken; pair with a dead-letter queue for poison messages. **Claim-Check** — replace a large payload on a message with a reference (a URI/key) to the payload stored in a durable store; account for fallacy 3 (bandwidth not infinite); precondition: a durable, accessible store exists and the reference retrieval is reliable; tradeoff: decouples message size from payload size but adds a store dependency and a retrieval round-trip.

- **Data management group** (account for fallacies 1, 2, consistency ↔ availability): **Cache-Aside** — load data into cache on demand (read-miss populates the cache); precondition: the read pattern is read-heavy and the data can tolerate bounded staleness; tradeoff: reads are fast after warm-up but the first read after eviction hits the origin (stampede risk — pair with TTL + probabilistic early expiration); consistency cost named per driver. Full treatment in `workflows/20-data/caching-strategy.md`.

- **Design & implementation group** (account for Conway's law, evolvability, fallacy 8): **Ambassador** — co-locate a proxy sidecar that handles cross-cutting concerns (retries, circuit-breaking, mTLS, observability) on behalf of the primary service; precondition: the platform supports process co-location (container per-pod, or similar) and the cross-cutting concern is consistent across multiple services justifying the abstraction; tradeoff: adds a network hop + process-management overhead; overkill for a single service. **Sidecar** — the general form of Ambassador: a co-located process that extends or augments the primary container without modifying it; precondition: same as Ambassador; tradeoff: operational complexity scales with sidecar count; worth it only when the augmentation is applied consistently across many services. **Gateway Aggregation** — aggregate multiple downstream calls into one response at the gateway/BFF so the client makes one round-trip instead of N; accounts for fallacy 2 (latency not zero) and the chatty-interface anti-pattern; precondition: the aggregated calls can be parallelized or have acceptable sequential latency; tradeoff: the gateway becomes a coupling point — a change in any downstream contract requires a gateway update; cross-reference `workflows/30-integration/api-gateway-bff.md`. **Anti-Corruption Layer (ACL)** — translate between the domain model and a foreign/legacy model at an integration boundary so the foreign model's concepts do not bleed into the domain; precondition: the caller owns (or can introduce) the translation layer; tradeoff: adds a translation layer to maintain but protects domain purity — the cost of not having one is domain contamination that is hard to reverse (one-way-door risk); cross-reference `workflows/30-integration/integration-patterns.md`. **Strangler Fig** — incrementally replace a legacy system by routing new behavior to the new implementation while the legacy handles the rest; precondition: a routing façade exists at the entry point; migration slices are independently deployable; tradeoff: the routing layer is a new SPOF (must be reliable and observable); the legacy and new systems must run concurrently for the migration duration (cost + consistency complexity); cross-reference `workflows/30-integration/integration-patterns.md`.

- **Security group** (account for fallacy 4 — network not secure): **Valet Key** — grant a client time-limited, scoped access to a specific resource (e.g. a pre-signed URL for blob storage) so the service does not handle large payload transfer and the client does not hold permanent credentials; precondition: the storage system supports short-lived scoped tokens; tradeoff: adds token-generation latency per operation but removes the service from the data path.

- **Carbon-aware group** (Green Software Foundation — sustainability soft oracle, `shared/quality-attribute-rubric.md` → *Sustainability*; SCI per ISO/IEC 21031:2024): these patterns trade *when*, *where*, or *how much* compute runs against carbon intensity, never against a hard response-measure — a scenario with a latency/availability response-measure always wins over a carbon-aware shift unless the driver set says otherwise. **Temporal Shifting** — move time-flexible workloads (batch jobs, non-urgent reports, model-retraining, backups) to run when the local grid's carbon intensity is forecast lower, using a signal API (e.g. WattTime, Electricity Maps) to decide *when* to fire the job; precondition: the workload has a slack window (a deadline looser than "now") and is idempotent/resumable if the window is missed; tradeoff: adds scheduling complexity and a dependency on an external carbon-intensity signal (itself a fallacy-1/5 risk — the signal API can be unavailable or stale, so a "run anyway by deadline" fallback is mandatory); failure mode: if the signal API is unreachable, the job must still run by its committed deadline on a fixed schedule — silently skipping the workload is not acceptable degradation. **Spatial Shifting** — route or deploy time-and-location-flexible workloads to the region/AZ with the lowest current or forecast carbon intensity among the regions that already satisfy data-residency and latency constraints; precondition: multi-region deployment already exists (or is independently justified by an availability/latency driver — never stood up *for* carbon-awareness alone, since cross-region replication itself carries a carbon and consistency cost) and the candidate regions are pre-filtered by compliance/residency (`shared/quality-attribute-rubric.md` → *Data lifecycle lawful*) before carbon intensity is used as the tiebreaker; tradeoff: shifting load to a greener but geographically distant region can worsen latency (fallacy 2) and cross-region egress cost — carbon-awareness never overrides a latency response-measure; failure mode: the greenest region is unreachable or non-compliant → fall back to the nearest compliant region, never to an unbounded retry across regions. **Demand Shaping** — shape user-facing demand itself rather than just scheduling around it: offer a lower-carbon/lower-fidelity mode (reduced video bitrate, longer batch windows instead of real-time, degraded-but-functional responses) and let carbon intensity (or cost, as a proxy) influence the default; precondition: the product has a legitimate degraded-mode path already defined for graceful degradation (reuse the same mechanism the resilience group uses for fallback, not a bespoke one); tradeoff: demand shaping is a product/UX decision as much as an architectural one — it must be opt-in or clearly disclosed, never a silent quality reduction; failure mode: none structural, but an un-disclosed degradation is a trust/UX risk, not a carbon-pattern risk. **Energy-Proportional / Right-Sizing** — size and autoscale compute so utilization tracks load (avoid idle over-provisioned capacity, which burns energy proportionally to *provisioned* not *used* capacity); precondition: the workload's autoscaling signal (CPU/RPS/queue-depth) is already defined for a performance/cost driver — right-sizing reuses that signal, it does not invent a new one; tradeoff: aggressive scale-to-zero adds cold-start latency (fallacy 2) — the same latency-vs-elasticity tradeoff named in `workflows/50-infrastructure/` capacity/scaling flows, now carrying a carbon dimension alongside the cost dimension; failure mode: scale-to-zero on a driver that cannot tolerate cold-start latency is a driver violation, not a valid right-sizing choice — gate right-sizing depth by the tightest latency response-measure in scope.

**Named catalog grouping criteria** (per the Azure Well-Architected + AWS Well-Architected frameworks): each pattern addresses one or more of the five pillars — Reliability, Security, Performance Efficiency, Operational Excellence, Cost Optimization — and the flow maps every recommended pattern to its primary pillar(s) to surface coverage gaps. This mapping is enforced via a hard assertion in the exit gate: any active prioritized driver (`QAS-*`) whose governing Well-Architected pillar has no applied catalog pattern is a finding (`oracle:"hard"`, `severity:"major"` — a driver-threatening gap). Pillar coverage is verified in the report's `decisions[]` block, where each ADR records the pillar(s) it addresses; missing-pillar gaps appear as `ARCH-CDP-NNN` risk entries per `finding-schema.md`. The carbon-aware group maps to Cost Optimization + Operational Excellence (the Well-Architected frameworks do not carry a standalone sixth "Sustainability" pillar in this catalog's treatment) plus the standalone **Sustainability** soft oracle (`shared/quality-attribute-rubric.md`) — a missing carbon-aware pattern is **never** counted toward the hard pillar-coverage gate above; it is only ever a soft, uncertainty-flagged finding (`oracle:"soft"`), and only when sustainability is a stated driver in `project-architecture-config.md`.

**ATAM sensitivity and tradeoff points** (`shared/evaluation-method.md`): every pattern recommendation names the sensitivity point (the parameter that most determines whether the driver is met — e.g. circuit-breaker threshold voltage and timeout) and the tradeoff point (the driver the pattern helps vs the driver it worsens — e.g. retry improves reliability but worsens latency-under-failure).

## Step sequence

- **audit:** Read `project-architecture-config.md` and the active architecture description (`architecture/<slug>/design-doc.md`, ADRs) → identify all cross-service calls, queue interactions, caching layers, and integration boundaries → for each, check whether a named catalog pattern is applied or a bespoke mechanism is used → verify each applied pattern's preconditions are met (fallback present for circuit-breaker, idempotency for competing-consumers, etc.) → check that each pattern is traced to a driver and its tradeoff is recorded in an ADR → check that failure modes are defined per call → check that diagrams reflect the pattern placement and compile → where sustainability is a stated driver in `project-architecture-config.md`, additionally check whether time/location-flexible workloads have a temporal/spatial-shifting or right-sizing pattern applied and whether an SCI estimate is attempted (soft only — never escalate an absent carbon-aware pattern to `oracle:"hard"`) → emit findings per `shared/finding-schema.md` grouped by catalog concern group then severity (critical → cosmetic); emit a risk register for un-applied patterns where a driver is threatened. Read-only; authors nothing.

- **design:** Frame (read `project-architecture-config.md` + utility tree from `workflows/00-drivers/quality-attribute-scenarios.md` + existing ADRs — confirm driver set including whether sustainability is stated, read-only) → Explore (for each open driver in the resilience/messaging/data-mgmt/security/carbon-aware concern groups, propose ≥2 concrete pattern options as ADR-shaped specs — including "do nothing" or "bespoke mechanism" as a genuine alternative, not a straw man; read-only) → Evaluate (skeptical evaluator scores each option vs the relevant QAS scenarios and hard oracles per `shared/evaluation-method.md`; order-swapped to kill position bias; penalizes accidental complexity; picks the simplest sufficient pattern; names sensitivity + tradeoff points; carbon-aware options are scored against the soft Sustainability criterion and are rejected outright if they would trade away a hard response-measure) → Document (one writer authors the ADRs in `architecture/<slug>/adr/`, updates the C4 diagrams-as-code in `architecture/<slug>/diagrams/` to reflect pattern placements with sync/async labeling and trust boundaries, updates `architecture/<slug>/fitness-functions.md` with pattern-specific fitness checks — retry/backoff present, circuit-breaker state instrumented, queue depth alarmed, consumer idempotency tested, and where applicable a carbon-aware check such as "job has a fixed-schedule fallback if the carbon-intensity signal is unreachable") → Verify (fitness-function checks + diagram-compile + traceability: every applied pattern traces to a QAS; every QAS in the resilience/messaging/data-mgmt scope has a met response-measure; no cross-service call without timeout+retry+fallback; evaluator re-scores from the authored artifacts; loop ≤10 or pass).

## Assertions & exit gate

- Every applied pattern is named from the Azure/AWS cloud design-pattern catalog and matched to a named driver (`QAS-*`); no orphan pattern recommendation.
- Each pattern's preconditions are met: circuit-breaker → fallback defined; competing-consumers → idempotency confirmed; queue-based load-leveling → async path only (no synchronous-response driver violated); claim-check → durable store exists and reference retrieval is reliable; strangler-fig → routing façade identified and owns the entry point; ACL → translation layer owned and isolated.
- Each pattern's tradeoff against at least one other driver is recorded in the ADR (consequences negative — mandatory per `shared/adr-template.md`).
- Every cross-service call introduced or formalized by a pattern has a bounded timeout, retry policy (with backoff+jitter where idempotent), and a fallback or fail-fast — no unbounded synchronous call chains (`shared/quality-attribute-rubric.md` → *Failure mode defined*).
- The 8 fallacies of distributed computing are honored per pattern placement (`shared/fallacies-and-tradeoffs.md`); no design assumes a reliable network, zero latency, or infinite bandwidth.
- Diagrams-as-code compile without error and reflect pattern placements with sync/async distinction, trust boundaries, and element labels per `shared/diagramming-standards.md`.
- Every ADR has context · ≥2 options (including a genuine "do nothing" or "bespoke" alternative) · decision · consequences (positive AND negative) · one-way-door flag where applicable.
- Catalog patterns are preferred over bespoke mechanisms; no bespoke mechanism is introduced where a catalog pattern applies without a recorded justification.
- Where sustainability is a stated driver, carbon-aware patterns (temporal shifting, spatial shifting, demand shaping, right-sizing) are considered and their absence, if warranted, is a recorded soft finding — never a gate-blocking one unless the project has explicitly promoted a carbon budget to a hard constraint.
- **Gate:** hard oracles green (driver traceability, preconditions met, tradeoffs recorded, failure modes defined, fallacies honored, diagrams compile, ADRs complete) AND (design) rubric mean ≥ 0.8 (conceptual integrity — consistent pattern naming; simplicity — no cargo-culted over-composition; evolvability — one-way-door decisions flagged; tradeoff honesty — no ADR with only positive consequences; sustainability — carbon-aware choices considered where driven, uncertainty-flagged, never traded against a hard response-measure).

## Output

Write `artifacts/cloud-design-patterns/report.json` per `shared/report-format.md` — findings and risks per `finding-schema.md` (`id: ARCH-CDP-NNN`; `type` from: availability / integration / data / security / evolvability / risk; `oracle: hard` for a missing precondition, an orphan pattern, an un-traced driver, a missing failure-mode, or a fallacy violation; `oracle: soft` for a bespoke mechanism where a catalog pattern applies, a cargo-culted over-composition, inconsistent naming, or a carbon-aware pattern omitted/misapplied — the last always `severity:"minor"` minimum and flagged `uncertainty: true` given SCI's embodied-carbon measurement gap); the `decisions[]` block listing each pattern-application ADR with its `driver_ref` and, where applicable, its Well-Architected pillar(s); and for design mode the verification block (fitness-function results, diagram-compile results, traceability coverage, hard-oracle results, rubric score).

Authored artifacts in design mode land in `architecture/<slug>/`:
- `adr/NNNN-<pattern-name>-application.md` — one MADR per significant pattern decision (context, ≥2 options, decision, consequences positive+negative, one-way-door flag).
- `diagrams/` — updated C4 container or component diagrams-as-code (Mermaid / PlantUML+C4) showing pattern placements, with sync/async distinction, trust boundaries, and element labels.
- `fitness-functions.md` — pattern-specific checks: retry+backoff present on every fallible call; circuit-breaker state exposed as a metric; queue depth alarm defined (queue-based load-leveling); consumer idempotency covered by a contract test (competing-consumers); ACL translation layer has no domain-model import in the foreign side; where a carbon-aware pattern is applied — temporal-shifted job has a fixed-schedule fallback if the carbon-intensity signal is stale/unreachable; spatial-shifted workload's candidate region set is pre-filtered by residency/compliance before carbon intensity is applied as tiebreaker; right-sized/scale-to-zero path has its cold-start latency checked against the tightest response-measure in scope.

## Guardrails

Per `shared/guardrails.md`: propose & document — never write product code, never apply IaC, never edit `src/**` / `frontend/**` / `backend/**`. Every pattern recommendation is an ADR-shaped proposal (decision + tradeoff + rejected alternatives) for the controller to implement. Reuse the named catalog pattern before inventing a bespoke mechanism; the named pattern carries known contracts and failure modes the bespoke one does not. One writer authors the architecture artifacts for conceptual integrity; the evaluator is read-only and scores against traced scenarios and structural facts, not the writer's prose. The evaluator penalizes over-composition and unjustified novelty; the simpler sufficient pattern wins. Never grade the work without a driver; a pattern with no driver is accidental complexity — flag it, do not adopt it. Carbon-aware recommendations are held to the same rigor as any other pattern (named precondition, tradeoff, failure mode) but are graded on the soft Sustainability oracle only — the flow never fabricates a hard carbon threshold the project has not set, and never lets a carbon-aware choice degrade a hard response-measure (latency, availability, consistency) that outranks it.
