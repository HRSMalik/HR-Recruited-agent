# resilience-patterns

**Group:** 80-patterns
**Runs as:** subagent: ../.claude/agents/solution-architect.md
**Mode:** audit + design
**Default model:** sonnet

## Purpose

Catalogue, analyse, and compose the core resilience patterns — **timeout**, **retry with backoff+jitter**, **circuit-breaker**, **bulkhead**, **rate-limiter**, **load-shedding**, **graceful degradation**, and **fallback** — so that every cross-service call has a *bounded failure envelope* and the blast radius of any single failure is contained. This flow does not design for a specific call site; it builds the **pattern reference** and the **composition rules** that `../30-integration/service-communication-resilience.md` applies per interaction and `../40-cross-cutting/availability-resilience.md` places into its availability model. It answers: *which patterns are mandatory for which failure modes, in what order do they compose, and what are the traps (retry-storm, cascading failure) each is designed to prevent?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack, deployment target, and external boundaries (read the config — do not hardcode any project's values here); the planned integration points whose failure modes must be pre-specified before wiring; the team-topology constraint (e.g. small team, low-ceremony preference) and any stated SLOs or concurrency targets.
- The **drivers / utility tree** from `architecture/<slug>/quality-attributes.md` (`../00-drivers/quality-attribute-scenarios.md`) — specifically the availability and performance scenarios this pattern set must honour: e.g. BRD-seeded `dashboard render < 2s`, `document-processing extract < 30s`, `payment/settlement action < 60s` (express as 6-part QAS with response measures before designing; substitute the project's actual scenarios per `project-architecture-config.md`). **Availability target / RTO is not yet stated in the BRD** — if absent, surface as an open driver question; do not invent a number, do not design HA topology without it.
- The **failure-mode oracle** from `../../shared/quality-attribute-rubric.md` (*Failure mode defined* hard oracle): every cross-process call must have a bounded timeout, a retry policy or explicit no-retry, a circuit-breaker and/or bulkhead, and a fallback or fail-fast.
- The **8 fallacies of distributed computing** (`../../shared/fallacies-and-tradeoffs.md`) — especially fallacy #1 (*the network is reliable*) and fallacy #2 (*latency is zero*): the entire pattern stack exists to honour these where they apply.
- Preconditions: `project-understanding.md` and `docs/` (BRD/PRD) read first; the C4 L2 Container view (or enough topology to enumerate cross-process call sites) available; the interaction inventory from `../30-integration/service-communication-resilience.md` used as the consumer of this pattern catalogue; `../40-cross-cutting/availability-resilience.md`'s FMEA register as the upstream failure-mode source.

## Oracle (source of truth)

The **failure-mode hard oracle** (`../../shared/quality-attribute-rubric.md`) **+ the availability/performance quality-attribute scenarios** (the response measures the composed pattern stack must hold under a degraded environment) **+ the 8 fallacies** (`../../shared/fallacies-and-tradeoffs.md` §1–8). "We use resilience patterns" is not an oracle; "every cross-service call composes timeout (bounded connect+read) + retry only if idempotent (backoff+jitter, max-attempts cap, retry budget) + circuit-breaker (closed/open/half-open, error-rate threshold N or consecutive-failures N) + bulkhead (per-dependency pool) + fallback/fail-fast, and the composed stack demonstrably prevents retry-storm and cascading failure" is.

- **hard:**
  - **Timeout always, first.** Every cross-process or external call has a bounded **connect timeout** and a bounded **read/response timeout** — never `None`, never the driver default, never infinite. The timeout is set to less than the parent scenario's latency budget (so a hung callee fails fast rather than blocking the caller until the caller's own timeout fires — the *budget-allocation* rule: a child timeout must be strictly less than its parent's remaining budget). A call with no configured timeout is a `oracle:"hard"` finding.
  - **Retry only on idempotent operations with storm defense.** Retries are applied **only** to `GET`/`PUT`/`DELETE`-shaped operations or operations carrying an **idempotency key**. Every retry policy uses **exponential backoff** (e.g. base 200ms, multiplier 2ⁿ, maximum cap — never unbounded growth) **plus jitter** (full or decorrelated — to de-synchronize a thundering herd of callers hitting the same recovering dependency simultaneously) **plus a max-attempts cap** (e.g. 3 attempts) **plus a retry budget** (a token-bucket or percentage cap — retries cannot exceed X% of total request volume) so that a degraded dependency is not amplified into a **retry storm**. A retry on a non-idempotent `POST` with no idempotency key is a `oracle:"hard"` finding (double-charge / double-submit risk). A retry with no jitter on a shared dependency is a `oracle:"hard"` finding (thundering-herd amplification). A retry with no budget on a degraded path is a `oracle:"hard"` finding (storm amplification).
  - **Circuit-breaker on every path to a slow/unreliable dependency.** The breaker transitions: `CLOSED` (normal; monitors error rate or consecutive failure count) → `OPEN` after **5 consecutive failures** (traced to QAS availability driver) **or error-rate ≥ 50% over a 60s sliding window** (traced to QAS performance driver; both thresholds must be instantiated per-interaction against the relevant `QAS-*` scenario before the gate passes — a design that leaves these as unconfigured defaults is a `oracle:"hard"` finding) → `HALF-OPEN` after a configured probe interval (let a limited number of probe requests through; if they succeed, transition back to CLOSED; if they fail, back to OPEN). The probe interval must be ≤ the dependency's expected recovery time as stated in the availability scenario (e.g. 30s for a fast-recovering internal callee; up to 60s for a slow external provider). The breaker is the primary **cascading-failure** defense: it converts a slow callee into a fast, bounded failure at the caller, stopping the slow-callee-exhausts-caller-pool cascade. A critical-path cross-service call with no breaker where the callee can hang is a `oracle:"hard"` finding.
  - **Bulkhead isolates per dependency.** Each external or cross-service dependency is assigned its own **connection pool / thread pool / concurrency semaphore** so that saturation of one dependency cannot exhaust the shared resource pool and starve unrelated call paths. The *Titanic* metaphor: compartments contain flooding. The bulkhead's pool size is sized to the scenario's concurrency target, not left at a driver default that shares the pool with everything else. A design where a single shared pool (one database driver's connection pool, one HTTP client with no concurrency limit) fronts all dependencies simultaneously, and no driver requires one path to survive another's failure, is not a finding; a design where one slow dependency can starve an unrelated critical path with no isolation **when a driver requires that path to survive** is a `oracle:"hard"` finding.
  - **Rate-limiter protects the producer.** The service applies a **rate-limiter** (token bucket or leaky bucket) at its own ingress to shed excess load early — a `429 Too Many Requests` with a `Retry-After` header — rather than accepting work it cannot complete within the latency budget. The rate-limiter fires *before* the internal resource is exhausted; it is the first line of defence against **overload collapse** (the failure mode where accepting more work causes overall throughput to drop). The rate ceiling is derived from the scenario's demonstrated throughput capacity and the stated concurrency target in `project-architecture-config.md` — it is **not** an arbitrary constant; a rate-limiter with a ceiling that is not traced to a `QAS-*` scenario or a stated capacity constraint is a `oracle:"soft"` finding (ceremony with no driver). As a concrete calibration anchor: derive the token-bucket refill rate from `(throughput_SLO_rps × headroom_factor)`, e.g. if the system's stated peak load target is 50 RPS and demonstrated capacity is 80 RPS, set the limiter at 60 RPS with a burst cap of 80 — both values must be recorded with their `QAS-*` driver ref before the gate passes.
  - **Load-shedding under systemic overload.** When the system is overloaded beyond what rate-limiting on individual callers can handle (queue depth rising, latency budget blowing, error rate climbing), the system **sheds load** — rejects low-priority work early (admission control) to protect the core critical path. Load-shedding is triggered by **at least one measurable signal with a concrete threshold traced to a `QAS-*` driver** — for example: queue depth ≥ `max_queue_depth` (sized to `throughput_SLO_rps × max_acceptable_queuing_latency_s`), CPU utilisation ≥ 80% of demonstrated capacity, or p99 latency ≥ 2× the scenario's SLO response measure (e.g. if QAS-PERF-01 states `dashboard render p99 ≤ 2s`, the shedding trigger is p99 ≥ 4s). A load-shedding trigger that leaves all three signals as uninstantiated placeholders is a `oracle:"hard"` finding — the gate requires at least the primary trigger to carry its driver-traced value before the design artifact is accepted. Load-shedding is a last-resort escalation above rate-limiting; it is a structural behaviour, not an operational runbook step.
  - **Fallback or explicit fail-fast — never a silent hang.** Every call where a breaker fires OPEN or a timeout expires has a defined **fallback**: a stale cached response, a queued-for-later write, a degraded-feature placeholder, or an **explicit fail-fast** (a structured error to the caller with a clear reason and no retry). "Let it hang" is not a fallback. The fallback itself must have a correctness bound (e.g. stale-read staleness window, queued-write durability guarantee). Graceful degradation means the system sheds the failing feature and keeps the core path serving — not an all-or-nothing outage.
  - **Retry-storm and cascading-failure traps are explicitly defended.** The **retry-storm trap**: naïve retries under load amplify requests onto a degraded dependency, turning a brownout into an outage; defence = jitter + budget + breaker. The **cascading-failure trap**: a slow dependency holds caller threads/connections; the caller's pool exhausts; callers of the caller time out; the outage spreads upstream; defence = timeout (fail fast before pool exhaust) + breaker (stop sending once the dependency is degraded) + bulkhead (pool isolation so the cascade stops at the boundary). These two failure modes must be named in every retry and breaker decision (they are **tradeoff points** per `../../shared/evaluation-method.md`).
  - **Fallacies honored** (`../../shared/fallacies-and-tradeoffs.md`): every remote call accounts for #1 (unreliable network — timeout+retry+breaker+fallback), #2 (non-zero latency — a latency budget allocated per hop, child timeout < parent budget), #3 (finite bandwidth — pagination/projection/backpressure on large payloads), #5 (changing topology — no hardcoded hosts, connection re-establishment). A design that assumes any of these away fails this oracle.

- **soft:**
  - **Conceptual integrity of the composed pattern set.** The patterns compose into one coherent failure strategy across all services — not three call sites each doing retries a different way, not a breaker on some paths and nothing on equivalent paths. A single resilience library or middleware carries the policy (conceptual integrity per Brooks); bespoke per-call implementations of the same pattern are a craft gap.
  - **Each pattern is sized to a real failure mode, not cargo-culted.** A bulkhead on an in-process call where pool exhaustion is impossible is ceremony with no driver; a breaker with a threshold of 1 consecutive failure is hair-trigger and causes nuisance tripping. Every parameter (timeout value, max-attempts, breaker threshold, pool size) traces to a scenario's latency budget or an availability driver — not an arbitrary constant copied from a tutorial. Cargo-culting penalised; right-sizing rewarded.
  - **Observable.** Breaker state (CLOSED/OPEN/HALF-OPEN), retry counts, timeout rates, and load-shed counts are exported as metrics (cross-link `../40-cross-cutting/observability-architecture.md`) — the pattern is not a black box; its state is part of the system's health signal.

## Standards & techniques

- **Nygard *Release It!* (Michael Nygard)** — the primary source for stability patterns. Timeout, circuit-breaker, bulkhead, and fail-fast are described with their failure modes, anti-patterns (the *Cascading Failure*, the *Integration Points* chapter), and composition rules. Reference by name.
- **The Azure cloud-design-pattern catalog** (docs.microsoft.com/azure/architecture/patterns/) — the canonical named-pattern source for retry, circuit-breaker, bulkhead, throttling, queue-based load leveling, and ambassador patterns. Reuse the named pattern before inventing a bespoke mechanism (`../../shared/guardrails.md` §3).
- **Exponential backoff with jitter** (AWS: "Exponential Backoff And Jitter", aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/) — the canonical reference for de-synchronised retry under contention. Full jitter: `sleep = random_between(0, min(cap, base * 2^attempt))`. Decorrelated jitter: `sleep = random_between(base, previous_sleep * 3)`. Both prevent the thundering-herd amplification that equal-backoff retries produce.
- **Retry budget / token bucket** — a token bucket (rate = `budget_fraction * rps`, burst = `max_outstanding_retries`) caps total retry volume as a function of total traffic volume; a simpler percentage cap (`retries ≤ 10% of requests`) is acceptable where a full token-bucket implementation is not justified by the driver. Either form must be present on any retry policy where the callee can be degraded.
- **Circuit-breaker state machine** (closed/open/half-open) — parametrised by: **error-rate threshold** (e.g. ≥ 50% failure rate over a sliding 60s window) *or* **consecutive-failures threshold** (e.g. 5 consecutive failures), **open-duration** (how long to stay open before probing, e.g. 30s), **half-open probe count** (e.g. 1–3 requests). Parameters must be traced to the scenario's RTO and the dependency's expected recovery time — a 30s open window on a third-party payment provider known to recover in ≤ 60s is defensible; the same window on a database primary is not (the RTO driver sets the bound). Hystrix, resilience4j, and language/framework-specific retry-and-breaker libraries (e.g. `tenacity` + `circuitbreaker` in Python, `polly` in .NET) are example implementations; the choice is stack-specific and traces to `project-architecture-config.md`.
- **Bulkhead sizing** — connection/concurrency pool per dependency sized by: `pool_size ≥ (max_concurrent_requests_to_that_callee) + headroom`. For a database driver: the max-pool-size setting per client instance; for outbound HTTP calls: a per-host connection limit on the HTTP client session. Never share a single pool across callees where a driver requires one callee's saturation to not affect another.
- **Rate-limiter algorithms** — **token bucket** (allows bursts up to bucket capacity, then throttles at the refill rate — preferred for API ingress) and **leaky bucket** (smooths bursty traffic into a constant output rate — preferred for protecting a resource with a fixed service rate). Token bucket is the most common choice for HTTP rate-limiting (a language/framework rate-limiting middleware, or a gateway-level directive such as nginx `limit_req_zone`). The rate limit is derived from the scenario's target throughput and the system's demonstrated capacity, not arbitrary.
- **Load-shedding triggers** — a measurable signal (e.g. `len(request_queue) ≥ max_queue_depth`, CPU utilisation ≥ 80%, p99 latency ≥ 2× SLO) triggers admission control (return 503 with `Retry-After`). An async-capable web framework naturally defers work; explicit backpressure (a semaphore or queue-depth gate at the route level) is the structural mechanism. Load-shedding is a last-resort escalation above rate-limiting; it fires after the rate-limiter has already rejected excess callers.
- **Graceful degradation + feature-toggle** — the fallback strategy for each external dependency is a designed path: (a) **stale-read** (serve cached data with a staleness window); (b) **queued write** (accept the write, persist locally, retry when the callee recovers — requires idempotency on re-delivery); (c) **degraded-feature placeholder** (return a partial response flagging the missing data — e.g. document-extract field marked `status: pending`); (d) **fail-fast / explicit error** (return a structured 503 immediately). The choice among these is a decision traced to the scenario's consistency + availability tradeoff (`../../shared/fallacies-and-tradeoffs.md` → *Consistency ↔ Availability/Latency*).
- **The 8 fallacies** (`../../shared/fallacies-and-tradeoffs.md`) — the law this pattern set enforces. Cited by number in each pattern's failure-mode entry: #1 → timeout+retry+breaker+fallback; #2 → per-hop latency budget (child timeout < parent); #3 → pagination/projection/backpressure; #5 → no hardcoded hosts, connection re-establishment.
- **Composition order** — the patterns are not applied independently; they compose in a defined application order around each call: ①Rate-limiter (ingress gate, protects the producer) → ②Bulkhead (resource isolation, protects the caller's pool) → ③Timeout (bounds the call duration) → ④Circuit-breaker (fail-fast if the callee is degraded) → ⑤Retry (only if idempotent + no breaker open + within budget) → ⑥Fallback/fail-fast (on breaker-OPEN or final timeout/retry exhaustion). Load-shedding is ① at the system boundary; graceful degradation is the outcome of ⑥. This order prevents the retry-storm (③ fires before ⑤ can amplify) and the cascading-failure (④ stops sending to a degraded callee before ② is exhausted).
- **MADR ADRs** (`../../shared/adr-template.md`) — each pattern adoption and each parameter-level decision (a timeout value, a breaker threshold, the choice between full-jitter and decorrelated-jitter, the decision to not retry a given operation) is an ADR with context · ≥2 options · decision · positive *and* negative consequences, traced to a `QAS-*` driver ref.
- **Diagrams-as-code** (`../../shared/diagramming-standards.md`) — a C4 dynamic/sequence view of the composed failure path (breaker fires OPEN → fallback activates) for each high-priority scenario, sync calls as solid lines, async as dashed, breaker state annotated.

## Step sequence

- **audit:** Frame the availability/performance drivers → enumerate the **interaction inventory** from the C4 L2 + API contracts (every cross-process call site: caller, callee, protocol, idempotent?) → walk **each** call through the composition order (rate-limiter? bulkhead? bounded timeout? breaker? retry-if-idempotent-with-jitter+budget? fallback/fail-fast?) → check the fallacies-honored oracle per call → flag: missing/unbounded timeouts, retries on non-idempotent ops, retries with no jitter or no budget (storm risk), missing breakers on critical paths, shared pools with no bulkhead where a driver requires isolation, fallback absent, load-shedding absent where overload is a plausible failure mode → **name the retry-storm and cascading-failure risks explicitly** in the risk register — read-only, authors nothing, edits no source.
- **design:** **Frame** (extract/confirm the availability + performance scenarios as 6-part QAS with response measures; build the interaction inventory, read-only; surface unstated availability targets as open driver questions — do not assume) → **Explore** (read-only: ≥2 divergent resilience postures per contested interaction — e.g. for a payment/settlement integration: *sync + timeout(45s) + breaker(5 failures/60s window) + fail-fast + manual retry UI* vs *async queue + idempotent consumer + status-poll endpoint*; each posture specifies its timeout value, backoff parameters, breaker thresholds, bulkhead pool size, fallback strategy, and the failure modes it prevents, *not* the same call recoloured) → **Evaluate** (`architecture-evaluator`, ATAM, order-swapped to kill position bias: score each posture against the scenarios + hard oracles; identify the **sensitivity points** — e.g. "p99 is sensitive to the retry budget ceiling: above 15% of RPS the dependency is amplified into outage" — and the **tradeoff points** — e.g. "async-queue *improves* availability and blast-radius but *worsens* immediacy and adds eventual-consistency + a broker; sync-with-breaker *improves* immediacy but *worsens* blast-radius and requires the breaker to trip before the caller is freed"; penalise accidental complexity — no breaker on an in-process DB call where pool exhaustion is not a scenario; pick the winner, graft from runners-up) → **Document** (the **one writer**, `solution-architect`: author the **pattern catalogue** and the **per-call composition table** into `architecture/<slug>/design-doc.md` — for each interaction: the pattern stack applied, the parameter values with their driver traceability, the failure modes prevented, the fallback strategy; the C4 dynamic failure-path views in `diagrams/`; one **MADR ADR per significant decision** in `adr/`; the retry-storm / cascading-failure risk entries in `tradeoffs.md`; the pattern-application fitness-function specs in `fitness-functions.md`; reuse named patterns, preserve one coherent resilience policy) → **Verify** (the hard gates below; evaluator re-scores from the artifacts, not the prose) → loop ≤ 10 or pass.

## Assertions & exit gate

- Every call in the interaction inventory has, in composition order: ① a rate-limiter at the producer ingress (where overload is a plausible scenario); ② a bulkhead (per-dependency pool) where a driver requires one callee's saturation to not affect another path; ③ a **bounded** connect *and* read timeout (no `None`, no driver default, value < parent latency budget); ④ a circuit-breaker (or an explicit decision not to use one, with the blast-radius risk accepted) on every path to an external / slow-degradable callee; ⑤ a retry policy **if and only if** the operation is idempotent, with exponential backoff + jitter (full or decorrelated) + max-attempts cap + retry budget — or an explicit no-retry recorded with reason; ⑥ a fallback or explicit fail-fast (no silent hang).
- **No retry on a non-idempotent operation** without an idempotency key; every retry uses **jitter** and a **retry budget** (zero-defence-against-storm retries block regardless of backoff).
- **No unbounded synchronous call chain**: summed worst-case timeouts across a sync chain ≤ the originating scenario's latency budget; each child timeout strictly less than its parent's remaining budget.
- **Retry-storm and cascading-failure traps explicitly named** in the risk register with their structural defence (jitter+budget+breaker) — not just listed as a risk without a disposition.
- **Fallacies honored** (#1 timeout+retry+breaker+fallback, #2 per-hop latency budget, #3 pagination/backpressure, #5 no hardcoded hosts) — each accounted for where it applies to the system's topology.
- Every pattern parameter (timeout value, backoff base, breaker threshold, pool size) traces to a `QAS-*` scenario or a stated constraint in `project-architecture-config.md` — no magic constants.
- Every significant pattern decision is a **complete MADR ADR** (context · ≥2 options · decision · positive **and** negative consequences) traced to a `Driver refs`; any failure-path diagram **compiles** without error (`../../shared/diagramming-standards.md`).
- **Gate:** hard oracles green (every call's composition complete + idempotent-only retries with jitter+budget + bounded sync chain + storm+cascade traps defended + fallacies honored + parameters driver-traced + ADRs complete + diagrams compile) AND (design) rubric mean ≥ 0.8 (conceptual integrity of the composed policy / right-sized to real failure modes / observable / tradeoff-honest).

## Guardrails

Per `../../shared/guardrails.md`: **propose & document — never write product code and never configure a live retry/breaker/rate-limiter**; the writer authors the pattern catalogue, the composition table, the ADRs, and the fitness-function *specs* into `architecture/<slug>/`; the **controller** wires the timeout/retry/breaker configuration in the backend service code (e.g. a retry-decorator library, circuit-breaker annotations, the database driver's connection-pool-size setting, a rate-limiter middleware) per the stack named in `project-architecture-config.md`. A runnable fitness-function *check script* into `tests/`/`fitness/` is the only thing that may land outside the architecture folder (that is a verification artifact, not product behavior). **Ground every pattern adoption in a driver** — no breaker/bulkhead/rate-limiter on a call where no availability or latency scenario demands it is accidental complexity; flag and reject it; right-size to the team-topology and scale constraints stated in `project-architecture-config.md` (e.g. a small team with no high-RPS driver today prefers boring, well-understood tech over novel frameworks). **Reuse named patterns** (Nygard, Azure cloud-design-pattern catalog) before inventing a bespoke resilience mechanism; prefer the project's existing stack's idiomatic retry/breaker libraries (per `project-architecture-config.md`) over a novel framework with no driver justifying it. **Record the tradeoff and the rejected alternatives** on every decision — what the pattern costs (latency overhead of the breaker check, operational complexity of managing pool sizes, eventual-consistency tradeoff of async-queue fallback) and what it beat. **One writer** authors the pattern catalogue and composition table so the resilience policy has conceptual integrity — not a different retry library per endpoint, not a different jitter strategy per team member. The evaluator scores from the *artifacts*, never edits them. Read-only on any live system; never inject failures into a real environment — the architect specifies the failure-mode coverage; `qa-agent` runs the chaos/resilience verification against a seeded sandbox. Never write to project memory.
