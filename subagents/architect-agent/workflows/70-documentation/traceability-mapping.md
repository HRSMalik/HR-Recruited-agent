# traceability-mapping

**Group:** 70-documentation
**Runs as:** subagent: ../.claude/agents/solution-architect.md
**Mode:** audit + design   ·   **Default model:** opus

## Purpose
Build the **traceability matrix** that proves the architecture is fully grounded: the four-link chain **driver → decision (ADR) → component → fitness-function**, walked in both directions so that no requirement is unaddressed and no decision is unjustified. This is the **backbone of the hard-oracle gate** — every other flow's "every ASR traces to a decision, every decision to a driver" claim is only as real as this matrix, which makes it mechanically checkable instead of asserted. Answers: *does every architecturally-significant requirement reach a component that satisfies it and a fitness function that guards it, and does every significant decision trace back to a driver that demanded it — or is it accidental complexity?* The matrix is the artifact that lets the evaluator and the review-checklist verify the traceability gate without re-deriving the design.

## Inputs & preconditions
- From `project-architecture-config.md`: where the architecture description / ADR log / diagram source / fitness-function register live (`architecture/<slug>/` — `design-doc.md`, `quality-attributes.md`, `adr/`, `diagrams/`, `fitness-functions.md`, `tradeoffs.md`); the modular-monolith shape (so "component" means a **module boundary** — auth/RBAC, claim-lifecycle, company-onboarding, users, audit — not a service in a mesh); the locked rule that an unjustified decision is accidental complexity to be flagged, not shipped.
- The **drivers**: the ASR catalog from `../00-drivers/architecture-drivers.md` (`architecture/<slug>/adr/0001-asr-catalog.md`) is the **root of the matrix** — every ASR is a row that must reach a decision, a component, and (for an invariant) a fitness function; the utility tree from `../00-drivers/quality-attribute-scenarios.md` (`quality-attributes.md`) supplies the prioritized `QAS-*` scenarios with response measures that the `(H,*)` rows are checked hardest against.
- The **decisions**: the ADR log from `architecture-decision-records.md` (`architecture/<slug>/adr/`) — each accepted MADR ADR's `Driver refs` (back-trace) and `Decision outcome` (the component(s) it shapes) and `Follow-ups / fitness functions` (forward to the guard); the **fitness-function register** from `../60-evaluation/fitness-functions.md` (`architecture/<slug>/fitness-functions.md`) supplying the `decision_ref`/`driver_ref` on each function.
- Target: the existing/proposed architecture package under review or being completed; the relevant method is requirements traceability + the coverage matrix. Preconditions: the ASR catalog, the utility tree, and the ADR log are all readable. A run with **no ASR catalog has no matrix root** — it downgrades scope and says so (`status:error`, not a fabricated matrix); a run with ADRs but no utility tree can still back-trace decisions but cannot prove forward-coverage of response measures — it reports that gap rather than guessing.

## Oracle (source of truth)
The architecture package's **own cross-references** — the matrix is not an opinion, it is a join over the ASR catalog, the ADR `Driver refs`/`Follow-ups`, the component model in `design-doc.md`/the C4 container view, and the fitness-function register; a cell is either present (a real id-to-id link) or it is a hole. The grounding rule (`quality-attribute-rubric.md`): **no number + no traced driver = a preference, not a finding** — a "missing trace" claim must name the specific ASR/decision id and the link that is absent, never a vibe that "this feels untraced."

- **hard:**
  - **No orphan requirement (forward coverage).** Every ASR in the catalog, and every prioritized `(H,*)` quality-attribute scenario in the utility tree, traces **forward** to (a) at least one **decision** (ADR) that addresses it AND (b) at least one **component/module** that satisfies it. An ASR or `QAS-*` that reaches no component is an orphan requirement — a structure-shaping concern with nothing built for it — and **blocks** (`oracle:"hard"`). The forward chain must be complete: `driver → decision → component`.
  - **No unjustified decision (back coverage).** Every significant **decision** (every accepted ADR, every component/container that appears in the design-doc or the C4 container view, every introduced technology beyond the stack lock) traces **back** to at least one driver — a `QAS-*` scenario or a named constraint — that demanded it. A decision/component no driver demands is **accidental complexity** (a box with no reason, a datastore no scenario needs) and **blocks**. The back chain must be complete: `component → decision → driver`.
  - **Invariants are guarded (the fitness link).** Every ASR/decision that asserts a **structural invariant or a response measure** (every `(H,H)`/`(H,M)` leaf, every ADR with a `Follow-ups / fitness functions` obligation) traces to a **fitness function** in the register (`../60-evaluation/fitness-functions.md`). A key invariant with no guard is unguarded drift — the four-link chain `driver → decision → component → fitness-function` is broken at the last hop and the row is flagged (`oracle:"hard"`).
  - **The matrix is internally consistent + complete.** Every id referenced in the matrix resolves to a real artifact (no dangling `ADR-0099` that doesn't exist, no `QAS-PERF-04` absent from the utility tree); every ASR row and every ADR row appears exactly once; the counts reconcile (`asrs == traced + orphan_requirements`, `decisions == justified + unjustified_decisions`). A matrix that doesn't reconcile is a broken oracle and **blocks**.
- **soft:** the matrix is **complete and honest** — it is the single artifact that lets the `architecture-evaluator` and `architecture-review-checklist` verify the traceability gate **mechanically** (read the matrix, count the holes) rather than re-walking the whole design; the link rationale is legible (each cell says *how* the component satisfies the driver, not just that an arrow exists), and the matrix surfaces the *weak* links (a driver satisfied by a `mock`/not-yet-built module, a decision traced to a driver only by a stretch) rather than papering them green.

## Standards & techniques
- **Requirements traceability** — the discipline of maintaining bidirectional links from a requirement to the artifact that satisfies it and back, so coverage and impact are mechanically checkable. Here the requirement is an **ASR / quality-attribute scenario** (not a feature), and the satisfying artifact is a **decision + component + fitness-function**, not a line of code. **Forward traceability** answers "is every requirement addressed?"; **backward traceability** answers "is every artifact justified?" — both are required; one direction alone hides half the defects.
- **The driver → decision → component → fitness-function chain** — the four-link spine this flow exists to close. *Driver* = a `QAS-*` scenario or constraint from the utility tree (`../00-drivers/quality-attribute-scenarios.md`); *decision* = an ADR (`architecture-decision-records.md`); *component* = a module/container in the design-doc / C4 container view; *fitness-function* = the guard in the register (`../60-evaluation/fitness-functions.md`). A row is **fully traced** only when all four links resolve (or the missing link is an explicit, accepted N/A — e.g. a functional ASR with no measurable invariant legitimately has no fitness function, recorded as N/A, not a hole).
- **Orphan-requirement detection** — the forward-coverage check: a driver (ASR or `(H,*)` scenario) that no component satisfies. An orphan requirement is the most expensive defect to find late — the architecture was *declared* to meet a scenario it has no structure for. The detection is a left-anti-join: `drivers − drivers_with_a_component`.
- **Unjustified-decision detection** — the back-coverage check: a decision/component that no driver demands. This is **accidental complexity** made visible (Brooks; `quality-attribute-rubric.md` → simplicity) — a service, a datastore, a cache, a config knob, a microservice split that exists for résumé reasons rather than a driver. The detection is the mirror anti-join: `decisions − decisions_with_a_driver`. On this project specifically, an unjustified decision is the *primary* over-engineering smell — flag any proposed Kafka/k8s/multi-region/extra-datastore that no `QAS-*` demands (`project-architecture-config.md` → locked rules).
- **Coverage matrices** — the deliverable form: a matrix (rows = drivers/ASRs, columns = decisions/components/fitness-functions, or a normalized link table) where a populated cell is a real id-to-id reference and an empty cell is a coverage hole. Two summary tallies fall out of it: **forward coverage** (% of drivers reaching a component) and **back coverage** (% of decisions reaching a driver) — both must be 100% (or every exception an explicit accepted N/A) to pass the gate.
- **Impact analysis (a by-product)** — once the matrix exists, it answers "if `QAS-AVAIL-02` changes, which ADRs/components/fitness-functions are affected?" and "if we drop this module, which drivers go un-served?" — the matrix is the change-blast-radius oracle, reused by `migration-modernization` and the review checklist. Author it so a column/row slice is a clean impact set.

## Step sequence
- **audit:** Frame the chain (read the ASR catalog + utility tree + ADR log + component model + fitness register, read-only) → **join** the four artifacts into the link table (each ASR/`QAS-*` → its ADR(s) → its component(s) → its fitness-function(s), pulling the links from the ADR `Driver refs`/`Follow-ups`, the design-doc component model, and the register's `driver_ref`/`decision_ref`) → run the two anti-joins: **orphan-requirement** (a driver reaching no component) and **unjustified-decision** (a decision/component reaching no driver) → run the **fitness-link** check (every `(H,*)` leaf / invariant ADR reaches a function) → run the **consistency** check (every referenced id resolves; counts reconcile) → emit findings (each names the specific broken link: the orphan ASR id, the unjustified ADR/component, the unguarded invariant) — read-only, authors no matrix, edits no source.
- **design:** **Frame** (confirm the ASR catalog, utility tree, and ADR log are present and consistent — read-only) → **Explore** (this flow has little to diverge on — the matrix is a join, not a design space; "explore" here = decide the matrix *form* that best serves mechanical checking: a normalized link table vs a dense grid, and whether N/A links need a recorded reason — pick the form that makes a hole obvious at a glance) → **Evaluate** (the skeptical `architecture-evaluator` checks the *draft* matrix for honesty: are weak links — a `mock`/unbuilt module counted as satisfying a driver, a stretch back-trace — surfaced or papered green? are the N/A exceptions legitimate or holes in disguise? order-swapped to kill bias; it does **not** add or remove links, only judges whether the matrix tells the truth) → **Document** (ONE writer authors the matrix into the architecture package — typically appended to/linked from `architecture/<slug>/design-doc.md` as the **traceability section**, or a standalone `architecture/<slug>/traceability.md` — one row per ASR/`QAS-*` with the columns `driver_id · decision_ref(s) · component(s) · fitness_function(s) · status(traced|orphan|N-A) · rationale`, plus the inverse table of decisions/components → drivers for the back-coverage view; reuses the ids already minted in the catalog/ADRs/register, never re-numbering) → **Verify** (the verification hard gates: **forward coverage** = 100% of ASRs/`(H,*)` reach a component or are an explicit accepted N/A; **back coverage** = 100% of decisions/components reach a driver; **fitness-link** = every invariant reaches a function; **consistency** = every id resolves and the counts reconcile; the `architecture-evaluator` re-scores the *matrix*, not prose) → loop ≤ 5 or pass. The writer authors **only** the matrix — it never edits the ASR catalog, the ADRs, or the register to *force* a green; a hole is reported back to the controller to fix at the source (add the missing decision, remove the unjustified component, write the missing fitness function), then the matrix is re-joined.

## Assertions & exit gate
- Every ASR in the catalog and every `(H,*)` `QAS-*` scenario in the utility tree appears as a row; none is missing (the matrix root = the full driver set).
- **Forward coverage = 100%:** every driver row traces to ≥1 decision AND ≥1 component, OR is an explicit accepted N/A with a recorded reason. Zero orphan requirements (a driver reaching no component blocks).
- **Back coverage = 100%:** every accepted ADR and every component/container in the design-doc / C4 container view traces back to ≥1 driver (a `QAS-*` or a named constraint). Zero unjustified decisions (a box no driver demands blocks as accidental complexity).
- **Fitness-link present:** every `(H,H)`/`(H,M)` leaf and every ADR carrying a structural invariant / response measure traces to ≥1 fitness function in the register; an unguarded key invariant is a `major+` finding.
- **Consistency + reconciliation:** every id referenced in the matrix resolves to a real artifact (no dangling/typo refs); every ASR and every ADR appears exactly once; `asrs == traced + orphan_requirements` and `decisions == justified + unjustified_decisions` both reconcile.
- **Honesty:** weak links are surfaced, not hidden — a driver satisfied only by a `mock`/not-yet-built module (documents/release/payment/closure on this project) is marked as a **planned/seam** trace, not counted as fully satisfied; every N/A is a real N/A, not a disguised hole.
- **Gate:** hard oracles green (forward coverage 100%, back coverage 100%, fitness-link complete, matrix consistent + reconciled) AND (design) evaluator rubric mean ≥ 0.8 on completeness + honesty.

## Output
Write `artifacts/traceability-mapping/report.json` per `shared/report-format.md` — findings per `shared/finding-schema.md` (`type:"documentation"` for a matrix consistency/dangling-ref defect; `type:"driver"` for an **orphan requirement** — `oracle:"hard"`, `driver_ref` = the un-served ASR/`QAS-*`, `evidence` = "the driver reaches no component in the design-doc / C4 container view"; `type:"evolvability"` or `type:"risk"` for an **unjustified decision** — `oracle:"hard"`, `evidence` = "this ADR/component traces to no driver → accidental complexity", `recommendation` = "remove it, or surface the driver it actually serves and add the ASR"; `type:"evolvability"` for an **unguarded invariant** — the fitness-link hole). The report's `verification.traceability` block carries the reconciliation counts `{ "asrs": N, "traced": N, "orphan_requirements": 0, "unjustified_decisions": 0 }` (per `shared/report-format.md`), and `verification.artifacts[]` lists the authored matrix path. For **design** mode the authored matrix — the human-facing deliverable — lands in **`architecture/<slug>/`** (the traceability section of `design-doc.md` or a standalone `traceability.md`) per `project-architecture-config.md`, never scattered; the run report stays in `artifacts/`. The matrix is the artifact `../60-evaluation/architecture-evaluation-atam.md` and `architecture-review-checklist` read to verify the traceability gate mechanically.

## Guardrails
Per `shared/guardrails.md`: **propose & document — never write product code.** This flow authors the *traceability matrix* (an architecture artifact) into `architecture/<slug>/`; it never edits `src/**`/`frontend/**`/`backend/**`, never runs infra. **It never edits the source artifacts to force a green** — the matrix only *reports* the joins; a hole (an orphan requirement, an unjustified decision, an unguarded invariant) is reported back to the controller, who fixes it at the source (`../00-drivers/architecture-drivers.md` for a missing ASR, `architecture-decision-records.md` for a missing/unjustified ADR, `../60-evaluation/fitness-functions.md` for a missing guard), then the matrix is re-joined — the traceability agent does not silently invent a link to close a hole. **Ground every cell in a real id-to-id reference** — a trace with no resolvable id on both ends is not a trace; an "untraced" claim must name the specific id and the absent link, never a vibe. **One writer** authors the matrix so it has conceptual integrity (consistent ids, one coherent link form). The audit path **reads** the architecture package but writes nothing back. **Never write to project memory** — the matrix and the findings carry the lessons; the controller curates memory.
