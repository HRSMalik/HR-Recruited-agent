# architecture-description-arc42

**Group:** 70-documentation
**Runs as:** subagent: ../.claude/agents/solution-architect.md
**Mode:** audit + design
**Default model:** sonnet

## Purpose

Write the architecture description — the stakeholder-oriented document covering context, building blocks, runtime behaviour, deployment, crosscutting concepts, decisions, and risks — structured as the **arc42 12 sections** (ISO/IEC/IEEE 42010:2022 viewpoints), supplemented by the **Views-and-Beyond** module/C&C/allocation view types. Answers: *do we have a complete, traceable, non-drifted record of what we built, why, and for whom?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack + deployment target, the `architecture/<slug>/` artifact home, diagram tooling (Mermaid + PlantUML+C4-PlantUML), compliance regime, team topology, constraints, and all existing architecture artifacts to read before writing — `adr/`, `diagrams/`, `quality-attributes.md`, `threat-model.md`, `capacity-model.md`, `tradeoffs.md`.
- The **architecture drivers**: the prioritized utility tree (`quality-attributes.md` — 6-part scenarios with response measures per `workflows/00-drivers/quality-attribute-scenarios.md`); constraints and assumptions log (`constraints-assumptions-risks` flow); the functional scope from the domain analysis.
- **Stakeholder register**: the audiences arc42 section-1 must enumerate — at minimum: technical leads (container + component views, ADRs), operations/SRE (deployment + crosscutting/operability, failure modes), security reviewer (threat model, trust boundaries, STRIDE controls), product/business (context view, business rules, risks), new team members (sections 1–3 + rationale). Each group's concerns must map to a named view; ISO/IEC/IEEE 42010:2022 §5.3 requires that every concern is covered by at least one view. (Under the 2022 edition these groups are "entities of interest," not "stakeholders of the system of interest.")
- The **ADRs** authored by `workflows/70-documentation/architecture-decision-records.md`, the **C4 diagrams** authored by `workflows/70-documentation/c4-model-diagrams.md`, the **design-doc/RFC** authored by `workflows/70-documentation/design-doc-rfc.md`. The arc42 description is the **integrating document** — it references these; it does not duplicate them.
- Preconditions: `project-architecture-config.md` read; the utility tree present (if absent, this flow downgrades to `status:error` and requests the `quality-attribute-scenarios` flow runs first — a description without drivers is ceremony, not architecture); the ADRs + diagrams referenced exist and compile; the `solution-architect` is the only writer (per `../shared/guardrails.md` §4).

## Oracle (source of truth)

The description is judged against two external sources: (1) the **ISO/IEC/IEEE 42010:2022 architecture-description standard** — entities of interest + concerns → viewpoints → views → correspondences (note: 2022 edition replaced "system of interest" with "entity of interest" and dropped the UML metamodel annex; architecture evaluation authority now sits with ISO/IEC/IEEE 42030:2019 and process in 42020, both cross-referenced in `../shared/evaluation-method.md`) — and (2) the **arc42 section completeness contract** (all 12 sections present and non-empty for the sections the system demands). Drift against the ADRs or diagrams is a structural failure, not a style note.

**hard:**
- The description covers **every view the stakeholders' concerns demand** (`views_labeled_stakeholders_concerns`): Context (arc42 §3 — scope + external actors; ISO/IEC/IEEE 42010:2022 entity-of-interest/concern coverage); Building-block/static structure (arc42 §5 — module view per Views-and-Beyond); Runtime (arc42 §6 — C&C / dynamic scenarios, at least one per prioritized quality-attribute scenario in the utility tree); Deployment (arc42 §7 — allocation view, C4 deployment diagram as diagrams-as-code); Crosscutting concepts (arc42 §8 — recurring architectural decisions: auth/RBAC, error handling, logging/tracing, config/secrets, data validation). A missing view for a stakeholder concern with no accepted deferral ADR blocks.
- Every view **names its entities of interest and concerns** (ISO/IEC/IEEE 42010:2022 §5.3): "Deployment view — audience: ops/SRE, concerns: AZ topology, container restart policy, secrets injection, datastore connection failover." An unlabeled view fails this oracle (`views_labeled_stakeholders_concerns`).
- **Rationale is captured, not just structure** (`rationale_coverage`): each architecturally-significant building block + crosscutting concept traces to either an ADR (the *why*) or an explicit in-line rationale statement. A "what" without a "why" blocks for any component that represents a driver-satisfying decision.
- **Links to ADRs + diagrams + utility tree are present and non-broken** (`adr_refs_non_broken`, `diagram_refs_non_broken`): every section that references an ADR cites it by ID (`ADR-NNNN`); every referenced diagram is identified by its diagrams-as-code path under `architecture/<slug>/diagrams/`; the utility tree reference in §2 (architecture goals) points to `quality-attributes.md`. A broken cross-reference is a documentation defect, not a cosmetic.
- **No drift between the description and the authoritative artifacts** (`no_drift_vs_adrs`, `no_drift_vs_diagrams`): the containers named in arc42 §5 match the containers in `c4-model-diagrams.md`; the decision outcomes in arc42 §9 match the `accepted` ADRs; the risks in arc42 §11 are a superset of or identical to the risk register in `tradeoffs.md`. Any named inconsistency is a `major` finding (`type:"documentation"`, `oracle:"hard"`).
- **Diagrams referenced in the description are diagrams-as-code** (`diagrams_compile`): Structurizr DSL, PlantUML+C4-PlantUML, or Mermaid — and render without error per `../shared/diagramming-standards.md`. An embedded PNG or an un-compiled diagram source blocks the `diagrams_compile` hard oracle.
- **All 12 arc42 sections are present** (`arc42_sections_complete`): §1 Introduction+goals, §2 Constraints, §3 System scope+context, §4 Solution strategy, §5 Building-block view, §6 Runtime view, §7 Deployment view, §8 Crosscutting concepts, §9 Architecture decisions, §10 Quality requirements, §11 Risks+technical debt, §12 Glossary — even if some are deliberately marked "N/A with rationale" for a section the system does not require (e.g., §12 may be thin for a small team). A section silently absent — no content AND no "N/A" rationale — blocks.
- **Arc42 §4 states the AKF scale-cube axis chosen** (`akf_axis_stated`): which axis the solution targets (x-axis: horizontal cloning; y-axis: functional decomposition; z-axis: data partitioning) and the concrete driver from the utility tree that would prompt a change to that axis. A §4 that does not name an axis blocks.
- **STRIDE per trust boundary is controlled** (`stride_per_trust_boundary`): every trust boundary in arc42 §8 names a control (or an accepted-risk ADR) for each STRIDE category — Spoofing → authn; Tampering → integrity/signing; Repudiation → audit log; Information disclosure → authz + encryption; Denial of service → rate limits; Elevation of privilege → least-privilege RBAC. A trust boundary with an uncontrolled STRIDE category and no accepted-risk ADR blocks.
- **Arc42 §10 cross-references `workflows/60-evaluation/fitness-functions.md`** (`fitness_functions_referenced`): the description states which architectural characteristics are continuously enforced (e.g. no-cyclic-module-deps, p95-budget per QAS scenario, no-PII-in-logs) and the CI hook that runs them. A quality requirement with no fitness function and no alternative enforcement mechanism is advisory only — flag the gap. A §10 that does not name or link `workflows/60-evaluation/fitness-functions.md` blocks.

**soft:**
- **Right-sized to the audience** (rubric dimension: *simplicity / right-sizedness*): the description is as long as its readers need and no longer. A 40-page doc for a two-service modular monolith is accidental ceremony; a 2-line "context" for a system with five external integrations is under-documented. Penalize both directions.
- **Consistent with the diagrams and ADRs** (rubric dimension: *conceptual integrity*): no stylistic drift — same names, same component labels, same technology names used throughout; Brooks's conceptual integrity runs through every section.
- **A reader can reconstruct what + why from the document alone** (rubric dimension: *rationale followability*): the rationale trail is followable, the constraints are stated, the rejected alternatives are referenced (via ADRs), the open risks are named — without querying the author.
- **Crosscutting concepts (arc42 §8) are design-level, not prose platitudes**: "We use a stateless signed-token scheme (per `project-architecture-config.md`) for authn; the signing secret is env-injected (never in code or diagram); token validation occurs at the API-gateway/middleware layer before any route handler" is a crosscutting concept. "We take security seriously" is not.

## Standards & techniques

- **arc42 (the 12 sections)** — the canonical template for software architecture descriptions: §1 Introduction+goals, §2 Constraints, §3 System scope+context, §4 Solution strategy, §5 Building-block view (decomposition levels), §6 Runtime view (dynamic scenarios), §7 Deployment view, §8 Crosscutting concepts, §9 Architecture decisions, §10 Quality requirements, §11 Risks+technical debt, §12 Glossary. Use the section numbers as structure headers; do not rename them (readers cross-reference by number across projects).
- **ISO/IEC/IEEE 42010:2022** — the entities-of-interest–concerns–viewpoints–views model (Nov 2022 edition): every view exists to address at least one concern of an entity of interest; correspondences (consistency rules between views) must be stated. Key 2022 changes from the 2011 edition: "system of interest" replaced by "entity of interest" throughout; the UML metamodel annex was removed; architecture evaluation is now normatively governed by **ISO/IEC/IEEE 42030:2019** (evaluation) and **ISO/IEC/IEEE 42020** (process) — this workflow owns the AD structure (42010:2022) and cross-references 42030:2019 for the evaluation lane (see `../shared/evaluation-method.md`). The hard oracle for "every view names its entities of interest and concerns" is direct from this standard (§5.3, §5.5).
- **Views-and-Beyond (Clements et al.)** — the three view-type taxonomy: **Module views** (decomposition, layers, uses, generalization — the "what compiles together"; arc42 §5 maps to this), **C&C views** (Component-and-Connector — runtime entities and their connectors; arc42 §6 maps here; maps to C4 Container/Component views), **Allocation views** (mapping of software elements to environment — deployment, work-assignment, install; arc42 §7 maps here; maps to C4 Deployment view). Naming the view type per section clarifies the audience's reading frame.
- **Rationale capture (arc42 §9 + §4)** — explicitly recording the *why* (forces, constraints, alternatives considered) at each significant decision point. The rationale is not the same as the decision: the rationale is the argument that the decision was the best response to the forces. Every arc42 §9 entry is an ADR reference (MADR format, `../shared/adr-template.md`) — the description does not duplicate the ADR, it cites and summarizes.
- **C4 model (Simon Brown)** — the four-level decomposition (System Context L1, Container L2, Component L3, Code L4) provides the diagram vocabulary for arc42 §3 (L1), §5 (L2/L3), §7 (Deployment view). All diagrams-as-code: Mermaid for §3 context (renders inline in Markdown/GitHub); PlantUML+C4-PlantUML for §5 Container/Component and §7 Deployment (richer layout). Per `../shared/diagramming-standards.md`: every element labeled (name + type + one-line responsibility), every relationship labeled and directed (verb + protocol/style), sync vs async visually distinct, trust boundaries drawn.
- **MADR (Markdown Architectural Decision Records)** — arc42 §9 lists ADRs by ID and decision outcome; the full MADR (context · options · decision · consequences) lives in `architecture/<slug>/adr/`. The description section is the index, not the content.
- **STRIDE** — crosscutting security concepts (arc42 §8) must name the control per STRIDE category at each trust boundary (Spoofing → authn; Tampering → integrity/signing; Repudiation → audit log; Information disclosure → authz + encryption; Denial of service → rate limits; Elevation of privilege → least-privilege RBAC). A trust boundary with an uncontrolled STRIDE category, and no accepted-risk ADR, fails the `stride_per_trust_boundary` hard oracle.
- **AKF scale cube** — arc42 §4 (solution strategy) states the scalability axis chosen (x-axis: horizontal cloning; y-axis: functional decomposition; z-axis: data partitioning). For a modular monolith with an enterprise-internal scale target, the right answer is typically x-axis only — state this explicitly and note what driver would prompt a y-axis split (per `project-architecture-config.md` locked rule: no microservices absent a concrete driver). Absence of an explicit axis statement in §4 blocks `akf_axis_stated`.
- **Fitness functions** — arc42 §10 (quality requirements) cross-references `workflows/60-evaluation/fitness-functions.md`; the description states *which* architectural characteristics are continuously enforced (e.g. no-cyclic-module-deps, p95-budget per QAS scenario, no-PII-in-logs) and the CI hook that runs them. A quality requirement with no fitness function and no alternative enforcement mechanism is advisory only — note the gap; absence of the cross-reference blocks `fitness_functions_referenced`.

## Assertions & exit gate

- All 12 arc42 sections present (§1 Introduction+goals, §2 Constraints, §3 System scope+context, §4 Solution strategy, §5 Building-block view, §6 Runtime view, §7 Deployment view, §8 Crosscutting concepts, §9 Architecture decisions, §10 Quality requirements, §11 Risks+technical debt, §12 Glossary); any section marked "N/A" carries an explicit rationale, not a blank.
- Every view (Context §3, Building-block §5, Runtime §6, Deployment §7) names its entities of interest and concerns in the section header or a labeled preamble (ISO/IEC/IEEE 42010:2022 §5.3).
- Every architecturally-significant building block and every crosscutting concept (§8) in the description traces to either a cited ADR (`ADR-NNNN`) or an in-line rationale statement — no "what" without a "why."
- All ADR cross-references in §9 use the zero-padded ID form (`ADR-NNNN`) and each cited file exists under `architecture/<slug>/adr/`; all diagram cross-references point to a diagrams-as-code file (`.puml` / `.md` Mermaid) under `architecture/<slug>/diagrams/` that compiles without error.
- No drift: container/component names in §5 match `c4-model-diagrams.md`; decision outcomes in §9 match `accepted` ADRs; §11 risks are consistent with `tradeoffs.md` risk register.
- Arc42 §4 (solution strategy) states the AKF scale-cube axis chosen (x/y/z) and the driver that would prompt a change to that choice.
- Arc42 §8 crosscutting concepts are design-level specifics (named pattern/tool + the binding point in the codebase), not prose platitudes.
- Arc42 §10 quality requirements cross-references `quality-attributes.md` (the utility tree) and `workflows/60-evaluation/fitness-functions.md` (the enforcement hooks); every prioritized (H,*) scenario from the utility tree appears here.
- **Gate:** all hard oracles green (`arc42_sections_complete`, `views_labeled_stakeholders_concerns`, `rationale_coverage`, `adr_refs_non_broken`, `diagram_refs_non_broken`, `no_drift_vs_adrs`, `no_drift_vs_diagrams`, `diagrams_compile`, `akf_axis_stated`, `stride_per_trust_boundary`, `fitness_functions_referenced`) AND (design) rubric mean ≥ 0.8 (right-sizing, conceptual integrity, rationale followability).

## Output

Write `artifacts/architecture-description-arc42/report.json` per `../shared/report-format.md` — findings per `../shared/finding-schema.md` with `type:"documentation"`, `oracle:"hard"` for any of: missing arc42 section, unlabeled view (no stakeholder+concern statement), rationale gap on a significant building block, broken ADR/diagram reference, drift against ADRs or diagrams, un-compiled diagram, missing AKF axis statement in §4, STRIDE gap on a trust boundary, missing `workflows/60-evaluation/fitness-functions.md` linkage in §10; `oracle:"soft"` for right-sizing violations, platitude crosscutting concepts, readability gaps. The `verification` block records: `artifacts` list (the authored `design-doc.md` path), `traceability` (ASRs traced, orphan requirements, unjustified decisions), `hard_oracles` (all 11 named: `arc42_sections_complete`, `views_labeled_stakeholders_concerns`, `rationale_coverage`, `adr_refs_non_broken`, `diagram_refs_non_broken`, `no_drift_vs_adrs`, `no_drift_vs_diagrams`, `diagrams_compile`, `akf_axis_stated`, `stride_per_trust_boundary`, `fitness_functions_referenced`), `rubric_score` from the evaluator.

For design mode: the authored architecture description lands at `architecture/<slug>/design-doc.md` (the arc42 document). It is the integrating document — it references but does not duplicate:
- `workflows/70-documentation/c4-model-diagrams.md` (the diagram source files under `architecture/<slug>/diagrams/`)
- `workflows/70-documentation/architecture-decision-records.md` (the ADR files under `architecture/<slug>/adr/`)
- `workflows/70-documentation/design-doc-rfc.md` (the RFC if an upstream design-doc/RFC exists for the mandate)
- `architecture/<slug>/quality-attributes.md` (the utility tree, referenced in arc42 §10)
- `architecture/<slug>/threat-model.md` (referenced in arc42 §8 crosscutting concepts — security)
- `architecture/<slug>/tradeoffs.md` (risk register, referenced in arc42 §11)

No ADR content is duplicated into `design-doc.md`; no diagram source is duplicated — only the reference path. The description is the index and the rationale narrative; the ADRs and diagrams are the authoritative detail.

## Guardrails

Per `../shared/guardrails.md`: the `solution-architect` is the one writer (conceptual integrity — one coherent set of ideas, not a committee's seams); proposer and evaluator agents are read-only. The description **proposes and documents** — it never writes product code (`src/**`, `frontend/**`, `backend/**`) and never applies infrastructure. Every section's rationale must trace to a driver from the utility tree; a section that asserts a design choice without a traced driver is accidental complexity made visible — flag it, don't include it. Reuse named patterns (arc42's own crosscutting-concept idioms, the C4 notation, MADR) before inventing bespoke formats. Record the description-format tradeoff (single-file vs per-section files) as a design choice with its cost, per `../shared/adr-template.md`. The evaluator penalizes over-ceremony — a voluminous description for a small system is a rubric failure, not a thoroughness virtue. The evaluator never edits the description it grades.
