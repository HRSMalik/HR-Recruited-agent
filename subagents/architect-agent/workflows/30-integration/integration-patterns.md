# integration-patterns

**Group:** 30-integration  ·  **Runs as:** subagent: ../.claude/agents/integration-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Apply the **Enterprise Integration Patterns** (EIP) catalog and the **anti-corruption layer** (ACL) to connect services, modules, and external systems with clean contracts — selecting named message-routing, transformation, and endpoint patterns — and apply the **strangler-fig** for incremental legacy or external-system replacement. Answers: *which named integration pattern is the right seam at each boundary, and are foreign/legacy models isolated so they cannot leak concepts into the core domain?*

## Inputs & preconditions

- From `project-architecture-config.md`: the project's existing stack (services, client, datastore, deployment topology, whether a broker is already in place), deployment target, team topology (team size and ownership boundaries per service/branch), compliance regime (data-sensitivity classes, RBAC scoping rules), and the constraint that new datastores/brokers/runtimes require a driver-justified ADR.
- The architecture **drivers** from the utility tree (`architecture/<slug>/quality-attributes.md`): the prioritized quality-attribute scenarios (QAS-*) that bear on integration — core-entity lifecycle consistency, any near-term bounded-context seams not yet wired end-to-end, any deferred/future integration seam named in the config, and the evolvability driver (keep seams for future services without building their internals). If a QAS target (e.g. message-delivery latency, at-least-once vs exactly-once guarantee) is not stated, surface it as an open driver question — do not invent a number.
- The **integration inventory**: every point where the system sends or receives data across a process or module boundary — per `project-architecture-config.md`: the client-to-backend boundary, the backend-to-datastore boundary, and any near-term sub-system boundaries (e.g. a document-processing sub-system, a payment sub-system, a workflow-closure sub-system) that are currently mocked or unbuilt — treat these as planned bounded contexts with seams to design, not yet built; plus any future deferred integration seam (e.g. an agent layer or a plugin surface) named in the config (integration seam required, internals out of scope).
- Preconditions: `project-architecture-config.md` read (never hardcode project values); the utility tree (`quality-attributes.md`) read for the prioritized scenarios this decision must serve; `workflows/30-integration/event-driven-patterns.md` read for the current event topology decision (choreography vs orchestration choice and dual-write risk assessment); `workflows/10-styles-decomposition/api-first-contracts.md` read for the existing API contract baseline (OpenAPI) that integration seams must be consistent with.

## Oracle (source of truth)

**hard:**
- **ACL isolation (structural, traceable):** every integration point with a foreign system or a legacy/external model has an explicit **anti-corruption layer** — a translation boundary that converts the foreign concept into the core domain's ubiquitous language. No foreign-model type, field name, or concept appears in the core domain (the backend's module boundaries, per `project-architecture-config.md`) without translation. Verified by static inspection of the import graph and schema objects: a `vendor.*` / `external.*` type surfacing directly in a domain aggregate or a backend route handler is an `ARCH-INTPAT-*` finding of `severity:"major"`, `oracle:"hard"`, traced to the evolvability driver (change in the external model forces a change in the core domain — the cardinal coupling sin).
- **Named EIP with a stated contract (traceable):** every integration seam uses a **named EIP** — Message Channel, Message Router, Message Translator, Message Filter, Aggregator, Splitter, Process Manager, Polling Consumer, Event-Driven Consumer, Dead Letter Channel — and the pattern's contract (delivery guarantee, ordering constraint, idempotency requirement) is documented in the relevant ADR. An integration mechanism with no named pattern and no stated contract is an `ARCH-INTPAT-*` finding of `severity:"major"`, `oracle:"hard"`, because its failure modes are unknown.
- **Canonical data model is a deliberate tradeoff, not an accidental god-schema (structural):** if a canonical data model is introduced as a shared representation across module/service boundaries, it must be recorded as a one-way-door ADR tradeoff — coupling cost stated, migration path stated, per-context translation layer specified. A canonical model that silently absorbs all domain fields from all bounded contexts without a tradeoff record is an `ARCH-INTPAT-*` finding of `severity:"major"`, `oracle:"hard"` — the canonical model as a god-schema is the anti-pattern, not the pattern itself.
- **Strangler-fig increments are independently deployable and reversible:** each strangler-fig increment (legacy path → new path) must be verifiable via a feature flag or routing rule, with the legacy path kept live until the new path is proven. An increment that cannot be toggled back is a one-way-door finding requiring explicit ADR sign-off.
- **Failure modes defined per integration seam:** every cross-process integration point has explicit failure handling: a bounded timeout (value stated), a retry policy (with backoff+jitter for idempotent operations), and a fallback or fail-fast. Applies to every call from the backend to an external system (e.g. a payment integration, a document-processing provider, a third-party lookup) and every future bounded-context boundary. An integration seam with no failure-path documentation blocks (Fallacy #1: the network is not reliable; Fallacy #2: latency is not zero — `shared/fallacies-and-tradeoffs.md`).
- **Consistency model stated per integration write path:** every integration write path across a boundary declares its consistency model (at-least-once / exactly-once / at-most-once) and, where applicable, its idempotency mechanism (idempotency key, deduplication log). An integration that relies on synchronous two-phase commit (2PC) across process boundaries blocks unless a concrete driver justifies the latency and availability cost (CAP/PACELC — `shared/fallacies-and-tradeoffs.md`).
- **Diagram compiles:** every C4 diagram produced is diagrams-as-code (Mermaid or PlantUML+C4-PlantUML per `project-architecture-config.md`) and renders without error. Each integration boundary appears as a labeled relationship with the protocol, pattern name, and sync/async distinction explicit (per `shared/diagramming-standards.md`).
- **ADR complete (MADR):** every significant integration decision — the ACL placement at each foreign boundary, the chosen EIP at each seam, the strangler-fig routing strategy, the decision to use or reject a canonical data model — is recorded as an ADR per `shared/adr-template.md` with context · ≥2 options · decision · consequences (positive AND negative) · follow-up fitness functions. A decision with no recorded alternatives or no negative consequences blocks.

**soft:**
- The integration reuses a **named EIP** over a bespoke mechanism — the named pattern carries known failure modes, a documented contract, and community-understood tradeoffs; a bespoke one carries only unknowns. Prefer the boring named pattern (Polling Consumer, Message Translator, Dead Letter Channel) over a hand-rolled variant unless a concrete driver demands deviation.
- The **ACL is placed at every foreign boundary** without exception — not just at the currently-active ones, but at every seam where a deferred future service or a third-party system will plug in (the seam is the ACL placeholder, kept thin until the boundary is live).
- The strangler-fig and branch-by-abstraction approaches are preferred over big-bang replacement; each increment is small enough to ship without a multi-day freeze.
- Conceptual integrity: the same integration idiom is applied consistently to like boundaries — not a Message Translator at one external API and a raw HTTP deserialization import at the next.

## Standards & techniques

- **Enterprise Integration Patterns (EIP)** (Hohpe & Woolf, 2003) — the canonical catalog. Key pattern families applied by this flow:
  - *Message Channel* — the communication medium (queue, topic, synchronous RPC). The first decision: what kind of channel does this boundary need? Sync HTTP call = Point-to-Point Channel; async queue = Point-to-Point or Publish-Subscribe Channel.
  - *Message Router* — Content-Based Router (dispatch on message content), Message Filter (drop messages that don't match criteria), Recipient List (fan-out to N channels). Use when the entity type or event type determines routing logic (e.g. routing by order type, document category, or event kind).
  - *Message Translator* (the structural EIP for the ACL) — transform the external/legacy message into the canonical or core-domain message. Every ACL is at minimum a Message Translator; complex ACLs add Canonical Data Model or Context Map translation layers.
  - *Message Filter* — discard messages not relevant to this consumer; keeps consumers from processing irrelevant events and reduces coupling.
  - *Aggregator* — collect and combine related messages into a single message (e.g. aggregate chunked results from a document-processing provider into one completed payload). Requires correlation identifier and completion condition.
  - *Splitter* — decompose one message into a sequence (e.g. a batch import into individual per-record messages). Pairs with Aggregator in a Scatter-Gather.
  - *Process Manager* (stateful) — tracks multi-step workflows across messages; the EIP analog of the saga orchestrator. For core-entity lifecycle steps that span multiple async calls (e.g. submit → review → approval), the Process Manager holds state and routes next steps — distinct from choreography (where each step fires the next event without central coordination).
  - *Dead Letter Channel* — where unroutable or repeatedly-failed messages go. Every channel that can receive a delivery failure must declare its dead-letter destination; a channel with no dead-letter path is a silent data loss risk.
  - *Idempotent Receiver* — the consumer end of an at-least-once channel; the consumer deduplicates using an idempotency key. Required at every at-least-once delivery boundary.
  - *Polling Consumer* vs *Event-Driven Consumer* — Polling: the consumer pulls on a schedule (simpler, higher latency, works without a broker); Event-Driven: the broker pushes on message arrival (lower latency, requires a broker, e.g. a broker like Kafka or RabbitMQ). For a stack with no broker in place (per `project-architecture-config.md`), Polling Consumer over a DB outbox table is the pragmatic path to async integration without introducing a new infrastructure dependency.

- **Anti-Corruption Layer (ACL)** (Evans, *Domain-Driven Design*, ch. 14) — a translation boundary between two bounded contexts (or between the core domain and an external system) that converts foreign concepts into the receiving context's ubiquitous language. The ACL is the DDD answer to the integration coupling problem: without it, the foreign model's types, field names, and implicit constraints bleed into the core domain and every change in the external API forces a core-domain change. Implementation: a dedicated translation module/class that the core domain calls; it speaks "core domain" outward and "external API" inward. The ACL is placed at the boundary, not at the point of use — one ACL per foreign system, not one per call site.

- **Strangler-Fig pattern** (Fowler, 2004; also in the cloud-design-pattern catalog `workflows/80-patterns/cloud-design-patterns.md`) — incremental replacement of a legacy or external integration by routing a subset of calls to the new implementation while keeping the old path live. The strangler wraps the legacy with a façade; new logic is added in the new path; legacy is retired incrementally as the new path is proven. Key implementation constraint: the routing rule must be togglable (feature flag, header-based routing) so each increment is reversible. Pairs with **branch-by-abstraction** (`workflows/90-specialized/migration-modernization.md`) when the strangler needs an in-process seam rather than an HTTP-level façade.

- **Canonical Data Model (CDM)** and its risk — a shared representation used as the integration hub language between bounded contexts or services (Hohpe & Woolf, EIP §7). Benefit: reduces N×M point-to-point translators to N+M translators (each context translates to/from the CDM). Risk: the CDM becomes a **god-schema** — it absorbs all fields from all contexts, couples every context to its evolution, and becomes the integration bottleneck. The CDM is a **recorded tradeoff**, not a default. Rule for this flow: introduce a CDM only when ≥3 bounded contexts share non-trivial overlap AND translator proliferation is a confirmed cost; record it as a one-way-door ADR with the coupling cost made explicit. Otherwise prefer per-boundary Message Translators (simpler, more evolvable, less coupling).

- **Context Map** (Evans, DDD) — the explicit map of relationships between bounded contexts: Shared Kernel, Customer/Supplier, Conformist, Anti-Corruption Layer, Open-Host Service, Published Language. The context-map relationship at each boundary determines where an ACL is warranted (every Conformist relationship needs an ACL; a Shared Kernel does not, but must be governed). This flow produces or references the context map from `workflows/00-drivers/domain-analysis.md`.

- **Outbox + CDC** (Transactional Outbox + Change Data Capture) — the canonical mechanism for reliable async integration without 2PC: write the domain event to an outbox table/collection in the same transaction as the domain write (the datastore's transactional guarantee, per `project-architecture-config.md`); a CDC or polling relay publishes it to the channel. Eliminates the dual-write race (write to DB + write to queue; one can fail silently). For a stack with no broker in place (per `project-architecture-config.md`), the outbox relay is a Polling Consumer reading the outbox table/collection on a schedule — no broker (e.g. Kafka/Kinesis) required until a latency driver demands it. See `workflows/20-data/consistency-transactions.md` for the full consistency model.

- **The 8 fallacies of distributed computing** (`shared/fallacies-and-tradeoffs.md`) — integration seams are where the fallacies bite hardest. Fallacy #1 (network is reliable) → every integration call can fail; every seam must have a timeout + retry + dead-letter. Fallacy #2 (latency is zero) → every synchronous integration hop consumes latency budget; the latency cost of each hop must be in the capacity model. Fallacy #8 (network is homogeneous) → the external system may use a different schema version, encoding, or protocol; the ACL absorbs this heterogeneity so the core domain doesn't have to.

- **ATAM sensitivity/tradeoff analysis** (`shared/evaluation-method.md`) — the key integration tradeoff points: *Synchronous ↔ Asynchronous* (sync is simpler but couples and cascades failures; async decouples but is eventually consistent and harder to trace); *Canonical model ↔ per-boundary translator* (CDM reduces translator count but couples all contexts to the model's evolution; per-boundary is more evolvable but multiplies translators); *Strangler-fig ↔ big-bang* (strangler is safer and reversible but takes longer; big-bang is faster but one-way-door). Name the point chosen on each tradeoff curve per boundary, not just the winner.

- **MADR decision records** (`shared/adr-template.md`) — every integration decision (ACL placement, EIP selection at each seam, CDM adoption/rejection, strangler-fig routing strategy) is an ADR with ≥2 genuine options, stated consequences (positive AND negative), and follow-up fitness functions.

## Step sequence

- **audit:** Frame the integration drivers from `project-architecture-config.md` + `quality-attributes.md` → enumerate every integration seam (process boundaries, external system calls, bounded-context handoffs) → for each seam: (a) identify whether a named EIP is used and documented; (b) check whether any foreign-model types cross the boundary without an ACL (inspect backend route schemas and domain models for external-API field names appearing in core aggregates); (c) check whether the consistency model and delivery guarantee are stated; (d) check failure-path completeness (timeout value stated, retry policy, dead-letter or fail-fast); (e) check whether a canonical data model is in use and whether its coupling cost is recorded → emit findings per `shared/finding-schema.md` (`type:"integration"`; `oracle:"hard"` for unnamed-EIP seam, ACL violation, unstated consistency model, missing failure path; `oracle:"soft"` for bespoke-over-named-pattern, inconsistent ACL placement, CDM-as-god-schema risk without a recorded tradeoff) — read-only, no artifact authored, no source edited.

- **design:** Frame (extract/confirm integration drivers + the integration inventory — every boundary in scope — read-only) → Explore (produce ≥2 genuinely divergent directions per significant boundary: e.g. for a payment integration, Direction A — synchronous REST call wrapped in an ACL + circuit-breaker; Direction B — async outbox + Polling Consumer with ACL at the relay; each direction is a concrete spec: the named EIP, the ACL shape, the consistency model, the failure path, the latency budget impact — read-only proposers) → Evaluate (one skeptical ATAM evaluator scores each direction against the utility tree + hard oracles: ACL isolation check, named-EIP contract stated, consistency model stated, failure modes defined, CDM tradeoff recorded if used; scores soft oracles: reuse of named pattern vs bespoke, ACL at every foreign boundary, strangler-fig reversibility; order-swapped to kill position bias; picks winner + grafts superior elements from runners-up; documents sensitivity points — e.g. "core-entity status latency is sensitive to whether the document-processing sub-system integration is sync or async: sync couples availability; async adds eventual-consistency lag" — and tradeoff points) → Document (one writer — `solution-architect` — authors all artifacts: C4 L2 diagram with each integration boundary as a labeled relationship with EIP name and delivery guarantee; ACL module map; ADRs for each significant integration decision; the context map; outbox/relay design if async paths are introduced; fitness-function specifications) → Verify (fitness functions: static import-graph check that no foreign-model type crosses a boundary without an ACL module in between; contract test that each named EIP's idempotency/dead-letter contract is implemented; diagram-compile; traceability: every integration driver traces to a seam decision; evaluator re-scores from artifacts → loop ≤8 or pass).

## Assertions & exit gate

- Every integration seam in the inventory uses a named EIP with its contract (delivery guarantee, ordering constraint, idempotency) documented — no unnamed bespoke mechanism.
- Every foreign-system or legacy-model boundary has an explicit ACL module. Verifiable by static inspection: no external-API type or field name appears in a core-domain aggregate or backend route schema without a translation layer in between. Violations block.
- If a canonical data model is introduced, it is recorded in a one-way-door ADR with the coupling cost and migration path stated. A CDM with no ADR, or whose schema silently absorbs all contexts' fields, blocks.
- The context map (from `workflows/00-drivers/domain-analysis.md` or authored here) classifies every bounded-context relationship (ACL, Customer/Supplier, Conformist, Shared Kernel, Open-Host); every Conformist relationship has an ACL.
- Every strangler-fig increment has a togglable routing rule (feature flag, HTTP header, config value) and the legacy path is provably kept live until the new path is validated — no irreversible increments without an accepted-risk ADR.
- Every integration write path across a boundary states its consistency model (at-least-once / exactly-once / at-most-once) and its idempotency mechanism. No 2PC across process boundaries without a driver-justified ADR.
- Every integration seam has a defined failure path: bounded timeout (value stated), retry policy (idempotent paths only; backoff+jitter), and a dead-letter channel or fail-fast disposition. No unbounded synchronous call chains across process boundaries.
- All C4 diagrams (L2 Container view with integration boundaries as labeled relationships — EIP name, protocol, sync/async, delivery guarantee) are diagrams-as-code (Mermaid or PlantUML+C4-PlantUML) and render without error.
- Every significant integration decision has a complete MADR ADR: context · ≥2 options · decision · consequences (positive AND negative) · follow-up fitness functions. A decision with only positive consequences blocks.
- Any deferred future integration seam named in `project-architecture-config.md` (e.g. a future agent layer or plugin surface) is marked as an ACL placeholder at the module boundary — a thin, named interface that the future service will plug into — without building its internals into the current architecture.

**Gate:** hard oracles green (every seam uses a named EIP with stated contract; every foreign boundary has an ACL; CDM if present has a one-way-door ADR; strangler-fig increments are togglable; every write path states its consistency model; every seam has a defined failure path; diagrams compile; every ADR complete) AND (design) rubric mean ≥ 0.8 across conceptual integrity, right-sizedness, appropriate coupling, evolvability, and tradeoff honesty.

## Guardrails

Per `shared/guardrails.md`: propose and evaluate — never write product-source files (`backend/**`, `frontend/**`), never write integration glue code, never apply schema migrations or infrastructure changes. Author only architecture artifacts (ADRs, C4-as-code diagrams, context map, fitness-function specifications) into `architecture/<slug>/`. Writing a fitness-function check script into the project's `tests/` or `fitness/` directory for CI enforcement is permitted — that is a verification artifact, not product behavior. The controller implements the chosen integration design.

Ground every integration decision in a driver: an ACL that protects no driver is accidental complexity — but an integration seam without an ACL at a foreign boundary is a hard-oracle violation regardless of how simple the integration seems today. Evolvability is the driver: the cost of a missing ACL is paid later, when the external API changes and the core domain must absorb it.

Reuse the named EIP before inventing: the EIP catalog (Hohpe & Woolf) covers the full space of integration problems — if the problem has a name there, use the name, state the contract, and record the tradeoff. A bespoke mechanism for a problem with a named EIP solution is a soft-oracle finding (YAGNI is not the right framing here — the named pattern is simpler because its failure modes are known).

One writer (`solution-architect`) authors all artifacts in sequence — never concurrent partial authoring. Proposers and the evaluator are read-only. The evaluator scores against traced scenarios and structural facts, not the writer's prose; it penalizes accidental complexity (a CDM where per-boundary translators would do; a Process Manager where a simple event sequence would do) and unjustified novelty (introducing a message broker without a latency driver that demands it over a DB outbox).

Bound the loop: ≤8 iterations of document→verify before escalating to the controller with the blocker finding.
