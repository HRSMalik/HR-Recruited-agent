# async-messaging

**Group:** 30-integration  ·  **Runs as:** subagent: ../.claude/agents/integration-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Architect the asynchronous messaging layer — choosing the right primitive (point-to-point queue vs pub/sub topic vs durable log), justifying the broker against the project's concrete drivers, specifying delivery semantics and ordering guarantees per use, and wiring in dead-letter queues and poison-message handling. Answers: *which messaging topology, with which delivery contract, is the simplest that decouples the producers/consumers the drivers actually require?*

## Inputs & preconditions

- **From `project-architecture-config.md`:** existing stack + deployment target; scale/SLO targets (throughput, latency budgets); compliance regime (PII/audit-trail rules); team topology + operational headroom; constraints (stack lock — no new broker/runtime without a driver-justified ADR; boring-tech-preferred).
- **The architecture drivers:** the prioritized quality-attribute scenarios from `workflows/00-drivers/quality-attribute-scenarios.md` (the utility tree, 6-part, with response measures) that demand decoupling, durability, throughput, or ordering. A messaging design with no traced driver — "Kafka because it's modern" — is accidental complexity and fails the guardrails immediately.
- **Target:** the integration boundary or subsystem introducing async messaging; the integration event contracts (from `workflows/10-styles-decomposition/api-first-contracts.md` — AsyncAPI or equivalent as source of truth); the consistency model per write path (from `workflows/20-data/data-modeling.md` or the outbox/saga analysis in `workflows/20-data/consistency-transactions.md`).
- **Preconditions:** the source-of-truth docs (`docs/`, `project-understanding.md`) read; existing messaging topology (if any) documented; AsyncAPI/event contracts available or scoped in this flow's design step; if a driver (peak message rate, durability SLO, ordering requirement) is *not stated*, surface it as an open driver question — do not invent a number.

## Oracle (source of truth)

The external truth is the **quality-attribute scenario's response measure** (from the utility tree) that the messaging design must satisfy, plus the named pattern's structural guarantee.

- **hard:**
  - The **delivery semantic** (at-most-once / at-least-once / exactly-once) is stated and matched to the use case: audit-log events must not be lost → at-least-once; an idempotent downstream → at-least-once with idempotent consumer; financial settlement exactly-once where supported (and its overhead is driver-justified).
  - **Consumers are idempotent** for every at-least-once delivery path — either by a natural idempotency key (claim number, event ID + write-if-not-seen), or by a deduplication mechanism (Redis set, DB unique constraint on event ID). A non-idempotent consumer on an at-least-once queue is an `oracle:"hard"` finding (ARCH-MSG-001, `severity:"critical"`).
  - A **DLQ (Dead-Letter Queue) + retry policy** exists for every queue/subscription: max-retries stated (e.g. 3×), backoff strategy (exponential + jitter per `shared/fallacies-and-tradeoffs.md` §1 — network is unreliable), DLQ destination named, and an alerting/drain runbook referenced. A message path with no DLQ or an unbounded retry loop blocks.
  - The **broker choice is justified by a named driver**, not defaulted: log-replay needed (audit, event-sourcing projection replay, multi-consumer fan-out at scale) → Kafka/Pulsar; work-queue with visibility timeout + managed DLQ, low operational cost → RabbitMQ or SQS; managed cloud-native pub/sub → SNS+SQS (fan-out + queue per subscriber). "We use Kafka" with no driver that demands log-replay or consumer-group replay is unjustified novelty — flag it.
  - **Ordering guarantees are stated per channel**: Kafka guarantees per-partition order (per-key if key-routed); SQS standard offers best-effort (FIFO queue for strict ordering, at a throughput cost); RabbitMQ queues are per-queue FIFO by default. A scenario that needs strict ordering (claim state transitions must be applied in sequence) must route to a topology that can provide it; the throughput cost of that choice is stated.
  - Every messaging boundary has **failure modes defined**: producer failure (message lost before publish? → use outbox pattern; cross-reference `workflows/30-integration/event-driven-patterns.md`), broker unavailability (circuit-breaker on the publish path, fallback to outbox/retry table), consumer crash before ack (redelivery via at-least-once + idempotent handler). No unhandled partial-failure path.
  - **Fallacies honored** (`shared/fallacies-and-tradeoffs.md` §1–8, all eight checked): §1 network is unreliable → no fire-and-forget publish with no delivery guarantee where durability is a driver; DLQ + retry + backoff required; §2 latency is non-zero → async chosen deliberately when decoupling > latency, never assume broker publish is instantaneous; §3 bandwidth is not infinite → large payloads use the claim-check pattern (publish a reference/ID, consumer fetches the payload from blob/DB); §4 network is secure → broker endpoint requires authn (API key / IAM role / mTLS); no anonymous publish or subscribe; this gates the trust-boundary secured check; §5 topology changes → broker endpoint via service discovery / managed endpoint, not hardcoded IP; §6 one administrator → async channel contracts (event schemas, topic names, delivery semantics) must be versioned explicitly (AsyncAPI spec) because multiple independent teams own producers and consumers and cannot be coordinated for a simultaneous deploy; schema-evolution strategy (Avro/Protobuf backward-compat, or JSON Schema version field) is stated per channel; §7 transport cost is not zero → cross-AZ / cross-region message volume is accounted for in the capacity model; §8 network is homogeneous → schema format and serialization are declared in the AsyncAPI binding (JSON Schema / Avro / Protobuf) so heterogeneous consumers can deserialize without runtime negotiation; no implicit type-coercion across service boundaries.
  - **Consistency model stated** for every write path that crosses a messaging boundary: dual-write (DB + publish directly) is flagged as a race condition — the outbox pattern or CDC is the correct fix where at-least-once + no-lost-event is a driver.
  - **Trust boundary secured**: the broker endpoint is an internal trust boundary — authn required (API key / IAM role / mTLS), no anonymous publish; topic/queue ACLs so a producer can only publish to its own topics; secrets (broker creds) externalized via env, never in code or diagram (`project-architecture-config.md` compliance regime).
  - **ADR complete** per `shared/adr-template.md`: context · ≥2 considered options · decision · consequences (positive AND negative) · driver refs to the utility tree. A broker decision with no recorded alternatives blocks.
  - **Diagrams compile**: all C4 Container / Dynamic views showing async flows use dashed arrows for async, labeled with the event name + protocol (e.g. "publishes ClaimStatusChanged to, SQS (at-least-once)"); rendered without error from diagrams-as-code (`shared/diagramming-standards.md`).
  - **No un-accepted broker SPOF** (`quality-attribute-rubric.md` — No un-accepted SPOF): the broker itself is on the critical path of every availability-driver scenario it serves. The design must demonstrate that the broker has redundancy or failover (managed multi-AZ: SQS/SNS are inherently multi-AZ; Kafka/RabbitMQ self-hosted require explicit cluster config — ≥3 brokers, replication factor ≥ 2, ISR ≥ 2 stated; Confluent Cloud/MSK — HA tier selected and confirmed). If a single-node or single-AZ broker is accepted for cost/simplicity, it must be an explicit entry in the risk register (`type:"risk"`, `severity:"major"`, named scenario threatened, likelihood×impact, owner) — a silent broker SPOF on any availability-driver path blocks.
  - **Cost bounded** (`quality-attribute-rubric.md` — Cost bounded): the capacity model must include an order-of-magnitude monthly monetary cost estimate — not just storage volume — against a stated budget constraint from `project-architecture-config.md`. At minimum: broker tier cost (managed SQS/SNS per-request pricing at peak message rate × retention; Kafka MSK/Confluent cluster tier; self-hosted instance cost); cross-AZ data-transfer cost if multi-AZ; DLQ storage. Example gate: "SQS at 50k msg/day × 1 KB × 30-day retention ≈ $0.02/day ($0.60/mo), within the ops budget constraint of $X/mo." A design whose broker cost is unstated where cost is a driver blocks.
  - **Data lifecycle lawful** (`quality-attribute-rubric.md` — Data lifecycle lawful): where the compliance regime from `project-architecture-config.md` includes PII, GDPR, HIPAA, or audit-trail obligations — (a) every message schema is classified for PII content (claim PII — patient name, SSN, health data — must not appear as broker payload unless encrypted and explicitly justified); (b) broker message retention limits are set at or below the regulation's maximum retention period (e.g. GDPR: no indefinite retention of personal data — Kafka topic `retention.ms` and SQS message retention ≤ stated window); (c) an erasure/suppression path exists for right-to-erasure requests (event log compaction key-delete or pointer-based reference with PII stored only in the authoritative DB). An unclassified-PII message schema or an unbounded broker retention window under a regulated regime blocks.

- **soft:**
  - The messaging topology is the **simplest that decouples the producers/consumers the drivers require** — do not add a broker where an in-process call or a DB poll (outbox + polling consumer) is sufficient; do not add Kafka where SQS solves the problem with far lower operational overhead.
  - **Ordering is only enforced where the domain demands it** — it costs throughput and adds partitioning complexity. A channel that only needs loose ordering (notification emails, analytics events) should not be on a FIFO queue or a single Kafka partition. Conversely, a channel driving a state machine (claim lifecycle) needs ordering within the entity key; the design reflects this, not a blanket choice.
  - Fan-out topology is right-sized: SNS→SQS fan-out when ≥2 independent consumers need the same event; a single queue when there is exactly one consumer — no premature fan-out for a future consumer that doesn't yet have a driver.
  - Operational surface area is bounded by the team topology: a small team should default to a managed broker (SQS/SNS, Cloud Pub/Sub) before self-hosted Kafka unless a driver (log-replay, high-throughput, multi-datacenter replication) demands the operational cost.

## Standards & techniques

- **Point-to-point queue vs pub/sub topic vs durable log** (the EIP catalog primitives):
  - *Queue (point-to-point / competing consumers):* each message consumed by exactly one worker; natural work-distribution; dead-lettering built-in on managed queues (SQS, RabbitMQ). Use for: task offload (OCR job, document processing), work-queue decoupling, ordered per-entity processing (FIFO queue or partitioned queue).
  - *Topic (pub/sub):* each message delivered to all current subscribers; fan-out to ≥2 independent consumers. Use for: notification events (ClaimStatusChanged consumed by email service + audit service + analytics), where consumers are independently versioned and shouldn't share a queue.
  - *Durable log (Kafka/Pulsar):* messages persisted for a retention window; consumer groups replay from any offset; partitioned for throughput; per-partition ordering. Use when: multiple consumer groups need independent offsets, replay is a requirement (projection rebuild, audit replay, event sourcing), or throughput demands horizontal partitioning.
- **Broker selection by concrete driver:**
  - **Log-replay required / event sourcing / multi-consumer groups at scale → Kafka or Pulsar.** Kafka's log model is its defining guarantee: consumers can replay; partitions scale throughput horizontally. Cost: self-hosted operational burden (ZooKeeper/KRaft, schema registry, topic management); Confluent Cloud or MSK reduce ops cost but add vendor lock-in.
  - **Work-queue, low ops, managed DLQ, visibility timeout → SQS (or RabbitMQ if self-hosted).** SQS standard is extremely low ops, infinite scale, per-message visibility timeout (natural exactly-once window), built-in DLQ. SQS FIFO adds strict ordering + exactly-once deduplication within a 5-min window at reduced throughput (3000 msg/s with batching). RabbitMQ adds routing flexibility (exchanges: direct/topic/fanout/headers) but requires cluster management.
  - **Fan-out to multiple independent queues → SNS (topic) + SQS (per-subscriber queue).** SNS delivers to all subscribed SQS queues; each consumer controls its own DLQ and retry independently. Low operational cost; managed. Use when ≥2 services need the same event and must fail/retry independently.
  - **Cloud-native cross-service within a single cloud → managed equivalents** (Google Pub/Sub, Azure Service Bus) — same selection criteria; match `project-architecture-config.md` deployment target.
- **Delivery semantics — state the guarantee, design to it:**
  - *At-most-once:* fire and forget; messages may be lost; acceptable only for ephemeral / non-critical events (heartbeats, debug traces). Never use for claim data, audit events, or financial triggers.
  - *At-least-once:* the default safe choice; the consumer may receive duplicates on redelivery (broker timeout + retry); **consumer must be idempotent**. Kafka consumer-group with manual commit; SQS with visibility timeout and manual delete after processing; RabbitMQ with manual ack.
  - *Exactly-once:* guaranteed no-loss and no-duplicate; requires coordination (Kafka transactions + idempotent producer; SQS FIFO + deduplication ID). High overhead; use only where the domain truly cannot tolerate either loss or duplicate (e.g. payment trigger that cannot be retried externally). Evaluate the PACELC cost: exactly-once typically adds latency — name the tradeoff (`shared/fallacies-and-tradeoffs.md` Consistency ↔ Latency).
- **Consumer idempotency patterns:**
  - Natural key check: write-if-not-exists on a DB unique constraint keyed by `event_id`; fastest, no extra infra.
  - Idempotency table: a `processed_events(event_id, processed_at)` table; TTL it per the retention window.
  - Redis set with TTL: fast deduplication for high-throughput consumers; TTL must exceed the max redelivery window.
- **Outbox pattern** (EIP "Transactional Outbox"): to avoid dual-write (DB commit + publish in separate transactions = a race where the publish can be lost), write the event to an `outbox` table in the same DB transaction as the business entity update; a separate relay process (CDC via Debezium, or a polling publisher) reads committed outbox rows and publishes to the broker. Cross-reference `workflows/30-integration/event-driven-patterns.md`.
- **DLQ + poison-message handling:** every queue has a configured `maxReceiveCount` (e.g. 3) after which the message is moved to the DLQ. The DLQ is monitored (CloudWatch alarm / Datadog monitor — alert if DLQ depth > 0 for > 5 min). A drain runbook exists: inspect the message → fix the consumer → replay or discard. A "poison message" (structurally valid but semantically impossible) requires a separate dead-parsing fallback path.
- **Claim-check pattern** (EIP): for large payloads (e.g. scanned claim documents), publish only a reference (claim ID + blob URI); the consumer fetches the full payload. Avoids bandwidth fallacy §3 and broker message-size limits.
- **ATAM evaluation** for broker selection: score each option against the utility tree's high-priority scenarios on throughput, latency, ordering, ops cost, and replay need; build the decision matrix; kill position bias (swap order, evaluate twice); name sensitivity points (throughput is *sensitive* to partition count; exactly-once is *sensitive* to idempotent-producer overhead).
- **AsyncAPI** as the contract-first source of truth for async channels (parallel to OpenAPI for sync): channel name, message schema (JSON Schema / Avro / Protobuf), binding (Kafka topic, SQS queue), operation (publish/subscribe). Cross-reference `workflows/10-styles-decomposition/api-first-contracts.md`.

## Step sequence

- **audit:** Read `project-architecture-config.md` + `project-understanding.md` + the existing event/queue definitions → confirm or derive the drivers that justify async messaging → walk every messaging path and check: delivery semantic stated and matched to use; consumers idempotent for at-least-once paths; DLQ + retry policy exists with stated `maxReceiveCount`; broker choice traced to a driver; ordering guarantee stated and topology can provide it; no dual-write (outbox or CDC present where needed); no fire-and-forget on durable-event paths; failure modes (producer failure, broker down, consumer crash) all handled; trust boundary secured (broker authn + ACLs); claim-check used for large payloads → emit findings + risks per `shared/finding-schema.md` (read-only, no artifacts authored, no source edited).

- **design:** Frame (confirm drivers + utility tree scenarios that demand async — cross-reference `workflows/00-drivers/quality-attribute-scenarios.md`; surface open driver questions: peak message rate? ordering requirement per channel? durability SLO?) → Explore (≥2 divergent broker/topology directions — e.g. Direction A: SQS+SNS managed, no replay; Direction B: Kafka/MSK, log-replay + consumer groups; each with delivery semantics, DLQ strategy, and consumer idempotency approach concretely specified, not just named) → Evaluate (ATAM: score each direction vs high-priority scenarios on throughput response-measure, ordering, ops cost, evolvability; swap evaluation order; name sensitivity + tradeoff points; pick the simplest that meets every driver — penalize over-engineering; graft the DLQ/retry design from the better runner-up) → Document (ONE writer: AsyncAPI channel definitions; C4 Container diagram showing producers, broker, queues/topics, consumers with dashed async arrows labeled with event name + delivery semantic; ADRs — one per significant decision: broker choice, delivery semantic per channel, outbox vs dual-write, ordering approach; capacity model — back-of-envelope: peak message rate × message size × retention window = broker storage; consumer lag budget at peak → confirms response-measure is met; `architecture/<slug>/fitness-functions.md` entries: consumer-is-idempotent contract test, DLQ-depth alert threshold, no-sync-call-where-async-required dependency rule) → Verify (fitness-function specs compile and are runnable; C4 diagrams render without error; every channel in the AsyncAPI spec traces to an ADR; every ADR has driver refs to the utility tree; STRIDE per broker trust boundary reviewed; evaluator re-scores from the artifacts → loop ≤ 8 or pass).

## Assertions & exit gate

- Every async channel has a stated delivery semantic matched to its use case; at-most-once is absent from any durable-event path (audit, claim state, financial trigger).
- Every consumer on an at-least-once path is structurally idempotent — an idempotency key, a natural DB unique constraint, or an explicit deduplication mechanism is named and traceable to the consumer's implementation boundary.
- Every queue/subscription has a DLQ configured with a stated `maxReceiveCount` (≥ 1), a backoff strategy (exponential + jitter), and an alert threshold on DLQ depth; a drain runbook is referenced.
- The broker choice is justified by a named driver from the utility tree — log-replay / multi-consumer-group replay if Kafka, low-ops / work-queue / managed DLQ if SQS, fan-out if SNS+SQS; "default" or "familiarity" is not a driver.
- Ordering guarantees are stated per channel; channels that enforce ordering call out the throughput cost and confirm the driver demands it.
- No dual-write on any durable-event write path: either the outbox pattern or CDC is used, or the write path is single-transaction atomic; cross-reference `workflows/30-integration/event-driven-patterns.md`.
- Every cross-service messaging call has a bounded failure mode: producer publish failure, broker unavailability, consumer crash before ack — each handled or explicitly accepted as risk.
- All 8 fallacies of distributed computing are gated: §1 unreliable network → DLQ + retry + backoff on every durable path; §2 non-zero latency → async chosen deliberately; §3 finite bandwidth → claim-check for large payloads; §4 network not secure → broker authn (mTLS / IAM / API key) enforced, no anonymous publish; §5 topology changes → managed/discovered endpoints, no hardcoded broker IPs; §6 one administrator not assumed → event schemas versioned in AsyncAPI spec with backward-compat strategy named per channel; §7 transport cost non-zero → cross-AZ message volume in capacity model; §8 homogeneous network not assumed → serialization format declared in AsyncAPI binding (Avro / Protobuf / JSON Schema) for all consumers.
- C4 diagrams compile from diagrams-as-code; async flows use dashed labeled arrows; every relationship is typed with event name + delivery semantic + protocol.
- Every significant decision is an ADR (context · ≥2 options · decision · consequences positive AND negative · driver refs); no unjustified decision, no orphan ASR.
- The broker has no un-accepted SPOF on any availability-driver path: managed broker is confirmed multi-AZ, OR self-hosted cluster config (replication factor, ISR) is stated, OR the single-point risk is an explicit, signed-off entry in the risk register.
- Capacity model confirms peak-message-rate × message-size × retention ≤ broker storage budget AND includes an order-of-magnitude monthly monetary cost estimate against the stated budget constraint; consumer lag at peak does not violate any stated latency response-measure.
- Where a compliance regime applies: every message schema is PII-classified; broker retention windows are set within the regulatory limit; an erasure/suppression path for right-to-erasure is named.
- **Gate:** all hard oracles above are green AND (design) rubric mean ≥ 0.8 (simplest topology that meets every driver; ordering only where the domain demands it; managed/boring broker preferred unless a concrete driver justifies the operational cost of self-hosted).

## Guardrails

Per `shared/guardrails.md`: propose & document — never write product code (`src/**`, `frontend/**`, `backend/**`); never apply IaC, create broker topics/queues in a live environment, or touch credentials. Author AsyncAPI specs, ADRs, C4-as-code diagrams, fitness-function *specifications*, and capacity models into `architecture/<slug>/` — the controller implements them.

Ground every broker and topology decision in a driver from the utility tree (`workflows/00-drivers/quality-attribute-scenarios.md`). A broker with no driver that demands it is accidental complexity — flag it with `severity:"major"` and `oracle:"soft"` (Simplicity / YAGNI) rather than recommending it. Boring, managed technology (SQS, SNS) beats self-hosted (Kafka, RabbitMQ) unless a concrete driver — log-replay, throughput beyond managed limits, multi-datacenter replication — justifies the operational cost; state that cost in the ADR.
