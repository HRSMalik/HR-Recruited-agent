# service-communication-resilience

**Group:** 30-integration  ·  **Runs as:** subagent: ../../agents/integration-architect.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
Make every cross-process call survive **partial failure** — the normal state of a distributed system, not the exception. This flow owns the failure path of each remote interaction (service↔service, app→datastore, app→external API): a **bounded timeout**, a **retry policy** (backoff + jitter, idempotent-only, with a retry budget) *or* an explicit no-retry, a **circuit breaker** and/or **bulkhead** to stop a slow dependency from saturating the caller, and a **fallback or fail-fast** so the caller degrades instead of hanging. It answers the decision: *for each remote interaction, sync or async, and what does it do when the callee is slow, down, or lying — such that one dependency's failure cannot cascade into a system-wide outage?* Grounded in the **8 fallacies of distributed computing** — chiefly "the network is reliable" and "latency is zero" — the deliverable is a per-call **resilience contract** (ADR-shaped) that the availability scenarios can be argued against, not a vague "we use retries."

## Inputs & preconditions
- From `project-architecture-config.md`: the current shape (e.g. a **modular-monolith pair** — a frontend SPA → a backend API → a primary datastore, same-origin behind a reverse proxy, **no broker/queue/mesh today** — or whatever topology the project actually has), so the in-scope remote calls are mostly **SPA → backend API** (HTTP) and **backend → datastore** (driver-level), plus any **planned external boundaries** (e.g. a document-processing provider, a payment integration, a deferred integration seam) that must carry a failure contract *before* they are wired, not after.
- The **drivers / utility tree** from `architecture/<slug>/quality-attributes.md` (`workflows/00-drivers/quality-attribute-scenarios.md`) — specifically the availability + performance scenarios this flow protects: e.g. a BRD-seeded `payment action < 60s`, `document-extract < 30s`, `dashboard render < 2s` (express each as a 6-part QAS with a response measure before designing). **Availability target / RTO is not yet stated** — if absent, surface it as an open driver question (`requirements-analyst`), do not assume a number; a no-driver call downgrades scope and says so.
- The **interaction inventory**: the list of cross-process call sites (from the C4 L2 Container view + the API contracts), each with caller, callee, payload shape, sync/async, and whether the operation is **idempotent**. A call site missing from the inventory is an un-analyzed failure path — an audit finding in itself.
- Preconditions: the source-of-truth docs (BRD / PRD / understanding log) named in `project-architecture-config.md`, the C4 Container diagram, and any existing `architecture/<slug>/threat-model.md`/`capacity-model.md` read first; the relevant patterns from `../80-patterns/resilience-patterns.md` and `../80-patterns/cloud-design-patterns.md` reviewed (reuse the named pattern, don't invent a bespoke mechanism).

## Oracle (source of truth)
The external truth is the **availability/performance quality-attribute scenarios** (the response measures the failure handling must hold under a degraded environment) **+ the failure-mode hard oracle** (`shared/quality-attribute-rubric.md` → *Failure mode defined*) **+ the 8 fallacies** (`shared/fallacies-and-tradeoffs.md` §1–8). "We're resilient" is not an oracle; "payment-submit completes < 60s or fails-fast with a queued-retry, and a 30s gateway stall on the payment provider trips the breaker within ≤ 5 consecutive failures without exhausting the backend worker pool" is.

- **hard:**
  - **Every** cross-service / cross-process / external call (the full interaction inventory, including app→datastore and SPA→API) has, explicitly: (1) a **bounded timeout** — connect *and* read, never infinite/default-unbounded; (2) a **retry policy with backoff + jitter** *or* a recorded **no-retry** decision; (3) a **circuit breaker and/or bulkhead** isolating the dependency; (4) a **fallback or an explicit fail-fast**. A call site missing any of the four is an `oracle:"hard"` finding (`type:"integration"`/`"availability"`).
  - **Retries only on idempotent operations.** Every retried call is GET/PUT/DELETE-shaped or carries an **idempotency key**; no naked retry on a non-idempotent POST (double-charge / double-submit risk). A retry on a non-idempotent write with no dedupe key blocks.
  - **No unbounded synchronous call chains.** No A→B→C→D synchronous fan-out whose summed worst-case timeout exceeds the originating scenario's latency budget (the timeout budget must *decrease* down the chain — a child timeout ≥ its parent's is a bug). The end-to-end latency budget is allocated across hops (fallacy #2).
  - **Backoff is bounded + jittered + budgeted.** Retries use exponential backoff with jitter (full or decorrelated), a **max attempt cap**, and a **retry budget / token bucket** (retries capped as a fraction of total requests) so a struggling dependency is not amplified — the **retry-storm / cascading-failure** trap is explicitly defended, not assumed away.
  - **Sync-vs-async stated per interaction.** Each call records whether it is synchronous request/response or asynchronous (queue/event) **with the reason** — never a blanket "everything is sync" or a broker introduced with no driver. Where async is chosen it cross-references `async-messaging.md` for delivery semantics; where sync is kept, the failure contract above covers it.
  - **Fallacies honored** (`shared/fallacies-and-tradeoffs.md`): the design accounts for unreliable network (#1: timeouts/retries/breaker/fallback on every call), non-zero latency (#2: a per-hop latency budget), finite bandwidth (#3: pagination/projection/backpressure on large payloads), changing topology (#5: no hardcoded hosts — discovery/config), and partial failure as the normal case. A design that assumes any of these away fails this oracle.
- **soft:** the failure handling is **consistent across the system** (**conceptual integrity** — one resilience policy/library/middleware, not three call sites doing retries three different ways); the **blast radius of any one dependency is bounded** (a slow document-processing provider degrades only that feature, never the whole API — bulkhead/pool isolation); **sync vs async is chosen per interaction on its merits** (`shared/fallacies-and-tradeoffs.md` → *Synchronous ↔ Asynchronous*, *Coupling ↔ Autonomy*), not globally defaulted; the resilience config is **observable** (breaker state, retry counts, timeout rates exported — cross-link `../40-cross-cutting/observability-architecture.md`) and **right-sized** (no breaker/bulkhead on an in-process call where it adds latency for no driver — right-sizedness penalizes ceremony the topology doesn't need).

## Standards & techniques
- **The 8 fallacies of distributed computing** (Deutsch/Gosling, via `shared/fallacies-and-tradeoffs.md`) — the law this flow enforces. #1 *network is reliable* → timeout+retry+breaker+fallback per call; #2 *latency is zero* → a latency budget spent across hops; #3 *bandwidth is infinite* → pagination/projection/claim-check/backpressure; #4 *network is secure* → mTLS/authn at the boundary (defer the control to `../40-cross-cutting/security-architecture.md`, but the resilient call still authenticates); #5 *topology doesn't change* → discovery, no hardcoded hosts, connection re-establishment.
- **The resilience pattern stack** (`../80-patterns/resilience-patterns.md`, Nygard *Release It!*, the Azure cloud-design-pattern catalog):
  - **Timeout** — always bounded; set connect and read timeouts; the timeout is the *first* line of defense (a hung call you never time out is a leaked thread/connection and the seed of cascading failure).
  - **Retry** (with backoff + jitter) — only for **transient** faults and **idempotent** operations; **exponential backoff** (e.g. 200ms·2ⁿ) + **jitter** (full or decorrelated, to de-synchronize a thundering herd); a **max-attempts cap** and a **retry budget** (token-bucket / % cap) to prevent retry storms; classify retryable (429/503/timeout) vs non-retryable (400/401/409) errors.
  - **Circuit breaker** (closed → open → half-open) — trip after *N* consecutive failures or an error-rate threshold; **fail fast while open** (don't pile requests onto a dying dependency); probe with limited traffic in half-open. This is the primary cascading-failure defense — it converts a slow dependency into a fast, contained failure.
  - **Bulkhead** — isolate resources per dependency (separate connection pool / thread pool / concurrency limit) so one saturated callee can't exhaust the caller's whole pool (the *Titanic* metaphor). Pairs with the breaker.
  - **Fallback / graceful degradation** — a default value, a cached/stale response, a queued-for-later, or a degraded feature — *or* an explicit **fail-fast** with a clear error. Cross-link `../40-cross-cutting/availability-resilience.md` for the degradation strategy and blast-radius analysis.
  - **Load shedding / rate limiting** — under overload, reject early (429 + Retry-After) to protect the core; admission control over collapse.
- **Sync vs async per interaction** — choose on the `shared/fallacies-and-tradeoffs.md` *Synchronous ↔ Asynchronous* and *Coupling ↔ Autonomy* tradeoffs: sync is simple/immediate but couples and cascades; async (queue/event) decouples and absorbs load (queue-based load-leveling) but is eventually consistent and harder to trace. A broker is **operational surface area** — introduce it only on a driver (`project-architecture-config.md` locked rule: no Kafka/broker without a driver-justified ADR); see `async-messaging.md` and `event-driven-patterns.md`.
- **The retry-storm / cascading-failure trap** — naïve retries *amplify* load on a degraded dependency, turning a brownout into an outage; defenses are jitter + a retry budget + a circuit breaker + load shedding, applied together. Name this trap explicitly in any retry decision (it is a tradeoff point per `shared/evaluation-method.md`).
- **MADR ADRs** (`shared/adr-template.md`) — each resilience decision (a timeout value, a breaker threshold, sync→async, a no-retry choice) is recorded with its driver, tradeoff, and rejected alternatives; **diagrams-as-code** (`shared/diagramming-standards.md`) — a C4 dynamic/sequence view of the failure path (breaker-open, fallback) where a scenario warrants it, sync solid / async dashed.

## Step sequence
- **audit:** Frame the availability/performance drivers + build the **interaction inventory** from the C4 L2 + API contracts (every cross-process call site: caller, callee, sync/async, idempotent?) → walk **each** call against the four-part failure-mode hard oracle (bounded timeout? retry-or-explicit-no-retry with backoff+jitter+budget? breaker/bulkhead? fallback/fail-fast?) and against the fallacies (latency budget across the chain, no unbounded sync fan-out, idempotency on every retry) → flag retry-on-non-idempotent, infinite/default timeouts, missing breakers on the critical path, shared pools with no bulkhead, and any retry with no storm defense → emit findings + a **risk register** (read-only, authors nothing, edits no source).
- **design:** **Frame** (confirm the drivers + the interaction inventory, read-only) → **Explore** (read-only: ≥2 divergent resilience postures per contested interaction — e.g. for a document-processing or payment integration: *sync-with-breaker-and-fail-fast* vs *async-queue-with-retry-and-status-poll*; each a concrete spec with its timeout/retry/breaker/bulkhead/fallback values + tradeoffs, not the same call recolored) → **Evaluate** (`architecture-evaluator`, ATAM, order-swapped to kill position bias: score each posture against the availability/latency scenarios + the hard oracles; name the **sensitivity points** — e.g. "p99 is sensitive to the retry budget" — and the **tradeoff points** — e.g. "async-queue *improves* availability/blast-radius but *worsens* immediacy/consistency and adds a broker"; penalize accidental complexity — no breaker on an in-process call; pick a winner, graft from runners-up) → **Document** (the **one writer**, `solution-architect`: author the **per-call resilience contract** into `architecture/<slug>/` — the timeout/retry/breaker/bulkhead/fallback table, the sync-vs-async decision per interaction, the C4 dynamic failure-path view, and one **MADR ADR per significant decision** with tradeoff + rejected alternatives; reuse the named patterns, preserve one coherent resilience policy for conceptual integrity) → **Verify** (the hard gates below; evaluator re-scores craft from the artifacts) → loop ≤ 10 or pass.

## Assertions & exit gate
- **Every** call in the interaction inventory has all four: a **bounded** (connect+read) timeout, a retry policy **or** a recorded no-retry, a **circuit breaker and/or bulkhead**, and a **fallback or explicit fail-fast** — traced in a per-call table; **zero** call sites with an unhandled failure path.
- **No retry on a non-idempotent operation** without an idempotency key; every retry uses **backoff + jitter**, a **max-attempt cap**, and a **retry budget** (storm defense present).
- **No unbounded synchronous call chain**: the summed worst-case timeouts down any sync chain fit inside the originating scenario's latency budget, and each child timeout < its parent's (the budget decreases down the chain).
- **Sync-vs-async is stated per interaction with a reason**; any new broker/queue traces to a driver-justified ADR (no broker by default — `project-architecture-config.md`).
- **Fallacies honored** (`shared/fallacies-and-tradeoffs.md`): network-unreliable (#1), latency-budget-across-hops (#2), large-payload backpressure/projection (#3), no hardcoded hosts (#5) — each accounted for where it applies.
- Every prioritized availability/performance scenario has a **met response measure** argued by the resilience contract (a breaker-trip threshold, a fail-fast bound, a fallback path) — by a capacity/failure model or a named pattern's guarantee, not assertion; every resilience decision is a **complete MADR ADR** (context · ≥2 options · decision · positive+negative consequences) with its `Driver refs`; any failure-path diagram **compiles** (`shared/diagramming-standards.md`).
- **Gate:** hard oracles green (every call's failure mode defined + idempotent-only retries with storm defense + no unbounded sync chain + sync/async stated + fallacies honored + scenarios met + ADRs complete + diagrams compile) AND (design) rubric mean ≥ 0.8 (consistent policy / bounded blast radius / right-sized — no resilience ceremony the topology doesn't need).

## Output
Write `artifacts/service-communication-resilience/report.json` per `shared/report-format.md` — the `drivers[]` block (the availability/performance scenarios protected), `findings[]` per `shared/finding-schema.md` (`type:"integration"`/`"availability"`/`"risk"`; `oracle:"hard"` for an unbounded/default timeout, a missing breaker/bulkhead on a critical path, a retry on a non-idempotent op, an unbounded sync call chain, or a retry with no storm defense; each finding traces its `driver_ref` (`QAS-AVAIL-*`/`QAS-PERF-*`) and carries `tradeoff` + `alternatives_rejected`), and for design mode the `verification` block (the per-call failure-mode coverage matrix, the latency-budget check across each sync chain, hard-oracle results, the evaluator's `rubric_score`). Authored artifacts land in `architecture/<slug>/` — the resilience-contract table + sync/async decisions in `design-doc.md`, the C4 dynamic failure-path view in `diagrams/`, one MADR ADR per decision in `adr/`, the storm/blast-radius reasoning in `tradeoffs.md`, and any fitness-function spec (e.g. "no call site without a configured timeout" as a static check, "no synchronous call from API→payment-provider outside the breaker" as a dependency rule) in `fitness-functions.md` — never scattered.

## Guardrails
Per `shared/guardrails.md`: **propose & document — never write product code or wire the resilience library/middleware**; the writer authors the resilience contract (ADRs, the per-call table, the failure-path diagram, the fitness-function *spec*) into `architecture/<slug>/`, the **controller** implements it (the timeout/retry/breaker config in `backend/**` is the controller's job; a runnable fitness-function *check script* into `tests/`/`fitness/` is the only thing that may land outside the architecture folder). **Ground every resilience decision in a driver** — no breaker/bulkhead/async with no availability or latency scenario behind it is accidental complexity (right-size; don't add ceremony the topology doesn't need — flag the over-engineering). **Reuse the named patterns** (`../80-patterns/resilience-patterns.md`) before inventing a bespoke retry/breaker mechanism; prefer the existing stack and **no new broker without a driver-justified ADR** (`project-architecture-config.md`). **One writer** preserves a single coherent resilience policy (conceptual integrity). **Record the tradeoff and the rejected alternatives** on every decision (sync-vs-async, the breaker threshold, the no-retry choice) — a resilience recommendation with only upside is incomplete. Read-only on any live system; never run a failure-injection / chaos experiment against a real environment — the architect authors the *plan*, the operator/`qa-agent` runs it. Never write to project memory.
