# dependency-management

**Group:** 10-styles-decomposition  ·  **Runs as:** subagent: ../../agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Manage coupling and cohesion across module and service boundaries — keep the dependency graph acyclic, oriented toward stability, and carrying only the kind of coupling each edge warrants. Answers: *does the decomposition's dependency structure support independent changeability, or is it a distributed monolith disguised as bounded modules?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack and deployment target (e.g. a modular-monolith pair — a frontend SPA + a backend API over a primary datastore); the team topology (which team/owner is responsible for which module); constraints (no new datastore/broker without driver-justified ADR); the artifact locations (`architecture/<slug>/`).
- The architecture **drivers**: the functional decomposition produced by `component-design.md` (the layering/hexagonal/ports-and-adapters choices + the module/component boundaries already drawn); the prioritized quality-attribute scenarios from `../00-drivers/quality-attribute-scenarios.md` (the utility tree) — in particular the **evolvability / modifiability** and **maintainability** leaves, plus any security or availability scenario that places constraints on shared state across boundaries.
- The dependency graph source: the codebase's import graph (per the module layout named in `project-architecture-config.md` — e.g. frontend `src/**`, backend module imports across the API entrypoint, utilities, routers, services) — extracted by static analysis (Madge, `pydeps`, `deptrac`, or `grep -r "^from\|^import"`). A flow with no import graph to inspect downgrades to a structural/topology-only audit and says so.
- Preconditions: the module boundaries from `component-design.md` are identified (or the target decomposition is stated as the design scope); the utility tree contains at least one evolvability or modifiability scenario (else this flow surfaces that gap and pends on `quality-attribute-scenarios.md`).

## Oracle (source of truth)
The **acyclic-dependencies principle** (ADP, Robert C. Martin — *Agile Software Development*), the **stable-dependencies principle** (SDP), and the **stable-abstractions principle** (SAP) — the three structural coupling laws — are the external truth. A dependency graph that violates them fails a hard oracle regardless of how clean the code looks otherwise.

The governing evolvability scenario is:

> **QAS-EVO-01** (business value: H · technical risk: H)
> *Source:* a developer on the team. *Stimulus:* modifies the implementation of one bounded context (e.g. changes an API route handler, a schema/validation model, or a frontend feature slice). *Artifact:* the module/bounded-context boundary containing the change. *Environment:* normal development — the rest of the system is unchanged. *Response:* only the containing bounded context requires rebuild and retest; no other context is forced to rebuild, retest, or redeploy. *Response measure:* **zero import cycles in the dependency graph** (topological sort exits 0); **all SDP instability-direction assertions pass in CI** (every depender has `I ≥ I` of its dependency; no stable module imported by a volatile one); a developer can change, build, and deploy the modified context independently within the deployment topology named in `project-architecture-config.md` without touching a second context's source or tests.

Every hard oracle below is evaluated against QAS-EVO-01. Any ADP or SDP violation is a `driver_ref: "QAS-EVO-01"` finding — independent changeability is defeated when the graph has cycles or the stability direction is inverted.

- **hard:**
  - **Acyclicity (ADP):** the module/package/service dependency graph is a DAG. Any cycle — `A → B → A`, or any longer chain — is a hard finding (`ARCH-DEPM-NNN`, `oracle:"hard"`, `severity:"major"` minimum, `driver_ref:"QAS-EVO-01"`). A cycle means you cannot release A without releasing B, and vice versa: independent deployability and changeability are defeated — the QAS-EVO-01 response measure (zero cycles, independent rebuild) fails. Cycles are structurally checkable (topological sort on the import graph; exit code non-zero if a cycle exists) and enforceable as a fitness function (`../60-evaluation/fitness-functions.md`).
  - **Stability direction (SDP):** dependencies point from volatile (frequently-changed) modules toward stable (rarely-changed) modules. A volatile module depended upon by many stable modules is a hard finding (`driver_ref:"QAS-EVO-01"`) — instability metric `I = Ce / (Ca + Ce)` (where `Ca` = afferent couplings, `Ce` = efferent couplings): a module with high `I` (many efferents, few afferents — volatile) must not be imported by modules with low `I` (stable). Measurable per module; every violation names the instability values (`I_depender`, `I_dependency`) and cites QAS-EVO-01 as the driver (a stable module forced to rebuild when a volatile dependency changes defeats the independent-changeability response measure).
  - **Appropriate coupling type at each edge:** no inappropriate coupling class crosses a boundary — specifically, **common coupling** (shared mutable global state, shared in-memory store, shared module-level singletons across bounded contexts) and **content coupling** (one module directly modifies another's internals) are hard findings at any named boundary. Stamp coupling (passing whole objects when only a subset is needed) and control coupling (passing flags that drive another module's logic) at a boundary are major findings. Data coupling (passing only the data the callee needs) is the target at every cross-boundary edge.
  - **Connascence ≤ contract (the ranked coupling metric):** every cross-boundary edge is additionally classified on the **connascence hierarchy** (Page-Jones, *What Every Programmer Should Know About Object-Oriented Design*, 1996) — static forms ranked weakest → strongest: **Name < Type < Meaning < Position < Algorithm**; dynamic forms ranked weakest → strongest: **Execution < Timing < Value < Identity**. The rule: **minimize the *strength* of connascence, especially across a quantum/service boundary** — the further apart two elements are (across a process, a service, a team), the weaker the connascence between them must be. At any named module/service/quantum boundary, connascence stronger than **Connascence-of-Name** (a named, versioned contract — an API schema, an event schema, a published interface) is a hard finding (`ARCH-DEPM-NNN`, `oracle:"hard"`, `driver_ref:"QAS-EVO-01"`), mirroring the `Connascence ≤ contract` oracle in `shared/quality-attribute-rubric.md`. Concretely: Connascence-of-Position (two services agreeing on positional tuple/array shape instead of named fields), Connascence-of-Meaning (a shared magic number/enum-as-int with no schema), Connascence-of-Algorithm (both sides independently re-implementing the same business rule, e.g. a hash or a validation routine, instead of calling a shared, versioned service), Connascence-of-Timing (a caller that only works if the callee responds within an unstated window), and Connascence-of-Identity (two components required to share the literal same object/row reference rather than an equivalent copy) are each a hard finding when found *across* a boundary — the fix is always to narrow the coupling to a named contract (an explicit schema, a documented ID, an idempotent async message) rather than to accept the stronger form. The same connascence, found *within* a single module/bounded context, is acceptable (even Connascence-of-Position or -Algorithm inside one team's code is a normal, cheap-to-refactor internal coupling) — the hierarchy is a boundary-crossing gate, not a blanket ban. Enforceable as a fitness function via **ArchUnitTS** (JS/TS architecture-conformance rules), **dependency-cruiser** (custom `forbidden` rules matching cross-boundary import/call patterns), or **deptrac** (PHP/Python layer-contract enforcement) — wire the check into `architecture/<slug>/fitness-functions.md` alongside the acyclicity and SDP checks.
  - **Fan-out failure surface (Metcalfe / `fallacies-and-tradeoffs.md`):** for service-level dependencies (if any exist or are proposed), the N×M failure surface is quantified: N services × M average external dependencies = the blast-radius upper bound. A design that grows N×M without a driver is a hard finding (accidental complexity, `guardrails.md §1`).
  - **No orphan dependency:** every import/dependency traces to a functional or quality driver. A dependency with no driver is accidental complexity (guardrails §2) — flag it for removal.

- **soft:**
  - **Cohesion:** elements within a named boundary change together and for the same reason (Yourdon/Constantine: functional > sequential > communicational > procedural > temporal > logical > coincidental — target functional or sequential cohesion within each module). Low cohesion within a boundary is a soft finding (evolvability rubric dimension).
  - **Graph legibility:** the dependency graph, rendered as a C4 Component view (`diagramming-standards.md`), is readable at a glance — no hairball. A module with afferent or efferent coupling counts beyond a threshold the team agrees on (e.g. `Ca > 10` or `Ce > 10` in a modular monolith context) is a legibility smell worth surfacing, but not a hard gate.
  - **Coupling minimization:** coupling at each edge is the minimum required for the interaction — no stamp coupling where data coupling would serve; no shared table where a narrowed API contract would serve.

## Standards & techniques
- **Coupling taxonomy (Yourdon/Constantine, 1979):** content → common → control → stamp → data — the six classes from tightest (worst) to loosest (best). Every cross-boundary edge is classified against this taxonomy; the finding names the class and the target class.
- **Connascence hierarchy (Page-Jones, 1996) — the ranked coupling metric:** a finer-grained ranking than the Yourdon/Constantine taxonomy, applied *in addition to it* at every cross-boundary edge. Static connascence (detectable by reading the code): **Name < Type < Meaning < Position < Algorithm** (weakest → strongest). Dynamic connascence (detectable only by running the code): **Execution < Timing < Value < Identity** (weakest → strongest). The governing rule is to **minimize the strength of connascence, especially across a quantum/service boundary** — degree and locality both matter, but strength is the axis this flow gates on: a boundary should carry nothing stronger than Connascence-of-Name (a published, versioned contract). Use this hierarchy to name *which* dynamic or static form a suspect edge exhibits (not just "stamp coupling" but "Connascence-of-Position across the `POST /orders` boundary") — the precision is what makes the finding actionable and what a fitness function can pattern-match.
- **Cohesion taxonomy (Yourdon/Constantine):** functional → sequential → communicational → procedural → temporal → logical → coincidental — target functional or sequential; flag temporal/logical/coincidental within a named bounded context as a split signal.
- **Acyclic-dependencies principle (ADP, Martin):** the dependency graph of modules/packages must be a DAG. Verify with a topological sort; encode as a fitness function in CI.
- **Stable-dependencies principle (SDP, Martin):** depend in the direction of stability; measure with the instability metric `I = Ce / (Ca + Ce)` per module. `I` = 0 is maximally stable (no outgoing deps); `I` = 1 is maximally volatile (no incoming deps). SDP: `I` of a dependency must be ≤ `I` of the depender — volatile modules must not be depended upon by stable ones.
- **Stable-abstractions principle (SAP, Martin):** stable modules (`I` ≈ 0) should be maximally abstract; volatile modules (`I` ≈ 1) may be concrete. The "distance from the main sequence" (`D = |A + I − 1|`) places each module on the Martin metric plane — the zone of pain (stable + concrete, hard to change) and the zone of uselessness (abstract + unstable) are both findings.
- **Fan-out failure surface (Metcalfe / `shared/fallacies-and-tradeoffs.md`):** N×M coupling cost grows super-linearly; quantify the blast radius before adding a new cross-module or cross-service dependency.
- **AKF scale cube (axis Y — functional decomposition):** dependency structure must support Y-axis splits (by business capability) without tangled cross-capability dependencies that force co-deploy. Cross-reference `decomposition-strategy.md` for the split criteria.
- **ATAM sensitivity/tradeoff point framing (`shared/evaluation-method.md`):** coupling at a boundary is always a **tradeoff point** (consistency/simplicity/coupling vs autonomy/evolvability); name the point, the attributes affected, and the chosen position on the curve.
- **EIP anti-corruption layer (ACL) pattern (`../80-patterns/cloud-design-patterns.md`):** where a volatile external system or a future deferred-integration boundary is crossed, an ACL decouples the internal model from the external contract — the seam for any deferred integration (per `project-architecture-config.md` evolvability constraint) is designed here as a one-way-door consideration.
- **Fitness functions (`../60-evaluation/fitness-functions.md`):** the acyclicity check and the SDP instability-direction check are the canonical fitness functions for this flow — they are executable, automatable, and belong in CI.

## Step sequence
- **audit:** Read `project-architecture-config.md` + the utility tree → extract the module/bounded-context boundary map from `component-design.md` (or derive from the existing codebase structure if the flow doc is absent) → extract the import graph via static analysis (per the language/stack named in `project-architecture-config.md` — e.g. Madge for a JS/TS module graph; `pydeps` / `grep` for a Python module graph) → run a topological sort; record every cycle → classify each cross-boundary edge by coupling class (Yourdon/Constantine) **and** by connascence form (Page-Jones static/dynamic hierarchy) — name the strongest form present at each edge → compute afferent/efferent counts and instability `I` per module → check stability direction (SDP) → check for common/content coupling violations and for any cross-boundary connascence stronger than Name → emit findings per `shared/finding-schema.md` grouped by type then severity (critical → cosmetic); build the risk register. Read-only; authors no artifacts.
- **design:** Frame (confirm drivers — evolvability QAS + decomposition boundaries from `component-design.md`; read-only) → Explore (≥2 divergent structural options for resolving each cycle or SDP violation — e.g. option A: extract a shared kernel abstraction; option B: invert the dependency via an interface/port; option C: merge the coupled modules into one cohesive unit; read-only, no artifact authored) → Evaluate (`architecture-evaluator` scores each option against the utility tree + hard oracles using ATAM; names the sensitivity point — "the extracted abstraction is a tradeoff point: it eliminates the cycle but adds an indirection layer that worsens readability" — and the tradeoff; kills position bias by order-swapping; picks a winner; grafts from runners-up; read-only) → Document (one writer — `solution-architect` — authors the C4 Component view as diagrams-as-code in `architecture/<slug>/diagrams/` showing the corrected dependency graph with sync/async edges labeled and trust boundaries drawn; one ADR per structural decision per `shared/adr-template.md`; adds fitness-function specifications to `architecture/<slug>/fitness-functions.md` for the acyclicity + SDP checks; updates `architecture/<slug>/tradeoffs.md` with sensitivity/tradeoff points; traces every decision to a driver) → Verify (run the acyclicity fitness function against the proposed graph structure; diagram-compile check; traceability completeness — every corrected edge traces to a driver; evaluator re-scores the rubric from the authored artifacts; loop ≤10 or pass).

## Assertions & exit gate
- The dependency graph is a DAG: topological sort exits 0; no cycle survives. Any cycle remaining is a blocker.
- Every cross-boundary edge is classified by coupling type; no common or content coupling at a named boundary.
- Every cross-boundary edge is additionally classified on the connascence hierarchy; no edge exceeds Connascence-of-Name across a quantum/service boundary — any stronger static (Type/Meaning/Position/Algorithm) or dynamic (Timing/Value/Identity; Execution is acceptable) form crossing a boundary is a blocker unless narrowed to a named contract or accepted as a driver-justified ADR exception.
- Every dependency points from higher `I` (volatile) to lower `I` (stable) — SDP holds for every module pair in the graph; any SDP violation names the instability values and the driver it threatens.
- Fan-out is quantified for any proposed service-level dependency; an N×M blast radius above the team-agreed threshold has a driver justifying it or is reduced.
- Every C4 diagram compiles and renders; every container/component in the diagram appears in the dependency graph and vice versa — no orphan boxes, no unlabeled arrows.
- Every structural decision (cycle break, abstraction introduction, boundary merge) is captured as a complete ADR (context · ≥2 options · decision · positive + negative consequences) tracing to a `QAS-*` in the utility tree — the primary driver for coupling decisions in this flow is `QAS-EVO-01`; additional QAS-* refs apply when a security or availability scenario constrains a boundary.
- Fitness-function specifications (acyclicity check, SDP instability-direction check) are written to `architecture/<slug>/fitness-functions.md` and are executable in CI without additional tooling beyond Madge/`pydeps`/`deptrac`.
- **Gate:** hard oracles green (acyclic graph, no common/content coupling at boundaries, no cross-boundary connascence stronger than Name, SDP direction holds, all diagrams compile, all ADRs complete, every decision traces to a driver) AND rubric mean ≥ 0.8 (cohesion within boundaries, graph legibility, coupling minimized to interaction need).

## Output
Write `artifacts/dependency-management/report.json` per `shared/report-format.md` — findings and risks per `shared/finding-schema.md` (`type:"decomposition"` for coupling/cycle/connascence findings; `type:"evolvability"` for cohesion and legibility findings; `oracle:"hard"` for cycles, SDP violations, common/content coupling at a boundary, cross-boundary connascence stronger than Name, missing ADR or driver trace; `oracle:"soft"` for cohesion and legibility gaps). Each connascence finding names the specific form (e.g. "Connascence-of-Position") and the boundary it crosses. The report includes the verification block (fitness-function results: acyclicity pass/fail + SDP pass/fail per module pair + connascence-≤-contract pass/fail per boundary edge, executable via ArchUnitTS/dependency-cruiser/deptrac; diagram-compile result; traceability coverage count; evaluator rubric score) for design mode.

Authored design artifacts land in `architecture/<slug>/` (per `project-architecture-config.md`):
- `architecture/<slug>/diagrams/dependency-graph.{puml|md}` — C4 Component view of the corrected module graph as diagrams-as-code (PlantUML + C4-PlantUML or Mermaid `graph`); every edge labeled with coupling type, protocol, and direction; sync/async visually distinct; trust boundaries drawn; compiles without error.
- `architecture/<slug>/adr/NNNN-<decision>.md` — one MADR ADR per structural change (cycle break, boundary redraw, abstraction introduction, ACL placement for a deferred integration seam).
- `architecture/<slug>/fitness-functions.md` — acyclicity check (Madge/`deptrac`/`pydeps` command + expected exit code 0) and SDP instability-direction check (per-module `I` computed + direction assertion) as CI-executable specifications.
- `architecture/<slug>/tradeoffs.md` (appended) — sensitivity points and tradeoff points identified during ATAM evaluation, with the winning position on each axis named and justified.

## Guardrails
Per `shared/guardrails.md`: this flow proposes and documents structural decisions — it never writes product code (`src/**`, `frontend/**`, `backend/**`), never edits import statements, and never applies IaC. Static-analysis tools may read the codebase (import graph extraction, coupling metric computation) but write nothing back. The evaluator is always a separate agent from the writer; it grades from the artifacts, not the writer's prose claims. Every finding is driver-traced (`finding-schema.md` §driver_ref required) — a cycle or coupling violation with no driver it threatens is logged as a risk, not a critical finding. Reuse the existing module boundaries from `component-design.md` before redrawing them; a boundary redraw is a one-way-door decision and requires a driver-justified ADR. Any deferred integration seam (per `project-architecture-config.md` evolvability constraint) is designed here as an ACL placement recommendation — the ACL is specified in an ADR, never built by this flow. Cross-reference `../60-evaluation/fitness-functions.md` for the fitness-function encoding conventions and `shared/fallacies-and-tradeoffs.md` for the coupling↔autonomy canonical tradeoff that every structural decision is evaluated against.
