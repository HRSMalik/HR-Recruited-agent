# api-first-contracts

**Group:** 10-styles-decomposition  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Design and validate the **boundary contracts before the implementation** — OpenAPI (REST), AsyncAPI (events), and protobuf/gRPC schemas as the machine-readable source of truth for every cross-boundary interface — with an explicit versioning policy, a backward/forward compatibility definition, and a contract-diff fitness function that gates breaking changes. Answers: *what is the contract, who owns it, what constitutes a breaking change, and how is it enforced?*

## Inputs & preconditions

- From `project-architecture-config.md`: the project's stack (e.g. an API framework with typed request/response schemas, a client application consuming the API, a document or relational data store), the deployment target (per `project-architecture-config.md` — same-origin, gateway-fronted, or public API), the compliance regime (per `project-architecture-config.md` — if the domain carries PII or regulated data, schema changes that expose or restructure sensitive fields are a compliance event, not just a breaking change), team topology (per `project-architecture-config.md` — number of teams/tracks and whether there is a coordinated-deploy guarantee across them).
- The architecture **drivers**: the prioritized quality-attribute scenarios from `workflows/00-drivers/quality-attribute-scenarios.md` — in particular the **evolvability driver** (per `project-architecture-config.md`, the bounded contexts or integration seams most likely to be sources of new contracts — e.g. a deferred integration seam, a payment or notification integration), the **security driver** (any trust boundary carrying sensitive data; authn/authz must be explicit in the contract, not implementation-implicit), and the **maintainability / modifiability driver** (per `project-architecture-config.md` — if client and server, or multiple services, are developed on separate tracks, independent evolution without a coordination meeting is a stated team need).
- Preconditions: at least one cross-boundary interface exists or is being designed; the functional scope (which capabilities cross which boundary) is established; the decomposition from `decomposition-strategy.md` (sibling: `workflows/10-styles-decomposition/decomposition-strategy.md`) identifies the named boundaries. If no decomposition exists, frame the contract boundaries from the C4 L2 containers identified so far (per `project-architecture-config.md` — e.g. client → API → data store; any future integration seam).

## Oracle (source of truth)

**hard:**
- Every cross-boundary interface has a **machine-readable contract** (OpenAPI 3.x for REST endpoints, AsyncAPI 2.x/3.x for event channels, protobuf IDL for gRPC if introduced) committed to the repository and treated as the primary source of truth — the implementation must match the contract, not the reverse. An interface documented only in code comments or a wiki page fails this oracle.
- A **versioning policy** is stated and applied to every contract: semantic versioning (`MAJOR.MINOR.PATCH`) for REST contracts; event schema versioning with an explicit compatibility matrix (backward-compatible additions vs. breaking field removal/rename). The policy must name *what constitutes a breaking change* and *what constitutes an additive/non-breaking change* — "it changed" is not a policy.
- A **contract-diff fitness function** is specified: on every PR that touches a contract file, a diff is computed and classified as breaking/non-breaking per the versioning policy; breaking changes require a MAJOR version bump (or an explicit, signed-off accepted risk). This is an executable architectural characteristic check (`workflows/60-evaluation/fitness-functions.md`), not a manual code-review step.
- The **Postel tension is resolved per boundary**: for each contract, the policy states whether consumers validate strictly (reject unexpected fields) or liberally (ignore unknown fields) — this decision is explicit and consistent across consumers of the same contract; a boundary where some consumers validate strictly and others liberally is a finding.
- **Consumer-driven contract (CDC) tests** are specified for every consumer–provider pair where the consumer is a separately deployable/releasable unit: the consumer publishes a pact/contract fragment asserting what it uses; the provider verifies against it. For a client↔server boundary developed on separate tracks with no coordinated deploy (per `project-architecture-config.md`), a CDC test is the hard oracle for "the client won't break when the server deploys."

**soft:**
- Contracts are designed for **additive evolution**: field additions and optional extensions are always non-breaking; required-field addition, field removal, type narrowing, and enum-value removal are always breaking. This asymmetry should be structurally reflected in the schema (use `required` sparingly; prefer nullable/optional for fields that may evolve).
- The contract is the **agreement between teams**, not the implementation's shadow. Schemas authored after the fact from a running implementation drift from the real contract surface and hide implicit dependencies; the contract-first posture means schema authoring precedes code authoring.
- Contracts expose **capability, not internals**: internal storage field names, internal enum values, and auto-generated IDs should not leak into the API contract as-is; a mapping/translation layer between the persistence model and the contract model is the expected pattern.

## Standards & techniques

- **Contract-first / API-first design** — the schema file (`.yaml`, `.json`, `.proto`) is authored first and is the input to code generation (server stubs, client SDKs, mock servers); the implementation is a conformant realization of the contract, not vice versa. Toolchain: OpenAPI Generator, Spectral (linting), Prism (mock server from the spec).
- **OpenAPI 3.x** for REST — paths, request/response schemas, security schemes (`bearerAuth` / JWT, or whatever authn scheme the project's trust boundary requires per `project-architecture-config.md`), error shapes (RFC 7807 `Problem+JSON` for consistent error contracts across all endpoints), pagination envelopes. Every endpoint's request and response body must be schema-typed — `type: object` with no `properties` is not a contract.
- **AsyncAPI 2.x/3.x** for event channels — if/when a deferred integration seam or a downstream integration (e.g. payments, notifications) introduces an event bus. Channels, message schemas, bindings (broker type), publish/subscribe roles. The same breaking-change rules apply: a message schema change that removes a field or changes a field type is breaking for downstream consumers.
- **Protobuf / gRPC** — only introduce if a concrete driver demands it (e.g. a latency-sensitive, high-throughput internal service call where HTTP/JSON overhead is a bottleneck). Per `project-architecture-config.md` ("boring tech + existing stack first" or equivalent guidance): default is REST/OpenAPI; protobuf needs a driver-justified ADR. The serialization-cost fallacy (Fallacy 7 — transport cost is zero) is the tradeoff to name if protobuf is proposed.
- **Semantic versioning of contracts** — `MAJOR` for breaking changes (existing consumers must update); `MINOR` for additive backward-compatible changes (new optional fields, new endpoints); `PATCH` for documentation/description corrections. The version appears in the contract file's `info.version` and, for MAJOR, in the URL path (`/api/v2/`) or a content-type negotiation header. Never silently break a contract at the same version number.
- **Consumer-driven contracts (CDC)** — Pact (or compatible framework): the consumer writes a pact file asserting the subset of the contract it actually uses; the provider runs pact verification as part of its CI. This inverts the typical "provider changes, consumer breaks at runtime" flow into a "provider can't merge a change that breaks a known consumer" gate. Cross-reference: `workflows/30-integration/event-driven-patterns.md` for the event-channel CDC variant.
- **Backward vs. forward compatibility** — backward compatibility: a new provider version is compatible with old consumers (old consumers continue to work). Forward compatibility: an old provider version is compatible with new consumers (new consumers can be deployed before the provider catches up). State which direction(s) are required per boundary; a client↔server boundary where the client is deployable independently (per `project-architecture-config.md`) requires at minimum backward compatibility; forward compatibility is an (H,H) driver if client and server must be deployable in any order.
- **Postel's Law tension** — Postel's Law ("be conservative in what you send, liberal in what you accept") aids robustness under schema evolution but can mask contract drift: a consumer silently ignoring unknown fields never notices when the field it stopped receiving was the one it needed. Pair liberal acceptance with CDC tests (which assert what the consumer uses) to surface hidden dependencies. Per `shared/fallacies-and-tradeoffs.md`: the network-is-homogeneous fallacy (Fallacy 8) is exactly the failure mode liberal parsing avoids; but without CDC, it replaces a hard runtime failure with a silent data-quality regression.
- **Breaking-change classification** — the canonical taxonomy: **breaking**: removing a field, renaming a field, changing a field's type, narrowing a field's allowed values (enum reduction), adding a required request field, changing HTTP method or path, changing error status codes used by consumers. **Non-breaking**: adding an optional response field, adding a new endpoint, adding a new optional request parameter, adding an enum value (consumers must handle unknown values — document this expectation in the contract). The fitness function classifies every diff against this taxonomy.
- **STRIDE on the contract boundary** — the API contract is a trust boundary: spoofing (is the caller who they claim? → an explicit authn scheme required in the OpenAPI security section), tampering (are contract fields validated on ingress? → framework-level validators are the mechanism; schema must define `pattern`, `minLength`, `maxLength`, `enum` constraints to make the validation surface explicit), repudiation (audit logging of API calls is a driver — the contract should define which operations trigger audit events), information disclosure (does the error response leak internals? → RFC 7807 shapes should be audited against this), DoS (are there rate-limit / payload-size constraints? → document as contract-level non-functional properties). Cross-reference: `workflows/40-cross-cutting/security-architecture.md` for full STRIDE.

## Step sequence

- **audit:** Read the existing API surface → enumerate every cross-boundary interface (route → request/response schema in the project's schema layer; event channels if any exist) → check each interface against the hard oracles: (1) does a machine-readable contract file exist as the source of truth, or is the implementation's schema layer the implicit contract? (2) is a versioning policy stated anywhere? (3) is there a breaking-change detection step in CI? (4) is the Postel stance stated per consumer? (5) are CDC tests specified? → emit findings (read-only; no contract files authored; the controller decides which to act on). Cross-reference `workflows/70-documentation/traceability-mapping.md` to flag any contract-to-decision traceability gaps.

- **design:** Frame (confirm which boundaries need contracts from `project-architecture-config.md` + the utility tree; identify consumer–provider pairs; confirm the versioning scope — internal-only today, or must the contract remain stable as a public/partner API in future) → Explore (≥2 divergent directions — e.g. Direction A: generate OpenAPI from existing implementation schemas with a strict CDC gate; Direction B: hand-author the OpenAPI spec first and regenerate implementation validators from it; each as a concrete spec with the versioning policy + breaking-change enforcement mechanism + tradeoff) → Evaluate (skeptical evaluator scores each against the utility tree — evolvability, maintainability, security hard oracles; ATAM tradeoff analysis: code-gen-from-model is simpler today but makes the contract the implementation's shadow; spec-first is the contract-first ideal but requires toolchain investment; order-swapped to kill position bias; picks winner and grafts best ideas from runner-up) → Document (one writer authors: the OpenAPI spec(s) in `architecture/<slug>/contracts/`; the versioning policy ADR; the CDC test specification and the breaking-change fitness function gate appended into `architecture/<slug>/fitness-functions.md`; the breaking-change taxonomy; the Postel policy per boundary; C4 L2 diagram updated to show contract artifacts as annotations on the client→API relationship) → Verify (fitness-function spec verifiable? — run `spectral lint` on the authored spec file; check every endpoint has a typed request + typed response + security scheme; check the CDC test specification is executable against a stub of the provider; traceability: every contract traces to a boundary decision, every boundary decision traces to a driver; STRIDE coverage on the API trust boundary; evaluator re-scores from the authored artifacts) → loop ≤10 or pass.

## Assertions & exit gate

- Every cross-boundary interface has a machine-readable contract file committed to the repo as the source of truth; no interface documented only in code or prose.
- The contract file passes `spectral lint` (or equivalent OpenAPI/AsyncAPI linter) with zero errors; every endpoint has a typed request schema, a typed response schema, and a security scheme entry.
- The versioning policy is stated in the ADR: what version field is used, what increment signals a breaking change, where the version appears in the URL or header.
- The breaking-change taxonomy is documented and the fitness function specification is present: a CI step is specified (not just described) that diffs the contract on each PR and blocks merge on an unversioned breaking change.
- The Postel stance (strict vs. liberal validation) is stated per consumer; CDC tests are specified for every separately-deployable consumer–provider pair (per `project-architecture-config.md`: client↔server at minimum).
- Traceability: every contract boundary traces to a decomposition decision (ADR) and a driver (quality-attribute scenario); no contract with no driver (that is accidental surface area).
- STRIDE controls are stated on the API trust boundary: authn scheme in the OpenAPI security section, input-validation constraints in the schema, error shapes audited for information disclosure.
- **Gate:** hard oracles green (machine-readable contract exists + versioning policy stated + breaking-change fitness function specified + Postel stance explicit + CDC tests specified for separately-deployable pairs + STRIDE on the boundary) AND rubric mean ≥ 0.8 (evolvability, conceptual integrity, right-sizedness — no over-specified contract surface that makes every change breaking).

## Output

Write `artifacts/api-first-contracts/report.json` per `shared/report-format.md` — findings per `shared/finding-schema.md` (`type:"integration"` for missing contracts, `type:"evolvability"` for versioning/compatibility gaps, `type:"security"` for trust-boundary STRIDE gaps; `oracle:"hard"` for missing machine-readable contract / missing versioning policy / no breaking-change gate / unresolved Postel stance; `oracle:"soft"` for contract-is-implementation-shadow / over-specified surface / conceptual integrity craft issues). Every finding carries `driver_ref` tracing to a quality-attribute scenario from `workflows/00-drivers/quality-attribute-scenarios.md`. The risk register includes: (1) the risk that client and server drift silently with no CDC gate (likelihood×impact against the evolvability driver); (2) the risk that the implementation's own schemas remain the implicit contract surface and an internal refactor becomes an undetected breaking change.

For design mode, include the verification block in the report: (a) **fitness-function results** — `spectral lint` pass/fail count on the authored OpenAPI spec, breaking-change gate classification result (breaking / non-breaking / unclassifiable per the taxonomy), CDC test specification executability check (does the pact file resolve against a stub of the provider?); (b) **traceability coverage** — every contract boundary traces to a decomposition ADR and a quality-attribute scenario, every quality-attribute scenario with an API-surface implication has a corresponding contract entry (coverage %, any orphan scenarios or orphan contracts called out explicitly); (c) **hard-oracle results** — machine-readable contract exists (Y/N per interface), versioning policy stated in ADR (Y/N), breaking-change fitness function specified and CI-integrated (Y/N), Postel stance explicit per consumer (Y/N), CDC tests specified for all separately-deployable pairs (Y/N), STRIDE controls present on the API trust boundary (authn scheme / input-validation constraints / error-shape audit — each Y/N); (d) **rubric score** — mean across the three soft dimensions (evolvability: additive-evolution posture and conservative required-field use; conceptual integrity: one consistent contract-first criterion system-wide; right-sizedness: no over-specified surface that makes every change breaking) scored 0.0–1.0 each, mean reported; gate requires mean ≥ 0.8.

Design artifacts land in `architecture/<slug>/`:
- `contracts/` — the authored OpenAPI/AsyncAPI spec files (e.g. `contracts/<system>-api-v1.yaml`)
- `adr/NNNN-api-contract-first.md` — the contract-first decision ADR (context: client and server on separate tracks, per `project-architecture-config.md`; options: code-gen-from-implementation-schema vs. spec-first with code-gen; decision + consequences + fitness-function follow-ups)
- `adr/NNNN-api-versioning-policy.md` — the versioning policy ADR (semver scheme, breaking-change taxonomy, URL-path vs. header versioning, one-way-door flag: the version scheme is hard to change once consumers exist)
- `fitness-functions.md` — the breaking-change fitness function spec (tool: `oasdiff` / `openapi-diff` / custom Spectral ruleset; inputs: current contract + PR diff; output: breaking/non-breaking classification; CI integration point; the contract-diff gate entry is appended to this file alongside any other fitness checks for this slug)
- `diagrams/` — updated C4 L2 container diagram annotating the client→API relationship with the contract artifact and the CDC test gate

## Guardrails

Per `shared/guardrails.md`: propose and document — never write product code. The OpenAPI spec files authored here are *architecture artifacts* (the contract as the source of truth), not route definitions or model files in the implementation's own framework (those are the implementation that the controller writes to conform to the contract). Reuse the existing schema surface from the project's implementation (per `project-architecture-config.md`) as the *reference* input for the initial contract draft; the authored `.yaml` spec is what drives conformance. One writer (`solution-architect`) authors all contract files and ADRs — no concurrent writes to the same spec file. Record the Postel-vs-strict tension as the tradeoff it is: neither is universally correct, and the chosen stance must survive the breaking-change taxonomy it pairs with. Ground every contract boundary and versioning decision in a driver (`project-architecture-config.md`, `shared/quality-attribute-rubric.md` grounding rule); a contract with no driver is accidental surface area — flag it and recommend removal. Cross-reference: `workflows/30-integration/api-design.md` for REST design conventions; `workflows/30-integration/event-driven-patterns.md` for AsyncAPI event-channel contracts; `workflows/70-documentation/traceability-mapping.md` for contract-to-driver traceability coverage.
