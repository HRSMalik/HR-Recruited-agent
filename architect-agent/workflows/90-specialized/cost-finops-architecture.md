# cost-finops-architecture

**Group:** 90-specialized  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
Model the **cost of the architecture** — make cost a first-class quality attribute with a **bounded, order-of-magnitude estimate**, not an afterthought discovered on the invoice. This flow attaches a **cost-of-each-decision** to every architecturally-significant topology choice: it builds the **TCO model** (build + run + egress + operational toil over a stated horizon), **right-sizes** each resource to its real load (the AKF scale cube applied to spend, not just throughput), computes the **serverless-vs-provisioned crossover** and the **build-vs-buy / managed-TCO** point rather than guessing them, prices the often-forgotten **cross-AZ/region data-transfer + egress** (fallacy #7 — transport cost is not zero), and installs **FinOps practices** (cost visibility, allocation/attribution, optimization) so spend is *observable and owned*, not a black box. It answers the *decision*: *for each component, what does it cost to build and run against the stated budget, is it right-sized to its driver, and where is the managed-vs-self / serverless-vs-provisioned crossover that flips the economics?* — never "make it cheap." Cost is weighed **against** the driver (you do not starve an RTO or a `QAS-PERF-*` budget to save a dollar); the deliverable is a cost model + cost-bearing ADRs, never a reflexive "go serverless to save money."

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target (per `project-architecture-config.md` — e.g. a same-origin SPA + API-service-behind-reverse-proxy topology with a single primary datastore; deploy-ready containers; **prod deploy = a separate ticket** — registry images, real secrets, env-driven origins, managed DB are *priced here, not assumed built*), the cloud target (if none stated, the unit-price source is itself an open driver — surface it, don't assume AWS list price), the **scale/SLO targets** (the cost model is sized to the *real* concurrency stated in `project-architecture-config.md` — an internal enterprise tool with a handful of internal users is not consumer-scale; right-size, never résumé-drive), the compliance regime (PII residency can force a region/managed-service choice that *costs more* — that's a justified cost, not waste), the team topology (small team, Conway — **operational toil is a dominant run-cost line**, because what the team can't operate at 3am costs in incidents and headcount), the stated budget/constraint, and where `architecture/<slug>/` + the ADR log + `capacity-model.md`/`tradeoffs.md` live.
- The architecture **drivers** — the prioritized utility tree from `../00-drivers/quality-attribute-scenarios.md` (→ `architecture/<slug>/quality-attributes.md`): cost is itself a leaf (a `QAS-COST-*` budget — the monthly/annual run-cost ceiling) **and** the constraint every *other* scenario's solution is priced against (the redundancy a `QAS-AVAIL-*` RTO demands has a price; the headroom a `QAS-PERF-*` p99 demands has a price). **No budget driver stated → the cost model still runs** (every design needs a bounded estimate), but the gate downgrades to "estimate produced + each decision's cost recorded" and the *missing budget* is surfaced as an open driver question — never invent a budget number.
- The **capacity model** (`../40-cross-cutting/performance-capacity.md` → `architecture/<slug>/capacity-model.md`): the back-of-envelope sizing (RPS, concurrency, data volume, growth) is the **input to the cost model** — you cannot price a topology you haven't sized. No capacity model → request/derive the sizing first, or `status:error` (a cost estimate with no load model is a fabricated number).
- The topology under review — the C4 deployment view (`../50-infrastructure/deployment-topology.md`), the proposed managed-vs-self splits (`../50-infrastructure/cloud-well-architected.md`), the current `tradeoffs.md`.
- Preconditions: the source-of-truth docs' NFRs + budget read (the BRD/PRD/understanding log named in `project-architecture-config.md`) — **never invent a price, an RPS, or a budget**; the **unit-price source** identified (provider list price / negotiated rate / committed-use discount) before any cost number is written — a cost model on the wrong rate card is a wrong model.

## Oracle (source of truth)
The **stated budget/constraint + the utility tree + the capacity model + the provider's actual unit prices** — never "it's cheap," never "it scales" with no cost attached, never a vendor's "this is cost-optimized" in the abstract. The grounding rule holds: *no number + no traced driver = a preference, not a finding* (`shared/quality-attribute-rubric.md`). Cost is the AWS/Azure Well-Architected **Cost Optimization pillar** made concrete against *this* budget and *this* load.

- **hard (deterministic — measurable / traceable / structural):**
  - **Order-of-magnitude cost estimate against the stated budget** (the `cost_bounded` hard oracle, `shared/quality-attribute-rubric.md`) — the design carries a `cost = build (eng-effort) + run (infra + operational + opportunity)` estimate over the **stated horizon** (e.g. 1–3 yr TCO), compared to the budget/constraint. A ±2× order-of-magnitude estimate that *names its assumptions* (the rate card, the RPS, the data volume) is a pass; a precise-looking number with no traced load model is a *worse* fail than an honest range. "Scale infinitely, cost unstated" blocks.
  - **Cost-of-each-decision recorded per major topology choice** — every architecturally-significant component (the datastore, the compute model, the broker/queue if any, the cache, the load balancer, the egress path) carries a **cost line** and the **cost tradeoff** of the choice (what the chosen option costs vs the rejected one, on dollars *and* on the attribute it bought). A topology decision with no cost tradeoff in its ADR fails `adrs_complete` — cost is a mandatory consequence, not optional.
  - **Egress + cross-AZ/region data-transfer accounted** (fallacy #7 — *transport cost is not zero*, `shared/fallacies-and-tradeoffs.md`) — wherever the topology crosses an AZ boundary, a region boundary, the public internet, or a managed-service boundary, the **data-transfer/egress cost is a line in the model**. A chatty cross-AZ call path, a cross-region replica, or a large-payload egress with the transfer cost silently assumed zero blocks (this is the single most common surprise on a cloud invoice). Serialization/marshaling CPU cost counts where it dominates.
  - **No "scale infinitely, cost unstated" component** — every elastic/autoscaling component (serverless, an autoscale group, a managed service that bills per-request or per-GB) has a **cost ceiling or a cost-per-unit-of-load** stated, so the bill is bounded under the *worst-case* load the capacity model allows. An unbounded-spend component (autoscale with no max, serverless with no concurrency cap, a per-request service with no rate ceiling) under a budget driver blocks — that is a financial SPOF.
  - **Serverless-vs-provisioned crossover computed, not guessed** — where the choice is on the table, the **break-even utilization** is computed: serverless (pay-per-request, scale-to-zero) wins below the crossover; provisioned (reserved/always-on, lower marginal cost) wins above it. The decision names *which side of the crossover this load sits on* (e.g. "at ~30% steady utilization an always-on small instance is cheaper than per-request FaaS for this access pattern; below ~10% serverless wins on scale-to-zero"). A serverless-vs-provisioned choice asserted without the crossover number is a hard finding.
  - **Build-vs-buy / managed-TCO computed, not guessed** — every "managed X vs self-run X" choice (a managed database service vs a self-hosted datastore on a VM, a managed load balancer vs a self-managed reverse proxy, a managed queue vs a self-run broker) carries a **TCO comparison** (build + run + **operational toil** over the horizon) — *not just the sticker price*. For a small team, operational toil (patching, on-call, the 3am incident) is frequently the dominant line and flips the managed option to lower TCO even at a higher list price; the model must include it (`shared/fallacies-and-tradeoffs.md` — *Build ↔ Buy*). A managed-vs-self choice priced on sticker alone, omitting toil, blocks.
  - **Cost attribution / allocation defined** — the design states **how spend is attributed** (per component / per environment / per tenant-or-company where multi-tenancy is a driver) — the FinOps *allocation* capability. A shared, un-tagged, un-attributable spend pool under a budget driver blocks the *visibility* the budget needs to be governed (you cannot optimize what you cannot attribute).
  - **Every cost number traces to a load input + a unit price** — `cost = load (from the capacity model) × unit_price (from the named rate card)`; an untraced cost figure (a number with no load driver and no price source behind it) is a preference, not a finding, and is dropped.

- **soft (rubric / craft — scored only after hard oracles are green):**
  - **Cost is weighed against the driver, not minimized blindly** — the cheapest option that *misses* an RTO or a p99 budget is **not** the right answer; cost trades against `QAS-AVAIL-*`/`QAS-PERF-*`, it does not override them (`shared/fallacies-and-tradeoffs.md` — *Cost ↔ Resilience/Performance*). A design that starves a required SLO to shave spend fails this; so does one that gold-plates redundancy no driver demanded. Size cost to the *driver*, not to the minimum or the maximum.
  - **The crossover + the TCO are computed, not hand-waved** — the serverless/provisioned and build/buy points are derived from the load + the rate card, not asserted from fashion. "Serverless is cheaper" with no utilization figure, or "managed is expensive" with no toil line, is a guess the rubric penalizes.
  - **Right-sizedness honored (the cardinal FinOps virtue)** — match supply to demand: an internal tool with predictable low overnight load is a candidate for **scheduled scale-down / scale-to-zero off-peak**, not an always-on max-size cluster; reserved/committed-use discounts are reasoned about where utilization is steady. Provisioned-but-idle is waste; so is autoscale headroom no `QAS-PERF-*` demands. The simplest, right-sized spend that meets every driver scores higher — accidental complexity *is* accidental cost.
  - **FinOps is designed in, not bolted on** — cost visibility (tagging/attribution), allocation (per-component/env/tenant), and an optimization cadence (review unit-cost trends, kill idle, rightsize) are part of the architecture's operability story, not a later spreadsheet. Cost is observable like latency is observable.
  - **Cost honesty** — the model states its assumptions (rate card, RPS, growth), its sensitivity (which assumption the estimate is most sensitive to — usually the load forecast or the egress volume), and what the rejected cheaper option would have cost on the *other* attributes. A cost estimate with only a headline number and no assumptions or sensitivity is incomplete craft.

## Standards & techniques
- **TCO cost modeling (build + run + egress + operational, over a horizon)** — total cost of ownership = **build** (one-time eng-effort to implement) + **run** (recurring: infra unit cost + **data-transfer/egress** + **operational toil** + opportunity cost) over the relevant horizon (1–3 yr). Operational toil is a first-class line, not a footnote — for a small team it often dominates. The horizon matters: a higher build cost amortizes; a higher run cost compounds. This is the CBAM **cost** input (`shared/evaluation-method.md`); `../60-evaluation/tradeoff-analysis-cbam.md` *consumes* this model to rank options by ROI — this flow *produces* the rigorous number CBAM uses.
- **Right-sizing + autoscaling-for-cost (AKF scale cube applied to spend)** — match provisioned resource to *measured* demand from the capacity model, not to a guessed peak. The AKF scale cube (X = clone/scale-out, Y = functional split, Z = data partition) is a *cost* lever as much as a throughput lever: scale-out adds linear cost; functional split adds operational cost per service (Metcalfe/fan-out — N×M surface, `shared/fallacies-and-tradeoffs.md`); over-partitioning adds idle replicas. **Match supply to demand** — scheduled scale-down off-peak, scale-to-zero where the access pattern allows, reserved/committed-use where utilization is steady. Right-sizing is the highest-ROI FinOps lever because idle capacity is pure waste.
- **The serverless-vs-provisioned crossover** — serverless/FaaS bills **per-request + per-GB-second** and scales to zero (no idle cost); provisioned (reserved instances / always-on containers) bills **per-hour regardless of load** but at a lower marginal cost per request. There is a **break-even utilization**: below it, serverless wins (you don't pay for idle); above it, provisioned wins (the always-on cost is amortized across enough requests). Compute the crossover from the capacity model's utilization curve; factor cold-start latency against `QAS-PERF-*` (serverless cost savings that breach a latency budget are a false economy — the cost↔performance tradeoff).
- **Build-vs-buy / managed-TCO** (`shared/fallacies-and-tradeoffs.md` — *Build ↔ Buy*) — building fits exactly but costs to build *and own*; buying/managed is faster, lower operational TCO, carries a real SLA — at the cost of a dependency + lock-in (a one-way-door cost, `shared/adr-template.md`). **Default to managed/boring unless a driver demands bespoke.** The TCO comparison must include operational toil; for a small team the managed option frequently wins on TCO even at a higher sticker price. The lock-in reversal cost (data egress, re-architecture) is itself a cost line and a one-way-door flag.
- **The cost of cross-AZ/region egress (the transport-cost fallacy)** — fallacy #7: *transport cost is zero* is false. Cross-AZ traffic, cross-region replication, public-internet egress, and inter-service marshaling all cost CPU **and money** (egress is often the single most surprising invoice line). Account for it wherever the topology crosses a boundary: a chatty cross-AZ call path, a large-payload cross-region replica, or an un-paginated egress is a cost finding even if it "works" functionally. Co-locate latency- and cost-sensitive components; use claim-check/compression for large payloads (`shared/fallacies-and-tradeoffs.md`, fallacies #3/#7).
- **FinOps — the three capabilities** (FinOps Foundation framework): **Visibility/Inform** (tag and report spend so it is *seen* — unit cost per component/env/tenant, not one opaque bill), **Allocation/Optimize** (attribute spend to its owner and right-size — kill idle, rightsize, buy commitments where steady), **Operate/Govern** (a cadence: review unit-cost trends, set budgets/alerts, make cost a shared engineering responsibility). FinOps makes cost an *observable, owned, continuously-optimized* attribute — the architecture analog of designing-in observability for latency.
- **Cost as a Well-Architected pillar** (AWS WAF / Azure WAF — **Cost Optimization**): right-size, match supply to demand, stop paying for what you don't use, measure & attribute, adopt a consumption model. Walk the pillar's questions but score against *this* budget and *this* load (`../50-infrastructure/cloud-well-architected.md` assesses the pillar at topology level; this flow builds the rigorous number behind it). Cross-checks the other pillars: cost trades against Reliability (redundancy costs) and Performance (headroom costs) — name the point chosen on the curve, never minimize one pillar blindly.
- **CBAM / one-way vs two-way doors** (`shared/evaluation-method.md`, `../60-evaluation/tradeoff-analysis-cbam.md`) — the datastore, the cloud provider, the serverless-vs-provisioned runtime, and any committed-use/reserved purchase are **one-way doors** (a 1-yr reservation is a sunk commitment; a managed-service lock-in costs egress to reverse) — price and analyze them deeply. Reversible cost knobs (instance size, autoscale policy, off-peak schedule) are **two-way doors** — set a sensible default cheaply, tune from real spend data, don't over-analyze.

## Step sequence
- **audit:** Frame the drivers + pull the budget (or flag it missing), the utility tree, the capacity model, and the **named unit-price source** → walk the existing/proposed topology component-by-component and attach a **cost line** to each (datastore, compute, cache, broker, LB, **every egress/cross-AZ-region transfer path**) → check the hard structural gates (order-of-magnitude estimate vs budget; cost-of-each-decision recorded; egress accounted per boundary; no unbounded-spend component; serverless/provisioned crossover computed where relevant; build/buy TCO incl. operational toil; cost attribution defined; every number traced to load × price) → compute the serverless-vs-provisioned and build-vs-buy crossovers from the load + rate card → flag right-sizing gaps (idle/over-provisioned/always-on-where-scale-down-fits) and any SLO-starving cheapness → emit findings + a **cost-risk register** (the spend-surprise risks: an unbounded autoscale, an un-modeled egress path, a load-forecast the estimate is sensitive to) **read-only — authors no artifact, edits no source, applies no infra**. Each finding names the component, the driver (`QAS-COST-*` or the SLO it trades against), the cost fact (load × price), the cost tradeoff, the rejected cheaper/dearer alternative, and the ADR-shaped recommendation.
- **design:** **Frame** (extract/confirm the budget driver + the per-component load from the capacity model + the named rate card + the horizon; surface a missing budget as an open driver, don't invent one — read-only) → **Explore** (≥2 *divergent* cost postures as concrete specs — e.g. *provisioned always-on small instances + a self-hosted datastore on a VM (lowest sticker, highest toil)* vs *a managed database service + container service with scheduled off-peak scale-down (higher sticker, lower toil + bounded run cost)* vs *serverless where the access pattern is bursty/low-utilization* — each a real topology + a TCO line + the crossover math + the egress accounting + an attribution scheme, **not the same diagram with cheaper labels**; read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each posture against the utility tree via **CBAM** — TCO vs benefit per option, the **sensitivity points** (which assumption the ranking is most sensitive to — usually the load forecast or the egress volume; re-rank under pessimistic/optimistic load) and the **tradeoff points** (where a cost choice helps spend but hurts an attribute — *scale-to-zero serverless cuts off-peak cost but adds cold-start latency vs `QAS-PERF-*`*; *a self-hosted datastore cuts sticker cost but raises toil + introduces a SPOF vs `QAS-AVAIL-*`*); **order-swapped to kill position bias**; penalizes the more elaborate/idle-heavy posture, never the simpler right-sized one; picks the highest-ROI-under-budget winner, grafts the superior cost ideas from the runners-up) → **Document** (the ONE writer — `solution-architect` — authors the **cost model** into `architecture/<slug>/capacity-model.md` (the TCO table: per-component build + run + egress + toil over the horizon, the crossover math, the attribution scheme, the assumptions + sensitivity), the cost lines + cost-risk register into `tradeoffs.md`, and the cost-bearing decisions as **MADR ADRs** (`shared/adr-template.md`) each with the cost tradeoff + the rejected cheaper/dearer alternative + a one-way-door flag on reservations/lock-in — preserving conceptual integrity, reusing the existing stack + boring tech, every cost traced to load × price) → **Verify** (the hard gates re-run from the artifacts — order-of-magnitude estimate vs budget present, cost-of-each-decision recorded, egress/cross-AZ accounted per boundary, no unbounded-spend component, crossover + build/buy TCO computed incl. toil, attribution defined, every number traced; the evaluator re-scores craft from the artifacts not the prose) → loop ≤ 10 or pass.

## Assertions & exit gate
- The design carries an **order-of-magnitude `build + run` TCO estimate over the stated horizon, compared to the budget**, with its assumptions (rate card, RPS, data volume, growth) named — or, if no budget is stated, the estimate is still produced and the **missing budget is an open driver question** (never invented).
- **Every major topology decision records its cost-of-each-decision** — a cost line + the cost tradeoff (dollars *and* the attribute bought) in its ADR; cost is a mandatory consequence.
- **Egress + cross-AZ/region data-transfer is a line in the model wherever the topology crosses a boundary** (fallacy #7); no transfer cost silently assumed zero.
- **No "scale infinitely, cost unstated" component** — every elastic/per-request/autoscaling component has a cost ceiling or cost-per-unit-of-load bounded against the capacity model's worst-case load (no financial SPOF).
- The **serverless-vs-provisioned crossover** and the **build-vs-buy / managed-TCO** (including **operational toil**) are **computed from the load + the named rate card**, not asserted; the decision states which side of the crossover this load sits on.
- **Cost is attributable** (per component/env/tenant as the drivers demand) — the FinOps visibility/allocation capability is defined, not a single opaque pool.
- **Cost is weighed against the driver** — no option that starves a required `QAS-AVAIL-*`/`QAS-PERF-*` to shave spend, and no redundancy/headroom gold-plating no driver demanded; the spend is right-sized to the driver.
- Every cost number **traces to a load input × a unit price**; cost-bearing decisions are **ADRs** (context · ≥2 options · decision · consequences incl. the **negative**/cost) with the rejected alternative and a one-way-door flag on any reservation/lock-in.
- **Gate:** hard oracles green (order-of-magnitude estimate vs budget · cost-per-decision recorded · egress accounted · no unbounded-spend component · serverless/provisioned + build/buy crossovers computed incl. toil · attribution defined · every number traced to load × price · ADRs complete) AND (design) rubric mean ≥ 0.8 (cost weighed against the driver not minimized blindly · crossover + TCO computed not hand-waved · right-sized not idle-heavy · FinOps designed in · assumptions + sensitivity stated).

## Output
Write `artifacts/cost-finops-architecture/report.json` per `shared/report-format.md` — `drivers[]` (the `QAS-COST-*` budget + the SLO scenarios the spend trades against), `findings[]` per `shared/finding-schema.md` (`type:"cost"`; `oracle:"hard"` for an unbounded-spend component, an un-modeled egress/cross-AZ path, a serverless-or-build-vs-buy choice with no computed crossover, a cost number with no traced load × price, a missing cost-of-each-decision on a topology ADR, or an un-attributable spend pool under a budget driver; `type:"risk"` for the spend-surprise risks — an autoscale with no max, a load-forecast the estimate is sensitive to, a reserved-capacity one-way-door commitment — with likelihood×impact and an accept/mitigate/monitor disposition; each finding carries `driver_ref`, the cost fact in `evidence` (load × price), the `tradeoff`, and `alternatives_rejected`), the `decisions[]` index (the cost-bearing ADRs + driver_refs), and the `verification` block (estimate-vs-budget present, cost-per-decision coverage, egress accounted, no-unbounded-spend check, crossover/TCO computed, attribution defined, traceability of every number, rubric score). The design artifacts land in **`architecture/<slug>/`**: the TCO model (per-component build + run + egress + operational toil over the horizon, the serverless/provisioned + build/buy crossover math, the FinOps attribution scheme, the assumptions + sensitivity) in **`capacity-model.md`**; the cost lines + the cost-risk register in **`tradeoffs.md`**; the cost-bearing decisions as **`adr/NNNN-*.md`**. Never scatter the cost model or the ADRs elsewhere — the machine record stays in `artifacts/`, the durable artifact in `architecture/<slug>/`.

## Guardrails
Per `shared/guardrails.md`: **propose, price, and document — never write product code, never apply infra, never provision a resource or buy a reservation** (this flow produces the cost model + the cost-bearing decisions; the controller builds the winner, the operator provisions it). Ground every cost number in a **load input from the capacity model × a unit price from the named rate card** — **no invented prices, RPS, or budgets**; if a budget or a unit price is unknown, surface it as an open question and run the sensitivity what-if, never guess a winner on a fabricated number. **Right-size — do not let a cost calculation rationalize accidental complexity** *nor* starve a required SLO to shave spend (per `project-architecture-config.md`'s locked rules on real scale/user count; the cheapest option that misses an RTO/p99 is wrong, and so is gold-plated redundancy no driver demands). Reuse the existing stack + boring tech before pricing in a new datastore/broker/runtime — every new technology is operational toil, a run-cost line. Account for the transport-cost fallacy (#7) — egress/cross-AZ is real money. A separate skeptical evaluator scores craft (never the proposer grading itself), order-swapped to kill position bias, penalizing the idle-heavy/over-engineered posture over the right-sized one. Record the cost tradeoff and the rejected alternatives + what each was better at — a cost recommendation with only a headline number is incomplete. Read-only audit authors nothing; exactly one writer authors the cost model + ADRs for conceptual integrity; never write to project memory — lessons go in the returned report/ADR.
