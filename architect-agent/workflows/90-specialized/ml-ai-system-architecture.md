# ml-ai-system-architecture

**Group:** 90-specialized · **Runs as:** subagent: ../.claude/agents/solution-architect.md · **Mode:** audit + design · **Default model:** opus

## Purpose
Architect an ML / AI / LLM system as a structural decision, not a model-picking exercise. Owns the **train/serve split** (offline training vs online inference, the feature store, the model registry, the data→training→serving skew), the **inference topology** (online/batch/streaming, the cost+latency budget under token economics), and the **LLM/RAG/agentic stack** (retrieval, chunking, vector store, context assembly, orchestration, the eval + guardrail loop). The governing question: *how do we depend on a **probabilistic, drifting, externally-priced** component — one that returns a different answer to the same input, degrades silently as the world moves, and bills per token — without that dependency defeating an availability, latency, cost, safety, or correctness driver?* The model is a dependency with the failure modes of a flaky network call **plus** non-determinism, drift, and a per-call cost — design for all three. This flow is the **structural orchestrator**, not the site of every deep-dive: it decides train/serve, inference topology, and whether RAG/agentic complexity is driver-justified, then **delegates** the specialist batteries to the dedicated flows and gates on their results — it does not re-derive the RAG eval battery, the GenAI risk sweep, the agentic threat model, or the interop-protocol contract inline. (Where the project has a deferred integration seam for a future agentic capability — per `project-architecture-config.md` Evolvability — the live mandate is typically to design that integration boundary / anti-corruption layer the future agent services plug into, not to build the full agent-fleet internals up front.)

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack (the project's stack per that config — e.g. a modular monolith with a single primary datastore and **no broker/queue/vector-store today**, or whatever the config states), the deployment target, the scale/SLO targets, the compliance regime (e.g. regulated personal or financial data — per the config), the team topology (team size, branch/ownership model — per the config), the constraints (stack-lock: a new datastore/broker/vector-DB needs a driver-justified ADR; boring-tech-preferred), and where the architecture description / ADR log / diagrams live (`architecture/<slug>/`).
- The **drivers / utility tree** from `../00-drivers/quality-attribute-scenarios.md` — the prioritized 6-part scenarios this AI subsystem must satisfy, each ending in a response measure. For an AI system the drivers that decide the architecture are usually: an **inference-latency** scenario (`p99 ≤ N ms` / `≤ N s` end-to-end including retrieval + generation), an **inference-cost** scenario (`≤ $X / 1k requests` or a monthly token budget), a **quality/correctness** scenario (a measurable eval threshold — faithfulness/groundedness/answer-relevance ≥ θ, or task success-rate ≥ θ — never "good answers"), a **safety/PII** scenario (`0 unredacted-PII egress to the model provider`, jailbreak/prompt-injection refusal rate ≥ θ), and a **drift/freshness** scenario (model/data drift detected within window W; eval regression alerts before users notice). If the BRD/PRD states no eval threshold, cost ceiling, or latency budget for AI features, **surface it as an open driver question — never invent the number** (`../shared/quality-attribute-rubric.md` grounding rule).
- Target: the AI/ML subsystem under design or review — e.g. a document-processing/extraction path with a stated latency budget, any classification/triage step, and a deferred agentic integration seam if one exists; or a greenfield AI feature. Preconditions: the source-of-truth docs read (the BRD / PRD / understanding log named in `project-architecture-config.md`); the existing data flow understood (where the core business data lives, what would feed retrieval or a feature). A flow with no eval/cost/latency driver downgrades scope and says so — you cannot evaluate a probabilistic component against "feels right."
- **The dedicated AI flows this orchestrator dispatches to and gates on** (read their current oracles before framing scope — this flow does not duplicate their batteries): `rag-evaluation.md` (the RAGAS-style eval battery — Faithfulness, Context Precision, Context Recall, Answer Relevancy, Tool-Call Accuracy — for any retrieval-augmented path); `genai-risk-assessment.md` (the NIST AI 600-1 Generative AI Profile risk sweep — confabulation, information security/integrity, value-chain/provenance, bias/homogenization, dangerous content — for any generative path); `agentic-threat-model.md` (the OWASP-LLM-Top-10 × CSA MAESTRO × MITRE ATLAS sweep — for any tool-using or multi-agent path); and `../30-integration/agent-interop-protocol.md` (the MCP/A2A boundary contract — protocol, auth model, trust scope — for any agent↔tool or agent↔agent boundary, including a deferred agentic-seam placeholder where the project has one). A RAG design in scope with no `rag-evaluation.md` run, or an agentic/tool-using design in scope with no `agentic-threat-model.md` + `agent-interop-protocol.md` run, is an incomplete frame — not a pass.

## Oracle (source of truth)
The prioritized quality-attribute scenarios + the named AI-architecture standards (ML system design, MLOps, the LLM/RAG reference architecture, the agentic patterns, drift-monitoring), each as a measurable response measure or a structural contract. Split:

**hard:** (deterministic — measurable, traceable, or structurally checkable)
- **Inference path meets its latency + cost scenario, proved by a model.** A **capacity/cost model** (`capacity-model.md`) accounts for the *real* inference cost: tokens-in + tokens-out × per-token price (or GPU-seconds for a self-hosted model), retrieval round-trips (vector-search + rerank latency), context-assembly size, and the **fan-out multiplier of agentic loops** (an orchestrator-worker or tool-use loop that calls the model K times multiplies both latency and cost by ~K — model it, don't assume one call). The end-to-end `p99` and `$/1k-requests` are derived from this model and meet the scenario, or it's a hard finding. Cross-ref `../40-cross-cutting/performance-capacity.md` (latency budget) and `cost-finops-architecture.md` (token economics, serverless-vs-provisioned-GPU).
- **The model is treated as a drifting dependency.** The design carries (a) **drift/quality monitoring** — input/feature-distribution drift, prediction drift, and an **online eval signal** (groundedness/faithfulness for RAG, task-success for agents) sampled in production, not just an offline test set; (b) an **eval/guardrail loop** — an offline eval suite (a golden set with measurable metrics) gates promotion, an online guardrail checks each response (schema/grounding/safety) before it reaches the user; (c) a **fallback for low-confidence / failure** — when the model errors, times out, returns low confidence, fails a guardrail, or the provider is down, there is a defined path (deterministic fallback, cached/last-known answer, human-in-the-loop handoff, or fail-fast with a typed error) — **never** an unbounded wait or a silent bad answer. A model dependency with no drift signal, no eval gate, or no fallback is a hard finding (it violates *Failure mode defined* and the *fallacies* — the model call is a remote call: §1 network-unreliable, §2 latency-nonzero — and a non-deterministic, decaying one at that).
- **RAG retrieval quality + the data pipeline are designed (not assumed).** If retrieval is in scope: the **chunking strategy, embedding model, vector store, and retrieval+rerank topology** are decided with a tradeoff; retrieval quality has a **measurable target** (recall@k / context-precision / a faithfulness metric on a labeled set) — "it retrieves relevant docs" is not an oracle; the **index freshness / re-embedding pipeline** is defined (how new source documents enter the index, at what lag); and the **data→serving skew** is addressed (the same preprocessing/embedding at index time and query time). The vector store choice is driver-justified against the stack-lock (reusing the existing primary datastore's native vector-search capability, where the project's stack has one, vs introducing a dedicated vector DB — state the tradeoff per the boring-tech constraint and `project-architecture-config.md`). An undesigned retrieval path or an unmeasured retrieval quality blocks.
- **A RAG design must have a grounded-eval + fallback, dispatched to `rag-evaluation.md`.** Any RAG path in scope is not judged on this flow's prose claim of "it retrieves relevant docs" — it must have run the **RAGAS-style eval battery** (`../90-specialized/rag-evaluation.md`: Faithfulness, Context Precision, Context Recall, Answer Relevancy, plus Tool-Call Accuracy for an agentic RAG path), with retrieval-failure and generation-failure reported **separately** (not a blended score), each metric holding a **project-calibrated threshold traced to a driver** (never invented), and a **CI-gated promotion policy** blocking a model/prompt/index change on a threshold miss. Paired with the eval battery, the design must carry a **bounded fallback for a low-confidence RAG answer** on every decision path (a typed refusal, a human-in-the-loop handoff, or a cached/deterministic answer — never a silent hallucination or an unbounded wait) — the same fallback contract as the drifting-dependency oracle above, but scoped to RAG-specific failure (retrieval-miss, retrieval-noise, generation-hallucination, irrelevant-generation). A RAG design with no `rag-evaluation.md` run, no separated retrieval/generation metrics, no CI gate, or no fallback on a decision path **blocks**. Cross-ref `../90-specialized/rag-evaluation.md` (the eval battery + CI gate) and `../90-specialized/genai-risk-assessment.md` (the Confabulation risk dimension this control serves).
- **PII / safety guardrails exist on the model boundary.** The provider boundary is a **trust boundary** (data leaves the system to an external model): PII sent to the model is **classified and minimized/redacted** per the compliance regime (personal data + financial data, or whatever is stated in `project-architecture-config.md`), the prompt/response path has **input-side guardrails** (prompt-injection / jailbreak defense, the *liberal-in-what-you-accept-but-validate* rule) and **output-side guardrails** (PII-leak, toxicity, schema/grounding validation before the answer is shown or an action is taken), and **secrets** (the provider API key) are externalized via env, never in code/prompt/diagram. Each STRIDE category on the model trust boundary has a control or an accepted risk. Cross-ref `../40-cross-cutting/security-architecture.md` and `../40-cross-cutting/privacy-compliance-architecture.md`. An unredacted-PII egress to an external provider under a regulated regime blocks (`Data lifecycle lawful`). Where the system generates content or influences a consequential decision, this oracle is a **subset** of the fuller NIST AI 600-1 sweep owned by `../90-specialized/genai-risk-assessment.md` (confabulation, information integrity, value-chain/provenance, bias/homogenization, dangerous content) — a generative path in scope with no `genai-risk-assessment.md` run is an incomplete PII/safety check, not a pass.
- **An agentic design must have an agent-boundary threat model + bounded agency, dispatched to `agentic-threat-model.md` and `agent-interop-protocol.md`.** Any design in scope where the model invokes tools or delegates to a peer/sub-agent is not judged on this flow's own STRIDE pass alone — it must have run (a) `../90-specialized/agentic-threat-model.md` (the full OWASP-LLM-Top-10 × CSA MAESTRO 7-layer × MITRE ATLAS sweep, with every agent↔tool and agent↔agent boundary carrying authn, a documented least-privilege trust scope, and an orchestrator-enforced action budget — never model self-governance), and (b) `../30-integration/agent-interop-protocol.md` (every such boundary names MCP or A2A, or a driver-justified alternative, states its auth model and trust scope, and treats every tool-output/task-response re-entering the model's instruction plane as **untrusted input requiring validation** before use). **Bounded agency** is the hard gate: an explicit, orchestrator-enforced ceiling on autonomous action per session (max tool calls, max irreversible-write actions) and a human-in-the-loop gate on any irreversible-write action — an agent with unbounded tool-call budget or no HITL gate on an irreversible action is a hard finding regardless of how narrow its task looks. An agentic/tool-using design with no `agentic-threat-model.md` sweep, no `agent-interop-protocol.md` boundary contract, or no bounded-agency ceiling **blocks**.
- **No un-accepted SPOF on the inference path.** Every component on the **critical path of an availability-SLO scenario** has redundancy/failover, OR the single point of failure is an **explicit, signed-off accepted risk** in the register (`../shared/quality-attribute-rubric.md` → *No un-accepted SPOF*). For an ML system the **external model provider is a SPOF on the inference path** — a single managed API (one vendor, one region, one rate-limit pool) with no secondary provider, no self-hosted fallback model, and no cached/degraded path means any provider outage or throttle takes the AI feature down. A provider, a single vector-store node, or a single inference replica on an availability-SLO path with no failover and no accepted-risk sign-off **blocks**. (Distinct from the drifting-dependency fallback above: that handles a *bad/late answer*; this asserts *redundancy for total loss of the component*.)
- **Consistency model stated for every write path.** Each write path on the AI subsystem **declares its consistency model** (strong / eventual / causal) and its mechanism (`../shared/quality-attribute-rubric.md` → *Consistency stated*; CAP/PACELC per `shared/fallacies-and-tradeoffs.md`). The two ML write paths that matter: (a) the **index write path** — new source documents embedded into the vector store are **eventually consistent** with the source corpus; the design states the **re-embedding lag** and that retrieval may miss just-ingested docs within that window (or chooses a stronger guarantee with its cost); (b) the **model-registry promotion path** — staging→prod promotion is **eventually consistent** across inference replicas (a rolling/canary rollout means replicas serve different model versions during the window); the design states the consistency mechanism (atomic registry pointer flip / version-pinned request routing / drain-and-swap) so two concurrent requests don't silently get two model versions without it being a stated, accepted property. A write path with "it's consistent" and no model **blocks**.

**soft:** (craft — rubric-scored by `architecture-evaluator` after hard oracles are green)
- **AI complexity is driver-justified (no résumé-driven intelligence).** The *simplest* mechanism that meets the driver wins: a deterministic rule or a SQL query over an LLM where one suffices; a single retrieval-augmented call over an agentic loop where multi-step reasoning isn't required; a small/cheap model over a frontier model where the eval passes. An agentic flow where a query would do is accidental complexity — penalize it explicitly (the rubric's *simplicity / right-sizedness*; per `project-architecture-config.md`'s "right-size, don't résumé-drive" rule where the project locks one).
- **The eval + guardrail loop is designed in, not bolted on.** Evals exist as a first-class, version-controlled asset (a golden set + metrics that gate model/prompt changes), not an afterthought; the loop is operable (an eval regression is observable and alertable). The non-determinism is **handled** — the design states how it copes with same-input-different-output (idempotency keys, output schema validation + repair, temperature/seed where it matters, tolerance bands on evals) — not *assumed away*.
- **Conceptual integrity of the model dependency** — one coherent strategy for how the system talks to models (one client/ACL, one guardrail pipeline, one eval harness), not a different ad-hoc integration per feature.
- **Orchestration discipline — this flow decides and delegates, it does not re-derive.** The soft rubric penalizes this flow for duplicating a dedicated battery inline (re-deriving RAGAS metrics instead of dispatching `rag-evaluation.md`, hand-rolling a partial OWASP sweep instead of dispatching `agentic-threat-model.md`) — the craft is in correctly scoping *which* dedicated flow a component needs and consuming its verified output, not in re-authoring the specialist's oracle.

## Standards & techniques
- **ML system design — the train/serve split.** Offline (training: data collection → feature engineering → train → evaluate → register) vs online (serving: feature lookup → inference → monitor). The **feature store** as the contract that kills **training/serving skew** (the same feature computed the same way offline and online); the **model registry** as the versioned, promotable artifact store (staging→prod, rollback). Online vs **batch** vs **streaming** inference chosen by the latency/freshness scenario. (Huyen, *Designing ML Systems*; the MLOps maturity model.)
- **MLOps** — CI/CD/CT (continuous *training*): a model is data + code + config; promotion is gated by an **offline eval** on a held-out/golden set, deployed behind a **shadow / canary / champion-challenger** rollout, monitored for drift, and **rollback-able** via the registry. The model is a versioned dependency with a release process, not a frozen constant.
- **The LLM / RAG reference architecture** — **retrieval** (query → embed → vector-search top-k → optional rerank), **chunking** (size/overlap/structure-aware splitting — a sensitivity point for retrieval quality), the **vector store** (HNSW/IVF index; managed vs embedded; reuse-the-datastore vs dedicated), **context assembly** (top-k + instructions within the context window; claim-check the large payloads — fallacy §3 bandwidth), and **orchestration** (the prompt/flow controller). RAG turns a frozen model into a freshness-current one without retraining — but only if the **index pipeline** keeps the corpus current and the **retrieval quality** is measured; the measurement itself (RAGAS Faithfulness/Context Precision/Context Recall/Answer Relevancy, the golden set, the CI gate) is owned by `../90-specialized/rag-evaluation.md` — this flow decides the topology, that flow proves the quality.
- **Agentic patterns** (Anthropic's *Building effective agents* + the orchestrator-worker harness): **tool-use** (the model calls typed tools — each tool call is a cross-process call needing timeout/retry/fallback), **prompt-chaining / routing**, **orchestrator-worker** (a lead decomposes, workers execute — the K× fan-out multiplier), the **evaluator-optimizer / eval loop**, and **guardrails** as a wrapping layer. The reflex: reach for a **workflow** (fixed, predictable, cheaper) before an **autonomous agent** (flexible, costly, harder to bound) — agency is a cost you justify with a driver, not a default. Where tool-use or multi-agent delegation is chosen, the boundary threat model (OWASP-LLM/MAESTRO/ATLAS) is owned by `../90-specialized/agentic-threat-model.md` and the boundary contract (MCP/A2A, auth model, trust scope) is owned by `../30-integration/agent-interop-protocol.md` — this flow decides *whether* agency is justified, those flows prove the boundary is safe and well-formed.
- **Drift & monitoring** — **data drift** (input distribution moves: covariate shift), **concept drift** (the input→output relationship moves: the world changed), **prediction drift**, and for LLMs **eval drift** (groundedness/faithfulness/task-success decaying in production). Detect with distribution tests (PSI/KL) on features and a **sampled online eval** on outputs; alert before the user notices. Drift is the *expected* state of an ML dependency, not an incident.
- **The probabilistic-dependency + cost/latency reality** — same input → different output (non-determinism); the model **decays** as the world moves while the weights stay frozen (drift); every call **costs** tokens (money) and **takes** variable time (latency tail). These three are the durable laws of an LLM dependency — analogous to the 8 fallacies for a network call (`shared/fallacies-and-tradeoffs.md`): a design that assumes determinism, freshness, or zero cost/latency fails the oracle. Pair with **CAP/PACELC** thinking on the index (the vector store is eventually consistent with the source corpus — state the lag) and the canonical **build↔buy** tradeoff (managed model API vs self-hosted) and **latency↔cost** tradeoff (a bigger model / more retrieval / more agent steps buys quality at latency+cost).
- **NIST AI 600-1 (Generative AI Profile)** — the 12-dimension GenAI risk taxonomy (confabulation, harmful bias/homogenization, information security, information integrity, value-chain/data-provenance, dangerous/CBRN content, and six more) that ISO 25010 and classical availability/latency scenarios don't cover; owned end-to-end by `../90-specialized/genai-risk-assessment.md` — this flow's PII/safety oracle is the model-boundary slice of that larger sweep, dispatched whenever the design generates content or influences a consequential decision about a person.

## Step sequence

Emit findings + a risk register, read-only — author nothing, edit no source.
- **design:** **Frame** (confirm the utility-tree scenarios + the in/out scope of the AI subsystem; where `project-architecture-config.md` scopes a deferred agentic seam rather than full agent internals, confirm the design stays at the *seam*, not the agent internals; names the **sensitivity points** — e.g. *retrieval quality is sensitive to chunk size; cost is sensitive to agent step-count; p99 is sensitive to the rerank hop* — and the **tradeoff points** — *more retrieval/agent-steps improves quality but worsens latency+cost*; incorporates the dispatched flows' artifacts by reference rather than re-authoring them; reuses the established RAG/agentic patterns, preserves one coherent model-dependency strategy) → **Verify** (the cost+latency capacity model is computed and meets the scenario; the eval/guardrail loop + drift signal + fallback are present per the hard oracle; retrieval quality has a measured target and, where RAG is in scope, `rag-evaluation.md`'s hard oracles are green; STRIDE per the model boundary is covered and, where generative/agentic, `genai-risk-assessment.md` / `agentic-threat-model.md` / `agent-interop-protocol.md` hard oracles are green; diagrams compile; ADRs complete; the evaluator re-scores from the artifacts) → loop ≤ ~10 or pass.

## Assertions & exit gate
- Every AI ASR traces to a decision and every decision to a driver; no orphan AI requirement, no un-driver-justified model/agent/vector-store (an agentic loop with no driver demanding multi-step reasoning is accidental complexity → finding).
- The **inference latency + cost** scenarios have a **met response measure**, proved by the capacity/cost model that accounts for tokens, retrieval round-trips, and the **agent-loop fan-out multiplier** — not asserted.
- The model is a **handled drifting dependency**: drift/quality monitoring present, an offline-eval gate on promotion + online guardrails per response, and a **defined fallback** for error/timeout/low-confidence/guardrail-fail/provider-outage (bounded — never an unbounded wait or a silent bad answer).
- Retrieval (where in scope) is **designed and measured**: chunking + embedding + vector store + rerank decided with a tradeoff; a retrieval-quality metric target stated; the **index freshness / re-embedding** pipeline defined; train/serve (index/query) preprocessing skew addressed.
- **A RAG design has a grounded-eval + fallback:** where retrieval is in scope, `rag-evaluation.md`'s RAGAS-style battery has run (Faithfulness, Context Precision, Context Recall, Answer Relevancy — Tool-Call Accuracy for agentic RAG), retrieval/generation failure is separable, thresholds trace to a driver, a CI gate blocks promotion on a miss, and a bounded fallback exists on every decision path — absence of any of these blocks.
- The **model trust boundary** is secured: PII classified + minimized/redacted before egress, input+output guardrails present, provider secret externalized, STRIDE per boundary covered or an accepted risk.
- **An agentic design has an agent-boundary threat model + bounded agency:** where the design invokes tools or delegates to a peer/sub-agent, `agentic-threat-model.md`'s OWASP-LLM × MAESTRO × ATLAS sweep and `agent-interop-protocol.md`'s MCP/A2A boundary contract (protocol, auth model, trust scope, tool-output validation) have run, and an orchestrator-enforced action budget + HITL gate on irreversible actions is present — absence of any of these blocks.
- **No un-accepted SPOF on the inference path:** every component on an availability-SLO critical path (the **external model provider**, the vector store, an inference replica) has redundancy/failover OR is a **signed-off accepted risk** in the register — never a silent single point of failure.
- **Consistency model stated per AI write path:** the **index write path** (new docs → vector store) declares its model + re-embedding lag (eventually consistent unless a stronger guarantee is chosen with its cost), and the **model-registry promotion path** declares its model + mechanism (atomic pointer flip / version-pinned routing / drain-and-swap) so a rollout doesn't silently serve two model versions un-stated. "It's consistent" with no model is a finding.
- **Non-determinism is handled** (output-schema validation/repair, idempotency, tolerance bands) — not assumed away; the **eval suite is a version-controlled, gating asset**, not an afterthought.
- (design) **Diagrams compile and ADRs are complete:** every C4/threat-model diagram is diagrams-as-code that renders without error (`shared/diagramming-standards.md`); every ADR (agentic-vs-workflow, managed-vs-self-hosted, vector-store choice, eval-gate policy) has **context · decision drivers · considered options · decision · consequences (positive AND negative)** per `shared/adr-template.md` — a diagram that doesn't render or an ADR missing options/consequences blocks.
- **Gate:** hard oracles green (inference latency+cost met via the token-aware capacity model · drift-monitoring + eval-gate + guardrails + fallback present · **RAG grounded-eval + fallback via `rag-evaluation.md` where retrieval in scope** · **agent-boundary threat model + bounded agency via `agentic-threat-model.md` + `agent-interop-protocol.md` where agentic/tool-use in scope** · **no un-accepted SPOF on the inference path** · **consistency model stated for the index + model-registry write paths** · retrieval designed + quality-measured + index-freshness defined · PII/safety guardrails + STRIDE on the model boundary · non-determinism handled · **diagrams compile · ADRs complete**) AND (design) rubric mean ≥ 0.8 (AI complexity driver-justified · eval/guardrail loop designed-in · conceptual integrity of the model dependency · dedicated flows dispatched rather than re-derived inline).

## Guardrails
Per `shared/guardrails.md`: **propose & document — never write product code, never train a model, never call a live model provider, never apply infra.** This flow authors the AI architecture *artifacts* (ADRs, C4-as-code, the inference design doc, the token-aware capacity model, the provider-boundary threat model, the eval/guardrail *spec*) into `architecture/<slug>/` — the controller builds the pipeline and the operator provisions the model/vector store. Ground every AI decision in a driver — **no agentic loop, no LLM, no vector DB without a quality/latency/cost/safety scenario that demands it** (the "right-size, don't résumé-drive" rule, per `project-architecture-config.md` where the project locks one); if an eval threshold, cost ceiling, or latency budget is unstated, it's an **open driver question, not a guessed number**. Reuse the established RAG/agentic/MLOps patterns and the existing stack (the project's existing primary datastore before a new vector DB, unless a driver justifies it) before inventing. One writer preserves the conceptual integrity of the model-dependency strategy. Record the tradeoff and the rejected alternatives on every decision (agentic vs workflow, managed vs self-hosted, the vector store) — never just the winner. Treat the provider API key as a read-scoped secret injected via env; never log a prompt, a key, or PII. Where `project-architecture-config.md` scopes a deferred agentic integration seam: design **that integration boundary / anti-corruption layer**, **not** the full agent-fleet internals (per the config's Evolvability + Scope-out sections). **This flow orchestrates, it does not duplicate:** a RAG-shaped component must be routed to `../90-specialized/rag-evaluation.md` for its eval battery and CI gate rather than judged on this flow's prose; a generative or consequential-decision component must be routed to `../90-specialized/genai-risk-assessment.md` for its NIST AI 600-1 sweep; any tool-using or multi-agent component must be routed to `../90-specialized/agentic-threat-model.md` for its OWASP-LLM/MAESTRO/ATLAS sweep and to `../30-integration/agent-interop-protocol.md` for its MCP/A2A boundary contract. Never hand-roll a partial substitute for a dedicated flow's oracle inline — dispatch and gate on the result, or flag the dispatch as missing.
