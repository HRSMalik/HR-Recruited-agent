# migration-modernization

**Group:** 90-specialized  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Plan a system migration or modernization — strangler-fig, branch-by-abstraction, monolith↔microservices direction, the 6 R's, and expand-contract — **incrementally and reversibly**. The design question this flow answers: *given this utility tree and these constraints, what is the safest sequence of steps to move from the current system to the target, keeping the system working at each step and preserving a rollback path at each increment?* The cardinal failure mode it exists to prevent: a big-bang cutover that leaves the system broken mid-migration with no way back.

## Inputs & preconditions

- From `project-architecture-config.md`: the **current system shape** (the "working simple system" — its architecture style, deployment unit, data model, integration points, team topology, known pain points), the **stack lock and constraints** (new datastores/runtimes require a driver-justified ADR; boring-tech-preferred), the **team topology** (Conway's Law: the migration sequence must fit the team's ability to own the transition, not just the target state), the **compliance regime** (PII/financial data in flight across the migration must remain classified and residency-compliant at every step, not only at the end).
- The **drivers / utility tree** from `../00-drivers/quality-attribute-scenarios.md`: the prioritized 6-part quality-attribute scenarios this migration must satisfy. A migration with no drivers — no availability target (what uptime must be preserved during cut-over?), no rollback constraint (what is the RTO if a step fails and must revert?), no evolvability target (what *specific* pain does the current system cause that the target fixes?) — downgrades scope to `status:error`. You cannot evaluate a migration path without knowing what the target state is supposed to fix and what it must not break.
- The **migration inventory**: the concrete assets to move — code modules, schemas, APIs, data, integrations, deployment units. For each asset: its current state, its target state, the invariants it must preserve during transition (data integrity, API compatibility, session continuity), and the blast radius if the step fails. If an asset's invariant is unstated, surface it as an open driver question — do not assume it is safe to break.
- Preconditions: `project-architecture-config.md` read (never hardcode project values); the utility tree exists with at least the top-(H,*) scenarios identified; the source-of-truth docs (BRD / PRD / understanding log named in `project-architecture-config.md`, any prior ADRs) read; the current system is understood well enough to enumerate the migration steps (if not, frame this as a prerequisite, not a skip).

## Oracle (source of truth)

The migration is judged against **three structural facts** — not against how elegant the target state looks — because a migration is only as safe as the weakest step in its sequence:

- **hard** (measurable / traceable / structurally checkable):
  - **Incrementality:** no migration step is a big-bang — each step moves one bounded concern (one module, one schema, one integration seam, one deployment unit) and leaves the rest working. Verified structurally: every step in the migration plan has a defined scope, an entry state, an exit state, and an in-production verify before the next step begins. A step that requires multiple coordinated changes across the entire system in one deployment is a big-bang and blocks.
  - **Reversibility at every step:** every migration step has a named, exercisable rollback path — a feature flag, a routing rule, a schema expansion that adds but does not drop, a parallel-run leg that can be deactivated. A step whose only recovery is "redo everything" is a one-way door and must be recorded as an accepted-risk ADR with explicit sign-off. Verified structurally: each step carries a `rollback:` field in the plan; a step with `rollback: n/a` and no accepted-risk ADR blocks.
  - **System stays live at each step (expand-contract / parallel-run):** API and schema changes follow **expand-contract** (parallel-change): the expand phase adds the new capability alongside the old (new column, new endpoint, new service path) with the old kept live; the migration phase moves consumers; the contract phase removes the old. No step removes the old contract while any consumer still depends on it. Verified structurally: every schema change that removes or renames a field/column follows the expand-before-contract sequence; a `DROP COLUMN` / endpoint removal before consumer migration is a hard finding.
  - **Target is driver-justified, not résumé-driven:** the target architecture (microservices, new datastore, new runtime) is demanded by a named `(H,*)` quality-attribute scenario or a named constraint. Gall's Law (`shared/fallacies-and-tradeoffs.md`): you cannot design a complex working system from scratch — a complex system that works evolved from a simple one. The monolith-first + decompose-when-justified stance means: do **not** migrate a working modular monolith to microservices unless a concrete driver (an independent-deploy/team-scale/fault-isolation/scale scenario in the `(H,*)` band) is present and cited. Absent that driver, the migration plan is accidental complexity and blocks the `driver-traceability` hard oracle. The accidental-complexity check from `../10-styles-decomposition/architecture-style-selection.md` applies here: "you must be this tall to decompose."
  - **Rollback RTO matches the availability driver:** if the system's availability driver specifies an RTO (e.g. QAS-AVAIL-01: RTO ≤ 5 min on service failover), the rollback path for every migration step must be exercisable within that RTO. A rollback that takes 2 hours when the availability driver is RTO ≤ 5 min fails the `response-measure-met` hard oracle for the availability scenario during migration.
  - **Data integrity across the transition:** every data migration step (schema change, data copy, dual-write period) preserves the business invariants stated in the utility tree (e.g. QAS-DATA-*: no financial amount lost; no business-entity identifier duplicated). Dual-write periods — where both the old and new path write — are governed by the consistency model from `../20-data/consistency-transactions.md`: no silent dual-write, idempotency keys on every replayed write, a reconciliation gate before old path is retired. A data step with no stated invariant-preservation mechanism blocks.
  - **Fallacies honored at every new service boundary:** any step that introduces a new process boundary (a new service, a new data store, a new async channel) must account for the 8 fallacies of distributed computing (`shared/fallacies-and-tradeoffs.md`) from the moment that boundary exists — not from the moment the migration is complete. A step that wires up a synchronous service call with no timeout/retry/fallback "until we finish the migration" is a hard finding: the fallacies apply the instant the wire is cut.
  - **ADR complete per significant decision:** every significant migration decision — the chosen 6-R disposition per asset, the strangler-fig vs branch-by-abstraction choice per seam, the dual-write window and its consistency model, the expand-contract sequence per schema change — is recorded as a complete MADR ADR (`shared/adr-template.md`) with context · ≥2 options · decision · consequences (positive AND negative) · driver refs tracing to real `QAS-*` leaves. A decision with no recorded alternatives or no negative consequences blocks.
  - **Diagram compiles:** every migration sequence diagram (the "before → during → after" topology at each step) is diagrams-as-code (Mermaid or PlantUML+C4-PlantUML per `project-architecture-config.md`) and renders without error per `shared/diagramming-standards.md`.
  - **No un-accepted SPOF:** every component on the critical path of an availability-SLO scenario in the migration plan has redundancy/failover, OR the single point of failure is an explicit signed-off accepted risk in the risk register. A migration step that routes all traffic through a new singleton (a single strangler-fig façade instance, a single dual-write coordinator) with no redundancy and no accepted-risk ADR is a silent SPOF and blocks. Verified per `shared/quality-attribute-rubric.md`.
  - **Trust boundary secured:** every new trust boundary introduced by the migration — a new service-to-service call, a new async channel, a new data store crossed by a migrated module — has authn + authz in place, secrets externalized (not in config or diagram literals), and each STRIDE category has a stated control or an accepted risk. A new process boundary wired up mid-migration "until we finish" with no trust controls is a hard finding. Verified per `shared/quality-attribute-rubric.md` and `../40-cross-cutting/security-architecture.md`.
  - **Data lifecycle lawful:** where a compliance regime applies (GDPR/HIPAA/PCI/SOC2), PII that crosses a new service boundary or lands in a new datastore during the migration is classified at that moment — not only at the end of the migration. Residency must be honored at every step; retention and erasure paths must exist before the new store is written to. An unclassified-PII store under a regulated regime, or a migration step that moves PII to a new region without residency verification, blocks. Verified per `shared/quality-attribute-rubric.md`.

- **soft** (craft, rubric-scored only after hard oracles are green):
  - **Gall's Law respect:** the migration plan evolves the system from its current *working* simple state — it does not discard the working system and start over. Each step makes one improvement and keeps everything else working. A plan that rewrites the monolith while the monolith serves production is a big-bang in disguise.
  - **Right-sized target state:** the target architecture is the *simplest* state that satisfies the stated drivers — not the most impressive end-state the team could imagine. If the driver is "we need independent deploy of the payments module," the target is "extract that one module into a separately-deployable unit," not "rewrite every module into a microservice."
  - **Honest about risky steps:** the plan names its **pivot steps** — the irreversible moments (data format change with no legacy reader, a breaking API change, a dependency cutover that cannot be undone) — and places them late, after earlier reversible steps have proven the concept. A plan that front-loads the risky steps without justification is a craft failure.
  - **Tradeoff honesty:** the plan names what the migration costs on the other attributes during the transition period (e.g. "during the dual-write window, write latency on the core entity rises ~15% due to the second write path; consistency is eventual until the reconciliation gate"). A plan that presents only the target-state benefits with no transition-period costs is incomplete.

## Standards & techniques

- **The 6 R's of cloud/system migration** (AWS, Azure, and widely adopted in migration practice) — the disposition taxonomy for every asset in the migration inventory. Apply per-asset before sequencing:
  - **Rehost (lift-and-shift)** — move the asset to the target environment with no code change. Lowest risk, lowest value; appropriate when the target environment change (e.g. containerizing a service, moving to managed DB) is the sole driver and the code is not the problem.
  - **Replatform (lift-and-optimize)** — minimal changes to take advantage of the target platform (e.g. moving from a self-hosted Postgres to a managed cloud DB, swapping an HTTP poller for a managed queue trigger) without changing the core architecture. Higher value than rehost; still relatively safe.
  - **Repurchase (drop-and-shop)** — replace the asset with a commercial SaaS or off-the-shelf product (e.g. replace a hand-rolled auth layer with Auth0/Okta). Fastest time-to-value if the buy fits; introduces a vendor dependency and cedes control — record the tradeoff in an ADR.
  - **Refactor / re-architect** — redesign the asset's architecture to satisfy a quality-attribute driver that the current design cannot meet (e.g. extract a module to enable independent deploy; introduce the outbox pattern to cure a dual-write). The highest-value and highest-risk disposition; every refactor step must be incremental and reversible (strangler-fig or branch-by-abstraction, not a big-bang rewrite). Demands the "you must be this tall" bar: a concrete driver that the current shape fails.
  - **Retire** — decommission the asset (eliminate dead code, remove a deprecated integration, shut down an unused service). Clean and simple; requires confirming no remaining consumer before removal (verified by the static import graph and the integration inventory).
  - **Retain** — leave the asset in place; the cost or risk of migration exceeds the benefit. A deliberate decision, not a default. Record the retained asset and its known limitations in the risk register so it is not silently forgotten.

- **The strangler-fig pattern** (Fowler, 2004; cloud-design-pattern catalog `../80-patterns/cloud-design-patterns.md`) — route a subset of requests to a new implementation while keeping the old path live, then incrementally expand the new path's share until the old path can be retired. The routing rule (a façade, a feature flag, an HTTP header, a Content-Based Router EIP — `../30-integration/integration-patterns.md`) must be **togglable** so each increment is reversible. The strangler-fig is the canonical pattern for incremental migration at the HTTP/API layer; it applies to: replacing a legacy endpoint, migrating from a monolith to a service, replacing an external integration. Key constraint: the old path must stay functionally equivalent while the new path is proven — not "good enough for now" but actually equivalent on the invariants the utility tree specifies.

- **Branch-by-abstraction** (Fowler, *Refactoring*, 2nd ed., 2018; used in conjunction with the strangler-fig for in-process seams) — introduce an abstraction layer (an interface, a port, an ACL from `../30-integration/integration-patterns.md`) over the code you want to replace; point the callers at the abstraction; implement the new behavior behind the abstraction; reroute through the new implementation; remove the abstraction once the old is fully retired. Branch-by-abstraction is the in-process analog of the strangler-fig: it is used when the seam is a module/function boundary, not an HTTP boundary. The two are frequently paired: branch-by-abstraction creates the in-process seam; the strangler-fig routes traffic at the HTTP/API edge.

- **Expand-contract (parallel-change)** (Fowler, *Refactoring Databases*; also called "parallel-change") — the safe sequence for any schema or API change with live consumers: (1) **Expand** — add the new column/field/endpoint alongside the old; both old and new are live; consumers are not yet migrated; (2) **Migrate** — move consumers one at a time to the new shape; both paths still live; (3) **Contract** — remove the old shape after all consumers are migrated and verified. This is the only safe sequence for a backward-incompatible change in a live system; any variant that removes the old before all consumers are migrated is a hard finding. Applies to: document/collection or table/column changes on the project's datastore, API endpoint versioning on the project's service framework (per `project-architecture-config.md`), any shared schema in a multi-consumer context. Cross-reference `../20-data/consistency-transactions.md` for the consistency model of the dual-write window during the Migrate phase.

- **Monolith-first + decompose-when-justified** (Newman, *Monolith to Microservices*; Fowler, "MonolithFirst") — start with a well-factored modular monolith and extract a service boundary only when a concrete driver (independent-deploy cadence, a real Conway-law team split, a scale or fault-isolation scenario the monolith cannot meet) forces it. The modular monolith buys clean module boundaries and low operational cost; it sells the option to extract later at the cost of a refactor. Do not extract prematurely — each service boundary is an operational surface area, a network hop (fallacy #2: latency is not zero), and a consistency boundary that converts a local transaction into a cross-service write (requiring saga or outbox — `../20-data/consistency-transactions.md`). Cross-reference `../10-styles-decomposition/architecture-style-selection.md` for the full "you must be this tall to use microservices" accidental-complexity gate.

- **Gall's Law** (`shared/fallacies-and-tradeoffs.md`) — a complex system that works evolved from a simple system that worked. You cannot design a complex system correct from scratch — every big-bang rewrite that discards the working system violates this and historically fails or regresses. The migration plan's job is to be the bridge between the current working simple system and the target complex system, one proven step at a time.

- **Feature flags** (progressive delivery) — the operational primitive that makes each strangler-fig increment and each branch-by-abstraction cutover reversible in production. A feature flag controls which implementation path is live for which fraction of traffic; it is togglable without a deployment and can be evaluated at the routing layer. Feature flags are the rollback mechanism for migration steps where re-deployment is too slow for the availability driver's RTO.

- **ATAM sensitivity/tradeoff analysis** (`shared/evaluation-method.md`) — the key migration tradeoff points: *Strangler-fig ↔ big-bang* (strangler is safe and reversible but slow; big-bang is fast but a one-way door); *Expand-contract ↔ versioned-endpoint* (expand-contract keeps one schema, simpler to reason about, but requires two writes during the Migrate phase; versioned endpoints are parallel contracts, explicit, but add routing complexity); *Refactor ↔ retain* (refactor fixes the pain but takes time and introduces transition risk; retain avoids transition risk but leaves the pain). Name the point chosen on each tradeoff curve per asset, not just the winner.

- **AKF scale cube** as the right-sizing lens (`../40-cross-cutting/scalability-architecture.md`) — confirm whether the scale driver (if any) actually demands a Y-axis split (service decomposition). Most scale needs up to moderate load are met by X-axis (horizontal stateless clones of the monolith); Y-axis decomposition is warranted when a module's scale profile is fundamentally different from the rest (e.g. a document-processing pipeline that needs 10× more CPU than the core CRUD API). Absent a confirmed Y-axis driver, a service split for "scale" is accidental complexity.

- **MADR decision records** (`shared/adr-template.md`) — every migration decision (the 6-R disposition per asset, the strangler-fig routing strategy, the expand-contract sequence, the dual-write window consistency model, the feature-flag governance policy) is an ADR with ≥2 genuine options, stated consequences (positive AND negative), and follow-up fitness functions that confirm the step completed and the rollback path is valid.

## Step sequence

- **audit:** Frame the migration drivers from `project-architecture-config.md` + the utility tree → enumerate the migration inventory (every asset: current state, target disposition per the 6 R's, known invariants) → for each asset: (a) check whether the current design already has an accidental big-bang embedded (a planned cutover with no parallel-run, a schema DROP without a prior expand phase, a step with no stated rollback); (b) check whether each target-state decision is driver-justified (does the refactor disposition have a named `(H,*)` driver demanding it?); (c) check whether the expand-contract sequence is in place for every schema/API change; (d) check whether the dual-write windows have a stated consistency model and idempotency mechanism; (e) check whether the 8 fallacies are accounted for at each new process boundary introduced mid-migration; (f) check whether rollback RTOs are within the availability driver's RTO target → emit findings per `shared/finding-schema.md` (`type:"evolvability"` or `type:"risk"`; `oracle:"hard"` for a big-bang step, a missing rollback, a remove-before-migrate schema change, an un-driven refactor, a dual-write with no consistency model, a new process boundary with unaccounted fallacies; `oracle:"soft"` for Gall's-law-violating sequencing, an over-ambitious target state, front-loaded risky steps) — read-only, authors nothing, edits no source.

- **design:** **Frame** (extract/confirm the migration drivers + the utility tree `(H,*)` scenarios + the migration inventory + the invariants per asset; confirm the team topology and the transition-period availability/rollback drivers; read-only) → **Explore** (≥2 divergent migration paths as concrete step-by-step plans — e.g. *Path A: strangler-fig at the HTTP edge with a feature flag, monolith kept fully live until each module is proven in the new path*; *Path B: branch-by-abstraction at the module level, in-process seam introduced first, then split to a process boundary once the abstraction is proven* — each path is a real spec: the step sequence, the expand-contract phases per schema/API change, the dual-write window and its consistency model, the rollback mechanism per step, the team-ownership hand-off point; not the same plan relabeled; read-only proposers) → **Evaluate** (one skeptical ATAM evaluator scores each path against the utility tree + hard oracles: incrementality check, rollback-per-step, expand-contract present for schema changes, driver-justified target, consistency model stated for dual-writes, fallacies honored at new boundaries, ADR-completeness; scores soft oracles: Gall's-law respect, right-sized target, honest sequencing of risky steps, tradeoff honesty about transition costs; order-swapped to kill position bias; penalizes accidental complexity and over-ambitious targets; picks winner + grafts superior rollback/sequencing ideas from runners-up) → **Document** (the ONE writer, `solution-architect`, authors: the migration plan document with per-asset 6-R disposition table, the step sequence with entry/exit states and rollback fields, the expand-contract sequence diagrams-as-code per schema/API change, C4 deployment diagrams for each migration phase ("before," "during strangler-fig," "after"), one MADR ADR per significant migration decision, the risk register for accepted one-way-door steps, fitness-function specifications that confirm each step's exit gate) → **Verify** (fitness functions: the no-big-bang structural check — each step touches ≤1 bounded concern; the rollback-per-step check — no step has `rollback: n/a` without an accepted-risk ADR; the expand-before-contract check — no schema/API removal without a prior expand phase and a consumer-migration gate; the driver-justified-target check — each refactor step traces to a `QAS-*`; diagrams compile; traceability: every migration driver traces to a plan step; evaluator re-scores from the authored artifacts, not the prose) → loop ≤8 or pass.

## Assertions & exit gate

- Every asset in the migration inventory has a **6-R disposition** (rehost / replatform / repurchase / refactor / retire / retain) that traces to a stated driver or constraint; a "refactor" disposition without a named `(H,*)` driver blocks the accidental-complexity gate.
- Every migration step is **incremental** — it moves ≤1 bounded concern and leaves the rest of the system working. No step requires simultaneous coordinated changes across the full system.
- Every step has a **named, exercisable rollback path** (`rollback: <mechanism>` in the plan); a step with no rollback and no accepted-risk ADR blocks.
- Every schema and API change follows the **expand-contract (parallel-change) sequence**: expand → migrate consumers → contract. No removal of an old shape before all consumers are migrated and verified. The consistency model and idempotency mechanism for the dual-write Migrate phase are stated and trace to `../20-data/consistency-transactions.md`.
- Every new process boundary introduced mid-migration has **failure handling from day one** — bounded timeout, retry policy, fallback/fail-fast per `shared/fallacies-and-tradeoffs.md` fallacies #1 and #2; no "we'll add the timeout once the migration is done."
- **Rollback RTOs** for every step are within the availability driver's stated RTO target (or the deviation is an accepted-risk ADR); the rollback mechanism is togglable within that window (feature flag, routing-rule change, or re-deploy within the RTO bound).
- Every significant migration decision is a **complete MADR ADR** with context · ≥2 options · decision · positive AND negative consequences · driver refs to real `QAS-*` leaves; no decision with only positive consequences.
- All migration-phase diagrams (C4 deployment view: before / during / after per step or phase) are **diagrams-as-code** and compile without error.
- The **target architecture is driver-justified** — the "you must be this tall" bar from `../10-styles-decomposition/architecture-style-selection.md` is met with cited `(H,*)` scenarios before any service decomposition is included in the plan.
- **No un-accepted SPOF** on the critical path at any migration phase — every new singleton or choke point introduced by a migration step (strangler-fig façade, dual-write coordinator) has redundancy/failover or is recorded as an accepted risk in the register.
- **Trust boundary secured** — every new trust boundary created mid-migration (new service-to-service call, new async channel, new datastore) has authn + authz and STRIDE controls in place from the moment the boundary exists; externalized secrets; no "we'll secure it when migration is done."
- **Data lifecycle lawful** — PII that enters a new service boundary or new datastore during the migration is classified and residency-compliant at that migration step; retention and erasure paths exist before the new store is written to; no unclassified-PII store under a regulated regime at any phase.

**Gate:** hard oracles green (no big-bang steps · rollback path per step · expand-before-contract on every schema/API change · driver-justified target · dual-write windows consistency-stated · fallacies honored at new boundaries · ADRs complete with negative consequences · diagrams compile · no un-accepted SPOF · trust boundary secured · data lifecycle lawful) AND (design) rubric mean ≥ 0.8 across Gall's-law-respect, right-sizedness, risky-steps-honest, and tradeoff-honesty.

## Output

Write `artifacts/migration-modernization/report.json` per `shared/report-format.md` — findings/risks per `shared/finding-schema.md` (`type:"evolvability"` for big-bang, missing rollback, un-driven target, expand-before-contract violation; `type:"risk"` for accepted one-way-door steps; `type:"data"` for dual-write window findings; `oracle:"hard"` for all hard-gate failures; `oracle:"soft"` for Gall's-law, right-sizedness, risky-step-sequencing craft gaps), plus the verification block (fitness-function results — no-big-bang check, rollback-per-step check, expand-before-contract check, driver-justified-target check — traceability coverage, hard-oracle results, rubric score) in design mode.

Design artifacts land in `architecture/<slug>/` per `project-architecture-config.md` (where `<slug>` is the migration's slug, e.g. `core-platform-migration`):

```

├── design-doc.md                       # arc42 migration plan: current state, target state, step sequence, rollback matrix

├── adr/NNNN-expand-contract-*.md       # MADR: expand-contract sequence per breaking schema/API change
├── adr/NNNN-dual-write-window-*.md     # MADR: consistency model + idempotency during the parallel-run window
├── diagrams/migration-before.md        # C4 deployment view: current state (diagrams-as-code)
├── diagrams/migration-during-*.md      # C4 deployment view per phase: strangler-fig façade, dual-write window, etc.
├── diagrams/migration-after.md         # C4 deployment view: target state (diagrams-as-code)
├── diagrams/expand-contract-*.md       # Sequence diagram per schema/API change (expand → migrate → contract phases)
```

Audit mode authors nothing — findings + risk register only, returned to the controller to decide which to act on. Open driver questions (unstated rollback RTO, unknown consumer inventory for a schema change, undecided team-ownership hand-off for a new service boundary) are listed explicitly for the controller, never silently assumed or resolved by the agent.

## Guardrails

Per `shared/guardrails.md`: **propose and document the migration plan — never write product code, apply a migration, run a schema change, or touch a live system.** This flow authors the migration plan, the step sequence, the expand-contract diagrams, and the ADRs; the controller implements the strangler-fig routing rules and the branch-by-abstraction seams; the operator applies schema changes and the infra topology changes per the authored plan.

**Ground every migration step in a driver.** A refactor step with no named `(H,*)` scenario it serves is accidental complexity — remove it from the plan, don't ship it. Gall's Law: the working current system is the asset; the migration plan's job is to evolve it, not discard it. A plan that rewrites the system rather than incrementally improving it fails the grounding rule even if the target state is well-designed.

**Reuse the named patterns** — strangler-fig, branch-by-abstraction, expand-contract — before inventing bespoke migration mechanisms. Each named pattern carries known contracts and known failure modes; a bespoke "we'll just cut over on the weekend" is not a named pattern, it is a big-bang.

**Never migrate microservices without the "you must be this tall" bar** from `../10-styles-decomposition/architecture-style-selection.md`. The monolith-first stance applies equally to migrations: if the current modular monolith is working and no `(H,*)` driver demands a service split, the right migration target is a *better-factored* modular monolith, not a microservice fleet. Introducing a service boundary is introducing: a network hop (latency non-zero, fallacy #2), a consistency boundary (every cross-service write becomes a saga or outbox — `../20-data/consistency-transactions.md`), and an operational unit the team must own. Name all three costs in the ADR.

One writer (`solution-architect`) authors all artifacts in sequence, preserving conceptual integrity. Proposers and the evaluator are read-only. The evaluator scores against the drivers and the structural facts, not the writer's prose; it penalizes over-ambitious targets, front-loaded risky steps, and accidental complexity in the migration plan itself.
