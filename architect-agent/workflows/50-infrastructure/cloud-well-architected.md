# cloud-well-architected

**Group:** 50-infrastructure   ·   **Runs as:** subagent: ../.claude/agents/infrastructure-architect.md   ·   **Mode:** audit + design   ·   **Default model:** opus

## Purpose
Evaluate (and where greenfield, shape) the deployment architecture against the cloud **Well-Architected** pillars — operational excellence, security, reliability, performance efficiency, cost optimization, sustainability — and decide the **cloud posture**: cloud-native (managed services / serverless / elasticity) vs lift-and-shift vs re-platform vs hybrid. Answers the *decision* "for each pillar, where does this design fall short of its drivers, and is the managed-vs-self / lock-in-vs-leverage tradeoff on each component the right one?" — not "is it a good cloud setup." The output is a per-pillar findings set plus driver-justified posture ADRs, never a reflexive "go serverless / go k8s."

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target (per `project-architecture-config.md` — e.g. a same-origin SPA + API service behind a reverse proxy, a single primary datastore; deploy-ready container images; **prod deploy may be a separate ticket** — registry images, real secrets, env-driven origins, managed DB are *designed here, not assumed built*), the cloud target (if none stated, that is itself an open driver — surface it, don't pick a provider reflexively), the scale/SLO targets, the compliance regime (per `project-architecture-config.md`), the team topology (per `project-architecture-config.md`, Conway), the constraints (stack lock, boring-tech-preferred), and where `architecture/<slug>/` lives.
- The architecture **drivers** — the prioritized utility tree from `../00-drivers/quality-attribute-scenarios.md`: each pillar must be assessed against the QAS it serves (e.g. cost optimization against a `QAS-COST-*` budget; reliability against a `QAS-AVAIL-*` RTO/RPO; performance efficiency against a `QAS-PERF-*` latency measure). **A pillar with no driver is assessed at "are we over/under-provisioning vs the stated scale?" only** — you cannot score a pillar against an attribute the system isn't optimizing for; say so and downgrade scope rather than invent a target.
- The deployment topology (or the proposed one) under review — the C4 deployment view from `deployment-topology.md`, the IaC design from `infrastructure-as-code.md` if authored, the current `architecture/<slug>/capacity-model.md` and `tradeoffs.md`.
- Preconditions: the source-of-truth docs (BRD / PRD / understanding log, per `project-architecture-config.md`) NFRs read (never invent a budget, an RTO, or an availability number); the **shared-responsibility model** of the target platform identified before any security/reliability finding is written (what the provider secures vs what we secure shifts with every managed-vs-self choice).

## Oracle (source of truth)
The six Well-Architected pillars (AWS Well-Architected Framework / Azure Well-Architected Framework), each made concrete as the **driver it must satisfy** — a quality-attribute scenario's response measure, a stated budget, a compliance obligation — never a vendor "best practice" in the abstract. The grounding rule holds: *no number + no traced driver = a preference, not a finding* (`shared/quality-attribute-rubric.md`).
- **hard:**
  - **Per-pillar coverage** — the design is assessed against **all six** pillars; each pillar produces either a finding (a gap vs its driver) or an explicit recorded non-risk ("operational excellence adequate: health/readiness + structured logs + runbook present, traced to `QAS-OPS-*`"). A pillar silently skipped blocks — you cannot claim "Well-Architected" with two pillars unexamined.
  - **Managed-vs-self carries a TCO + lock-in tradeoff** — every "use managed X" vs "self-run X" choice (a managed datastore vs a self-hosted one, a managed load balancer/reverse-proxy vs self-managed, serverless vs container vs VM) is an ADR with a **TCO comparison** (build + run + operational toil over the relevant horizon, per `../60-evaluation/tradeoff-analysis-cbam.md`) AND a **named lock-in cost** (what reversing it costs — the one-way-door flag). A managed choice with no TCO and no lock-in note blocks.
  - **Cloud-native vs lift-shift is driver-justified** — the posture decision (cloud-native / lift-and-shift / re-platform / hybrid) names the *driver* that demands it. "Re-platform to serverless" with no elasticity/cost/ops driver is reflexive résumé-driving and blocks; "lift-and-shift the container as-is because no driver demands re-platforming and it's the cheapest reversible path" is a valid, recorded decision.
  - **Shared-responsibility boundary explicit** — the threat/reliability model states, per component, **who is responsible for what** (provider vs us) under the chosen managed-vs-self split; a security or reliability claim that silently assumes the provider covers something it doesn't (or that we cover something the provider already does, paying twice) blocks. Cross-checks `../40-cross-cutting/security-architecture.md` (STRIDE per trust boundary) and `../40-cross-cutting/reliability-slo.md`.
  - **Cost bounded** — the design carries an order-of-magnitude monthly run-cost estimate against the stated budget/constraint (`cost_bounded` hard oracle); no component is "elastic, cost unstated." Egress, cross-AZ/region transfer (fallacy #7 — transport cost is not zero), and per-request serverless pricing are in the model where they apply. Defers the deep model to `../90-specialized/cost-finops-architecture.md` but must not be silent.
  - **Reliability response-measure met** — every `QAS-AVAIL-*`/`QAS-PERF-*` the topology is on the critical path of has a met measure proven by the capacity model or a managed service's published SLA; no un-accepted SPOF on an availability-SLO path introduced by an infra choice (e.g. a single self-hosted datastore primary under an RTO driver). Honors the 8 fallacies where the topology spans AZs/regions/networks.
- **soft (rubric-scored, after hard green):**
  - **Right-sized cloud posture** — managed/boring where it *lowers* TCO and operational toil for a small team (a typical default: a managed datastore over self-running a replica set the team can't operate at 3am); lock-in accepted *only* where the leverage (lower TCO, less toil, a real SLA) outweighs the reversal cost — not for fashion. The simplest posture that meets every driver scores higher; a multi-region active-active or a k8s control plane with no driver demanding it is penalized as accidental complexity (`shared/quality-attribute-rubric.md` — penalize over-engineering).
  - **Sustainability + cost are first-class, not afterthoughts** — right-sizing, scale-to-zero / scale-down off-peak (an internal enterprise tool has predictable low overnight load), region/instance selection, and the carbon/efficiency view are reasoned about up front, not bolted on. An always-on max-size cluster for a tool used by a handful of internal seniors fails this even if it "works."
  - **Operational excellence designed in** — deployability (the deploy-ready Dockerfiles, env-driven config, immutable images), observability hooks, and a runbook-able failure story exist as part of the posture, not as a later phase.

## Standards & techniques
- **The six Well-Architected pillars** (AWS WAF / Azure WAF, same six): **Operational Excellence** (run + evolve: IaC, observability, runbooks, small reversible changes), **Security** (identity, detective controls, data protection, the shared-responsibility model), **Reliability** (recover from failure, scale horizontally, stop guessing capacity — RTO/RPO, redundancy, no-SPOF), **Performance Efficiency** (use the right resource type/size, serverless where it fits, review as services evolve), **Cost Optimization** (right-size, match supply to demand, stop paying for what you don't use, measure attribution), **Sustainability** (maximize utilization, minimize provisioned-but-idle, choose efficient regions/services). Each pillar has its own question set (the WAF "pillar questions") — walk them, but score against *our* drivers, not the questionnaire in the abstract.
- **Cloud posture taxonomy:** **cloud-native** (managed services, serverless/FaaS, elasticity, design-for-scale) vs **lift-and-shift / rehost** (move the container/VM unchanged) vs **re-platform / replatform** (minimal changes to exploit managed services — e.g. swap a self-hosted datastore for its managed equivalent) vs **hybrid** (some on-prem/self-managed, some cloud). Maps to the 6 R's in `../90-specialized/migration-modernization.md` (Rehost, Replatform, Repurchase, Refactor, Retain, Retire) — pick the R per component, driver-justified, cheapest reversible path absent a driver to refactor.
- **The managed-vs-self (build-vs-buy) tradeoff** (`shared/fallacies-and-tradeoffs.md` — *Build ↔ Buy*): managed = faster, lower operational TCO, a real SLA, less surface area — at the cost of control + a dependency + lock-in. **Default to managed/boring unless a driver demands bespoke.** Quantify with CBAM (`../60-evaluation/tradeoff-analysis-cbam.md`): TCO = build + run (infra + operational toil + opportunity) over the horizon; managed often wins for a small team even at a higher sticker price because operational toil is the dominant cost.
- **Vendor lock-in vs leverage** — lock-in is a one-way-door cost (the reversal price: data egress, re-architecture, retraining); leverage is the value bought (lower TCO, less toil, the SLA, time-to-market). Accept lock-in *deliberately* where leverage > reversal cost; use portable seams (an anti-corruption layer, a storage abstraction, OCI containers over a proprietary runtime) where the reversal risk is high and the leverage thin. Name the point chosen on the curve per component — never "avoid lock-in at all costs" (that gold-plates) nor "lock in everywhere" (that traps).
- **The shared-responsibility model** — the provider secures *of* the cloud (hardware, hypervisor, managed-service internals, patching the managed engine); we secure *in* the cloud (our data, IAM, network config, app code, the OS on anything self-managed). The boundary **moves with every managed-vs-self choice** (managed DB → provider patches the engine; self-hosted on a VM → we do). The threat model and the reliability model must both be drawn *against this boundary* — a control assumed on the wrong side is a gap (`../40-cross-cutting/security-architecture.md`).
- **AKF scale cube / elasticity** (for performance-efficiency + cost): scale-out (X), functional decomposition (Y), data partition (Z) — but **right-sized**: an internal enterprise tool sizes to its real concurrency, not to a consumer peak. Match supply to demand (scheduled scale-down off-peak) rather than always-on max.
- **CBAM / one-way vs two-way doors** (`shared/evaluation-method.md`, `../60-evaluation/tradeoff-analysis-cbam.md`) — the datastore, the cloud provider, the serverless-vs-container runtime, and any proprietary managed service are typically **one-way doors**; flag and analyze them deeply. Reversible config (instance size, autoscale policy) is a two-way door — decide cheaply, don't over-analyze.

## Step sequence
- **audit:** Frame the drivers (pull the utility tree + the stated budget/RTO/availability; identify the cloud target + shared-responsibility boundary; flag any pillar with no driver as reduced-scope) → walk the existing/proposed topology against **each of the six pillars** in turn, scoring each against the QAS it serves and the hard structural checks (per-pillar coverage, managed-vs-self TCO+lock-in on each component, posture driver-justification, shared-responsibility boundary, bounded cost, no infra-introduced SPOF, fallacies where the topology spans the network) → emit findings + a risk register (read-only; authors no artifact, edits no source, applies no infra). Each finding names the pillar, the driver, the structural/cost fact, and the ADR-shaped recommendation.
- **design:** **Frame** (extract/confirm the drivers + the per-pillar targets + the cloud target + the shared-responsibility split, read-only) → **Explore** (≥2 *divergent* postures as concrete specs — e.g. *lift-and-shift the existing container behind a managed load balancer with a self-hosted datastore on a VM* vs *re-platform to a managed datastore + container service with scheduled scale-down* vs *serverless where the access pattern fits* — each a real topology + per-pillar profile + TCO + lock-in note, not the same diagram recoloured; read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each posture against the utility tree + the six pillars via CBAM — TCO vs benefit per option, the sensitivity points, the **tradeoff points** where a pillar choice helps one attribute and hurts another, e.g. *serverless improves cost+sustainability off-peak but worsens cold-start latency vs `QAS-PERF-*`*; order-swapped to kill position bias; penalizes the more elaborate posture, not the simpler one; picks a winner, grafts the superior ideas from the runners-up) → **Document** (the ONE writer — `solution-architect` — authors the posture ADRs + the per-pillar assessment into the design-doc + the deployment-view note + the cost lines into `capacity-model.md`/`tradeoffs.md`, preserving conceptual integrity, reusing the existing stack and boring tech, every decision traced to a driver) → **Verify** (the hard gates re-run from the artifacts — per-pillar coverage complete, every managed-vs-self ADR carries TCO + lock-in, the posture is driver-justified, the shared-responsibility boundary is in the threat/reliability model, cost is bounded, no infra-introduced un-accepted SPOF, diagrams compile, ADRs complete; the evaluator re-scores from the artifacts, not the prose) → loop ≤ 10 or pass.

## Assertions & exit gate
- **All six pillars assessed** — each of operational excellence, security, reliability, performance efficiency, cost optimization, sustainability produces a finding or a recorded non-risk traced to its driver; no pillar silently skipped.
- **Every managed-vs-self choice is an ADR** with a TCO comparison AND a named lock-in / reversal cost, with the one-way-door flag set where it applies.
- **The cloud-native vs lift-shift vs re-platform vs hybrid posture is driver-justified** — the decision names the driver demanding it; a reflexive re-platform/serverless/k8s choice with no driver is rejected as accidental complexity.
- **The shared-responsibility boundary is explicit** per component and consistent across the threat model and the reliability model — no control assumed on the wrong side, no double-paid control.
- **Cost is bounded** — an order-of-magnitude monthly run-cost estimate against the stated budget, with egress/transfer/per-request pricing where it applies; defers the deep model to `../90-specialized/cost-finops-architecture.md` but is never silent.
- **No infra-introduced un-accepted SPOF** on an availability-SLO path; every `QAS-AVAIL-*`/`QAS-PERF-*` on the topology's critical path has a met response-measure (capacity model or a managed SLA); the 8 fallacies honored where the topology spans AZs/regions/networks.
- Every ADR is complete (context · ≥2 options · decision · positive AND negative consequences · driver refs); every diagram referenced compiles; every finding carries a `driver_ref` (`shared/finding-schema.md`).
- **Gate:** hard oracles green (six-pillar coverage · managed-vs-self TCO+lock-in · posture driver-justified · shared-responsibility explicit · cost bounded · no un-accepted SPOF · fallacies honored · ADRs complete · diagrams compile) AND (design) rubric mean ≥ 0.8 (right-sized posture · sustainability+cost first-class · operability designed in).

## Output
Write `artifacts/cloud-well-architected/report.json` per `shared/report-format.md` — `drivers[]` (the per-pillar targets), `findings[]` per `shared/finding-schema.md` (`type:"cost"|"availability"|"security"|"operability"|"performance"|"risk"` as the pillar dictates; `oracle:"hard"` for a skipped pillar, a managed-vs-self choice with no TCO/lock-in, a reflexive un-driver-justified posture, a wrong-side shared-responsibility assumption, an unbounded cost, or an infra-introduced un-accepted SPOF; `oracle:"soft"` for an over-provisioned / not-right-sized / sustainability-as-afterthought craft gap), each finding carrying its `driver_ref` + the pillar + the TCO/lock-in tradeoff + the rejected alternatives + an ADR-shaped recommendation, plus a **per-pillar coverage matrix** (pillar → finding/non-risk → driver) and the **risk register** (likelihood×impact + disposition). For **design** mode, the authored artifacts land in `architecture/<slug>/` per `project-architecture-config.md` — the posture ADRs in `adr/NNNN-*.md` (e.g. `NNNN-managed-datastore-over-self-hosted.md`, `NNNN-lift-and-shift-over-serverless.md`), the per-pillar assessment + the cloud-native-vs-lift-shift decision in `design-doc.md`, the run-cost lines in `capacity-model.md`/`tradeoffs.md`, the shared-responsibility split folded into `threat-model.md` — and the `report.json` `verification` block records the six-pillar coverage, the hard-oracle results, and the evaluator's rubric score. Open driver questions (no stated budget, no RTO/availability, no chosen cloud target) are listed explicitly for the controller — never silently assumed.

## Guardrails
Per `shared/guardrails.md`: **propose & document — never write product code and never apply infrastructure.** This flow authors the posture ADRs, the per-pillar assessment, and the cost/shared-responsibility notes into `architecture/<slug>/`; it never edits `src/**`/`backend/**`/`frontend/**`, never runs `terraform apply` / a migration / a live cloud change, and never touches a real environment — the controller builds, the operator applies the plan. Ground every pillar finding in a driver (no decision without a quality-attribute scenario or a budget it serves; no "best practice" without the driver it protects). **Right-size, don't résumé-drive** — default to managed/boring where it lowers TCO for a small team (e.g. a managed datastore over a self-run replica set, lift-and-shift over a reflexive serverless re-platform); accept lock-in only where leverage beats the reversal cost; surface over-engineering as a risk. Reuse the existing stack and the deploy-ready container images before introducing a new runtime; a new managed service needs a driver-justified ADR with its TCO + lock-in tradeoff. Record the tradeoff and the rejected alternatives on every decision, never just the winner. Cross-reference the sibling flows rather than duplicating them: `deployment-topology.md` (the C4 deployment view + environments/regions the pillars are argued against), `infrastructure-as-code.md` (the IaC design + immutable infra + environment parity this posture is realized through), and `../90-specialized/cost-finops-architecture.md` (the deep cost model + FinOps + serverless-vs-provisioned economics this flow bounds but does not fully derive).
