# containerization-orchestration

**Group:** 50-infrastructure  ·  **Runs as:** subagent: ../.claude/agents/infrastructure-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Decide the runtime packaging and orchestration strategy — container image discipline, the deployment platform (Kubernetes topology vs managed PaaS vs serverless), service-mesh adoption, sidecar patterns — including the honest "do you actually need Kubernetes?" check against the concrete drivers. Answers the structural question: *which orchestration platform is right-sized for this system's scale, team size, and operational complexity, and what does choosing it cost on the other attributes?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack and deployment target (per the project's config — e.g. reverse-proxy same-origin, deploy-ready Dockerfiles, dev-only compose/Makefile at root), scale and SLO targets (confirm whether the system is internal/enterprise-scale or consumer-scale; concurrent-user expectations and tenant count to be confirmed), team topology (small team vs platform-ops function), compliance regime (data-sensitivity class and any audit-logging hard control), and the artifact location (`architecture/<slug>/`).
- The architecture **drivers**: the prioritized quality-attribute scenarios (utility tree from `workflows/00-drivers/quality-attribute-scenarios.md`) that the orchestration choice must satisfy — specifically the scale scenario (peak concurrency and RPS), the availability scenario (RTO/RPO, if stated — surface as an open driver question if not), and the operational-cost constraint (the team cannot operate what it cannot reason about). A flow run with no drivers confirmed downgrades scope and says so.
- Preconditions: the project's Dockerfiles (per `project-architecture-config.md`) and compose file have been read; the deployment-topology design (if authored) has been read (`workflows/50-infrastructure/deployment-topology.md`); the observability design (if authored) has been read (`workflows/40-cross-cutting/observability-architecture.md`); the infrastructure-as-code design (if authored) has been read (`workflows/50-infrastructure/infrastructure-as-code.md`). Confirm the system's current topology (per `project-architecture-config.md`) — e.g. a modular-monolith pair vs a microservices mesh — and do not propose a service-mesh or multi-service k8s topology unless a concrete driver (multi-service proliferation, mTLS requirement across many autonomous services) demands it.

## Oracle (source of truth)

- **hard:**
  - The orchestration-platform choice is **driver-justified** — the decision ADR names the specific quality-attribute scenario(s) it satisfies (scale: peak concurrency, RTO/RPO requirement, team-size and operational-load constraint). Choosing Kubernetes without a driver that demands it (N independent services, team with k8s operational capability, availability SLO that a managed PaaS cannot meet) is an unjustified decision — flag it `ARCH-CONT-001`, `oracle:"hard"`, `severity:"major"`.
  - Every container **defines resource requests and limits** (CPU and memory) — a container with no limits on a shared node is a noisy-neighbour bomb and an un-accepted SPOF on the node's stability. Absence of limits blocks (`oracle:"hard"`).
  - Every service exposes a **health/readiness probe** (HTTP or exec) — an orchestrator without probes will route to an unready replica or fail to restart a crashed one silently. Absence blocks (`oracle:"hard"`).
  - Every cross-container call has a **defined timeout and retry policy** — the fallacy "the network is reliable" (Fallacy 1) and "latency is zero" (Fallacy 2) apply; an unbounded synchronous call across container boundaries blocks (`oracle:"hard"`, `finding-schema.md`).
  - The **operational cost of the chosen platform** is acknowledged as a tradeoff in the ADR — a k8s cluster has a control-plane cost, a cert-manager, an ingress controller, a CNI plugin, an etcd, and requires runbooks; this must be weighed against the team's capacity. A design with no operational-cost estimate where cost is a stated constraint blocks (`quality-attribute-rubric.md` → *Cost bounded*).
  - **Immutable images** — the container image is the unit of deployment; no `apt-get` or `pip install` at runtime, no mutable shared volumes for code, no image tagged `latest` in a production manifest. A mutable-image pattern blocks (the immutability guarantee is the orchestration contract) (`oracle:"hard"`).
  - **STRIDE per container trust boundary** — for every trust boundary between containers (including east-west traffic, service-to-service calls, and any service mesh control plane), the threat model must include a STRIDE entry with an explicit control or an accepted risk per category. Where the project's compliance regime (per `project-architecture-config.md`) includes sensitive data classes and an audit-logging hard control, east-west trust boundaries are in scope regardless of whether a service mesh is adopted — if no mesh is present, the threat model must name the alternative control (e.g., in-pod colocation, network policy, mTLS via sidecar). A trust boundary with no STRIDE entry blocks (`oracle:"hard"`; `security-architecture` trust-boundary oracle; `quality-attribute-rubric.md` → *Trust boundary secured*). Where a service mesh is proposed, the mTLS trust model is a required sub-entry of the relevant STRIDE boundary record.
  - The **platform ADR is complete** (per `shared/adr-template.md` and `quality-attribute-rubric.md` → *ADR complete*): it must contain context, ≥ 2 genuine options including the simpler alternative, a decision outcome that names the specific quality-attribute response-measure it now satisfies, and consequences that enumerate both positive effects and negative costs (operational burden, lock-in, migration cost). An ADR missing any of these fields — particularly one with no recorded alternatives or no negative consequences — blocks (`oracle:"hard"`).
  - The **C4 deployment diagram is authored as diagrams-as-code** (Structurizr / PlantUML / Mermaid / C4, per `shared/diagramming-standards.md`) and must render without error; every container named in the L2 Container view must appear as a deployment node in the deployment view; names must be consistent with the ADR and design doc. A diagram that does not compile or that has inconsistent naming blocks (`oracle:"hard"`; `quality-attribute-rubric.md` → *Diagram compiles*).

- **soft:**
  - The runtime platform is **right-sized** (YAGNI / simplicity rubric): a managed PaaS (Cloud Run, App Service, Elastic Beanstalk, Fly.io) or a single-host Docker Compose-in-prod beats Kubernetes for a small system where the team topology cannot sustain k8s operations. The evaluator penalizes k8s adoption that earns its complexity only by scale and team drivers the system does not yet have.
  - A **service mesh** (Istio, Linkerd, Cilium) is adopted only where mTLS and traffic-management across many independently-owned services earn their complexity cost — for a two-container (SPA + API) modular monolith, a mesh is accidental complexity and scores 0.0 on the simplicity rubric dimension.
  - The **sidecar pattern** (ambassador, adapter, or logging/metrics sidecar) is used where a cross-cutting concern genuinely cannot be in the main process, not as a default packaging habit.
  - Container image layering follows a **build-time discipline** (multi-stage Dockerfiles, minimal base images, non-root user, pinned base digests) — not gated, but absence is a craft finding.

## Standards & techniques

- **Immutable images / the container contract** — the image contains exactly what runs; the environment (config, secrets) is injected via env vars or mounted secrets at deploy time, never baked in. Multi-stage Dockerfiles separate the build toolchain from the runtime image. Pin base images by digest in production manifests; `latest` is forbidden on critical paths.
- **Kubernetes primitives** — `Deployment` (declarative replica management, rolling-update strategy), `Service` (stable DNS + load-balanced ClusterIP/NodePort/LoadBalancer), `Ingress` / `Gateway API` (L7 routing, TLS termination), `HorizontalPodAutoscaler` (HPA — scale on CPU/custom metrics to a response-measure), `Namespace` (isolation boundary), `ResourceQuota` and `LimitRange` (enforce requests/limits cluster-wide), `PodDisruptionBudget` (preserve availability during voluntary disruptions). These are the vocabulary; use them only where k8s is the justified choice.
- **The k8s-is-overkill check** — evaluate the orchestration choice against three axes before defaulting to k8s: (1) *service count*: one or two containers do not need a service mesh, a CNI plugin, or etcd HA; a managed PaaS is sufficient; (2) *team size and k8s operational competence*: a small team without a dedicated platform-ops function pays a disproportionate operational tax for k8s; (3) *availability driver*: if the RTO/RPO is unstated or the system is internal-only with a moderate availability SLO, a managed PaaS with built-in rolling deploys meets the driver at a fraction of the operational cost. Apply Gall's Law: start with the simpler platform that works; evolve to k8s when a concrete driver demands it.
- **Managed PaaS / serverless alternatives** — Cloud Run (GCP), Azure Container Apps, AWS App Runner, Fly.io, Render: fully managed container runtimes that handle scheduling, health-checking, rolling deploys, and autoscaling without the control-plane tax. Prefer over k8s unless the driver demands k8s-specific capabilities (custom scheduling, pod-level networking, a service mesh, stateful workloads requiring PersistentVolumeClaims, or multi-tenant node isolation).
- **Service mesh** (Istio / Linkerd / Cilium) — provides mTLS between services, traffic management (weighted routing, retries, circuit-breaking at the sidecar), and L7 observability (golden-signal metrics per service pair). Complexity cost: a mesh adds a control-plane (istiod/linkerd-control-plane), a per-pod sidecar proxy (Envoy / linkerd-proxy), and a set of CRDs that must be understood and operated. The **CAP-style tradeoff**: a mesh buys zero-trust east-west security and fine-grained traffic control; it costs operational surface area, increased memory/CPU per pod, and a new failure domain (the sidecar). Only justified when (a) there are many independently-deployed services at a trust boundary requiring mTLS and (b) the team has the operational competence to run the control plane.
- **Sidecar pattern** (from the cloud-design-pattern catalog) — co-locate a helper container in the same pod for concerns the main process cannot or should not own: log forwarding (Fluent Bit), metrics scraping (Prometheus exporter), mTLS termination (Envoy sidecar, without a full mesh). Prefer in-process solutions first; reach for the sidecar only when the concern is genuinely cross-cutting and injecting it into the application would violate the separation of concerns or the language boundary.
- **AKF scale cube** — position the system on the cube per its current drivers (per `project-architecture-config.md`): the X-axis (horizontal duplication / stateless replicas) is typically the right-sizing for an early-stage system — add replicas behind a load balancer. The Y-axis (functional decomposition into services) and Z-axis (data partitioning) are only pulled forward when a concrete driver demands them (e.g. a deferred integration seam becoming a live scale driver) — otherwise they remain out of scope. The orchestration choice must match the current scale-cube position, not an anticipated future one.
- **Well-Architected pillars (infra lens)** — Operational Excellence (automated deploy, runbooks, health probes), Reliability (no-SPOF replicas, graceful drain, PDB), Performance Efficiency (right-sized instance types, HPA only where scale is a driver), Cost Optimization (no idle control-plane, right-size before scaling out), Security (least-privilege service accounts, no root containers, secrets from a secret store, not env baked in the image).
- **STRIDE at the container boundary** — Spoofing: service-account tokens and mTLS where services communicate; Tampering: read-only root filesystems where practical; Repudiation: audit logs for privilege escalation and image pulls; Information Disclosure: no secrets in image layers or env vars baked at build time; Denial of Service: resource limits + PDB; Elevation of Privilege: non-root UID, no `privileged: true`, drop Linux capabilities.
- **The 8 fallacies at the container boundary** — Fallacy 1 (network reliable): every inter-container HTTP call needs a timeout + retry + circuit-breaker, not naked `requests.get()`; Fallacy 5 (topology doesn't change): service discovery via DNS (`ClusterIP` service name or the container-network alias), never hardcoded IPs; Fallacy 4 (network is secure): zero-trust between containers if they cross a trust boundary — don't assume internal traffic is safe.
- **Cross-references:** `workflows/50-infrastructure/deployment-topology.md` (environments, regions/AZs, blue-green vs rolling — the orchestration platform must match the deploy strategy), `workflows/50-infrastructure/infrastructure-as-code.md` (IaC tooling owns the manifest/Helm chart authoring — this flow decides the topology; IaC flow operationalizes it), `workflows/40-cross-cutting/observability-architecture.md` (logs/metrics/traces must be designed for the chosen platform: stdout-to-collector for containers, sidecar exporters if needed, OpenTelemetry pipeline that works without a mesh).

## Step sequence

- **audit:** Frame the drivers (read `project-architecture-config.md` + the utility tree; identify the scale scenario and the availability scenario; surface open driver questions — RTO/RPO, peak concurrency — as explicit gaps, never assume) → Read the existing Dockerfiles and compose file → Walk the existing/proposed deployment against the hard oracles in order: (1) orchestration-choice driver traceability, (2) resource requests/limits defined, (3) health/readiness probes present, (4) cross-container timeout/retry defined, (5) operational-cost acknowledged, (6) immutable image discipline, (7) STRIDE per container trust boundary (unconditional — note alternative control if no mesh), (8) ADR complete (context + ≥2 options + decision + consequences), (9) C4 deployment diagram compiles → For each hard-oracle failure emit a `finding-schema.md` finding (`ARCH-CONT-NNN`, `oracle:"hard"`, traced to the driver + the structural fact) → Run the k8s-is-overkill check against the three axes → Emit the risk register for `oracle:"soft"` findings (right-sizedness, mesh complexity, sidecar overuse) → Return findings + risks, read-only, no artifact authored.

- **design:** Frame (extract/confirm drivers + open questions, read `project-architecture-config.md` + utility tree + existing Dockerfiles, read-only) → Explore (≥2 divergent platform directions, e.g. [A] single-host Docker Compose-in-prod behind a reverse proxy with a managed database; [B] managed PaaS — Cloud Run / App Service / Fly.io — with container-registry CI push; [C] k8s on a managed cluster — only if the driver check passes — with Deployment+Service+Ingress+HPA; each direction is a concrete spec: platform choice, replica strategy, health-probe design, resource limits, TLS termination point, cost estimate, and the fallacies it accounts for; read-only) → Evaluate (skeptical ATAM evaluator scores each direction against the utility tree + hard oracles; applies the k8s-is-overkill check; runs the decision matrix with position-swap to kill bias; picks a winner, grafts superior ideas from runners-up; names sensitivity points — e.g. "p99 latency is sensitive to the number of network hops the chosen platform adds" — and tradeoff points — e.g. "managed PaaS is a tradeoff point: wins on operational simplicity / cost, loses on scheduling flexibility"; read-only) → Document (one writer — `solution-architect` — authors: ADR `NNNN-orchestration-platform.md` per `shared/adr-template.md`; C4 deployment view as diagrams-as-code in `architecture/<slug>/diagrams/` per `shared/diagramming-standards.md`; resource-limits + probe spec per service; fitness-function spec for image-immutability and probe-presence checks; operational-cost model in `architecture/<slug>/capacity-model.md`; threat-model entries for the container trust boundary in `architecture/<slug>/threat-model.md`) → Verify (fitness functions: image-immutability check compiles, probe-presence check compiles; diagram renders without error; ADR has context + ≥2 genuine options including the simpler one + decision naming the response-measure satisfied + consequences positive and negative; cost model present; traceability matrix shows every QAS traced to a decision; STRIDE entries present per container trust boundary unconditionally — including the alternative-control entry where no mesh is adopted; evaluator re-scores from the artifacts) → loop ≤ 8 or pass.

## Assertions & exit gate

- The orchestration-platform ADR exists and is complete: context states the driver that justifies the choice, ≥2 genuine options considered (including the simpler one), decision outcome states the response-measure it now satisfies, consequences lists both positive and negative (operational cost is a named negative), one-way-door flag set if the platform is hard to change later.
- The k8s-is-overkill check ran: if k8s is chosen, the three axes (service count, team k8s competence, availability SLO) are all documented as justifying the choice; if k8s is rejected, the winning platform is named with the driver it satisfies that k8s did not earn.
- Every container in the design has `resources.requests` + `resources.limits` defined (CPU + memory); no `latest` tag in a production manifest.
- Every service exposes a health probe (liveness) and a readiness probe (distinct — a pod that is alive but not ready must be drained from the load balancer before termination).
- Every cross-container call has a documented timeout (bounded, not infinite) and a retry policy (count, backoff, idempotency assumption); fallacies 1 and 2 honored.
- The threat model contains a STRIDE entry per container trust boundary; if a service mesh is adopted, the mTLS trust model is explicitly stated; if no mesh, the threat model names the alternative control (in-pod colocation, network policy, or other stated mechanism) — an uncontrolled east-west boundary blocks.
- A sidecar, if present, has a traced driver (a cross-cutting concern the main process cannot own) — its absence is acceptable and penalizing absence is not the rule; its unjustified presence is the finding.
- The C4 deployment diagram renders without error; every container in the L2 Container view maps to a deployment node in the deployment view; names are consistent with the ADR and design-doc.
- The operational-cost estimate (order-of-magnitude, $/month for the chosen platform at the stated scale) is present; no component is "scale infinitely, cost unstated."
- Traceability: every QAS that drove the orchestration choice is traced to the platform ADR; no unjustified decisions.
- **Gate:** hard oracles green (driver-justified choice, resource limits, health probes, bounded timeouts, immutable images, cost model, ADR complete, diagram compiles, STRIDE coverage) AND (design) rubric mean ≥ 0.8 (right-sized platform, no accidental complexity, operational cost acknowledged, sidecar/mesh only where earned).

## Output

Write `artifacts/containerization-orchestration/report.json` per `shared/report-format.md` — findings and risks per `finding-schema.md` (IDs: `ARCH-CONT-NNN`; type: `operability` for probe/limit gaps, `scalability` for HPA/autoscaling decisions, `security` for trust-boundary and sidecar-mTLS gaps, `cost` for unbounded platform cost, `risk` for the k8s-overkill risk if k8s is chosen without a driver). The verification block (audit: hard-oracle results + risk register; design: fitness-function results + traceability coverage + hard-oracle results + rubric score per soft dimension) is included. Design artifacts land in `architecture/<slug>/`:

```
architecture/<slug>/
├── adr/NNNN-orchestration-platform.md      platform decision (MADR; one-way-door flagged)
├── adr/NNNN-service-mesh-decision.md       mesh adopt/reject decision (if the question was live)
├── diagrams/deployment-view.puml|.md       C4 deployment view — containers → nodes/regions/AZs
├── capacity-model.md                       $/month estimate for the chosen platform at stated scale
├── threat-model.md                         STRIDE entries for container trust boundaries
└── fitness-functions.md                    image-immutability check + probe-presence check specs
```

Open driver questions (unstated RTO/RPO, unconfirmed peak concurrency, undecided cloud target) are listed explicitly in the report's `open_questions[]` block — never silently assumed.

## Guardrails

Per `shared/guardrails.md`: propose and document — never write product code, never apply IaC, never run `docker build`, `kubectl apply`, or a live cloud change. The agent authors the ADRs, diagrams, and fitness-function *specifications*; the controller builds the manifests and the operator applies them. Every decision traces to a driver from the utility tree; a choice no driver demands is accidental complexity — flag it and do not ship it. Reuse the existing Dockerfiles and the existing deployment pattern (per `project-architecture-config.md`) before introducing a new runtime; boring, proven technology is preferred unless a driver demands novelty. The evaluator is separate from the writer — it never grades its own work, and it explicitly penalizes the k8s/mesh/sidecar choices that earn complexity without a driver paying for it. Never write to project memory; lessons go in the ADRs and the returned report.
