# anti-patterns-fallacies

**Group:** 80-patterns  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Catalog and scan the architecture against the known failure modes that have killed real systems: the **8 fallacies of distributed computing**, the **distributed monolith**, **big-ball-of-mud**, **golden-hammer**, **premature optimization**, **accidental/incidental complexity**, **God-object/God-service**, **resume-driven development**, and the **spec/inner-platform/lava-flow** patterns. Answers: *which of these failure modes are present in this design, which driver do they harm, and what is the simplest structural remedy — not the cleverest one?* An honest scan that names the simpler alternative beats a thorough one that hides it.

## Inputs & preconditions

- From `project-architecture-config.md`: the current architecture shape, the existing module boundaries, the stack lock, the right-size and boring-tech constraints, the team topology, and the locations of the architecture description, ADR log, and diagram source.
- The **drivers / utility tree** from `../00-drivers/quality-attribute-scenarios.md` — the prioritized 6-part scenarios each anti-pattern finding must name as a harmed driver. A finding with no harmed driver is a preference, not a finding (`../../shared/quality-attribute-rubric.md` grounding rule).
- The **decomposition and dependency graph** from `../10-styles-decomposition/decomposition-strategy.md` — the module/service boundaries and the call/data-dependency map the scan walks.
- The **architecture review checklist** from `../60-evaluation/architecture-review-checklist.md` — the anti-pattern scan is a focused pass within the broader gate review; read it first to avoid duplicating findings already covered by dedicated flows.
- Preconditions: `project-understanding.md` + `docs/` (BRD/PRD) read; the utility tree exists or this flow surfaces missing drivers as open questions. If the utility tree is absent, the scan is degraded (cannot name harmed drivers) and the flow says so explicitly — it does not invent numbers.

## Oracle (source of truth)

The **anti-pattern catalog** (§Standards & techniques) applied structurally to the design: each anti-pattern is either present or absent, and if present, a finding is emitted naming the harmed driver, the structural evidence, and the simplest remedy. The catalog is the external oracle — not the agent's taste.

- **hard:**
  - **fallacies_honored** — the 8 fallacies of distributed computing (`../../shared/fallacies-and-tradeoffs.md` §1–8) are the oracle for the networked-system failure modes. Where a fallacy applies, the design must account for it. For each of the 8: if the design *assumes the fallacy away* (e.g. a cross-service call with no timeout = "the network is reliable"; an unbounded synchronous chain = "latency is zero"; no pagination/projection on large fan-out payloads = "bandwidth is infinite"; no mTLS/authz between services = "the network is secure"; hardcoded hostnames = "topology doesn't change"; assumed coordinated deploy = "there is one administrator"; no cross-AZ egress cost in the cost model = "transport cost is zero"; no schema evolution/ACL at foreign boundaries = "the network is homogeneous") → a `hard` finding with the specific fallacy violated, the structural fact, the harmed driver, and the remedy. A design that does NOT distribute (a modular monolith, single-process) is NOT expected to account for fallacies that do not apply — the scope is explicit: mark each fallacy as applicable/not-applicable with a one-line justification.
  - **distributed_monolith_absent** — if the system claims independent-deploy autonomy for multiple services/units but any of these hold: (a) they share a write path on the same DB collection with no ownership contract, (b) they must release in lockstep, (c) they form an unbounded synchronous chain — then the design is a distributed monolith (`../10-styles-decomposition/decomposition-strategy.md`). This is a `hard` finding: all the cost of distribution, none of the autonomy. Where `project-architecture-config.md` declares a modular-monolith style, the single-deploy is an *accepted* decision, not a finding — but any proposal that introduces service-splitting without a driver-justified ADR is a distributed-monolith risk.
  - **accidental_complexity_flagged** — every architectural element (a layer, a pattern, a technology, a seam) that traces to no driver in the utility tree is accidental complexity (`../../shared/guardrails.md` §2 — "a choice no driver demands is accidental complexity"). Each instance is a `hard` finding with the element, the missing driver, and the remedy (remove it, or surface the hidden driver). This is the hardest oracle to apply honestly because an evaluator must resist rewarding complexity.
  - **anti_pattern_findings_complete** — for each of the 13 named anti-patterns in §Standards, the scan produces: (a) present/absent verdict with structural evidence, (b) if present: the harmed driver (`driver_ref`), the severity (`critical`/`major`/`minor`), and an ADR-shaped remedy naming the simpler alternative rejected alternatives. An anti-pattern scan that produces no findings on a non-trivial design is suspect and must be called out.

- **soft:**
  - **simpler_alternative_named** — for every anti-pattern found, the remedy names the *simplest* structural alternative (a table instead of a queue, a module instead of a service, a library instead of a platform, an existing pattern instead of a bespoke one). The scan is honest about over-engineering — the most common anti-pattern in ambitious designs — and does not soften a finding because the complexity was impressive to build. A scan that names only what is wrong, without naming what is simpler, is an incomplete craft result.
  - **scan_proportionate** — the finding set is proportionate to the design's actual scope and the project's right-size constraints (`project-architecture-config.md` locked rules). A scan of a correct, simple, right-sized design that manufactures findings to appear thorough is itself a form of over-engineering in the audit.
  - **architectural_smell_detection_advisory** — the five **structural** architectural smells this catalog overlaps with a deterministic detector — **god-component** (→ #7 God-object/God-service), **cyclic dependency** (→ #3 big-ball-of-mud's acyclic-dependency check), **hub-like dependency** (a component with abnormally high combined afferent+efferent coupling — the same signal as #7's Ca check and #12's chatty-interface fan-out), **ambiguous interface** (an interface whose callers each use a different, non-overlapping slice of it — the connascence/stamp-coupling read of #9 inner-platform and #13 shared-database), **unstable dependency** (a stable module depending on a volatile one, violating the Stable-Dependencies Principle — the directional check behind #3's "impose acyclic dependencies" remedy) — detection is **static-tool-first**, never pure-LLM. Run `dependency-cruiser`/**ArchUnitTS** (or `deptrac`/`madge`/`import-linter` per the stack, matching `../60-evaluation/architecture-conformance.md`'s tool catalog) against the real import graph and let the tool's afferent/efferent counts, cycle report, and instability metric (`I = Ce/(Ca+Ce)`) decide presence/absence deterministically — the same static-first, LLM-second split `../60-evaluation/architecture-conformance.md` uses for layering conformance. The LLM's job on these five is strictly **explain, prioritize, and cluster** the tool's output (which smell threatens which driver, which to fix first, which cluster of violations is one root cause) — **never** to independently assert "this is a God-component" from reading source in isolation: published architectural-smell-detection studies put pure-LLM precision/F1 at **~40%**, far too low to gate a build on. A pure-LLM call on any of these five smells is therefore always `oracle:"soft"` and **advisory only** — it never blocks the gate by itself, and it is promoted to a `hard` finding (folded into the matching numbered anti-pattern's finding above, e.g. #7/#3/#12/#9/#13) only once the static tool corroborates it with a concrete metric or reported cycle. The remaining anti-patterns in this catalog that have no equivalent structural detector (the 8 fallacies, distributed-monolith's deploy-lockstep/ownership check, RDD, lava-flow, stovepipe, golden-hammer, premature-optimization, accidental-complexity) need the driver/decision/runtime context a dependency-graph tool can't see, so they stay LLM-reasoned per the oracles above; this advisory narrows to the five smells with a genuine structural equivalent.

## Standards & techniques

The 13 named failure modes this flow scans against. Each is cited by its canonical name; the scan names which apply and why:

1. **The 8 fallacies of distributed computing (Deutsch/Gosling)** — the eight false assumptions naïve designs make about a network (`../../shared/fallacies-and-tradeoffs.md`): (1) the network is reliable, (2) latency is zero, (3) bandwidth is infinite, (4) the network is secure, (5) topology doesn't change, (6) there is one administrator, (7) transport cost is zero, (8) the network is homogeneous. Each is a hard-oracle check where applicable; each violation names the cross-service call or integration point that assumes it away.

2. **Distributed monolith** — services (or modules presented as services) that share a database, must deploy in lockstep, or form a synchronous dependency chain. The defining diagnostic: *you get all the operational cost of a distributed system with none of the autonomy* (`../10-styles-decomposition/decomposition-strategy.md`). Simpler alternative: revert to an explicit modular monolith (one deploy, clear module boundaries, no false independence claim) — or fix the coupling so independence is real.

3. **Big-ball-of-mud (Foote/Yoder)** — no discernible module/bounded-context boundary; everything calls everything; changes propagate unpredictably. Diagnostic: the call/dependency graph is a dense mesh with no clusters corresponding to business capabilities. Harmed driver: every modifiability/evolvability scenario. Remedy: decompose on the criteria in `../10-styles-decomposition/decomposition-strategy.md`; impose acyclic dependencies.

4. **Golden hammer (Maslow's hammer)** — a single technology/pattern applied to every problem regardless of fit (e.g. Kafka for every state change in a system with no stream-processing driver, MongoDB for every data shape when a relational model fits, a message bus for operations that are naturally synchronous). Diagnostic: the same tool appears across contexts where a simpler alternative would serve the driver at lower operational cost. Remedy: match the tool to the driver — not the driver to the tool already on the résumé.

5. **Premature optimization (Knuth)** — structural complexity introduced for a performance/scale concern that no stated driver demands at the current scale. Diagnostic: a caching layer, a CQRS split, a sharding scheme, or a CDN tier added before a performance QAS with a response measure exists and is shown to be unmet by the simpler design. Harmed driver: evolvability and simplicity — the complexity is paid now for a benefit not yet needed. Remedy: add the optimization when the driver appears and a capacity model shows the simple design can't meet it; not before.

6. **Accidental/incidental complexity (Brooks)** — complexity the problem did not require; complexity introduced by the solution, not inherent in the domain. Distinguished from *essential complexity* (the inherent difficulty of the problem domain — unavoidable) vs *accidental complexity* (the infrastructure, the tooling ceremony, the layering tax — avoidable). Diagnostic: remove an element; if no driver or business outcome degrades, it was accidental. Every layer, every abstraction, every pattern adds accidental complexity; it is only justified when it buys a named quality attribute. Remedy: remove or simplify the element; state the driver it would need to justify its presence.

7. **God-object / God-service** — one module or service that owns too many responsibilities, making it a coupling magnet and a deployment/change bottleneck. Diagnostic: a module's change log shows unrelated concerns changing together; afferent coupling (Ca) is very high relative to the rest of the graph; a single component appears in the critical path of nearly every scenario. Harmed driver: every independent-deploy, modifiability, and availability scenario that passes through it. Remedy: split on business capability per `../10-styles-decomposition/decomposition-strategy.md`; each resulting unit must trace to its own driver.

8. **Resume-driven development (RDD)** — technology or pattern choices driven by engineering interest or market cachet rather than a traced driver. Diagnostic: a new technology (k8s, Kafka, event sourcing, GraphQL federation, a service mesh) that appears in a proposal for a system whose drivers are fully satisfied by the boring stack, with no driver-referenced ADR justifying the novelty. The grounding rule (`../../shared/quality-attribute-rubric.md`) is the oracle: no driver + no number = a preference. Remedy: the boring technology that meets the driver wins; record the rejected flashy option and *why* it was rejected (the ADR's rejected-alternatives block).

9. **Spec / abstract factory overload / inner-platform effect** — a bespoke abstraction or configuration layer that reimplements the platform it sits on (a custom ORM over the ORM, a homegrown service-locator over the DI container, a meta-configuration engine that configures the configuration system). Diagnostic: the abstraction's API surface mirrors the platform it wraps, but with less documentation, less testing, and more unknown failure modes. Remedy: use the platform directly; introduce an abstraction only at a boundary where the underlying technology is likely to change (and a driver demands that isolation).

10. **Lava flow** — dead code, unused modules, stale integration points, deprecated APIs, or skeleton services left in the architecture because they are "too risky to remove." They accumulate until their purpose is unknown and removal feels riskier than keeping them. Diagnostic: an element with no callers, no driver-traced responsibility, and no owner in the team topology. Harmed driver: every evolvability and operability scenario. Remedy: flag + remove; if uncertain, add a usage fitness function (assert no callers → merge if green).

11. **Stovepipe / silo architecture** — independent vertical stacks that solve the same cross-cutting concern (auth, audit, data access) redundantly, with no shared abstraction, so a cross-cutting change requires N parallel edits. Diagnostic: the same concern is implemented in K>1 modules with no shared library/service/contract. Remedy: extract the shared concern into a module or library with a single owner; not a shared service unless an independent-deploy driver justifies it.

12. **Chatty interface / nano-service** — service or module boundaries cut so fine that a single business operation fans out across N≥4 synchronous hops, compounding latency (fallacy 2), failure probability (fallacy 1), and serialization cost (fallacy 7). Diagnostic: a latency budget trace shows the majority of the budget consumed in inter-unit transport, not in computation. Remedy: coarsen the boundary to reduce the hop count — merge, co-locate, or batch; the AKF Y-axis split buys independent-deploy not latency.

13. **Shared-database integration** — multiple independent modules/services using the same DB schema as their integration contract, coupling their data models and deployment schedules invisibly. The data-coupling form of the distributed monolith. Diagnostic: two units write to the same collection/table without a recorded ownership contract and access API. Remedy: assign a single writing owner per collection; reads cross-module via a module API or an event, never a foreign write (`../10-styles-decomposition/decomposition-strategy.md` — the no-silent-shared-write-path hard oracle).

## Step sequence

- **audit:** Frame the drivers (read the utility tree; confirm which fallacies apply given the architecture style — distributed, modular-monolith, single-process) → walk the call/dependency graph and the data-access map, checking each of the 13 anti-patterns structurally: for fallacies, trace every cross-process call for its timeout/retry/fallback; for distributed-monolith, check the deploy-lockstep + shared-write + sync-chain triple; for accidental complexity, check every element against the utility tree for a traced driver; for RDD, check every non-boring technology against its driver-referenced ADR → emit findings per `../../shared/finding-schema.md` (`type:"evolvability"` for big-ball-of-mud/lava-flow/stovepipe, `type:"scalability"` for chatty/nano-service, `type:"integration"` for shared-database/distributed-monolith, `type:"risk"` for premature-optimization, `type:"decomposition"` for God-service/RDD/golden-hammer, `type:"driver"` for accidental-complexity) ranked critical→cosmetic; each finding carries its `driver_ref`, `evidence` (the structural fact — not an assertion), `tradeoff` (what the anti-pattern bought and what it costs), and `recommendation` (the simpler alternative, ADR-shaped). Read-only — authors no artifact, edits no source.

- **design:** **Frame** (confirm the utility tree and the decomposition graph; mark each of the 13 anti-patterns as in-scope/not-applicable with a one-line justification; read-only) → **Explore** (`architecture-explorer` ×N produces divergent remediation directions for the highest-severity findings: e.g. (a) remove the accidental layers entirely, (b) consolidate via a shared-kernel module, (c) introduce a lightweight abstraction at the true change boundary — each a concrete spec with its structural change, the driver it serves, and its Coupling↔Autonomy tradeoff point; read-only) → **Evaluate** (`architecture-evaluator` scores each direction against the utility tree + the anti-pattern hard oracles via ATAM; finds sensitivity points — *the God-service is sensitive to the team's ability to split ownership* — and tradeoff points — *removing shared-DB integration buys deploy-independence but costs the simple single-query join*; order-swapped to kill position/sophistication bias; the simpler remedy that removes the anti-pattern and meets the driver scores higher than the elegant refactor that introduces new patterns; grafts the best ideas from runners-up) → **Document** (the ONE writer — `solution-architect` — authors an **ADR per significant anti-pattern remedy**: context (the anti-pattern found, the driver harmed) · ≥2 options (keep/accept as risk vs. the remediation directions) · decision · consequences positive and negative · the rejected alternatives with their tradeoffs; no product-code changes; diagrams updated as needed to reflect the structural change; the fitness-function specs that enforce the remedy in CI — e.g. a dependency-rule check that prevents the God-service's callers from growing, a lava-flow check that asserts no callers on a flagged module) → **Verify** (fitness-function specs reviewed for completeness; traceability check: every finding has a corresponding ADR or accepted-risk record; the `fallacies_honored` and `distributed_monolith_absent` hard oracles re-checked against the post-remedy design; diagram-compile; evaluator re-scores) → loop ≤10 or pass.

## Assertions & exit gate

- Every finding names the **specific anti-pattern** from the §Standards catalog (not "bad design"), the **driver it harms** (`driver_ref` → a `QAS-*` in the utility tree), and the **structural evidence** (the traced fact — a specific component, collection, call chain, or missing timeout — not an assertion).
- Every finding carries a **simpler alternative** as its `recommendation`: an ADR-shaped statement naming the structural change and the response-measure it now satisfies, plus the alternatives rejected and their costs.
- The **fallacies_honored** check is explicit per fallacy: each of the 8 is marked applicable/not-applicable with justification; each applicable fallacy is either satisfied (with the mechanism named) or a `hard` finding.
- The **distributed_monolith_absent** check passes: no unit claims independent-deploy autonomy it cannot exercise, OR the coupling is an explicit ADR-recorded accepted decision.
- The **accidental_complexity_flagged** check passes: every architectural element traces to a driver in the utility tree, OR is flagged as accidental complexity and a removal/simplification recommendation is recorded.
- Every anti-pattern finding is matched by either a remediation ADR (design mode) or an accepted-risk record (audit mode) — no finding is left open without a disposition.
- For design mode: every remediation ADR is complete (context · ≥2 options · decision · positive AND negative consequences · rejected alternatives · fitness-function follow-up) per `../../shared/adr-template.md`; diagrams compile (`../../shared/diagramming-standards.md`).
- The scan is honest: a trivially simple, right-sized design that genuinely has few applicable anti-patterns produces few findings — the absence of findings is evidence of a correct design, not a shallow scan, provided the scan shows its work (the per-fallacy applicable/not-applicable table; the per-anti-pattern present/absent verdict with evidence).
- **Gate:** hard oracles green (`fallacies_honored` · `distributed_monolith_absent` · `accidental_complexity_flagged` · `anti_pattern_findings_complete`) AND (design) rubric mean ≥ 0.8 (`simpler_alternative_named` · `scan_proportionate`).

## Output

Write `artifacts/anti-patterns-fallacies/report.json` per `shared/report-format.md` — findings per `finding-schema.md` grouped by anti-pattern then severity (critical → cosmetic), each with `driver_ref`, `evidence`, `tradeoff`, `alternatives_rejected`, and `recommendation`; a **per-fallacy applicability table** (8 rows: fallacy name · applicable? · mechanism or finding-id); a **per-anti-pattern verdict table** (13 rows: name · present/absent · finding-id or "n/a"); the **risk register** (`type:"risk"` items with likelihood×impact and accept/mitigate/monitor disposition); and the verification block (hard-oracle results, traceability coverage, rubric score).

For design mode, authored artifacts land in `architecture/<slug>/`:
- `adr/NNNN-<anti-pattern-slug>.md` — one MADR ADR per significant anti-pattern remedy (context, ≥2 options, decision, consequences positive+negative, rejected alternatives, fitness-function follow-up).
- `fitness-functions.md` — the executable/automatable checks that enforce the remedies in CI: e.g. a dependency-rule fitness function asserting no module calls the God-service directly; a lava-flow check asserting a flagged module has no callers; a foreign-write check asserting no module writes a collection it does not own.
- `diagrams/` — updated C4 Container/Component views (diagrams-as-code, Mermaid/PlantUML per `project-architecture-config.md`) reflecting the structural changes; verify they compile before counting them done.

Run reports (the per-anti-pattern evidence, the option matrix, the sensitivity/tradeoff points) go to `artifacts/anti-patterns-fallacies/`; the architecture itself lives in `architecture/<slug>/`. Never scatter artifacts outside these two trees.

## Guardrails

Per `../../shared/guardrails.md`: **propose & document — never write product code** (this flow authors ADRs, fitness-function specs, and updated diagram source; the controller refactors the modules/services). **Ground every finding in a driver** — a finding with no harmed `driver_ref` is a preference; drop it or tie it to an attribute the system must satisfy (`../../shared/quality-attribute-rubric.md` grounding rule). **Name the simpler alternative** — the anti-pattern's remedy is never "introduce a more sophisticated pattern"; it is always "remove the unnecessary complexity" or "use the boring, proven path that meets the driver." **One writer** authors all remediation ADRs in a sequenced pass to preserve conceptual integrity across the remedy set; a second concurrent writer would produce inconsistent decomposition criteria in the ADRs. **The evaluator never grades its own work** — the `architecture-evaluator` runs as a separate pass, penalizes accidental complexity and unjustified novelty explicitly, and does not reward an elaborate remediation plan over a simple one that meets the drivers. **Cross-reference** `../../shared/fallacies-and-tradeoffs.md` (the canonical source for the 8 fallacies + the canonical tradeoffs), `../10-styles-decomposition/decomposition-strategy.md` (the distributed-monolith check and the no-silent-shared-write-path oracle), `../60-evaluation/architecture-conformance.md` (the static-tool-first, LLM-second pattern this flow's `architectural_smell_detection_advisory` soft oracle mirrors — that flow gates layering/dependency conformance on the same tool catalog; this flow gates the five overlapping structural smells the identical way), and `../60-evaluation/architecture-review-checklist.md` (the gate review this scan feeds into — do not duplicate findings already owned by the dedicated availability/security/data flows; this flow owns the anti-pattern and fallacy catalog specifically).
