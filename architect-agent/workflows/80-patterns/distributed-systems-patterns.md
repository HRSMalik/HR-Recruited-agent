# distributed-systems-patterns

**Group:** 80-patterns  ·  **Runs as:** subagent: ../../agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
The distributed-systems **pattern playbook** — the canonical, named mechanisms for the hard problems that appear the instant a write, a decision, or a piece of state has to live across more than one process: **consensus/quorum** (Paxos/Raft), **leader election**, **replication** (leader-follower / multi-leader / leaderless), **sharding/partitioning**, the **saga + outbox + CDC** trio, **idempotency + dedup**, **CRDTs** for conflict-free merge, and **logical/vector clocks** for the ordering problem. Each pattern is catalogued here with its **contract** (what it promises), its **consistency/availability guarantee** (its point on the CAP/PACELC curve), its **failure modes**, and — the part that matters most — **when NOT to use it**. The design question this flow answers: *for a given cross-process invariant, which established pattern holds it, at which guarantee, against which failures — and is it the simplest one that holds, or résumé-driven over-reach?* This is the reference the data and integration flows reach into; it exists so a cross-boundary problem is solved with a **known contract** (known tradeoffs, known failure modes) rather than a bespoke mechanism whose unknowns surface in production. The cardinal failure mode it kills: a hand-rolled "leader" elected by a `if (i_am_first)` check, a "retry" with no idempotency key, or a "highest timestamp wins" merge using wall-clock time — bespoke distributed mechanisms that are subtly, silently wrong.

## Inputs & preconditions
- From `project-architecture-config.md`: the **existing stack and its distribution primitives** — e.g. many projects start as a **modular-monolith over a single-primary datastore** (no replica set, no broker, no queue, no quorum, no leader election in the application); read the actual current-state topology from `project-architecture-config.md`, never assume it. Where the current stack has no distribution primitives, this is load-bearing: most of these patterns **must not be introduced today** (there is no second node to replicate to, no broker to be at-least-once over, no partition to merge after) — they are the **seam vocabulary** for near-term bounded contexts and any deferred integration/agent layer noted in `project-architecture-config.md`, not the current build. The constraint against introducing a new datastore/broker/runtime without a driver-justified ADR (boring-tech-preferred) governs every pattern here.
- The **drivers / utility tree** (`../00-drivers/quality-attribute-scenarios.md`): the prioritized 6-part quality-attribute scenarios this decision must satisfy — the data-integrity invariants (e.g. *QAS-DATA-01: no financial/monetary record is ever lost or double-applied; QAS-DATA-02: minting a business-entity identifier never produces a duplicate*), the availability/recoverability scenarios the pattern trades against (PACELC: a quorum-write costs latency; a leaderless read costs staleness), and the scale scenario (a sharding decision needs a stated partition-key cardinality + a hot-spot bound). A pattern with **no driver demanding it is accidental complexity** — flag it, don't ship it (`../shared/guardrails.md` §2).
- The **target**: the specific cross-process problem under review/design — *what state lives across how many nodes/services, what must stay true across them, and which failure (crash, partition, message loss, duplicate delivery, concurrent write, clock skew) threatens the invariant.* The relevant methods: consensus (Paxos/Raft) + quorum, leader election, the three replication topologies, sharding, saga/outbox/CDC, idempotency/dedup, CRDTs, logical/vector clocks, the CALM theorem.
- **Preconditions**: the utility tree exists (else `status:error` — you cannot pick a pattern with no invariant it protects); `../20-data/consistency-transactions.md` is read (the per-write-path consistency **model** is chosen *there*; this flow supplies the **mechanism** that realizes it — the two must agree); `../30-integration/event-driven-patterns.md` is read where the pattern crosses the DB↔broker boundary (the outbox/CDC choice is shared between these flows). Audit mode reads the current write/coordination paths from the code/schemas and edits nothing.

## Oracle (source of truth)
The **business invariant** each cross-process operation must preserve, made into a hard, structurally-checkable contract by **naming the pattern, its guarantee, and its failure modes** — never "it's distributed and it works." A pattern is only correct *relative to a stated invariant and a stated failure model*; the oracle is the invariant expressed against CAP/PACELC and the named pattern's published contract.

- **hard** (measurable / traceable / structurally checkable):
  - **Every applied pattern is named with its contract.** Each pattern used in the design is recorded as `{pattern, invariant it holds, consistency/availability guarantee (its CAP/PACELC point), failure modes it tolerates, failure modes it does NOT (and the residual risk), when-NOT-to-use}`. A "we'll coordinate the nodes" / "we'll replicate it" with no named pattern and no guarantee **blocks** — a bespoke mechanism with no published contract is an un-evaluatable unknown.
  - **The choice is grounded in a CAP/PACELC position + a driver.** Every pattern that touches consistency states its point on the curve **per data path** and traces to the `QAS-*` it serves: a **quorum/consensus** write is **CP** (consistent, unavailable under partition without quorum) — justified only by a strong-consistency invariant; a **leaderless / multi-leader replication** read is **AP** (available, eventually consistent) — justified only where staleness is harmless. A pattern whose CAP/PACELC point contradicts its driver's invariant (e.g. AP leaderless replication chosen for an account-balance write that must be CP) is a **hard** finding (the **Consistency stated** oracle, `../shared/quality-attribute-rubric.md`).
  - **Idempotency is mandatory wherever at-least-once delivery or retries exist.** Any consumer of an outbox/CDC event, any saga step, any retried RPC, any redelivered message **carries an idempotency key (or a natural dedup key) and a dedup store/check** so at-least-once does not become apply-twice. A retriable/replayable write with no dedup is a **hard** finding — it defeats the at-least-once guarantee the rest of the design relies on (this is the same gate `../20-data/consistency-transactions.md` and `../30-integration/event-driven-patterns.md` enforce; the three must agree).
  - **No silent dual-write; the saga/outbox/CDC contract is honored end-to-end.** Any operation that mutates two systems (two stores, or a store **and** a broker/external API) inside one logical step uses **outbox+CDC** (event row written in the same local transaction as the business write; a relay tails/polls and publishes at-least-once to an **idempotent** consumer) or a **saga** (a sequence of local transactions, each with a **complete, semantically-correct compensating action** for every step that can fail, the un-compensatable step ordered last as the pivot) — never a bare `save()` then `publish()`. A dual-write with no atomic link is a **critical** hard finding (defers to `../20-data/consistency-transactions.md`).
  - **Quorum math is stated and correct where quorum is used.** A leaderless/Dynamo-style replication path states **N (replicas), W (write quorum), R (read quorum)** and shows **W + R > N** for read-your-writes / strong-quorum consistency (or explicitly accepts a sloppy quorum + hinted-handoff with its staleness window). A consensus path names the protocol (**Raft / Paxos / Multi-Paxos**) and its quorum (⌊N/2⌋+1) and states the **even-cluster-size split-vote** and **minority-partition-unavailability** failure modes. "We replicate to 3 nodes" with no W/R/N math **blocks**.
  - **Leader election has a fencing mechanism.** Any leader-election / single-writer pattern (Raft leader, a lease, a distributed lock) carries a **fencing token / epoch / term number** so a **stale leader** (one that GC-paused, was partitioned, or lost its lease but doesn't know it yet) cannot corrupt state after a new leader is elected — the classic *two leaders during a partition* (split-brain) failure mode is explicitly handled, not assumed away (violates *fallacy #1 the network is reliable* and *#5 topology doesn't change*, `../shared/fallacies-and-tradeoffs.md`). Election with no fencing **blocks**.
  - **Ordering is established by a logical mechanism, never wall-clock.** Any path that depends on event/operation ordering across nodes uses **logical clocks (Lamport)** for a total order or **vector clocks** for causal order + concurrency detection — **not** `Date.now()` / server timestamps to decide "latest wins" (clock skew makes wall-clock ordering across nodes wrong, and silently — this is the **"last-write-wins on wall-clock"** anti-pattern). A cross-node ordering/merge that relies on wall-clock time is a **hard** finding.
  - **Sharding states its partition key + hot-spot + rebalancing story.** Any sharding/partitioning decision names the **partition key**, shows its **cardinality and access distribution** (so it doesn't create a hot shard — e.g. partitioning by `tenant_id` where one tenant holds 60% of volume is a hot-spot), and states the **rebalancing mechanism** (consistent hashing to bound re-mapped keys on a node add/remove; or range-split) and the **cross-shard query/transaction cost** (a query spanning shards is scatter-gather; a transaction spanning shards needs a saga, not a local txn). A shard key with no distribution analysis **blocks**.
  - **CRDTs are used only where the merge is genuinely commutative, and the type is named.** A CRDT choice names the **specific type** (G-Counter, PN-Counter, OR-Set, LWW-Register, RGA…) and shows the operation is **associative + commutative + idempotent** so concurrent replicas converge **without coordination** (per the **CALM theorem**: a computation has a coordination-free, consistent implementation **iff it is monotone**). A CRDT proposed for a non-monotone invariant (e.g. a balance that must never go negative — a constraint CRDTs cannot enforce without coordination) is a **hard** finding: the merge "converges" to a wrong-but-agreed state.
- **soft** (craft, rubric-scored only after hard oracles are green):
  - **The named pattern is reused over a bespoke mechanism.** A published pattern (Raft, the saga, outbox+CDC, consistent-hashing, a named CRDT, vector clocks) carries a **known contract, known tradeoffs, known failure modes**; a hand-rolled coordinator/lock/merge carries unknowns and re-litigates solved problems. Inventing a bespoke distributed mechanism where a catalog pattern fits is a craft failure even if it "works in the demo" (`../shared/guardrails.md` §3).
  - **The simplest pattern that holds the invariant is chosen (right-sizedness / YAGNI).** Climb the ladder only as far as the invariant + the failure model force: **a single-node local transaction** (no distribution) beats **single-leader replication** beats **a quorum/consensus protocol** beats **multi-leader/CRDT merge**; **a redrawn boundary that makes the write local** beats **a saga**; **an atomic single-document counter update** (e.g. `$inc`/`findAndModify`, or the storage engine's equivalent) beats **a Raft-backed sequence service**. No Raft where one primary node holds the invariant; no sharding where the data fits one node; no CRDT where there are no concurrent writers. This is the over-engineering gate the config's *Locked rules* demand for a right-sized internal/enterprise tool (per `project-architecture-config.md`).
  - **The guarantees are stated honestly — no exactly-once theater.** The design says **at-least-once + idempotent-consumer** or **at-most-once**, never "exactly-once delivery" (which is impossible over an unreliable network — *exactly-once **processing*** is the achievable goal, via at-least-once delivery + dedup). It says **eventual consistency with a stated convergence/staleness window**, not "consistent." It names the **residual failure** every pattern leaves (the saga's intermediate-visible inconsistency window; the CDC relay's oplog-window data-loss risk if it falls behind; the quorum's minority-partition unavailability). A design that presents only upside is incomplete craft (`../shared/quality-attribute-rubric.md` → tradeoff honesty).

## Standards & techniques
- **Consensus — Paxos / Raft / Multi-Paxos** (Lamport; Ongaro & Ousterhout). A protocol for a set of nodes to **agree a single value / an ordered log** despite crashes, using a **majority quorum (⌊N/2⌋+1)**. Contract: linearizable, **CP** — progresses only while a majority is reachable; the **minority partition is unavailable** (by design — it cannot have a quorum). Failure modes: split-vote on an even cluster (use odd N); leader churn under flapping networks; the log compaction/snapshot burden. *When NOT to use:* when one node owns the invariant (no agreement needed); when the throughput cost of every write going through a quorum exceeds the latency budget; when eventual consistency is acceptable (use replication, not consensus). Used **under the hood** by the datastores/coordinators you'd *buy* (etcd, ZooKeeper, a managed DB's replica-set election) — the architect's job is to **choose the managed primitive, not implement Raft** (boring-tech, build-vs-buy).
- **Quorum (Dynamo-style)** — for **leaderless replication**: N replicas, a write needs **W** acks, a read reads **R**; **W + R > N** guarantees a read overlaps the latest write (read-your-writes); **W > N/2** prevents two concurrent writes both succeeding. Tunable: `W=N,R=1` (read-fast, write-slow, strongly-consistent reads), `W=1,R=N` (write-fast, read-slow), `W=R=⌈(N+1)/2⌉` (balanced). Failure modes: **sloppy quorum + hinted handoff** keeps writes available under partition but widens the staleness window and needs **read-repair / anti-entropy (Merkle trees)** to converge; concurrent writes to the same key produce **siblings** that need a merge (vector clocks or a CRDT). *When NOT to use:* when you need linearizability (use consensus) or when there is one writer (use single-leader).
- **Leader election + fencing** — elect **one** node to be the single writer/coordinator (via Raft, a lease in etcd/ZooKeeper, or a DB advisory lock) so a sequence of operations has a single serialization point. The **non-negotiable companion**: a **fencing token** (a monotonically increasing epoch/term) attached to every write the leader makes, checked by the resource — so a **stale leader** (paused, partitioned, lease-expired-but-unaware) is rejected and **split-brain** cannot corrupt state. Failure modes: split-brain without fencing; election storms; the leader as a throughput bottleneck and a SPOF if failover is slow (the leader is on a critical path → it needs the **No un-accepted SPOF** treatment, `../shared/quality-attribute-rubric.md`).
- **Replication topologies** — the three shapes, each a distinct CAP/PACELC point:
  - **Single-leader (leader-follower)** — all writes to one leader, async/sync-replicated to followers; reads can fan to followers (read-scaling). **Strong on the leader, eventually-consistent on followers** (replication lag → stale reads + the *read-your-own-writes* problem). Sync replication = CP + higher write latency (PACELC: E→C); async = AP-ish + lower latency but a **lost-write window** on leader failover (RPO > 0). The default and the **right near-term shape for most projects** the day they need HA: a **replica set** on the project's chosen datastore (one primary, electable secondaries — Raft-like election under the hood) gives failover (RTO) + read-scaling without the application owning any consensus code.
  - **Multi-leader** — multiple write-accepting leaders (e.g. multi-region, or offline-capable clients), async-replicated between them. Buys write-availability/locality but **introduces write-write conflicts** that must be resolved (LWW — lossy; a merge function; or a CRDT). *When NOT to use:* when conflicts are unacceptable on the invariant (money) — the conflict resolution will silently drop a write.
  - **Leaderless (Dynamo/quorum)** — clients write to many replicas directly under W/R quorum (above). High availability, no failover step, but eventual + needs read-repair/anti-entropy and sibling-merge. *When NOT to use:* internal enterprise CRUD with no availability driver demanding it — it's operational complexity with no payoff.
- **Sharding / partitioning** — split data across nodes by a **partition key** to scale beyond one node's capacity. **Hash partitioning** (even spread, but range scans become scatter-gather) vs **range partitioning** (efficient scans, but hot ranges). **Consistent hashing** (a hash ring) bounds the keys remapped when a node joins/leaves to ~`K/N` instead of nearly all — the standard rebalancing mechanism. The hard problems: **hot spots** (a high-cardinality, evenly-distributed key avoids them; a low-cardinality or skewed key — a tenant/customer id with one dominant tenant — creates them), **cross-shard queries** (scatter-gather + merge), and **cross-shard transactions** (no local txn — needs a saga). *When NOT to use:* when the data fits one node (true for most internal/enterprise tools at their current scale, per `project-architecture-config.md`) — sharding is the last resort after vertical scaling, read-replicas, and caching, never the first.
- **The saga + outbox + CDC trio** (Garcia-Molina & Salem; Chris Richardson; the dual-write cure — shared with `../20-data/consistency-transactions.md` and `../30-integration/event-driven-patterns.md`):
  - **Saga** — a long-lived transaction as a sequence of **local** transactions; each step triggers the next; on failure, **compensating transactions** semantically undo the completed steps (refund, not delete; release-hold, not roll-back-time). Two coordination styles: **orchestration** (a central process-manager drives + compensates — visible, testable, a coupling point) vs **choreography** (each service reacts to events + emits the next — decoupled, but the flow is implicit). Failure mode: the **intermediate-state visibility window** (the system is observably half-done mid-saga — isolation is sacrificed, not atomicity) and the **un-compensatable step** (order it last as the pivot, or make it idempotently retriable).
  - **Outbox** — to write a store **and** emit an event atomically, write the event into an **outbox table/collection in the same local transaction** as the business write; a separate relay publishes it. Converts a dual-write into one local write + an at-least-once relay.
  - **CDC (Change-Data-Capture)** — the relay: tail the store's change log (e.g. MongoDB change streams / oplog tailing, or Debezium on a relational WAL) and publish the change as an event — no polling, near-zero latency. Failure mode: the relay must be **HA and keep up** — if it falls behind the **change-log window**, changes are lost on the relay side (not the primary); it typically requires a **replica set / multi-node cluster** (not a standalone single-node store). Where the project's current stack (per `project-architecture-config.md`) is a single primary with no replica set, CDC needs replica-set + an HA relay as **prerequisites** — surface them as open drivers, don't assume them.
- **Idempotency + dedup** — the property that applying a write ≥1 times has the same effect as applying it once: an **idempotency key** (a client-supplied or natural unique id) + a **dedup store** (a unique index on `processed_events` / a Redis seen-set), or a **naturally idempotent op** (`set`/upsert, not `increment`). The precondition that makes **at-least-once delivery safe** and turns *exactly-once delivery* (impossible) into **exactly-once *processing*** (achievable). Without it, every retry corrupts.
- **CRDTs (Conflict-free Replicated Data Types)** (Shapiro et al.) — data types whose concurrent replicas **merge deterministically without coordination** because their merge is a **commutative, associative, idempotent** join (a semilattice). Named types: **G-Counter / PN-Counter** (counters), **G-Set / 2P-Set / OR-Set** (sets with add/remove), **LWW-Register / MV-Register** (registers), **RGA / Logoot** (sequences, for collaborative text). Contract: **strong eventual consistency** — replicas that have seen the same ops are in the same state, no merge conflict ever. The hard limit (**CALM theorem**, Hellerstein): coordination-free consistency is achievable **iff the computation is monotone** — CRDTs **cannot** enforce a non-monotone global invariant (uniqueness, a non-negative balance, a capacity cap) without falling back to coordination. *When NOT to use:* any invariant that's a global constraint (money that can't go negative, a unique business-entity identifier) — the merge converges to an agreed-but-invalid state.
- **Logical & vector clocks** (Lamport; Fidge/Mattern) — the **ordering problem**: there is **no global clock** in a distributed system, and **wall-clock timestamps are unreliable for ordering** across nodes (skew, NTP jumps). **Lamport clocks** give a **total order** consistent with causality (if A→B then L(A)<L(B), but not the converse). **Vector clocks** give **causal order + concurrency detection** (they can tell whether two events are causally related or genuinely concurrent — the basis for detecting Dynamo siblings). **Hybrid Logical Clocks (HLC)** combine physical + logical for an order that's close to wall-clock but monotone. Use these — never `Date.now()` — wherever cross-node ordering or last-writer-wins resolution matters.
- **The 8 fallacies + the canonical tradeoffs** (`../shared/fallacies-and-tradeoffs.md`) — every pattern here is an *answer to a fallacy*: idempotency/retries/timeouts answer *#1 the network is reliable*; quorum/consensus answer *partial failure*; fencing answers *#5 topology changes*; CDC backpressure + claim-check answer *#3 bandwidth* and *#2 latency*; schema/versioning across replicas answers *#6 one administrator*. Name the **tradeoff and the point chosen on it** for every pattern: consistency↔availability/latency (CAP/PACELC), sync↔async replication, orchestration↔choreography, coupling↔autonomy.

## Step sequence
- **audit:** Frame the cross-process invariants from the utility tree (which state lives across ≥2 processes; which must stay strongly consistent vs may be eventual) → enumerate every **coordination / replication / cross-process write / ordering / merge** point in the existing or proposed architecture and, for each, identify the pattern in use (or the **bespoke mechanism standing in for one**) and walk: *is the pattern named with a contract? does its CAP/PACELC point match the invariant's driver? is there a fencing token on any leader/lock? an idempotency key on every retried/replayed write? quorum math (W+R>N) where quorum is claimed? a hot-spot/rebalancing story on any shard key? a logical (not wall-clock) ordering where order matters? a CALM-monotone check on any CRDT?* → flag every **bespoke distributed mechanism with no contract** (hard), every **CAP/PACELC point contradicting its driver** (hard), every **un-fenced leader/lock** (hard, split-brain), every **non-idempotent retriable write** (hard), every **wall-clock ordering across nodes** (hard), every **quorum with no W/R/N math** (hard), every **shard key with no distribution analysis** (hard), every **CRDT on a non-monotone invariant** (hard), and — the right-sizing finding for *this* project — every **pattern introduced with no driver demanding it** (over-engineering, `soft`→ `major` if it adds a datastore/broker the team must operate) → emit findings + the risk register (read-only; authors nothing, edits no source).
- **design:** **Frame** (extract/confirm the per-problem invariant + its CAP/PACELC requirement + the failure model from the utility tree, read-only) → **Explore** (≥2 divergent pattern choices per problem as concrete specs — e.g. for a future *sequential business-identifier minting at scale* invariant (an order/invoice/entity number that must never repeat): *(A) keep a single atomic counter update on one primary (e.g. `$inc`/`findAndModify`, or the storage engine's equivalent) — no distribution, the current pattern, correct while one primary owns it*; *(B) a Raft-backed sequence service (etcd lease) — CP, fenced, but a new component to operate*; *(C) per-shard ID ranges handed out in blocks + a node-id prefix — no central coordination, monotone-per-shard, gaps tolerated — each spec naming its pattern, its CAP/PACELC point, its failure modes, and its when-NOT; read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each against the invariant + the hard oracles via ATAM — does the guarantee actually hold the invariant under the stated failures? where's the **sensitivity point** (e.g. "consensus latency is sensitive to cluster RTT — a cross-AZ quorum blows the write budget")? the **tradeoff point** (e.g. "quorum W=N buys read-your-writes *but* makes a single slow replica stall every write")? — order-swapped to kill bias; **the simplest pattern that holds the invariant wins, accidental complexity and unjustified novelty penalized explicitly** per the config's Locked rules) → **Document** (the **one writer**, `solution-architect`, authors: the **pattern register** (the table of `{pattern, invariant, guarantee/CAP point, failure modes, when-NOT}`), the saga/quorum/replication-topology **sequence + deployment diagrams-as-code**, and one **MADR ADR** per significant pattern choice with its rejected alternatives, its CAP/PACELC point, and — for any one-way-door pattern (event-sourcing, a sharding scheme, a multi-leader topology) — the **one-way-door flag**) → **Verify** (the hard gates below as fitness functions + the evaluator re-scores from the artifacts) → loop ≤10 or pass.

## Assertions & exit gate
- **Every applied pattern has a row** in the pattern register: `{pattern, invariant it holds, CAP/PACELC guarantee, failure modes tolerated, failure modes NOT tolerated + residual risk, when-NOT-to-use}`. No blank cells on any architecturally-significant cross-process mechanism; no bespoke mechanism without a named-contract equivalent considered.
- **Every pattern's CAP/PACELC point matches its driver's invariant** — no AP mechanism on a strong-consistency money/uniqueness invariant, no CP mechanism (paying latency/availability) on a harmless-staleness read; the point is stated **per path** and traces to a `QAS-*`.
- **Every retried / replayed / at-least-once write is idempotent** (named key + dedup store, or a naturally idempotent op); the guarantee is stated as **at-least-once + idempotent ⇒ exactly-once *processing***, never "exactly-once delivery."
- **No silent dual-write survives** — every two-system write is outbox+CDC, a saga (with complete compensations + the pivot ordered last), or a redrawn-to-local boundary (defers to `../20-data/consistency-transactions.md`).
- **Every leader/lock is fenced** (epoch/term/fencing token); split-brain under partition is explicitly handled, not assumed away.
- **Every quorum states W/R/N and shows W+R>N** (or an explicit sloppy-quorum accepted-risk with its staleness window + the anti-entropy mechanism).
- **Every shard key has a distribution + hot-spot + rebalancing analysis**; cross-shard query/transaction cost is stated.
- **Every cross-node ordering/merge uses a logical/vector clock, never wall-clock**; every CRDT names its type and passes the **CALM-monotone** check for its invariant.
- **The simplest pattern that holds the invariant is chosen** — any pattern adding a datastore/broker/coordinator the team must operate carries an ADR whose rejected-alternatives show no simpler pattern holds (right-sizing gate; over-engineering is a finding, not a feature).
- Each pattern choice is an **ADR** (context · ≥2 options · decision · positive+negative consequences · the CAP/PACELC point · driver refs · one-way-door flag) tracing to a `QAS-*`; diagrams compile (`../shared/diagramming-standards.md`).
- **Gate:** hard oracles green — **every pattern named with a contract, CAP/PACELC point matching its driver, idempotency on every retriable write, zero silent dual-writes, fencing on every leader, W+R>N math on every quorum, distribution analysis on every shard key, logical-clock (not wall-clock) ordering, CALM-monotone on every CRDT** — AND (design) rubric mean ≥ 0.8 (named-pattern-reused + simplest-pattern-that-holds + guarantees-stated-honestly).

## Output
Write `artifacts/distributed-systems-patterns/report.json` per `../shared/report-format.md` — findings/risks per `../shared/finding-schema.md` (`type` ∈ `{data, integration, availability, scalability, risk}`; `oracle:"hard"` for a bespoke mechanism with no contract, a CAP/PACELC point contradicting its driver, an un-fenced leader, a non-idempotent retriable write, a quorum with no W/R/N math, a shard key with no distribution analysis, a wall-clock cross-node ordering, or a CRDT on a non-monotone invariant; `oracle:"soft"` for a bespoke mechanism where a catalog pattern fits, an over-sized pattern with no driver, or exactly-once theater; `driver_ref` = the `QAS-*` invariant it serves; every finding carries its `tradeoff` and `alternatives_rejected`, ADR-shaped). The `verification` block lists the fitness-function results (the pattern-register-complete check, the idempotency check, the no-dual-write structural check, the no-wall-clock-ordering check, the fencing-present check) + hard-oracle results + the rubric score. Design-mode authored artifacts land in `architecture/<slug>/` per `project-architecture-config.md`: the **pattern register** in `design-doc.md` (or `tradeoffs.md`), the saga/quorum/replication **sequence + deployment diagrams-as-code** in `diagrams/`, the per-pattern fitness-function entries in `fitness-functions.md`, and one **MADR ADR** per significant pattern choice in `adr/NNNN-*.md` — never scattered. Open questions (an unstated availability target that would justify a replica set, an undecided RPO that drives sync-vs-async replication, a missing tenant-volume distribution that a shard key would need) are listed for the controller, never silently assumed.

## Guardrails
Per `../shared/guardrails.md`: **propose & document — never write product code, never stand up a broker/coordinator/replica set, never run a migration or change a live datastore.** This flow authors the pattern register, the ADRs, and the saga/quorum/replication diagrams; the controller implements the idempotency/outbox/saga/fencing code and the operator provisions any replica set or coordinator. Ground every pattern in a **driver** (the cross-process invariant from the utility tree) — **no pattern without an invariant it protects; no pattern a driver doesn't demand** (the cardinal over-engineering sin for a right-sized internal/enterprise tool). **Reuse the named catalog pattern** (Raft via a managed primitive, the saga, outbox+CDC, consistent-hashing, a named CRDT, vector clocks) before inventing a bespoke coordinator/lock/merge — and prefer the **simplest pattern that holds the invariant** (no-distribution local-txn > single-leader > quorum/consensus > multi-leader/CRDT), right-sized to the project's actual current-state stack (per `project-architecture-config.md`) — surface the heavier patterns as **seams for near-term bounded contexts and any deferred integration/agent layer** named in `project-architecture-config.md`, not as things to build now. State the guarantee **honestly** — at-least-once + idempotent (exactly-once *processing*), eventual-with-a-stated-window, the residual failure named — never exactly-once-delivery theater. Record the **tradeoff and the rejected alternatives** on every pattern (its CAP/PACELC point, sync↔async, orchestration↔choreography), never just the winner. Read-only audit edits nothing; one writer (`solution-architect`) preserves conceptual integrity; the evaluator re-checks each hard oracle from the artifacts, never the writer's prose; never write to project memory — lessons live in the ADRs and the tradeoffs register.
