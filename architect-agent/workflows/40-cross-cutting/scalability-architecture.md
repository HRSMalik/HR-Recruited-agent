# scalability-architecture

**Group:** 40-cross-cutting  ·  **Runs as:** subagent: ../.claude/agents/reliability-architect.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose

Decide how the system scales — *along which axis, in which order, and to what bound* — so a stated load-growth scenario is met without scaling the wrong dimension or scaling at all where no driver demands it. Uses the **AKF scale cube** to name the axis: **X** (clone the whole behind a load balancer), **Y** (split by function / capability), **Z** (split by data / customer). Answers: *which axis does the scale driver actually require, is the service on that axis stateless (or does it carry an explicit state-externalization plan), what are the autoscaling bounds and the load-shedding policy under surge, and where is the bottleneck the capacity model says will break first?* The default bias is **X-axis horizontal scaling of stateless replicas** — the cheapest, most reversible move — and **adding scale only to the axis a prioritized scenario demands**, not résumé-driving toward sharding or microservices.

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack (per `project-architecture-config.md` — e.g. a web SPA + API service over a primary datastore, same-origin behind a reverse proxy; note whether a broker, mesh, or autoscaler exists today), the deployment target (containerized/deploy-ready per config; a managed-datastore prod deploy may be a *separate, unbuilt* ticket), the scale/SLO targets, the team topology (per config — draw boundaries this team can operate, per Conway's law), and the `architecture/<slug>/` artifact location.
- The architecture **drivers**: the prioritized quality-attribute scenarios (the utility tree) from `workflows/00-drivers/quality-attribute-scenarios.md` and the `architecture/<slug>/quality-attributes.md` it produces. **Do not enter this flow without a load-growth or throughput scenario** — "scale" with no number (concurrent users, RPS, tenant/customer count, data-volume growth) is not a driver. Per `project-architecture-config.md`, if the source-of-truth requirements docs do **not** state a consumer-scale number (e.g. an internal enterprise tool with no public-scale driver), and the concurrency/tenant target is unstated, surface it as an open driver question and **downgrade to audit-only** — never invent a 10k-RPS target to justify scaling machinery.
- The **capacity model** from `workflows/40-cross-cutting/performance-capacity.md` (the back-of-envelope sizing) — the bottleneck on the scale path is *identified by the capacity model*, not guessed; if it is absent, run performance-capacity first or downgrade scope and flag the gap. The **partition/replication** strategy from `workflows/20-data/partitioning-sharding.md` is the Z-axis mechanism — only reached after X and Y are shown insufficient.
- **Preconditions:** `project-architecture-config.md` and the source-of-truth docs (BRD / PRD / understanding log, as named in `project-architecture-config.md`) read; the utility tree (or stub) available; current topology (replica count, statefulness of each service on the request path, session/state storage) read from the codebase. A flow with no load-growth scenario or no capacity model **downgrades to audit-only** and states so.

## Oracle (source of truth)

The **scaling axis is chosen and justified by a load-growth quality-attribute scenario, cheapest-axis-first (X before Y before Z)**, and the bottleneck that the chosen axis relieves is the one the **capacity model** names. Statefulness on the scale path and the autoscaling/load-shedding policy are structural facts, checked — not asserted. Everything is evaluated against the utility tree's response measures; no number + no traced driver = a preference, not a finding (`shared/quality-attribute-rubric.md`, the grounding rule).

- **hard:**
  - **The scaling axis is named on the AKF cube and justified by a scale scenario, cheapest-first.** The chosen axis (X / Y / Z) traces to a specific load-growth QAS (`QAS-SCALE-*`: source · stimulus · *environment = peak/surge* · response · **response-measure** — e.g. "300 concurrent users at month-end close, dashboard p95 ≤ 2s, error-rate ≤ 0.1%"). X (clone) must be shown insufficient before Y (functional split) is proposed, and Y insufficient before Z (data partitioning). An axis chosen out of order — Z-sharding or a service split where horizontal cloning would meet the measure — blocks as accidental complexity. An axis with no traced scenario blocks (no number = preference).
  - **Every service on the scale path is stateless, or carries an explicit state-externalization strategy.** A service that is to be X-axis cloned must hold no per-request affinity in process: session/auth state, in-memory caches that must be coherent, file-upload scratch, sticky counters (e.g. a sequence-counter store that mints a business identifier such as an order/invoice number) are named, and each is either externalized (shared store / token-borne / DB sequence with the right concurrency guarantee) or the service is declared explicitly stateful with sticky-session/affinity as a *signed-off* tradeoff. A silently stateful service on the clone path — load-balanced replicas that disagree, or a sequence that double-mints under concurrency — blocks.
  - **Autoscaling bounds and a scale-in safety policy exist (where autoscaling is in scope).** If the topology autoscales, the policy declares: the scaling signal (a *saturation* metric — CPU/RPS/queue-depth/concurrent-connections, never a lagging mean-latency alone), `min`/`max` replica bounds (an unbounded `max` is an unbounded cost — fails `cost_bounded`), the cooldown/stabilization window to prevent flapping, and **scale-in safety** (graceful connection draining + a termination grace period so in-flight requests are not killed). Where autoscaling is *not* yet in scope (per `project-architecture-config.md` — e.g. no autoscaler today), this collapses to a stated static capacity headroom factor against the peak scenario; "scale infinitely" with no bound or no drain policy blocks.
  - **A load-shedding / backpressure policy is defined for surge beyond the autoscale ceiling.** Because autoscaling is not instantaneous and capacity is finite, the design names what happens when arrival rate exceeds provisioned capacity: a load-shedding rule (reject low-priority/over-budget requests with `429` + `Retry-After` to protect the SLO of admitted requests — fail-fast, not collapse), and backpressure where a queue or pool fronts a slow dependency (bounded queue + queue-based load-leveling, not an unbounded in-memory buffer that OOMs). A scale path with no surge behavior beyond "add more replicas" assumes infinite capacity and instantaneous scaling — it blocks (this is the *finite bandwidth* + *latency-is-not-zero* fallacies; `shared/fallacies-and-tradeoffs.md`).
  - **The bottleneck is identified by the capacity model, and the chosen axis relieves it.** The single resource that saturates first under the scale scenario (typically the **primary datastore**, or a connection-pool ceiling, or an unindexed scatter query) is named with a number from `performance-capacity.md` — not "the app." X-axis cloning of stateless app replicas does **not** relieve a database write-throughput bottleneck (it amplifies it); if the model says the bottleneck is the shared datastore, the recommendation must address *that* (read replica → CQRS read model → cache → Y/Z split), not just add app replicas. A scaling decision that does not move the modeled bottleneck blocks (it spends cost without meeting the measure).
  - **Every cross-process call introduced by the scaling decision has a defined failure mode.** Adding a load balancer, a shared session/cache store, or splitting a function across a network boundary adds remote calls — each gets a bounded timeout, a retry policy (idempotent + backoff/jitter where safe), and a fallback or fail-fast per `shared/quality-attribute-rubric.md` (*Failure mode defined*) and `workflows/40-cross-cutting/availability-resilience.md`. A new synchronous hop on the scale path with an unbounded/missing timeout blocks (the *network-is-reliable* fallacy).
  - **Every scaling ADR is complete and traced.** Context · ≥2 options · decision · consequences (positive AND negative); `Driver refs` points at a `QAS-*` in `architecture/<slug>/quality-attributes.md`; the partition-key / service-split one-way-door decisions are flagged as such. An ADR with no rejected alternatives or no negative consequences blocks (`shared/adr-template.md`).

- **soft:**
  - **Scale is added to the axis the driver demands, cheapest-first — X before Y before Z.** Stateless horizontal cloning (X) is the default because it is cheap, linear, and a two-way door; functional decomposition (Y) and data partitioning (Z) are progressively more expensive, more operationally complex, and (Z especially) one-way doors. Reaching for Y/Z when X meets the measure is the cardinal accidental-complexity sin here and scores low on simplicity/YAGNI. For a project with no stated consumer-scale driver (`project-architecture-config.md`: *right-size, don't résumé-drive*) — e.g. an internal enterprise tool — the correct answer is frequently **"a single right-sized instance + a static headroom factor, no autoscaler, no sharding" — and saying so is a pass, not a gap.**
  - **Stateless-by-default is the design posture.** New services and request-path components are designed stateless from the start (externalized session, token-borne identity, DB-or-cache-backed shared state) so the X-axis stays open for free; baking affinity in early (in-process session, local file scratch, in-memory coherent cache) forecloses the cheapest scaling axis and scores lower even if it "works" at today's single replica.
  - **The scaling topology is right-sized and reversible.** Prefer the simplest axis and the smallest replica/headroom factor that meets the SLO with margin; over-provisioning redundancy or pre-sharding against an *unstated* growth driver is penalized as accidental complexity. Two-way-door moves (more replicas, a bigger instance, an external session store) are made cheaply; one-way-door moves (a Y-split into services this small team must then operate, a Z partition key) demand a flagged ADR with the operational-cost tradeoff stated.

## Standards & techniques

- **The AKF scale cube** (Abbott & Fisher, *The Art of Scalability*) — the axis taxonomy that makes "scale" a *decision* instead of an adjective:
  - **X-axis — horizontal duplication / cloning.** Run N identical stateless replicas of the same code behind a load balancer; each handles a fraction of requests. Linear, cheap, the most reversible move, and the default. *Requires statelessness* — replicas must be interchangeable. Relieves *request-handling* capacity; does **not** relieve a shared-datastore write bottleneck (it concentrates it). This is the first axis to reach for and usually the only one an internal tool needs.
  - **Y-axis — functional / service decomposition.** Split the system by *verb/noun* — by business capability or bounded context (e.g. intake vs documents vs settlement/payment vs audit) — so each scales, deploys, and fails independently. Buys independent scaling of a hot capability and team autonomy (Conway), at the cost of a network boundary, distributed-failure surface, and operational complexity. Justified by *a specific capability being the bottleneck* or *independent-deploy/team-ownership* drivers — not by fashion. Where the system is a modular monolith, its module boundaries are the *seam* for a future Y-split; do not actually split absent a driver (`project-architecture-config.md`).
  - **Z-axis — data / customer partitioning (sharding).** Split by a *lookup* on the data itself — by customer/company/tenant or key range — so each shard owns a slice of the data and load. The last resort: needed only when a single data node cannot hold the data or sustain the write load after X and Y. A one-way-door decision (the partition key is expensive to change). Delegated to `workflows/20-data/partitioning-sharding.md`; this flow only confirms X and Y are insufficient before recommending it.
- **Horizontal vs vertical scaling.** *Vertical* (scale-up: a bigger instance — more CPU/RAM) is the simplest move and the right first step for a single-instance internal tool with headroom, but it has a hard ceiling and a single-point-of-failure profile (one box). *Horizontal* (scale-out: more instances) removes the ceiling and adds redundancy but *requires statelessness* and a load balancer. The standard progression: scale **up** until the box-ceiling or the availability driver demands redundancy, then scale **out** (X-axis). Name which, and the ceiling/driver that triggers the switch.
- **Stateless service design + externalized state.** A service is *X-axis scalable* iff it holds no client-affine state in process across requests. Externalize: **session/auth** → stateless tokens (JWT or equivalent — per `project-architecture-config.md`'s existing model) or a shared session store (e.g. Redis); **coherent caches** → a shared cache, not per-replica in-memory; **sequences/counters** → a datastore sequence with the right atomicity (a datastore-native atomic increment/find-and-modify is atomic *at the datastore* — confirm it is not shadowed by an in-process counter before cloning); **file scratch** → object storage / shared volume. The sticky-session escape hatch (LB affinity) is a tradeoff that *re-couples* replicas — allowed only as a flagged, signed-off ADR.
- **Load balancing (L4 vs L7).** *L4 (transport)* balances by IP/port — fast, protocol-agnostic, no payload inspection. *L7 (application)* balances by HTTP semantics — path/host/header routing, TLS termination, retries, health-aware routing — and is what an HTTP API + SPA front (a same-origin reverse-proxy edge, per `project-architecture-config.md`) typically wants. Name the LB algorithm (round-robin / least-connections / consistent-hash for cache affinity) and the **health-check** (readiness probe) that takes an unhealthy replica out of rotation — a load balancer with no health check sends traffic into a dead replica.
- **Autoscaling (reactive vs predictive) + scale-in safety.** *Reactive* scales on a live saturation signal crossing a threshold (target-tracking on CPU/RPS/queue-depth/concurrency); simple and the default, but lags the spike by the boot+warm-up time. *Predictive/scheduled* pre-scales ahead of a *known* pattern (e.g. a month-end close or a scheduled batch-processing surge) — appropriate only when the load curve is predictable. Always set `min`/`max` bounds (bounded cost), a cooldown/stabilization window (anti-flapping), and **scale-in safety**: on scale-down, drain connections and honor a termination grace period so in-flight requests complete (never kill a replica mid-request). Choose the signal as a *leading saturation* metric, not a lagging latency mean.
- **Load-shedding + backpressure under surge** (resilience patterns; `workflows/80-patterns/resilience-patterns.md`). Capacity is finite and autoscaling is not instantaneous, so the design must degrade gracefully past the ceiling rather than collapse: **load-shedding** — reject excess/low-priority load fast (`429 Too Many Requests` + `Retry-After`, or shed by request class) to preserve the SLO of admitted work (a fast `429` beats a slow timeout for everyone). **Backpressure** — propagate "slow down" upstream rather than buffering unboundedly; front a slow dependency with a **bounded** queue (queue-based load-leveling), apply the **bulkhead** pattern to isolate a saturating dependency's pool from the rest, and a **rate limiter** at the edge. The anti-pattern is an unbounded in-memory buffer that converts a load spike into an OOM crash.
- **The fallacies this flow must honor** (`shared/fallacies-and-tradeoffs.md`): *bandwidth is infinite* and *latency is zero* (finite capacity + non-instant scaling → load-shedding/backpressure are mandatory, not optional); *the network is reliable* (every new hop introduced by scaling gets timeout+retry+fallback); *transport cost is zero* (cross-AZ/region replica chatter and egress show up in the cost model). The canonical tradeoffs in play: **Latency ↔ Throughput** (batching/queueing for throughput costs latency), **Cost ↔ Resilience/Performance** (replicas and headroom cost money — size to the driver, not the maximum), **Coupling ↔ Autonomy** (a Y-split decouples scaling but adds eventual consistency + operational surface). Name the point chosen on each.
- **Well-Architected — Performance Efficiency + Reliability + Cost pillars** as the cross-check (`workflows/50-infrastructure/cloud-well-architected.md`): scale horizontally and statelessly, adopt managed/elastic capacity where a driver demands elasticity, and right-size against a cost ceiling — the same "scale to the driver, not the maximum" posture this flow enforces.

## Step sequence

- **audit:** Frame the load-growth drivers (the `QAS-SCALE-*` scenarios + the capacity model's named bottleneck) → walk the current topology against each: which axis (if any) is being used; is every request-path service actually stateless (grep for in-process session/cache/counter/file state — confirm the sequence/counter mint is datastore-atomic and not shadowed in-process); is there a load balancer + health check; are there autoscaling bounds + scale-in drain; is there any load-shedding/backpressure past the ceiling; does the modeled bottleneck (the primary datastore) get relieved by the present approach → emit findings + a risk register (a silent SPOF on the scale path, a stateful clone-path service, an unbounded buffer, a wrong-axis scaling plan). **Read-only — authors no artifact, edits no source.** Crucially, audit also flags the *inverse* finding: scaling machinery (autoscaler / sharding / a service split) present with **no driver demanding it** is accidental complexity — a `soft`/over-engineering finding, not a pass.
- **design:** **Frame** (extract/confirm the `QAS-SCALE-*` scenarios + the capacity model's bottleneck; if no load-growth number is stated, downgrade and raise the open driver question — never invent one). → **Explore** (≥2 *divergent* scaling directions as concrete specs, read-only — e.g. *A:* vertical scale-up + static headroom, no autoscaler (the right-sized internal-tool default); *B:* X-axis stateless clones behind L7 LB + readiness probe + a read-replica for the read bottleneck + token-borne session; *C:* a Y-split of the bottleneck capability *only if* a driver demands independent scaling — each spec names the axis, the statelessness plan, the LB/autoscale/shed policy, the bottleneck it relieves, and the cost). → **Evaluate** (the skeptical `architecture-evaluator` scores each option against the utility tree's `QAS-SCALE-*` measures + the hard oracles via ATAM; finds the **sensitivity point** — e.g. "throughput is sensitive to the datastore write ceiling, not the app replica count" — and the **tradeoff point** — e.g. "X-clones improve request capacity but worsen DB connection pressure"; order-swapped to kill position bias; *penalizes the more elaborate option* — the simplest axis that meets the measure wins; picks a winner, grafts superior ideas from runners-up). → **Document** (the **one** writer — `solution-architect`, with `reliability-architect` advising read-only — authors the C4 Container + **Deployment** view showing replicas/LB/autoscale-group as diagrams-as-code, the scaling ADR(s) naming the axis + bounds + shed policy with rejected alternatives, the externalized-state plan, and the fitness-function specs; reuses the established patterns — LB, queue-based load-leveling, bulkhead, rate-limiter — before inventing). → **Verify** (fitness-functions + diagram-compile + traceability + failure-mode coverage as **hard** gates below; evaluator re-scores from the artifacts) → loop ≤10 or pass.

## Assertions & exit gate

- Every prioritized `QAS-SCALE-*` scenario traces to a chosen AKF axis (X/Y/Z) justified cheapest-first, and the chosen axis relieves the bottleneck the **capacity model names** (not a different one).
- Every service on the scale path is stateless **or** carries an explicit, signed-off state-externalization strategy; any sequence/counter mint (e.g. an order/invoice number generator) is confirmed datastore-atomic under concurrent replicas (no in-process shadow).
- Autoscaling (where in scope) declares a leading saturation signal, `min`/`max` bounds (bounded cost), a cooldown window, and scale-in connection-drain + grace period; where not in scope, a static headroom factor against the peak scenario is stated.
- A load-shedding rule (`429` + `Retry-After` / by request class) and backpressure (bounded queue / bulkhead / rate-limiter) are defined for surge beyond the ceiling; no unbounded in-memory buffer on the scale path.
- Every new cross-process hop introduced by the scaling decision has a bounded timeout + retry(backoff/jitter) + fallback/fail-fast; no new unbounded synchronous call.
- The C4 Deployment/Container diagrams compile (diagrams-as-code) and show the replicas/LB/scaling group; every scaling ADR is complete (context · ≥2 options · decision · consequences) and `Driver refs`-traced; one-way-door axes (Y-split, Z partition key) flagged; cost bounded against the budget.
- The "no scaling needed — right-sized single instance + headroom" outcome is a **valid pass** when no load-growth driver is stated; the inverse (scaling machinery with no driver) is flagged as accidental complexity.
- **Gate:** hard oracles green (axis-traced-cheapest-first · stateless-or-externalized · autoscale-bounds-or-headroom · load-shed/backpressure defined · failure-modes on new hops · bottleneck-relieved · diagrams compile · ADRs complete · cost bounded) AND (design) rubric mean ≥ 0.8.

## Output

Write `artifacts/scalability-architecture/report.json` per `shared/report-format.md` — the `drivers[]` block (the `QAS-SCALE-*` scenarios with response measures + priority), findings/risks per `shared/finding-schema.md` (`type:"scalability"`; `oracle:"hard"` for a wrong-axis/out-of-order scaling decision, a silently-stateful clone-path service, missing autoscale bounds or load-shedding, a new unbounded hop, or a decision that does not relieve the modeled bottleneck; `oracle:"soft"` for premature Y/Z scaling or baked-in statefulness against no driver — each with `driver_ref` + `tradeoff` + `alternatives_rejected`), plus the `verification` block (fitness-function results, traceability coverage, hard-oracle results, rubric score) for design mode. Authored design artifacts land in **`architecture/<slug>/`** per `project-architecture-config.md`: the scaling ADR(s) in `adr/NNNN-*.md`, the C4 **Deployment** + Container views in `diagrams/` (diagrams-as-code), the externalized-state + autoscale/load-shed policy folded into `design-doc.md`, and the scale capacity numbers + headroom factor into `capacity-model.md` — never scattered. Open driver questions (an unstated concurrency/tenant/RPS target) are listed explicitly for the controller to resolve — never silently assumed into a scale number.

## Guardrails

Per `shared/guardrails.md`: **propose & document — never write product code, never apply infra.** This flow authors the scaling ADRs, the deployment/topology *design*, the externalized-state plan, and the fitness-function *specs* into `architecture/<slug>/`; it never edits `src/**`/`frontend/**`/`backend/**`, never runs `terraform apply`, never provisions a load balancer / autoscaler / replica, and never touches a live environment — the controller builds it and the operator applies the plan. Ground every scaling decision in a stated load-growth driver — **no driver, no scaling machinery**: if the concurrency/tenant/RPS target is unstated, it is an open question, not a guessed number, and the right answer may be "a single right-sized instance is correct." Reach for the cheapest axis (X / stateless clone) and the established patterns (LB, queue-based load-leveling, bulkhead, rate-limiter, load-shedding) before inventing or before Y/Z. One writer preserves conceptual integrity; the read-only proposer/evaluator never edit the architecture they assess. Record the tradeoff and the rejected alternatives on every scaling decision, never just the winning axis. Cross-reference `workflows/20-data/partitioning-sharding.md` (the Z-axis mechanism — last resort), `workflows/40-cross-cutting/performance-capacity.md` (the bottleneck-naming capacity model this flow consumes), `workflows/40-cross-cutting/availability-resilience.md` (the no-SPOF + failure-mode contract for new hops on the scale path), and `shared/architecture-principles.md` (the senior right-sizing / cheapest-reversible-move habits).
