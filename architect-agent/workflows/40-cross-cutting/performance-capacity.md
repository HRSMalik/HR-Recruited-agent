# performance-capacity

**Group:** 40-cross-cutting  ·  **Runs as:** subagent: ../../agents/reliability-architect.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
Engineer performance **by design**, not by later profiling: for each performance/throughput quality-attribute scenario, allocate a **latency budget** across the call path (every hop, query, serialization, and queue wait gets a millisecond allowance that sums under the response measure), build a **back-of-the-envelope capacity model** that demonstrates — with arithmetic, not assertion — that the design meets its throughput and tail-latency target at *peak*, and set the **USE/RED** metrics as design-time targets so the system is observable against the budget the day it ships. Answers the decision: *given this response measure, where does the time and the load actually go, what is the saturation point, and how much headroom does the design carry?* — not "is it fast?" but "does the math close, and where is it sensitive?"

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment topology (e.g. a web SPA → gateway/reverse-proxy → application service(s) → primary datastore, plus whatever broker/queue exists or doesn't), the scale/SLO seeds (e.g. dashboard/list render < 2s; a heavy async operation such as document extraction < 30s; a critical write/transaction action < 60s — *sourced from the project's BRD/NFRs, never invented*), the expected tenant/account count + concurrent-user expectations (an **open driver** — confirm with the controller; right-size to the stated scale, do not model an arbitrary high RPS absent a driver), and where the architecture package / capacity model live (`architecture/<slug>/capacity-model.md`).
- The **drivers / utility tree** — the prioritized 6-part performance/throughput scenarios from `../00-drivers/quality-attribute-scenarios.md`, each ending in a response measure (a percentile + a time bound + an environment). The `(H,H)` performance leaves (e.g. a primary dashboard render at peak concurrency) get the deepest capacity model; `(L,L)` leaves get a sizing sanity-check only.
- **Target:** the system/subsystem under design or review (a request path end-to-end, e.g. a dashboard GET, a document-processing extract, a payment/settlement action) and the resources it crosses (application worker pool, database connection pool + primary, background-worker pool, gateway/reverse-proxy).
- **Preconditions:** the utility tree exists with measurable performance scenarios (else `status:error` — there is no oracle without a response measure; do not assume a number); the C4 Container view (or the proposed one) is available so the **call path is known** (you cannot budget across hops you can't see); the **environment** per scenario is stated (peak vs average vs degraded) — a budget against average load is not a budget. A scenario whose load/concurrency is unstated → surface as an open driver question, model nothing.

## Oracle (source of truth)
The prioritized performance/throughput **quality-attribute scenarios and their response measures** (`../00-drivers/quality-attribute-scenarios.md`), proven against a capacity model and a per-hop latency budget — never the writer's claim that it "scales" or "is fast." Per the grounding rule (`../../shared/quality-attribute-rubric.md`): no number + no traced driver = a preference, not a finding.

**hard:**
- **Capacity model meets the measure (the math, not an assertion).** Every prioritized performance/throughput scenario has a **back-of-the-envelope capacity model** that *demonstrably* meets its response measure by arithmetic — **Little's Law** (`concurrency L = arrival-rate λ × residence-time W`), `required throughput = QPS × per-request work`, and the powers-of-ten latency numbers (L1/main-memory/SSD/network-round-trip/disk-seek) — applied to the actual call path. A scenario "met" by a bare claim with no model **blocks** (`response_measures_met` hard oracle).
- **Latency budget allocated across the call path.** The response measure is decomposed into a per-hop budget — every hop (gateway/reverse-proxy → application service → database → back), every DB query, every serialization, and every queue/pool wait gets an allocated `ms`, and the allocation **sums to ≤ the response measure** with explicit slack for the tail. A scenario with a single end-to-end number and no per-hop allocation has no traceable budget → blocks.
- **Saturation point + headroom estimated.** The model states the **knee** — the load at which a resource (application worker pool, database connection pool / primary IOPS, background-worker pool) saturates (utilization → 1, queue depth → ∞ per the M/M/1 latency-vs-utilization curve) — and the **headroom** between peak-design load and that knee (e.g. "peak 40 concurrent dashboard loads vs worker-pool knee at ~120 → 3× headroom"). "Scales infinitely" with no estimated saturation point blocks.
- **Tail, fan-out, and peak modeled — not average.** The model is computed at **peak** (the scenario's stated environment), gates on **p99** (not mean), and accounts for **fan-out amplification** — where a request issues N parallel sub-calls, end-to-end latency is the **max** of N, so the effective tail is worse than any single call's p99 (`p99_observed ≈ 1 − (1 − p99_single)^N` exposure). A model that uses the average, ignores the tail, or ignores fan-out is dishonest and blocks.
- **USE/RED defined as design targets.** Every request-handling service carries its **RED** targets (Rate, Errors, Duration-as-percentiles) and every bottleneck resource its **USE** targets (Utilization, Saturation, Errors) — wired to the latency budget so the system is observable against it. A performance scenario with no instrumentation target is unverifiable in production → blocks (hands the SLI definitions to `observability-architecture.md`).
- **Failure-mode + fallacy honored on the path.** Latency-is-zero is not assumed away (`../../shared/fallacies-and-tradeoffs.md` §2): every cross-process call in the budgeted path has a **bounded timeout** that fits inside the budget (an infinite/absent timeout makes the tail unbounded → the budget is meaningless), and load-shedding / queue-based load-leveling is named where peak can exceed the saturation point. A budgeted path with an unbounded call blocks (`failure_modes_defined`).

**soft:**
- **The capacity model is honest** — it reasons about p99 (and fan-out tail), models peak rather than average, states its assumptions (request mix, cache hit-rate, payload size) as explicit inputs, and shows its working so a reviewer can falsify it. A model that quietly picks the favorable number scores low even if it "passes."
- **Performance is designed in where a scenario demands it, not micro-optimized everywhere** (YAGNI; `../../shared/fallacies-and-tradeoffs.md` Performance↔Simplicity). The budget spends effort on the `(H,*)` paths; it does not add a cache/read-model/denormalization to a path with no performance driver — accidental complexity dressed as optimization is penalized.

## Standards & techniques
- **Latency budgets** — decompose the response measure top-down: spend the `ms` across hops, queries, serialization, and queue waits; the sum (plus tail slack) must fit under the target. Ground each hop in the **powers-of-ten latency numbers** (main-memory read ~100ns, SSD random read ~16µs, same-DC network round-trip ~0.5ms, rotational disk seek ~10ms) so an allocation is realistic, not wishful. A hop whose allocated budget is below its physical floor is impossible — flag it.
- **Back-of-envelope capacity estimation** — **Little's Law** (`L = λW`) to convert a throughput + latency target into required concurrency (and thus pool/worker sizing); `required_capacity = peak_QPS × per-request_resource_cost`; the M/M/1 intuition that latency rises hyperbolically as utilization → 1 (so you size to a utilization ceiling, e.g. 70%, *not* 100%, leaving headroom). Always compute at **peak**, with stated assumptions, and round to the order of magnitude — the goal is to prove the design is in the right ballpark and find the bottleneck, not false precision.
- **The USE method** (Brendan Gregg) — per resource: **U**tilization (% busy), **S**aturation (queue depth / waiting work), **E**rrors. Use it to locate the bottleneck resource (the one that saturates first sets the system's ceiling) and to define the resource-side design targets (application worker-pool, database connection-pool, primary IOPS, background-worker pool).
- **RED** (Tom Wilkie) — per request-handling service: **R**ate (req/s), **E**rrors (rate/%), **D**uration (latency **distribution**, gated on p95/p99 — never the mean). RED is the externally-observed user experience and maps directly to the scenario's response measure; USE explains a RED breach.
- **Tail latency & fan-out amplification** — gate on **p99**, not average (a system fast on average can be slow for the user that matters); where one request fans out to N services/queries, the user-observed latency is the **max** of N, so a 1%-tail per call becomes a much larger tail at the aggregate — the cure is hedged requests, fewer fan-out hops, or tighter per-call tails (cite this as the reason to prefer fewer, well-bounded calls per `../../shared/fallacies-and-tradeoffs.md` Metcalfe/fan-out reality).
- **Canonical tradeoffs named** (`../../shared/fallacies-and-tradeoffs.md`): Latency↔Throughput (batching/queueing raises throughput but adds latency), Performance↔Simplicity (caches/denormalization/read-models speed reads but add invalidation + moving parts — only where a driver demands), Cost↔Performance (headroom and lower latency cost money — size to the driver, not the maximum). The budget names the point chosen on each, not just the winner.
- **Cloud-design patterns for the load side** (`../80-patterns/cloud-design-patterns.md`): **queue-based load-leveling** + **competing-consumers** for a heavy async/document-processing path (smooth a spike below the worker saturation point rather than over-provision for peak), **cache-aside** for read-heavy dashboard/list data (defer the hit-rate assumption and stampede protection to `../20-data/caching-strategy.md`), **bulkhead** to stop one saturated path starving another's pool.

## Step sequence
- **audit:** Frame the prioritized performance/throughput scenarios + their environments → for each, reconstruct the **actual call path** from the C4 Container view and walk the latency budget hop-by-hop (is there a per-hop allocation that sums under the measure?) → check the capacity model exists and the **math closes at peak** (Little's Law, QPS×cost) → identify the **bottleneck resource** and whether its **saturation point + headroom** are estimated → check the model gates on **p99** and accounts for **fan-out** → check every budgeted cross-process call has a **timeout inside the budget** → check **USE/RED** targets are defined → emit findings + risks (read-only; authors nothing, edits no source). A scenario with a response measure but no capacity model is a `major+` `performance` finding (`response_measures_met` un-met); an unbounded saturation claim or a budget built on the average is `major`.
- **design:** **Frame** (extract/confirm the performance scenarios + peak environment + assumptions; surface unstated load as an open driver, read-only) → **Explore** (≥2 divergent ways to *meet the budget* as concrete specs — e.g. (a) synchronous read-optimized path with cache-aside + a database projection vs (b) precomputed read-model/materialized dashboard refreshed async; for a heavy document-processing path, (a) inline-synchronous-with-timeout vs (b) queue-based load-leveling + competing-consumers worker pool — each a real spec with its own budget and saturation profile, read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each option's capacity model against the utility tree per ATAM `../../shared/evaluation-method.md`: does the math close at peak? what is the **sensitivity point** — e.g. "p99 is sensitive to cache hit-rate: below 90% the budget blows" — and the **tradeoff point** — e.g. "the async read-model improves read p99 but worsens freshness/consistency"? order-swapped to kill bias; penalize the over-engineered option that meets no extra driver) → **Document** (the ONE writer authors `capacity-model.md` — per-scenario latency budget table, the Little's-Law/QPS arithmetic at peak, the bottleneck + saturation point + headroom, the fan-out/tail analysis, the USE/RED target table — plus an ADR per significant choice (`../../shared/adr-template.md`) and a sequence/deployment diagram of the budgeted path (`../../shared/diagramming-standards.md`); reuses the load-leveling/cache-aside patterns rather than inventing) → **Verify** (a **performance fitness function** spec — an automatable budget assertion, e.g. "dashboard GET p99 ≤ 1.6s at 40 concurrent over a seeded set," handed to `qa-agent`'s `performance-load` to *execute*; the budget arithmetic re-checked; the evaluator re-scores from the artifact) → loop ≤10 or pass.

## Assertions & exit gate
- Every prioritized performance/throughput scenario has a **capacity model** whose arithmetic (Little's Law / QPS×cost at peak) **demonstrably meets** its response measure — modeled, not asserted.
- The response measure is **decomposed into a per-hop latency budget** across the call path; the allocation **sums to ≤ the measure** with explicit tail slack; no hop is budgeted below its physical floor.
- The **bottleneck resource, its saturation point (the knee), and the headroom** to peak-design load are estimated and stated; no "scales infinitely / cost unstated."
- The model is computed at **peak**, gates on **p99**, and accounts for **fan-out amplification** (max-of-N tail), not the average.
- Every request-handling service has **RED** targets and every bottleneck resource has **USE** targets, wired to the budget (SLI definitions cross-handed to `observability-architecture.md`).
- Every cross-process call on the budgeted path has a **bounded timeout that fits inside the budget**; load-shedding / queue-based load-leveling is named where peak can exceed the saturation point.
- Each significant choice is an ADR with context · ≥2 options · decision · consequences (positive AND negative — what the optimization costs on simplicity/consistency/cost) and a traced `Driver ref`.
- **Gate:** hard oracles green (capacity model meets each measure by arithmetic · per-hop budget sums under the measure · saturation point + headroom estimated · p99 + peak + fan-out modeled · USE/RED targets defined · timeouts bounded inside the budget) AND (design) rubric mean ≥ 0.8 (`../../shared/quality-attribute-rubric.md`).

## Output
Write `artifacts/performance-capacity/report.json` per `../../shared/report-format.md` — `drivers[]` (the prioritized performance/throughput scenarios with response measures + priority), `findings[]` per `../../shared/finding-schema.md` (`type:"performance"`; `oracle:"hard"` for a scenario with no capacity model / a budget that doesn't sum / an unestimated saturation point / a model on the average instead of peak / an unbounded call on the budgeted path; each finding carries its `driver_ref` (`QAS-PERF-*`), the traced arithmetic in `evidence`, and the `tradeoff` + `alternatives_rejected`), and for design mode the `verification` block (the latency-budget table result, the capacity-model arithmetic per scenario, the saturation/headroom estimate, the fitness-function spec, and the evaluator's `rubric_score`). Authored artifacts land in `architecture/<slug>/`: the **`capacity-model.md`** (the deliverable — per-scenario budget + sizing arithmetic + saturation/headroom + USE/RED targets), the ADR(s) in `adr/`, the budgeted-path diagram in `diagrams/`, and the performance fitness-function spec appended to `fitness-functions.md` — never scattered. Open driver questions (unstated peak concurrency, tenant/account count) are listed explicitly for the controller to resolve — never silently assumed (`project-architecture-config.md`: right-size, never invent a load number).

## Guardrails
Per `../../shared/guardrails.md`: **propose & document the capacity model — never write product code and never run a load test** (this flow authors the *budget and the fitness-function spec*; `qa-agent`'s `performance-load` *executes* against a non-prod SUT; the controller implements the design). Ground every budget in a **stated** performance scenario — never invent a peak-load or concurrency number; if it's unstated, it's an open driver question, not a guess (`project-architecture-config.md`). **Right-size:** add a cache, read-model, queue, or denormalization **only** where a `(H,*)` scenario's budget demands it — micro-optimizing a path with no performance driver is accidental complexity, flag it rather than ship it. Reach for the established load-side pattern (queue-based load-leveling, cache-aside, bulkhead) before inventing a bespoke mechanism, and prefer the existing stack (per `project-architecture-config.md`) unless a driver-justified ADR demands new infrastructure. The model reasons about **p99 at peak with fan-out**, not the average. Read-only audit authors nothing; one writer authors `capacity-model.md` for conceptual integrity. Cross-reference: `scalability-architecture.md` (the horizontal-scale + AKF-scale-cube model the saturation point feeds), `../20-data/caching-strategy.md` (the cache hit-rate the read-path budget is sensitive to + stampede protection), `observability-architecture.md` (where the USE/RED design targets become live SLIs), and `../../shared/evaluation-method.md` (the ATAM sensitivity/tradeoff-point analysis the evaluator runs over the capacity model).
