# multi-tenancy

**Group:** 40-cross-cutting  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Design **tenant isolation** — choosing silo, pool, or bridge partitioning per driver (security, noisy-neighbor blast radius, cost) — so every tenant's data and compute are bounded, cross-tenant leakage is structurally impossible, and per-tenant cost attribution, onboarding/offboarding, and compliance-erasure paths are defined. Answers the structural decision: *at each tier (data / compute / network), what isolation level does each tenant class require, and what quota/bulkhead bounds the noisy-neighbor risk?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack + deployment target (per the project's stack; a common shape is a service + a primary datastore in a modular monolith, with tenant-scoped RBAC via a tenant-membership field on the user/principal — e.g. `assigned_tenant_ids`); the tenant unit itself (e.g. an organization, a company, an account — named in `project-architecture-config.md`); the compliance regime (PII and/or other regulated data, per the project); the team topology constraint (small team — favor the simplest isolation that satisfies the security driver, not a silo-per-tenant deployment that the team cannot operate).
- The **drivers / utility tree** (`workflows/00-drivers/quality-attribute-scenarios.md`): the prioritized 6-part security and compliance scenarios this decision must satisfy — specifically the tenant-scoping hard driver (a top-level admin role sees all; all other roles scoped strictly to their assigned tenant(s), per `project-architecture-config.md`), the data-erasure/export compliance driver (PII and any other regulated data), and any noisy-neighbor / blast-radius scenario (if not yet stated in the source-of-truth docs, surface as an open driver question — do not invent a response measure). The cost driver must be stated: pool-by-default is the right-sized default for an internal enterprise tool; silo is justified only where a concrete security or isolation driver demands it.
- **Preconditions:** source-of-truth docs read (the BRD / PRD / understanding log named in `project-architecture-config.md`); the RBAC/auth boundary confirmed (the project's role names and their tenant-scoping field, per `project-architecture-config.md`); the data model and datastore collection/table structure known (the tenant-scoped entities — e.g. the core business entity, tenant records, users, audit logs — and which store(s) they live in, per `project-architecture-config.md`); the deployment topology known (per `project-architecture-config.md`). Cross-check with `workflows/40-cross-cutting/security-architecture.md` for the STRIDE controls already in place on the auth boundary, and `workflows/20-data/partitioning-sharding.md` for the partition-key and hot-spot conventions this flow's data-isolation decision must honor.

## Oracle (source of truth)

The tenant-scoping **security quality-attribute scenarios** (each with a response measure), the **cross-tenant data access structural check**, the **noisy-neighbor bound**, and the **per-tenant lifecycle** (onboarding / offboarding / erasure). Never "it's isolated" / "tenants can't see each other" without a traced driver and a structural or measurable proof (the grounding rule, `shared/quality-attribute-rubric.md`).

Illustrative QAS IDs that this flow's oracles are traced to (actual response measures are authored by `workflows/00-drivers/quality-attribute-scenarios.md` from the source-of-truth docs — do not invent numbers here; open measures are surfaced as driver questions):
- **QAS-MT-01** — *cross-tenant data isolation:* an authenticated member-level user of tenant A (source) calls the core API with a valid token for tenant A (stimulus) on a pool-model core-entity collection/table (artifact) under normal load (environment); the system returns only tenant-A records and writes an audit event (response) — **0 cross-tenant records returned, 100% of denied cross-tenant reads audited** (response-measure).
- **QAS-MT-02** — *compliance erasure:* a compliance officer (source) requests full erasure of a decommissioned tenant B (stimulus) across the core-entity, user, and audit-log stores (artifact) under a regulatory deadline (environment); the system enumerates and purges all `tenant_id = B` records (response) — **100% of tenant-B PII/regulated records enumerable and erasable within the stated retention window** (response-measure; the specific window is a driver question if not stated in the source-of-truth docs).
- **QAS-MT-03** — *noisy-neighbor:* a high-volume tenant C (source) bursts API requests beyond its quota (stimulus) on the shared service + datastore pool (artifact) under peak load (environment); tenant C's excess requests are rate-limited and other tenants' p99 latency is unaffected (response) — **per-tenant_id request rate capped at a stated threshold; no cross-tenant latency degradation exceeding the performance QAS bound** (response-measure; exact cap stated in the driver or recorded as an accepted risk if the scale driver classifies this Low/Low).

- **hard:**
  - **Isolation model is driver-chosen** — the silo / pool / bridge tier is selected by naming the security, blast-radius, or cost driver it serves, and documenting the rejected alternatives as an ADR (`shared/adr-template.md`). A model chosen by convention alone (not a driver) fails the `driver_traceability` oracle.
  - **Cross-tenant data access is structurally impossible** — not just policy-policed. At the data tier, every query that touches tenant-scoped data carries a `tenant_id` (or the project's equivalent tenant-membership field, per `project-architecture-config.md`) filter as a structural query invariant — not a caller-convention. If the isolation model is pool (shared collections/tables), the partition predicate must be enforced at the repository/data-access layer so no code path can omit it. A design where cross-tenant access is prevented only by "callers must remember to filter" blocks — the enforcement must be structural, checkable by a fitness function. *Traceable:* a named fitness function asserts that no query on a tenant-scoped collection runs without the tenant predicate.
  - **Noisy-neighbor is bounded** — every shared-compute or shared-data resource path serving multiple tenants has an explicit quota, rate-limit, or bulkhead with a stated threshold (e.g. "no single tenant_id may hold more than 25% of the connection pool at any instant, enforced at the application-layer rate-limiter"). "We trust the tenants not to flood" is not a bound — the threshold must be stated or the driver for omitting it must be stated. If the scale driver (per `project-architecture-config.md`: internal enterprise tool, not consumer-scale) means noisy-neighbor is Low/Low priority, record that disposition explicitly as an accepted risk in the register rather than silently ignoring it.
  - **Per-tenant data is exportable and erasable** — a compliance path must exist: given a `tenant_id`, the system can enumerate all PII/regulated records belonging to that tenant (across the core entity, users, and audit-log stores) and either export or erase them. No collection may store tenant data in a form that makes per-tenant enumeration structurally impossible (e.g. a shared aggregate with no tenant discriminator). *Traceable:* the data model audit confirms a `tenant_id` discriminator on every tenant-scoped collection, and the `data-governance-lifecycle.md` erasure path is referenced. This is a hard gate where a compliance regime applies (`quality-attribute-rubric.md` → *Data lifecycle lawful*).
  - **Tenant context threads consistently** — the tenant identifier (`tenant_id` or the decoded tenant-membership claim from the auth token, per `project-architecture-config.md`) is extracted at the auth boundary (the API gateway / service middleware layer) and propagated to every data-access call as a first-class argument, never re-derived from request body or caller convention at the repository layer. *Structural check:* a static-analysis fitness function verifies that tenant-scoped data-access functions require the tenant argument (not optional / nullable).
  - **Onboarding/offboarding is defined** — adding a new tenant and decommissioning one are defined procedures: what records are created/deleted, whether data is exported before erasure, how the role/user assignments are cleaned up, and whether the tenant's quota entry is removed. An undefined offboarding path means tenant data persists indefinitely — a compliance gap under retention obligations.

- **soft:**
  - **Isolation tier matches sensitivity + cost target** — pool (shared collections, tenant-discriminated) is the correct default for an internal enterprise tool at this scale; silo (per-tenant database or schema) is justified only where a concrete security driver (e.g. a contractual data-residency or legal hold requirement from a specific tenant) demands it. A design that defaults to silo for all tenants for "extra safety" without a driver is accidental over-engineering (penalized by the simplicity rubric). The bridge model (silo for high-sensitivity tenants, pool for standard) is a valid response if the driver set is mixed — document the classification criteria.
  - **Tenant context is threaded without coupling** — the mechanism for propagating `tenant_id` (e.g. a request-scoped context object injected via the framework's dependency-injection mechanism, not a global/threadlocal) is consistent across all modules; it does not create a hidden coupling channel or make unit testing hard. Conceptual integrity (Brooks): the same isolation mechanism is used uniformly — not three different ways to scope a query across different routes.

## Standards & techniques

- **Silo / pool / bridge isolation models** — the three canonical multi-tenancy tiers (SaaS Architecture Playbook / AWS SaaS Lens):
  - **Silo (dedicated):** one database / schema / deployment per tenant. Maximum blast-radius isolation; highest cost and operational overhead; justified when a contractual, legal, or security driver demands it (e.g. "Tenant A's data must never share a storage node with any other tenant").
  - **Pool (shared):** all tenants share the same collections/tables; rows/documents are discriminated by a `tenant_id`. Lowest cost; highest operational simplicity; correct default for an internal tool at this scale. Cross-tenant leakage is the primary risk — mitigated by structural enforcement at the data-access layer, not caller convention.
  - **Bridge (mixed):** high-sensitivity tenants get silo; standard tenants share a pool. Requires a tenant-classification scheme and a routing layer that directs queries to the right store. More complex to operate; justified when the driver set is genuinely heterogeneous.
- **Tenant isolation dimensions** — isolation must be designed at three tiers independently:
  - *Data isolation:* partition key (`tenant_id`) enforced structurally at the repository layer; index per tenant discriminator for query performance; per-tenant export/erasure path.
  - *Compute isolation:* rate-limiting and bulkhead per tenant at the API layer (service middleware / API gateway); no shared thread-pool starvation path across tenants.
  - *Network isolation:* relevant at silo level (VPC-per-tenant) or when regulatory data-residency demands it; not required at pool level for an internal tool — record this as an accepted risk if confirmed out of scope.
- **Noisy-neighbor mitigation** — the bulkhead pattern (`workflows/80-patterns/resilience-patterns.md`) applied per tenant: rate-limit per `tenant_id` at the API boundary (token-bucket or sliding-window with a stated threshold); connection-pool quota per tenant at the datastore layer if pool exhaustion is a risk. The Cloud Design Pattern catalog's **Throttling** and **Bulkhead** patterns are the mechanisms; cite them by name in the ADR.
- **Cross-tenant-leak threat** — STRIDE Spoofing + Information Disclosure at the data layer: an attacker (or a bug) presents a `tenant_id` they do not own, or omits the predicate and gets back all tenants' records. Defense: (1) authn/authz at the API boundary validates the token's tenant claims before any DB call; (2) the repository layer enforces the predicate structurally — not via caller discipline. Cross-reference `workflows/40-cross-cutting/security-architecture.md` STRIDE analysis for the data-access trust boundary.
- **Per-tenant data partitioning** — coordinate with `workflows/20-data/partitioning-sharding.md` for the choice of `tenant_id` as the partition key: verify it avoids hot-spots (a dominant tenant_id skewing load), supports range/hash distribution, and allows per-tenant index creation for export/erasure enumeration.
- **AKF scale cube** — multi-tenancy relates directly to the Y-axis (functional decomposition) and Z-axis (data partitioning by customer/tenant) of the AKF scale cube. A pool model occupies the Z-axis (same service, data split by tenant key). A silo model adds Y-axis decomposition per tenant. Right-size: for an internal-enterprise-scale system, Z-axis pool partitioning is the correct default; Y-axis silo is a one-way-door decision (`adr-template.md` one-way-door flag) with significant operational cost.
- **Tenant onboarding / offboarding** — define as a first-class operation with: (a) idempotent provisioning (creating a tenant record + seeding roles is safe to re-run), (b) per-tenant cost attribution (track storage and request counts by `tenant_id` for internal chargeback or capacity planning), (c) offboarding with data export before erasure (right-to-erasure path per `workflows/20-data/data-governance-lifecycle.md`), and (d) cleanup of orphaned user assignments and quota entries.
- **CAP / PACELC** — the consistency model for tenant-scoped writes (e.g. creating a core-entity record under `tenant_id X`) must be stated; for a single-primary pool model, the write path is typically strong-consistent within a session but offers no cross-replica guarantee until a replica-set ADR exists. State this per write path explicitly (`fallacies-and-tradeoffs.md` CAP/PACELC), against the storage choice recorded in `project-architecture-config.md`.
- **MADR** — every silo/pool/bridge decision is an ADR with context · options · decision · consequences (positive AND negative) · status, per `shared/adr-template.md`. The isolation model is a **one-way-door decision** at the data tier: migrating from pool to silo post-launch requires a full data migration.

## Step sequence

- **audit:** Frame the tenant model (who is a tenant, what is the discriminator) → walk every data-access path in the existing codebase for the structural `tenant_id` predicate (Grep/Glob read-only) → check the three isolation dimensions (data / compute / network) against the drivers → verify noisy-neighbor mitigation exists and is bounded → check onboarding/offboarding is defined and the erasure/export path is enumerable → emit findings per `finding-schema.md` (type: `security` / `privacy` / `compliance` / `evolvability`; hard oracle for each structural failure; soft for isolation-tier mismatch). Read-only, authors nothing.

- **design:** Frame (confirm drivers: company-scoping QAS, compliance/erasure QAS, noisy-neighbor QAS or accepted risk, cost target; read-only) → Explore (≥2 divergent isolation directions: e.g. Direction A — pool with structural repo-layer enforcement + bulkheads; Direction B — bridge with silo for contractual tenants + pool for standard; each is a concrete spec covering data / compute / network / onboarding / lifecycle) → Evaluate (ATAM skeptic scores each vs utility tree + hard oracles, order-swapped; surfaces the silo↔pool tradeoff point: blast-radius isolation vs operational cost + migration risk; picks a winner and grafts the best ideas from the runner-up) → Document (one writer: ADR for the isolation model, C4 Deployment view showing tenant data partitioning as diagrams-as-code, fitness-function spec for the structural predicate check, the per-tenant quota policy, and the onboarding/offboarding procedure into `architecture/<slug>/`) → Verify (fitness-function spec checked for compilability; traceability matrix confirms every company-scoping QAS traces to a decision; STRIDE cross-tenant-leak control confirmed in the threat model; hard oracles re-scored by evaluator) → loop ≤ 10 or pass.

## Assertions & exit gate

- The silo/pool/bridge model is chosen and every driver it serves is named; rejected alternatives are recorded with explicit reasons (ADR complete per `shared/adr-template.md`; one-way-door flag set for the data-tier decision).
- Cross-tenant data access is structurally impossible at the data-access layer — a named fitness function asserts the `tenant_id` predicate is required on every tenant-scoped collection query; no query path omits it by caller convention alone.
- Noisy-neighbor is bounded: a stated quota/rate-limit threshold per `tenant_id` exists at the API and (if relevant) connection-pool layer, **or** the noisy-neighbor driver is confirmed Low/Low priority and recorded as an accepted risk in the risk register with the scenario it threatens.
- Per-tenant data is exportable and erasable: a `tenant_id` discriminator exists on every tenant-scoped collection; the erasure path can enumerate all records for a given tenant (`data-governance-lifecycle.md` erasure procedure referenced).
- Tenant context is extracted at the auth boundary and propagated to all data-access calls via a consistent mechanism (not re-derived at the repository layer); a fitness function or static-analysis rule makes the tenant argument mandatory on scoped data-access functions.
- Onboarding and offboarding procedures are defined: idempotent provisioning, per-tenant cost attribution point, and erasure-before-deletion are all specified.
- The consistency model for tenant-scoped write paths is stated (CAP/PACELC position named per write path).
- The C4 Deployment diagram (or Container view) shows the tenant data partitioning and compiles without error (diagrams-as-code, per `shared/diagramming-standards.md`).
- Every QAS the isolation model is designed to satisfy has a met response measure (proven structurally or by capacity model — not by assertion).
- **Gate:** hard oracles green (driver-traced isolation model, structural cross-tenant predicate, bounded noisy-neighbor or accepted risk, erasure path enumerable, tenant context consistent, ADR complete, diagram compiles, QAS response measures met) AND (design) rubric mean ≥ 0.8.

## Output

Write `artifacts/multi-tenancy/report.json` per `shared/report-format.md` — `flow:"multi-tenancy"`, `mode:"audit|design"`, `status`, `drivers[]` (the tenant-scoping, compliance, noisy-neighbor QAS IDs with response measures and priorities), `summary`, `findings[]` per `shared/finding-schema.md` (type `security` / `privacy` / `compliance` / `evolvability` / `cost` / `risk`; `oracle:"hard"` for cross-tenant structural failures, un-traced isolation decisions, missing erasure path, unbounded noisy-neighbor; `oracle:"soft"` for isolation-tier mismatch vs cost target, inconsistent tenant-context threading), `decisions[]` (the ADR for the silo/pool/bridge choice), and — in design mode — the `verification{}` block (traceability coverage, fitness-function compile result, hard-oracle results, rubric score).

Design artifacts land in `architecture/<slug>/` (per `project-architecture-config.md`):
- `adr/NNNN-tenant-isolation-model.md` — the silo/pool/bridge decision (context · options · decision · consequences · one-way-door flag)
- `diagrams/tenant-partitioning.{puml|md}` — C4 Container or Deployment view showing `tenant_id` partitioning (diagrams-as-code, compiles)
- `fitness-functions.md` — structural check: "every tenant-scoped query carries a `tenant_id` predicate" (assertable in CI via static analysis or an integration test that queries without the predicate and expects rejection)
- `tradeoffs.md` — the silo↔pool tradeoff point (blast-radius isolation vs operational cost + migration risk + silo one-way-door), the sensitivity point (isolation level is sensitive to whether any tenant carries a contractual data-residency obligation)

## Guardrails

Per `shared/guardrails.md`: propose and document — never write product code, never implement the RBAC predicate in `src/**` or `backend/**`, never run a migration. The isolation model ADR and the fitness-function *specification* are the deliverables; the controller writes the enforcement code.

Ground every recommendation in a named driver: "pool because the cost driver and team-topology constraint demand it, not because it is simpler in the abstract" — the simplicity that matters is the simplicity the *drivers allow*. A silo recommendation with no security or contractual driver is accidental complexity and must be flagged.

Reuse known patterns before inventing: the Throttling + Bulkhead patterns from the cloud-design-pattern catalog cover noisy-neighbor; the STRIDE Information Disclosure control at the data-access boundary covers cross-tenant leak; the right-to-erasure procedure from `workflows/20-data/data-governance-lifecycle.md` covers the compliance erasure path (e.g. GDPR/CCPA or the regime named in `project-architecture-config.md`). Do not re-derive what the sibling flows already specify — cross-reference and compose.

The isolation model at the data tier is a **one-way-door decision** — flag it as such in the ADR and in any recommendation, so the controller understands the reversal cost (a full data migration) before committing. One writer (`solution-architect`) authors all artifacts in this flow's design mode; domain specialists (`security-architect`, `data-architect`) advise read-only and verify their dimension.
