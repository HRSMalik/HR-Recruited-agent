# security-architecture

**Group:** 40-cross-cutting  ·  **Runs as:** subagent: ../.claude/agents/security-architect.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
Make security a **structural property of the architecture, not a feature bolted on at the end** — threat-model every element and trust boundary with **STRIDE**, apply **zero-trust** (never trust the network; verify every call) and **defense-in-depth**, and place **authentication + authorization at every boundary** with **externalized secrets** and **least privilege**. Where the design has an **LLM / RAG / agent component**, STRIDE alone under-covers the attack surface — this flow additionally sweeps the **OWASP Top 10 for LLM Applications (2025)** and the **CSA MAESTRO 7-layer agentic framework**, and references **MITRE ATLAS** for ML-specific technique coverage, per `../shared/agentic-threat-taxonomy.md`. Answers: *for each trust boundary, which threats apply — classical and, where AI is present, LLM/agentic — what control stops each, and what residual risk is accepted — traced to a security/compliance driver?* The deliverable is a threat model + control map + ADRs, not "it's secure."

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + auth model (e.g. a signed-token scheme over the API framework, password hashing, field-level encryption, the datastore's replication topology, same-origin SPA+API behind an edge proxy — per the project's actual stack), the compliance/security regime (e.g. **PII + financial or other regulated data**, as applicable), the RBAC scoping rule (e.g. a top-level admin role sees all; other roles scoped to their assigned org/tenant + assigned manager/member), any **BE↔FE role-name skew** the project has (naming inconsistencies between backend and frontend role identifiers), the secrets in play (signing secret, encryption key, datastore creds), and where the threat model + ADRs live (`architecture/<slug>/threat-model.md`, `adr/`).
- The architecture **drivers** — the prioritized quality-attribute scenarios from `../00-drivers/quality-attribute-scenarios.md` (the utility tree). Security drivers must be *security QAS* with a measurable response: e.g. `QAS-SEC-01` "an authenticated member of tenant A (source) requests a record of tenant B (stimulus) at the core API (artifact) under normal load (environment); the system denies the read and writes an audit event (response) — **0 cross-tenant reads succeed, 100% denied-access attempts audited** (response-measure)." If a security target (residency, retention, the applicable regulatory regime) is **unstated**, surface it as an open driver question — never invent a control's strength or a compliance obligation.
- Target: the system/subsystem trust-boundary map under design or review — the SPA↔API edge, the API↔datastore boundary, the auth/RBAC boundary, the audit-log write path, any future bounded contexts, and any **deferred integration seam** (e.g. a planned agent layer, a future payment integration) named in `project-architecture-config.md` (design the boundary's authn/authz now even though the component itself is deferred). Where any element **is** or **calls** an LLM/RAG/agent component (including a deferred-but-designed seam), that component and its boundaries are additionally in scope for the AI-specific sweep below.
- Preconditions: the utility tree exists (a flow with no security drivers downgrades scope and says so — you cannot size controls without the data sensitivity + the threats they protect); the C4 Container view + the trust boundaries from `../00-drivers/context-scoping` are available (STRIDE is *per element / per boundary* — no boundary map, no threat model). Where an LLM/RAG/agent component is in scope, `../shared/agentic-threat-taxonomy.md` is read before the sweep — it is the taxonomy this flow applies, not re-derives.

## Oracle (source of truth)
The **STRIDE-per-trust-boundary threat model** + the security QAS response-measures + the **OWASP ASVS** verification requirements — never "it looks secure" or a vendor's "best practice." A control with no threat it stops and no driver it serves is theater, not security; a threat with no control and no accepted-risk entry is a silent hole.

- **hard** (deterministic — traceable / structural):
  - **Every trust boundary has authn + authz.** Each boundary in the threat model (SPA→API, API→datastore, service→service, edge→BFF, the agent seam) carries an authentication mechanism *and* an authorization decision point. A boundary that authenticates but never authorizes (or vice-versa) blocks. (This is the **Trust boundary secured** hard oracle in `../shared/quality-attribute-rubric.md`.)
  - **Every STRIDE category, per element/boundary, has a control or an accepted-risk entry.** For each element the model enumerates Spoofing/Tampering/Repudiation/Information-disclosure/Denial-of-service/Elevation-of-privilege; each cell is either mitigated by a named control or recorded as an explicit, signed-off accepted risk in the register. An empty STRIDE cell (threat neither controlled nor accepted) blocks.
  - **AI components additionally clear the OWASP-LLM + MAESTRO + ATLAS sweep — this is part of the Trust boundary secured oracle, not a separate gate.** Where the design has an LLM, a RAG pipeline, or an agent, STRIDE alone does not cover the boundary: it must **also** clear the **OWASP Top 10 for LLM Applications (2025, published Nov 2024)** — with named-priority focus on **LLM01 prompt injection** (every ingestion path to the model is trust-tiered; no untrusted content enters the instruction plane unsanitized), **LLM06 excessive agency** (every tool grant is minimum-required, irreversible-write actions gated behind a human-in-the-loop token), **LLM07 system-prompt leakage** (output scanned for prompt fingerprints; no secret/PII placed in the system prompt), and **LLM08 vector/embedding weaknesses** (ABAC enforced at the retrieval layer itself, cross-tenant retrieval verified empty) — plus the full LLM02–LLM05/LLM09–LLM10 set; the **CSA MAESTRO 7-layer sweep** (Foundation Model → Data → Agent Frameworks → Deployment Infra → Observability → Security Controls → Agent Ecosystem), walked layer-by-layer against the component diagram; and, for each ML component, the **MITRE ATLAS** tactic sweep (technique IDs named, not just referenced). An in-scope OWASP-LLM item, MAESTRO layer, or ATLAS-applicable ML component with no named control and no accepted-risk ADR blocks, exactly like an empty STRIDE cell. Full detail + the per-item design controls live in `../shared/agentic-threat-taxonomy.md`; the dedicated deep-dive flow (multi-agent topologies, red-team probe verification, the coverage-matrix artifact) is `../90-specialized/agentic-threat-model.md` — this flow performs the sweep as part of its trust-boundary pass and defers to that flow for full agentic-topology depth when the system *is* substantially agentic.
  - **Secrets are externalized — never in code, config-in-repo, or any diagram.** The signing secret, encryption key, datastore creds (and every future credential) are injected via env / a vault / KMS. A secret literal in source, a committed `.env`, or a credential printed in a diagram/ADR blocks (`grep`-checkable). Crypto is *named and current* — bcrypt/argon2 for passwords, an authenticated cipher (e.g. AES-256-GCM) for field encryption, TLS in transit; the token-algorithm choice (e.g. HS256 vs RS256) is a stated, driver-traced choice, not an accident.
  - **Service-to-service / cross-process calls are authenticated (zero-trust).** No call is trusted by network position alone — internal calls carry a verifiable identity (mTLS / signed token / workload identity), not "it's inside the VPC so it's fine." A call that trusts an un-verified caller identity violates fallacy #4 (*the network is secure*, `../shared/fallacies-and-tradeoffs.md`) and blocks.
  - **Least privilege is applied to every identity.** Every human role and every machine/service identity has the *minimum* grants its function requires (the RBAC org/tenant-scoping is enforced server-side at the data boundary, not just hidden in the FE; the API's datastore credential is scoped, not root; any agent-seam identity is read-scoped to exactly what it needs). A wildcard/over-broad grant with no driver demanding it blocks.
  - **Authorization is enforced at the trust boundary that owns the data, server-side.** Tenant/org-scoping and role checks live at the API/data boundary; a control that exists only in the SPA (client-side hide) is *not* a control (trivially bypassed) — it blocks as an Elevation/Information-disclosure hole. Any **BE↔FE role-name skew is reconciled** so an authz check isn't silently bypassed by a name mismatch.
- **soft** (craft, rubric-scored after hard oracles green):
  - **Security is structural** — threat-modeled at *design* time per boundary, with defense-in-depth (layered controls so no single control failing exposes the asset), not a perimeter-only or bolt-on posture.
  - **Proportionate to threat × data sensitivity** — control strength matches the asset (PII + financial/payment data ⇒ stronger; an internal audit-read ⇒ lighter). No gold-plating a low-value path; no under-protecting a regulated one. (Right-sizedness, `../shared/quality-attribute-rubric.md`.)
  - **No trust assumed from network position** — zero-trust runs through the whole design coherently (conceptual integrity), not zero-trust at the edge and implicit trust everywhere behind it.
  - **AI guardrail depth matches autonomy × data sensitivity** — a read-only, non-agentic LLM feature over public data warrants a lighter OWASP-LLM/MAESTRO control set than a tool-equipped agent with write access to PII/financial records; guardrails (prompt sanitization, output scanning, HITL gates) are designed in as first-class components on the C4 diagram, not described as future work.

## Standards & techniques
- **STRIDE** (Microsoft threat-modeling taxonomy) — per element / per data-flow / per trust boundary: **S**poofing (→ authentication), **T**ampering (→ integrity: signing, hashing, AEAD, input validation), **R**epudiation (→ non-repudiation: audit logging, signed events), **I**nformation disclosure (→ confidentiality: encryption in transit + at rest, least-privilege reads, field encryption), **D**enial of service (→ availability: rate-limiting, quotas, timeouts, load-shedding — cross-link `availability-resilience.md`), **E**levation of privilege (→ authorization: RBAC/ABAC, server-side enforcement, deny-by-default). The mapping STRIDE→control class is the backbone of the threat model.
- **The trust-boundary map** — draw the boundaries (C4 Container view + a security perimeter overlay per `../shared/diagramming-standards.md`: dashed boundary boxes). A trust boundary is any place data/identity crosses from one trust level to another (browser→edge, edge→service, service→datastore, service→external/agent). STRIDE is applied *at and across* each boundary, not globally.
- **OWASP Top 10 for LLM Applications (2025, published Nov 2024)** — the application-layer vulnerability taxonomy for any in-scope LLM/RAG/agent component, applied *in addition to* STRIDE, not instead of it: LLM01 Prompt Injection, LLM02 Sensitive Information Disclosure, LLM03 Supply Chain, LLM04 Data/Model Poisoning, LLM05 Improper Output Handling, **LLM06 Excessive Agency**, **LLM07 System Prompt Leakage**, **LLM08 Vector/Embedding Weaknesses**, LLM09 Misinformation, LLM10 Unbounded Consumption. Full mechanism/control detail per item lives in `../shared/agentic-threat-taxonomy.md` Part 1 — this flow sweeps it, not re-derives it.
- **CSA MAESTRO** (Multi-Agent Environment Security Threat and Risk Ontology, Cloud Security Alliance) — the 7-layer agentic-system decomposition (L1 Foundation Model, L2 Data, L3 Agent Frameworks, L4 Deployment Infrastructure, L5 Observability & Monitoring, L6 Security Controls, L7 Agent Ecosystem), walked layer-by-layer against the component diagram so coverage doesn't stop at the classical application boundary (an agentic STRIDE-only sweep misses L1–L3 and L7). Detail per `../shared/agentic-threat-taxonomy.md` Part 2.
- **MITRE ATLAS** (Adversarial Threat Landscape for AI Systems) — referenced, not restated, for ML-specific technique-level coverage (model inversion, training-data poisoning, model-extraction, adversarial examples) on any ML component (foundation model, embedding model, reranker, classifier/guardrail). Cite the ATLAS technique ID per finding. Detail per `../shared/agentic-threat-taxonomy.md` Part 3; the full multi-agent-topology deep-dive (red-team probe verification, the OWASP×MAESTRO×ATLAS coverage-matrix artifact) is owned by `../90-specialized/agentic-threat-model.md`.
- **Zero-trust architecture** (NIST SP 800-207) — *never trust, always verify*: no implicit trust from network location; every request authenticated + authorized + (where possible) encrypted; verify identity, device/workload, and policy on **every** call, not once at the edge. Per-request authz, short-lived credentials, micro-segmentation.
- **Defense-in-depth** — layered, independent controls (edge authn + service authz + data-boundary scoping + field encryption + audit) so a single control's failure doesn't expose the asset. Each layer named and traced to the STRIDE cell it covers.
- **Authentication** at every boundary — **OIDC/OAuth2** (user identity; where an SSO seam is mock-now/real-later per config, design the boundary for the real OIDC flow), **mTLS / workload identity** (service-to-service), API keys/tokens at the edge. State the mechanism per boundary.
- **Authorization** at every boundary — **RBAC** (role → permission) and/or **ABAC** (attribute/policy-based: e.g. *org/tenant-scoping* is ABAC — `user.assigned_org_ids ∋ resource.org_id`), enforced **server-side, deny-by-default**, at the boundary owning the data. Externalize policy where it grows (policy-as-code) rather than scattering `if role ==` checks.
- **Secrets management** — vault / KMS / env-injected; **never in code, repo config, logs, or diagrams**; rotation + least-scope; encryption keys (any field-level encryption key) managed and rotatable, not hardcoded. Passwords hashed (bcrypt/argon2), tokens signed, transport encrypted (TLS).
- **Least privilege** (Saltzer & Schroeder) — minimal grants per identity (human and machine); scoped DB credentials; deny-by-default; just-enough, just-in-time.
- **OWASP Top 10 + OWASP ASVS** as the verification checklist — walk the Top-10 risk classes (broken access control, crypto failures, injection, insecure design, security misconfiguration, vulnerable components, identification/authn failures, software/data-integrity failures, logging/monitoring failures, SSRF) against the design; use **ASVS** as the leveled requirement set (L1/L2/L3 by data sensitivity — PII+financial ⇒ ≥ L2) so coverage is a checklist, not a vibe.
- **The 8 fallacies** — especially **#4 *the network is secure*** (drives zero-trust service-to-service authn) — per `../shared/fallacies-and-tradeoffs.md`. Pair Postel's-law leniency with **strict schema validation at trust boundaries** (input is hostile by default).

## Step sequence
- **audit:** Frame the security drivers (the security QAS + the data-sensitivity classes + the compliance regime) → draw/confirm the trust-boundary map from the C4 Container view → run **STRIDE per element/boundary**, filling each cell with the control present *or* the gap → **where an element is or calls an LLM/RAG/agent component, additionally sweep OWASP-LLM01–10 + walk MAESTRO L1–L7 + run the MITRE ATLAS tactic sweep per ML component** (per `../shared/agentic-threat-taxonomy.md`), filling each item with its control, "out of scope + reason," or an accepted-risk flag → check the hard structural set (authn+authz at every boundary, secrets externalized via `grep` over source/config/diagrams, service-to-service calls authenticated, least privilege per identity, server-side authz enforcement, any BE↔FE role-skew reconciled) → walk **OWASP Top-10 (web) / ASVS** as a checklist → emit findings (`type:"security"`, `oracle:"hard"` for an empty STRIDE cell / an uncovered in-scope OWASP-LLM item or MAESTRO layer / a secret-in-code / a client-only authz check / an unauthenticated internal call) + a **risk register** of accepted/residual risks. Read-only — authors nothing, edits no source.
- **design:** **Frame** (extract/confirm the security QAS + data-sensitivity + compliance drivers, the trust-boundary map, and whether any element is or calls an LLM/RAG/agent component; read-only) → **Explore** (N divergent control topologies as concrete specs — e.g. *gateway-centralized authn+authz at a BFF* vs *service-local authz with a shared policy module* vs *sidecar/ACL-mediated* — each with its STRIDE coverage + tradeoff, not the same model recolored; where AI components are in scope, each topology also states where its guardrails (prompt sanitization, output scanning, HITL gates) sit in the C4 topology; read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each vs the security QAS + the hard oracles using ATAM; finds the **sensitivity points** — e.g. "confidentiality is *sensitive* to where the org-scope check sits" or "a centralized guardrail sidecar is a SPOF on the agent path" — and **tradeoff points** — e.g. "centralized edge authz simplifies but couples; a shared-secret token scheme is simple but can't separate sign/verify trust" — order-swapped to kill bias; picks a winner, grafts runner-up ideas) → **Document** (the ONE writer — `solution-architect`, advised read-only by `security-architect` — authors `threat-model.md` (the STRIDE matrix per boundary, **extended with the OWASP-LLM × MAESTRO × ATLAS coverage matrix where AI components are in scope** — full artifact detail owned by `../90-specialized/agentic-threat-model.md` when the system is substantially agentic), the control map, the trust-boundary overlay diagram, and one **MADR ADR per significant decision** (auth model, token algorithm, where authz is enforced, secrets strategy, and — where applicable — guardrail topology + inter-agent authn + HITL gate policy) per `../shared/adr-template.md`; reuses named patterns — *gateway-offloading*, *ambassador/sidecar*, *valet-key*, *federated-identity* from the cloud-design-pattern catalog — before inventing) → **Verify** (the hard gates below run as **fitness functions**: a `secrets-not-in-repo` scan, a `every-boundary-has-authz` traceability check across the boundary map, a `no-client-only-authz` check, a STRIDE-cell completeness count, an ASVS-L2 checklist pass, and — where AI components are in scope — the OWASP-LLM/MAESTRO/ATLAS coverage-matrix completeness check; the evaluator re-scores from the authored artifacts) → loop ≤10 or pass.

## Assertions & exit gate
- Every trust boundary in the model has a named **authentication** mechanism **and** an **authorization** decision point (server-side, deny-by-default); any BE↔FE role-name skew is reconciled so no authz check is bypassed by a name mismatch.
- Every **STRIDE** category for every element/boundary is either mitigated by a named control or recorded as an explicit, signed-off **accepted risk** — no empty cell.
- **Where an element is or calls an LLM/RAG/agent component**, the **OWASP Top 10 for LLM Applications (2025)** is fully swept (LLM01–LLM10, each covered / out-of-scope-with-reason / accepted-risk), the **CSA MAESTRO** L1–L7 layers are walked against the component diagram, and every **MITRE ATLAS**-applicable ML component has its tactic sweep recorded with technique IDs — per `../shared/agentic-threat-taxonomy.md`; an uncovered in-scope item is treated exactly like an empty STRIDE cell.
- **Secrets are externalized** — a repo/config/diagram scan finds no secret literal (signing secret, encryption key, datastore creds, any token); crypto is named + current (bcrypt/argon2 passwords, authenticated encryption at rest, TLS in transit), and the token-algorithm choice (e.g. HS256 vs RS256) is a traced ADR.
- **Service-to-service / cross-process calls are authenticated** — no call trusted by network position; internal identity is verifiable (mTLS / signed token / workload identity). Fallacy #4 honored.
- **Least privilege** holds for every human role and machine identity (RBAC org/tenant-scoping enforced at the data boundary; DB credential scoped not root; any agent-seam identity read-scoped); no un-driver-justified wildcard grant.
- **OWASP ASVS** at the level the data sensitivity demands (PII + financial or other regulated data ⇒ ≥ L2) walks clean; the Top-10 access-control + crypto + injection + logging classes each have a traced control.
- Every security decision is a complete **ADR** (context · ≥2 options · decision · positive **and** negative consequences) traced to a security `QAS-SEC-*`; the threat-model + boundary diagram **compile/render** (`../shared/diagramming-standards.md`).
- **Gate:** all hard oracles green (authn+authz per boundary · every STRIDE cell controlled-or-accepted · **OWASP-LLM/MAESTRO/ATLAS swept and controlled-or-accepted where AI components are in scope** · secrets externalized · service-to-service authenticated · least privilege · server-side authz · ASVS-level clean · ADRs complete · diagram compiles) AND (design) evaluator rubric mean ≥ 0.8.

## Output
Write `artifacts/security-architecture/report.json` per `shared/report-format.md` — findings/risks per `finding-schema.md` (`type:"security"` or `"privacy"`/`"compliance"`; `oracle:"hard"` for an empty STRIDE cell, an uncovered in-scope OWASP-LLM item / MAESTRO layer / ATLAS-applicable ML component, a secret in code/config/diagram, a client-only or missing authz check, an unauthenticated internal call, or an over-broad grant; each finding carries its `driver_ref` to a `QAS-SEC-*`, the `iso25010` characteristic *Security ›* refinement, the STRIDE category **or, for an AI-component finding, the OWASP-LLM ID / MAESTRO layer / ATLAS technique ID**, the `tradeoff` (what the control costs — latency/coupling/operational surface), and the `alternatives_rejected`), plus the verification block (STRIDE-cell coverage count, the secrets-scan result, the every-boundary-has-authz traceability, the ASVS-level result, **the OWASP-LLM × MAESTRO × ATLAS coverage-matrix result where AI components are in scope**, hard-oracle results, rubric score). The **risk register** lists every accepted/residual risk with likelihood×impact, disposition (accept/mitigate/monitor), and the `QAS-SEC-*` it threatens — *disposition is proposed; the final accept call is the controller's/human's.* The authored security artifacts — `threat-model.md` (STRIDE matrix per boundary, **extended with the OWASP-LLM/MAESTRO/ATLAS matrix where AI components are in scope**), the trust-boundary diagram, the control map, and the security ADRs — land in `architecture/<slug>/` (per `project-architecture-config.md`), never scattered. Cross-link **`privacy-compliance-architecture.md`** (data residency / retention / erasure / regulatory regime — security controls *the access*; privacy governs *the data lifecycle*), **`../30-integration/api-gateway-bff.md`** (the edge is where centralized authn / rate-limit / token-translation lives), **`../50-infrastructure/networking-architecture.md`** (network segmentation / mTLS / zero-trust transport are the infra realization of these boundaries), **`../shared/agentic-threat-taxonomy.md`** (the OWASP-LLM / MAESTRO / ATLAS / NIST-AI-600-1 catalog this flow sweeps against for any AI component), and **`../90-specialized/agentic-threat-model.md`** (the dedicated deep-dive flow for multi-agent topologies, inter-agent authn, HITL gate policy, and red-team probe verification — hand off there when the system is substantially agentic rather than carrying one AI-touched boundary).

## Guardrails
Per `shared/guardrails.md`: **propose & document — never write product code, never apply infra, never touch a live system or a real credential.** The fleet authors the threat model + control map + ADRs + the boundary diagram into `architecture/<slug>/`; the *controller* implements the controls and the *operator* configures secrets/network — the agent produces the plan, not the running defense. **Ground every control in a driver** — no control without a STRIDE threat (or, for an AI component, an OWASP-LLM/MAESTRO/ATLAS threat) it stops *and* a `QAS-SEC-*` it serves (a control no threat demands is theater; flag it, don't ship it); no threat without a control or an accepted-risk entry. **Reuse named patterns** (gateway-offloading, federated identity/OIDC, valet-key, ambassador/ACL, sidecar mTLS) before inventing a bespoke mechanism — a named pattern carries a known contract and known failure modes. **Never invent a compliance obligation or a control's strength** — if the regulatory regime/residency/retention is unstated, it's an open driver question for the controller, not a guess. **One writer** preserves conceptual integrity (zero-trust runs through the whole design, not just the edge); the `security-architect` advises read-only and verifies its dimension, the `solution-architect` folds it in. **Record the tradeoff and the rejected alternatives** on every decision (centralized vs service-local authz; HS256 vs RS256; mTLS vs signed-token; centralized guardrail sidecar vs per-agent embedded policy), and **treat every credential as read-scoped — never log or echo a secret.** Read-only audit authors nothing and edits no source. **Never write to project memory** — lessons and residual risks go in the returned report and the ADRs.
