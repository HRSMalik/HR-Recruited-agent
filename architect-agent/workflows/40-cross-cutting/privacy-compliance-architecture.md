# privacy-compliance-architecture

**Group:** 40-cross-cutting  ·  **Runs as:** subagent: ../.claude/agents/security-architect.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose

Build **privacy-by-design** into the structure and map the **regulatory regime** (GDPR / HIPAA / PCI DSS / SOC 2) into architectural controls — data classification, residency, consent, purpose limitation, minimization, and the **LINDDUN** privacy threats per data flow — so compliance is *structural*, not policed after the fact. Answers: *which regime applies, what obligations does it impose, and which architectural control (segmentation, tokenization, minimization, an erasure path, a residency boundary) satisfies each — so the regulated scope is reduced by the architecture rather than carried everywhere?*

This is the **privacy/regulatory** sibling of `security-architecture.md` (STRIDE / confidentiality-integrity-availability of the system) and the upstream half of `../20-data/data-governance-lifecycle.md` (the lifecycle mechanics — retention, lineage, erasure jobs). Security asks *can an attacker break in?*; this flow asks *is the lawful processing of personal data designed in, and is the regulated footprint as small as the architecture can make it?*

## Inputs & preconditions

- From `project-architecture-config.md`: the project's stack (per `project-architecture-config.md`) and its data stores (e.g. a primary datastore plus collections/tables for the core business entity, accounts/organizations, users, an audit log, and a sequence-counter store), the compliance regime (PII = the personal-data fields the project handles + any financial data; the applicable industry/PII obligations; RBAC scoping + audit logging as existing controls, per `project-architecture-config.md`), the constraints (stack lock; the deployment topology; secrets externalized via env, per `project-architecture-config.md`), the deployment target (per `project-architecture-config.md`'s prod-deploy ADR scope), and the `architecture/<slug>/` artifact location.
- The architecture **drivers**: the prioritized quality-attribute scenarios (utility tree) from `../00-drivers/quality-attribute-scenarios.md` and `architecture/<slug>/quality-attributes.md`. The compliance/privacy scenarios — a Data Subject Request (DSR / right-to-erasure), a minimization mandate, a residency constraint, a consent/purpose-limitation requirement, PCI/PHI scope reduction — must appear as named `QAS-*` leaves or be surfaced as **open driver questions**. Never invent a regulatory obligation or a retention window that the source-of-truth docs (BRD / PRD / understanding log, per `project-architecture-config.md`) do not state; if the regime is unconfirmed (which exact obligations apply to this project's domain and jurisdiction — state-level data-breach laws? CCPA? GLBA financial-privacy? PCI if card data is touched?), surface it as an open driver and downgrade to audit-only, saying so.
- The **data classification + PII inventory** from `../20-data/data-governance-lifecycle.md` and the data-modeling output. If the field-level classification table in `architecture/<slug>/threat-model.md` has not been authored, this flow reads the schema directly (per the schema/validator locations named in `project-architecture-config.md`) to build the personal-data inventory and flags the missing classification table as a `critical` precondition gap.
- The **trust boundaries + DFD** from `security-architecture.md` — the same data-flow diagram that anchors STRIDE anchors LINDDUN; reuse it, do not re-derive. Each data flow that carries personal data gets a LINDDUN pass.
- Target: the system/subsystem under design or review (the core platform, a planned payment/financial bounded context, a document-processing context) and the personal-data + financial-data it processes.
- **Preconditions:** `project-architecture-config.md` read; the utility tree (or stub) available; the personal-data inventory built or read; `security-architecture.md`'s DFD available or built. A flow with **no identified regime and no PII inventory** downgrades to audit-only and says so — you cannot map obligations to controls without knowing which obligations apply.

## Oracle (source of truth)

The **applicable regulatory regime's obligations**, expressed as **quality-attribute scenarios with measurable response measures**, plus the **privacy-by-design 7 principles** and the **LINDDUN** privacy-threat taxonomy as the structural coverage check. The regime is the external truth — not the architect's taste, not "we handle privacy well." Example DSR scenario: *"A data subject (source) submits a verified erasure request (stimulus) to the system (artifact) in normal operation (environment); the system erases or anonymizes every personal-data field linked to the subject across the primary store, all denormalized copies, the audit log, and any backup within the retention window (response) within the statutory window (e.g. 30 days for GDPR Art. 17) with a traceable completion event (response-measure)."* Every privacy/compliance decision — classification, residency boundary, consent mechanism, minimization rule, scope-reduction segmentation — must trace to a regime obligation expressed this way or to a stated constraint. **A privacy rule with no number and no traced regulatory driver is a preference, not a finding** (`../shared/finding-schema.md`, the grounding rule).

- **hard:**
  - **The applicable regime is identified and its obligations are mapped to architectural controls.** For each regime in scope (GDPR / HIPAA / PCI DSS / SOC 2 / state-PII law — whichever the drivers confirm), every material obligation has a named architectural control or an explicit accepted-risk ADR. A regime in scope with an unmapped obligation (e.g. "GDPR applies" but no DSR path) blocks. An *unconfirmed* regime is an open driver question, not a guessed obligation.
  - **PII / PHI / PCI data is classified, minimized, and residency-honored.** Every field carrying end-user/subject personal data or financial data is classified (cross-reference `../20-data/data-governance-lifecycle.md`'s four-tier scheme, sub-classified by regime); every personal-data field maps to a named functional/business purpose (purpose limitation + minimization — a field with no purpose is an over-collection finding, `major` where PII); and where a residency constraint is a driver, every store, replica, and backup path satisfies it or carries an accepted-risk ADR. An unclassified personal-data field, an unjustified PII field, or an unaddressed residency constraint blocks.
  - **Erasure / DSR paths exist for every personal-data store.** Every store holding personal data has a Data-Subject-Request fulfilment path — access (export), rectification, and erasure/anonymization — that reaches the primary store, denormalized copies, the audit log, logs/events, and in-window backups, OR an accepted-risk entry stating why a path is structurally impossible and what compensating control exists (e.g. "audit-log entries are immutable for SOC 2 integrity; PII is anonymized in place rather than hard-deleted"). No DSR/erasure path → blocks. (The erasure *mechanics* — the runnable job, the TTL index, the cascade — are owned by `../20-data/data-governance-lifecycle.md`; this flow asserts the path *exists and is reachable per regime*.)
  - **The regulated / in-scope footprint is minimized by segmentation, not just policed.** Where a high-obligation data class is present (PCI cardholder data, PHI, or other sensitive financial fields), the architecture reduces what falls *in scope* — tokenization/vaulting so raw card/account numbers never enter the main store, a segmented collection/table/boundary so the regulated data is isolated from the general business data, or a third-party processor that takes the data out of scope entirely. A design that lets the highest-obligation data spread across every collection (so the *whole* system is in PCI/PHI scope) when segmentation would shrink the footprint is a `major` hard finding. *(Confirm whether raw card/bank-account numbers are actually stored — if payment/settlement flows use a processor token instead, PCI scope may already be out; assert the scope boundary explicitly per `project-architecture-config.md`, do not assume it.)*
  - **Every LINDDUN privacy threat per data flow has a control or an accepted risk.** Walk each personal-data flow in the DFD against the seven LINDDUN categories — **L**inkability, **I**dentifiability, **N**on-repudiation (in the privacy sense — a subject cannot deny an action where they should be able to), **D**etectability, **D**isclosure of information, **U**nawareness (the subject is uninformed / cannot exercise control), **N**on-compliance — and for each applicable threat record a mitigation (a control: pseudonymization, k-anonymity/aggregation, access control, consent capture, transparency notice, minimization) or an explicit accepted-risk entry. A personal-data flow with an unaddressed applicable LINDDUN threat and no accepted-risk blocks. *(Note: LINDDUN's privacy "Non-repudiation" is a threat — plausible deniability lost — which can directly conflict with the security/audit-log Non-repudiation requirement; name that tradeoff where it occurs.)*
  - **Consent + purpose limitation are represented in the architecture where the regime requires them.** Where the regime mandates a lawful basis / consent (GDPR Art. 6, consent capture; CCPA opt-out), the architecture has a place to record the basis/consent and its scope, and processing is gated to the captured purpose. A regime requiring consent with no architectural slot to record or enforce it blocks. *(For an internal B2B enterprise tool the lawful basis is more likely contract/legitimate-interest than end-user consent — confirm the basis with the driver per `project-architecture-config.md`; do not bolt on a consent banner the regime does not require.)*
  - **No personal data in logs / events / diagrams without classification + a bounded path.** Raw PII (SSN, DOB, bank-account, subject identifiers) must not appear in structured logs, the audit-log payload, emitted events, or the architecture diagrams unless classified, retention-bounded, and erasure-reachable. Unclassified PII in a log/event/diagram is a `critical` finding (cross-reference `../20-data/data-governance-lifecycle.md`'s log-scrubbing rule and `../shared/diagramming-standards.md` — secrets/PII never in a diagram).
  - **Every significant privacy/compliance decision is a complete ADR** — context · ≥2 options · decision · consequences (positive AND negative) · status — with `Driver refs` tracing to a `QAS-*` compliance scenario in `architecture/<slug>/quality-attributes.md` (`../shared/adr-template.md`). A privacy decision (tokenize-vs-encrypt-in-place, segment-vs-tag, consent-vs-legitimate-interest, regional-pinning-vs-accepted-risk) with no recorded alternatives or no negative consequences blocks. Flag the **one-way-door** ones (the data-residency topology, the classification scheme, whether a payment processor is in or out of PCI scope).

- **soft:**
  - **Privacy is designed in, not retrofitted** — minimization and purpose limitation happen at *modeling time* (the schema collects the minimum personal data each purpose requires; every personal-data field is added with its purpose and classification), not as a pre-audit cleanup. A design that needs a manual field-sweep before a regulator visit scores lower than one where the schema layer carries the data-class + purpose and new regulated fields inherit the policy. (Privacy-by-design principle 2: *privacy as the default setting*; principle 7: *respect for user privacy / data-subject-centric*.)
  - **Compliance scope is reduced by architecture, not just covered by policy** — the staff-architect move is to make the *whole system* not be in PCI/PHI scope by isolating or tokenizing the regulated data (scope reduction), so most components carry no regulated obligation at all. A design that satisfies every control but leaves the regulated data smeared everywhere (so every component is in scope and must be audited) is correct-but-heavier; the segmented design that shrinks the audit surface scores higher. Penalize the reflex to encrypt-everything-and-call-it-compliant where a boundary would have removed the obligation.
  - **Right-sized to the project's actual scale and audience** (per `project-architecture-config.md`) — an internal enterprise tool with a small, known user base is not a consumer platform with data subjects at scale; do not import a full consent-management platform, a data-residency multi-region topology, or a privacy-engineering tooling stack the confirmed regime does not demand. The end-to-end-visible, minimal-personal-data, segmented-financials design beats a privacy-theater stack. (Penalize accidental complexity; `../shared/quality-attribute-rubric.md`.)
  - **Tradeoff honesty** — each privacy control records what it costs (tokenization adds a vault dependency + a detokenize hop on every read of the real number; pseudonymization weakens analytics joins; data-residency pinning constrains the deployment topology and raises cost; minimization can remove a field a future feature wanted). The chosen point on each privacy tradeoff is named, not hidden.

## Standards & techniques

- **Privacy-by-Design — the 7 foundational principles (Cavoukian)** — (1) **Proactive not reactive / preventive not remedial**; (2) **Privacy as the default setting** (no action required by the subject to be protected); (3) **Privacy embedded into design** (a core function, not a bolt-on); (4) **Full functionality — positive-sum, not zero-sum** (privacy *and* functionality, not a trade); (5) **End-to-end security — full lifecycle protection** (cradle-to-grave, ties to `../20-data/data-governance-lifecycle.md`); (6) **Visibility and transparency** (the processing is open to verification — the audit log + a data-flow that can be shown); (7) **Respect for user privacy — keep it user-centric** (data-subject-centric: access, consent, minimization, transparency). Each material design decision is checked against these; principles 2 and 3 are the ones a checklist forgets and the soft oracle weights.
- **LINDDUN privacy threat modeling** — the privacy analog of STRIDE, applied to the **same DFD** as `security-architecture.md`. The seven categories per data flow / data store / process: **Linkability** (two items can be linked to the same subject → unlinkability via pseudonyms/aggregation), **Identifiability** (a subject can be singled out → anonymity/pseudonymity, k-anonymity), **Non-repudiation** (the subject cannot deny — privacy threat: plausible deniability lost → repudiable channels where appropriate; note the conflict with the security audit requirement), **Detectability** (it's observable that data about a subject *exists* → undetectability/unobservability), **Disclosure of information** (data leaks to the unauthorized → confidentiality, access control, encryption), **Unawareness** (the subject is uninformed or cannot exercise control → consent, transparency notices, data-subject rights), **Non-compliance** (the system violates a regime → policy + technical compliance controls). Run LINDDUN GO or the full methodology; map each elicited threat to a mitigation strategy (the LINDDUN mitigation taxonomy) or an accepted risk.
- **Data residency + data sovereignty** — *residency* = the data physically sits in a named jurisdiction; *sovereignty* = it is subject to that jurisdiction's laws regardless of where it sits. Where a driver states a residency/sovereignty constraint, the control is regional pinning of the store + all replicas/backups, in-region processing, and egress controls; where none is stated (confirm against `project-architecture-config.md`), record "no residency constraint confirmed" with the regulatory basis rather than designing a multi-region topology no driver demands.
- **Consent + purpose limitation** — purpose limitation (GDPR Art. 5(1)(b)): personal data is collected for specified, explicit purposes and not further processed incompatibly. The architecture represents the **lawful basis** per processing activity (consent / contract / legitimate interest / legal obligation) and gates processing to the recorded purpose; for an internal B2B tool the basis is often contract/legitimate-interest, not end-user consent — confirm before adding consent machinery. Consent (where required) is captured with scope + timestamp + version and is revocable, which links to the DSR/erasure path.
- **Data minimization** — collect and retain the *minimum* personal data each purpose requires (GDPR Art. 5(1)(c); HIPAA *minimum necessary*). The check is an access-pattern audit: every personal-data field maps to a named functional requirement / business capability; unmapped PII fields are over-collection candidates. Minimization at modeling time is the privacy-by-design default (principle 2).
- **The specific regime obligations (map only the ones the drivers confirm):**
  - **GDPR** — Data Subject Rights (Art. 15–22: access, rectification, erasure/Art. 17, portability, restriction, objection); lawful basis + consent (Art. 6); data minimization + purpose + storage limitation (Art. 5); residency / international transfer (Ch. V); breach notification (72h); records of processing + DPIA where high-risk. → architectural controls: a DSR fulfilment path, a lawful-basis registry, a minimized + classified schema, a residency boundary, a breach-detectable audit trail.
  - **HIPAA** — applies only if PHI is processed (confirm against `project-architecture-config.md` and the domain — some verticals carry medical/injury data on a record, which *is* PHI). Privacy Rule *minimum-necessary*; Security Rule administrative/physical/technical safeguards; access controls + audit controls + transmission security; a Business Associate Agreement where a processor touches PHI. → controls: PHI sub-classification, minimum-necessary access (ties to RBAC scoping), encrypted transmission (per the project's crypto standard), audit-log of PHI access.
  - **PCI DSS** — applies only where cardholder data is stored/processed/transmitted; the dominant move is **scope reduction**: tokenize via a PCI-compliant payment processor so raw PAN never enters the system (taking most of the system *out of scope*), segment any cardholder-data environment, and never store the CVV. For any payment integration in scope: confirm whether raw bank-account/routing or card numbers are stored — if so they are sensitive financial data warranting tokenization/vaulting + segmentation even outside strict PCI.
  - **SOC 2** — the Trust Services Criteria (Security + the relevant of Availability / Processing Integrity / Confidentiality / **Privacy**); the Privacy criterion maps to notice, choice/consent, collection, use/retention/disposal, access, disclosure, quality, monitoring. → controls: the audit log as the monitoring control, retention/disposal from `../20-data/data-governance-lifecycle.md`, access control from `security-architecture.md`.
- **STRIDE ↔ LINDDUN relationship** — STRIDE (`security-architecture.md`) covers the *security* threats (an attacker compromises CIA); LINDDUN covers the *privacy* threats (a legitimate or illegitimate actor harms the data subject's privacy). They share the DFD and overlap at Information Disclosure ≈ Disclosure, but LINDDUN adds Linkability/Identifiability/Detectability/Unawareness that STRIDE never sees. Run both against the same boundaries; never assume STRIDE coverage implies privacy coverage.
- **Well-Architected Security pillar + the data-protection patterns** — encryption in transit (TLS) + at rest (per the project's crypto standard, per `project-architecture-config.md`), tokenization/vaulting for the highest-obligation data, pseudonymization/anonymization as a LINDDUN control, least-privilege access (RBAC scoping) — reach for the **established control** before inventing one (`../shared/guardrails.md`, reuse known patterns).
- **ATAM / utility tree** — privacy options (tokenize-vs-encrypt-in-place; segment-the-regulated-collection-vs-field-tag-in-place; consent-capture-vs-legitimate-interest; regional-pin-vs-accept-residency-risk) are scored against the utility tree (compliance, operability, performance, evolvability) for **sensitivity** and **tradeoff** points. The canonical tradeoff here: **compliance-scope-reduction ↔ operational/architectural complexity** — tokenization/segmentation shrinks the audit surface but adds a vault dependency + a detokenize hop + a boundary to operate; field-tagging-in-place is simpler but leaves the whole store in scope. Name the chosen point on this curve (`../shared/fallacies-and-tradeoffs.md`).

## Step sequence

- **audit:** Frame the drivers (which regime is confirmed in scope; which compliance/privacy scenarios are prioritized `(value × risk)`) → build/read the personal-data inventory + the field-level classification (from `../20-data/data-governance-lifecycle.md` or the schema) → reuse `security-architecture.md`'s DFD and run **LINDDUN** per personal-data flow → walk each confirmed regime's obligations against the existing/proposed architecture (DSR/erasure path reachable? minimization satisfied? residency honored? regulated scope segmented or smeared? consent/lawful-basis represented? PII out of logs/events/diagrams?) → emit findings + the privacy risk register (read-only; **no artifact authored, no source edited, no infra touched**). Surface every unconfirmed regime/obligation as an open driver question rather than guessing.
- **design:** **Frame** (extract/confirm the regime + the prioritized compliance scenarios + the personal-data inventory, read-only) → **Explore** (N divergent privacy strategies as concrete specs — e.g. *tokenize-financials-via-processor + segment* vs *encrypt-in-place + field-tag* vs *minimize-and-pseudonymize-at-ingest*; each with its scope-footprint, its DSR path, its residency stance, its LINDDUN coverage and tradeoffs, read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each vs the utility tree + the hard oracles; finds the scope-reduction sensitivity point and the compliance-↔-complexity tradeoff point + the privacy risks; order-swapped to kill bias; picks a winner, grafts the superior ideas from runners-up) → **Document** (the **one writer** `solution-architect` folds the security-architect's privacy findings into the architecture: the regime→control mapping, the field-level classification + purpose registry, the LINDDUN threat-model section in `architecture/<slug>/threat-model.md`, the DSR/erasure architecture, the residency stance, the scope-reduction segmentation, and the ADRs — conceptual integrity preserved, established controls reused) → **Verify** (LINDDUN coverage per flow, regime-obligation→control traceability, DSR-path reachability across every personal-data store, classification completeness as a **fitness function**, diagram-compile, ADR-completeness as **hard gates**; the evaluator re-scores from the authored artifacts) → loop ≤ 10 or pass.

## Assertions & exit gate

- The applicable regime(s) is/are identified (or surfaced as an open driver); every confirmed obligation maps to an architectural control or an accepted-risk ADR — no unmapped obligation in scope.
- Every personal-data / financial field is classified and sub-classified by regime; every PII field maps to a named purpose (minimization + purpose limitation); over-collection candidates are flagged (`major` for PII).
- A DSR/erasure path exists and is reachable for every personal-data store (primary, denormalized copies, audit log, logs/events, in-window backups) — or an accepted-risk entry names the structural reason + the compensating control.
- Where a residency constraint is a driver, every store/replica/backup satisfies it or carries an accepted-risk ADR; where none is confirmed, "no residency constraint" is recorded with its basis (not a guessed multi-region design).
- The highest-obligation data (PCI cardholder / PHI / bank-account financials) is **scope-reduced by architecture** — tokenized/vaulted/segmented/processor-offloaded — or the decision to keep it in-scope is a deliberate, traced ADR; not smeared across every collection by default.
- Every personal-data flow in the DFD has run **LINDDUN**; each applicable Linkability/Identifiability/Non-repudiation/Detectability/Disclosure/Unawareness/Non-compliance threat has a control or an accepted risk; the LINDDUN-vs-STRIDE Non-repudiation conflict is named where it occurs.
- Consent / lawful basis is represented where the confirmed regime requires it (and *not* bolted on where contract/legitimate-interest is the basis); no raw PII in logs/events/diagrams without classification + a bounded, erasure-reachable path.
- Every significant privacy decision is a complete ADR (context · ≥2 options · decision · consequences pos+neg · status) tracing to a `QAS-*` compliance scenario; one-way-door decisions (residency topology, classification scheme, PCI-scope boundary) flagged.
- **Gate:** hard oracles green (regime mapped · personal data classified + minimized + residency-honored · DSR/erasure path per store · regulated scope segmented or traced-accepted · LINDDUN per flow controlled-or-accepted · consent/lawful-basis represented where required · no unclassified PII in logs/events/diagrams · ADRs complete) AND (design) the evaluator's rubric mean ≥ 0.8. Any unmapped in-scope obligation, any unclassified personal-data field, any personal-data store with no DSR/erasure path, any uncontrolled applicable LINDDUN threat, or any PII in a log/event/diagram → **fail**, loop back to the writer.

## Output

Write `artifacts/privacy-compliance-architecture/report.json` per `../shared/report-format.md` — findings/risks per `../shared/finding-schema.md` (`type: "privacy"` or `"compliance"`; `oracle: "hard"` for an unmapped in-scope obligation / unclassified PII / a missing DSR-erasure path / an uncontrolled LINDDUN threat / unjustified-residency / PII-in-logs-or-diagram; `oracle: "soft"` for retrofit-not-designed-in or scope-not-reduced-by-architecture craft gaps), each carrying its `driver_ref` (the `QAS-*` compliance scenario or the regime obligation), its `tradeoff`, and its `alternatives_rejected`. Include the `drivers[]` block (the prioritized compliance/privacy scenarios) and, for design mode, the verification block (LINDDUN coverage per flow + regime-obligation→control traceability + DSR-path reachability + classification-completeness fitness-function results + hard-oracle results + rubric score). Authored design artifacts land in `architecture/<slug>/` (never scattered): the regime→control mapping and the LINDDUN threat model in `architecture/<slug>/threat-model.md` (the privacy section alongside the STRIDE section from `security-architecture.md`); the field-level classification + purpose/lawful-basis registry, the DSR/erasure architecture, and the residency stance folded into `design-doc.md`; the privacy ADRs in `architecture/<slug>/adr/`; the classification-completeness + LINDDUN-coverage checks specified in `fitness-functions.md`. Open driver questions (unconfirmed regime, unstated retention window, unconfirmed PCI/PHI applicability) are listed explicitly for the controller to resolve — never silently assumed.

## Guardrails

Per `../shared/guardrails.md`: **propose & document — never write product code, never apply infra, never run a migration or touch a live data store.** This flow authors the regime→control mapping, the LINDDUN threat model, the classification/purpose registry, the DSR/erasure architecture, the ADRs, and the fitness-function specs into `architecture/<slug>/` — the controller implements the controls and the operator applies any infra. Ground every privacy rule in a confirmed regulatory driver or a stated constraint — **never invent an obligation or a retention window**; an unconfirmed regime is an open question, not a guess (no number + no traced regime = a preference, not a finding). Reuse the established privacy/security controls (classification scheme, tokenization, pseudonymization, RBAC scoping, the project's crypto standard, the audit log — per `project-architecture-config.md`) before inventing a mechanism, and right-size to the project's actual scale and audience — do not import a consent-management platform or a multi-region residency topology the confirmed regime does not demand. The read-only audit authors nothing and edits nothing; in design mode the security-architect advises read-only and the single `solution-architect` writer folds the findings in to preserve conceptual integrity. Record the tradeoff and the rejected alternatives on every decision; **never write to project memory** — lessons and risks live in the returned report and the ADRs. Never echo or place a secret, a raw PII value, or a credential into a report, an ADR, or a diagram.
