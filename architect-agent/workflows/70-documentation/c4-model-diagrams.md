# c4-model-diagrams

**Group:** 70-documentation  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Produce the architecture diagrams as diagrams-as-code — C4 Context, Container, Component, Deployment, and Dynamic views — that compile without error, are internally consistent across levels, and trace every element and relationship to a documented decision or driver. Answers: *is every structurally significant architecture decision made legible, correctly leveled, and verifiably current — not as a pasted PNG that rots, but as version-controlled source that renders?*

## Inputs & preconditions
- From `project-architecture-config.md`: the diagram tooling (Mermaid for C4 Context/Container in-Markdown; PlantUML+C4-PlantUML for richer Component/Deployment views where Mermaid's C4 support is thin; D2 for deployment topology); the `architecture/<slug>/diagrams/` artifact location; the project's stack and deployment target (per `project-architecture-config.md`); and the ADR log location (`architecture/<slug>/adr/`).
- The architecture **drivers**: the prioritized quality-attribute scenarios (the utility tree) from `workflows/00-drivers/quality-attribute-scenarios.md` — every container, component, and deployment node drawn must trace to a driver it serves; an element with no driver behind it is accidental complexity made visible. If the utility tree is absent, this flow surfaces it as a blocking precondition.
- The completed C4 Level-1 system context from `workflows/00-drivers/context-scoping.md` — the actor and external-system names and the in/out scope boundary established there are the source of truth for every L1 element in the diagram set; this flow must not introduce new actors or external systems the context-scoping flow did not scope in.
- The architecture description (arc42 views) from `workflows/70-documentation/architecture-description-arc42.md` if present — names, boundaries, and decisions must be consistent with it.
- The ADR log (`architecture/<slug>/adr/`) — every named container, component, and deployment node must correspond to at least one accepted ADR (or a `proposed` ADR flagged as open); a drawn element with no ADR behind it is an unjustified decision.
- Preconditions: the source-of-truth docs (BRD / PRD / understanding log) named in `project-architecture-config.md` are readable; diagram tooling confirmed in `project-architecture-config.md`; `architecture/<slug>/diagrams/` directory writable; the context-scoping flow has either completed or its actor/scope output is available as a read input.

## Oracle (source of truth)
The C4 model (Simon Brown) applied via the diagrams-as-code convention (`shared/diagramming-standards.md`), with every element traceable to a driver and every diagram verified to compile. Split hard and soft:

- **hard:**
  - Every diagram is **diagrams-as-code** (Mermaid `C4Context`/`C4Container`, PlantUML+C4-PlantUML `!include <C4/C4_Container>`, or D2) and **compiles/renders without error** — run the toolchain; a syntax error or render failure is a broken artifact and blocks (`shared/diagramming-standards.md` → *Diagram compiles* hard oracle; `shared/quality-attribute-rubric.md` → *Diagram compiles* hard oracle).
  - **C4 levels are consistent across the diagram set:** every Container named in the L2 view appears in L3 (Component) if drilled into; every Container and Component that appears in the deployment-view mapping corresponds to a real Container/Component in L2/L3; names match the ADRs and the architecture description exactly — any drift between a diagram element's name and its ADR or the design-doc text is a hard finding.
  - Every **element** (Person, Software System, Container, Component, Node, Deployment Node) carries a **name + type + one-line responsibility** (`[Container: API service]`, "handles core-entity lifecycle, RBAC, and audit logging"). An unlabeled or untyped box fails this oracle.
  - Every **relationship** carries a **label + direction + sync/async distinction**: the verb + the protocol/style + whether the call is synchronous (solid line) or asynchronous/event (dashed line): "submits request, JSON/HTTPS (sync)", "publishes domain-event to, internal queue (async)". An unlabeled, undirected, or unclassified arrow fails this oracle.
  - **Trust boundaries** are drawn wherever the threat model has them (as boundary boxes or dashed security perimeters) — at minimum: the client vs the ingress/gateway; the ingress/gateway vs the application backend; the backend vs the primary data store; any external-system boundary (auth provider, payment integration, document-processing provider, third-party autocomplete/lookup service). Missing a trust boundary where STRIDE identifies one is a hard finding.
  - Every diagram element traces to a **decision** (an accepted or proposed ADR in the ADR log) or to a **driver** (a quality-attribute scenario in the utility tree). A box or relationship with no traceable decision or driver is accidental complexity drawn in — flag it with `ARCH-C4-NNN`, `type:"documentation"`, `severity:"major"`, `oracle:"hard"`.
  - **No orphan elements** — every box connects; every external system in L1 reappears as the other end of a labeled relationship in L2; every container drilled into in L3 appears in L2. An element with no relationships fails this oracle.
  - The C4 **altitude is correct per level**: L1 (System Context) shows one system box + Persons + System_Ext; L2 (Container) shows deployable/runnable units, not code modules; L3 (Component) shows major modules inside one container, not classes; Deployment shows containers mapped onto infrastructure nodes. Mixing levels within a single diagram fails this oracle.

- **soft:**
  - Diagrams are at the **right zoom for the audience** — L1 is legible to a non-technical reader without a legend for internal structure; L2 communicates the major tech choices and integration contracts to a developer; L3 is drawn only for containers that are complex enough to warrant it (not every container needs a Component view).
  - **No accidental complexity drawn** — every box reflects a real decision with a documented rationale; the diagrams do not show aspirational structure or deferred integration-seam elements as if built (they are scoped as `System_Ext` placeholders or explicitly labeled `[deferred]`).
  - **Naming is consistent** with the domain ubiquitous language (`workflows/00-drivers/domain-analysis.md`), the project's RBAC role names (per `project-architecture-config.md`), and the ADR log — no synonyms for the same concept across levels.
  - **Legend included** when notation is not self-evident (sync/async line styles, boundary types, data-store shapes); the diagram communicates trust boundaries cleanly to a security reviewer.
  - The supplementary Deployment view argues the **availability and deployment-topology quality-attribute scenarios** — containers mapped onto real infrastructure nodes (per the project's deployment target in `project-architecture-config.md`); the mapping is honest (does not imply multi-AZ or redundancy the design does not provide).

## Standards & techniques
- **The C4 model** (Simon Brown) — the canonical four levels: Level 1 (System Context), Level 2 (Container), Level 3 (Component), Level 4 (Code — rarely hand-maintained; generated on demand). Supplementary views: Deployment (containers → infrastructure nodes, required to argue availability/scalability scenarios), Dynamic/Sequence (a specific runtime scenario's flow, e.g. a core-entity intake saga or a failover path). System Landscape (multiple systems in an enterprise context) is out of scope unless `project-architecture-config.md` says otherwise.
- **Diagrams-as-code toolchain** per `project-architecture-config.md` and `shared/diagramming-standards.md`:
  - **Mermaid** (`C4Context`, `C4Container`, `sequenceDiagram`) — default for in-Markdown rendering (GitHub-native, embeds in `design-doc.md`); lightest weight; use for L1 and L2 where Mermaid's C4 support is sufficient.
  - **PlantUML + C4-PlantUML stdlib** (`!include <C4/C4_Container>`, `!include <C4/C4_Component>`, `!include <C4/C4_Deployment>`) — for richer Container/Component/Deployment views where Mermaid's C4 rendering is thin; renders anywhere; the most faithful C4 tool after Structurizr.
  - **D2** — modern auto-layout, clean; use for Deployment and topology views where spatial clarity matters more than C4 stereotypes.
  - **Structurizr DSL** — purpose-built for C4; one model, many auto-laid-out views; preferred when the diagram set grows beyond 4–5 views and drift between levels becomes a maintenance risk (Structurizr enforces consistency structurally via a single model). Flag as an ADR choice if adopted.
  - All diagram source files land in `architecture/<slug>/diagrams/`; the rule is invariant: **text, must compile, must be version-controlled**. A pasted PNG is not an architecture artifact.
- **Notation conventions** (`shared/diagramming-standards.md`): name + type + one-line responsibility on every element; label + direction + verb + protocol/style + sync/async marker on every relationship; solid lines for synchronous request/response, dashed for asynchronous/event; trust boundary boxes drawn where the threat model has them; legend included if notation is non-obvious; consistent names across levels (no synonyms).
- **Traceability discipline**: map each diagram element to its ADR in `architecture/<slug>/adr/` and to its QAS in `quality-attributes.md` — document the mapping in `architecture/<slug>/tradeoffs.md` or a dedicated traceability table in the design-doc. The `traceability-mapping` flow (`workflows/70-documentation/traceability-mapping.md`) owns the full matrix; this flow contributes the per-diagram element mapping as its share.
- **MADR decision records** (`shared/adr-template.md`) — the diagram tooling choice (Mermaid vs PlantUML vs Structurizr) is an ADR if it affects how the team reviews and maintains diagrams; record it once and reference it from every diagram file header comment.
- **Supplementary Deployment view**: map every Container from L2 onto a named infrastructure node (per the project's deployment target in `project-architecture-config.md` — e.g. a reverse-proxy/ingress node, a container host, a managed database node); show deployment regions/AZs only if the availability scenarios require them (for a project whose current drivers call for a single-node deployment, that is honest — do not imply redundancy the design does not provide); this view is the architecture's argument against availability and scale scenarios.
- **Dynamic/Sequence view**: author a Dynamic view for each `(H,H)` quality-attribute scenario that involves a multi-step runtime interaction (e.g. a core-entity intake end-to-end: client → gateway/ingress → application backend → data store → audit log; the auth + RBAC enforcement path). Use PlantUML `sequenceDiagram` or Mermaid `sequenceDiagram`. These make the failure modes and trust-boundary crossings visible at the scenario level.

## Step sequence
- **audit:** Read `project-architecture-config.md` → enumerate any existing diagram files in `architecture/<slug>/diagrams/` → for each file: attempt to compile/render (Mermaid CLI / PlantUML / D2 — whichever the project uses); check every element is labeled name+type+responsibility; check every relationship is directed+labeled+sync/async; check trust boundaries are drawn where the threat model has them; check C4 level altitude is correct; check every element traces to an ADR or driver (spot-check the ADR log in `architecture/<slug>/adr/`); check cross-level consistency (L2 container names == names referenced in L3 and Deployment views); check no orphan elements → emit findings per `finding-schema.md` (`type:"documentation"`, `oracle:"hard"` for compile failure, unlabeled element/relationship, missing trust boundary, altitude violation, orphan element, name drift from ADR/design-doc; `oracle:"soft"` for zoom/audience and naming-consistency craft gaps). Read-only; no artifact authored.

- **design:** Frame (confirm the scope of views to author from `project-architecture-config.md` — at minimum L1 Context, L2 Container, Deployment view; add L3 Component only for containers complex enough to warrant it; confirm which `(H,H)` scenarios need a Dynamic view; read the utility tree and the ADR log to establish the element-to-driver mapping before drawing anything) → Explore (produce ≥2 candidate diagram-set designs as prose+pseudo-DSL specs — e.g. Option A: Mermaid for all views, tradeoff = limited C4 stereotype support in complex diagrams; Option B: Mermaid for L1/L2, PlantUML+C4-PlantUML for Component/Deployment, tradeoff = two toolchains but richer fidelity; evaluate each against the compile-oracle and the project's diagram-tooling constraint in `project-architecture-config.md`) → Evaluate (skeptical evaluator scores each option against the hard oracles + the tooling-fit soft oracle; order-swapped to kill position bias; picks the option with the fewest toolchain dependencies that satisfies every hard oracle and the project's existing tool preference; records the decision as an ADR-shaped choice) → Document (one writer authors all diagram source files in `architecture/<slug>/diagrams/` — one file per view: `context.md` or `context.puml`, `container.md` or `container.puml`, `component-<name>.puml` per drilled container, `deployment.puml` or `deployment.d2`, `dynamic-<scenario>.md` per `(H,H)` scenario; every element labeled; every relationship labeled+directed+sync/async; trust boundaries drawn; names match ADRs + design-doc; each file's header comment cites the ADR and the QAS it serves) → Verify (run the toolchain on every authored file — compile/render each diagram; check every hard oracle against the rendered output: element labels, relationship labels+direction+sync/async, trust boundaries, C4 altitude, cross-level name consistency, no orphans, traceability spot-check against ADR log; evaluator re-scores rubric; loop ≤ 5 or pass).

## Assertions & exit gate
- Every diagram file in `architecture/<slug>/diagrams/` compiles/renders without error — run the toolchain (Mermaid CLI, PlantUML, D2), not a manual syntax review; a render error is an unambiguous hard-oracle failure.
- Every element (Person, Container, Component, Node) carries name + type + one-line responsibility; no unlabeled or untyped boxes remain.
- Every relationship carries a label, a direction, a verb + protocol/style, and a sync/async marker; no unlabeled, undirected, or unclassified arrows remain.
- Trust boundaries are drawn at every boundary the threat model (`architecture/<slug>/threat-model.md`) identifies; at minimum: client ↔ ingress/gateway; ingress/gateway ↔ application backend; application backend ↔ primary data store; application backend ↔ any external system (auth provider, payment integration, document-processing provider, third-party autocomplete/lookup service).
- C4 levels are consistent: every L2 Container name matches exactly the name used in any L3 Component view and in the Deployment view; no name drift from the ADR log or `architecture/<slug>/design-doc.md`.
- No orphan elements: every box is the endpoint of at least one labeled relationship; every external system in L1 appears as the other end of a labeled relationship in L2.
- Every Container and Component element traces to an accepted (or explicitly `proposed`) ADR in `architecture/<slug>/adr/` and to a quality-attribute scenario in `quality-attributes.md`; the mapping is documented in the diagram file's header comment or in `tradeoffs.md`.
- The Deployment view maps every Container from L2 onto a named infrastructure node; the topology is honest (does not imply redundancy or multi-AZ that the design does not provide; deferred integration-seam elements are scoped as explicit placeholders, not drawn as built).
- Dynamic/Sequence views exist for every `(H,H)` quality-attribute scenario that involves a multi-step runtime interaction across trust boundaries.
- (Design mode) Rubric scores (soft oracles — evaluated only after all hard oracles are green): audience zoom correctness ≥ 0.8; traceability narrative quality and completeness in header comments ≥ 0.8; naming consistency with domain ubiquitous language and the source-of-truth docs named in `project-architecture-config.md` ≥ 0.8; legend clarity and trust-boundary legibility for a security reviewer ≥ 0.8.
- **Gate:** hard oracles green (every diagram compiles without error; every element labeled+typed+one-line-responsibility; every relationship labeled+directed+sync/async; trust boundaries present at all STRIDE-identified boundaries; C4 levels consistent with no name drift; no orphan elements; every element traces to an accepted or proposed ADR and a quality-attribute scenario) AND (design) rubric mean ≥ 0.8.

## Output
Write `artifacts/c4-model-diagrams/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (`type:"documentation"`; `oracle:"hard"` for a diagram that fails to compile, an unlabeled element or relationship, a missing trust boundary, a C4 altitude violation, an orphan element, a name drift from the ADR log, an element with no traceable driver; `oracle:"soft"` for zoom/audience craft gaps, naming inconsistency, or a missing legend; `driver_ref` traces to the QAS or constraint that the missing/broken diagram element threatens — e.g. a missing trust boundary traces to the security scenario `QAS-SEC-*`). For audit mode: a prioritized findings list (critical → cosmetic) and a risk register for every structural inconsistency or untraceable element. For design mode: the verification block (per-file compile result, element/relationship label coverage, trust-boundary presence, cross-level consistency check, traceability spot-check result, rubric score per soft oracle) plus the paths to the authored diagram files:

```
architecture/<slug>/diagrams/
├── context.md              C4 L1 System Context (Mermaid C4Context)
├── container.md            C4 L2 Container (Mermaid C4Container or PlantUML+C4-PlantUML)
├── component-<name>.puml   C4 L3 Component per drilled container (PlantUML+C4-PlantUML)
├── deployment.puml|.d2     C4 Deployment view (PlantUML or D2)
└── dynamic-<scenario>.md   Dynamic/Sequence view per (H,H) scenario (Mermaid sequenceDiagram)
```

Open diagram questions (a container whose internal structure is undocumented, an external-system integration whose protocol is unstated, a trust boundary whose STRIDE coverage is unknown) are listed explicitly for the controller to resolve — never silently elided.

## Guardrails
Per `shared/guardrails.md`: propose and document — never write product code; ground every drawn element in a stated source (ADR, the BRD/PRD/understanding-log named in `project-architecture-config.md`); do not invent containers, components, or deployment nodes that no driver or decision demands (accidental complexity). Read-only audit authors nothing. Exactly one writer (`solution-architect`) authors all diagram source so the set has conceptual integrity (one coherent visual language, one decomposition criterion, consistent names — no committee seams). The diagram-source files (`*.puml`, `*.md`, `*.d2`) in `architecture/<slug>/diagrams/` are the deliverables, never PNGs; the compile oracle is the gate. Deferred integration seams (any planned-but-not-built subsystem) are drawn as `System_Ext` placeholders with an explicit `[deferred]` label if shown at all, never as built containers. Cross-reference `../shared/diagramming-standards.md` for notation rules and the compile hard oracle; cross-reference `../00-drivers/context-scoping.md` for the L1 actor/scope contract (L2 must not introduce actors or external systems the context-scoping flow did not scope in); cross-reference `architecture-description-arc42.md` for the full architecture-description package this diagram set must be consistent with.
