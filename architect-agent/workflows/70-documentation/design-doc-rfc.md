# design-doc-rfc

**Group:** 70-documentation  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Run the design-doc / RFC process — problem statement, explicit goals and non-goals, the proposal, ≥2 considered alternatives, cross-cutting impact, rollout, and open questions — as the lightweight collaborative review structure before a significant decision is locked. Answers: *is there a reviewable, decision-forcing document that the team can challenge before the choice becomes a one-way door?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target, team topology (Conway), constraints (stack lock, phase scope), compliance regime, where ADRs + architecture artifacts live (`architecture/<slug>/`).
- The architecture **drivers**: the functional scope, the prioritized quality-attribute scenarios from `workflows/00-drivers/quality-attribute-scenarios.md`, and the constraints + assumptions that scope the decision under review. A design doc with no utility-tree reference cannot be evaluated as a decision-forcing document.
- Target: the proposed decision, change, or architectural direction that warrants a design doc / RFC — stated as the scope, the component or subsystem affected, and the driver(s) it must satisfy. If the proposal lacks a concrete driver (`QAS-*` ref or a named constraint), that is the first finding.
- Preconditions: the drivers and quality-attribute scenarios are available or the `00-drivers` flows have been run first (`workflows/00-drivers/architecture-drivers.md`, `workflows/00-drivers/quality-attribute-scenarios.md`); existing ADRs in `architecture/<slug>/adr/` read before drafting so this doc does not re-litigate settled ground.

## Oracle (source of truth)
The external truth for a design doc / RFC is whether it is **structurally complete as a decision-forcing document** — not whether it reads well. A doc that has no explicit non-goals, only one alternative, or no open questions is not a design doc; it is a status update masquerading as one.

- **hard:** (1) the doc states the problem + context explicitly — a reader who was not in the room understands what is being decided and why now; (2) explicit **goals AND non-goals** are both present — the non-goals must be as precisely stated as the goals (scope discipline: a non-goal that is vague or absent means the scope is open-ended and the proposal cannot be evaluated); (3) **≥2 genuinely considered alternatives** are described with their tradeoffs — an alternative that is named but not analyzed fails this oracle; (4) a **cross-cutting impact section** covers the affected quality attributes, trust boundaries, data paths, and operational concerns — not just the happy path; (5) the proposal **traces to at least one named driver** (`QAS-*` scenario or constraint from `project-architecture-config.md`) — a proposal with no driver is accidental complexity; (6) **open questions are enumerated explicitly** (not buried) — a document with no open questions has either answered everything (rare) or has not looked hard enough; (7) the document is **decision-forcing**: it identifies who decides, by when, and what the default is if no decision is made.
- **soft:** the non-goals are as crisp as the goals — they cut scope at a real boundary, not at "everything else"; the alternatives are genuine (not a straw man next to the obviously favored option); the proposal drives toward a decision that will land as one or more ADRs (`workflows/70-documentation/architecture-decision-records.md`); the doc is concise enough to be acted on in a single review session — a target of ≤ 2,500 words (excluding appendix/references) and ≤ 10 open questions; longer docs are a signal that scope has not been bounded by the non-goals section.

## Standards & techniques
- **The design-doc / RFC format** (the standard collaborative structure, as used across Google/Rust RFC/IETF/Python PEP/AWS): *title + status + author + date · problem statement · context and background · goals · non-goals · proposal (the design) · considered alternatives · cross-cutting impact · rollout / migration plan · open questions · appendix / references*. Each section is mandatory for a decision-forcing document; an absent section is a hard-oracle gap.
- **Goals-vs-non-goals as scope control** — the non-goals section is a first-class architectural artifact, not a footnote. It names the related problems that are **deliberately not being solved** by this proposal. A tight non-goals list bounds the proposal's blast radius and prevents scope creep during review. The test: can a reviewer use the non-goals list to close an out-of-scope question? If not, the non-goals are too vague.
- **The review-and-comment workflow** — a design doc is circulated for comments before the decision is made; the RFC status lifecycle (`proposed → accepted / rejected / withdrawn / superseded`) mirrors the MADR ADR lifecycle (`shared/adr-template.md`). An accepted design doc's key decisions are immediately translated into ADRs in `architecture/<slug>/adr/`; the design doc is the narrative; the ADR is the durable, immutable record.
- **One-way vs two-way door test** (Bezos) — the doc must declare whether the proposal is a one-way door (hard to reverse — state why, and justify the depth of analysis) or a two-way door (reversible — lighter review is appropriate; do not over-analyze). A one-way-door proposal without deep alternatives analysis is incomplete.
- **ATAM sensitivity and tradeoff points** (`shared/evaluation-method.md`) — the cross-cutting impact section must name the **sensitivity points** (decisions that strongly determine whether a quality-attribute scenario is met) and the **tradeoff points** (decisions that improve one attribute and worsen another). An impact section with no tradeoffs is incomplete craft.
- **The 8 fallacies of distributed computing** (`shared/fallacies-and-tradeoffs.md`) — the cross-cutting impact section checks the proposal against the fallacies where they apply; a proposal that assumes zero latency, a reliable network, or a homogeneous topology without accounting for it fails the `fallacies_honored` hard oracle at the ADR stage.
- **STRIDE per new trust boundary** (`workflows/40-cross-cutting/security-architecture.md`) — if the proposal introduces or re-draws a trust boundary, the cross-cutting impact section must include a minimal STRIDE pass for that boundary. An unanalyzed trust boundary is a security finding before the proposal even reaches the ADR.
- **Driver traceability** — every section of the proposal traces to one or more drivers (`QAS-*` or constraint ID). A section that serves no driver is a candidate for the non-goals list. This is the architecture analog of the grounding rule in `shared/quality-attribute-rubric.md`: no number + no traced driver = a preference, not a proposal.

## Step sequence
- **audit:** Read the existing design doc or RFC draft → check structural completeness against the hard oracle checklist (problem, goals, non-goals, ≥2 alternatives with analysis, cross-cutting impact, open questions, driver traces, one-way-door declaration) → flag every missing or vague section as a hard-oracle gap finding (`type:"documentation"`, `oracle:"hard"`) → check that alternatives are genuine (not straw-men) and that the proposal's driver traces are real QAS refs or named constraints → flag any unanalyzed trust boundary or unacknowledged fallacy violation as a cross-cutting finding → emit findings prioritized (critical → cosmetic) as a read-only report. No artifacts authored, no edits made.
- **design:** Frame (read `project-architecture-config.md` + the utility tree; confirm the drivers the doc must satisfy; confirm what is in/out of scope by checking existing ADRs for settled ground) → Explore (draft ≥2 structurally distinct proposal directions — each a concrete sketch with tradeoffs, not the same idea re-coloured; read-only at this stage) → Evaluate (skeptical evaluator scores each direction against the utility-tree drivers, the one-way-door test, and the hard structural oracle; order-swapped to kill position bias; names sensitivity and tradeoff points; picks a direction or synthesizes the best-of) → Document (ONE writer authors the design doc in `architecture/<slug>/design-doc-rfc-<decision-slug>.md` following the standard format: problem · context · goals · non-goals · proposal · alternatives · cross-cutting impact with STRIDE pass if a trust boundary is affected · rollout · open questions; every proposal section traces to a `QAS-*` or constraint; a one-way-door declaration is present; the alternatives section includes the evaluation rationale) → Verify (check every hard oracle: structural completeness, ≥2 alternatives analyzed, driver traces, non-goals sharp, open questions enumerated, one-way-door declared; STRIDE present if new trust boundary; cross-cutting impact names at least one sensitivity point and one tradeoff point; evaluator re-scores rubric from the authored artifact) → loop ≤ 5 or pass gate. On acceptance, draft the ADR stubs in `architecture/<slug>/adr/` (`shared/adr-template.md`) for each key decision surfaced.

## Assertions & exit gate
- The doc contains all mandatory sections: problem statement, context, goals, non-goals, proposal, ≥2 considered alternatives (each with a named tradeoff, not just a label), cross-cutting impact, open questions, rollout plan, one-way-door declaration.
- Non-goals are as precisely stated as goals — each names what is *deliberately excluded* at a level a reviewer can use to close out-of-scope questions.
- Every section of the proposal traces to at least one named driver (`QAS-*` or a named constraint from `project-architecture-config.md`); no section is driverless.
- The alternatives are genuine: each has an analysis of why it was not chosen, naming the driver or tradeoff that ruled it out.
- The cross-cutting impact section names at least one **sensitivity point** and one **tradeoff point** (per ATAM, `shared/evaluation-method.md`).
- If the proposal introduces or re-draws a trust boundary, a STRIDE pass for that boundary is present in the cross-cutting impact section.
- Open questions are enumerated in a dedicated section — not embedded in prose; each carries an owner and a resolution target date.
- The document is **decision-forcing**: it names who decides, by when, and what the default is if consensus is not reached.
- On design exit: ADR stubs are drafted in `architecture/<slug>/adr/` for each key decision, ready for the controller to promote to `accepted`.
- **Gate:** hard oracles green (structural completeness, ≥2 analyzed alternatives, driver traceability, sharp non-goals, open questions section, STRIDE if new trust boundary, one-way-door declared, decision-forcing closure) AND (design) rubric mean ≥ 0.8.

## Output
Write `artifacts/design-doc-rfc/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (`type:"documentation"`, `oracle:"hard"` for any missing mandatory section, single alternative, absent driver trace, undeclared one-way door, or missing STRIDE on a new trust boundary; `oracle:"soft"` for non-goals that are present but imprecise (cannot be used to close an out-of-scope question), straw-man alternatives, buried open questions, or a proposal that will not reduce to actionable ADRs); plus the verification block (structural-completeness checklist results, driver-trace coverage, STRIDE flag, rubric score) for design mode. Authored artifacts land in `architecture/<slug>/`:
- `design-doc-rfc-<decision-slug>.md` — the full design doc in standard format
- `adr/<NNNN>-<kebab-title>.md` (stub, status `proposed`) — one per key decision surfaced by the doc, per `shared/adr-template.md`

Cross-links: `workflows/70-documentation/architecture-decision-records.md` (for the ADR stubs that follow acceptance), `workflows/00-drivers/architecture-drivers.md` (for the driver + utility-tree inputs).

## Guardrails
Per `shared/guardrails.md`: propose and document — never write product code; ground every section of the proposal in a driver (no proposal section without a QAS ref or named constraint, no driver without a proposal section that satisfies it); reuse existing patterns and settled ADRs before re-opening decisions; one writer authors the design doc so it has conceptual integrity (Brooks) — not a committee's seams; record the tradeoffs and rejected alternatives (both the chosen direction and what it costs are mandatory — a proposal with only upside is incomplete); the design doc is the narrative, the ADR is the durable immutable record — never let the doc substitute for the ADR on the decisions it surfaces.
