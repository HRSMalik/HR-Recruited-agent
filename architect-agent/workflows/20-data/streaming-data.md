# streaming-data

**Group:** 20-data  ·  **Runs as:** subagent: ../.claude/agents/data-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Architect data-in-motion: the batch-vs-stream choice, the Lambda/Kappa topology decision, delivery and processing semantics (exactly-once vs at-least-once + idempotent consumers), event-time vs processing-time with watermarks, windowing strategies, and backpressure — for pipelines, analytics, and event-driven read models. Answers: *what streaming topology and processing contract does this system need, and what does each cost on latency, consistency, and operational complexity?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack + deployment target (e.g. today: no broker + no queue, or whatever the project's baseline is); the stated NFRs and SLO targets (BRD/PRD — never invent numbers); the compliance regime (per the project's data-sensitivity classification); team topology (per the project's team size — Conway applies); the artifact locations (`architecture/<slug>/`).
- The architecture **drivers**: the prioritized quality-attribute scenarios (utility tree from `workflows/00-drivers/quality-attribute-scenarios.md`) that streaming must serve — specifically the latency/freshness scenario (e.g. "an order-status read model updated within N seconds of a state transition") and the throughput scenario (events/sec at peak). A flow with no measured freshness or throughput driver downgrades to batch + defers; it does not impose a broker or a streaming topology on a system that has no such driver.
- The **functional scope**: which data flows are in motion — core-entity lifecycle events, audit log entries, document-processing results, payment notifications, report aggregations. The distinction between *event notification* (something happened, small payload) and *event-carried state* (the new state is in the payload) changes the consumer design.
- The **write paths** for which a consistency model must be stated (per the `consistency_stated` hard oracle): every stream-produced read model or serving layer must declare whether it is eventually consistent with the source-of-truth store and what the bounded staleness window is.
- **Preconditions**: utility tree read and the relevant `(H,H)` freshness/throughput scenarios identified; `project-architecture-config.md` read (stack, constraints); `workflows/20-data/cqrs-event-sourcing.md` consulted if read/write separation is a related driver (the streaming serving layer may be a CQRS read model); `workflows/30-integration/async-messaging.md` consulted for broker selection + delivery semantics when a broker is already in scope; `workflows/90-specialized/realtime-streaming.md` consulted when sub-second latency or WebSocket/SSE push to clients is a stated driver. If no freshness driver is stated in the utility tree, surface as an open question — do not assume one.

## Oracle (source of truth)

The driver-traced quality-attribute scenarios from the utility tree, made concrete:

- **hard:**
  - **Processing semantics stated and matched:** every stream consumer declares its delivery guarantee — at-least-once or exactly-once — and the design matches the use. At-least-once delivery is the default; consumers are designed idempotent (deduplication key, natural idempotency of the operation, or an idempotency store) so duplicate delivery is a no-op. Exactly-once processing (via transactional producers + atomic offset commit, or an outbox + CDC) is used only where the driver requires it and a non-idempotent consumer makes at-least-once unsafe; the cost (throughput ~30–50% lower, Kafka exactly-once semantics requiring same-cluster transactions, or 2PC-equivalent coordination) is stated in the ADR. "Exactly-once theater" — claiming exactly-once semantics via a framework flag while the consumer performs non-idempotent writes — is a hard oracle failure.
  - **Late/out-of-order data handled:** for any flow that uses event time (vs processing time), watermarks are specified — the maximum expected lateness (e.g. "events arrive ≤ 30 s late; watermark = event_time − 30 s") — and a drop-or-reprocess policy for events that arrive after the watermark is stated. A flow that processes event time without watermarks has an unbounded window and blocks.
  - **Backpressure designed:** no unbounded in-process buffering — the consumer signals capacity to the producer (e.g. poll-based Kafka consumer naturally applies backpressure; a push-based HTTP webhook fan-out to a slow consumer is a fallacy-3 violation). Where the producer is faster than the consumer, the overflow strategy is explicit: bounded queue + block, bounded queue + drop-with-metric, or scale-out the consumer tier. An architecture that silently buffers without bound fails `fallacies_honored` (bandwidth is not infinite) and the `failure_mode_defined` hard oracle.
  - **Serving-layer consistency stated:** every stream-produced read model (a materialized view, a search index, a reporting aggregate) declares its consistency relationship with the source-of-truth store: the bounded staleness window (e.g. "read model lags ≤ 10 s at p99 under expected event rate") and the mechanism for handling a consumer restart (replay from offset-0, or snapshot + delta). A read model with no stated consistency model blocks.
  - **Every ASR traces to a decision** (driver traceability) and every decision traces to a driver. A streaming topology chosen without a freshness or throughput driver that demands it is accidental complexity — flag it, do not design it.
  - **Failure mode defined:** every cross-process hop in the pipeline (producer → broker → consumer → serving store) carries a bounded timeout, a retry policy (idempotent retries with backoff+jitter where applicable), and a dead-letter / poison-pill strategy. An event that cannot be processed must not block the partition/queue indefinitely.

- **soft:**
  - **Kappa over Lambda unless a driver forces both.** The Lambda architecture (a batch layer producing correct-but-slow views + a speed layer producing approximate-but-fresh views + a serving layer merging both) is the correct answer only when the batch reprocessing output is structurally *different* from the stream output and the system must serve both simultaneously. Where the stream processor can reprocess from a durable log (Kafka retention, S3-backed log), Kappa (stream-only, reprocess from the beginning of the log when the logic changes) is simpler, has one code path, and eliminates the batch/speed merge complexity. Propose Lambda only when a concrete driver (e.g. ML feature computation that requires full historical scans too expensive to re-stream) demands it.
  - **Simplest topology that meets the freshness scenario.** A database change-data-capture (CDC) feed or a simple outbox poll is the right answer for many "near-real-time read model" cases — a full broker + stream-processing cluster is not. A table-driven polling approach (e.g. a background worker tailing an audit-log table at 5-second intervals) satisfying a "freshness ≤ 30 s" driver is simpler and more boring than a Kafka + Flink deployment; prefer it unless the throughput or latency driver rules it out.
  - **No exactly-once theater.** If the consumer operation is idempotent (inserting a record keyed on an immutable natural ID, incrementing a counter via an atomic update with a processed-event idempotency store), exactly-once delivery machinery is accidental complexity. State this explicitly; the evaluator penalizes the more elaborate mechanism if idempotency suffices.

## Standards & techniques

- **Batch vs stream processing** — batch: scheduled, bounded dataset, high latency/high throughput; stream: unbounded, event-triggered, low latency. The choice is driven by the freshness scenario's response measure. If no sub-minute freshness driver exists, batch is simpler and correct (Gall's Law: start simple).
- **Lambda architecture** (Jay Kreps) — batch layer (correct, slow, full recompute) + speed layer (approximate, fast, incremental) + serving layer (merges both). Introduces two code paths for the same logic (a tradeoff point: accuracy vs latency vs complexity). Use only when both layers are genuinely needed.
- **Kappa architecture** (Jay Kreps) — stream-only; the immutable log is the source of truth; reprocessing = replay from offset-0 with a new consumer version. One code path. Wins on simplicity when the stream processor can handle the reprocessing volume. The evaluator defaults to Kappa and requires a Lambda driver.
- **Delivery + processing semantics:**
  - *At-most-once*: fire-and-forget, no retry — acceptable only for non-critical telemetry where loss is tolerable (not core business records).
  - *At-least-once* + *idempotent consumer*: the default; retries on failure; consumer must handle duplicates (natural idempotency or a dedup key). The idempotency contract is the design's obligation.
  - *Exactly-once*: either transactional producers + atomic offset commit (Kafka transactions, same-cluster only) or outbox + CDC (see `workflows/20-data/cqrs-event-sourcing.md`). Higher latency, higher operational complexity, requires the broker to support it end-to-end. Use only when the consumer cannot be made idempotent.
- **Event time vs processing time:** event time = when the event occurred (in the source system); processing time = when it arrived at the processor. Event-time processing is correct for out-of-order data but requires watermarks and a late-data policy. Processing-time processing is simpler but produces incorrect results for out-of-order data; use it only when the ordering guarantee is provided by the source (e.g. a single-writer append-only log with monotonic timestamps).
- **Watermarks:** a heuristic estimate of how far behind real time the event stream is. A watermark at `max_event_time_seen − max_lateness_bound` triggers window close. Too tight → late events dropped; too loose → window close delayed, memory held. The max_lateness_bound must be derived from observed or estimated network/source delay, not invented.
- **Windowing:** tumbling (fixed non-overlapping), sliding (fixed overlapping, heavier), session (activity-gap based), global (no window — aggregate everything). Window choice is driven by the analytics question; session windows are natural for core-entity lifecycle aggregations; tumbling windows for time-boxed throughput metrics.
- **Backpressure:** the mechanism by which a slow consumer signals the producer to slow down, preventing unbounded buffer growth (Fallacy 3: bandwidth is not infinite). Pull-based consumers (Kafka poll loop) implement backpressure natively; push-based systems (HTTP callbacks, gRPC streams, SSE) require explicit flow control or a bounded queue between the push source and the processing stage. Reactive Streams (Project Reactor, Akka Streams, RxJava) standardize the backpressure protocol for in-process pipelines.
- **Outbox pattern + CDC:** the canonical at-least-once-or-exactly-once bridge between a transactional write and an event bus — write to the DB transactionally + write to an outbox table in the same transaction; a CDC reader (Debezium) or a polling worker reads the outbox and publishes to the broker. Removes the dual-write / lost-event failure mode. Cross-reference: `workflows/20-data/cqrs-event-sourcing.md`.
- **Dead-letter queue (DLQ) / poison-pill handling:** events that fail processing after N retries are routed to a DLQ for inspection/replay; the partition is unblocked. A pipeline with no DLQ strategy blocks on any malformed or persistently-failing event.
- **AKF scale cube (X/Y/Z-axis):** stream consumers scale on the X-axis (horizontal clone) by adding partitions; Y-axis (functional decomposition) by splitting the event type into separate consumer groups; Z-axis (data partitioning) by partition key. Consumer group scale-out is bounded by partition count — design partition count to meet the peak-throughput scenario.
- **PACELC applied to stream serving:** the serving layer is a read model updated asynchronously — it is inherently on the Latency side of the PACELC curve (PA/EL). State the bounded staleness window as the consistency model; don't claim the read model is strongly consistent with the source store.
- **ATAM evaluation** — score each streaming direction against the (H,H) freshness/throughput scenarios; identify sensitivity points (e.g. idempotency is the sensitivity point for at-least-once correctness — a non-idempotent consumer write makes duplicate delivery a data-integrity failure) and tradeoff points (e.g. broker introduction trades operational complexity and a new stack component for throughput headroom and producer/consumer decoupling). Run order-swapped to kill position bias; the simpler direction that meets every driver scores higher. Per `shared/evaluation-method.md`.

## Assertions & exit gate

- Every data-in-motion flow has a freshness or throughput driver from the utility tree; any flow proposed without one is flagged as accidental complexity (`ARCH-STREAM-NNN`, `oracle:"soft"`, severity `minor`).
- Every stream consumer states its delivery guarantee (at-least-once or exactly-once) and the design matches: at-least-once consumers are provably idempotent (natural idempotency or a stated deduplication mechanism); exactly-once consumers state the transactional mechanism and the cost.
- "Exactly-once theater" is absent: no flow claims exactly-once semantics while performing a non-idempotent consumer write without a transactional mechanism end-to-end.
- Every event-time flow specifies watermarks (the max_lateness_bound and its derivation) and a late-data policy (drop / side-output / reprocess).
- Backpressure is designed for every pipeline: no unbounded in-process buffer; overflow strategy is explicit (block, drop-with-metric, or scale-out).
- Every stream-produced serving layer (read model, materialized view, aggregate) states its consistency model and bounded staleness window; no serving layer claims strong consistency with an asynchronous source.
- Every pipeline hop (producer → broker → consumer → serving store) has a bounded timeout, a retry policy with backoff+jitter, and a DLQ / poison-pill strategy.
- Lambda topology is used only when a driver demands both batch and speed layers; Kappa or simpler (outbox-poll, CDC without a broker) is the default.
- Every ADR has context, ≥2 options, decision, and consequences (positive AND negative), per `shared/adr-template.md`.
- All diagrams-as-code compile and render without error; every pipeline hop is labeled with direction, protocol, and delivery semantics; sync vs async is visually distinct.
- The capacity model proves the freshness response measure at the stated peak event rate; no claim of "scales to X events/sec" without a back-of-envelope model.

**Gate:** hard oracles green (processing semantics stated + matched; no exactly-once theater; event-time flows have watermarks and late-data policy; backpressure designed; serving-layer consistency stated; failure modes defined on every hop; Lambda justified by driver or Kappa/simpler chosen; traceability complete; diagrams compile; ADRs complete; capacity model proves freshness QAS) AND (design) rubric mean ≥ 0.8.

## Guardrails

Per `shared/guardrails.md`: propose & evaluate — never write product code (`src/**`, `frontend/**`, `backend/**`). Fitness-function *specifications* and diagram source in `architecture/<slug>/` are the deliverable; a runnable fitness-function check script into `tests/fitness/` (e.g. a script that asserts read-model lag ≤ N seconds against a running environment) is a verification artifact, not product code. Ground every decision in a driver: no broker, no Kafka, no exactly-once machinery without a quality-attribute scenario that demands it — the project's stack baseline (per `project-architecture-config.md`) governs what's already in scope; a new technology component requires a driver-justified ADR with operational-cost tradeoff and a flag that it is outside the current stack constraint. One writer authors all streaming artifacts (conceptual integrity). Reuse outbox + CDC patterns from `workflows/20-data/cqrs-event-sourcing.md` and broker/delivery semantics from `workflows/30-integration/async-messaging.md` before designing bespoke mechanisms. For real-time push to clients (sub-second, WebSocket/SSE), cross-reference `workflows/90-specialized/realtime-streaming.md`. Record the rejected topology and its cost — a recommendation with only upside is incomplete. Never write to project memory; lessons and findings go in the returned report and the ADRs.
