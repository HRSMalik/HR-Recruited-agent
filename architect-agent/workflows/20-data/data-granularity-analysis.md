# data-granularity-analysis

**Group:** 20-data  ·  **Runs as:** subagent: ../.claude/agents/data-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Decide how to break apart — or deliberately keep together — the system's **data**: which service/module owns which store, when a joint/shared table is fine versus when it must be split, and which datastore-per-service topology (monolithic DB vs domain/decomposed databases) the drivers justify. This is the **data analog of service granularity** (`10-styles-decomposition/service-granularity-analysis.md`): that flow decides how big a *service* should be; this flow decides how big a *data store* should be and who's allowed to touch it — and the two must agree, because a service split with no matching data split (or a data split with no matching service split) produces a distributed monolith at the persistence layer.

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack and its current data shape — the project's stack (per `project-architecture-config.md`), whether that is a single shared database behind a single service, or an already-decomposed set of stores; the config names the current datastore(s), the service(s) fronting them, and the collections/tables in play; note whether polyglot persistence or a per-service database already exists. Note the near-term bounded contexts the config calls out as seams to keep (e.g. a document-processing capability, a payment/settlement capability, a closure/workflow capability) — this flow decides, for each, whether its data should land in the same store (joint ownership, module-level separation) or a genuinely separate store, and says so explicitly rather than defaulting either way.
- The **drivers / utility tree** (`../00-drivers/quality-attribute-scenarios.md`): specifically the scale/throughput scenarios that would force a data split (a hot collection outgrowing a shared instance), the availability scenarios (does one team's schema change risk another team's read path?), and the team-topology constraint (Conway — who can operate what).
- The **decomposition strategy** already decided (`../10-styles-decomposition/decomposition-strategy.md`, `../10-styles-decomposition/service-granularity-analysis.md`) — data granularity follows service/module granularity, it does not precede it. If no decomposition decision exists yet, this flow evaluates the *current* monolithic-DB shape against the drivers and flags whether a split is even on the table, rather than inventing service boundaries of its own.
- The **target**: the specific data store(s)/collection(s)/table(s) under review or design, and — for design mode — the specific data-ownership question being decided (e.g. "who owns the core business entity once a document-processing capability and a payment capability become real services: one owner + two consumers, or a split?").
- Preconditions: the utility tree exists (else `status:error`); the aggregate/entity model from `data-modeling.md` is read (aggregates are the candidate seams for a data split); the consistency model per write path (`consistency-transactions.md`) is read where a split is being evaluated, because splitting a store turns a local transaction into a distributed one.

## Oracle (source of truth)
The **data-decomposition forces** from Richards & Ford's *Software Architecture: The Hard Parts* (Part II, "Pulling Apart Operational Data") — the **disintegrators** (reasons to split) weighed explicitly against the **integrators** (reasons to keep together) — never a split (or a non-split) asserted without naming which forces were weighed and how they netted out.

- **hard** (measurable / traceable / structurally checkable):
  - **Single ownership per store.** Every shared data table/collection/store has exactly **one owning service/module** that may write to it directly. Any other consumer reads via that owner's API/contract or an explicit, justified joint-ownership pattern (below) — never a second writer touching the same table. An un-designated or dual-writer store is a `critical` finding (the data-layer distributed-monolith smell).
  - **Joint ownership, where it exists, is one of the four named patterns** (Richards & Ford) and is named explicitly, not left implicit: *table split* (physically split the shared table along ownership lines, denormalize/duplicate as needed), *data-ownership-by-transaction-context* (whichever service is transactionally in-flight owns the write, others delegate to it), *delegate* (one owner absorbs writes-on-behalf-of the others via its API), or *service consolidation* (fold the services sharing the data back into one, when the data coupling is stronger than the service-boundary driver justifying the split). A design with two writers and none of these four named is a hard failure.
  - **Every disintegrator that motivates a split is named and weighed against every relevant integrator** — the six disintegration driving forces (change control, connection management, scalability, fault tolerance, architectural quanta, database-type optimization) and the two integration forces (data relationships, database transactions) are the fixed checklist; a split (or a stay-together) decision that doesn't show the weighing is unjustified — accidental complexity if it splits with no disintegrator, an availability/consistency risk left unaddressed if it stays together against a strong disintegrator.
  - **A data split preserves every transactional invariant it used to hold locally**, or explicitly moves that invariant to a named saga pattern (`../shared/transactional-saga-taxonomy.md`) with its compensations designed, or to an outbox+CDC per `consistency-transactions.md`. Splitting a table that two write paths update inside one local transaction, with no compensation/saga/outbox named for the now-distributed write, is a `critical` finding — it is the data-granularity flow's version of the dual-write anti-pattern.
  - **Architectural quantum boundaries match data boundaries.** Where the decomposition strategy asserts a service is an independent architectural quantum (`../shared/evaluation-method.md` → Step 0), its data must not require synchronous joins/transactions against another quantum's store — a quantum whose "independent" deploy still needs a cross-store join at request time is not actually independent; flag the mismatch.
  - **The DB-type-optimization force is justified, not assumed.** If a store is proposed to split *because* one part of the data wants a different database type (e.g. a search index, a graph store, a time-series store), the design names the specific access pattern that the current store handles poorly (structurally: a query shape, a write-amplification number, a scale ceiling) — "NoSQL vs SQL" without a named access-pattern mismatch is not a justification.
- **soft** (craft, rubric-scored only after hard oracles are green):
  - **Data is split only when the forces net toward splitting** — a shared DB is fine (and simpler) until a disintegrator genuinely forces decomposition; the default, absent a driver, is to keep data together (YAGNI at the persistence layer, mirroring the modular-monolith default in `project-architecture-config.md`). Splitting because "microservices should each own their database" with no disintegrator named is over-engineering.
  - **The consistency model survives the split.** After a split, every write path that used to be local-transactional states its new consistency model and mechanism (per `consistency-transactions.md`) and the design shows the business invariant is still held — not merely "eventually" waved at.
  - **The chosen joint-ownership pattern (if any) is the simplest that resolves the specific force** — prefer delegate/table-split for a narrow shared field before reaching for full service consolidation; don't consolidate services back together over a single low-volatility shared lookup table.
  - **Change control and connection-management pressure are actually present**, not assumed, when cited as the reason to split (e.g. "two teams deploy schema changes to this table on independent cadences and have collided" / "the shared connection pool is saturated at peak" — a named, evidenced pressure, not a hypothetical).

## Standards & techniques
- **Data decomposition — the Hard Parts (Richards & Ford, ch. 9–11).** The six **disintegrators** (reasons to split shared data) and two **integrators** (reasons to keep it together), used as the fixed pro/con checklist for every ownership decision:
  - *Disintegrators:* **change control** (independent schema/data evolution cadence per owner), **connection management** (pool exhaustion / contention from many services hitting one store), **scalability** (one part of the data needs to scale independently of another), **fault tolerance** (isolate blast radius — one store's outage shouldn't take down an unrelated capability), **architectural quanta** (the service boundary demands independent deployability, which a shared synchronous-transaction store defeats), **database-type optimization** (a different data shape/access pattern wants a different database technology).
  - *Integrators:* **data relationships** (foreign-key-like structural relationships that are cheap inside one store, expensive/eventually-consistent across two), **database transactions** (an ACID guarantee across rows/collections that a split turns into a distributed transaction problem — see `consistency-transactions.md`).
  - The decision procedure: for the store in question, score each disintegrator (present / strength) and each integrator (present / strength); if disintegrators dominate, split (and pick the split *pattern*, below); if integrators dominate, keep together and record why splitting was rejected.
- **The "who owns this data" question** — every store/table gets a named single owner (a service/module boundary), mirroring the DDD aggregate-ownership rule (`data-modeling.md`) at the store level, not just the object level. Shared-data situations are resolved by exactly one of the **four joint-ownership patterns**: table split, data-ownership-by-transaction-context, delegate, service consolidation (Richards & Ford ch. 11) — never left as an implicit shared table with two writers.
- **CAP/PACELC + the saga taxonomy** (`../shared/fallacies-and-tradeoffs.md`, `../shared/transactional-saga-taxonomy.md`) — every split that breaks a local transaction is re-expressed as a named saga pattern (default: Parallel Saga, async+eventual+orchestrated, absent a hard atomicity driver) with compensations, or as outbox+CDC (`consistency-transactions.md`).
- **Architectural quantum** (Richards & Ford; `../shared/evaluation-method.md` Step 0) — the unit that must scale/fail/deploy together; used here to check that the *data* boundary doesn't silently re-couple two quanta that the *service* decomposition claims are independent.
- **DDD aggregates** (`data-modeling.md`) — the aggregate is the natural seam for a data split (never split *inside* an aggregate's consistency boundary without moving to a saga); this flow operates one level up, at the store/service-ownership level, and must stay consistent with the aggregate boundaries already decided.

## Step sequence
- **audit:** Frame the drivers (which scale/availability/team-topology forces are live) → enumerate every shared data store/collection/table in the existing architecture and, for each: *who writes to it today, is there more than one writer, and if so which of the four joint-ownership patterns (if any) governs it?* → for every store touched by more than one write path inside what used to be a single local transaction, check the consistency model is still stated and the invariant still holds → weigh the six disintegrators against the two integrators for each candidate split point already proposed elsewhere (e.g. a decomposition-strategy ADR proposing to split the core business entity from a future document-processing service) → emit findings (un-owned/dual-writer stores, un-named joint-ownership patterns, splits with no preserved invariant, splits/non-splits with no forces weighed) + the risk register. Read-only; authors nothing, edits no source.
- **design:** **Frame** (confirm the data-ownership question in scope — e.g. "a document-processing capability becomes a real service: does it get its own store, or stay in the shared store under the core entity's ownership?" — read-only) → **Explore** (≥2 divergent ownership options as concrete specs: e.g. *(A) keep together* — the document-processing metadata stays in the shared datastore instance, the core-entity service remains sole writer, the document-processing service reads via API (delegate pattern); *(B) split* — document-processing gets its own store (object metadata) with an outbox-relayed integration event back to the core entity, saga-compensated on failure; *(C) table split* — physically separate the document-metadata fields out of the core entity's table/collection into its own table/collection in the same datastore instance, each service owning its own table/collection, joined by a foreign-key-style reference only — each option scored against the six disintegrators/two integrators, read-only) → **Evaluate** (`architecture-evaluator` scores each option: does it net the disintegrators/integrators correctly for the stated forces? does the chosen joint-ownership pattern (if any) actually resolve the cited force? is the invariant preserved or moved to a named saga/outbox with real compensations? — order-swapped to kill bias; the simplest option that resolves the actual force wins, unjustified splits penalized as accidental complexity) → **Document** (the **one writer**, `solution-architect`, authors: the data-ownership map, the joint-ownership-pattern choice with its ADR, and — where a split is chosen — the updated consistency table + saga/outbox diagram cross-referencing `consistency-transactions.md`) → **Verify** (the hard gates below as fitness functions/structural checks + the evaluator re-scores from the artifacts) → loop ≤10 or pass.

## Assertions & exit gate
- **Every data store has exactly one row in the ownership map**: `{store, owning_service, other_readers/writers, joint_ownership_pattern (none|table-split|ownership-by-transaction-context|delegate|service-consolidation), disintegrators_present, integrators_present, net_decision}`. No store with two undeclared writers.
- **Every proposed or existing split states which disintegrator(s) forced it** and shows the integrators (data relationships, transactions) were weighed and outweighed — a split (or non-split) with no forces named is not a decision, it's a preference.
- **Every split that breaks a prior local transaction names its replacement mechanism** — a saga pattern from `../shared/transactional-saga-taxonomy.md` (with compensations) or outbox+CDC (`consistency-transactions.md`) — and the invariant it used to hold is restated and shown still held (eventually, with a real recovery path, or via a redrawn boundary that keeps it local).
- **Every joint-ownership situation names one of the four patterns** explicitly (table split / ownership-by-transaction-context / delegate / service consolidation); an implicit shared table with silent dual writes never survives to `accepted`.
- Each ownership/split decision is an **ADR** (context · ≥2 options · decision · positive+negative consequences · driver refs) and traces to a `QAS-*`; diagrams (the ownership map / the split topology) compile per `../shared/diagramming-standards.md`.
- **Gate:** hard oracles green — **single-owner-or-named-joint-pattern on every store, every split's forces weighed and stated, every broken invariant moved to a named saga/outbox with compensations, no orphan dual-writer** — AND (design) rubric mean ≥ 0.8 (force-fit correctness + invariant preservation + simplest-mechanism-wins).

## Output
Write `artifacts/data-granularity-analysis/report.json` per `../shared/report-format.md` — findings/risks per `../shared/finding-schema.md` (`type:"data"`; `oracle:"hard"` for an un-owned/dual-writer store, an un-named joint-ownership pattern, a split with no preserved/relocated invariant, or a split/non-split decision with no disintegrator/integrator named; `driver_ref` = the scale/availability/team-topology `QAS-*` or constraint the ownership call serves; each finding carries its `tradeoff` and `alternatives_rejected`, ADR-shaped). The `verification` block lists the fitness-function results (the ownership-map-complete check, the dual-writer structural scan, the invariant-preserved-or-relocated check) + hard-oracle results + the rubric score. Design-mode authored artifacts land in `architecture/<slug>/` per `project-architecture-config.md`: the **data-ownership map** in `design-doc.md` (or `tradeoffs.md`), the split topology as a **diagrams-as-code** view in `diagrams/`, and one **MADR ADR** per significant ownership/split decision in `adr/NNNN-*.md` — never scattered. Open questions (an un-stated scale threshold that would force a split, an un-confirmed team boundary for a proposed data owner) are listed for the controller, never silently assumed.

## Guardrails
Per `../shared/guardrails.md`: **propose & document — never write product code and never apply a schema migration or touch a live datastore.** This flow authors the ownership map, the ADRs, and the split/joint-ownership diagrams; the controller implements any collection split or migration and the operator runs it. Ground every ownership/split call in a **driver** — no split without a named disintegrator it resolves, no disintegrator left unaddressed if it's strong and prioritized. **Reuse the four named joint-ownership patterns** (table split, ownership-by-transaction-context, delegate, service consolidation) before inventing a bespoke shared-table arrangement, and default to **keeping data together** (the existing shared-store shape) absent a disintegrator the drivers actually raise — do not propose polyglot persistence or a datastore-per-service split with no scale/fault-isolation/change-control pressure named (per `project-architecture-config.md`'s stated default, where a modular-monolith default applies). Record the **tradeoff and the rejected alternatives** on every decision (which disintegrators won, which integrators were given up), never just the winner. Read-only audit edits nothing; one writer preserves conceptual integrity. Cross-references: `../10-styles-decomposition/service-granularity-analysis.md` (the service-side granularity call this flow's data call must agree with), `consistency-transactions.md` (the consistency model/mechanism a split's broken invariant moves to), `../shared/transactional-saga-taxonomy.md` (the named saga pattern + compensations when a split can't stay local).
