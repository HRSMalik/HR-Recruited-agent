# cqrs-event-sourcing

**Group:** 20-data  ·  **Runs as:** subagent: ../../agents/data-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Decide whether to separate the read and write models (CQRS) and whether to persist state as an ordered log of domain events (event sourcing) — and when the complexity tax makes both the wrong choice. Answers: *does the read/write asymmetry or the auditability driver justify the machinery of separate models, projections, and event replay, or does plain CRUD with a single read/write model satisfy all drivers at lower cost and operational risk?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack (per `project-architecture-config.md` — e.g. an API service + a single-primary datastore, no broker, no queue today; stack lock — no new datastore/broker without a driver-justified ADR), deployment target, compliance regime (per `project-architecture-config.md` — e.g. PII or financial data where audit logging is a hard compliance driver), constraints, and the `architecture/<slug>/` artifact location.
- The architecture **drivers**: the prioritized quality-attribute scenarios (utility tree) from `workflows/00-drivers/quality-attribute-scenarios.md` and the `architecture/<slug>/quality-attributes.md` it produces. Specifically look for:
  - **Read/write asymmetry** — a scenario where read load, read shape, or read latency requirement is materially different from write load/shape (e.g. "dashboard renders aggregate status counts at < 2s from a system under concurrent writes"). Without this driver, CQRS is accidental complexity.
  - **Auditability / temporal query** — a scenario requiring a full history of state transitions, point-in-time reconstruction, or event replay (e.g. "compliance audit can reconstruct the exact entity state at any past timestamp within the retention window"). Without this driver, event sourcing is accidental complexity.
  - **Conflict resolution / multi-writer** — a scenario requiring deterministic conflict resolution across concurrent writes to the same aggregate.
  - **Integration event production** — a scenario where downstream consumers (a deferred integration seam, per `project-architecture-config.md` evolvability driver) must react to every state change, not just the current state.
- The **domain analysis** (bounded contexts, aggregate boundaries) from `workflows/00-drivers/domain-analysis.md` — CQRS/ES boundaries are drawn on aggregate boundaries, not on service convenience; the core business entity's lifecycle FSM (the allowed status transitions protected as aggregate invariants in `workflows/20-data/data-modeling.md`) is the primary candidate aggregate.
- The **consistency/transaction model** from `workflows/20-data/consistency-transactions.md` — event sourcing imposes eventual consistency between the write side (the event store) and the read side (projections/materialized views); this must be weighed against the consistency scenario response measures in the utility tree.
- **Preconditions:** `project-architecture-config.md` read; utility tree available (or stub with open driver questions surfaced); existing write paths read from the codebase (per the source layout in `project-architecture-config.md`); the access-pattern inventory from `workflows/20-data/data-modeling.md` available (CQRS is meaningless without knowing the actual read and write shapes). A flow with no access-pattern inventory and no stated auditability/asymmetry driver **rejects CQRS/ES as unwarranted** and records that decision as an ADR.

## Oracle (source of truth)

The **named quality-attribute scenarios** in the utility tree that correspond to read/write asymmetry, auditability, or integration-event production — each 6-part, each ending in a measurable response — and the **aggregate invariant contract** from `workflows/20-data/data-modeling.md`. A recommendation for CQRS or event sourcing with no traced driver that demands it is a preference, not a finding. The complexity cost of CQRS/ES is always weighed against the driver that justifies it.

- **hard:**
  - **If CQRS/ES is recommended:** the read/write consistency lag between the command side (write model) and the query side (projections/materialized views) is **stated and bounded** per a named scenario — e.g. "projection lag ≤ 500ms at p95 under nominal write load" — and that lag is acceptable to the scenario's response measure. A CQRS design with an unstated or unbounded consistency lag blocks.
  - **If CQRS/ES is recommended:** every **projection is rebuildable from scratch by replaying the event log** — the projection rebuild mechanism (full replay, snapshotted replay) is described, the rebuild time at current event-log size is estimated, and the operational runbook step for triggering a rebuild is specified. A projection with no replay path blocks.
  - **If CQRS/ES is recommended:** **event schema evolution** is handled by a named strategy — versioned event types (`OrderPlaced_v1`, `OrderPlaced_v2`), upcasters, or schema-registry contract — so old events remain deserializable as new consumers are deployed. An event store with no schema-evolution strategy blocks.
  - **If CQRS/ES is recommended:** every **command produces at most one aggregate write** (the unit-of-work matches the aggregate boundary); cross-aggregate consistency uses a saga or process manager with compensation, not a distributed transaction. A command that spans two aggregates without a saga blocks.
  - **If CQRS/ES is rejected (the right-sized choice):** the simpler CRUD model (single read/write model, direct query of the write store) is **explicitly recorded as the chosen design** in an ADR, with the tradeoff stated (lower operational complexity, no projection lag, no replay machinery) and the rejected alternatives (CQRS, ES) with why the drivers did not justify them.
  - All **write paths declare their consistency model** (strong / eventual / causal) and mechanism per `workflows/20-data/consistency-transactions.md`. Event sourcing does not exempt this oracle — the event-append write is strong, the projection is eventual; both must be stated.
  - Every **cross-service or cross-process call** that the CQRS/ES design introduces (e.g. event bus publish, projection subscriber) has a bounded timeout, a retry policy (idempotent + backoff/jitter), and a fallback or fail-fast, per the fallacies hard oracle (`shared/fallacies-and-tradeoffs.md` Fallacy 1: the network is reliable).
  - Every **ADR for the CQRS and/or ES decision** is complete: context · ≥2 options · decision · consequences (positive AND negative) · status. An ADR missing alternatives or consequences blocks.

- **soft:**
  - CQRS and event sourcing are used **only where a driver justifies the complexity tax** — not by default, not because the domain "feels event-driven," not as a résumé pattern. The simpler design that meets every driver scores higher (per `shared/quality-attribute-rubric.md` simplicity oracle; vendor/novelty bias penalized).
  - **Conceptual integrity** between commands, events, and projections: command names are imperative (`SubmitOrder`, `ApproveOrder`), event names are past-tense domain facts (`OrderSubmitted`, `OrderApproved`), projection names describe the read model they serve (`OrderStatusView`, `WorklistView`). Mixed naming conventions, or events shaped as commands, are a craft gap.
  - **Right-sized scope**: CQRS/ES applied to the whole system when only one bounded context (e.g. the core business entity's lifecycle with its audit trail) justifies it is accidental complexity. The recommendation names exactly which aggregates/bounded contexts get CQRS/ES and why each qualifies; all others stay CRUD.
  - **Evolvability seam for deferred integrations**: if integration-event production is the driver, the event design provides a clean integration boundary for future downstream consumers (the outbox or event-log is the seam), without coupling the current system to the consumer's internals.

## Standards & techniques

- **CQRS (Command Query Responsibility Segregation)** — Greg Young's pattern: commands mutate state (return void or an error); queries read state (return data, no side effects). The command side writes to the write model (normalized, invariant-protecting); the query side reads from one or more materialized read views (denormalized, shaped for the query). The two models evolve independently. Key consequences: the consistency gap (reads are eventually consistent with writes), the need for a synchronization mechanism (projections), and the ability to optimize each side independently. CQRS *does not require* event sourcing — it can be implemented with a dual-table CRUD store and a change-data-capture (CDC) fan-out.
- **Event sourcing (ES)** — the aggregate's current state is *derived* by replaying an ordered, append-only log of domain events (the event store is the source of truth; the current-state document is a cache / projection). Enables: full audit trail, point-in-time replay, temporal queries, and decoupled consumers (events as integration primitives). Key consequences: projection lag (eventual consistency between event append and projection update), schema evolution complexity (old events must remain deserializable by new code), replay cost at scale (mitigated by snapshots at interval N), and a steeper cognitive and operational burden for a team accustomed to CRUD.
- **Snapshots** — at interval N events, a snapshot of the aggregate's current state is persisted alongside the event log so replay starts from the latest snapshot (not event 0). The snapshot interval N is a tuning parameter; at small event-log sizes (< ~1k events/aggregate for an enterprise internal tool) snapshots may add complexity without benefit — evaluate per driver.
- **Materialized read views / projections** — a read-side handler subscribes to the event stream (via an in-process event bus, a CDC subscriber, or a message broker) and updates a denormalized read model shaped for a specific query. Projection lag = time from event append to read-model update; this is the consistency window that must be stated and bounded against the scenario's response measure.
- **The outbox pattern** — to avoid the dual-write problem (write the aggregate state *and* publish the event in two separate I/O operations), persist the event to an outbox table/document in the same atomic write as the aggregate state, then use a CDC relay (or a polling publisher) to forward events to consumers. This satisfies the at-least-once delivery guarantee without a distributed transaction. Cross-reference `workflows/30-integration/event-driven-patterns.md` for the dual-write risk and the outbox/CDC contract.
- **Saga / process manager** — when a business operation spans multiple aggregates (e.g. an order submission triggers both an `Order` write and a related activity-log write), a saga orchestrates the steps with explicit compensation on failure. CQRS/ES does not eliminate the need for sagas; it sharpens the aggregate boundaries that make sagas necessary. Cross-reference `workflows/20-data/consistency-transactions.md` for the saga/outbox decision.
- **Event schema evolution strategies**: (1) versioned event types with explicit upcasters (`OrderSubmitted_v1` → `OrderSubmitted_v2` via an upcaster function); (2) schema-registry contract (Avro/Protobuf with a compatibility rule — BACKWARD, FORWARD, or FULL); (3) tolerant reader / Postel's Law (consumers ignore unknown fields — simple but hides contract drift). Strategy 1 or 2 is required when the event store outlives a single deploy cycle; strategy 3 alone is insufficient for a durable event log.
- **CAP/PACELC applied to CQRS/ES** (`shared/fallacies-and-tradeoffs.md`): the write side of CQRS (the event store or write collection) targets CP (strong consistency per aggregate, single-writer); the read side (projections) trades consistency for latency (PA/EL in PACELC terms). Both choices must be stated and traced to the scenario that justifies them.
- **AKF scale cube** — CQRS is an X-axis (replication) + Y-axis (functional decomposition of reads from writes) pattern; it does not address Z-axis partitioning (sharding by tenant/customer). For an enterprise-scale internal tool (not consumer-scale) per `project-architecture-config.md`, read/write decomposition is unlikely to be the correct scale lever — confirm the scale scenario first (per the locked rule in `project-architecture-config.md`: do not design for a high-RPS target without a driver).
- **ATAM evaluation** — score the CQRS/ES option against each `(H,H)` scenario in the utility tree; build a sensitivity point analysis (e.g. "projection p95 is *sensitive* to the projection-fan-out: if 5 projections update on every event, the lag budget is split 5 ways") and tradeoff points (e.g. "event sourcing is a tradeoff point: it *improves* auditability/temporal-query scenarios but *worsens* operational complexity, schema-evolution surface, and query flexibility for ad-hoc reads"). Per `shared/evaluation-method.md`.

## Assertions & exit gate

- If CQRS/ES is chosen: every command is scoped to one aggregate boundary; every event is past-tense and carries the aggregate ID + version; every projection has a named rebuild mechanism and an estimated rebuild time at current log size.
- If CQRS/ES is chosen: the read/write consistency lag is stated, bounded per a named scenario response measure, and accepted as within that measure.
- If CQRS/ES is chosen: the event schema evolution strategy is named (versioned types + upcasters, schema-registry compatibility rule, or tolerant reader with documented limitations) and applied to every event type.
- If CQRS/ES is chosen: every cross-process call introduced (event-bus publish, projection subscriber, outbox relay) has a bounded timeout, retry policy (idempotent + backoff/jitter), and a fallback or fail-fast — per the 8-fallacies hard oracle (`shared/fallacies-and-tradeoffs.md`).
- If CQRS/ES is rejected: the ADR records the simpler CRUD choice, the drivers that were evaluated and found insufficient to justify CQRS/ES, and the two alternatives rejected — it is an explicit right-sized decision, not a gap.
- Every write path (event-append or CRUD) declares its consistency model (strong / eventual) and mechanism (single-document atomic / saga+outbox / application-level transaction) — per `workflows/20-data/consistency-transactions.md`.
- Every significant decision (the CQRS adoption choice; the ES adoption choice; the projection mechanism; the snapshot interval if ES chosen; the event-schema evolution strategy) is an ADR with context · ≥2 options · decision · consequences (positive AND negative). One-way-door decisions (event sourcing — expensive to reverse once events accumulate) are flagged as such.
- Every projection / materialized read view traces to ≥1 named query scenario in the utility tree; no projection exists without a named consumer query that justifies it.
- Diagrams-as-code (C4 Container view of the command/query split; dynamic sequence diagram of the event-append → projection-update → query path) compile without error per `shared/diagramming-standards.md`.
- **Gate:** hard oracles green (consistency lag stated + bounded; projections rebuildable; schema evolution handled; commands aggregate-scoped; cross-process failure modes defined; ADRs complete with alternatives; diagrams compile; every decision traces to a `QAS-*`) AND (design) rubric mean ≥ 0.8 (CQRS/ES used only where a driver justifies the complexity; conceptual integrity between commands/events/projections; right-sized scope; evolvability seam for deferred integrations where applicable).

## Guardrails

Per `shared/guardrails.md`:

- **Propose & document — never write product code.** The data architect authors ADRs, diagrams-as-code, event-schema definitions, and projection-rebuild specs. It does not write API route handlers, event schema implementations, database projection update queries, or any file under `src/**` / `frontend/**` / `backend/**` (per the source layout in `project-architecture-config.md`). The controller implements the chosen design.
- **Ground every decision in a driver — especially the adoption decision.** CQRS and event sourcing are not default choices; they are complex patterns with a real complexity tax (projection lag, schema evolution, rebuild operations, dual model maintenance). The adoption decision must trace to a `QAS-*` scenario that demands the capability (read/write asymmetry, auditability, integration-event production). No driver at `(H,*)` priority → reject and record that as the ADR. The evaluator penalizes CQRS/ES applied without a justifying driver, per the simplicity oracle in `shared/quality-attribute-rubric.md`.
- **Consistency model is not optional.** Event sourcing imposes eventual consistency between the event-append (write side) and the projection (read side). This is not a gap to ignore; it is a design choice to state and bound against the scenario. Route the consistency-model decision to `workflows/20-data/consistency-transactions.md` — do not duplicate that oracle here.
- **Cross-reference siblings — do not duplicate their oracles:**
  - Consistency model + saga/outbox decisions → `workflows/20-data/consistency-transactions.md`
  - Event-driven integration and dual-write risk → `workflows/30-integration/event-driven-patterns.md`
  - Stream processing (batch vs stream, watermarks, exactly-once) → `workflows/20-data/streaming-data.md`
  - Access-pattern inventory and aggregate boundary source of truth → `workflows/20-data/data-modeling.md`
- **The 8 fallacies apply to every cross-process boundary CQRS/ES introduces.** The event bus is not a reliable in-memory call; the projection subscriber is not guaranteed to keep up; the outbox relay is not zero-latency. Every one of these boundaries gets a timeout, retry, fallback, and a failure-mode path — or the fallacies hard oracle blocks.
- **ES is a one-way door; flag it.** Once events accumulate in an event store and consumers build on the event schema, reversing the decision is expensive (all history must be migrated or discarded). The ADR must flag this, and the decision must be justified by a driver that demands the capability at the current phase of the system — not speculative future-proofing.
- **Record the tradeoff and the rejected alternatives on every decision.** The canonical tradeoffs that apply here — Consistency ↔ Availability/Latency (the projection lag), Coupling ↔ Autonomy (the event bus decouples writers and readers but adds operational complexity), Performance ↔ Simplicity (materialized views speed reads but add invalidation/rebuild complexity) — are named and the chosen point on each curve is stated, per `shared/fallacies-and-tradeoffs.md`.
- **One writer preserves conceptual integrity.** The `data-architect` is the sole writer for the CQRS/ES artifacts. Proposer and evaluator agents are read-only. A second writer (e.g. `adr-author`) runs only sequenced, never concurrent, on a non-overlapping artifact.
