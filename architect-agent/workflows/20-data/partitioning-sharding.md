# partitioning-sharding

**Group:** 20-data  ·  **Runs as:** subagent: ../.claude/agents/data-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Decide how data is partitioned and sharded across nodes, which replication strategy governs reads and failure tolerance, and how hot spots are avoided before they silently collapse a performance or availability scenario. Answers: *is the partition key the right one — does it distribute load evenly, support the dominant queries, and survive a celebrity-key event — and is the replication consistency model explicitly chosen and bounded?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack (datastore, topology, and connection details as configured — e.g. a single-primary document store, single node), deployment target, scale/SLO targets, compliance regime (as declared — e.g. PII or other regulated data), constraints (stack lock — no new datastore without a driver-justified ADR; boring-tech-preferred), and the `architecture/<slug>/` artifact location.
- The architecture **drivers**: the prioritized quality-attribute scenarios (utility tree) from `workflows/00-drivers/quality-attribute-scenarios.md` and the `architecture/<slug>/quality-attributes.md` it produces. **Do not enter this flow without a utility tree** — partitioning decisions are meaningless without knowing which query pattern dominates, what the throughput / latency SLOs are, and what the data volume growth driver is. If the utility tree is absent, surface that as an open driver question and downgrade scope.
- The **data model and access-pattern inventory** from `workflows/20-data/data-modeling.md` — the partition key must serve the dominant queries; the access-pattern inventory is the primary input that determines whether a key distributes evenly or creates a hot shard. Run data-modeling first if its output is not available.
- The **scalability drivers** from `workflows/40-cross-cutting/scalability-architecture.md` — the AKF Z-axis (data partitioning) is the *last* resort after X-axis (horizontal clone) and Y-axis (functional decomposition); confirm that X/Y scaling has been considered and rejected or is insufficient before recommending sharding.
- **Preconditions:** `project-architecture-config.md` read; the utility tree (or stub) available; the access-pattern inventory from data-modeling available; current node topology, estimated data volume per collection, and projected growth rate read from the codebase and the source-of-truth docs (BRD / PRD / understanding log) named in `project-architecture-config.md`. A flow with no access-pattern inventory or no growth driver **downgrades to audit-only** and flags the gap explicitly.

## Oracle (source of truth)

The **partition key choice is justified against the access-pattern inventory and the distribution uniformity test** — not by intuition, not by a field's semantic meaning in isolation. The replication strategy's **consistency model is explicitly stated** (sync / async, quorum size, CAP/PACELC point) per write path. Both are evaluated against the utility tree's response measures.

- **hard:**
  - The **partition key is justified by the dominant query pattern and distribution uniformity**: the key must appear in the `WHERE`/filter of the most frequent high-priority queries (so they are shard-targeted, not scatter-gathered); the key's cardinality and value distribution must be demonstrated to be uniform — a key whose values are skewed toward a small set of "hot" values is a hot-spot failure of the hard oracle. Evidence must show the distribution (histogram, cardinality estimate, or a capacity model), not assert it. A key justified only by "it makes sense semantically" blocks.
  - **Cross-partition queries and cross-shard transactions are identified and bounded**: every query that cannot be served by a single shard (a scatter-gather fan-out) is enumerated; each carries its fan-out factor and the latency / consistency cost. An unidentified cross-partition query on a high-priority latency scenario blocks. Cross-shard transactions (ACID across shards) require a named mechanism (two-phase commit / saga + compensation / outbox) or are explicitly prohibited as a design constraint.
  - **The replication consistency model is stated per write path**: for every write path (e.g. core-entity creation, status transition, audit-log append, payment/settlement write), the design declares: sync or async replication, quorum size (if quorum-based), CAP/PACELC point chosen (consistency vs latency vs availability under partition), and RPO consequence. "Replication is on" with no stated model blocks.
  - **Rebalancing does not require downtime**: the rebalancing strategy (how shards are moved when nodes are added/removed — chunk migration, consistent hashing ring, directory updates) must not require application downtime or a global lock that defeats an availability scenario. A rebalancing approach with unacceptable downtime vs the availability SLO blocks.
  - **Hot-spot / celebrity-key avoidance is designed-in**: if the key distribution analysis reveals a potential hot shard (e.g. a time-based key that funnels all writes to the "current" shard; a tenant-ID key where one tenant accounts for >20% of writes), a mitigation is named and an ADR is authored (compound key, synthetic randomization prefix, hash of the natural key, write-time fan-out). An unmitigated hot-spot on a throughput or latency driver blocks.
  - **Every ADR for a partitioning/replication decision is complete**: context · ≥2 options · decision · consequences (positive AND negative). The `Driver refs` field traces to a `QAS-*` in `architecture/<slug>/quality-attributes.md`. An ADR with no alternatives or no consequences blocks.
  - **Response-measure met** (rubric: `quality-attribute-rubric.md` — "a scenario with no met measure blocks"): every prioritized QAS that this flow touches — throughput (e.g. sustained write RPS across shards), read latency (e.g. p99 ≤ X ms for shard-targeted queries vs scatter-gather fan-out), and availability (e.g. RTO/RPO target) — must be demonstrably satisfied by the capacity model in `architecture/<slug>/capacity-model.md`. The capacity model must work through the arithmetic: writes-per-shard vs shard throughput ceiling, per-shard p99 vs the QAS p99 after fan-out multiplication, replication-lag bound vs the read-staleness tolerance. A QAS whose numeric response measure is asserted but not traced to a capacity-model calculation blocks. If the utility tree response measures are unstated, surface that as an open driver question and block on it — do not assume numbers.
  - **No un-accepted SPOF on an availability-SLO path** (rubric: `quality-attribute-rubric.md` — "silent SPOFs block"): every component on the critical path of an availability scenario must have explicit redundancy/failover or be a recorded accepted risk. For this flow that means: (a) the **query-router / proxy tier** (if the chosen datastore uses one) — a single router instance is a SPOF for all application-layer database traffic; the design must deploy ≥2 router instances behind a load-balancer or document the accepted-risk entry; (b) the **metadata/config server tier** (if the chosen datastore uses one) — a config-server quorum with fewer than 3 members, or whose loss halts chunk migration and routing-table reads, is a SPOF; the design must specify a properly-sized quorum or record the accepted risk; (c) the **existing single-primary node** (current stack: per `project-architecture-config.md`) — a single node with no replica set is a SPOF against any availability scenario; the design must either introduce a replica set or log the explicit accepted-risk entry (including the blast radius and RTO consequence). A sharding proposal that introduces a router or config-server tier without addressing their redundancy blocks.
  - **Data lifecycle lawful for sharded PII** (rubric: `quality-attribute-rubric.md` — "an unclassified-PII store under a regulated regime blocks"; compliance regime: as declared in `project-architecture-config.md`, e.g. PII or financial data): distributing regulated records (e.g. subject identity, ownership/assignment fields, financial amounts) across shard nodes introduces new obligations — data residency (which shard node holds which subject's record), retention (TTL or purge jobs must locate and delete records across all shards, not just one node), and erasure (a right-to-erasure request requires locating all shard nodes that hold the subject's data, not a single-collection delete). The design must: (i) classify every collection by sensitivity before a partition key is chosen; (ii) demonstrate that the shard key does not prevent targeted erasure (a hash shard key on `_id` distributes a single subject's records — verify that a subject-scoped delete can still be executed efficiently across shards or that the key choice enables it); (iii) confirm that shard node placement does not violate any data-residency constraint in the applicable compliance regime. An unclassified regulated-data collection proposed for sharding, or a shard key that makes subject-targeted erasure impractical, blocks.

- **soft:**
  - **Partitioning is introduced only when single-node cannot hold the data or load** (the AKF Z-axis is the last resort after X/Y). If the current data volume and projected growth fit comfortably on a single-node primary + read replica (with headroom for the SLO), recommending sharding is accidental complexity — flag it with the YAGNI soft-oracle and the simplicity score. The burden of proof is on introducing partitioning, not on avoiding it.
  - **The key choice is robust to skew**: beyond the mechanical distribution test, the evaluator must assess whether the key is *likely* to remain uniform over the data's lifecycle — seasonality, power-law access patterns (a single large tenant that accounts for 80% of records), or a data model where one partition key value accumulates unboundedly over time all produce future skew even if the current distribution appears uniform. A key that passes the current-distribution test but is fragile to skew scores lower on this dimension.
  - **The replication topology is right-sized to the availability driver**: multi-replica, multi-region, and leaderless quorum are progressively more complex and expensive; the chosen topology should be the simplest one that meets the stated availability SLO. Over-provisioning redundancy against an unstated availability driver is penalized as accidental complexity.

## Standards & techniques

- **Partitioning strategies** (choose by access pattern, not by convention):
  - *Range partitioning* — splits the key space into contiguous ranges (e.g. `created_date` ranges, `tenant_id` alphabetical ranges). Simple and supports range scans; high risk of hot spots when inserts are monotonically increasing (all writes land on the "last" range shard). Avoid as a primary partition key for append-heavy workloads.
  - *Hash partitioning* — applies a deterministic hash to the key before assigning to a shard; distributes uniformly by construction but destroys range-query locality (a range scan becomes a full scatter-gather). The canonical tradeoff: distribution uniformity ↔ range-query efficiency. Use when writes dominate and point lookups / equality filters are the dominant read pattern.
  - *Directory / lookup partitioning* — a routing table maps each key (or key range) to a specific shard; maximally flexible and allows manual rebalancing or tenant isolation, but the routing table is a SPOF unless it is itself replicated and highly available. Use when tenant isolation or geography-driven placement is a hard constraint.
  - *Geo / zonal partitioning* — a variant of directory partitioning where the partition criterion is geographic region or compliance zone (data residency). Required when a regulatory driver (GDPR data residency, insurance jurisdiction) demands it; not required absent that driver.
  - *Compound keys* — prefix the natural key with a distribution element (e.g. `<bucket_id>_<timestamp>`, `<random_suffix>_<tenant_id>`). The standard hot-spot avoidance technique when the natural key is time-based or otherwise monotonically skewed; adds application-side key-assembly complexity and complicates range queries.

- **The partition-key choice (the make-or-break decision)**:
  The partition key is a one-way-door decision in most distributed datastores — it is expensive to change after data is in production (requires a full data migration). The decision must be treated as a one-way-door ADR. Evaluation criteria: (1) does the key appear in the filter of the dominant queries so those queries are shard-targeted? (2) is the key's cardinality high enough to distribute across N shards without collision? (3) is the value distribution uniform — neither time-skewed nor power-law-skewed? (4) does the key remain stable over the record's lifetime (a mutable partition key requires moving the record between shards on update)? (5) does the key avoid cross-shard joins for the dominant read patterns? All five must be answered with evidence (cardinality estimate, distribution histogram from sample data, or a worked access-pattern trace), not by assertion.

- **Replication models** (choose the model that meets the consistency scenario and name the CAP/PACELC point):
  - *Leader-follower (single-leader / primary-secondary)* — all writes go to one leader; followers replicate asynchronously (default) or synchronously. Async replication: low write latency, non-zero replication lag → stale reads on followers, non-zero RPO on leader failure. Sync replication: zero-RPO for one follower, higher write latency, follower unavailability blocks writes. Many datastores' replica sets are leader-follower with a configurable write-concern (e.g. "acknowledge on primary only" → async; "acknowledge on majority" → sync-to-quorum). State the write concern per collection/table and its RPO implication.
  - *Multi-leader (multi-master)* — multiple nodes accept writes; conflicts are inevitable and require a resolution strategy (last-write-wins, application-defined merge, CRDTs). Complexity cost is high; use only when write-side geo-distribution is a hard latency driver (writes in two regions) and the conflict surface is bounded. Not the default.
  - *Leaderless / quorum (Dynamo-style)* — writes and reads go to any node; consistency is controlled by quorum parameters (W + R > N). Tunable consistency: `W=1, R=1` → fast but stale; `W=majority, R=majority` → strong consistency but higher latency and reduced availability under node loss. PACELC point: on Partition, choose A (low W) or C (high W); Else (normal operation), choose L (low W+R) or C (high W+R). Name the quorum chosen and the scenario it satisfies.
  - *Replication lag as a design input*: async replication creates a lag window. For the project's domain, identify which read paths can tolerate stale data (e.g. dashboard summaries, audit-log queries) vs which require monotonic reads or read-your-writes consistency (e.g. entity status immediately after a write, an RBAC-scoped list immediately after an assignment change). The read-concern level (e.g. "local", "majority", "snapshot" — per the chosen datastore's terminology) must be stated per read path.

- **Rebalancing**: adding or removing shards requires moving data. The two canonical approaches: (a) *fixed partition count with reassignment* (Elasticsearch, pre-split MongoDB) — the total number of logical partitions is fixed at design time; nodes own subsets of partitions; adding a node moves whole partitions (no single-key move). Simple, but the partition count must be chosen large enough at creation to avoid repartitioning later. (b) *Consistent hashing with virtual nodes* (Cassandra, DynamoDB) — the hash ring is divided into many virtual nodes; each physical node owns a slice; adding a node moves a proportional slice with minimal data movement. More complex to operate; rebalancing is continuous and automatic. For a sharded document store, chunk migration is a common mechanism: the balancer moves chunks between shards; the application sees no downtime but chunk migration consumes I/O — size chunks and balance windows to avoid contention with peak write load.

- **Hot-spot / celebrity-key avoidance** (the operational failure most often encountered after launch):
  - *Time-based key hot spot*: `created_at` or `_id` (a monotonically-increasing surrogate key) as the shard key sends all new inserts to the "last" shard. Mitigation: hash the `_id`, or use a compound key `(tenant_id, _id)` that distributes writes across tenants. A hashed shard key on `_id` is the canonical fix.
  - *Celebrity / power-law key hot spot*: a single `tenant_id` that accounts for a disproportionate share of records funnels all its writes to one shard. Mitigations: (a) compound key `(tenant_id, record_id_prefix)` where `record_id_prefix` is a random bucket `[0..B)`; (b) fan-out write at application level (the celebrity tenant writes to B buckets, reads merge B results); (c) directory partitioning with a dedicated shard for the celebrity tenant (the "silo isolation" pattern from `workflows/40-cross-cutting/scalability-architecture.md`). Each carries a different application complexity cost — record in an ADR.
  - *Cardinality floor*: the shard key's distinct value count must exceed the number of shards by at least an order of magnitude, otherwise shards cannot be evenly filled. A binary or boolean shard key (e.g. `is_active`) is a cardinality failure.

- **AKF Scale Cube** (the right-sizing framework — cite before recommending sharding):
  - *X-axis*: horizontal cloning — run N identical instances behind a load balancer. Stateless application tier; shared data store. Cheapest and simplest; handles throughput and availability but not data size.
  - *Y-axis*: functional decomposition — split by bounded context (e.g. an orders service, a payments service, a users service). Data is partitioned by domain ownership, not by key range. Reduces coupling and allows independent scaling. Appropriate when `decomposition-strategy` (`workflows/10-styles-decomposition/decomposition-strategy.md`) has already justified splitting.
  - *Z-axis*: data partitioning / sharding — split the *same* data horizontally by a key. Handles data volume that exceeds a single node; the most complex axis. **Introduce only after X and Y have been evaluated and found insufficient**. For a system operating at modest internal-tool scale, X-axis read replicas often satisfy the read-latency scenario before Z-axis is needed — the flow must make this assessment explicit against the actual scale drivers in `project-architecture-config.md`.

- **CAP / PACELC** (from `shared/fallacies-and-tradeoffs.md`): every replication configuration is a choice on the CAP triangle under partition and on the latency/consistency curve during normal operation. State the chosen point and the scenario it serves; "we chose eventual consistency" without naming the bounded staleness window or the read-path that tolerates it is not a stated model.

- **ATAM / utility tree**: every partitioning and replication option is scored against the prioritized quality-attribute scenarios from `workflows/00-drivers/quality-attribute-scenarios.md`. Sensitivity points (e.g. "p99 list-records latency is *sensitive* to whether a scatter-gather fan-out is required: a non-shard-targeted query on 8 shards multiplies latency by ~8x at the same per-shard p99") and tradeoff points (e.g. "hash shard key on `_id` is a tradeoff point: distributes writes uniformly / destroys range-query locality for time-window reports") are the deliverable of the evaluate step.

- **Distributed-systems patterns**: the named patterns from `workflows/80-patterns/distributed-systems-patterns.md` that apply here — consistent hashing, virtual nodes, quorum reads/writes, sloppy quorum + hinted handoff (Dynamo), read repair, vector clocks / version vectors (for multi-leader conflict detection), the saga + outbox pattern for cross-shard writes that must be atomic at the business level.

## Assertions & exit gate

- The partition key (if sharding is introduced) satisfies all five criteria: (1) present in the filter of dominant queries, (2) cardinality > 10× shard count, (3) distribution is demonstrably uniform (evidence: histogram or capacity model), (4) key is immutable or a migration strategy for key changes is authored, (5) dominant read patterns are shard-targeted, not scatter-gather.
- Every cross-partition / cross-shard query is enumerated with its fan-out factor and the latency impact quantified against the relevant QAS response measure; no unidentified scatter-gather on a high-priority latency scenario.
- Every write path declares its replication consistency model (sync/async, write concern / quorum size, CAP/PACELC point chosen) and the RPO consequence of that choice.
- Rebalancing strategy is stated; the data-movement window is compatible with the availability SLO (no required downtime exceeding the SLO's RTO).
- Hot-spot / celebrity-key risk is assessed for every candidate key; any identified hot spot has a named mitigation with an ADR; no unmitigated hot spot on a throughput or latency driver.
- The AKF Z-axis assessment is explicit: X-axis (replica set, horizontal clone) and Y-axis (functional decomposition) adequacy are evaluated before sharding is recommended; if single-node + read replicas satisfy the utility tree's response measures, sharding is deferred and the rationale is recorded.
- Every partitioning/replication ADR is complete: context · ≥2 options · decision · consequences (positive AND negative) · one-way-door flag for the partition key. Every ADR's `Driver refs` traces to a `QAS-*` in `architecture/<slug>/quality-attributes.md`.
- The C4 deployment diagram showing shard/replica topology compiles without error and is consistent with the ADRs and design doc.
- **Gate:** hard oracles green (key justified by access patterns + distribution evidence; cross-partition queries enumerated; consistency models stated; rebalancing downtime bounded; hot spots mitigated; ADRs complete; diagram compiles) AND (design) rubric mean ≥ 0.8 (Z-axis introduced only when X/Y insufficient; key is skew-robust; topology right-sized to the availability driver; simplicity rewarded over over-engineering).

## Guardrails

Per `shared/guardrails.md`:

- **Propose & document — never write product code.** The data architect authors ADRs, the partitioning design doc, the capacity model, and diagrams-as-code. It does not write shard-configuration commands, migration scripts, query-router config, or any file under `src/**` / `frontend/**` / `backend/**`. The controller/operator applies the plan.
- **Ground every decision in a driver.** The partition key must trace to a named query in the access-pattern inventory. The replication model must trace to a named consistency or availability QAS. A choice no driver demands is accidental complexity — flag it, do not recommend it. If the data volume and throughput drivers are unstated, surface them as open questions; do not assume sharding is needed.
- **AKF Z-axis is the last resort** — assert X-axis (read replicas, horizontal clone) and Y-axis (functional decomposition) adequacy before recommending sharding. For a typical internal enterprise tool, the default posture is: right-size, avoid sharding unless a concrete volume or throughput driver justifies it. The simplest topology that meets the utility tree's response measures wins.
- **The partition key is a one-way door** — treat it with the rigor a one-way-door decision demands: enumerate ≥2 options, evaluate against all five criteria, record the distribution evidence, and flag the migration cost of changing it. Do not recommend a key informally.
- **Reuse named patterns** — consistent hashing, virtual nodes, quorum reads/writes, sloppy quorum + hinted handoff, vector clocks, saga + outbox — from `workflows/80-patterns/distributed-systems-patterns.md` before inventing bespoke replication or rebalancing mechanisms. Named patterns carry known contracts and known failure modes.
- **Record tradeoffs and rejected alternatives** on every ADR. The canonical tradeoffs here — distribution uniformity ↔ range-query efficiency (hash vs range key), consistency ↔ latency/availability (sync vs async replication, PACELC), simplicity ↔ scalability (no sharding vs sharding), hot-spot avoidance ↔ application complexity (compound key / bucket prefix) — are always present; name the chosen point on each curve.
- **Cross-reference siblings** — storage engine and datastore selection decisions flow to `workflows/20-data/storage-selection.md`; consistency model and saga/outbox patterns for cross-shard writes flow to `workflows/20-data/consistency-transactions.md`; AKF axis and horizontal-scaling strategy flow to `workflows/40-cross-cutting/scalability-architecture.md`; distributed-systems patterns (consistent hashing, quorum, CRDT, vector clocks) are sourced from `workflows/80-patterns/distributed-systems-patterns.md`. Do not duplicate their oracles; reference the paths.
- **One writer preserves conceptual integrity.** The `data-architect` is the sole writer for the partitioning artifacts. Proposer and evaluator agents are read-only (no `Edit`/`Write` tool access).
- **Non-destructive and safe.** Read-only on any live environment. Never run `sh.shardCollection()`, `rs.add()`, chunk-migration commands, or any shard-topology change against a real database. The data architect produces the design plan; the operator applies it.
