# data-modeling

**Group:** 20-data  ·  **Runs as:** subagent: ../.claude/agents/data-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Model the data from conceptual to logical to physical — entities, aggregates, relationships, and invariants — and decide normalization vs denormalization per concrete access pattern. Answers: *which consistency boundary owns which data, and what shape does the storage model take given the actual read/write mix?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack (data-store model and topology, core collections/tables), deployment target, compliance regime (PII + any regulated financial or sensitive data; RBAC scoping), constraints (stack lock — no new datastore without a driver-justified ADR), and the `architecture/<slug>/` artifact location.
- The architecture **drivers**: the prioritized quality-attribute scenarios (utility tree) from `workflows/00-drivers/quality-attribute-scenarios.md` and the `architecture/<slug>/quality-attributes.md` it produces — every modeling decision traces to a scenario here; if no utility tree exists this flow creates a stub and surfaces each missing measure as an open driver question. Functional scope confirms the read/write access patterns (core entity lifecycle transitions, RBAC-scoped list queries, audit/append-only writes, any regulated-data writes) — these are the primary inputs that *drive* the model, not copied-from-default schema.
- The **domain analysis** (bounded contexts, ubiquitous language, context map) from `workflows/00-drivers/domain-analysis.md` — aggregates are drawn on DDD consistency boundaries, not on table convenience; run domain-analysis first if it has not been authored.
- **Preconditions:** `project-architecture-config.md` read; the utility tree (or stub) available; the existing data model (collections/tables, key documents/records, indexes) read from the codebase per the source locations named in `project-architecture-config.md`; a list of the concrete queries/mutations the system issues (the access-pattern inventory). A flow with no access-pattern inventory downgrades scope and says so.

## Oracle (source of truth)

The **access-pattern inventory** (the concrete queries and write paths the system issues, drawn from the codebase and the utility tree) and the **DDD aggregate contract** (an aggregate is the unit of transactional consistency, sized to protect one invariant per write). Every modeling decision must be justified by one of these; a schema choice with neither a traced access pattern nor a consistency-boundary reason is a preference, not a finding.

- **hard:**
  - Every aggregate has an **explicit consistency boundary** — the set of entities it owns in a single transaction — and the **invariant it protects** is named (e.g. "order aggregate protects: order-number is unique per tenant-year; status transitions follow the allowed FSM; all line-items belong to exactly one order"). An aggregate with no named invariant blocks.
  - The **model is justified by the access patterns**, not copied from a default or imported from a relational habit. For each collection / document shape, the dominant read query and dominant write mutation are named; the shape serves them. A schema choice with no traced access pattern blocks.
  - Every **denormalization is a recorded tradeoff** (read speed vs update-anomaly risk + the concrete update cost). A denormalized field with no ADR entry — stating the read pattern it serves and the update-anomaly surface it creates — blocks.
  - Every **write path declares its consistency model** (strong / eventual / causal) and the mechanism (single-document atomic write / application-level saga + compensation / outbox). "It's consistent" with no model blocks. (Consistency model hard oracle is shared with `workflows/20-data/consistency-transactions.md` — cross-check.)
  - **PII is classified** at the field level: each field carrying end-user PII or other regulated data is tagged; residency, retention, and erasure paths exist (or are flagged as an open compliance gap) per the regime (`project-architecture-config.md`). An unclassified-PII field in a regulated collection blocks.
  - Every **ADR for a non-trivial modeling decision** (aggregate boundary, denormalization, normalization breaking an access pattern, a new index) is complete: context · ≥2 options · decision · consequences (positive AND negative). An ADR with no alternatives or no consequences blocks.

- **soft:**
  - The model **fits the domain** — aggregate names align with the ubiquitous language from `workflows/00-drivers/domain-analysis.md`; bounded contexts have clear data ownership (no collection shared by two bounded contexts without an explicit interface or a duplication-with-sync pattern recorded in an ADR).
  - **No premature denormalization** — a field is not embedded/duplicated unless a concrete access pattern justifies it and the update-anomaly cost is acceptable at the current scale. The temptation to "flatten everything" for a document store is not a justification.
  - **No over-normalized chatty model** — relations are not artificially split into separate documents if the dominant access is always joint and the consistency requirement is single-transaction. Joins in a document store are application-level scatter-gather: pay only when the decomposition earns it (independent lifecycle, large sub-document, or high-cardinality fan-out).
  - The model is **evolvable** — new fields can be added without rewriting existing documents (schema evolution strategy noted); bounded-context seams exist where future or deferred integration services will produce/consume data (the integration seam driver from `project-architecture-config.md`).

## Standards & techniques

- **Conceptual → logical → physical modeling** — three distinct passes: (1) *conceptual*: entities + relationships in domain language, independent of storage (Entity-Relationship or a domain diagram); (2) *logical*: normalized/denormalized structure independent of the specific database engine (tables / document shapes / key schemas); (3) *physical*: the actual collections/tables, field names, indexes, shard/partition keys, and row/document size constraints for the project's storage engine (per `project-architecture-config.md`). Never collapse these — a design that jumps from domain to physical skips the logical justification.
- **DDD aggregates** — an aggregate is a cluster of entities and value objects treated as a single unit for data changes (Evans, *Domain-Driven Design*). An aggregate root controls access; no external object holds a reference to an interior entity. The consistency boundary *is* the aggregate boundary — all invariants enforced within an aggregate are guaranteed by a single atomic write to its root. Aggregate sizing: as small as possible (one invariant ↔ one aggregate); resist the "god aggregate" that absorbs unrelated entities just for convenience.
- **Normalization (3NF) vs denormalization** — normalize to 3NF as the *default* (eliminate transitive and partial dependencies; reduce update anomalies); denormalize *deliberately* when a concrete read pattern justifies it and the update-anomaly surface is bounded. In a document store, denormalization (embedding) is the default for 1:1 and bounded 1:few relationships where the sub-document has no independent lifecycle; referencing (normalized) is correct for high-cardinality, independently-evolving, or large sub-documents. In a relational store, normalize by default and denormalize only where a proven hot read path justifies it. The canonical tradeoff — "Normalization ↔ Read performance" from `shared/fallacies-and-tradeoffs.md` — is named and recorded per denormalized field.
- **Access-pattern-drives-the-model rule** — for NoSQL especially: model the data around the queries, not around the entities in isolation (Fowler, *NoSQL Distilled*; DynamoDB single-table design principle generalized). The query set is read from the application code; the model serves the dominant patterns; secondary patterns get indexes. The inverse — model the entities first, then add indexes to compensate — produces chatty, slow, or over-scanned access paths.
- **Invariant identification** — for each aggregate, enumerate the business rules that must hold after every write: uniqueness constraints, cardinality limits, state-machine transitions, referential integrity within the boundary. These are the aggregate's contract; violations are bugs, not edge cases.
- **Index design** — compound indexes serve the most selective query predicates first (ESR rule: Equality → Sort → Range). Covering indexes eliminate collection/table scans for frequent read paths. Each index has a named read query it supports; indexes with no traced query are candidates for removal (write overhead with no read benefit).
- **Schema evolution** — schema evolution is an explicit strategy per the project's storage engine: for a schemaless document store, optional fields + default-on-read, versioned documents (a `_schema_version` field), or an explicit migration (backfill); for a relational store, versioned migrations with backward-compatible rollout (expand/contract). The strategy is chosen per collection/table and recorded; uncontrolled schema drift is an evolvability risk.
- **PII classification** — field-level tagging using the regime from `project-architecture-config.md`; retention windows and erasure paths are modeled as first-class fields (`deleted_at`, `anonymized_at`) or as a separate erasure-log record, not as an afterthought. Cross-reference `workflows/40-cross-cutting/privacy-compliance-architecture.md` for the LINDDUN threat model and data-residency driver.
- **ATAM / utility tree** — every modeling option is scored against the prioritized quality-attribute scenarios (e.g. performance: a key dashboard/list render under its target latency; compliance: PII erasure; evolvability: a deferred integration seam). The evaluator locates sensitivity points (e.g. "the primary list-query p95 is *sensitive* to the index coverage: a missing compound index flips a covered query to a full collection scan") and tradeoff points (e.g. "embedding child records in the parent document is a tradeoff point: improves read latency / worsens concurrent updates across multiple writers").

## Step sequence

- **audit:** Read `project-architecture-config.md` + the existing schema (the project's schema/model definitions and validators, per the source locations named in `project-architecture-config.md`) + the codebase's query patterns → reconstruct the de-facto access-pattern inventory → check each collection/table against the hard oracles (consistency boundary named? invariant stated? every denormalization recorded? every write path's consistency model declared? PII classified? ADRs complete?) → flag missing indexes for high-frequency read paths → emit findings + risks per `finding-schema.md`, grouped by `type:"data"` then severity. Read-only, no artifact authored, no source edited.

- **design:** Frame (read `project-architecture-config.md` + utility tree + domain-analysis output + existing schema; confirm or build the access-pattern inventory; identify open driver questions — e.g. RPO for core business data, retention window for audit records — and surface them; read-only) → Explore (≥2 divergent logical models — e.g. a fully-embedded parent document vs a normalized header + child-record references; a single collection vs partitioned-by-status; each as a concrete spec with its dominant query shapes, aggregate boundaries, index plan, and consistency model; read-only) → Evaluate (skeptical evaluator scores each option against the utility tree + hard oracles using ATAM; builds a decision matrix: scenarios × options; identifies sensitivity points and tradeoff points; orders swapped to kill position bias; picks a winner and grafts superior ideas from runners-up; read-only) → Document (one writer — `data-architect` — authors: `architecture/<slug>/design-doc.md` data-architecture section; ADRs per significant decision in `architecture/<slug>/adr/`; a C4-compatible Entity-Relationship / logical schema diagram as diagrams-as-code in `architecture/<slug>/diagrams/`; PII classification table in `architecture/<slug>/threat-model.md`; index plan in the design doc; schema-evolution strategy per collection/table) → Verify (fitness functions: aggregate-boundary check — no cross-aggregate reference without a root identifier; index coverage check — every query in the access-pattern inventory has a supporting index or a documented exception; PII-classification completeness — every collection/table with a PII field has an entry in the classification table; ADR completeness — every denormalization and aggregate boundary has a complete ADR; diagram compiles; evaluator re-scores from artifacts) → loop ≤ 8 or pass.

## Assertions & exit gate

- Every aggregate names its consistency boundary (the entity set it owns) and the invariant it protects.
- Every collection's document shape traces to ≥1 named access pattern; no field exists without a read or write justification.
- Every denormalized / embedded field has a recorded ADR entry: the read pattern it serves + the update-anomaly surface + the alternatives rejected (e.g. reference + application-join).
- Every write path declares its consistency model and mechanism (single-document atomic / saga / outbox); the mechanism is appropriate to the consistency scenario (cross-reference `workflows/20-data/consistency-transactions.md`).
- PII fields are classified at the field level; retention + erasure paths are named for each PII-bearing collection.
- Every modeling ADR is complete: context · ≥2 options · decision · consequences positive AND negative. Every ADR's `Driver refs` trace to a `QAS-*` in `architecture/<slug>/quality-attributes.md`.
- The logical-to-physical mapping (collection names, indexes, document shapes) is captured as diagrams-as-code that compiles without error.
- The access-pattern inventory is complete (all high-frequency queries and every mutation path are listed); no high-frequency query is served by a full collection/table scan without an accepted-risk entry.
- The model fits the bounded contexts from `workflows/00-drivers/domain-analysis.md`: no collection is owned by two bounded contexts without a recorded sharing pattern.
- **Gate:** hard oracles green (aggregate boundaries + invariants named; access patterns traced; denormalizations recorded; consistency models stated; PII classified; ADRs complete; diagram compiles) AND (design) rubric mean ≥ 0.8 (model fits domain, no premature denormalization, no chatty over-normalization, evolvable).

## Output

Write `artifacts/data-modeling/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (`type:"data"`; `oracle:"hard"` for: unnamed invariant, untraceable schema choice, unrecorded denormalization, unstated consistency model, unclassified PII, incomplete ADR; `oracle:"soft"` for: premature denormalization, over-normalized chatty access, poor domain fit, missing evolvability seam); the verification block (fitness-function results, traceability coverage, hard-oracle results, rubric score) for design mode; open driver questions listed explicitly (unstated RTO for core business data, retention window, availability target) for the controller to resolve.

Authored design artifacts land in `architecture/<slug>/` per `project-architecture-config.md`:

- `architecture/<slug>/design-doc.md` — data-architecture section: aggregate inventory, access-pattern inventory, index plan, schema-evolution strategy per collection/table.
- `architecture/<slug>/adr/NNNN-*.md` — one MADR ADR per significant decision (aggregate boundary, denormalization, consistency model choice, new storage technology if any).
- `architecture/<slug>/diagrams/data-model.puml` (or `.md` for Mermaid) — logical ER / aggregate diagram as diagrams-as-code; compiles without error.
- `architecture/<slug>/threat-model.md` — PII classification table (collection → field → PII category → retention → erasure path).

## Guardrails

Per `shared/guardrails.md`:

- **Propose & document — never write product code.** The data architect authors ADRs, the logical/physical schema design doc, and diagrams-as-code. It does not write model/schema classes, migration scripts, or any file under `src/**` / `frontend/**` / `backend/**`. The controller implements the chosen design.
- **Ground every decision in a driver.** No aggregate boundary without a named invariant it protects. No denormalization without a traced read pattern. No index without a traced query. A modeling choice with no driver is accidental complexity — flag it, don't recommend it.
- **Access pattern first, shape second** — especially for a document/NoSQL store. The access-pattern inventory is non-optional; a flow with no inventory downgrades to audit-only and flags the gap.
- **Reuse known patterns.** The embedded-document pattern, the bucket pattern (for time-series data), the computed-pattern (pre-aggregation), the schema-versioning pattern — reach for the named pattern catalog appropriate to the project's storage engine before inventing bespoke shapes. Each named pattern carries a known contract and known failure modes.
- **Record the tradeoff and the rejected alternatives** on every significant decision. An ADR with only upside is incomplete (`shared/adr-template.md`). The normalization ↔ read-performance tradeoff and the consistency ↔ latency tradeoff (`shared/fallacies-and-tradeoffs.md`) are always present in a data modeling decision; name the chosen point on each curve.
- **Cross-reference siblings** — consistency model decisions flow to `workflows/20-data/consistency-transactions.md`; storage technology decisions (if a driver justifies a new store) flow to `workflows/20-data/storage-selection.md`; PII/retention obligations flow to `workflows/40-cross-cutting/privacy-compliance-architecture.md`; bounded-context and ubiquitous-language source of truth is `workflows/00-drivers/domain-analysis.md`. Do not duplicate their oracles; reference the paths.
- **One writer preserves conceptual integrity.** The `data-architect` is the sole writer for the data-modeling artifacts. Proposer and evaluator agents are read-only (no `Edit`/`Write` tool access). A second writer (e.g. `adr-author`) runs only sequenced, never concurrent, on a non-overlapping artifact.
- **Non-destructive and safe.** Read-only on any live environment. Never run a migration, alter a collection, or drop an index against a real database. The data architect produces the *design* (the ADR, the index-creation script as a proposed artifact, the migration plan); the controller/operator applies it.
