# distributed-transaction-design

**Group:** 20-data  ·  **Runs as:** subagent: ../.claude/agents/data-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Choose and document the **distributed-transaction strategy** for every multi-service write that needs coordination — by applying the **8 transactional-saga taxonomy** (Richards & Ford, *The Hard Parts* ch. 12) to classify each write on three binary dimensions (communication × consistency × coordination), defaulting away from the worst (Horror Story: async + atomic + choreographed), and designing the compensating-transaction machinery, outbox/idempotency strategy, and boundary redraw where applicable. The design question this flow answers: *for each write that spans a service or storage-root boundary, which of the eight named patterns governs it, what compensations does that choice require, and can the boundary be redrawn so the write is local — eliminating the saga entirely?* The cardinal rule: **no saga beats a saga** — a boundary redraw that collapses a cross-service write into a single local transaction is always preferred over any saga pattern.

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack and its transactional primitives (e.g. a single backend service + a single document-store instance, no broker, no queue — a *single-document* update is atomic; a write spanning two documents/collections is not unless wrapped in a multi-document transaction on a replica set; if there is no replica set, multi-doc writes are a latent dual-write); the near-term bounded contexts where cross-service/cross-context writes will appear (e.g. documents, payments, closure — keep seams now, design the saga when the boundary is drawn); the compliance regime (PII + financial data — payment/settlement writes are typically the highest-value invariants requiring the most complete compensation design).
- The **drivers / utility tree** (`workflows/00-drivers/quality-attribute-scenarios.md`): the prioritized 6-part quality-attribute scenarios this decision must satisfy — specifically data-integrity scenarios (e.g. *QAS-DATA-01: no financial record is ever lost or double-applied; QAS-DATA-02: a sequence-counter mint never produces a duplicate*) and any responsiveness / availability scenario the saga pattern trades against (sync paths block; atomic variants add compensation cost). A multi-service write with **no stated business invariant** downgrades scope — you cannot classify a saga without knowing what the write must preserve.
- The **target write paths** under review or design: what mutates, across how many storage roots / services / external systems (document-store collections, an external payment processor, a future event channel), and the business invariant each must preserve. The consistency model for each path must already be stated or be produced by `consistency-transactions.md` before this flow runs — this flow picks the *pattern* for writes that `consistency-transactions.md` has classified as requiring a cross-boundary mechanism.
- Preconditions: the utility tree exists (else `status:error`); the data model and aggregate boundaries (`data-modeling.md`) are read (the aggregate is the unit of a local transaction; a write within one aggregate needs no saga); the existing write paths are enumerable from the codebase/schemas (audit reads, never edits).

## Oracle (source of truth)
The **business invariant** each coordinated write must preserve — expressed as a measurable, structurally-traceable contract — **and** the **named saga pattern from the 8-pattern taxonomy** (`../shared/transactional-saga-taxonomy.md`) that governs it. A design that describes a coordinated write without naming its pattern is a hard-oracle failure.

- **hard** (deterministic gate — traceable, measurable, or structurally checkable):
  - **Every multi-service write that needs coordination names its saga pattern** from the 8-pattern taxonomy (`../shared/transactional-saga-taxonomy.md`): one of *Epic Saga, Phone Tag Saga, Fairy Tale Saga, Time Travel Saga, Fantasy Fiction Saga, Horror Story, Parallel Saga, Anthology Saga* — classified by `{communication: sync|async, consistency: atomic|eventual, coordination: orchestrated|choreographed}`. A coordinated write with an **unnamed pattern** is a `critical` finding (`oracle:"hard"`, `type:"data"`).
  - **Every pattern choice carries its consistency, coupling, and recoverability ratings** as stated in the taxonomy — not asserted by the designer. The ratings are fixed by the pattern definition; misrepresenting them to make a pattern look better than it is is a critical finding.
  - **Horror Story (async + atomic + choreographed) requires an explicit, written ADR** stating: (a) why atomicity is a non-negotiable hard driver (traced to a `QAS-…` scenario or a regulatory obligation); (b) why async communication is required; (c) why a central orchestrator cannot be introduced. Absence of that ADR on a Horror Story landing is a `critical` finding (`oracle:"hard"`, `severity:"critical"`). This is a non-negotiable gate from the taxonomy.
  - **Compensating transactions are complete for every atomic variant** (Epic Saga, Phone Tag Saga, Fantasy Fiction Saga): every step that can fail has a documented compensating action that semantically restores the invariant (a refund, not a delete; a release-hold, not a rollback). An atomic-variant design with any saga step whose failure leaves the system in a permanent partial state is a `critical` finding.
  - **Idempotency is stated for every eventual variant and every retried/replayed write** (Parallel Saga, Anthology Saga, Fairy Tale Saga, Time Travel Saga): every participant names its idempotency key or its naturally idempotent operation (upsert, `$set`, monotonic counter) so at-least-once delivery does not become apply-twice. An eventual variant with no idempotency mechanism is a `major` finding (`oracle:"hard"`).
  - **The outbox pattern is the mechanism wherever a write to the primary store must also trigger a downstream event or service call.** A `db.save()` immediately followed by a `bus.publish()` or a service call (without an outbox in the same transaction) is a dual-write: a crash between the two leaves systems disagreeing, and a retry double-applies. This structural check is inherited from `consistency-transactions.md` and carries `critical` severity here when it appears in a saga context.
  - **Anthology Saga (async + eventual + choreographed) carries a saga-completion monitor** — a dedicated process or timeout that detects a stuck/incomplete saga and surfaces it for reconciliation. Absent a monitor, the saga has no authority to detect completion; this is a `major` finding (`oracle:"soft"` per the taxonomy, treated as a gate prerequisite here because an undetectable stuck saga is operationally equivalent to silent data loss).
- **soft** (craft — rubric-scored only after hard oracles are green):
  - The **simplest saga that holds the invariant is chosen** (right-sizedness): Parallel Saga (async + eventual + orchestrated) is the default; a design that selects a more complex variant (atomic, choreographed at scale) carries the burden of showing that a simpler one fails the invariant.
  - The **boundary is drawn — or redrawn — before accepting a saga** (the "no saga beats a saga" principle): where two things that must change together can be placed inside one aggregate without violating cohesion drivers, the saga is replaced by a local transaction. A saga introduced where the boundary redraw was feasible is a craft finding.
  - **Choreography is avoided above 3 participants** — the flow becomes implicit and non-observable; orchestration is preferred above that threshold.
  - **Sync patterns are avoided when async holds the invariant** — synchronous variants (Epic, Phone Tag, Fairy Tale, Time Travel) block the caller for the full saga duration and cascade failures; an async alternative that satisfies the same invariant is the better choice absent an explicit inline-response driver.

## Standards & techniques
- **The 8 transactional-saga taxonomy** (`../shared/transactional-saga-taxonomy.md`, Richards & Ford *The Hard Parts* ch. 12): the canonical classification grid — three binary dimensions (communication × consistency × coordination) yielding 8 named patterns. This is the only valid classification vocabulary in this flow; bespoke names are not accepted. Cite the pattern by name; carry its defined consistency/coupling/recoverability ratings.
- **Orchestration vs choreography** — orchestration (a central coordinator drives steps, owns the error path, is visible and testable) vs choreography (each service reacts to events from the previous participant, no central component, implicit flow requiring distributed tracing to observe). The orchestrated variants expose the full saga logic in one place; choreographed variants distribute it. For correctness-sensitive writes, the hidden complexity of choreography is a design debt that grows with participant count.
- **Atomic vs eventual consistency** — atomic saga variants enforce global commit-or-rollback semantics via compensating transactions; eventual variants accept transient partial states and converge via reconciliation or out-of-band recovery. Atomic is not free: compensating transactions must be designed, tested, and exercised in failure drills; an undocumented compensation is not a compensation. Eventual is not loose: idempotency and a reconciliation path are mandatory, not optional.
- **Sync vs async communication** — synchronous sagas block the calling thread/request for the duration of every participant hop; total latency = Σ(participant latencies), and a slow participant cascades to the caller. Asynchronous sagas decouple the caller from the saga execution via a durable channel; total latency = dispatch time + async fan-out, and participant slowness does not cascade upstream. Async requires durable messaging infrastructure and idempotent consumers.
- **Compensating transactions** (`../shared/transactional-saga-taxonomy.md`, `consistency-transactions.md`) — a *semantic* undo for each saga step: the operation that restores the system to the pre-step state if the step must be rolled back. Not a database rollback (which is not available cross-service) but a business-level reversal (void a charge, release a hold, cancel a booking). Compensations must be: pre-designed (not an afterthought), idempotent (can be retried), and ordered from the most recent committed step backward. The **pivot step** (the one that cannot be compensated — e.g. a funds transfer that has settled) must be identified and placed last in the sequence, or made idempotently retriable instead.
- **The outbox pattern** (`consistency-transactions.md`, `../30-integration/event-driven-patterns.md`) — writing the trigger event into an outbox collection in the **same local transaction** as the business state change on the primary store, then relaying it via a poller or CDC tailing the store's change log (e.g. an oplog or WAL). Converts a dual-write into a single atomic write + an at-least-once relay. The outbox entry carries the idempotency key; the consumer deduplicates on it. This is the standard mechanism for the "write state + emit event" atomic step inside a Parallel Saga or Anthology Saga participant.
- **Idempotency** — the contract that makes at-least-once delivery safe: applying a write more than once has the same effect as applying it once. Enforced via: an **idempotency key** (a unique token per operation, stored on first process, rejected on replay); a **naturally idempotent operation** (a `$set`/upsert with a monotone version field, a conditional update that checks state before mutating); or a **dedup store** (a processed-events log keyed by event id). Every saga participant in an eventual variant must implement one of these.
- **CAP / PACELC and the canonical tradeoffs** (`../shared/fallacies-and-tradeoffs.md`) — every pattern choice is a point on the CAP/PACELC curve per write path; name the point. Atomic variants buy consistency (C) and pay latency/availability; eventual variants buy availability/latency and pay transient consistency. These are not rhetorical — they are the measurable tradeoff the evaluator scores.
- **The 8 fallacies of distributed computing** (`../shared/fallacies-and-tradeoffs.md`) — synchronous saga patterns violate Fallacy 1 (reliable network) and Fallacy 2 (zero latency) hardest: every blocked hop is a cascading-failure surface. Name which fallacies the chosen pattern is most exposed to and how the design mitigates them (timeouts, dead-letter queues, circuit breakers).

## Step sequence
- **audit:** Frame the invariants from the utility tree (which writes are financial/order-grade, which are analytics/audit-grade) → enumerate every write path that spans more than one aggregate, service, or external system → for each, classify: *which of the 8 taxonomy dimensions apply, what is the current de-facto pattern?* → flag every coordinated write **without a named taxonomy pattern** (critical), every Horror Story landing **without a justifying ADR** (critical), every atomic variant **without documented compensating transactions** (critical), every eventual variant **without an idempotency mechanism** (major), every Anthology Saga **without a completion monitor** (major), and every sync variant **where async would satisfy the same invariant** (minor, advisory) → emit findings + risk register (read-only; authors nothing, edits no source).
- **design:** **Frame** (extract/confirm the per-path invariants + coordination-relevant scenarios from the utility tree; enumerate which write paths cross boundaries; read-only) → **Explore** (for each path, produce ≥2 divergent concrete alternatives — e.g. for a future order→payment coordinated write: *(A) redraw the boundary: move payment-status into the order aggregate → single local transaction on the primary store, no saga needed; (B) Parallel Saga: order-service orchestrator writes `PaymentRequested` to the outbox in the same transaction as order state-change, payment-service consumes and settles, compensating action = issue void/refund on payment-service, idempotency via `order_id+event_seq`; (C) Epic Saga: synchronous orchestrator blocks on payment-service inline, all-or-nothing, compensation = reverse each step if any fails* — each a real spec with its taxonomy classification, CAP/PACELC point, compensation design, and failure path; read-only) → **Evaluate** (the skeptical `architecture-evaluator` scores each against the invariant + hard oracles via ATAM: does the compensation actually restore the invariant at the pivot? where is the sensitivity point — the step with no compensation? the tradeoff point — orchestration decouples participants but introduces an orchestrator to operate? order-swapped to kill position bias; the simplest pattern that holds the invariant wins; Horror Story and sync variants carry an accidental-complexity penalty unless a driver demands them) → **Document** (the **one writer**, `solution-architect`, authors: the per-path saga-pattern table, the saga sequence diagram as diagrams-as-code, the outbox/idempotency design notes, and one MADR ADR per significant pattern choice with its rejected alternatives and the CAP/PACELC point) → **Verify** (the hard gates below as fitness functions + the evaluator re-scores from the authored artifacts) → loop ≤10 or pass.

## Assertions & exit gate
- **Every multi-service / cross-aggregate write that needs coordination has a row** in the saga-pattern table: `{write_path, invariant, pattern_name, communication, consistency, coordination, compensation_steps, idempotency_key, outbox_used, horror_story_adr_ref}`. No blank cells on architecturally-significant writes.
- **No coordinated write lands on Horror Story without a written ADR** — the ADR must answer (a) why atomic is a hard driver (traced to `QAS-…`), (b) why async is required, (c) why orchestration is impossible. Absence blocks.
- **Every atomic variant (Epic, Phone Tag, Fantasy Fiction) has a complete compensation map** — every saga step that can fail names its compensating action, and the pivot step (the one that cannot be compensated) is identified. An orphan step with no compensation or pivot designation blocks.
- **Every eventual variant (Parallel, Anthology, Fairy Tale, Time Travel) states an idempotency mechanism** per participant — named key, natural idempotent op, or dedup store. Absence of an idempotency strategy on any eventual participant blocks.
- **Every Anthology Saga names a saga-completion monitor** — a detection mechanism for stuck or incomplete sagas. Absence blocks.
- **No unresolved dual-write** — every operation that mutates the primary store and triggers a downstream call uses the outbox pattern (same-transaction outbox write + at-least-once relay) or is classified as an accepted risk with a reconciliation job in the risk register.
- **Every pattern choice is an ADR** (context · ≥2 options · decision · positive + negative consequences · CAP/PACELC point · driver refs pointing to `QAS-DATA-*`) and traces to a stated invariant; diagrams compile (`../shared/diagramming-standards.md`).
- **Simplest pattern wins** — any design choosing an atomic variant or a sync variant where the evaluator confirms an eventual/async alternative holds the invariant is a soft finding; if the writer cannot rebut it with a driver, it is upgraded to major.
- **Gate:** hard oracles green — **every coordinated write named in the taxonomy, no Horror Story without ADR, atomic variants with complete compensation maps, eventual variants with idempotency, no silent dual-write, Anthology sagas with completion monitors** — AND (design) rubric mean ≥ 0.8 (invariant-fit + compensation-completeness + pattern-simplicity).

## Output
Write `artifacts/distributed-transaction-design/report.json` per `../shared/report-format.md` — findings/risks per `../shared/finding-schema.md` (`type:"data"`; `oracle:"hard"` for an unnamed pattern, a Horror Story without ADR, an atomic variant without compensation map, an eventual variant without idempotency strategy, an Anthology Saga without a completion monitor, or a silent dual-write; `driver_ref` = the `QAS-DATA-*` invariant the finding serves; every finding carries its `tradeoff` and `alternatives_rejected`, ADR-shaped). The `verification` block lists the fitness-function results (taxonomy-classification-complete check, no-Horror-Story-without-ADR check, compensation-completeness check, idempotency-present check, no-dual-write structural check) + hard-oracle results + the rubric score. Design-mode authored artifacts land in `architecture/<slug>/` per `project-architecture-config.md`:
- The per-path **saga-pattern table** in `design-doc.md` or `tradeoffs.md`
- The **saga sequence diagrams-as-code** (showing happy path + failure path + compensation sequence) in `diagrams/`
- One **MADR ADR** per significant pattern decision in `adr/NNNN-*.md`, including the mandatory Horror Story ADR if that pattern is chosen
- Open questions (an unstated invariant for a pending bounded context, an undecided compensation strategy for the pivot step) listed explicitly for the controller — never silently assumed.

## Guardrails
Per `../shared/guardrails.md`: **propose & document — never write product code and never apply a migration, deploy a broker, or change a live datastore.** This flow authors the saga-pattern table, the ADRs, and the sequence diagrams; the controller implements the outbox, idempotency, and saga-orchestrator code; the operator deploys any infrastructure the pattern requires. Ground every pattern choice in a **driver** (the business invariant from the utility tree) — no pattern without an invariant it protects, no invariant without a named pattern that governs it. **Reuse the named 8 patterns** before describing a bespoke coordination mechanism; bespoke mechanism names that do not resolve to one of the eight are rejected as unnamed-pattern findings. Prefer the **simplest pattern that holds the invariant** and prefer a **boundary redraw that makes the write local** over any saga — a local transaction is always simpler than the simplest saga and should be evaluated first. Do **not** introduce a broker, CDC pipeline, or durable queue where there is no second system to write to yet — model it as a seam for future bounded contexts named in `project-architecture-config.md`. Record the **tradeoff and the rejected alternatives** on every decision (CAP/PACELC point, sync↔async, orchestration↔choreography, atomic↔eventual, boundary-redraw as the always-considered alternative), never just the winner. Read-only audit edits nothing; one writer preserves conceptual integrity.
