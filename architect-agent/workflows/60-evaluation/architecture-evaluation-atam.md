# architecture-evaluation-atam

**Group:** 60-evaluation  ·  **Runs as:** subagent: ../.claude/agents/architecture-evaluator.md  ·  **Mode:** audit + design  ·  **Default model:** opus

## Purpose
Run a full **ATAM** evaluation of an existing or proposed architecture against its utility tree: **scope by architecture quantum first**, present the architectural approaches, confirm/build the prioritized scenarios, analyze each high-priority scenario's approach against its **response measure**, then extract the **sensitivity points**, **tradeoff points** (via a per-scenario pass *and* a dedicated tradeoff-mining sweep), **risks/non-risks**, and cluster the risks into **risk themes** tied to business drivers. Answers the decision question every later gate inherits: *does this architecture demonstrably meet its `(H,*)` quality-attribute scenarios per quantum, and where are the points — including the ones no single scenario walk surfaces — where one attribute is being bought at another's expense?* This is the analysis flow behind the `architecture-review-checklist` gate — it produces the scorecard, not a yes/no.

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target, the stated SLO/scale targets, the compliance regime (e.g. PII/financial data, per the project), the team topology (Conway), the constraints (stack lock, right-size, boring-tech-first), and where the architecture description / ADR log / utility tree live (`architecture/<slug>/`).
- The **quantum map**: the independently-deployable units and their per-quantum utility sub-trees, produced by `quantum-identification.md` (`shared/evaluation-method.md` Step 0). ATAM scores each quantum against *its own* sub-tree, never a system-wide tree that averages distinct operational characteristics away. If no quantum map exists, run `quantum-identification.md` first (or, in audit mode, run its quantum-scoping check inline before scoring and record that as a preceding step in the report — do not score a multi-quantum system off one flattened tree).
- The **drivers** as a utility tree: the prioritized 6-part quality-attribute scenarios with response measures, produced by `../00-drivers/quality-attribute-scenarios.md`, partitioned per quantum. If the tree is missing or scenarios lack measures, ATAM cannot score — run/repair that flow first (or, in audit mode, restate the stated NFRs as scenarios and flag every "-ility" with no measure as a gap; do not invent numbers).
- Target: the architecture under evaluation — the design doc + C4 views + ADRs in `architecture/<slug>/` (design mode), or the as-built system scanned from the repo (audit mode). Preconditions: the source-of-truth docs (BRD / PRD / understanding log, named in `project-architecture-config.md`) read; the approaches to analyze are *visible* (a decision with no recorded approach can't be analyzed — surface it as a documentation gap that blocks scoring).

## Oracle (source of truth)
The **utility tree** — per-quantum, the prioritized scenarios with response measures — judged by ATAM analysis (`shared/evaluation-method.md`). The architecture is scored against the drivers it must satisfy, never against how clever the diagram looks. Split:
- **hard:** **quantum scoping is complete first** — every scenario is bound to the quantum (or the shared root) whose sub-tree it belongs to per `quantum-identification.md`, and no scenario is scored against a flattened, cross-quantum tree; every high-priority `(H,*)` scenario (and every `(H,H)` first), per quantum, has an identified **architectural approach** whose response measure is judged **met or unmet by a model, a named pattern's guarantee, or a fitness function — proven, never asserted** (an unmet measure is a `critical`/`major` finding); every **sensitivity point** (a single property that determines whether a scenario is met) is named with the scenario it controls; every **tradeoff point** (a decision that is a sensitivity point for ≥2 attributes pulling opposite ways — see `shared/fallacies-and-tradeoffs.md`) found either in the per-scenario walk *or* the **tradeoff-mining sweep** is named with both attributes and the direction each moves; every sensitivity point surfaced by the mining sweep that turns out to be a tradeoff the design makes **with no recording ADR** is itself a finding (an undocumented tradeoff is a documentation-traceability gap, not a pass); every **risk** carries `likelihood × impact`, the `QAS-*` scenario it threatens, and a **risk theme** tied to a stated business driver; **non-risks** (decisions confirmed safe given the drivers) are recorded, not just risks; coverage is complete — no `(H,*)` scenario left un-analyzed, and the mining sweep has run over *every* sensitivity point named across *all* quanta, not just the ones surfaced per-scenario.
- **soft:** evaluation effort concentrates on the `(H,H)` leaves (where the architecture lives or dies) and treats `(L,L)` as noise; the tradeoff points read as genuine *and the chosen point on each tradeoff curve is justified by a driver*, not asserted; where two approaches both meet the drivers, the **simpler** one is preferred and the over-engineered alternative is named as a non-risk-to-acquire (penalize accidental complexity, per the rubric's length/novelty bias guards).

## Standards & techniques
- **ATAM (SEI, the 9-step method)** — the spine of this flow: (0) **scope by architecture quantum** (`quantum-identification.md`; `shared/evaluation-method.md` Step 0) → (1) present ATAM → (2) present business drivers → (3) present the architecture/approaches → (4) identify architectural approaches → (5) **generate/confirm the utility tree** (per quantum) → (6) **analyze the approaches** per high-priority scenario → (7) brainstorm & prioritize scenarios (confirm coverage) → (8) re-analyze, including the **tradeoff-mining sweep** → (9) **present results**: the sensitivity points, tradeoff points (per-scenario + mined), risks, non-risks, and risk themes. This flow runs step 0 as its precondition and steps 4–9 against an already-seeded, per-quantum tree (step 5 falls back to `../00-drivers/quality-attribute-scenarios.md` when the tree is absent, and to `quantum-identification.md` when the quantum map is absent).
- **Quantum scoping as the entry gate** (`quantum-identification.md`; `shared/evaluation-method.md` Step 0) — before any scenario is analyzed, confirm (or run) the quantum map: one utility sub-tree per independently-deployable unit bounded by synchronous connascence (Richards & Ford). Score each `(H,*)` scenario against *its own* quantum's sub-tree; shared system-wide attributes (security posture, compliance) stay at the root. A scenario scored against the wrong quantum's characteristics (e.g. the batch quantum's relaxed latency budget applied to the public-API quantum) is a mis-scoped analysis, not a valid verdict.
- **Sensitivity / tradeoff / risk / non-risk vocabulary** (the ATAM analysis outputs) — *sensitivity point* = a property of one decision strongly correlated to one scenario's response; *tradeoff point* = a sensitivity point for two-or-more conflicting attributes; *risk* = a decision (or its absence) that threatens a scenario; *non-risk* = a decision confirmed safe. Defined in `shared/evaluation-method.md`.
- **Tradeoff-mining sweep** (`shared/evaluation-method.md` — ATAM evaluation) — after the per-scenario walk, run one dedicated pass over **every sensitivity point named across every quantum** (not just the ones the scenario-by-scenario walk already flagged as tradeoffs) looking for cross-attribute tradeoff *pairs* the linear scenario walk misses — a decision that's a sensitivity point for scenario A (already analyzed) and *also*, silently, for scenario B (never checked against it). This is an LLM pass explicitly scoped to hunt for undocumented tradeoffs: for each sensitivity point, ask "which *other* `(H,*)` scenarios, in this quantum or the shared root, does this same decision also affect, and in which direction?" A tradeoff surfaced this way that the design makes **with no ADR recording it** is a finding in its own right (a documentation-traceability gap, distinct from the tradeoff itself being good or bad).
- **CBAM (Cost-Benefit Analysis Method)** — the cost-weighted extension: when two analyzed approaches both meet the drivers, hand the tradeoff to `tradeoff-analysis-cbam.md` to quantify `benefit = Σ(scenario-utility × priority-weight)` vs `cost = build + run` and rank by ROI — so the report doesn't gold-plate (e.g. a multi-region option with higher utility but sub-1 ROI against the actual availability driver is rejected with a number, not a vibe).
- **The canonical tradeoffs you choose but never escape** (`shared/fallacies-and-tradeoffs.md`) — CAP/PACELC (consistency↔availability/latency), latency↔throughput, coupling↔autonomy, sync↔async, normalization↔read-performance, cost↔resilience. The evaluator uses this list as the *locator* for tradeoff points in both the per-scenario pass and the mining sweep: every place the design sits on one of these curves is a candidate tradeoff point to name.
- **The 8 fallacies of distributed computing** (`shared/fallacies-and-tradeoffs.md`) — any approach that *assumes a fallacy away* on a `(H,*)` scenario's path is a risk, regardless of how the measure scores on the happy path.
- **Risk theming → premortem** — risks that cluster (e.g. "no coherent failure-handling strategy across the core entity's write paths") become a risk theme; hand systemic themes to `risk-storming-premortem.md` for likelihood×impact placement on the risk matrix and a pre-mortem on the highest-impact theme.

## Step sequence
- **audit:** **Scope by quantum** (confirm or run `quantum-identification.md` — one utility sub-tree per independently-deployable unit; system-wide attributes stay at the root) → Frame the business drivers + confirm the per-quantum utility tree (restate stated NFRs as 6-part scenarios; flag measureless "-ilities" as gaps, never invent) → for each `(H,*)` scenario, per quantum, walk the as-built architecture, identify its approach, and judge met/unmet against the response measure **by a capacity model / a named pattern's guarantee / a fitness function** (assertion is not analysis) → mark sensitivity points, tradeoff points (against the canonical-tradeoff list + the fallacies), risks (with likelihood×impact + threatened scenario), and non-risks → **run the tradeoff-mining sweep**: re-walk every sensitivity point named across all quanta and ask which *other* `(H,*)` scenarios the same decision affects that the per-scenario pass didn't check it against; flag any mined tradeoff with no recording ADR as a finding → cluster risks into themes tied to business drivers → emit the scorecard as findings + a risk register. Read-only: authors no artifact, edits no source.
- **design:** Frame (confirm the quantum map + the tree + the approaches under evaluation, read-only; run `quantum-identification.md` first if absent) → Explore (only when evaluating *competing* directions: score each option against every `(H,*)` scenario per quantum, building a decision matrix — does it meet the measure, at what tradeoff; read-only) → Evaluate (the ATAM analysis above, including the tradeoff-mining sweep; **order-swapped** when comparing options to kill position bias; pick a winner and graft the runners-up's superior approaches where they don't break conceptual integrity) → Document (the **one writer** `solution-architect` folds the analysis into `architecture/<slug>/tradeoffs.md` — the per-quantum scenario×approach scorecard, the sensitivity/tradeoff points (per-scenario + mined), the risk register with themes, the non-risks, and the option matrix; raises an ADR for any decision the evaluation *changes*, and for any mined tradeoff previously undocumented; the evaluator never edits the architecture it grades) → Verify (re-score the documented tradeoffs from the artifacts, not the prose: every `(H,*)` scenario has a met measure backed by a model/pattern/fitness-function scoped to its quantum, every risk has a theme + likelihood×impact, every tradeoff point — per-scenario and mined — names both attributes and cites its ADR; fitness functions for the sensitivity-point thresholds where automatable) → loop ≤10 or pass.

## Assertions & exit gate
- **Quantum scoping is confirmed before scoring**: every scenario analyzed is bound to the quantum (or the shared root) whose utility sub-tree it belongs to per `quantum-identification.md`; no scenario is scored against a flattened, cross-quantum tree.
- Every `(H,*)` scenario (every `(H,H)` first), per quantum, has an identified architectural approach with a **met/unmet** verdict **proven by a capacity model, a named pattern's guarantee, or a fitness function** — no scenario left un-analyzed, no verdict by assertion.
- Every **sensitivity point** is named with the scenario it controls; every **tradeoff point** — found in the per-scenario pass *or* the **tradeoff-mining sweep** — names the ≥2 conflicting attributes and the direction each moves, and the chosen point on the curve is driver-justified.
- The **tradeoff-mining sweep** has run over every sensitivity point named across every quantum (not just those already flagged per-scenario); any mined tradeoff with **no recording ADR** is emitted as a finding, distinct from the tradeoff's own risk rating.
- Every **risk** carries `likelihood × impact`, the threatened `QAS-*`, and a **risk theme tied to a business driver**; **non-risks** are recorded; risk themes are clustered (not a flat list).
- Where two approaches both meet the drivers, the **simpler** one is preferred (the over-engineered alternative is named and rejected, with a CBAM ROI number when cost is the differentiator — `tradeoff-analysis-cbam.md`).
- No `(H,*)` scenario relies on an approach that assumes a fallacy away on its critical path (`shared/fallacies-and-tradeoffs.md`).
- **Gate:** the quantum map is confirmed and every `(H,*)` scenario is scored against its own quantum's sub-tree; every scenario has a met (or explicitly accepted-risk) response measure proven by a model/pattern/fitness-function; every risk is themed to a business driver with likelihood×impact; every tradeoff point — per-scenario and mined — is named with both attributes and, if undocumented, raised as a finding — AND (design) the documented scorecard re-scores green with rubric mean ≥ 0.8.

## Output
Write `artifacts/architecture-evaluation-atam/report.json` per `shared/report-format.md` — the `drivers[]` block (the prioritized scenarios analyzed, per quantum), `findings[]` per `shared/finding-schema.md` (`type:"risk"` for register items with `severity = likelihood×impact`; `type:"decomposition"` for a scenario scored against the wrong quantum's tree; `oracle:"hard"` for an un-analyzed `(H,*)` scenario, an unmet/asserted response measure, an un-named tradeoff point on a conflicting decision, a mined tradeoff with no recording ADR, or a risk with no theme), and the verification block (the quantum-scoping confirmation, per-scenario met/unmet + the model/pattern/fitness-function that proves it, the sensitivity points, the tradeoff points (per-scenario + mined, each tagged with its source pass), the risk register with themes, the non-risks). In design mode the authored scorecard lands in `architecture/<slug>/tradeoffs.md` and any decision the evaluation changes — including a newly-documented mined tradeoff — is recorded as an ADR in `architecture/<slug>/adr/` (`shared/adr-template.md`); the report lists those paths. Open driver questions (an unstated RTO/availability/scale that blocks a `(H,*)` analysis, or a missing quantum map) are listed explicitly for the controller — never silently assumed.

## Guardrails
Per `shared/guardrails.md`: **propose & evaluate — never write product code and never edit the architecture being graded** (the evaluator is read-only; the one writer `solution-architect` folds findings in, sequenced). Ground every verdict in a driver — a "risk" with no threatened scenario is a preference, not a finding; drop it. Prove each response measure by a model/pattern/fitness-function, never by assertion (`shared/evaluation-method.md`). Score against the correct quantum's sub-tree, not a flattened system-wide tree (`quantum-identification.md`). Name the tradeoff and the chosen point on the curve — both the ones found scenario-by-scenario and the ones surfaced only by the tradeoff-mining sweep — and record non-risks alongside risks — a scorecard that shows only downside is as incomplete as one that shows only upside. Penalize accidental complexity and unjustified novelty: where the simpler approach meets the drivers, it wins. Never write to project memory — the analysis lives in the report and `tradeoffs.md`.

Cross-references: `quantum-identification.md` (Step 0 — run before this flow scores anything; supplies the per-quantum utility sub-trees this flow scores against) and `shared/evaluation-method.md` (the authoritative ATAM/utility-tree/CBAM method, including the Step 0 quantum-scoping rule and the tradeoff-mining-sweep definition this flow implements).
