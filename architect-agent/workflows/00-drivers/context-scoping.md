# context-scoping

**Group:** 00-drivers
**Runs as:** subagent: ../.claude/agents/requirements-analyst.md
**Mode:** audit (read-only review of an existing C4 L1 context diagram and scope boundary against the stated actors, external systems, and drivers) · design (writer + verify loop, authors the context diagram-as-code and the scope in/out boundary)   ·   **Default model:** sonnet

## Purpose
Draw the C4 Level-1 System Context — the system as a single black box, its named human actors/roles and external systems around it, and an explicit in/out scope boundary — so every downstream flow agrees on *what is being designed, who touches it, and what it touches*. Answers: *who are the human actors that interact with this system, what external systems does it depend on or serve, and where does this system's responsibility end?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack and deployment target, team topology, compliance regime, constraints, and the `architecture/<slug>/` artifact location + diagram tooling (Mermaid for C4 Context; PlantUML+C4-PlantUML for richer views).
- The architecture **drivers**: the functional scope and the prioritized quality-attribute scenarios from `quality-attribute-scenarios.md` (the utility tree already authored). If the utility tree is absent, this flow surfaces it as a blocking precondition — the context must be grounded in what the system must achieve, not just what it looks like from the outside.
- The system's source-of-truth docs named in `project-architecture-config.md` (typically a BRD, a PRD, and a durable understanding log covering roles, permissions, core-entity lifecycle, and module boundaries). Read these before framing actors or external systems; never invent a role or an external dependency.
- Preconditions: source docs readable; the quality-attribute scenarios flow (`workflows/00-drivers/quality-attribute-scenarios.md`) has either completed or its open driver questions are known; the diagram tooling in `project-architecture-config.md` is confirmed (Mermaid default; PlantUML+C4-PlantUML for deeper views).

## Oracle (source of truth)
The C4 Level-1 contract (Simon Brown's C4 model) applied to the stated functional scope and the system's documented roles/integrations. Split hard and soft:

- **hard:**
  - The context diagram is **diagrams-as-code** (Mermaid `C4Context` or PlantUML+C4-PlantUML `C4_Context.puml`) and **compiles/renders without error** — a diagram that does not compile is a broken artifact (`shared/diagramming-standards.md` → *Diagram compiles* hard oracle).
  - Every **human actor** is represented as a named `Person` or `Person_Ext` element typed with their role (per the project's RBAC roles in `project-architecture-config.md` — e.g. a generic `admin` / `manager` / `member` hierarchy) and labeled with a one-line responsibility — no unlabeled boxes.
  - Every **external system** the software system depends on or serves is represented as a named `System_Ext` element typed with its category (identity provider, payment processor, document store, mapping service, email/notification sink, etc.) — no unlabeled arrows into undeclared boxes.
  - Every **relationship** is directed and carries a label (the verb + the nature of the interaction: "submits orders via, HTTPS", "triggers document processing via, async webhook") — an unlabeled or undirected arrow fails this oracle.
  - The **scope in/out boundary** is explicit: a written list of what is *inside* this software system vs what is *outside* (an adjacent system, a deferred integration seam, a third-party) — nothing is left un-bounded or ambiguous.
  - Actors and external systems found in the source-of-truth docs (BRD/PRD/understanding log, per `project-architecture-config.md`) that *interact with the system* are all present — no silent omissions of a documented role or integration.
  - The diagram operates at **C4 Level 1 altitude** (one box for the entire software system, not split into containers or services — that is L2 scope handled by `workflows/70-documentation/c4-model-diagrams.md`).

- **soft:**
  - The boundary is at the **right altitude** — the system box represents one software system (the project under design, per `project-architecture-config.md`), not the whole enterprise and not one module or container; the L1 diagram communicates to a non-technical audience without needing a legend for internal structure.
  - **External dependencies are complete** — no known integration is missing; the diagram is a faithful external view, not a cherry-picked subset of the real dependency surface.
  - Actor names and external-system names are consistent with the ADRs, the understanding log named in `project-architecture-config.md`, and the domain ubiquitous language (established in `workflows/00-drivers/domain-analysis.md`) — drift between the context diagram and other artifacts is a craft gap.
  - The chosen diagram DSL and the layout communicate the boundary cleanly; trust boundaries (where authn/authz applies) are visually distinct.

## Standards & techniques
- **C4 Level 1 — System Context** (Simon Brown's C4): the system-as-a-black-box, surrounded by its `Person` (human actors) and `System_Ext` (external systems) elements; directed, labeled `Rel` edges. The fundamental audience is "everyone, including non-technical" — write labels accordingly.
- **Actor + external-system identification**: enumerate actors from the RBAC model in the project's understanding log (per `project-architecture-config.md`) — e.g. a top-level `admin` role fanning out to scoped roles such as `manager` and `member` — and from the BRD's stakeholder section. Enumerate external systems from the integration surface: authentication/identity, document processing, a payment integration, a mapping/geolocation service, email/notification sinks, and any deferred integration seam called out in the drivers (scope it as *out* today, mark the seam).
- **Scope boundary (in/out)**: explicit, written prose adjacent to the diagram — what is inside (the core platform and its primary bounded contexts, per `project-architecture-config.md`: e.g. auth, core-entity lifecycle, org/account management, users, audit) vs outside (third-party processors, any deferred integration seam, future bounded contexts still mocked). This is the scope the ATAM evaluation (`workflows/60-evaluation/architecture-evaluation-atam.md`) will use.
- **System-as-a-black-box**: C4 L1 shows *interactions with* the system, not *the internals of* the system. Container decomposition is L2 (`workflows/70-documentation/c4-model-diagrams.md`). No internal boxes at this level.
- **Diagrams-as-code** per `shared/diagramming-standards.md`: Mermaid `C4Context` for in-Markdown rendering (lightest weight, embeds in the design doc); PlantUML+C4-PlantUML (`!include <C4/C4_Context>`) for richer layout where Mermaid's C4 support is thin. Source lands in `architecture/<slug>/diagrams/`; verify it compiles before the artifact is counted done.
- **Notation conventions** (`shared/diagramming-standards.md`): every element labeled name + type + one-line responsibility; every relationship labeled and directed (verb + protocol/style); sync vs async visually distinct (solid vs dashed); trust boundaries drawn where they exist; legend included if notation is not self-evident.

## Step sequence
- **audit:** Read the source-of-truth docs named in `project-architecture-config.md` (understanding log, PRD, BRD) → enumerate every named human role and every external system mentioned → check whether an existing C4 L1 diagram is present in `architecture/<slug>/diagrams/` → if present: verify it compiles, check every actor and external system is present and labeled, check every relationship is directed and labeled, check the in/out scope is explicit, check the altitude is correct (one system box, not containers) → emit findings per `finding-schema.md`: missing actor (`ARCH-CTX-NNN`, type `documentation`, hard oracle); unlabeled relationship (hard oracle); diagram does not compile (hard oracle); missing external system (hard oracle); wrong altitude (hard oracle); missing in/out scope boundary (hard oracle). `driver_ref` for every non-documentation finding is assigned in the Output section — see below for the mapping to the functional-scope driver and the security/compliance scenario. Read-only; no artifact authored.

- **design:** Frame (extract actors + external systems from the source-of-truth docs named in `project-architecture-config.md`; confirm drivers from the utility tree; list open-boundary questions for the controller if any integration is undocumented — do not invent) → Explore (produce ≥2 candidate boundary framings as prose specs — e.g. framing A places a deferred integration seam as an explicit `System_Ext` placeholder; framing B treats it as entirely out-of-scope with a note; evaluate which more honestly represents the architecture contract the current team must honor) → Evaluate (single skeptical evaluator scores each framing against the hard oracles + the completeness and altitude soft oracles; order-swapped to kill position bias; picks the framing that is most complete, most accurate, and at the right altitude — not the most impressive) → Document (one writer authors the C4 L1 diagram-as-code in `architecture/<slug>/diagrams/context.{md|puml}` and the scope in/out list in `architecture/<slug>/design-doc.md`; names match the understanding log exactly; no internal structure exposed) → Verify (compile the diagram; check every actor, every external system, every relationship is labeled and directed; check scope in/out is explicit; evaluator re-scores rubric; loop ≤ 5 or pass).

## Assertions & exit gate
- The C4 L1 diagram compiles/renders without error (Mermaid `C4Context` or PlantUML+C4-PlantUML — run the toolchain, not just a syntax review).
- Every human actor documented in the understanding log named in `project-architecture-config.md` (at minimum: the project's top-level admin role and its scoped sub-roles, e.g. a generic `admin` / `manager` / `member` hierarchy) appears as a named, typed, labeled `Person` or `Person_Ext` element.
- Every external system the platform depends on or integrates with (at minimum: authentication/identity source, a document-processing provider, a payment integration, a mapping/geolocation service, email/notification sink) appears as a named, typed, labeled `System_Ext` element.
- Every relationship is directed and labeled (verb + interaction nature); no unlabeled or undirected edges.
- Scope in/out is explicit in prose adjacent to the diagram: a list of what is *inside* the system boundary and what is *outside* (third-party, deferred, mocked), with any deferred integration seam named in the drivers explicitly scoped out.
- The diagram operates at C4 L1 altitude — the system is one box; no containers, services, or modules are exposed inside it; the diagram is legible to a non-technical reader.
- (Design mode) Rubric scores: boundary altitude ≥ 0.8; external-dependency completeness ≥ 0.8; actor/external-system naming consistent with the understanding log and domain ubiquitous language (from `workflows/00-drivers/domain-analysis.md`) ≥ 0.8.
- **Gate:** hard oracles green (diagram compiles; all actors present and labeled; all external systems present and labeled; all relationships directed and labeled; scope in/out explicit; C4 L1 altitude correct) AND (design) rubric mean ≥ 0.8.

## Output
Write `artifacts/context-scoping/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (type `documentation` or `driver`; `oracle:"hard"` for a diagram that does not compile, a missing actor, an unlabeled relationship, a missing external system, an implicit scope boundary, or wrong altitude; `oracle:"soft"` for craft gaps in altitude or completeness; `driver_ref` traces to the functional-scope driver or the security/compliance scenario that the missing boundary decision threatens). For audit mode: a prioritized findings list (critical → cosmetic) and a risk register for any integration not accounted for. For design mode: the path to the authored artifacts (`architecture/<slug>/diagrams/context.{md|puml}`, the scope in/out section in `architecture/<slug>/design-doc.md`) plus the verification block (diagram-compile result, completeness check, rubric score per soft oracle). Open boundary questions (an integration mentioned in the BRD but undocumented — e.g. notification channel, payment processor identity) are listed explicitly for the controller to resolve, never silently assumed.

## Guardrails
Per `shared/guardrails.md`: propose and document — never write product code; ground every actor and external system in a stated source (the BRD/PRD/understanding log named in `project-architecture-config.md`); do not invent a role, an integration, or a response measure that is not documented. Read-only audit authors nothing. One writer authors the context diagram and the scope boundary; the diagram-source file (`context.puml` / `context.md`) is the deliverable, never a PNG. Cross-reference `workflows/70-documentation/c4-model-diagrams.md` for the full C4 model authoring convention; cross-reference `workflows/00-drivers/domain-analysis.md` for the ubiquitous language and bounded context names that must align with the actor and system names in this diagram; cross-reference `shared/diagramming-standards.md` for the notation rules and the diagram-compile hard oracle. The scope in/out list authored here becomes the boundary contract that `workflows/60-evaluation/architecture-evaluation-atam.md` and the domain-analysis flow both read — get it right before proceeding.
