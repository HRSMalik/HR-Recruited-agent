# domain-analysis

**Group:** 00-drivers  ·  **Runs as:** subagent: ../.claude/agents/requirements-analyst.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Apply DDD strategic design to build the **ubiquitous language**, identify **bounded contexts** with explicit boundaries, produce a typed **context map** (relationship types between every pair of contexts), and classify subdomains as **core / supporting / generic** — the boundary decisions every decomposition and integration flow downstream must follow. Answers: *where does the domain language change, and what does that tell us about where module/service boundaries belong?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing system shape (the project's stack, e.g. a modular-monolith-pair of a backend service + frontend SPA over a primary datastore), team topology (per Conway's Law), scope in/out, constraints, compliance regime, and the artifact output path (`architecture/<slug>/`).
- The architectural **drivers**: the prioritized utility tree from `workflows/00-drivers/quality-attribute-scenarios.md` — in particular the evolvability driver (any deferred integration seam, near-term bounded contexts for domains still mocked or stubbed) and the security/RBAC driver (tenant- or org-scoped data isolation). A domain analysis without drivers has no way to decide *which* subdomain is core (where to invest) vs generic (where to buy/outsource). If the utility tree is absent, surface as a blocking precondition gap.
- **Source-of-truth documents to read first** (read-only): the BRD/PRD and the confirmed-understanding log named in `project-architecture-config.md` (the project's requirements doc, product spec, and the running workflows/roles/permissions log). Do not re-derive what is already confirmed; reuse the existing understanding.
- **Preconditions**: at least one source-of-truth document readable; the utility tree present or a summary of top-priority quality-attribute scenarios; the functional scope confirmed (per `project-architecture-config.md` — the core entity lifecycle, intake, org/account onboarding, users, RBAC, audit, documents, payments, closure, or whichever domains apply; any domain still mocked should be noted as such).

## Oracle (source of truth)
The domain language and the functional boundaries drawn from the BRD/PRD, the confirmed workflows in the project's understanding log, and the structural fact that **language changes at context boundaries** (Evans's DDD, ch. 14 — a term that means something different in two parts of the system reveals a boundary). No boundary is accepted without evidence that the language shifts or that an isolation driver demands it.

- **hard:**
  - Every bounded context has an **explicit boundary statement** — a one-sentence scope definition — and a **ubiquitous-language glossary** of the terms that belong to it (terms that straddle multiple contexts without translation reveal a missing boundary).
  - Every pair of contexts that communicate has an **explicitly typed relationship** on the context map — one of: Partnership, Customer–Supplier (upstream/downstream), Conformist, Anticorruption Layer (ACL), Shared Kernel, Open-Host Service (OHS) / Published Language, or Separate Ways. An untyped arrow is a gap finding (`oracle:"hard"`, `type:"decomposition"`).
  - Every subdomain is classified: **core** (the business differentiator — where strategic investment belongs), **supporting** (necessary but not differentiating — build or outsource selectively), or **generic** (commodity — buy/SaaS/open-source; never invest core-team effort). A subdomain left unclassified blocks.
  - Core subdomains and generic subdomains are disjoint — a subdomain that is both is a classification error (usually the generic part of a core domain has been conflated with the differentiating part; split it).
  - Every ACL relationship names the **foreign model it protects against** and the translation responsibility (which team owns the translation layer). An ACL with no named foreign model is structurally incomplete.
  - The context map is a **diagrams-as-code artifact** (Mermaid `graph` or PlantUML) that renders without error — per `shared/diagramming-standards.md`. A context map that is prose-only or doesn't compile is a hard oracle failure.

- **soft:**
  - Boundaries are cut **where the domain language changes**, not on technical layers (database schema, REST endpoints, import paths). A boundary drawn on "frontend vs backend" with no language shift behind it scores low on conceptual integrity (`shared/quality-attribute-rubric.md`).
  - The context map relationship types are **honest about coupling** — a Conformist is not relabeled Customer–Supplier to look better; a Shared Kernel is not hidden as a Partnership when two contexts share a mutable model.
  - An ACL is placed **wherever the system integrates a foreign model** (a third-party service, a future deferred-integration API, an external data feed) — not only where the relationship is technically difficult. Undiscovered ACL needs are an evolvability risk.
  - The subdomain classification is **defensible from the BRD** — core subdomains are those where a better implementation than competitors' provides measurable business value (e.g. the core-entity-adjudication and RBAC-scoped-assignment logic); generic subdomains are those where commodity solutions exist (auth token issuance, PDF generation, email delivery).
  - Conway's Law is applied explicitly: the proposed context boundaries are checked against the actual team topology (`project-architecture-config.md`) — boundaries the team can't own are scored down (Brooks, `fallacies-and-tradeoffs.md`).

## Standards & techniques
- **DDD strategic design** (Evans, *Domain-Driven Design*, Part IV): ubiquitous language, bounded contexts, context mapping, core/supporting/generic subdomain classification. These are the primary method; every output maps to a named DDD concept.
- **Context map relationship types** (Evans ch. 14 + Brandolini *Introducing EventStorming*): Partnership (mutual dependency, fail together), Customer–Supplier (downstream shapes upstream's roadmap), Conformist (downstream has no influence — takes the upstream model as-is), ACL (downstream translates to protect its own model), Shared Kernel (two teams share a subset of the domain model, changes coordinated), OHS/Published Language (upstream provides a well-defined open API; downstream conformist or ACL), Separate Ways (no integration — explicit decision). Type every edge; unlabeled is not a type.
- **Subdomain classification** (Wlaschin, *Domain Modeling Made Functional*, ch. 2; Millett & Tune, *Patterns, Principles, and Practices of Domain-Driven Design*): core = differentiating; supporting = necessary but not unique; generic = commodity. Classification drives the build/buy/partner decision for each subdomain — only core subdomains warrant core-team implementation effort.
- **ATAM sensitivity analysis** (per `shared/evaluation-method.md`): boundary placement is a **sensitivity point** for the evolvability and security quality attributes — a boundary drawn in the wrong place is a one-way-door (expensive to move later). Flag every boundary decision with its reversibility (one-way-door) and the driver it serves.
- **Conway's Law / Inverse-Conway Maneuver** (`shared/fallacies-and-tradeoffs.md`): team structure and context boundaries must be consistent; where they conflict, either the boundary or the team structure needs deliberate adjustment. Name the tension explicitly in findings.
- **Anti-corruption layer** (Evans ch. 14, also referenced in `workflows/80-patterns/distributed-systems-patterns.md`): the canonical pattern for protecting a core context's model from a foreign model (a third-party API, a future deferred-integration service, an external data provider). Every trust boundary in the threat model that crosses a context boundary is a candidate for an ACL.
- **Diagrams-as-code** (per `shared/diagramming-standards.md`): context map authored in Mermaid `graph LR` or PlantUML; every node is a bounded context, every directed edge is a typed relationship, relationship type is on the arrow label.

## Step sequence

- **audit:** Read the BRD/PRD/understanding log named in `project-architecture-config.md` → enumerate the domain terms that appear in the codebase and documents → identify terms whose meaning shifts between parts of the system (each shift is a candidate boundary) → check whether the current module/package structure follows or contradicts the language boundaries → identify every inter-module dependency and type it against the context-map relationship taxonomy → classify each module/domain area as core/supporting/generic against the BRD's stated differentiators → emit findings (untyped relationships, missing glossary terms, miscounted ACL needs, subdomain classification gaps, Conway's Law conflicts, uncompilable diagrams). Read-only; no artifact authored.

- **design:** **Frame** (read source docs + confirm utility tree; extract domain terms and functional areas from the BRD/PRD/understanding log named in `project-architecture-config.md`; confirm scope with controller if ambiguous; read-only) → **Explore** (≥2 divergent context-map proposals covering: a language-boundary-first cut vs a volatility/team-boundary-first cut — both rendered as diagrams-as-code with all relationship types labeled; read-only, proposers are separate from the evaluator) → **Evaluate** (skeptical evaluator scores each proposal against the hard oracles + utility-tree drivers + Conway's Law, order-swapped per `shared/evaluation-method.md`; picks a winner and grafts superior ideas from runners-up; produces a decision matrix noting sensitivity/tradeoff points for boundary placement; read-only) → **Document** (one writer — `solution-architect` — authors the domain analysis artifacts: ubiquitous-language glossary, bounded-context catalog with boundary statements, context map diagram-as-code, subdomain classification table, ADRs for significant boundary decisions; all land in `architecture/<slug>/`; preserves conceptual integrity) → **Verify** (fitness functions: glossary completeness check, context-map edge type coverage, subdomain classification coverage, diagram compile, traceability to utility-tree drivers; evaluator re-scores from the authored artifacts; loop ≤ 10 or pass).

## Assertions & exit gate
- Every bounded context has a one-sentence boundary statement and a ubiquitous-language glossary (≥ the key domain terms; no term shared across contexts without an explicit translation mechanism on the context map edge).
- Every context-map edge is typed with one of the 7 canonical DDD relationship types; no unlabeled arrows.
- Every subdomain is classified exactly once as core, supporting, or generic; the core subdomains are traced to a BRD driver (the differentiating business capability they represent).
- Every ACL names the foreign model it protects against and states which team owns the translation layer.
- The context map diagram compiles (Mermaid/PlantUML renders without error; per `shared/diagramming-standards.md`).
- Every significant boundary decision has an ADR (context · ≥2 options · decision · consequences positive AND negative; per `shared/adr-template.md`); one-way-door boundaries are flagged explicitly.
- The context map is traceable to at least one utility-tree driver per boundary — a boundary with no driver is accidental complexity and is flagged as a `major` finding (`type:"decomposition"`, `oracle:"soft"`).
- Conway's Law tension (a boundary the team topology can't own) is named as an open risk if present, not silently ignored.
- **Gate:** all hard oracles green (typed context map compiles; every BC has a glossary + boundary statement; every subdomain classified; every ACL names its foreign model; every ADR complete) AND rubric mean ≥ 0.8 (boundary rationale, honest coupling, ACL placement, Conway's fit, subdomain classification defensibility).

## Output
- Write `artifacts/domain-analysis/report.json` per `shared/report-format.md` — findings per `finding-schema.md` (`type:"decomposition"`, `oracle:"hard"` for an untyped context-map edge, a bounded context with no boundary statement or glossary, an unclassified subdomain, an ACL with no named foreign model, or a context-map diagram that does not compile; `type:"decomposition"`, `oracle:"soft"` for a boundary drawn on a technical layer with no language shift behind it, a subdomain classification not defensible from the BRD, or a context boundary the team topology cannot own; `type:"risk"`, `oracle:"hard"` for a Conway's-Law conflict where the proposed boundary has no owning team — this is a one-way-door structural risk, not a craft gap; `type:"driver"`, `oracle:"soft"` for a context boundary with no traceable utility-tree driver — accidental complexity, not a hard failure but a gate blocker until the driver is named); the `drivers[]` block listing every quality-attribute scenario this boundary decision is traced to, and the verification block (hard-oracle results, rubric scores, fitness-function results) for design mode.
- Design artifacts land in `architecture/<slug>/` (slug per `project-architecture-config.md`):
  - `architecture/<slug>/domain-glossary.md` — the ubiquitous language: one table per bounded context, term → definition within this context, cross-context disambiguation where the same word means different things.
  - `architecture/<slug>/bounded-contexts.md` — the bounded-context catalog: one entry per context with boundary statement, team ownership, in-scope responsibilities, and excluded responsibilities.
  - `architecture/<slug>/diagrams/context-map.puml` (or `.md` if Mermaid) — the context map as diagrams-as-code; every node is a bounded context; every directed edge carries its DDD relationship type and the upstream/downstream direction for Customer–Supplier and Conformist.
  - `architecture/<slug>/subdomain-classification.md` — the classification table: subdomain → core/supporting/generic → BRD driver ref → build/buy/partner recommendation.
  - `architecture/<slug>/adr/NNNN-<boundary-decision>.md` — one MADR ADR per significant boundary placement or ACL decision; one-way-door decisions flagged.
- Open driver questions (any team-topology conflict, any subdomain whose classification is contested, any regulatory scope that changes a context boundary) are listed explicitly in the report for the controller to resolve — never silently assumed.
- Cross-flow outputs: the context map and subdomain classification become the canonical input for `workflows/10-styles-decomposition/decomposition-strategy.md` (which splits by subdomain/capability) and are checked against the system context in `workflows/00-drivers/context-scoping.md`; ACL placement decisions feed `workflows/80-patterns/distributed-systems-patterns.md`.

## Guardrails
Per `shared/guardrails.md`:
- **Propose & document, never write product code.** Domain artifacts (glossary, context map diagrams-as-code, bounded-context catalog, ADRs) land in `architecture/<slug>/`; never in `src/`, `frontend/`, `backend/`, or `tests/`. The controller implements the module boundaries; this flow defines them.
- **Ground every boundary in a driver.** A context boundary that serves no quality-attribute scenario or constraint is accidental complexity — flag it, do not author it (`guardrails.md §2`). The utility tree from `workflows/00-drivers/quality-attribute-scenarios.md` is the authority.
- **One writer.** The solution-architect authors the domain artifacts; proposer agents and the evaluator are read-only (`guardrails.md §4`). Multiple writers produce an incoherent glossary — the cardinal DDD failure.
- **Honest relationships.** Do not relabel a Conformist as Customer–Supplier or hide a Shared Kernel as Partnership to make the map look cleaner. The context map's value is its honesty about coupling; a polished map that misrepresents the relationships is worse than a messy honest one.
- **Never invent NFR numbers.** If a driver (availability SLO, scale target, data-residency obligation) that affects a boundary decision is not stated in the BRD/PRD, surface it as an open question — do not assume it.
- **Right-size the granularity.** Match the system's actual scale and team topology (see `project-architecture-config.md`). Where the project is a small-team system best served by a modular monolith, do not propose a context-per-microservice decomposition; the modular monolith is the default in that case. A context boundary in this flow maps to a **module boundary inside the monolith** unless a concrete driver (independent deploy, team split, scale isolation) demands a service split — which is `decomposition-strategy.md`'s decision, not this flow's.
