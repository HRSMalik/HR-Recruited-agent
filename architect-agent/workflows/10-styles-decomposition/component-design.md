# component-design

**Group:** 10-styles-decomposition  ·  **Runs as:** subagent: ../.claude/agents/solution-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Design the internal structure of a module or service — layering vs hexagonal (ports & adapters) vs clean/onion — and enforce the **dependency rule**: source-code dependencies point inward, toward higher-level policy, never outward toward volatile frameworks or I/O. Answers: *what internal structure best isolates the domain core from change-prone externals, given the evolvability and testability drivers this module must satisfy?*

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack and module boundaries, deployment target, team topology, constraints (stack lock, boring-tech-preferred), and the artifact locations for the architecture package.
- The **drivers** for this module: the prioritized quality-attribute scenarios (`architecture/<slug>/quality-attributes.md`) that this module's internal structure must satisfy — in particular any evolvability (can the datastore or framework be replaced without rewriting domain logic?), testability (can the core be tested without a live DB/broker?), and performance scenarios that constrain layering choices.
- **The upstream decomposition decision**: the output of `workflows/10-styles-decomposition/decomposition-strategy.md` (which bounded context or capability this module represents) and `workflows/10-styles-decomposition/dependency-management.md` (the inter-module coupling constraints that this internal structure must honor at its boundary).
- The module's current source layout (read-only scan: file/directory tree, import graph, framework coupling points) for an audit; a confirmed functional scope + the driver set for a design.
- Preconditions for **audit**: the source tree is readable and the import graph can be derived (static analysis — `grep`, `glob`, dependency-cruiser, or equivalent). Preconditions for **design**: the decomposition decision is accepted (the boundary of this module is settled) and the driver set is non-empty; a design with no evolvability or testability driver has no basis for choosing ports-and-adapters over a plain layered structure — surface this as an open question, do not default to the more complex style.

## Oracle (source of truth)

The external truth is the **structural contract** implied by the chosen internal style, verified against the module's source-code dependency graph and the evolvability/testability scenarios in the utility tree.

- **hard:**
  - **Dependency rule satisfied (acyclicity + inward direction):** every import / dependency edge in the module points from outer rings toward inner rings — framework and I/O adapters depend on the domain core; the domain core has **zero** imports of framework types, ORM types, HTTP client types, or any other volatile external. A single outward dependency from the domain core (e.g. `core/` importing an ORM session type directly) is a hard-oracle failure. Verified by static import-graph analysis: the domain ring has no dependency on `src/infrastructure/**`, `src/adapters/**`, `src/framework/**`, or any third-party I/O library. This maps to the **stable-dependencies principle** (depend in the direction of stability) and the **acyclic-dependencies principle** (no cycles in the component graph) from `workflows/10-styles-decomposition/dependency-management.md`.
  - **Every external concern behind a port/adapter:** each I/O boundary (database, external API, message broker, file system, clock, random) is accessed exclusively through an interface defined in the domain ring and implemented in the adapter ring. No direct call to a database driver, an HTTP client library, or equivalent appears in domain or application-logic code — only the port interface. Verified structurally: grep for framework/IO imports in domain-layer files; zero matches is the gate.
  - **Domain core testable without infrastructure:** the domain-core ring can be instantiated and exercised with no live database, no running broker, and no HTTP endpoint — either in-process pure-function style or with in-memory test doubles substituted through the port interface. A domain module that can only be tested with a real database connected has a failed hard oracle regardless of how the code is labeled.
  - **Consistency model stated per write path inside the module:** if the module performs more than one write (e.g. domain-event emission + persistence), the consistency mechanism (unit-of-work pattern, outbox, single atomic write) is explicit and documented. An implicit dual-write inside a module's use-case layer is a `major` hard-oracle failure (`consistency_stated`).
  - **Driver traceability:** the chosen internal style (layered / hexagonal / clean) and every port decision traces to a named quality-attribute scenario in the utility tree (`QAS-*`). A port with no driver reference is an unjustified decision — flag it.

- **soft:**
  - **Port placement is change-motivated:** every port/adapter seam corresponds to a real change vector or I/O boundary stated in the drivers (a datastore substitution scenario, a framework-migration evolvability scenario, an external-API integration). Ports placed for ceremony with no driver are accidental complexity — flagged as a `major` advisory finding (`oracle:"soft"`) against the YAGNI/right-sizedness rubric dimension.
  - **Conceptual integrity across like modules** (Brooks): if the codebase has ten modules, they all use the same internal style — the same ring names, the same port-naming convention, the same import discipline. A system where three modules are layered and two are hexagonal with different terminology fails this even if each is internally consistent.
  - **Port placement matches volatility:** ports sit where change or I/O *actually crosses*, not as a blanket pattern applied to every dependency. An in-process, same-language utility that will never be swapped out does not need a port — wrapping it in one adds cognitive overhead with no driver benefit.
  - **Ring granularity is right-sized:** the number of rings (2-ring layered: core + infra; 3-ring clean: domain + application + infra; 4-ring onion: domain model + domain services + application + infra) matches the complexity of the use-case logic. A CRUD adapter with no domain behavior does not need a 4-ring onion structure — the simplest layering that keeps I/O out of domain logic is correct.

## Standards & techniques

- **Layered architecture** — the classic N-tier internal cut: presentation → application → domain → infrastructure. Dependencies flow strictly downward; each layer knows only the one below it. The domain layer has no reference to infrastructure or presentation. Appropriate when the module has minimal cross-cutting concerns and no expectation of infrastructure substitution; the simplest correct choice for bounded CRUD modules.
- **Hexagonal architecture (ports & adapters)** — Alistair Cockburn (2005): the application core (domain + use cases) sits in the center; **ports** (interfaces defined in the core) are the application's intentions toward the outside; **adapters** implement those ports for specific externals (a Postgres adapter for a repository port, an HTTP adapter for a notification-service port). Driving adapters (primary: HTTP handlers, CLI) call in through an input port; driven adapters (secondary: DB, broker, email) are called out through output ports. The core has no knowledge of any adapter. Choose this when: the datastore or a key external is a plausible change vector, or when the domain logic must be testable in isolation with in-memory doubles.
- **Clean / onion architecture** — Uncle Bob / Jeffrey Palermo: a strict concentric-ring discipline where **entities** (enterprise business rules) sit in the innermost ring, **use cases** (application business rules) in the next, **interface adapters** (controllers, presenters, gateways) next, and **frameworks & drivers** (DB, web, UI) on the outside. The **dependency rule** is the invariant: *source-code dependencies can only point inward* — an outer ring may know an inner ring, never the reverse. Clean architecture extends hexagonal by further separating use-case orchestration (application layer) from domain entities, which is valuable when use-case logic is rich (complex workflows, domain events, policy objects) rather than thin CRUD.
- **The dependency rule** (Uncle Bob, *Clean Architecture*): "Source code dependencies must point only inward, toward higher-level policies." This is the single invariant all three styles enforce — they differ in how many rings and how ports are named, not in the rule itself.
- **Dependency inversion principle (DIP)** at boundaries: high-level modules (domain, use cases) define the interface; low-level modules (adapters, infrastructure) implement it. The interface ownership is inverted — the *consumer* (domain) owns the contract, the *provider* (adapter) satisfies it. This is what makes the dependency graph point inward even though data flows outward.
- **Stable-dependencies principle** and **acyclic-dependencies principle** (Robert C. Martin, *Agile Software Development*): packages/modules should depend in the direction of stability; no cycles in the dependency graph. Both are checked structurally. Cross-reference `workflows/10-styles-decomposition/dependency-management.md` for the inter-module application of these principles.
- **Anti-corruption layer (ACL)** pattern (from the cloud-design-pattern catalog, `workflows/80-patterns/cloud-design-patterns.md`): at the adapter boundary where a foreign model (external API, legacy system, third-party SDK) must be translated into the domain's own model — rather than letting the foreign types leak into the domain ring, the adapter implements the port by translating. The ACL is the adapter itself; the port is the domain's own language. Reference this when designing adapters for external services (e.g. a payment gateway, a third-party data provider API).
- **Unit-of-work pattern** for write-path consistency inside a module: the application-layer use case opens a unit of work (a transactional boundary), calls domain methods, emits domain events, and commits — all in one atomic scope. The port defines `UnitOfWork`; the adapter implements it against the actual datastore. This keeps the domain core unaware of the transaction mechanism while ensuring consistency.

## Step sequence

- **audit:** Read the module's source tree and derive the import graph (static analysis — directory structure, import statements, dependency-cruiser or `grep -r "^from\|^import"`) → map each file to a ring (domain / application / adapter / infrastructure / framework) based on its imports and responsibilities → for each cross-ring dependency, check the direction (does an inner ring import an outer ring?) → flag every outward dependency from domain code, every framework type imported into the domain or use-case layer, every I/O call not behind a port, and every port with no driver reference → check for acyclic dependency violations (import cycles within the module) → check the write paths for implicit dual-writes → compare ring/port naming and structure against sibling modules for conceptual-integrity failures → emit findings per `finding-schema.md` (read-only; no source edited; no architecture artifact authored).

- **design:** Frame (confirm the functional scope + extract/confirm the driver set from `architecture/<slug>/quality-attributes.md`; identify the change vectors and I/O boundaries that motivate a port; if evolvability/testability drivers are absent, note that a simpler layered structure is sufficient — do not default to hexagonal) → Explore (produce ≥2 divergent internal structures: e.g. plain 2-ring layered vs hexagonal with explicit port inventory vs clean/onion with use-case objects; each is a concrete spec naming the rings, the port interfaces, the adapter types, and the tradeoff against the drivers; read-only) → Evaluate (skeptical evaluator scores each against the utility tree + hard oracles — dependency rule, domain-core purity, testability, driver traceability — using ATAM sensitivity and tradeoff analysis; kills position bias by scoring in swapped order; picks a winner and grafts superior ideas from runners-up; penalizes accidental complexity — a hexagonal structure for a CRUD module with no substitution driver scores lower, not higher) → Document (one writer authors: C4 L3 Component diagram as diagrams-as-code showing rings + ports + adapters; the ADR for the chosen style; a port inventory table naming each port, its driver reference, and its adapter(s); fitness-function specifications for CI enforcement of the dependency rule; all artifacts land in `architecture/<slug>/`) → Verify (run the dependency-rule fitness function against the authored diagram and any draft source layout; check diagram compiles; check every port traces to a driver; check every ring label matches the style's invariant; evaluator re-scores from the artifacts; loop ≤ 10 or pass).

## Assertions & exit gate

- Every source-code dependency in the module graph points inward (toward the domain core); zero outward dependencies from domain or use-case code to framework/IO types — verified by static import-graph analysis or by the authored fitness-function specification.
- Every I/O boundary (database, external API, broker, clock, file system) is accessed exclusively through a port interface defined in the domain or application ring; adapters implement those ports in the outer ring.
- The domain-core ring can be instantiated and tested without a live database, broker, or external network dependency.
- Every port has a named driver reference (`QAS-*`) tracing it to an evolvability, testability, or integration scenario in the utility tree; ceremonial ports with no driver are flagged and removed.
- The chosen internal style (layered / hexagonal / clean) and ring structure are consistent with sibling modules in the codebase — same naming convention, same ring discipline, same port-interface-ownership rule (conceptual integrity).
- Every write path in the module's use-case layer names its consistency mechanism (unit-of-work, outbox, or single atomic operation); no implicit dual-writes.
- The C4 L3 Component diagram compiles and renders; every element is labeled with name, type, and one-line responsibility; every relationship is labeled with direction and protocol/invocation style; the diagram is consistent with the ADR and the port inventory.
- The ADR for the style choice is complete: context, ≥2 considered options (at minimum: layered vs hexagonal, or hexagonal vs clean), decision, positive consequences (what evolvability/testability scenarios it now meets), negative consequences (added abstraction overhead, port-maintenance cost), follow-up fitness functions.
- **Gate:** hard oracles green (dependency rule satisfied structurally; all I/O behind ports; domain core infrastructure-free; all ports driver-traced; write paths consistent; diagram compiles; ADR complete) AND (design) rubric mean ≥ 0.8 (conceptual integrity across modules, right-sized ring granularity, port placement matches volatility).

## Output

Write `artifacts/component-design/report.json` per `shared/report-format.md` — findings and risks per `finding-schema.md` with `type:"decomposition"` for structural violations, `type:"evolvability"` for dependency-rule failures, and `type:"risk"` for untraced ports or unmitigated write-path consistency gaps. Every design/decision finding carries the mandatory `driver_ref` (the `QAS-*` scenario it serves), `tradeoff` (what the chosen structure costs on other attributes — abstraction overhead, port-maintenance cost, additional indirection), and `alternatives_rejected` (the options considered and why not — e.g. "plain layered rejected: no isolation of volatile datastore, fails QAS-EVOL-01"). Hard-oracle findings (`oracle:"hard"`) block; soft-oracle findings (`oracle:"soft"`) are advisory above the rubric threshold.

For **design** mode the `report.json` also includes the **verification block**: fitness-function results (did the authored dependency-rule fitness function pass against the diagram/draft layout?), traceability coverage (count of ports with a `QAS-*` driver ref vs total ports — must be 100%), hard-oracle results (each named oracle and its pass/fail status), and the rubric score (the evaluator's mean across the soft-oracle dimensions from `shared/quality-attribute-rubric.md`, threshold ≥ 0.8). The orchestrator gate reads these fields directly — the verification block is required for the gate to determine pass/fail.

For **design** mode the following artifacts land in `architecture/<slug>/` (auto-create the directory if absent):
- `diagrams/component-<module-name>.puml` (or `.md` for Mermaid) — C4 L3 Component view as diagrams-as-code showing rings, ports, and adapters per `shared/diagramming-standards.md`.
- `adr/NNNN-component-structure-<module-name>.md` — the MADR ADR for the chosen internal style, per `shared/adr-template.md`; must include the port inventory as a table (port name | driver ref `QAS-*` | adapter(s)).
- `fitness-functions.md` (append or create) — the dependency-rule fitness function specification: the CI check that verifies no domain-ring file imports from `infrastructure/`, `adapters/`, or any external framework/IO package (e.g. a dependency-cruiser rule, a custom import linter, or an import-graph assertion script in `tests/fitness/`).

The per-flow run record (`report.json`) is the machine record consumed by the orchestrator gate. The architecture artifacts in `architecture/<slug>/` are the durable deliverables the controller implements. No product code is authored; no source files under `frontend/**`, `backend/**`, or `src/**` are written or edited.

## Guardrails

Per `shared/guardrails.md`:

- **Propose & evaluate — never write product code.** This flow authors ADRs, C4-as-code diagrams, a port inventory, and fitness-function *specifications*; it does not write module source, does not create `__init__.py`, `router.py`, or any file under `backend/**` or `frontend/**`. The controller writes the implementation from the authored architecture artifacts.
- **Ground every port in a driver.** A port with no quality-attribute scenario reference is accidental complexity — flag it and recommend removal (`shared/guardrails.md` §2). If the only argument for a port is "best practice," that is not a driver.
- **Right-size the style.** The simplest internal structure that satisfies the drivers is correct (`quality-attribute-rubric.md` soft oracle: Simplicity/YAGNI). A plain 2-ring layered split (domain + infrastructure) that keeps I/O out of domain logic and is testable with in-memory fakes beats a 4-ring onion for a module whose drivers do not require rich use-case orchestration or infrastructure substitution. Penalize accidental complexity.
- **One writer (conceptual integrity).** Exactly one `solution-architect` authors the C4 diagram and the ADR so the naming conventions, ring labels, and port-ownership rule are consistent across all modules (`shared/guardrails.md` §4). Domain specialists (data-architect, integration-architect) advise on their dimension read-only and verify their concerns are reflected.
- **External oracle over self-judgment.** The import-graph analysis and the fitness-function check are the gate — not the writer's assertion that the structure "looks clean." Run the structural check (or specify the CI fitness function that will run it) and report what it finds.
- **Cross-references:** `workflows/10-styles-decomposition/dependency-management.md` (inter-module coupling, stable-dependencies principle, ADP); `workflows/80-patterns/cloud-design-patterns.md` (anti-corruption layer at adapter boundaries for foreign models); `shared/architecture-principles.md` (dependency-rule operating habit, conceptual integrity, Conway's law, boring-tech preference); `shared/adr-template.md` (the MADR structure — context / ≥2 options / decision / consequences — consumed when authoring the component-structure ADR and verified at the exit gate).
