# service-granularity-analysis

**Group:** 10-styles-decomposition · **Runs as:** subagent: ../.claude/agents/solution-architect.md · **Mode:** audit + design · **Default model:** sonnet

## Purpose
Decide **whether to split or merge a service** — `decomposition-strategy.md` finds *where* a boundary is (capability / bounded-context / volatility, from the DDD context map); this flow decides *whether to act on it*, by scoring the **granularity disintegrators** (forces that push a service apart) against the **granularity integrators** (forces that pull it back together) for each candidate boundary. Answers: *for this specific boundary, do the disintegrating forces outweigh the integrating counterweights — and if so, by how much?* A boundary with a valid seam but no net disintegrator force should stay merged; splitting it anyway is résumé-driving, not architecture.

## Inputs & preconditions
- From `project-architecture-config.md`: the current shape (e.g. a modular monolith — do **not** propose splitting the system into services absent a concrete driver; where the config locks a modular-monolith default, this flow's default finding is "stay merged"), the existing module boundaries, the near-term bounded contexts, the team topology, the stack lock, and where the architecture description / ADR log / diagrams live.
- The **drivers / utility tree** from `../00-drivers/quality-attribute-scenarios.md` — in particular any scalability/throughput scenario, fault-isolation/availability scenario, security/compliance-isolation scenario, or extensibility/rate-of-change scenario that could motivate a split. A candidate boundary with no driver behind any disintegrator force cannot be scored above "keep merged" — do not invent a force.
- The **candidate boundary(ies)** from `decomposition-strategy.md` (the seam already validated as capability/context/volatility-based, not a technical layer) and, where relevant, the domain context map from `../00-drivers/domain-analysis.md`. This flow does not re-derive the seam; it scores whether to *act* on an already-identified one.
- The **call/data-dependency graph** and the **transaction/workflow inventory** for the candidate boundary and its neighbors — which operations currently span the boundary in a single database transaction, which tables/collections are read/written by both sides, and which business workflows choreograph across it.
- Preconditions: `project-understanding.md` + `docs/` read; the utility tree and the candidate boundary exist (or this flow surfaces the missing inputs as open questions and does not guess a split/merge verdict).

## Oracle (source of truth)
The **granularity disintegrators vs integrators** framework (Richards & Ford, *Software Architecture: The Hard Parts*, ch. 4–5) — six forces that argue for splitting a service further, weighed against four forces that argue for keeping it together. A split is justified only when the disintegrator score, weighted by driver priority, exceeds the integrator counterweight for that specific boundary; a merge (or "leave merged") is justified when it does not. Neither verdict is a default — both must be scored, named, and traced.

- **hard:**
  - **Every disintegrator force scored and named** — for the candidate boundary, each of the six disintegrators (service scope/cohesion, code volatility, scalability/throughput, fault tolerance, security, extensibility) is explicitly rated (None/Low/Medium/High) with the evidence that produced the rating. A disintegrator asserted without evidence (a topology fact, a metric, a named driver) is not scored — it is dropped, and the split cannot rely on it.
  - **Every integrator force scored and named** — for the same boundary, each of the four integrators (database transactions, data dependencies, workflow/choreography, [service scope/cohesion when it argues for merging]) is explicitly rated with evidence — the transaction inventory (does a single ACID transaction currently span the boundary?), the data-dependency graph (foreign-key/join density across the boundary), and the workflow inventory (does splitting turn a single in-process call into a multi-hop choreographed saga?).
  - **Disintegrator-vs-integrator arithmetic is shown, not asserted** — the decision states which forces were compared, their individual ratings, and the net verdict (split / merge / leave as-is); "it feels like it should be a service" without the scored comparison is not a finding.
  - **Split recommended only when disintegrators outweigh integrators** — a `split` verdict requires at least one disintegrator rated High with a traced driver AND the integrator counterweight (in particular database-transaction and data-dependency strength) rated no higher than Medium for that boundary. A High integrator counterweight (e.g. the split would break an existing ACID transaction with no compensating-transaction design) blocks a `split` verdict until `../20-data/consistency-transactions.md` / `../20-data/distributed-transaction-design.md` names the saga pattern that would carry it.
  - **The decision is an ADR** — every split/merge verdict is recorded as a MADR ADR (context = the boundary + the drivers in tension, options = split / merge / leave-as-is, decision = the scored verdict, consequences = what the winning side costs on the other attributes) per `../../shared/adr-template.md`. A verdict with no ADR is undocumented and blocks the gate.
  - **Driver-traced** — every disintegrator rated Medium or High traces to a quality-attribute scenario in the utility tree (a scale driver for scalability/throughput, an availability driver for fault tolerance, a compliance/PII driver for security, a rate-of-change signal — e.g. commit-churn or deploy-frequency skew — for code volatility). A disintegrator with no traced driver is accidental complexity if it drives a split.

- **soft:**
  - **Right-sized, not nano-serviced** — a High disintegrator score with no integrator counterweight recommends a split; anything short of that recommends keeping the boundary cohesive. Splitting on a Low/Medium disintegrator score "to be safe" is flagged as over-decomposition risk (N×M failure surface, Metcalfe coupling, more moving parts than the driver bought).
  - **The counterfactual merge is considered** — for every existing or previously-split service, the flow also asks whether the integrators now outweigh the disintegrators (i.e., should this already-split service be *merged back*); a one-way "always split further" bias is a soft finding.
  - **Sensitivity noted** — where the verdict is close (disintegrator and integrator scores within one band of each other), the flow states what evidence would flip it (e.g. "if document-processing volume triples, the scalability/throughput disintegrator moves Medium→High and the verdict flips to split").

## Standards & techniques
- **Granularity disintegrators** (Richards & Ford, *The Hard Parts* ch. 4) — six forces, each a reason to split a service *further*:
  1. **Service scope & cohesion** — the service does too many unrelated things (low functional cohesion); splitting isolates a distinct responsibility.
  2. **Code volatility** — a sub-part of the service changes at a materially different rate than the rest (commit-churn/deploy-frequency skew); splitting isolates the volatile part so its change cadence doesn't force redeploy/retest of the stable part.
  3. **Scalability & throughput** — a sub-part has different load/scaling characteristics (e.g. a document-processing/OCR-style extraction step is CPU-bound and bursty; core record CRUD is not); splitting lets each scale independently (AKF Y-axis).
  4. **Fault tolerance** — a sub-part's failure mode should not take down the rest (e.g. a flaky third-party payment integration should not block the core intake workflow); splitting bulkheads the blast radius.
  5. **Security** — a sub-part handles data under a stricter trust/compliance boundary (PII, financial/payment data) than the rest; splitting lets it carry its own trust boundary, access control, and audit surface without over-exposing the rest.
  6. **Extensibility** — a sub-part is expected to grow new variants/integrations frequently (e.g. future partner/payment integrations); splitting gives it a stable extension seam (plugin-style) without destabilizing the core.
- **Granularity integrators** (Richards & Ford, *The Hard Parts* ch. 5) — four forces, each a reason to keep (or bring back) a service *together*:
  1. **Database transactions** — an operation currently relies on a single ACID transaction across the candidate boundary; splitting breaks atomicity and requires a saga/compensating-transaction redesign (`../20-data/distributed-transaction-design.md`, the transactional-saga-taxonomy) — a real cost, not a formality.
  2. **Data dependencies** — high join/foreign-key density or shared-aggregate access across the boundary; splitting requires either data duplication (with a sync/CDC strategy) or a chatty cross-service read pattern (N+1 across the network).
  3. **Workflow & choreography** — a business process currently executes as a single in-process call sequence across the candidate boundary; splitting turns it into a distributed choreography/orchestration with its own failure-handling, idempotency, and observability cost (the 8 fallacies tax every new hop).
  4. **Service scope & cohesion** (the same force, read the other way) — the sub-parts are so tightly related (shared vocabulary, always-changed-together) that no independent lifecycle exists; splitting produces a distributed monolith, not two services.
- **The scored comparison, not a checklist tally** — this is not "6 boxes ticked beats 4 boxes ticked"; each force is weighted by the driver's (business value, technical risk) priority from the utility tree (`../shared/evaluation-method.md`), and a single High-priority disintegrator can outweigh several Low integrators, or vice versa. State the weighting explicitly per boundary.
- **AKF scale cube — Y-axis** (cross-reference `decomposition-strategy.md`) — the scalability/throughput disintegrator is the Y-axis case; this flow supplies the scored justification decomposition-strategy's granularity heuristic assumes.
- **The transactional-saga taxonomy** (`../../shared/transactional-saga-taxonomy.md`) — the required reference the moment the database-transactions integrator is rated above None; a split recommendation that crosses an existing ACID boundary must name which of the eight saga patterns would carry the now-distributed transaction, or the split verdict is incomplete.
- **Canonical tradeoff — Coupling ↔ Autonomy** (`../../shared/fallacies-and-tradeoffs.md`) — the disintegrator/integrator scoring *is* this tradeoff made concrete and measurable per boundary, rather than argued in the abstract.

## Step sequence
- **audit:** Frame the drivers + read the candidate boundary(ies) from `decomposition-strategy.md` → for each boundary, inventory the transaction/data-dependency/workflow facts that bear on the four integrators → score all six disintegrators and all four integrators with their evidence and traced drivers → compute the weighted comparison and state the verdict (split / merge / leave-as-is) for each existing or previously-proposed boundary → flag any **existing split with no scored disintegrator justification** as an over-decomposition finding (the distributed-monolith cost was paid with no force that demanded it) and any **existing merge with a High disintegrator + Low integrator score** as an under-decomposition finding → emit findings + risk register (read-only, authors no artifact, edits no source).
- **design:** **Frame** (confirm the utility tree + the candidate boundary from `decomposition-strategy.md`; restate which drivers could plausibly supply a disintegrator force — read-only) → **Explore** (`architecture-explorer` ×N produces divergent verdicts as concrete specs for the same boundary: e.g. (a) leave merged — score both sides, name the integrator that dominates; (b) split now — score both sides, name the disintegrator that dominates and the saga pattern that carries the broken transaction; (c) split later / seam-only — keep merged today but draw the internal module seam so a future split is cheap (Gall's Law: evolve from a simple system that worked); read-only) → **Evaluate** (`architecture-evaluator` scores each option's disintegrator/integrator arithmetic against the utility tree via ATAM; locates the sensitivity point — *the verdict is sensitive to [named metric, e.g. document-processing volume / payment-integration failure rate]* — and the tradeoff point — *splitting buys fault isolation but costs the ACID transaction, requiring a saga*; order-swapped to kill sophistication bias so the **option with the more honest, better-evidenced scoring** wins, not the more "microservices-native" one; picks a winner) → **Document** (the ONE writer — `solution-architect` — authors the disintegrator/integrator scorecard as a table (six + four forces, rating, evidence, driver_ref) per boundary, and **one ADR per split/merge verdict** with the rejected alternative(s) and, where a split crosses a transaction, the named saga pattern from `../20-data/distributed-transaction-design.md`) → **Verify** (check every force is scored with evidence, every Medium/High disintegrator has a driver_ref, every split verdict names a saga pattern where it breaks a transaction, diagram-compile if a boundary diagram changed, traceability boundary↔driver; the evaluator re-scores from the authored scorecard) → loop ≤ 10 or pass.

## Assertions & exit gate
- Every candidate boundary has all **six disintegrators and all four integrators explicitly rated** with evidence (no force silently skipped).
- Every disintegrator rated Medium or High **traces to a driver** in the utility tree; an untraced force cannot support a split verdict.
- A **split verdict requires** a traced High disintegrator AND an integrator counterweight no higher than Medium; where the database-transactions integrator is above None, the split names the saga pattern (`../../shared/transactional-saga-taxonomy.md`) that replaces the broken ACID transaction.
- A **merge/leave-as-is verdict is equally valid** and is the default where no disintegrator clears the bar — this flow does not have a pro-split bias.
- Every verdict is recorded as a **complete ADR** (context · ≥2 options · decision · positive and negative consequences · rejected alternatives) per `../../shared/adr-template.md`.
- **Gate:** hard oracles green (all ten forces scored with evidence · Medium/High disintegrators driver-traced · split verdicts satisfy the disintegrator-outweighs-integrator test and name a saga pattern where a transaction breaks · ADR complete) AND (design) rubric mean ≥ 0.8 (right-sized verdict, not nano-service bias · counterfactual merge considered · sensitivity stated).

## Output
Write `artifacts/service-granularity-analysis/report.json` per `../../shared/report-format.md` — findings/risks per `../../shared/finding-schema.md` (`type:"decomposition"`; `oracle:"hard"` for an unscored/unevidenced force, an untraced Medium/High disintegrator, a split verdict that crosses an ACID transaction with no named saga pattern, or a verdict with no ADR; `oracle:"soft"` for over-decomposition risk — an existing split with no disintegrator justification — or under-decomposition risk — a High disintegrator with no counterweight left merged), each carrying its `driver_ref`, the `tradeoff` (what the winning verdict costs — e.g. "split buys fault isolation on the payment integration but costs the atomic order-settlement transaction, requiring a saga with compensation") and `alternatives_rejected` (split vs merge vs seam-only, and why the losing option scored lower). For design mode, include the verification block (force-scoring completeness, driver-traceability coverage, saga-pattern-named-where-required check, hard-oracle results, rubric score). Authored artifacts land in `architecture/<slug>/` per `project-architecture-config.md` — the disintegrator/integrator scorecard folded into `tradeoffs.md`, the split/merge ADR(s) in `adr/NNNN-*.md`, any resulting boundary change reflected in `design-doc.md` and `diagrams/`. Open driver questions (an unmeasured volatility/scale signal that would firm up a force's rating) are listed explicitly for the controller — never silently assumed.

## Guardrails
Per `../../shared/guardrails.md`: **propose & document — never write product code** (this flow authors the scorecard and the ADR; the controller performs any resulting split/merge). **Ground every disintegrator in a driver** — a force with no traced quality-attribute scenario cannot justify a split (per `project-architecture-config.md`'s right-size rule: where the config locks a modular-monolith default, that shape is the correct default; do not recommend splitting absent a scored, evidenced disintegrator that clears the integrator counterweight). **Reuse `decomposition-strategy.md`'s candidate boundary** — this flow does not re-derive the seam, it decides whether to act on it. **One writer** preserves conceptual integrity of the scorecard and the ADR log. **Record the tradeoff and the rejected alternatives** on every verdict — a split with only upside, or a merge with no counterfactual considered, is incomplete. Cross-reference `decomposition-strategy.md` (the boundary this flow scores), `../20-data/data-granularity-analysis.md` (the data-sizing analog — table/collection split-or-merge using the same disintegrator/integrator lens on the data side), and `../../shared/adr-template.md` (the ADR every verdict must produce).
