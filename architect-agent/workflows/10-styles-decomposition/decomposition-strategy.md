# decomposition-strategy

**Group:** 10-styles-decomposition
**Runs as:** subagent: ../.claude/agents/solution-architect.md
**Mode:** audit + design
**Default model:** opus

## Purpose
Decide **how** the system is split into modules/services and **prove the split is not a distributed monolith.** Answers: *along which seam do we cut — business capability, bounded context, or rate-of-change (volatility) — at what granularity, and can each resulting unit be deployed, scaled, and owned without dragging the others along?* The style flow (`architecture-style-selection.md`) chooses monolith-vs-services; this flow draws the **internal seams** whichever style won — module boundaries in a modular monolith, service boundaries in a distributed system. The seam is the most expensive line in the system to move later (a one-way door), so it is judged against drivers, never drawn on a technical layer or an org-chart accident.

## Inputs & preconditions
- From `project-architecture-config.md`: the current shape (e.g. a **modular-monolith pair** — a single-page frontend + an API backend over one primary datastore; *do not propose service-splitting absent a concrete driver* — or whatever style the config actually locks), the existing module boundaries (e.g. auth/RBAC, the core business workflow + intake, account onboarding, users, audit), the near-term bounded contexts to keep seams for (e.g. documents, payment/settlement, closure), any deferred future integration seam (design the seam only), the team topology (size, ownership, branch conventions), the stack lock, and where the architecture description / ADR log / diagrams live.
- The **drivers / utility tree** from `../00-drivers/quality-attribute-scenarios.md` — the prioritized 6-part scenarios this decomposition must satisfy (independent-deploy/modifiability scenarios, isolation/security scenarios, the core-workflow consistency scenario, any team-scale driver). A decomposition with no driver to optimize *granularity* against downgrades scope and says so — you cannot say a boundary is "right" without the change it must absorb.
- The **domain model** from `../00-drivers/domain-analysis.md` — the DDD bounded contexts, the context map, and the core/supporting/generic subdomain classification. The bounded context is the *primary* candidate seam; this flow consumes the context map, it does not re-derive it.
- Target: the system/subsystem under decomposition (or the greenfield scope); the current module/service inventory and the call/data-dependency graph between candidate units.
- Preconditions: the source-of-truth docs (BRD / PRD / understanding log) named in `project-architecture-config.md` read; the utility tree exists (or this flow surfaces the missing drivers as open questions rather than guessing granularity). If no modifiability/isolation/team-scale driver is stated, **do not invent a reason to split** — record the over-decomposition risk (`project-architecture-config.md` "right-size, don't résumé-drive").

## Oracle
The **decomposition criterion** (capability / bounded-context / volatility) applied to drivers, plus the **independent-deployability** and **coupling-acyclicity** structural facts — *not* "the boxes look clean." A boundary is only justified by the change it isolates or the attribute it buys; if a split has no number and no traced driver, it is résumé-driving, not a finding (`shared/quality-attribute-rubric.md` grounding rule).
- **hard:**
  - **Independently deployable OR explicitly accepted coupling** — every proposed unit can be built, tested, deployed, and rolled back **without lockstep release of another unit**, OR the deploy-time coupling is a *named, signed-off accepted decision* in an ADR (a modular monolith deploys as one unit *by design* — that is an accepted coupling, recorded; a "microservices" set that must always deploy together is a **distributed monolith** and blocks). The test: can unit A ship a change while unit B stays on its current version? If no, and it isn't an accepted decision, that's a `hard` failure.
  - **Boundary follows capability/context/volatility, never a technical layer** — no unit is named for a tier (`*-controller`, `*-service-layer`, `*-dao`, a "frontend service" and "backend service" as the only split). A horizontal-layer cut is the canonical wrong seam (it makes every business change cross every unit). Each boundary maps to a named business capability, a DDD bounded context, or a distinct rate-of-change.
  - **No silent shared write path** — no two units write the *same* table/collection/aggregate without an explicit, ADR-recorded owner and access contract. Shared-database-as-integration across "independent" units is the distributed monolith's tell (data coupling defeats the deploy independence the split claimed). For a single-primary-datastore project (per `project-architecture-config.md`): each table/collection has exactly one writing module; cross-module reads go through a module API, not a foreign write. A silent dual-writer blocks (cross-link `dependency-management.md`).
  - **Acyclic, inward-pointing dependencies** — the candidate-unit dependency graph has **no cycles** (a cycle = two units that must change and deploy together = a hidden monolith); dependencies point toward stability/core subdomains (Stable-Dependencies Principle). A cycle blocks until broken (extract a shared kernel, invert with an event/ACL, or merge the two units — and record which, with its tradeoff).
  - **Driver-traced granularity** — every unit boundary traces to a driver: an independent-deploy/modifiability scenario, an isolation/security scenario (e.g. financial/payment data isolated from intake), a scale-axis (an AKF Y-axis split by function), or an accepted "keep-together" decision. A boundary no driver demands is accidental complexity; an unsplit blob a driver demands isolation for is an orphan requirement — both block.
- **soft:** boundaries are **stable against the likely next change** (near-term bounded contexts such as documents/payment/closure and any deferred integration seam land on their own boundaries, so building them doesn't reshape the existing ones); **granularity is right** — not nano-services (chatty, N×M failure surface, Metcalfe coupling) and not a big-ball-of-mud (no internal seams at all); **Conway-aligned** — each unit is ownable by a team that exists (for a small team, fewer well-bounded modules over a service sprawl no one can operate); **conceptual integrity** — one consistent decomposition criterion runs through the whole system (don't split half by capability and half by layer).

## Standards & techniques
- **The three decomposition criteria (cite which, and why):** **capability-based** (split by what the business *does* — e.g. order intake, settlement, onboarding; the default for aligning to business value and team ownership); **subdomain/bounded-context-based** (split on the DDD context map — each bounded context a unit, with its own model and ubiquitous language; the *primary* criterion when a rich domain model exists); **volatility-based** (Parnas information-hiding / "On the Criteria…" — encapsulate what changes at the same rate behind a stable interface, so a change is contained in one unit). State the chosen criterion per boundary; mixing them without intent fails conceptual integrity.
- **The distributed-monolith anti-pattern** (`../80-patterns/anti-patterns-fallacies.md`) — services that share a database, must deploy in lockstep, or form a synchronous chain are a monolith with network latency added: all the cost of distribution, none of the autonomy. The independent-deployability check above *is* the distributed-monolith gate. A modular monolith that deploys as one unit is **not** this anti-pattern — it never claimed deploy independence (right-sized; see `project-architecture-config.md`).
- **High cohesion, loose coupling** — maximize within-boundary cohesion (things that change together live together — functional + informational cohesion, not coincidental/temporal), minimize cross-boundary coupling (afferent/efferent coupling, instability metric `I = Ce/(Ca+Ce)`; cross-link `dependency-management.md` for the acyclic-dependencies + stable-dependencies + stable-abstractions principles that *enforce* this).
- **AKF scale cube — the Y-axis** — functional/service decomposition (split by *what it does*) is one of three independent scaling axes (X = clone, Z = shard by data). This flow owns the **Y-axis** seam; X/Z are `40-cross-cutting/scalability-architecture.md` + `20-data/partitioning-sharding.md`. Only split on Y when a driver demands functional isolation, not for resume value.
- **Conway's law & the inverse-Conway maneuver** — system structure mirrors org structure; draw boundaries the team can own, or deliberately reshape teams to *get* the boundaries you want (`shared/fallacies-and-tradeoffs.md`). For a small team, a sprawl of services is a Conway mismatch — the modular monolith is the Conway-aligned right size.
- **The canonical tradeoff — Coupling ↔ Autonomy** (`shared/fallacies-and-tradeoffs.md`): a shared DB / sync call is simple but couples (the monolith side); events/queues decouple but add eventual consistency + operational surface (the services side). Naming the point chosen on this curve — *and the driver that justifies it* — is the deliverable, not a generic "loosely coupled."
- **Granularity heuristics** — a unit should be cohesive enough to own one capability and small enough for a team to hold in its head, but not so small that a single business operation fans out across many (the N×M failure surface + distributed-transaction cost of nano-services). The 8 fallacies (`shared/fallacies-and-tradeoffs.md`) tax every boundary you cut — a boundary you didn't need is latency, partial-failure, and serialization cost you didn't need.

## Step sequence
- **audit:** Frame the drivers + read the domain context map → inventory the current units and build the **call/data-dependency graph** (who calls whom, who writes which table/collection) → for each candidate boundary, check the four hard structural facts (independently deployable or accepted-coupling · seam-on-capability/context/volatility-not-layer · no silent shared write path · acyclic & inward-pointing) and trace its granularity to a driver → run the **distributed-monolith test** (deploy-lockstep? shared DB? sync chain? cycle?) → emit findings + the risk register (read-only — authors no artifact, edits no source). Per the project's stack (per `project-architecture-config.md`): confirm each table/collection has a single writing module, flag any cross-module foreign write, and flag any proposed split with no modifiability/isolation/team-scale driver as an over-decomposition risk.
- **design:** **Frame** (confirm the utility tree + the context map; restate the modifiability/isolation/scale scenarios this decomposition must absorb — read-only) → **Explore** (`architecture-explorer` ×N produces *divergent* decompositions: e.g. (a) capability-based modules, (b) bounded-context-based with the near-term contexts (e.g. documents, payment, closure) as first-class seams, (c) volatility-based around what a deferred future integration layer will change — each a concrete spec with its unit list, the seam criterion, the dependency graph, and its Coupling↔Autonomy point; read-only) → **Evaluate** (`architecture-evaluator` scores each vs the utility tree + the four hard oracles via ATAM; locates the sensitivity points — *granularity is sensitive to team size and deploy-independence need* — and the tradeoff points — *finer granularity buys isolation/independent-deploy but costs cross-unit consistency + N×M failure surface*; order-swapped to kill position/sophistication bias so the **simpler, coarser** decomposition that meets every driver scores higher; picks a winner, grafts the superior seams from runners-up) → **Document** (the ONE writer — `solution-architect` — authors the module/service boundary view as C4-as-code (L2 Container + L3 Component where a unit is drilled), the dependency graph, the per-boundary criterion, and **one ADR per significant seam decision** with the rejected alternatives; preserves one consistent decomposition criterion across the system; reuses the existing module structure before re-cutting it) → **Verify** (the **acyclic-dependency fitness function** and the **no-cross-module-foreign-write / layering-rule fitness function** as hard gates, diagram-compile, traceability — every boundary↔driver, every driver↔boundary — and the distributed-monolith structural re-check; the evaluator re-scores from the authored artifacts) → loop ≤10 or pass.

## Assertions & exit gate
- Every proposed unit is **independently deployable**, OR its deploy-coupling is an explicit ADR-recorded accepted decision (a modular monolith's single-deploy is recorded as accepted, not silent).
- Every boundary maps to a **business capability / bounded context / volatility** — **no** unit named for a technical layer; the decomposition criterion is consistent system-wide (conceptual integrity).
- **No silent shared write path** — each table/collection/aggregate has exactly one writing unit with a recorded access contract; cross-unit data access goes through a unit API or an event, never a foreign write (per the project's stack in `project-architecture-config.md`: one writing module per table/collection).
- The candidate-unit **dependency graph is acyclic and inward-pointing** (no two units must deploy together via a cycle; dependencies point toward stable/core subdomains).
- Every boundary **traces to a driver** (modifiability/independent-deploy, isolation/security, or a scale axis) — no driver-less split (accidental complexity), no driver-demanded-but-unsplit blob (orphan requirement).
- The **distributed-monolith test passes** — no deploy-lockstep + shared-DB + synchronous-chain combination on units that claim independence.
- Each seam ADR is complete (context · ≥2 options · decision · positive **and** negative consequences · the Coupling↔Autonomy tradeoff · the rejected granularities) and diagrams compile.
- **Gate:** hard oracles green (independent-deploy-or-accepted-coupling · seam-not-layer · no-silent-shared-write · acyclic-inward deps · driver-traced granularity · distributed-monolith test) AND (design) rubric mean ≥ 0.8 (boundary stability against the likely next change · right-sizedness · Conway-alignment · conceptual integrity).

## Output
Write `artifacts/decomposition-strategy/report.json` per `shared/report-format.md` — findings/risks per `shared/finding-schema.md` (`type:"decomposition"`; `oracle:"hard"` for a distributed monolith / a technical-layer seam / a silent shared write path / a dependency cycle / a driver-less split or an orphan unsplit requirement; `oracle:"soft"` for wrong granularity / Conway-mismatch / criterion-inconsistency), each carrying its `driver_ref`, the `tradeoff` (what the seam costs on the other attributes — typically cross-unit consistency, latency, operational surface vs the isolation/independent-deploy it buys) and `alternatives_rejected` (the granularities and criteria considered and why not). For design mode, include the verification block (the acyclic-dependency + no-foreign-write fitness-function results, traceability coverage boundary↔driver, the distributed-monolith re-check, hard-oracle results, rubric score). Authored artifacts land in **`architecture/<slug>/`** per `project-architecture-config.md` — the boundary/Container view + dependency graph in `diagrams/`, the seam ADRs in `adr/NNNN-*.md`, the decomposition criterion + the unit catalog folded into `design-doc.md`, the option matrix + sensitivity/tradeoff points + risk register in `tradeoffs.md`, and the dependency-rule + foreign-write fitness checks specified in `fitness-functions.md` — never scattered. Open driver questions (an unstated team-scale or independent-deploy need that would justify finer granularity) are listed explicitly for the controller — never silently assumed to justify a split.

## Guardrails
Per `shared/guardrails.md`: **propose & document — never write product code** (this flow authors the boundary view, the ADRs, and the fitness-function *specs*; the controller refactors the modules/services). **Ground every boundary in a driver** — no split a driver doesn't demand (`project-architecture-config.md` right-size rule: do **not** propose splitting the project's modular monolith into services, a broker like Kafka, or k8s absent a concrete team-scale/independent-deploy/isolation driver — surface the over-engineering risk if asked to). **Reuse the existing module structure** before re-cutting it; prefer the **bounded context as the seam** (a known criterion with known tradeoffs) over a bespoke split. **One writer** preserves the single consistent decomposition criterion (conceptual integrity — don't let a committee leave seams of mixed criteria). **Record the tradeoff and the rejected granularities/criteria**, never just the chosen boundary — the Coupling↔Autonomy point chosen and the splits rejected are half the deliverable. Cross-reference `../00-drivers/domain-analysis.md` (the context map this consumes), `dependency-management.md` (the coupling/cohesion + acyclic-dependency principles that enforce the seam), and `../../shared/architecture-principles.md` (the senior operating habits + review checklist).
