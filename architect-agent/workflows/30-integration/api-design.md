# api-design

**Group:** 30-integration  ·  **Runs as:** subagent: ../.claude/agents/integration-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose
Design synchronous API surfaces — REST (resource modeling, Richardson maturity), GraphQL (schema design, N+1, cost limiting), or gRPC (internal high-throughput binary contracts) — choosing the style per use-case, with consistent pagination, filtering, error envelopes, idempotency, and versioning. Answers: *which API style is right for this surface, and what are the concrete structural rules every endpoint must follow?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack (API framework + request/response schema layer, the project's auth scheme, RBAC scoping), the deployment target (per the config's topology), the compliance regime (data-sensitivity classification, RBAC tenant/entity scoping, audit logging), and the artifact locations (`architecture/<slug>/`).
- The architecture **drivers**: the prioritized quality-attribute scenarios (the utility tree from `workflows/00-drivers/quality-attribute-scenarios.md`) + constraints + the open driver questions (availability SLO, concurrent-user expectations, and tenant count are not yet stated in the source-of-truth docs — surface them as open questions, never invent them). A flow with no drivers downgrades scope and says so.
- The **contract-first artifact** (OpenAPI/AsyncAPI/protobuf schema) produced by `workflows/10-styles-decomposition/api-first-contracts.md` — the machine-readable contract is the source of truth this flow refines, not the implementation. If the contract does not exist yet, this flow's design step authors it as its first deliverable.
- The **consumer surface map**: which clients consume which endpoints (SPA, mobile partner app, internal service, third-party), at what load, and whether the consumer is internal or external — this is the primary driver that selects the API style per surface (see Oracle below).
- **Preconditions:** the utility tree has measurable response measures for the performance scenarios (e.g. `dashboard render < 2s` → imposes a latency budget the API must stay within); the trust boundaries and RBAC rules are known (`security-architecture` baseline); and the `api-first-contracts.md` artifact or its open status is confirmed.

## Oracle (source of truth)

**hard:**
- **Style is driver-chosen, not fashion-chosen.** Public or partner-facing surfaces and browser SPAs with varied-projection needs → REST (Richardson level 2 minimum, level 3 HATEOAS only where a driver demands discoverability) or GraphQL (only where flexible projection eliminates a measurable over-/under-fetch problem traced to a stated driver). Internal high-throughput service-to-service calls (when a concrete latency or throughput scenario demands it, not by default) → gRPC/protobuf. A style chosen without a named driver is a hard-oracle failure (`oracle:"hard"`, `type:"integration"`).
- **Contract is machine-readable and schema-first.** Every API surface has an OpenAPI 3.1 (REST), GraphQL SDL, or `.proto` (gRPC) file as the *source of truth* — the implementation is generated from or validated against it. A surface with no machine-readable contract blocks (`api-first-contracts.md` hard oracle). This is the structural-checkability anchor: the CI gate lints the schema, diffs it against the prior version to enforce backward-compatibility, and generates client stubs.
- **Error envelope is consistent across all endpoints on a surface.** A single error shape (`{ "error": { "code": "<domain.SNAKE_CASE>", "message": "<human>", "detail": {} } }` for REST/GraphQL; a canonical `Status` + error details proto for gRPC) is used everywhere. Mixed error shapes on the same surface block.
- **Pagination, filtering, and sorting follow one convention.** Cursor-based pagination (a `next_cursor` token, opaque to the client) for large, append-heavy collections (e.g. an orders or events list — ordering is stable, offset pagination drifts on inserts); offset (`page`/`page_size`) only where the dataset is small, bounded, and random-access is a driver. Filtering by named query params (REST) or typed `filter` input types (GraphQL), never ad-hoc string injection. A surface that mixes cursor + offset without a driver-stated reason blocks.
- **Idempotency is defined for every unsafe operation.** HTTP `POST` endpoints that create resources (e.g. submit an order, create an account) carry an `Idempotency-Key` header (RFC-draft); the server returns the cached response for a duplicate key within a stated TTL. `PUT` and `PATCH` are idempotent by HTTP semantics and the design must honor that (same input → same state, no side-effect accumulation). An unsafe operation with no idempotency definition on a write path traced to a consistency/reliability driver blocks.
- **Versioning strategy is stated and enforced.** URI path versioning (`/api/v1/…`) for public/partner REST surfaces (most visible, least surprising, easy to route); version negotiation via `Accept` headers for partner APIs where multiple live versions must coexist without URL sprawl. GraphQL surfaces evolve by field-level deprecation (no breaking removal without a deprecation window). A surface with no stated versioning strategy blocks when external consumers exist.
- **GraphQL cost is bounded.** If GraphQL is chosen: query depth limit ≤ stated max (a fixed threshold declared in the service configuration and enforced by the execution engine — no stated limit blocks), query complexity budget (assign a cost to each field/resolver, reject queries exceeding a budget threshold), and DataLoader-based batching to eliminate N+1 resolver calls. An unbounded GraphQL surface (no depth limit, no cost limit, no DataLoader audit) blocks.
- **Failure handling per the 8 fallacies.** Every cross-service or external call made by the API layer (downstream enrichment, async dispatch, a third-party payment or document-processing integration) has: a bounded timeout (not infinite — Fallacy 1: the network is not reliable), a retry policy with exponential backoff + jitter (idempotent paths only), and a fallback or fail-fast that returns a meaningful error envelope to the caller. No unbounded synchronous call chain. Cross-reference `workflows/30-integration/service-communication-resilience.md` for the timeout/retry/circuit-breaker pattern catalog.
- **API contract clean — Spectral lint gates every contract change (`quality-attribute-rubric.md` → API contract clean).** Every OpenAPI 3.1 / AsyncAPI file lints with **zero errors** against a committed ruleset (`spectral lint api-contract.yaml --ruleset .spectral.yaml`, extending `spectral:oas` / `spectral:asyncapi`) *before* the contract is treated as reviewable — style rules (operationId present, tag consistency, description completeness) and hard structural rules (no undefined `$ref`, every response has a schema, security scheme applied) both run in this one pass. A contract that fails Spectral lint blocks the step-sequence Verify gate; it never reaches the evaluator. This is the **first** of two contract gates — lint checks the contract is well-formed, the breaking-change diff (below) checks it didn't silently change the deal with existing consumers.
- **API contract clean — breaking-change detection gates every REST/gRPC contract diff.** For REST/OpenAPI: **`oasdiff breaking <base>.yaml <head>.yaml`** runs on every PR that touches the contract file, classified against oasdiff's 490+ built-in change-type taxonomy (field removed, type narrowed, required-field added, enum value removed, response code removed, security requirement weakened, etc.); any change oasdiff marks `breaking` blocks the merge unless it ships with a MAJOR version bump (URI path or `Accept`-header) **and** a signed-off deprecation-window ADR — never a silent same-version break. For gRPC/Protobuf surfaces: **`buf breaking --against '.git#branch=main'`** runs the equivalent check against the `.proto` file (field-number reuse, type change, required-ness change, removed RPC) — the wire-compatibility analog of oasdiff for binary contracts. A surface with no breaking-change gate wired into CI is a hard-oracle failure regardless of how clean the current contract looks, because the next PR is the one that breaks a consumer. Cross-reference `workflows/10-styles-decomposition/api-first-contracts.md` (the contract-diff fitness function this gate implements) and `workflows/60-evaluation/fitness-functions.md` (CI wiring).
- **Stamp coupling bounded (`quality-attribute-rubric.md` → Stamp coupling bounded).** For every consumer–payload pair on the surface, compute `fields_consumed / fields_in_response`. A consumer using **< ~20% of a payload's fields** is a stamp-coupling finding: the endpoint is over-fetching for that consumer and coupling it to the whole shape, not the slice it needs. The fix is structural, not advisory — tighten via a **projection** (`?fields=`/sparse fieldset scoped to that consumer), a **BFF** (`workflows/30-integration/api-gateway-bff.md`) that shapes the response per client, or a **consumer-driven contract** (Pact-style, per `api-first-contracts.md`) that pins the subset actually used so the provider can evolve the rest freely. A shared "God response" reused by 5 consumers each touching a different 15% is 5 stamp-coupling findings, not one — measure per consumer, not per endpoint. This check runs from the consumer surface map (Inputs) cross-referenced against the response schema — a structural fact, not a judgment call.

**soft:**
- **Conceptual integrity across endpoints.** Resource naming is consistent (nouns, not verbs; `/orders/{id}/line-items/{item}` not `/getOrderLineItem`); HTTP method semantics are honored (`GET` is safe + idempotent; `POST` creates; `PUT` replaces; `PATCH` partial-updates); status codes are used precisely (201 on create, 202 on accepted-async, 409 on conflict, 422 on semantic validation failure — not all errors returning 400). A surface that solves the same retrieval problem three different ways (by ID, by filter, by search) without a driver for each fails conceptual integrity.
- **No structural over-fetching or under-fetching baked into the resource model.** REST responses return the resource that matches the use-case projection, not the whole domain object with every nested relation embedded. Where a SPA needs varied projections across views (e.g. a list view needs 5 fields; a detail view needs 30), either: sparse fieldsets via `?fields=` (REST), GraphQL, or distinct resource endpoints per projection — chosen by the number of consumer projections and the measurable payload cost, not by habit.
- **GraphQL cost model is operationally visible.** Cost and depth limits are logged per request so operations can tune the budget over time; a request that would exceed the budget is rejected with a structured error, not silently truncated or timed out.

## Standards & techniques
- **Richardson Maturity Model (RMM)** — the 4-level model (0: HTTP tunnel; 1: resources; 2: HTTP verbs; 3: HATEOAS). Level 2 is the minimum bar for any new REST surface; level 3 HATEOAS (`_links` in the response) is applied only where a driver demands runtime discoverability (e.g. a HAL-navigable partner integration) — not by default. The RMM levels are the structural gate for the REST-style hard oracle: a level-0 surface (every call POSTed to one URL) blocks.
- **Resource modeling + noun-first URL design** — identify the bounded resources (per the project's domain model — e.g. orders, accounts, users, line-items, documents), their canonical URL hierarchies, and the relationships between them. Sub-resource nesting (`/orders/{id}/line-items`, `/accounts/{id}/users`) when the child is always scoped to the parent; separate top-level resources when the child has independent identity. Max nesting depth of 2 (one level of sub-resource) before flattening.
- **GraphQL schema design + DataLoader N+1 mitigation** — define the SDL top-down (Query/Mutation root fields → types → scalars); use `@deprecated` for field removal with a migration window; batch all resolver calls that fetch by ID list through DataLoader (one SQL/Mongo round-trip per batch window, not one per object); measure resolver p95 latency per field to locate hot resolvers before they hit a cost budget. The N+1 problem: a list query that resolves N objects, each triggering 1 child fetch = N+1 queries — DataLoader collapses these to 2.
- **gRPC / Protocol Buffers for internal high-throughput** — `.proto` file is the contract; use proto3; name conventions follow Google API Design Guide (`snake_case` fields, `UpperCamelCase` messages); streaming RPCs for large result sets or server-push; use `google.rpc.Status` + `google.rpc.ErrorInfo` for structured errors. Only when a latency or throughput driver makes REST/JSON insufficient — binary framing and HTTP/2 multiplexing eliminate most REST overhead for internal calls.
- **Idempotency keys (RFC-draft / Stripe pattern)** — client-generated UUID4 in `Idempotency-Key` header; server stores `(key → response)` with a TTL (e.g. 24 h in a cache store or the existing durable audit/event store, per the project's scale — see `project-architecture-config.md`); returns the cached response on duplicate within TTL; after TTL expires the key is retired (a new request with the same key is a new operation). The key is scoped to the authenticated user/tenant to prevent cross-tenant replay.
- **Cursor-based vs offset pagination** — cursor pagination: the response includes a `next_cursor` (an opaque, base64-encoded sort key, e.g. `{id, created_at}`); the client passes `cursor=<token>&limit=N`; the server returns the next page after the cursor position. Stable under concurrent inserts (no page drift), efficient on large indexed collections (index seek, not `SKIP`). Offset: `page=N&page_size=M`; simple, random-access, but drifts on inserts and degrades with `SKIP` on large datasets. Choose cursor for a large, unbounded, append-heavy list (e.g. an orders or events feed); offset for bounded admin lists (a dozen records, e.g. account users).
- **Consistent error envelopes** — REST: `{ "error": { "code": "orders.ORDER_NOT_FOUND", "message": "Order ORD-2026-000123 not found", "detail": { "order_id": "ORD-2026-000123" } } }` with a machine-readable `code` in `domain.SCREAMING_SNAKE` format. GraphQL: errors in the `errors` array following the GraphQL spec, with `extensions.code` and `extensions.detail`. gRPC: `google.rpc.Status` with `details[]` of typed `ErrorInfo` protos. The code namespace (`domain.CODE`) makes errors greppable and traceable to the module that raised them.
- **Versioning strategies** — URI path (`/api/v1/`) for REST: route-level, cacheable, easy to deprecate at the gateway/reverse-proxy config layer. Header negotiation (`Accept: application/vnd.<project>.v2+json`) for multi-version coexistence without URL sprawl. GraphQL field deprecation: no version prefix; use `@deprecated(reason: "Use orderSummary instead")` and remove after a deprecation window (one sprint minimum for internal; one release cycle for external). Breaking change = new URI version (REST) or a new field alongside the old one (GraphQL) — never a silent contract change.
- **ATAM sensitivity + tradeoff analysis** for style selection: REST vs GraphQL is a tradeoff point — REST is simpler, cacheable at the HTTP layer, and requires no client-side query builder; GraphQL eliminates over/under-fetch variability but adds query-execution complexity, cost-limiting operational overhead, and N+1 risk. The decision matrix is: (consumer count × projection variability × caching requirement × team GraphQL familiarity). Evaluated per surface, not globally.
- **AKF scale cube (API dimension)** — the Z-axis (data partitioning) applied to API routing: route by tenant/account ID at the gateway layer for the multi-tenant RBAC model; this enables per-tenant rate limiting and the eventual shard-per-tenant path. Cross-reference `workflows/40-cross-cutting/scalability-architecture.md`.
- **API governance tooling — the two executable gates every contract change passes through.** (1) **Spectral** (Stoplight) — a JSON/YAML-path-based rules engine for OpenAPI 3.x and AsyncAPI; ships `spectral:oas`/`spectral:asyncapi` built-in rulesets and supports a project-custom ruleset (`.spectral.yaml`) for house style (naming, required descriptions, tag taxonomy) alongside structural correctness (dangling `$ref`, missing response schemas, undocumented security). Runs as a CLI (`spectral lint`) or a CI action — zero-config to start, ruleset grows with the project. (2) **oasdiff** — a purpose-built OpenAPI diff/breaking-change detector; classifies a contract diff against **490+ named change types** (its `checker` catalog covers request/response schema changes, parameter changes, security-requirement changes, server/path changes) and emits a `breaking` / `non-breaking` / `unclassified` verdict per change — this is what turns "did this PR break someone" from a manual reviewer judgment call into a deterministic CI check. (3) **`buf breaking`** — the Protobuf/gRPC analog: compares the current `.proto` set against a baseline (git ref, another module, or the BSR) and flags wire- and source-incompatible changes (field renumbering, type changes, removed RPCs/services) per a configurable rule category (`WIRE`, `WIRE_JSON`, `FILE`, `PACKAGE`). Use `buf breaking` instead of hand-rolled `protoc` diffing wherever gRPC is the chosen style (per the style-selection oracle above).
- **Stamp coupling (Page-Jones / connascence literature) — the measurement discipline.** Stamp coupling is the classic coupling type where a module receives a composite data structure but uses only part of it, coupling it to fields it never touches. The rubric's `< ~20%` threshold operationalizes this: for each consumer, walk the response schema fields actually read (from the consumer's known query/rendering path in the consumer surface map — SPA view, mobile screen, downstream service handler) vs. the total fields the endpoint returns; a ratio below the threshold is the finding. This is the API-layer instance of the same `connascence ≤ contract` hard oracle in `quality-attribute-rubric.md` — stamp coupling is dynamic connascence-of-meaning riding on top of a payload that's nominally just "the response."

## Assertions & exit gate
- Every API surface has a machine-readable contract (OpenAPI 3.1 / GraphQL SDL / `.proto`) that compiles/lints cleanly; the contract is the source of truth, not the implementation.
- API style (REST / GraphQL / gRPC) per surface is traced to a named driver (consumer type, projection variability, throughput scenario) in the ADR — no style choice without a driver.
- One error envelope convention across all endpoints on a surface; envelope has a machine-readable `code` in `domain.SCREAMING_SNAKE` format.
- One pagination convention per surface; cursor-based for large or append-heavy collections; convention is documented in the contract.
- Every `POST`/`PUT`/`PATCH`/`DELETE` path with a consistency or reliability driver has an explicit idempotency definition (key scheme, TTL, server-side store, duplicate-response behavior).
- Versioning strategy stated; breaking-change definition documented; deprecation window defined for external consumers.
- GraphQL surfaces (if present): depth limit ≤ stated max, complexity budget defined per field, DataLoader wrapping all list-resolver child fetches.
- Every cross-service call on the API path has a bounded timeout, a retry policy (idempotent paths only, with backoff + jitter), and a fallback or fail-fast — Fallacies 1 + 2 honored.
- **`spectral lint` runs on every contract (REST/GraphQL) with zero errors against the committed ruleset** — the API-contract-clean gate, executed *before* the breaking-change diff.
- **`oasdiff breaking` (REST) or `buf breaking` (gRPC/Protobuf) is wired into CI on every PR that touches a contract file** — every diff is classified breaking/non-breaking/unclassified; a `breaking` verdict blocks merge unless paired with a MAJOR version bump + a signed-off deprecation-window ADR.
- **Every consumer–payload pair has a computed stamp-coupling ratio (`fields_consumed / fields_in_response`); any consumer below ~20% carries a remediation decision** (projection/sparse-fieldset, BFF, or consumer-driven contract) recorded in the design doc or an ADR — not left as a silent over-fetch.
- Every ADR (style selection, pagination strategy, idempotency scheme, versioning policy, error envelope) has context · ≥2 options · decision · positive AND negative consequences (`adr-template.md`).
- Diagrams compile/render (Mermaid `sequenceDiagram`, OpenAPI spec renders in Redoc/Swagger UI without errors).
- Traceability matrix: every endpoint traces to a QAS or functional requirement; no orphan route; no unjustified endpoint.
- **Gate:** all hard oracles green (contract lints via Spectral, breaking-change gate wired via oasdiff/buf breaking, style-driver traced, consistent error envelope, pagination convention stated, idempotency defined for unsafe writes, versioning strategy present, failure handling per fallacies, stamp coupling bounded (≥ ~20% fields-consumed per consumer or a remediation decision recorded), ADRs complete, diagrams compile, traceability complete) AND rubric mean ≥ 0.8 (conceptual integrity across endpoints, no structural over/under-fetch, GraphQL cost operationally visible).

## Guardrails
Per `shared/guardrails.md`: propose & evaluate — never write product code, never edit `src/**` / `frontend/**` / `backend/**`, never run `terraform apply` or touch a live system. The contract stubs and ADRs are the deliverable; the controller writes the route implementations against them. Ground every style decision in a named driver — the consumer surface map + the utility tree are the oracle, not API fashion. Reuse the project's existing API framework + schema-validation stack (per `project-architecture-config.md`) before introducing a new one; a new dependency (e.g. Strawberry/Apollo for GraphQL, grpcio for gRPC) needs a driver-justified ADR with its operational-cost tradeoff stated. One writer (`solution-architect`) authors all artifacts in `architecture/<slug>/` so the error envelope, pagination convention, and versioning policy are a single coherent set of decisions, not a committee's seams. Record every rejected alternative (REST-only rejected for the partner surface because projection variability is a driver; gRPC rejected internally because the team lacks proto expertise and REST latency is within budget) — the rejected alternatives are half the value of the ADR. The API-governance gates (Spectral, oasdiff/buf breaking, the stamp-coupling scan) are *specified and wired into the fitness-functions spec*, never executed against a live registry/broker by this flow — the controller's CI runs them; this flow authors the ruleset, the CI step definition, and the pass/fail threshold. Cross-reference `workflows/10-styles-decomposition/api-first-contracts.md` (contract-first hard oracle, the machine-readable schema as source of truth, and the contract-diff fitness function this flow's oasdiff/buf-breaking gate implements concretely), `workflows/30-integration/api-gateway-bff.md` (gateway responsibilities, BFF per client, edge authn + rate-limit — a BFF is also the structural fix for a flagged stamp-coupling finding), `shared/quality-attribute-rubric.md` (the `API contract clean` and `Stamp coupling bounded` hard-oracle rows this flow implements), and `shared/diagramming-standards.md` (diagrams-as-code compile oracle, C4 notation conventions, Mermaid `sequenceDiagram` for the write-path flow).
