# api-gateway-bff

**Group:** 30-integration  ·  **Runs as:** subagent: ../.claude/agents/integration-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Decide the edge architecture: whether an API gateway is warranted for cross-cutting concerns, and whether a Backend-for-Frontend (BFF) per client type is warranted for per-surface aggregation — without turning either into a coupling chokepoint or a smart-pipe full of business logic. Answers: *which cross-cutting concerns belong at the edge, which are best served by a BFF, and which are scope creep that belongs in a backend service?*

## Inputs & preconditions

- From `project-architecture-config.md`: the project's stack (client + API tiers, edge/proxy layer, same-origin or cross-origin topology), deployment target, team topology, scale/SLO targets, compliance regime, constraint that new datastores/runtimes need a driver-justified ADR.
- The architecture **drivers** from the utility tree (`architecture/<slug>/quality-attributes.md`): the prioritized quality-attribute scenarios (QAS-*) that bear on the edge — authentication/authorization latency budgets, rate-limiting requirements, protocol-translation needs, client-type divergence, availability SLOs, security requirements (authn, TLS termination, CORS), and any stated NFRs from the source-of-truth docs (BRD / PRD / understanding log) named in `project-architecture-config.md`. If a QAS target (auth-latency, rate-limit threshold, per-client payload shape delta) is not stated, surface it as an open driver question — do not invent a number.
- The existing integration boundary: per `project-architecture-config.md` — typically the client proxies API calls through an edge/reverse-proxy layer to the backend; state explicitly whether a broker, queue, or gateway layer already exists today.
- The **client types** in scope: the primary client surface (per `project-architecture-config.md`) and any future consumer surfaces (mobile, third-party, a deferred integration seam — keep a seam per `project-architecture-config.md`).
- Preconditions: `project-architecture-config.md` read (never hardcode project values); the utility tree (`quality-attributes.md`) read for the prioritized scenarios this decision must serve; `workflows/30-integration/api-design.md` read for the existing API contract baseline; `workflows/40-cross-cutting/security-architecture.md` read for the current authn/authz threat model (trust boundaries, RBAC, STRIDE coverage) that the edge must not duplicate in business-logic form.

## Oracle (source of truth)

**hard:**
- **Gateway scope contract (structural, traceable):** the gateway owns only cross-cutting edge concerns — authn (token validation, JWT verify), rate-limiting, TLS termination, request routing, and optionally request/response header transformation. It contains **no business logic** (no core-entity state transitions, no RBAC rule evaluation beyond a coarse allow/deny at the perimeter, no data assembly). Every business rule that creeps into the gateway is an `ARCH-APIGW-*` finding of `severity:"major"`, `oracle:"hard"`, traced to the "dumb pipes, smart endpoints" principle (Richardson) and the coupling driver it violates.
- **BFF mapping (structural):** each BFF maps 1:1 to a real, distinguishable client experience — a different payload shape, a different auth flow, a different aggregation pattern driven by the client's constraints (mobile bandwidth, SPA render cycle, agent API surface). A BFF whose response shape is identical to a raw backend response with no client-specific transformation is scope creep; surface it as `ARCH-APIGW-*` finding `severity:"minor"` `oracle:"soft"` (YAGNI — the simplest edge that meets the drivers wins).
- **No un-accepted SPOF:** the gateway (if introduced) must not be a single point of failure on any path covered by an availability QAS; either redundancy/failover is designed in, or the SPOF is an explicitly accepted risk in the register (per `shared/quality-attribute-rubric.md`).
- **Failure modes defined:** every inbound path through the gateway has a bounded timeout, retry policy (with backoff+jitter for idempotent routes), and a fallback or fail-fast — no unbounded synchronous call chains (Fallacy #1 of distributed computing: the network is not reliable).
- **Trust boundary secured (per STRIDE):** the gateway's ingress perimeter is a trust boundary; authn and authz controls at that boundary must be traced to the threat model in `workflows/40-cross-cutting/security-architecture.md`; secrets (JWT secret, TLS certs) are externalized (never in diagram or gateway config source). A gateway-trust-boundary with no STRIDE entry blocks.
- **Diagram compiles:** every C4 diagram produced is diagrams-as-code (Mermaid or PlantUML+C4-PlantUML per `project-architecture-config.md`) and renders without error. The gateway and each BFF appear as Containers in the L2 Container view with labeled sync/async relationships (per `shared/diagramming-standards.md`).
- **ADR complete (MADR):** the decision to introduce (or explicitly not introduce) a gateway and each BFF is recorded as an ADR per `shared/adr-template.md` — context · decision drivers · ≥2 considered options · decision · consequences (positive AND negative) · follow-ups. A decision with no recorded alternatives or no stated negative consequences blocks.
- **Right-sized introduction gate:** a gateway is warranted only when there are ≥2 distinct cross-cutting concerns to centralize that cannot be addressed by the existing reverse-proxy layer. If the audit finds that the existing proxy already handles TLS + routing and the only new concern is auth, the finding must evaluate whether a lightweight middleware in the backend service satisfies the driver before recommending a new gateway process (YAGNI, Gall's Law — add a gateway only when a simple system has proven it needs one).

**soft:**
- The edge is right-sized: a gateway only when cross-cutting concerns exceed what the existing proxy + backend middleware stack can bear at the driver's priority; no gateway tax on simple CRUD paths.
- BFFs don't fan out into duplication — shared logic lives in a backend service, not copy-pasted across BFFs. The BFF is a thin aggregation + projection layer, not a second backend.
- The decomposition honors Conway's Law: the team topology (per `project-architecture-config.md`) can actually own the gateway and any BFF without creating an unsupported service.
- Conceptual integrity: the edge pattern choice (gateway, BFF, both, neither) is consistently applied — not a gateway on some paths and a raw reverse-proxy on others without a stated reason.

## Standards & techniques

- **API gateway pattern** (the cloud-design-pattern catalog: *Gateway Routing*, *Gateway Aggregation*, *Gateway Offloading*) — the three distinct gateway responsibilities: (1) routing: forward requests to upstream services without knowledge of business semantics; (2) aggregation: compose multiple upstream calls into one client response (the *Gateway Aggregation* pattern — reduces client round-trips, moves fan-out to the edge); (3) offloading: centralize cross-cutting concerns (authn, rate-limiting, TLS, CORS, logging, tracing) that all services need — *Gateway Offloading* — so each backend service stops reimplementing them. Offloading is the hardest oracle for correctness: only concerns that are genuinely cross-cutting belong here.
- **BFF (Backend-for-Frontend)** (Sam Newman / SoundCloud pattern) — one backend per client experience. The principle: a mobile client has different payload-size, auth-flow, and aggregation needs than a desktop SPA; a single "one-size-fits-all" API forces the SPA or the mobile client to do excess assembly. A BFF is a thin server-side projection layer authored and owned by the front-end team, not a second domain backend. The BFF anti-pattern: a BFF that grows business logic or a DB connection of its own is a second backend wearing a BFF label — flag it.
- **"Dumb pipes, smart endpoints"** (Martin Fowler / Microservices — the inverse of ESB-style orchestration) — integration infrastructure (the gateway) carries payloads and applies edge policies; the intelligence (business rules, domain logic, state) lives in the endpoint services. Violating this principle — adding routing logic that depends on the state of a core business entity, or rate-limit rules that depend on RBAC roles beyond a simple token-scope check — is the canonical gateway-creep finding.
- **Gateway Aggregation pattern** (cloud-design-pattern catalog) — when the client needs data from N backend calls, the aggregator fans them out in parallel server-side and returns one response, hiding N×latency and N×round-trips from the client. Hard tradeoff: the aggregator is now on the critical path of all N calls; one upstream timeout must not cascade to a full-response failure (bulkhead + per-upstream timeout, not a global timeout).
- **STRIDE threat modeling** (at the gateway trust boundary) — the gateway ingress is the principal external trust boundary; Spoofing (JWT forgery), Tampering (header injection), Repudiation (missing correlation-ID / audit log), Information Disclosure (response leakage), Denial of Service (no rate-limit = gateway SPOF), Elevation of Privilege (business-logic creep bypassing RBAC). Each STRIDE category must have a stated control or an accepted risk, per `workflows/40-cross-cutting/security-architecture.md`.
- **The 8 fallacies of distributed computing** (`shared/fallacies-and-tradeoffs.md`) — introducing a gateway adds a hop; Fallacy #1 (network is reliable) and Fallacy #2 (latency is zero) apply: every gateway path must carry a latency budget, a timeout, and a failure path. Fallacy #4 (network is secure) applies at the gateway boundary — zero-trust between gateway and upstream services (mTLS or at minimum mutual authn on the internal leg), not just at the perimeter.
- **AKF scale cube** (Abbot, Fisher, Ellis) — the gateway can be the X-axis scale point (horizontal replication for availability); the BFF is a Z-axis split (partition by client type). Neither replaces Y-axis decomposition (functional split by service capability) in the backend.
- **CAP/PACELC** — the gateway is stateless by design (sessions carried in the JWT); a stateful gateway (session store, per-user rate-limit counter) introduces a consistency model. If a rate-limit counter is in Redis, state the consistency model (eventual — a brief overshoot under partition is the tradeoff against availability of the rate-limiter itself).
- **ATAM sensitivity/tradeoff points** (`shared/evaluation-method.md`) — the gateway introduction is a tradeoff point: it *improves* security (centralized authn), observability (centralized correlation-ID injection), and client simplicity (one endpoint) while it *worsens* latency (one extra hop), availability risk surface (new SPOF without redundancy), and operational cost. Name the point on the tradeoff curve explicitly.
- **MADR decision records** (`shared/adr-template.md`) — every edge decision (introduce gateway? introduce BFF? use the existing reverse-proxy as gateway? add a rate-limiter at the proxy layer?) is an ADR with ≥2 genuine options and stated consequences.

## Step sequence

- **audit:** Frame the drivers from `project-architecture-config.md` + `quality-attributes.md` → read the existing edge: reverse-proxy/edge config, backend middleware, client-side API proxy (per `project-architecture-config.md`) → walk each cross-cutting concern (authn, rate-limiting, TLS, CORS, logging, routing) and assess whether it is already centralized or scattered across service code → check the gateway-scope hard oracle (any business logic in proxy/middleware layers? RBAC evaluation at the edge beyond token validation?) → check SPOF: is the existing proxy the sole ingress with no redundancy? is that accepted? → check failure modes: do all inbound paths through the proxy have a bounded timeout and upstream-error handling? → check trust boundary: is the proxy→backend leg secured (authn/mTLS) or assumed safe because it's internal? → emit findings per `shared/finding-schema.md` (type:`integration`, traced to the driver each concern violates) — read-only, no artifact authored, no source edited.

- **design:** Frame (extract/confirm edge drivers + the client-type map — which clients exist today, which are in the seam for a future integration — read-only) → Explore (produce ≥2 genuinely divergent edge directions: e.g. Direction A — proxy-as-gateway: extend the existing reverse-proxy layer with auth-module + rate-limit, zero new process; Direction B — lightweight gateway process: introduce a minimal gateway sidecar or an off-the-shelf gateway (e.g. Kong, Traefik, Envoy) proxying to the backend; Direction C — BFF per client type with the proxy routing to each BFF, each BFF aggregating backend calls — only if a client-type divergence driver is confirmed; each direction is a concrete spec: topology, C4 L2 containers, the cross-cutting concerns it centralizes, latency budget impact, operational cost — read-only proposers) → Evaluate (one skeptical evaluator ATAM-scores each direction against the utility tree + hard oracles — gateway-scope check, SPOF check, failure-mode coverage, STRIDE, driver traceability, ADR completeness; scores each soft oracle criterion; order-swapped to kill position bias; picks a winner + grafts superior elements from runners-up; documents the sensitivity points — e.g. "auth-latency p99 is sensitive to whether JWT validation is in-process at the proxy vs a gateway sidecar RTT" — and the tradeoff point: "a dedicated gateway process improves isolation and rate-limiting precision but worsens operational complexity and adds a hop latency budget") → Document (one writer — `solution-architect` — authors the artifacts: C4 L1 + L2 diagrams with the gateway/BFF as labeled containers; ADR for the gateway decision; ADR for each BFF decision or the decision not to introduce a BFF; threat model entry per STRIDE at the gateway trust boundary; capacity/latency budget addendum; fitness functions for the gateway-scope constraint) → Verify (fitness functions: static check that no business logic lives in gateway config; diagram-compile; traceability check: every cross-cutting driver traces to a gateway/BFF decision; STRIDE coverage per gateway trust boundary; evaluator re-scores from the artifacts — loop ≤8 or pass).

## Assertions & exit gate

- Every cross-cutting concern in scope (authn, rate-limiting, TLS, routing, CORS, correlation-ID injection) either maps to a gateway component with a stated control or is explicitly handled at the backend middleware layer with a documented rationale — no unaddressed concern.
- The gateway (if introduced) contains no business logic: no core-entity-state awareness, no RBAC rule evaluation beyond coarse token-scope/audience checks. Verifiable by reading the gateway config/middleware — a structural check.
- Each BFF introduced maps to a confirmed client-type divergence driver (different payload shape, different auth flow, or different aggregation pattern); a BFF with no such driver is flagged as YAGNI (soft, advisory).
- No un-accepted SPOF on any path covered by an availability QAS: the gateway has redundancy/failover designed in or the SPOF is in the accepted-risk register with an explicit sign-off.
- Every cross-service call through the gateway has a bounded timeout (value stated, not "default"), retry policy (idempotent routes only, with backoff+jitter), and a fallback or fail-fast path documented.
- The gateway ingress trust boundary has full STRIDE coverage (control or accepted-risk per category); secrets (TLS certs, JWT signing key) are externalized from all gateway config and diagrams.
- Every significant edge decision has a complete MADR ADR: context · ≥2 options · decision · consequences (positive AND negative) · follow-up fitness functions. A decision with only positive consequences blocks.
- All C4 diagrams (L1 System Context, L2 Container showing gateway + BFF as containers with labeled relationships) are diagrams-as-code (Mermaid or PlantUML+C4-PlantUML) and render without error.
- The latency budget is stated end-to-end (client → gateway → BFF → backend service → DB → response); the gateway hop's p99 contribution is bounded (not open-ended "it'll be fast").
- The decision honors the project constraint (boring tech first): if the existing proxy + backend middleware stack satisfies every cross-cutting driver, the finding is "no gateway process warranted now" with a dated seam note for any deferred integration surface — not a gateway for its own sake.
- **Cost bounded:** the decision to introduce a gateway process (vs. extending the existing reverse-proxy) carries an order-of-magnitude operational cost estimate — at minimum: additional process/instance hosting cost, any licensing cost (Kong, Traefik, cloud APIGW), and ops overhead per `shared/quality-attribute-rubric.md`. The estimate is stated against the project's constraint (no component "scale infinitely, cost unstated"). If no budget constraint is on record, the estimate is still produced and the absence of a stated constraint is surfaced as an open driver question. A gateway recommendation with no cost estimate where operational cost is a tradeoff factor (as it always is per ATAM Standards line) blocks.
- **Data lifecycle lawful:** the gateway boundary is the principal ingress for any PII in transit (credentials, bearer tokens, user-identifying headers, core-entity payloads). Under the compliance regime noted in Inputs (the source-of-truth docs named in `project-architecture-config.md`), the gateway's handling of PII must be classified: (1) PII in transit is not logged at the gateway layer in cleartext; (2) any request/response transformation or header enrichment that carries PII has a stated retention policy (typically ephemeral — in-flight only, not persisted at the gateway); (3) if the gateway introduces a log store or an audit trail, that store's residency and erasure path are stated. An unclassified PII path at the gateway ingress under a regulated regime blocks per `shared/quality-attribute-rubric.md`.
- **Consistency stated:** any stateful gateway path (per-user rate-limit counters, session state, circuit-breaker state) declares its consistency model and mechanism. The canonical case: a Redis-backed rate-limit counter under partition accepts a brief overshoot (eventual consistency — availability prioritized over strict enforcement, per CAP/PACELC analysis in Standards). That choice is stated explicitly as the model; it is not left implicit. A stateful gateway write path with no consistency model stated blocks.

**Gate:** hard oracles green (gateway-scope structural check passes; SPOF accepted or mitigated; failure modes defined per cross-service call; consistency model stated for every stateful gateway write path; STRIDE coverage complete; diagrams compile; every ADR complete; latency budget stated; order-of-magnitude cost estimate produced against stated constraint; PII at gateway ingress classified with residency and retention paths stated) AND (design) rubric mean ≥ 0.8 across conceptual integrity, right-sizedness, appropriate coupling, evolvability, operability & observability, and tradeoff honesty.

## Output

Write `artifacts/api-gateway-bff/report.json` per `shared/report-format.md` — findings per `shared/finding-schema.md` (type:`integration`; `oracle:"hard"` for gateway-scope violations, un-accepted SPOF, missing failure mode, STRIDE gap, incomplete ADR, uncompilable diagram; `oracle:"soft"` for YAGNI / over-engineered edge / BFF duplication), plus the verification block (fitness-function results, traceability coverage, hard-oracle results, rubric score) in design mode.

Each finding's `driver_ref` traces to a `QAS-*` in the utility tree or to the named constraint it serves. Each ADR's `Driver refs` traces to the same. No orphan findings; no unjustified decisions.

## Guardrails

Per `shared/guardrails.md`: propose and evaluate — never write gateway config, BFF source code, reverse-proxy vhosts, or any product file under `frontend/**` / `backend/**`. Author only architecture artifacts (ADRs, C4-as-code diagrams, threat model entries, fitness-function specifications) into `architecture/<slug>/`. The controller implements the chosen direction.

Ground every edge decision in a driver: a gateway introduced with no cross-cutting driver is accidental complexity — flag it and recommend against it. A BFF introduced with no client-type divergence driver is the same. The grounding rule from `shared/quality-attribute-rubric.md` applies: no number + no traced driver = a preference, not a finding.

Reuse the existing edge before adding a process: the existing reverse-proxy + backend middleware stack is the boring default; the burden of proof is on the driver that demands a new gateway process, not on the team to justify keeping the simple stack.

One writer (`solution-architect`) authors all artifacts in sequence — never concurrent partial authoring. Proposers and the evaluator are read-only. The evaluator scores against traced scenarios and structural facts, not the writer's prose; it penalizes accidental complexity and unjustified novelty.

Bound the loop: ≤8 iterations of document→verify before escalating to the controller with the blocker finding.
