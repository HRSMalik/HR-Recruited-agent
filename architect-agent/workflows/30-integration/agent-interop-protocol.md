# agent-interop-protocol

**Group:** 30-integration · **Runs as:** subagent: ../../agents/integration-architect.md · **Mode:** audit + design · **Default model:** sonnet

## Purpose
Design agent-to-agent and agent-to-tool integration as a **first-class integration concern** — not an afterthought bolted onto a REST/EIP diagram. Every agent↔tool boundary (Model Context Protocol) and agent↔agent boundary (Agent2Agent) is a **trust boundary with a protocol, an auth model, and a discovery mechanism**, because the Enterprise Integration Patterns catalog (Hohpe & Woolf, 2003) predates agent delegation and has no native vocabulary for "a peer that can invoke tools on my behalf" or "a tool whose output re-enters an LLM's instruction plane." Answers: *which protocol governs each agent-interop boundary, what is its auth model and trust scope, and where does an untrusted tool/agent output cross into the LLM context (the injection boundary)?*

## Inputs & preconditions
- From `project-architecture-config.md`: the existing stack + deployment target, the compliance regime, the team topology, and — specifically for this flow — whether an agentic/LLM layer is in current scope or **deferred** to a later phase (per `project-architecture-config.md`: if the agentic AI layer is deferred, keep the integration **seam**, do not build agent internals now). A flow with no agentic component in current scope still runs in **seam mode**: it designs the boundary contract the future agent layer will plug into, without authoring the agent internals.
- The architecture **drivers**: the prioritized quality-attribute scenarios bearing on the agent-interop seam — evolvability (the seam must absorb a future agent layer without a core-domain rewrite), security (no unredacted-PII egress to a tool/model-provider, no unauthenticated tool invocation), and any latency/availability scenario on a path an agent will sit on. If no security QAS names the agent boundary yet, surface it as an open driver question — do not invent a response measure.
- The **integration inventory** for this flow specifically: every point where an LLM/agent process exchanges data with (a) a tool/external-context source (the MCP lane) or (b) a peer agent/orchestrator (the A2A lane). Today (seam mode): the placeholder boundary between the existing backend (per `project-architecture-config.md`) and the future deferred agent services. When agent components exist: every MCP server/client pair, every agent-card-advertised capability, every orchestrator↔worker-subagent call.
- Preconditions: `shared/agentic-threat-taxonomy.md` read (the OWASP-LLM-Top-10 + MAESTRO + ATLAS sweep this flow's boundaries feed into); `integration-patterns.md` read (the EIP catalog and ACL discipline this flow extends into the agent lane); `shared/diagramming-standards.md` read (the agent-interop annotation convention — see "Agent-interop boundaries annotated" under Notation conventions); `../90-specialized/ml-ai-system-architecture.md` read if an LLM/RAG/agentic component is in scope (the orchestration-topology decisions this flow's boundaries sit inside).

## Oracle (source of truth)
The Model Context Protocol specification (Anthropic, Nov 2024), the Agent2Agent protocol specification (Google, Apr 2025, now under the Linux Foundation), and `shared/agentic-threat-taxonomy.md` (OWASP LLM Top 10 2025 + CSA MAESTRO) are the external truth for what a compliant, trust-scoped agent-interop boundary looks like.

**hard:** (deterministic — traceable, measurable, or structurally checkable; block on failure)
- **Every agent↔tool boundary names MCP (or a stated, justified alternative) as its protocol.** Where an LLM/agent process invokes an external tool, file, database, or context source, the integration is either MCP-conformant (client↔server over stdio/HTTP+SSE/Streamable-HTTP, tools/resources/prompts as MCP primitives) or the deviation is a driver-justified ADR (e.g., a direct SDK call for a single first-party tool with no reuse need). A tool call with no named protocol and no ADR is `type:"integration"`, `severity:"major"`, `oracle:"hard"`.
- **Every agent↔agent boundary names A2A (or a stated, justified alternative) as its protocol.** Where one agent delegates a task to a peer or sub-agent across a process/service boundary (not an in-process function call), the integration is either A2A-conformant (Agent Card capability advertisement, task lifecycle via JSON-RPC 2.0 over HTTP(S), `tasks/send` / `tasks/sendSubscribe` semantics) or the deviation is a driver-justified ADR. An unnamed bespoke agent-delegation mechanism is `type:"integration"`, `severity:"major"`, `oracle:"hard"`.
- **Every agent-interop boundary states its auth model.** Each MCP server and each A2A peer declares an explicit authentication mechanism (OAuth 2.1 for remote MCP servers per the 2025-03-26+ spec revision, mTLS or signed service-identity tokens for A2A peer calls, or an equivalent named mechanism) — never ambient trust ("the caller is on the same network" is not an auth model). A boundary with no stated auth model is `type:"security"`, `severity:"critical"`, `oracle:"hard"`.
- **Every agent-interop boundary states its trust scope.** The permission/capability scope granted across the boundary is named and is the minimum required (which tools an MCP client may invoke, which task types an A2A peer may accept, what data classes may cross). An unscoped ("full tool access") grant with no least-privilege justification is `type:"security"`, `severity:"major"`, `oracle:"hard"` (excessive agency — OWASP LLM06).
- **Tool output crossing into an LLM context is treated as untrusted input.** Any MCP tool result, resource fetch, or A2A task response that is concatenated into a prompt or otherwise enters the model's instruction plane is passed through an explicit sanitization/validation step (schema validation, content-type allowlisting, prompt-injection scanning, or a structured-output constraint) before use — it is never trusted verbatim as an instruction. This is the **injection boundary**: an MCP/A2A response is architecturally indistinguishable from adversarial input unless proven otherwise. A tool-output path with no validation step feeding an LLM context is `type:"security"`, `severity:"critical"`, `oracle:"hard"` (tool poisoning / prompt-injection-via-tool-output, OWASP LLM01).
- **Capability discovery is authenticated.** Where agent capability advertisement is used (an A2A Agent Card served at `/.well-known/agent.json` or equivalent, an MCP server's `tools/list` response), the discovery endpoint itself is authenticated before the capability list is trusted for invocation planning — an unauthenticated discovery response is a spoofing vector (agent-card spoofing: a malicious actor advertises a capability it doesn't actually have, or impersonates a trusted agent's identity to be selected for delegation). An unauthenticated discovery endpoint feeding an orchestrator's routing decision is `type:"security"`, `severity:"major"`, `oracle:"hard"`.
- **Diagram compiles and the agent-interop boundary is annotated.** Every diagram touching an agent-interop boundary is diagrams-as-code (per `project-architecture-config.md` tooling) and renders without error; the boundary is drawn per `shared/diagramming-standards.md` → Notation conventions → "Agent-interop boundaries annotated" — labeled with protocol (MCP/A2A), auth model, and trust scope, visually distinct from a plain synchronous/asynchronous call.
- **ADR complete (MADR).** The protocol choice at each significant agent-interop boundary (MCP vs a direct SDK call; A2A vs an in-process orchestration function; the auth mechanism; the trust-scope grant) is recorded per `shared/adr-template.md` with context · ≥2 options · decision · consequences (positive AND negative). A protocol choice with no ADR is `oracle:"hard"`.

**soft:** (craft — rubric-scored by the evaluator after hard oracles are green)
- **The protocol choice fits the interaction shape.** MCP is chosen for agent↔tool/context access (the agent needs data or a capability, not a peer relationship); A2A is chosen for peer-agent delegation (one agent hands off a task to another autonomous agent that plans and executes independently). Using A2A where a plain MCP tool call would do (wrapping a stateless function as a "peer agent") is over-engineering; using MCP to fake peer delegation (chaining tool calls to simulate agent autonomy) hides the actual control-flow. Named per interaction, not assumed.
- **The simplest sufficient integration is chosen.** A single in-process function call needs neither protocol — reach for MCP/A2A only when the boundary is genuinely cross-process/cross-trust. Reserve full A2A task-lifecycle machinery (streaming, push notifications, multi-turn negotiation) for interactions that need it; a single request/response exchange between agents may be the bare-minimum ADR tier.
- **Trust boundaries are explicit on the diagram**, not just in the ADR prose — a reviewer scanning the C4 diagram alone should be able to see every agent-interop edge, its protocol, and that it is a trust boundary (not inferred from a separate document).
- **Emerging protocols are watched, not chased.** ANP (Agent Network Protocol) and other emerging agent-interop proposals are tracked in the taxonomy note below as **watch-list, not a gate** — do not require conformance to an unratified or immature protocol; do not penalize a design for not adopting one.

## Standards & techniques
- **Model Context Protocol (MCP)** (Anthropic, Nov 2024; open spec, JSON-RPC 2.0 base) — the standard for connecting an LLM/agent to tools, data sources, and prompts. Client↔server architecture: an MCP **host** (the agent application) runs one or more MCP **clients**, each holding a 1:1 session with an MCP **server** that exposes **tools** (invokable functions), **resources** (readable context — files, DB rows, API responses), and **prompts** (reusable prompt templates). Transports: stdio (local process, trusted same-machine boundary) or Streamable HTTP with SSE (remote, crosses a network trust boundary — requires the OAuth 2.1 authorization framework per the 2025-03-26 spec revision, including PKCE and dynamic client registration). Use this flow to decide: is this integration MCP-shaped (tool/context access) — and if remote, is OAuth 2.1 wired, not just assumed?
- **Agent2Agent Protocol (A2A)** (Google, Apr 2025; donated to the Linux Foundation mid-2025) — the standard for agent-to-agent task delegation across organizational/framework boundaries. Core primitives: the **Agent Card** (a JSON capability manifest served at a well-known URL — declares skills, input/output modes, and the auth schemes the agent accepts), the **task** (the unit of work with a lifecycle: submitted → working → input-required → completed/failed/canceled), and the transport (JSON-RPC 2.0 over HTTP(S), with `tasks/send` for synchronous and `tasks/sendSubscribe`/push-notification webhooks for long-running async tasks). A2A is explicitly complementary to MCP, not competing: MCP connects an agent to its tools; A2A connects an agent to another agent. Use this flow to decide: is this a genuine peer-delegation relationship (A2A) or a tool call dressed up as one (MCP, or a plain internal function call)?
- **Agent capability discovery / Agent Cards** — the mechanism by which an orchestrator or peer agent learns what another agent can do before delegating. Treated as a **trust-sensitive read**: the discovery response feeds a routing/delegation decision, so an unauthenticated or unverified Agent Card is a spoofing vector (a hostile actor publishes a card claiming a trusted agent's identity or capabilities it will not actually honor). Mitigation: verify the serving domain/identity (TLS + domain ownership, or a signed card) before trusting the capability list for invocation planning.
- **Protocol trust boundaries** — the three concrete risk classes this flow gates on, per `shared/agentic-threat-taxonomy.md`:
  - **Tool poisoning** — a malicious or compromised MCP server returns a tool result crafted to manipulate the calling agent's subsequent behavior (distinct from prompt injection in the user input — this is injection *via the tool-output channel*, which developers reflexively trust more than user input and therefore validate less).
  - **Prompt-injection-via-tool-output** — any MCP resource fetch, tool result, or A2A task response that gets concatenated into the model's context window is a fresh injection surface; the fix is the sanitization/validation step named in the hard oracle above, not "the model will probably ignore it."
  - **Agent-card spoofing** — impersonating a trusted agent's identity or advertising false capabilities at the discovery layer, covered by the authenticated-discovery hard oracle above.
- **OWASP Top 10 for LLM Applications 2025** — this flow's hard oracles operationalize **LLM01** (prompt injection, via the tool-output injection boundary), **LLM06** (excessive agency, via the least-privilege trust-scope oracle), and **LLM07** (system-prompt leakage, relevant when a tool/agent response could be used to extract instructions) at the specific granularity of agent-interop boundaries; the full sweep across all ten categories and all MAESTRO layers is `90-specialized/agentic-threat-model.md` — this flow is narrower and protocol-specific, that flow is the exhaustive security sweep.
- **CSA MAESTRO** L3 (Agent Frameworks) and L7 (Agent Ecosystem) — the layers this flow's boundaries live in: L3 covers the orchestration framework's tool-permission configuration (which this flow's trust-scope oracle enforces at the individual-boundary level); L7 covers cross-organization/cross-framework agent ecosystems (which A2A's Agent Card + auth-scheme negotiation is designed to secure).
- **Anti-Corruption Layer (ACL)** (Evans, DDD ch. 14; `integration-patterns.md`) — applied at the agent-interop seam exactly as at any foreign-system boundary: a translation layer between an external agent's/tool's data shape and the core domain's ubiquitous language, so a third-party MCP server's schema drift or a peer agent's task-result shape does not leak into core-domain code. The seam-mode deliverable for a deferred agentic layer (per `project-architecture-config.md`) *is* this ACL placeholder — a thin, named interface the future agent services will plug into.
- **Enterprise Integration Patterns (EIP)** (Hohpe & Woolf, 2003; `integration-patterns.md`) — MCP and A2A are new **channel + endpoint** types the 2003 catalog didn't anticipate, not a replacement for it: an MCP tool call is functionally a synchronous Point-to-Point Channel with a Message Translator (the MCP JSON-RPC envelope ↔ the tool's native call shape); an A2A task delegation with `tasks/sendSubscribe` is functionally a Publish-Subscribe Channel plus a Process Manager (the task lifecycle state machine). Naming the EIP-equivalent alongside the protocol name keeps this flow's findings consistent with the rest of `30-integration/`.
- **The 8 fallacies of distributed computing** (`shared/fallacies-and-tradeoffs.md`) — apply at full force to agent-interop calls: Fallacy 1 (the network is not reliable — every MCP/A2A call needs a timeout + retry + fallback, same as any cross-process call); Fallacy 4 (the network is not secure — this is *why* the auth-model and trust-scope hard oracles exist); Fallacy 6 (there is not one administrator — a third-party MCP server or a partner's A2A agent is under someone else's change control; version and negotiate capabilities, don't assume stability).
- **MADR decision records** (`shared/adr-template.md`) — every protocol choice, auth mechanism, and trust-scope grant at a significant agent-interop boundary is an ADR with ≥2 genuine options and stated positive+negative consequences.
- **ATAM sensitivity/tradeoff analysis** (`shared/evaluation-method.md`) — the key agent-interop tradeoff points: *MCP tool call ↔ A2A task delegation* (a tool call is simpler and stateless but the calling agent retains all planning; a task delegation offloads planning to the peer but adds a full task-lifecycle state machine and a second point of autonomous decision-making to audit); *centralized Agent Card registry ↔ per-agent discovery endpoint* (a registry is a single place to audit and revoke but is a SPOF and a spoofing target if compromised; per-agent discovery is resilient but multiplies the number of authenticated endpoints to secure); *strict tool-output validation ↔ agent flexibility* (heavier sanitization closes the injection boundary but can reject legitimate edge-case tool output, forcing tighter schemas). Name the point chosen, not just the winner.

## Step sequence
- **audit:** Frame the agent-interop drivers (evolvability seam + security QAS + any latency/availability scenario on an agent-touched path) from `project-architecture-config.md` + the utility tree → enumerate the agent-interop inventory: every agent↔tool call (existing or seam-placeholder) and every agent↔agent/orchestrator↔sub-agent call → for each boundary: (a) check whether MCP or A2A (or a justified alternative) is named as the protocol; (b) check whether an auth model is stated and is not ambient trust; (c) check whether the trust/permission scope is stated and minimal; (d) check whether any tool-output/task-response path feeding an LLM context has a sanitization/validation step; (e) check whether capability-discovery (Agent Card / `tools/list`) is authenticated before being trusted for routing; (f) check ACL presence at the seam (no foreign tool/agent schema leaking into core domain) → emit findings per `shared/finding-schema.md` (`type:"integration"` or `"security"`; `oracle:"hard"` for missing protocol name, missing auth model, unscoped trust grant, un-sanitized tool-output-to-LLM path, unauthenticated discovery; `oracle:"soft"` for a protocol mismatch to the interaction shape or an over-built A2A task machine where a plain MCP call would do) — read-only, no artifact authored, no source edited.
- **design:** Frame (confirm the in-scope agent-interop boundaries — real or seam-placeholder — and the evolvability/security drivers; read-only) → Explore (≥2 genuinely divergent directions per significant boundary — e.g., for a deferred agent-layer seam: Direction A — a single MCP server fronting the backend's core-domain capabilities as tools, agents call in via one authenticated channel; Direction B — a per-bounded-context MCP server (one per core domain area) each with its own auth scope and ACL, composed by an A2A orchestrator; each direction stated as a concrete spec: protocol per boundary, auth model, trust scope, ACL placement, injection-boundary controls — read-only proposers) → Evaluate (one skeptical ATAM evaluator scores each direction against the utility tree + hard oracles: protocol named, auth model stated, trust scope minimal and justified, injection boundary controlled, discovery authenticated; scores soft oracles: protocol-to-interaction fit, simplicity, boundary explicitness on the diagram; order-swapped; names sensitivity points — e.g., "the evolvability of the deferred-agent-layer seam is sensitive to whether one shared MCP server or per-context servers are chosen: a shared server is simpler to stand up now but couples all future agent capabilities to one auth/trust-scope surface" — and tradeoff points) → Document (one writer — `solution-architect` — authors: the C4 diagram annotation for each agent-interop boundary per `shared/diagramming-standards.md`; ADRs for the protocol choice, auth mechanism, and trust-scope grant per boundary; the ACL/seam placeholder specification; the tool-output sanitization/validation control spec; fitness-function specifications for the injection-boundary and auth checks) → Verify (fitness functions: a structural check that every agent-interop boundary in the diagram carries a protocol label + auth model + trust scope; a check that no tool-output/task-response path reaches an LLM context without a named validation step; diagram-compile; traceability: every agent-interop driver traces to a boundary decision; evaluator re-scores from artifacts → loop ≤8 or pass).

## Assertions & exit gate
- Every agent↔tool boundary in the inventory names MCP (or a justified ADR'd alternative) as its protocol; every agent↔agent/orchestrator↔sub-agent boundary names A2A (or a justified ADR'd alternative).
- Every agent-interop boundary states an explicit auth model (OAuth 2.1 for remote MCP, mTLS/signed identity for A2A, or an equivalent named mechanism) — no ambient-trust boundary passes.
- Every agent-interop boundary states a trust/permission scope that is the minimum required, with the business justification for any write-capable or irreversible-action scope.
- Every tool-output or task-response path that feeds an LLM's instruction plane has a named sanitization/validation step (schema validation, content-type allowlisting, injection scanning, or structured-output constraint) — no path trusts tool/agent output verbatim.
- Every capability-discovery endpoint (Agent Card, `tools/list`) feeding a routing/delegation decision is authenticated before that decision is made.
- Every diagram touching an agent-interop boundary compiles and annotates the boundary per `shared/diagramming-standards.md` (protocol, auth model, trust scope, visually distinct from a plain call).
- Every significant protocol/auth/trust-scope decision has a complete MADR ADR: context · ≥2 options · decision · consequences (positive AND negative).
- Emerging protocols (ANP and successors) appear only in the standards taxonomy as watch-list items — never required as a gate.
- **Gate:** hard oracles green (every boundary protocol-named, auth-modeled, trust-scoped, injection-boundary controlled, discovery authenticated, diagrams compile, ADRs complete) AND (design) rubric mean ≥ 0.8 across protocol-to-interaction fit, simplicity, and tradeoff honesty.

## Output
Write `artifacts/agent-interop-protocol/report.json` per `shared/report-format.md` — findings per `shared/finding-schema.md` (`type:"integration"` for an unnamed protocol or missing ADR, `type:"security"` for an unauthenticated boundary, unscoped trust grant, or un-sanitized tool-output-to-LLM path, `type:"risk"` for an accepted-but-monitored gap; `driver_ref` tracing each finding to the evolvability seam driver or the relevant security QAS), plus the verification block (fitness-function results, traceability coverage, hard-oracle results, rubric score) in design mode. Design artifacts land in `architecture/<slug>/`:

```
architecture/<slug>/
├── diagrams/agent-interop-boundary.md      # C4 boundary annotation: protocol (MCP/A2A), auth model, trust scope per edge
├── adr/NNNN-mcp-tool-boundary-*.md         # MADR: MCP vs alternative per agent↔tool boundary
```

## Guardrails
Per `shared/guardrails.md`: propose and evaluate — never write product-source files (`src/**`, `frontend/**`, `backend/**`), never write MCP server/client implementation code or A2A agent-runtime code, never apply infrastructure. Author only architecture artifacts (ADRs, C4-as-code diagrams, the tool-output-validation spec, fitness-function specifications) into `architecture/<slug>/`. Writing a fitness-function check script into the project's `tests/`/`fitness/` directory for CI enforcement is permitted — that is a verification artifact, not product behavior. The controller implements the chosen MCP/A2A integration.

Ground every decision in a driver: a heavyweight A2A task-lifecycle machine that no evolvability or delegation-autonomy driver demands is accidental complexity — but an agent-interop boundary with no stated protocol, auth model, or trust scope is a hard-oracle violation regardless of how "obviously fine" the integration looks today, because a project's own deferred-agentic-layer posture (per `project-architecture-config.md`) means this seam is exactly the kind of boundary that gets built out later without re-deriving the contract from scratch.

Reuse the named protocol before inventing: MCP and A2A between them cover the agent↔tool and agent↔agent integration space with known auth models, known message shapes, and a growing ecosystem of clients/servers — a bespoke JSON-over-HTTP agent-delegation mechanism invented for this project is a soft-oracle finding unless a stated driver rules out both.

One writer (`solution-architect`) authors all artifacts in sequence — never concurrent partial authoring. Proposers and the evaluator are read-only. The evaluator scores against traced scenarios and structural facts (is the protocol named, is the auth model real, is the trust scope minimal), not the writer's prose, and penalizes both under-specification (ambient trust dressed up as "internal, so it's fine") and over-engineering (a full A2A negotiation handshake for a boundary that is really one stateless tool call).

Bound the loop: ≤8 iterations of document→verify before escalating to the controller with the blocking finding.
