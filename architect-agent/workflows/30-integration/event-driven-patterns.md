# event-driven-patterns

**Group:** 30-integration  ·  **Runs as:** subagent: ../.claude/agents/integration-architect.md  ·  **Mode:** audit + design  ·  **Default model:** sonnet

## Purpose

Choose the event integration style — event notification vs event-carried-state-transfer vs event sourcing — and solve the dual-write problem. Answers the decision: *which event style satisfies the coupling/consistency driver, and how is the boundary between the database and the broker kept atomic?* A flow that stops at "use events" without resolving dual-write or schema evolution produces an architecture with a race condition at its core.

## Inputs & preconditions

- From `project-architecture-config.md`: the existing stack (e.g. a backend framework + a primary datastore, no broker today), the modular-monolith shape, the constraint against introducing a new broker/runtime without a driver-justified ADR, the compliance regime (if applicable — e.g. PII or financial data → audit log is a hard driver), and where architecture artifacts land (`architecture/<slug>/`).
- The architecture **drivers**: the prioritized quality-attribute scenarios from `workflows/00-drivers/quality-attribute-scenarios.md` that bear on integration — at minimum the consistency scenario (the core business-entity lifecycle must not lose an event when the DB write and the downstream notification are split across two systems), the auditability scenario (every state transition must be durably recorded for compliance, where applicable), and the evolvability scenario (any deferred integration seam named in `project-architecture-config.md` must be integrable without rewriting the core flow). A flow with no drivers from the utility tree downgrades scope and says so.
- The **functional scope**: which bounded contexts produce and consume events today (e.g. core entity lifecycle, audit logging, account/user management — per `project-architecture-config.md`) and which are near-term (e.g. documents, payments, closure/fulfillment). The event-style choice must handle both present and near-term consumers without requiring a rearchitecture.
- **Preconditions**: `workflows/00-drivers/quality-attribute-scenarios.md` run (utility tree exists); `workflows/30-integration/async-messaging.md` consulted for broker selection and delivery semantics — event-style selection is downstream of, not a substitute for, broker selection; `workflows/20-data/consistency-transactions.md` read for the consistency model already chosen per write path (the dual-write solution must be consistent with it).

## Oracle (source of truth)

- **hard:**
  - The event style (notification / carried-state-transfer / sourcing) is chosen against a named coupling/consistency driver from the utility tree — not by fashion. A style chosen without a traced driver is accidental complexity (`guardrails.md` §2).
  - The dual-write between the DB and the broker is **solved** — either via the outbox pattern (write event to a local `outbox` collection/table in the same database transaction, relay with a CDC/polling publisher) or transactional log tailing (a change-stream/CDC mechanism appropriate to the project's datastore → publisher). A bare two-write sequence (write DB, then publish to broker) with no atomicity guarantee is a hard-oracle failure: under a process crash between the two writes, the event is silently lost or duplicated with no recovery path — this defeats the audit-log compliance driver (where applicable) and any downstream consistency scenario.
  - Event schemas are versioned and backward-compatible: consumers written against schema v1 must tolerate v2 events (Postel's Law applied to event contracts); breaking changes require a new event type or a versioned topic/exchange. An unversioned event schema on a multi-consumer integration is a hard failure (fallacy 6 — no coordinated deploy).
  - The end-to-end flow for **choreography** designs is documented: for each business process (e.g. order placed → payment authorized → fulfillment assigned), the full event chain is drawn as a sequence/dynamic C4 diagram (diagrams-as-code, compiles, per `shared/diagramming-standards.md`) so the implicit flow is made explicit and the absence of a saga/process-manager is a deliberate, justified choice — not an oversight.
  - Every event consumer is **idempotent**: at-least-once delivery (the guaranteed delivery mode for any outbox/CDC relay) requires consumers to handle duplicate events without producing duplicate side effects. This is structurally checkable via a contract test or fitness function.
  - Every cross-service/cross-process event publication path has a failure mode defined: what happens when the relay (CDC publisher / outbox poller) is down? At-least-once guarantees require the relay to be retried with backoff; the consumer's idempotency key is the compensating control. No unbounded retry without backoff violates fallacy 1 (network reliability).
  - Every trust boundary crossed by an event has authn + authz controls: an internal domain event published inside a monolith module is trusted; an event crossing a network boundary to an external consumer or a deferred integration seam (per `project-architecture-config.md`) requires an authn scheme and PII scrubbing where the consumer is outside the PII trust zone.

- **soft:**
  - **Orchestration vs choreography chosen deliberately** — not by default. Choreography (decoupled, event-reactive) is the natural fit for decoupled notifications and eventual-consistency flows; orchestration (a visible saga/process-manager) is the right fit when the business process has many steps, compensations, timeouts, or a compliance requirement to observe the in-flight state. The choice is not "choreography because events," it is "choreography because [named driver]; orchestration rejected because [named cost]." A design that uses choreography for a 6-step core-entity lifecycle with compensation paths and no process visibility is a craft gap.
  - **Events carry the right amount of state** — not too thin, not too fat. An event notification (carry only the ID) forces every consumer to query back for data, creating chatty fan-out (violates fallacy 2 — latency is not zero) and tight temporal coupling (the producer must still be available when the consumer queries). An event-carried-state-transfer (carry enough payload for the consumer to act without a callback) trades some payload size for consumer autonomy. Full event sourcing (the event log *is* the system of record) buys audit + temporal queries but costs projection complexity, schema evolution burden, and is a one-way-door decision. The right point on the spectrum is traced to a driver, not picked by convention.
  - **Tradeoff honesty** — the ADR for each style decision records what it costs: outbox adds a collection + a relay process (operational surface area); event sourcing adds projection rebuilds + snapshotting; choreography adds implicit flow (observability burden). Every upside is paired with its cost.

## Standards & techniques

- **The three event styles** (Greg Young / Martin Fowler taxonomy) — each with its coupling/consistency contract:
  - *Event notification* — carries only the event type + entity ID; consumer queries back for data. Minimal coupling (consumer controls the read model) but chatty (N consumers × M events = N×M upstream queries); requires the producer to stay available for the callback window. Right when consumers need very different projections and data volume is low.
  - *Event-carried state transfer* — carries enough payload for the consumer to act autonomously (the "fat event"). Removes temporal coupling and the callback query; consumer can process offline. Costs: larger payload (fallacy 3 — bandwidth is not infinite), schema evolution complexity (every consumer must tolerate new fields), and PII-in-transit surface if the payload contains regulated data.
  - *Event sourcing* — the append-only event log *is* the primary store; state is derived by replaying events. Buys a complete audit trail (a hard driver where compliance applies — e.g. financial or PII data), temporal queries, and event replay for new projections. Costs: projection rebuilds on schema changes, snapshotting for performance at scale, significantly higher operational complexity. A one-way-door decision (reversing it means migrating data out of the log) — flag as such in the ADR; justify with a driver, not fashion.

- **Choreography vs orchestration** — the canonical tradeoff (see `shared/fallacies-and-tradeoffs.md`, "Canonical tradeoffs" section, Orchestration ↔ Choreography bullet):
  - *Choreography* — services/modules react to events; no central coordinator. Decoupled; each consumer evolves independently; fits well when the flow is simple or when each step truly has no dependency on the others completing in order. Costs: the end-to-end flow is implicit (observability burden — you cannot look at one place and see where the core entity is in its lifecycle); compensating transactions are distributed and hard to reason about; temporal assertions across steps require cross-log correlation.
  - *Orchestration* — a process manager (saga orchestrator) drives the flow: calls each step, handles timeouts, triggers compensations. Centrally visible; compensation logic is in one place. Costs: the orchestrator is a coupling point (it knows about every participant) and a potential SPOF if not made resilient; adds a new component to operate. Required when: the flow has many steps, compensations are non-trivial, the business needs to observe in-flight state (e.g. for an operator to see "order placed → payment authorized → awaiting fulfillment" as a visible status), or SLAs attach to individual steps.

- **The dual-write problem + the outbox pattern** (Chris Richardson / microservices.io):
  - *The problem*: writing to the database and then publishing to a broker in two separate operations is not atomic. A process crash, OOM, or broker unavailability between the two leaves the system in a split state — DB updated, event not published (silent data loss) or event published, DB not updated (phantom event). Under a compliance driver (audit log) this is a hard-oracle failure.
  - *Outbox pattern*: write the domain state and an outbox record to the same local transaction (for a document store: a multi-document transaction, or for single-document writes, an embedded outbox subdocument / a session-scoped write; for a relational store: an `outbox` table in the same transaction as the domain write). A separate relay process polls or tails the outbox and publishes to the broker with at-least-once semantics; on acknowledgement it marks the outbox record delivered. This makes the DB the source of truth for both state and the pending event publication — no race condition.
  - *CDC / change streams*: a change-data-capture mechanism native to the project's datastore (e.g. change streams on a document store, logical replication/WAL tailing on a relational store) can tail the transaction log and publish events without a polling outbox table — zero-latency relay, no extra collection/table. Cost: the change-stream consumer must be highly available (if it falls behind, the log retention window may be exceeded — a data loss risk on the relay side, not on the primary store). Typically requires a replicated deployment (not standalone). Where the project runs a single primary today, the CDC option needs an HA relay and a replicated deployment as prerequisites (surface as an open driver if not yet in place; confirm the current topology in `project-architecture-config.md`).
  - *2PC (two-phase commit)*: rejected for this use case — a database and a message broker do not typically share a distributed transaction coordinator; 2PC across a data store and a broker adds significant latency, a coordinator SPOF, and is operationally fragile. Record as a rejected alternative in the ADR.
  - *Dual-write (bare)*: write DB, then broker, no atomicity — rejected as the race condition is a hard-oracle failure under any consistency or audit driver.

- **Event schema design + evolution** (AsyncAPI / CloudEvents):
  - Schema-first: define the event schema in AsyncAPI (for message brokers) or as a CloudEvents envelope before consumers are written. The schema is the contract; version it explicitly (`event_type: "order.placed.v2"`).
  - **Backward compatibility** (Postel + fallacy 6): new optional fields are backward-compatible; removing or renaming fields is a breaking change requiring a new version. Consumers must tolerate unknown fields (liberal in what they accept).
  - **Schema registry** (Confluent / Apicurio / inline JSON Schema): enforces compatibility at publish time — prevents accidental breaking changes from reaching consumers. Where a broker is introduced (`workflows/30-integration/async-messaging.md`), the schema registry is the enforcement mechanism.
  - **Idempotency key**: every event carries a stable, unique ID (`event_id`). Consumers deduplicate on this key against a seen-set (a cache such as Redis, or a unique index on a `processed_events` store) to absorb the at-least-once duplicates from the outbox relay.

- **EIP catalog** (Hohpe & Woolf) patterns relevant here: *Message Channel* (topic/queue), *Event Message* (the event type), *Dead Letter Channel* (unprocessable events → DLQ for inspection, never silently dropped), *Idempotent Receiver*, *Claim Check* (carry a reference for large payloads; retrieve on demand — addresses fallacy 3).

- **ATAM evaluation** (`shared/evaluation-method.md`): the event-style choice is a **tradeoff point** — it helps one attribute (decoupling / audit / autonomy) and hurts another (simplicity / latency / observability). The evaluator must name the tradeoff point explicitly and score against the utility tree, not against the elegance of the diagram.

- **AKF scale cube** (Abbott and Fisher): event-driven integration is the Y-axis (functional decomposition by bounded-context / service) working with the X-axis (horizontal clone). It enables independent scaling of producers and consumers — surface only if a scale driver demands it; it is not a free argument for introducing a broker. (The Z-axis is data partitioning/sharding — splitting a data store by key range or tenant, not by functional domain.)

## Step sequence

- **audit:** Frame the drivers relevant to integration (consistency, audit-log compliance, evolvability seam) from the utility tree → inspect each cross-boundary write path in the existing codebase for dual-write races (bare DB-write + broker/log publish sequences) → identify any event schemas in use and check for versioning/compatibility discipline → check whether the end-to-end flow for any choreography sequence is documented or implicit → inspect consumers for idempotency controls → emit findings per `shared/finding-schema.md` ordered critical→cosmetic; flag every un-solved dual-write as `critical` (hard oracle), every unversioned schema on a multi-consumer path as `major`, every implicit choreography flow with no documentation as `major`. Read-only — no artifact authored, no source edited.

- **design:** Frame (extract/confirm the coupling+consistency+audit drivers from the utility tree; read the existing consistency model from `workflows/20-data/consistency-transactions.md`; confirm broker selection and delivery semantics from `workflows/30-integration/async-messaging.md`; read-only) → Explore (produce ≥2 genuinely divergent directions: e.g. [A] event-notification + choreography + outbox-poll relay vs [B] event-carried-state-transfer + orchestration + CDC change-stream relay vs [C] event-sourcing for the core aggregate only + notification for supporting contexts; each direction is a concrete spec covering: event style, choreography/orchestration choice, dual-write mechanism, schema evolution strategy, idempotency approach, and the C4 dynamic view of the flow; read-only) → Evaluate (skeptical evaluator scores each against the utility tree + the hard oracles above; locates sensitivity points — e.g. "outbox relay availability is sensitive to process uptime: a crash drops in-flight publications unless the relay is HA or persists state"; locates tradeoff points — e.g. "event-carried-state vs notification is a tradeoff point between consumer autonomy [↑ ECST] and payload/schema-evolution cost [↑ ECST]"; order-swapped to kill position bias; picks winner and grafts superior elements from runners-up; verifies STRIDE has been applied per trust boundary — internal module boundaries vs cross-network event consumers vs any deferred integration seam named in `project-architecture-config.md`; read-only) → Document (ONE writer authors: the event-style ADR (`adr/NNNN-event-style.md`), the dual-write ADR (`adr/NNNN-dual-write-outbox.md`), the orchestration-vs-choreography ADR (`adr/NNNN-orchestration-vs-choreography.md`), and the schema-evolution ADR (`adr/NNNN-schema-evolution.md`); updates `architecture/<slug>/tradeoffs.md` with the sensitivity/tradeoff register from the evaluator) → Verify (diagram-compile: all .puml/.md diagrams render without error; traceability: every ADR's `driver_ref` points at a real QAS; dual-write fitness function: grep or AST check — no path in the codebase writes to the data store and then writes to the broker without the outbox intermediary; schema-backward-compat gate: schema registry compatibility check or a test asserting new-schema events parse under old-schema consumer; consumer idempotency: contract test asserts replaying the same `event_id` twice produces the same outcome; choreography-flow-documented gate: presence check that every multi-step business process has a corresponding dynamic diagram; ## Assertions & exit gate

- Every cross-boundary write that must produce an event uses the outbox pattern or CDC — no bare dual-write sequence exists anywhere on a path where event loss would violate a consistency or compliance driver. Traceable by a fitness function (static analysis or AST check) in `architecture/<slug>/fitness-functions.md`.
- The event style (notification / carried-state-transfer / sourcing) is recorded in an ADR with: the driver it satisfies, ≥2 rejected alternatives with their costs, and the positive + negative consequences. An ADR with no driver reference or no negative consequences fails the `adrs_complete` hard oracle.
- The choreography-vs-orchestration choice is recorded in its own ADR (they are separate decisions); any choreography design includes a C4 dynamic/sequence diagram for every multi-step business process making the end-to-end flow explicit (not implicit in individual service reaction handlers).
- All event schemas are versioned (`event_type` carries a version suffix or a version field); the schema evolution strategy (backward-compatible-only, new-type-on-break, registry-enforced) is documented and the registry (or equivalent) is named.
- Every consumer is demonstrably idempotent: the `event_id` deduplication path exists and is covered by a contract test referenced in `architecture/<slug>/fitness-functions.md`.
- All diagrams compile and render without error (Mermaid/PlantUML); every container in the L2 view that is drilled into has a matching L3 or dynamic view; sync vs async relationships are visually distinct (solid vs dashed per `shared/diagramming-standards.md`).
- The dual-write ADR explicitly records 2PC and bare dual-write as rejected alternatives with their failure modes named.
- Any deferred integration seam named in `project-architecture-config.md` (e.g. a future agent layer, a partner integration, a downstream analytics consumer) is identified — a named boundary (an anti-corruption layer boundary or a dedicated topic/channel) that the current design does not cross but which the future consumers will attach to without requiring changes to the core event producers.
- **STRIDE per trust boundary** (`shared/evaluation-method.md`): every trust boundary crossed by an event (internal module → broker, broker → external consumer, broker → any deferred integration seam) has a completed STRIDE analysis. At minimum: Spoofing (authn scheme on the event consumer), Tampering (integrity of event payload in transit), Repudiation (durable audit log of event publication), Information Disclosure (PII scrubbing or encryption for consumers outside the PII trust zone), Denial of Service (backpressure + DLQ for unprocessable events), Elevation of Privilege (authz — consumers can only subscribe to topics they are authorised for). STRIDE gaps at a cross-network boundary are a blocking hard-oracle failure.
- **Gate:** hard oracles green (dual-write solved + event style driver-traced + schemas versioned + consumers idempotent + choreography flows documented + diagrams compile + ADRs complete [event-style, dual-write-outbox, orchestration-vs-choreography, schema-evolution] + STRIDE applied per trust boundary with authn/authz and PII controls named at every cross-network event path) AND rubric mean ≥ 0.8 (orchestration/choreography chosen deliberately; events carry the right state; tradeoff honesty in every ADR).

## Output

Write `artifacts/event-driven-patterns/report.json` per `shared/report-format.md` — findings/risks per `shared/finding-schema.md` (type: `"integration"`; oracle: `"hard"` for un-solved dual-write / unversioned schema / undocumented choreography flow / non-idempotent consumer; `"soft"` for choreography-by-default without deliberate justification / chatty notification style where the latency driver demands ECST); the verification block for design mode (fitness-function results: no-bare-dual-write check, schema-backward-compat, consumer-idempotency-contract; diagram-compile results; traceability coverage: every ADR driver_ref checked against utility tree; rubric scores per soft criterion).

Authored design artifacts land in `architecture/<slug>/`:

## Guardrails

Per `shared/guardrails.md`: propose & evaluate — never write product code, never configure a live broker, never run a migration. The outbox collection design and the CDC relay configuration are ADR artifacts; the controller implements them. Every decision traces to a driver (no driver = accidental complexity; flag and drop). Reuse the EIP catalog and outbox/CDC patterns before inventing a bespoke relay. One writer (`solution-architect`) authors the ADRs and diagrams; the `integration-architect` advises read-only and verifies the integration dimension. The evaluator independently re-checks every hard oracle from the artifacts — it never grades the writer's prose. Never write to project memory — lessons go in the ADRs and the tradeoffs register. Cross-reference `workflows/30-integration/async-messaging.md` (broker selection + delivery semantics — prerequisites for this flow), `workflows/20-data/consistency-transactions.md` (the consistency model per write path — must be reconciled with the dual-write solution chosen here), and `workflows/80-patterns/distributed-systems-patterns.md` (the saga/outbox/CDC pattern contracts and the idempotency receiver pattern).
