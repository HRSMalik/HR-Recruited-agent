# data-quality

**Category:** 20-interface-data
**Runs as:** subagent: ../.claude/agents/data-quality-validator.md
**Default model:** sonnet   ·   **Mode:** read-only

## Purpose
Score data against the six DQ dimensions — completeness, accuracy, consistency, timeliness, validity, uniqueness — using declarative expectation/contract checks (Great Expectations / Soda / dbt-tests), plus freshness and distribution profiling. Goes beyond `database-validation`'s row-count + checksum reconciliation: this owns the dimension framework and drift/freshness detection. Read-only on live; profiling never mutates.

## Inputs & preconditions
- Required artifacts: the **declared expectations / data contract** (e.g. a Great Expectations suite `expectation_suite.json`, a Soda `checks.yml`, or dbt `schema.yml` tests) naming per-column expectations, freshness SLAs, and a profiling **baseline** (golden distribution snapshot) for drift comparison.
- Target: connection string + read credentials (via env) for the live table/warehouse; the expectation suite version pinned to the build under test.
- Preconditions: connection succeeds; host/DB asserted NON-PROD or read-replica before profiling; the expectation suite + baseline snapshot are present and version-matched to the SUT; no expectations are auto-generated from the SUT's current data (that would encode current state as truth — see oracle).

## Oracle (source of truth)
The **declared expectations / data contract** — the external, human-authored suite that states what each column *must* satisfy (non-null %, allowed set/regex, type/range, uniqueness, FK consistency, freshness SLA) and the **baseline distribution** for drift. NEVER profile the live table and assert it matches itself; a GE/Soda suite generated by profiling production is not an oracle. Per https://greatexpectations.io, Expectations are the verifiable, declarative assertions that encode intended data quality.

## Step sequence (Plan → Act → Verify)
1. **Plan** — map each contract assertion to its dimension: **completeness** (`expect_column_values_to_not_be_null`, required-field coverage %), **validity** (type/range/regex/allowed-set, e.g. email, ISO date, enum status), **uniqueness** (`expect_column_values_to_be_unique`, PK/business-key dup rate), **consistency** (cross-field + cross-table referential agreement, format standardization), **accuracy** (values match an authoritative reference/lookup, not just well-formed), **timeliness/freshness** (max load/event timestamp within the SLA window). Enumerate distribution profiles to compare against the baseline (null %, min/max, cardinality, category frequencies, mean/stddev).
2. **Act** — run the expectation suite read-only via GE/Soda/dbt against the live table (SELECT/profiling queries only). Compute the freshness lag = now − max(updated_at). Profile each column and diff its distribution against the baseline (e.g. PSI / null-rate delta / new-or-missing categories).
3. **Verify** — assert every expectation in the suite passes at its declared threshold; freshness lag ≤ SLA; distribution drift below the contract's tolerance. Classify each failed expectation per `shared/finding-schema.md` (`real_bug` vs `test_assertion_issue` — a too-strict expectation that encodes outdated reality is a contract fix, not a data defect).

## Assertions & exit gate
- Completeness: required-field non-null coverage ≥ contract threshold; no required column below its declared % .
- Validity: 0 values violating declared type/range/regex/allowed-set.
- Uniqueness: PK/business-key duplicate rate == 0 (or ≤ declared tolerance).
- Consistency: cross-field and referential rules hold; formats standardized per contract.
- Accuracy: sampled values reconcile against the authoritative reference/lookup.
- Timeliness: freshness lag ≤ the SLA window for every dataset checked.
- Distribution: no profiled column drifts beyond the contract's tolerance vs baseline.
- **Gate:** `dq_dimensions_pass` — all six dimensions pass at declared thresholds AND freshness ≤ SLA AND drift within tolerance.

## Output
Write `artifacts/data-quality/report.json` per `shared/report-format.md`:
`{ flow, status, summary{total,passed,failed,skipped}, findings[], gate{name:"dq_dimensions_pass",passed} }`, plus a per-dimension scorecard (pass/fail + score per dimension) and the freshness lag + drift deltas in `notes` or an attached profile artifact.
Each finding follows `shared/finding-schema.md`; `oracle` = the expectation id + dimension (e.g. `suite:candidates.email expect_column_values_to_match_regex [validity]`). Evidence = the failing expectation, its declared vs observed value, the offending count + sample violating keys (PK only, no PII), and the baseline-vs-current distribution delta for drift findings. Record the suite version + baseline snapshot id for determinism.

## Watch (do not gate)
Emerging: ML-based anomaly detection on profiles and contract auto-negotiation between producer/consumer — note any signal observed, but never let it fail the gate; only declared-contract violations gate.

## Guardrails
Per `shared/guardrails.md`: read-only on live (`disallowedTools: Edit, Write`; SELECT/profiling queries only — the PreToolUse hook blocks INSERT/UPDATE/DELETE/DROP/TRUNCATE). Assert NON-PROD or a read-replica before profiling large tables; respect query cost/rate limits and sample where the table is large. No real PII in evidence — emit keys/counts/distribution stats, never row contents. Connection string redacted. Cap turns.
